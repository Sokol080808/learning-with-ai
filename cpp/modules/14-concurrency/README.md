# Модуль 14 — Введение в многопоточность

Современные процессоры — многоядерные, и junior должен хотя бы понимать основы
параллелизма и его главную опасность. Большая идея: несколько потоков, меняющих **одни
и те же** данные без синхронизации, — это **гонка данных** (data race), то есть
неопределённое поведение. Почти весь модуль — про то, как этого избежать.

## Идея 1. Поток (`std::thread`)

```cpp
#include <thread>

void work(int id) { /* ... */ }

std::thread t(work, 1);   // запускает work(1) в отдельном потоке
t.join();                 // дождаться завершения (ОБЯЗАТЕЛЬНО до разрушения t)
```
Каждый `std::thread` нужно либо `join()` (дождаться), либо `detach()` (отпустить) —
иначе деструктор потока завершит программу. Обычно — `join()`.

## Идея 2. Гонка данных — корень зла

```cpp
long counter = 0;
auto inc = [&]{ for (int i = 0; i < 100000; ++i) ++counter; };  // ОПАСНО
std::thread a(inc), b(inc);
a.join(); b.join();
// counter, скорее всего, НЕ 200000 — операции потоков «перемешались»
```
`++counter` — это на самом деле «прочитать, прибавить, записать». Два потока могут
прочитать одно значение и затерять инкремент друг друга. Результат недетерминирован.

## Идея 3. Мьютекс (`std::mutex`) — взаимное исключение

Мьютекс гарантирует, что критическую секцию в каждый момент выполняет только один поток:

```cpp
#include <mutex>
std::mutex m;
long counter = 0;

auto inc = [&]{
    for (int i = 0; i < 100000; ++i) {
        std::lock_guard<std::mutex> lock(m);  // захват; освободится в конце блока (RAII!)
        ++counter;
    }
};
```
`std::lock_guard` — это RAII для мьютекса: захватывает в конструкторе, освобождает в
деструкторе. Не вызывай `lock()`/`unlock()` руками — забудешь `unlock` при исключении.
Видишь, как RAII (модуль 04) спасает и здесь?

## Идея 4. Атомарные переменные (`std::atomic`)

Для простых счётчиков мьютекс избыточен. `std::atomic<T>` делает операции неделимыми:

```cpp
#include <atomic>
std::atomic<long> counter{0};
auto inc = [&]{ for (int i = 0; i < 100000; ++i) ++counter; };  // безопасно и быстро
```
`++counter` для атомика гарантированно «прочитать-прибавить-записать» как одна операция.

## Идея 5. Деление работы и `std::async`

Часто потоки не делят данные, а **разбивают работу на независимые куски** — тогда
синхронизация почти не нужна: каждый поток считает свою часть, потом складываем
результаты. `std::async` упрощает «посчитать в фоне и забрать результат»:

```cpp
#include <future>
std::future<long> f = std::async(std::launch::async, [&]{ return heavy(); });
long result = f.get();   // дождаться и получить результат
```

## Задания

Интерфейс — `include/task14.hpp`, реализация — `src/task14.cpp`.

1. `long parallel_sum(const std::vector<long>& v, unsigned num_threads)` — посчитать
   сумму, **разделив** массив на `num_threads` примерно равных частей, по потоку на
   часть, затем сложить частичные суммы. Здесь общие данные только читаются — гонки
   нет. (`num_threads >= 1`; считай, что не больше размера массива в тестах.)
2. `long concurrent_increment(unsigned num_threads, unsigned per_thread)` — запустить
   `num_threads` потоков, каждый увеличивает **общий** счётчик `per_thread` раз.
   Защити счётчик `std::mutex` + `std::lock_guard`. Верни итог (== `num_threads*per_thread`).
3. `long atomic_increment(unsigned num_threads, unsigned per_thread)` — то же, но через
   `std::atomic<long>` **без** мьютекса.

### Подсказки

<details><summary>parallel_sum — разбиение</summary>
Посчитай размер куска `chunk = v.size() / num_threads`. Поток k обрабатывает диапазон
`[k*chunk, (k+1)*chunk)`, а последний поток забирает «хвост» до конца. Храни частичные
суммы в `std::vector<long> partials(num_threads)` — каждый поток пишет в СВОЙ элемент
(разные элементы — гонки нет). После `join()` всех — сложи partials.
</details>
<details><summary>Почему задание 2 и 3 дают один результат, но по-разному</summary>
И мьютекс, и атомик устраняют гонку, но разной ценой. Подумай (и спроси меня): какой из
них быстрее для простого счётчика и почему? Когда мьютекс всё же необходим?
</details>

Важно: если убрать синхронизацию в задании 2/3, тест может *иногда* проходить случайно —
это и есть коварство гонок. Поэтому корректность тут — про синхронизацию, а не про
«получилось один раз». Обсудим, если интересно, как такие баги ловят (ThreadSanitizer).
