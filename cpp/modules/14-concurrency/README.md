# Модуль 14 — Введение в многопоточность

Современные процессоры — многоядерные, и junior должен хотя бы понимать основы
параллелизма и его главную опасность. Большая идея: несколько потоков, меняющих **одни
и те же** данные без синхронизации, — это **гонка данных** (data race), то есть
неопределённое поведение. Почти весь модуль — про то, как этого избежать, и про
инструменты, которыми мы наводим порядок: `std::thread`, `std::mutex`,
`std::condition_variable` и `std::atomic`.

## Идея 1. Поток (`std::thread`): что это и как им управлять

Поток (thread) — это независимая последовательность исполнения внутри одного процесса.
Все потоки процесса делят общее адресное пространство: ту же кучу, те же глобальные
переменные, те же открытые файлы. Делят они почти всё, **кроме** собственного стека и
набора регистров. Именно общая память делает потоки одновременно мощными (обмен данными
бесплатен — просто читаем общую переменную) и опасными (два потока могут испортить друг
другу данные).

```cpp
#include <thread>

void work(int id) { /* ... */ }

std::thread t(work, 1);   // ОС создаёт поток и сразу запускает work(1)
t.join();                 // дождаться завершения (ОБЯЗАТЕЛЬНО до разрушения t)
```

Конструктор `std::thread` **копирует** (точнее — `decay`-копирует) свои аргументы во
внутреннее хранилище, а уже оттуда передаёт их функции. Это важная тонкость: если
функция принимает `int& x`, то по умолчанию она получит ссылку на **копию**, а не на
твою переменную. Чтобы передать настоящую ссылку, оборачивай аргумент в
`std::ref(x)`. Аналогично для передачи объекта по перемещению — `std::move`.

### Механика join/detach и почему это не опциональный жест

У каждого `std::thread` есть два конечных состояния, в которых его законно разрушать:
**joined** и **detached**. Если объект `std::thread`, всё ещё связанный с живым потоком
(`joinable() == true`), доходит до своего деструктора, тот вызывает
`std::terminate()` — программа аварийно падает. Это сделано намеренно: язык не знает,
хотел ли ты дождаться поток или отпустить его, поэтому отказывается «угадывать».

- `join()` — блокирует текущий поток, пока дочерний не завершится; после этого `t`
  больше не joinable. Используется в 99% случаев.
- `detach()` — разрывает связь: поток продолжает жить «сам по себе», его ресурсы
  освободит рантайм по завершении. Опасность: если detached-поток обращается к
  локальным переменным функции, которая уже вернула управление, — это обращение к
  мёртвому стеку (use-after-free).

Самая частая ошибка — забыть `join()` на пути исключения:

```cpp
void bad() {
    std::thread t(work, 1);
    may_throw();          // если бросит — t.join() ниже не выполнится
    t.join();             // -> деструктор t вызовет std::terminate()
}
```

Правильно — обернуть в RAII (или в C++20 использовать `std::jthread`, который сам
делает `join()` в деструкторе). В этом модуле мы остаёмся на `std::thread` и аккуратно
джойним вручную:

```cpp
std::vector<std::thread> pool;
for (unsigned k = 0; k < n; ++k) pool.emplace_back(work, k);
for (auto& t : pool) t.join();   // джойним ВСЕ, даже если что-то пошло не так
```

Ещё одна ловушка — «забыть имя переменной потока»:

```cpp
std::thread(work, 1);   // ВРЕМЕННЫЙ объект: создан и тут же разрушается -> terminate()
```

Временный `std::thread` joinable, его деструктор срабатывает в конце выражения — и
программа падает. Всегда давай потоку имя и джойни его.

## Идея 2. Гонка данных — корень зла

```cpp
long counter = 0;
auto inc = [&]{ for (int i = 0; i < 100000; ++i) ++counter; };  // ОПАСНО
std::thread a(inc), b(inc);
a.join(); b.join();
// counter, скорее всего, НЕ 200000 — операции потоков «перемешались»
```

Гонка данных (data race) по стандарту — это когда два потока обращаются к одной
ячейке памяти, **хотя бы одно** обращение — запись, и между ними **нет отношения
happens-before** (нет синхронизации). Результат — **неопределённое поведение** (UB).
Это не «иногда неверное число»: формально компилятор вправе сделать что угодно.

Почему теряются инкременты? `++counter` на уровне процессора — это три отдельных шага:

```
load   R <- counter     ; прочитать в регистр
add    R <- R + 1        ; прибавить
store  counter <- R      ; записать обратно
```

Два потока могут прочитать одно и то же значение `5`, оба прибавить и оба записать `6` —
один инкремент потерян. Это называется **read-modify-write race**, или «потерянное
обновление».

Коварство гонок в том, что они **недетерминированы**: тест может случайно пройти
тысячу раз и упасть на тысяча первом, на другой машине, под нагрузкой или в release-
сборке (оптимизатор меняет порядок инструкций). Поэтому «получилось один раз» — не
доказательство корректности. Ловят такие баги не глазами, а инструментами:
**ThreadSanitizer** (`g++ -fsanitize=thread`) инструментирует обращения к памяти и
кричит при первой же гонке. Привыкай прогонять многопоточный код под TSan.

Важно отделять **гонку данных** (UB, лечится синхронизацией) от **состояния гонки**
(race condition) — логической ошибки «результат зависит от того, кто успел первым».
Второе бывает даже при идеальной синхронизации, если сама логика предполагает
определённый порядок. Синхронизация убирает первое; правильный дизайн — второе.

## Идея 3. Мьютекс (`std::mutex`) — взаимное исключение

Мьютекс (mutual exclusion) — это «турникет»: пройти может только один поток за раз.
Поток, желающий войти в **критическую секцию**, делает `lock()`; если мьютекс уже
захвачен — поток **засыпает** (ОС снимает его с ядра) до освобождения. Выходя, поток
делает `unlock()`, и ОС будит одного из ждущих.

```cpp
#include <mutex>
std::mutex m;
long counter = 0;

auto inc = [&]{
    for (int i = 0; i < 100000; ++i) {
        std::lock_guard<std::mutex> lock(m);  // захват; освободится в конце блока (RAII!)
        ++counter;
    }
};
```

`std::lock_guard` — это RAII для мьютекса: захватывает в конструкторе, освобождает в
деструкторе, в том числе при выходе по исключению. **Не вызывай `lock()`/`unlock()`
руками** — рано или поздно забудешь `unlock` на каком-то пути и получишь намертво
зависшую программу. Видишь, как RAII (модуль 04) спасает и здесь?

### lock_guard vs unique_lock

В стандарте есть две основные RAII-обёртки:

- `std::lock_guard<std::mutex>` — минимальная и самая быстрая. Захватывает в
  конструкторе, держит до конца области видимости, отпускает в деструкторе. Ничего
  больше не умеет — и это её достоинство.
- `std::unique_lock<std::mutex>` — гибкая: умеет отложенный захват
  (`std::defer_lock`), ручные `lock()`/`unlock()` посреди жизни, передачу владения
  (move). **Главное:** только `unique_lock` совместим с `condition_variable` (см.
  Идею 4), потому что cv должна уметь временно отпустить мьютекс, пока поток спит.

Правило: бери `lock_guard` по умолчанию; переходи на `unique_lock`, только когда нужна
его гибкость (условные переменные, ручное отпускание).

```cpp
std::mutex m;
std::unique_lock<std::mutex> lk(m, std::defer_lock);  // ещё НЕ захвачен
prepare();                                            // что-то вне критической секции
lk.lock();                                            // захватываем здесь
/* критическая секция */
lk.unlock();                                          // можно отпустить досрочно
```

### Частые ошибки с мьютексами

1. **Слишком крупная критическая секция.** Если держать `lock` дольше необходимого,
   потоки выстраиваются в очередь и параллелизм испаряется. Захватывай мьютекс на
   минимальный участок — только вокруг общего состояния.

2. **Забыть, что `const`-метод тоже меняет мьютекс.** Геттер логически «не меняет»
   объект, но захват мьютекса физически меняет его поле. Решение — пометить мьютекс
   `mutable`:

   ```cpp
   class Counter {
       mutable std::mutex m_;     // mutable: можно лочить в const-методе
       long value_ = 0;
   public:
       long get() const { std::lock_guard<std::mutex> l(m_); return value_; }
   };
   ```

3. **Защитить запись, но не чтение.** Если один поток пишет под мьютексом, а другой
   читает **без** него — это всё ещё гонка данных (UB). Под защитой должны быть **все**
   обращения к общему изменяемому состоянию, и чтения тоже.

4. **Рекурсивный самозахват.** Обычный `std::mutex` нельзя захватить дважды в одном
   потоке — это deadlock (см. Идею 6). Если метод под локом вызывает другой метод,
   тоже берущий тот же лок, — вы повисли. Не вкладывай захваты одного мьютекса.

## Идея 4. Условная переменная (`std::condition_variable`)

Мьютекс отвечает на вопрос «можно ли мне трогать данные?». Условная переменная отвечает
на другой: «как **подождать**, пока данные придут в нужное состояние, не сжигая
процессор?». Наивный busy-wait — это катастрофа:

```cpp
while (queue.empty()) { }   // ПЛОХО: 100% загрузка ядра впустую
```

`condition_variable` позволяет потоку **уснуть** до тех пор, пока другой поток не
скажет «проснись, состояние изменилось». Связка всегда тройная: общие данные +
`std::mutex` + `std::condition_variable`.

```cpp
std::mutex m;
std::condition_variable cv;
std::queue<int> q;

// Поток-потребитель: ждёт, пока появится элемент.
std::unique_lock<std::mutex> lk(m);
cv.wait(lk, [&]{ return !q.empty(); });   // спит, ПОКА предикат ложен
int x = q.front(); q.pop();

// Поток-производитель: кладёт и будит.
{ std::lock_guard<std::mutex> l(m); q.push(42); }
cv.notify_one();                          // разбудить одного ждущего
```

### Что происходит под капотом у `wait`

`cv.wait(lk, pred)` — это аккуратно собранная атомарная операция, эквивалентная:

```cpp
while (!pred()) cv.wait(lk);   // версия без предиката
```

а сам `cv.wait(lk)` делает три вещи **неразрывно**: (1) отпускает мьютекс `lk`, (2)
усыпляет поток в очереди ожидания cv, (3) при пробуждении заново захватывает `lk` —
и только потом возвращает управление. Отпускание мьютекса критично: иначе производитель
не смог бы захватить тот же мьютекс, чтобы изменить данные и позвать `notify`.

### Почему предикат обязателен: spurious wakeups и потерянные сигналы

Условная переменная имеет право **проснуться сама по себе**, без всякого `notify` —
это так называемое **ложное пробуждение** (spurious wakeup), допускаемое стандартом
(оно реально случается на POSIX). Поэтому проверять условие надо **в цикле**, а не
один раз. Перегрузка `wait(lk, pred)` именно это и делает: проснувшись, она снова
вычисляет `pred()` и, если он ложен, засыпает обратно. **Никогда** не пиши голый
`if (q.empty()) cv.wait(lk);` — пиши `cv.wait(lk, [&]{ return !q.empty(); });`.

Вторая причина предиката — **потерянный сигнал** (lost wakeup). Если производитель
вызвал `notify_one()` в тот момент, когда ни один поток ещё не спит на cv, сигнал
пропадает в никуда. Но данные-то уже в очереди! Предикат спасает: потребитель, дойдя
до `wait`, сначала проверит `!q.empty()` — увидит, что элемент уже есть, и **вообще не
уснёт**. Без предиката он уснул бы навсегда.

### notify_one vs notify_all

- `notify_one()` будит **одного** произвольного ждущего. Достаточно, когда любой из
  потоков одинаково способен обработать событие (например, один новый элемент в
  очереди — нужен ровно один потребитель).
- `notify_all()` будит **всех**. Нужно, когда событие меняет ситуацию для всех сразу.
  Классический пример — закрытие очереди: «данных больше не будет» должны узнать все
  спящие потребители, чтобы корректно завершиться, а не висеть вечно.

Тонкость порядка: захватывай мьютекс при **изменении** общего состояния, но сам
`notify` можно (и часто полезно) звать **после** освобождения мьютекса — иначе
разбуженный поток проснётся лишь для того, чтобы упереться в ещё занятый мьютекс.

## Идея 5. Атомарные переменные (`std::atomic`)

Для простых счётчиков мьютекс избыточен. `std::atomic<T>` делает базовые операции
**неделимыми** на аппаратном уровне — без перевода потока в сон:

```cpp
#include <atomic>
std::atomic<long> counter{0};
auto inc = [&]{ for (int i = 0; i < 100000; ++i) ++counter; };  // безопасно и быстро
```

`++counter` для атомика гарантированно выполняется как одна неразрывная операция
read-modify-write (на x86 это инструкция вроде `lock xadd`). Гонки нет, потому что
аппаратура не даёт двум ядрам «вклиниться» в середину.

### Мьютекс или атомик?

- **Атомик** хорош для одиночных скалярных операций: счётчик, флаг, указатель. Он
  *lock-free* (обычно не усыпляет поток) и потому быстрее под низкой конкуренцией.
- **Мьютекс** нужен, когда критическая секция охватывает **несколько** действий,
  которые должны быть согласованы как одно целое (изменить два поля сразу, проверить-
  и-вставить в контейнер, обновить инвариант структуры данных). Атомик защищает одну
  ячейку — мьютекс защищает **участок логики**.

Важно: `std::atomic<long> a; a = a + 1;` — это уже **не** атомарно (это отдельные
чтение и запись с гонкой между ними). Используй цельные операции: `++a`,
`a.fetch_add(1)`, `a.compare_exchange_strong(...)`.

## Идея 6. Взаимная блокировка (deadlock)

Deadlock — это вечное ожидание по кругу: поток A держит ресурс 1 и ждёт ресурс 2, а
поток B держит ресурс 2 и ждёт ресурс 1. Никто не уступит — программа висит навсегда,
не падая (чем deadlock коварнее краша: его не видно в логе).

```cpp
std::mutex m1, m2;

// Поток A                         // Поток B
std::lock_guard l1(m1);            std::lock_guard l2(m2);
std::lock_guard l2(m2);  // ждёт   std::lock_guard l1(m1);  // ждёт
//   -> ОБА ВИСЯТ
```

Классические условия Коффмана: для deadlock одновременно нужны взаимное исключение,
удержание-и-ожидание, отсутствие вытеснения и **круговое ожидание**. Достаточно
разорвать любое — и тупик невозможен. На практике проще всего убить круговое ожидание.

### Как избегать

1. **Единый порядок захвата.** Если все потоки всегда берут мьютексы в одном и том же
   глобальном порядке (сначала `m1`, потом `m2`), цикла в графе ожидания не возникнет.

2. **Захватывать сразу несколько — атомарно.** Когда нужно держать два мьютекса
   одновременно, используй `std::scoped_lock` (C++17), который захватывает их **без
   риска тупика** (через алгоритм с откатом):

   ```cpp
   std::scoped_lock lk(m1, m2);   // берёт оба сразу, порядок не важен, deadlock-free
   ```

   Эквивалент на старом API — `std::lock(m1, m2)` плюс `lock_guard` с `adopt_lock`.

3. **Не вызывать чужой код под локом.** Колбэк, вызванный внутри критической секции,
   может попытаться взять другой (или тот же) мьютекс — и вы в тупике. Минимизируй то,
   что делается под захватом.

4. **Не лочить один и тот же `std::mutex` дважды в одном потоке** — это вырожденный
   deadlock с самим собой (см. ошибку 4 в Идее 3).

2. `long concurrent_increment(unsigned num_threads, unsigned per_thread)` — запустить

   Защити счётчик `std::mutex` + `std::lock_guard`. Верни итог (== `num_threads*per_thread`).
3. `long atomic_increment(unsigned num_threads, unsigned per_thread)` — то же, но через

   синхронизация спрятана **внутри** объекта (`mutable std::mutex` + `lock_guard`),
   чтобы любое число потоков могло звать методы одновременно и не терять обновлений.
   Геттер тоже должен читать под защитой.
5. **Producer–consumer.** Реализуй класс `BlockingQueue` (`push`, `pop`, `close`) на
   `std::mutex` + `std::condition_variable`: `pop()` **блокируется**, пока очередь
   пуста и не закрыта; на пустой **закрытой** очереди возвращает `std::nullopt`.
   Затем напиши `long producer_consumer_sum(num_producers, num_consumers,
   items_per_producer)`: producer'ы кладут `1..items_per_producer`, consumer'ы
   разбирают и суммируют. Итог должен совпасть с суммой всего положенного.
6. `long parallel_sum_balanced(const std::vector<long>& v, unsigned num_threads)` — как
   задание 1, но обязано работать при **любом** `num_threads >= 1`: больше размера

   потерялся и не посчитался дважды. Результат — в точности как у обычной суммы.

Посчитай размер куска `chunk = v.size() / num_threads`. Поток k обрабатывает диапазон
`[k*chunk, (k+1)*chunk)`, а последний поток забирает «хвост» до конца. Храни частичные
суммы в `std::vector<long> partials(num_threads)` — каждый поток пишет в СВОЙ элемент
(разные элементы — гонки нет). После `join()` всех — сложи partials.
</details>

<details><summary>Почему задания 2 и 3 дают один результат, но по-разному</summary>
И мьютекс, и атомик устраняют гонку, но разной ценой. Подумай (и спроси меня): какой из
них быстрее для простого счётчика и почему? Когда мьютекс всё же необходим? (Подсказка:
посмотри Идею 5 — атомик защищает одну ячейку, мьютекс — участок логики.)
</details>

<details><summary>SafeCounter — с чего начать (задание 4)</summary>
Нужны два поля: `mutable std::mutex m_;` и `long value_ = 0;`. Каждый публичный метод
начинается с захвата: `std::lock_guard<std::mutex> lock(m_);`. `increment()` проще
всего выразить через `add(1)`. Не забудь: `get()` помечен `const`, и именно поэтому
мьютекс должен быть `mutable` — иначе его нельзя будет залочить в const-методе.
</details>

<details><summary>SafeCounter — а зачем мьютекс, если это просто +=? (задание 4)</summary>
Потому что `value_ += delta` — это read-modify-write (Идея 2): прочитать, прибавить,
записать. Без защиты два потока затрут инкременты друг друга. Тест
`ConcurrentIncrementsAreNotLost` запускает 8 потоков по 50000 — без `lock_guard` итог
почти наверняка окажется меньше 400000.
</details>

<details><summary>BlockingQueue::pop — структура ожидания (задание 5)</summary>
Возьми `std::unique_lock<std::mutex> lock(m_);` (именно unique_lock — он нужен cv).
Затем `cv_.wait(lock, [&]{ return !q_.empty() || closed_; });` — поток спит, пока
очередь пуста И не закрыта. После пробуждения: если в очереди есть элемент — сними его
с фронта и верни; иначе (очередь пуста и closed_) — верни `std::nullopt`. Перечитай
Идею 4 про то, почему предикат обязателен.
</details>

<details><summary>BlockingQueue::push / close — про notify (задание 5)</summary>
`push`: под `lock_guard` положи значение в `q_`, затем `cv_.notify_one()` (один новый
элемент — будим одного потребителя). `close`: под `lock_guard` выставь `closed_ = true`,
затем `cv_.notify_all()` — все спящие потребители должны узнать, что данных больше не
будет, иначе повиснут навсегда. Почему здесь именно `notify_all`, а в `push` —
`notify_one`? См. Идею 4.
</details>

<details><summary>producer_consumer_sum — оркестровка потоков (задание 5)</summary>
Создай одну `BlockingQueue q`. Запусти `num_producers` потоков-производителей (каждый
`push`'ит `1..items_per_producer`) и `num_consumers` потоков-потребителей. Потребитель
крутит цикл: `while (auto x = q.pop()) local_sum += *x;` — выходит, когда `pop()` вернул
`nullopt`. Порядок завершения критичен: сперва `join()` ВСЕХ производителей, ПОТОМ
`q.close()`, и только затем `join()` потребителей. Если закрыть очередь раньше времени —
потеряешь данные; если не закрыть вовсе — потребители повиснут на `pop()`. Суммы
потребителей храни в `std::vector<long>` по индексу (каждый в свой слот — гонки нет),
в конце сложи.
</details>

<details><summary>parallel_sum_balanced — корректное разбиение остатка (задание 6)</summary>
Пусть `n = v.size()`, `base = n / num_threads`, `rem = n % num_threads`. Тогда первые
`rem` потоков получают по `base + 1` элементу, остальные — по `base`. Веди бегущий
указатель `start`: для потока k длина куска `len = base + (k < rem ? 1 : 0)`, диапазон
`[start, start + len)`, после чего `start += len`. Потоки с пустым куском просто дают 0.
Проверь мысленно крайние случаи: пустой вектор (все куски пустые, сумма 0) и
`num_threads` больше `n` (лишние потоки получают 0).
</details>

Важно: если убрать синхронизацию в заданиях 2/3/4/5, тест может *иногда* проходить
случайно — это и есть коварство гонок. Поэтому корректность тут — про синхронизацию, а
не про «получилось один раз». Обсудим, если интересно, как такие баги ловят
(ThreadSanitizer: `g++ -fsanitize=thread`).
