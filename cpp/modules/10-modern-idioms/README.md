# Модуль 10 — Современные идиомы C++

Этот модuль — про то, как пишут на C++ *сегодня*. Те же задачи, но короче, безопаснее и
выразительнее. Большая идея: современные возможности убирают шум и переносят больше
проверок на этап компиляции.

## Идея 1. `auto` — вывод типа и его подводные камни

`auto` просит компилятор вывести тип переменной из инициализатора. Внешне это «меньше
печатать», но механика глубже: правила вывода `auto` — это **те же правила, что у вывода
шаблонных аргументов** (template argument deduction). Поэтому, чтобы понимать `auto`, надо
понимать, что именно он отбрасывает.

### Как это работает под капотом

Когда вы пишете `auto x = expr;`, компилятор мысленно представляет шаблон
`template<class T> void f(T x);`, подставляет туда `expr` и смотрит, чему выведется `T`.
Ключевой момент: при выводе по значению (`auto x`) **отбрасываются** верхнеуровневые
`const`/`volatile` и ссылочность.
```cpp
const int  ci = 42;
auto a = ci;          // a -> int   (const отброшен!), a — независимая копия
a = 100;              // ок, ci не тронут

int& r = ...;
auto b = r;           // b -> int   (ссылочность отброшена), b — копия
```
Чтобы сохранить квалификаторы и не копировать, добавляйте их явно:
```cpp
const auto& c = ci;   // c -> const int&  (ссылка, без копии)
auto& d = r;          // d -> int&         (та же переменная)
```

### Зачем это нужно

1. **Громоздкие типы.** Тип итератора `std::map<std::string, std::vector<int>>::iterator`
   читать невозможно — `auto it = m.begin();` решает это.
2. **Типы, которых вы не можете назвать.** Тип лямбды уникален и не имеет имени — без
   `auto` его в переменную не положить:
   ```cpp
   auto cmp = [](int a, int b) { return a > b; };   // имя типа недоступно
   ```
3. **Защита от молчаливых сужающих конверсий.** `auto n = v.size();` всегда возьмёт точный
   тип `size_t`; а `int n = v.size();` молча сузит на больших векторах.

### Trade-offs и ловушки

- **Потеря читаемости.** `auto total = compute();` прячет тип. Если из имени функции тип
  не ясен — пишите тип явно. `auto` экономит автору, но может стоить читателю.
- **Случайная копия в range-for.** Самая частая дорогая ошибка:
  ```cpp
  for (auto s : strings) { ... }        // КОПИРУЕТ каждую строку!
  for (const auto& s : strings) { ... } // правильно: читаем без копий
  ```
- **`auto` никогда не выводит `&` сам.** `auto x = obj.big_member();` сделает копию, даже
  если функция вернула ссылку. Хотите ссылку — пишите `auto&`.
- **Proxy-объекты.** `std::vector<bool>` хранит биты, а `operator[]` возвращает прокси, а
  не `bool&`. `auto bit = vb[0];` поймает прокси-тип, а не `bool` — частый источник
  сюрпризов. Здесь лучше явный `bool bit = vb[0];`.

Правило большого пальца: **`auto&` / `const auto&` по умолчанию в циклах и при привязке к
существующим объектам; `auto` по значению — когда копия действительно нужна.**

## Идея 2. Structured bindings — разложить агрегат на имена

Structured bindings (C++17) позволяют за один шаг распаковать `pair`, `tuple`, массив или
структуру в несколько именованных переменных.
```cpp
std::pair<int, int> p = minmax_of(v);
auto [lo, hi] = p;                       // вместо p.first / p.second
```

### Как это работает под капотом

Запись `auto [a, b] = expr;` — это **синтаксический сахар**. Компилятор создаёт скрытую
переменную (назовём её `e`), инициализирует её из `expr`, а `a` и `b` делает **именами для
её частей**, а не самостоятельными переменными. Что считать «частью», зависит от типа:
- **массив** — поэлементно;
- **тип с `tuple_size`/`tuple_element`/`get` (`pair`, `tuple`, `array`)** — через `get<i>`;
- **простая структура без приватных полей** — публичные нестатические поля по порядку.

Из этого следуют важные практические выводы:
```cpp
auto [a, b] = p;        // e — КОПИЯ p; a,b ссылаются на части копии
auto& [a, b] = p;       // e — ссылка на p; запись в a меняет p.first
const auto& [a, b] = p; // только чтение, без копии — идеально для range-for
```
Квалификатор (`auto`, `auto&`, `const auto&`) применяется к **скрытой переменной `e`**, а
не к каждому имени по отдельности. Поэтому нельзя сделать одно имя ссылкой, а другое — нет.

### Зачем это нужно

Главный выигрыш — **читаемость в циклах по словарю** и при возврате нескольких значений:
```cpp
for (const auto& [name, count] : m)      // сразу ключ и значение, без it->second
    std::cout << name << " = " << count << "\n";

auto [quotient, remainder] = divmod(17, 5);   // говорящие имена вместо .first/.second
```

### Ловушки

- **Случайная копия.** `for (auto [k, v] : m)` копирует каждую пару (а ключ в `map` ещё и
  `const`, так что копия пары — это копия и ключа, и значения). В горячем цикле берите
  `const auto& [k, v]`.
- **Нельзя пропустить элемент.** В C++20 нет `_` как «игнорировать» — придётся завести имя
  (`auto [a, unused] = p;`) и, возможно, пометить его `(void)unused;`.
- **Имена — не настоящие переменные.** Их (в общем случае) нельзя захватить в лямбду по
  имени напрямую — это часто удивляет. Если нужно в лямбду, скопируйте в обычную переменную.
- Привязка к временному объекту по `const auto&` продлевает его жизнь (как обычная
  ссылка), а по `auto&` к rvalue — ошибка компиляции. Это защищает вас.

## Идея 3. `range-for` — как он устроен на самом деле

Цикл `for (auto x : range)` выглядит примитивно, но под ним прячется развёртка, понимание
которой убирает целый класс багов. Компилятор переписывает
```cpp
for (T x : range) body;
```
примерно в:
```cpp
{
    auto&& __r = range;                 // ссылка на диапазон, продлевает жизнь временного
    auto __b = std::begin(__r);
    auto __e = std::end(__r);
    for (; __b != __e; ++__b) {
        T x = *__b;                     // <-- здесь происходит копия или ссылка
        body;
    }
}
```
Отсюда два важных следствия.

**Первое — стоимость `x`.** Строка `T x = *__b;` — это то место, где `auto x` делает копию,
а `const auto& x` — нет. Для тяжёлых элементов (строки, вектора) это разница между O(n)
лишних аллокаций и нулём.

**Второе — опасность временных диапазонов.** `__r` связывается с `range`. Если `range` —
временный объект, он живёт до конца цикла (хорошо). Но если вы итерируетесь по *под-объекту*
временного, который сам не продлевается, получаете висячую ссылку:
```cpp
for (char c : make_string().substr(1)) { ... }   // ок: substr возвращает temporary, он жив
for (auto& kv : get_map_ref_member_of_temp()) {} // ОПАСНО, если temporary умирает раньше
```
В C++20 для этого добавили инициализатор в range-for, чтобы «удержать» владельца:
```cpp
for (auto&& obj = make_owner(); auto& part : obj.parts()) { ... }   // obj живёт весь цикл
```

Резюме: **`const auto&` — почти всегда правильный выбор элемента**; `auto&`, если вы
изменяете элементы на месте; `auto` по значению — только когда нужна именно копия.

## Идея 4. `enum class` — типобезопасные перечисления

```cpp
enum class Color { Red, Green, Blue };

Color c = Color::Red;          // обязательно с префиксом Color::
// int x = c;                  // ошибка — не приводится неявно к int (это хорошо!)
int x = static_cast<int>(c);   // если очень надо — явно
```
В отличие от старого `enum`, `enum class` не «протекает» именами в окружающую область и
не превращается случайно в int. Всегда предпочитай `enum class`. Можно задать и
базовый тип (`enum class Code : std::uint8_t { ... }`), чтобы зафиксировать размер.

## Идея 5. `constexpr` — перенос вычислений на этап компиляции

`constexpr` помечает функцию или переменную как **пригодную для вычисления компилятором**.
Это не «всегда вычисляется в компайл-тайме» — это «**может** быть вычислено в компайл-тайме,
если этого потребует контекст и если все аргументы — константные выражения».
```cpp
constexpr int square(int n) { return n * n; }

constexpr int k = square(5);          // ОБЯЗАН быть посчитан компилятором -> 25
static_assert(square(4) == 16);       // проверка корректности на этапе компиляции
int runtime = square(x);              // x неизвестен -> та же функция работает в рантайме
```

### Как это работает под капотом

Компилятор содержит мини-интерпретатор C++ («constant evaluator»). В **константном
контексте** (`constexpr`-переменная, `static_assert`, размер массива, нетиповой шаблонный
аргумент) он обязан выполнить функцию во время компиляции — иначе ошибка. В обычном
контексте `constexpr`-функция компилируется в обычный код и работает в рантайме. То есть
один и тот же исходник обслуживает оба мира.

### Зачем это нужно

- **Производительность даром.** Таблицы, хэши, разворачивание — посчитаны один раз
  компилятором, в бинарнике лежит уже результат.
- **Сильные гарантии.** Если выражение вычислилось в компайл-тайме, в нём **гарантированно
  нет UB** — компилятор откажется компилировать константное выражение с UB. Это бесплатный
  статический анализатор.
- **`static_assert` как тест.** Проверки уезжают из рантайма в сборку.

### Trade-offs и ловушки

- **`constexpr`-функции живут в заголовках.** Чтобы вычислить функцию в компайл-тайме в
  *другом* файле, компилятор должен видеть её **тело** — как и шаблоны (модуль 07).
  Объявление в `.hpp`, определение в `.cpp` лишит её способности считаться в компайл-тайме.
- **`constexpr` подразумевает `inline`** для функций — несколько определений в разных TU не
  нарушат ODR. Это ещё одна причина держать их в заголовке.
- **Не всё разрешено.** Внутри `constexpr` нельзя бросать неперехваченные исключения в
  константном контексте, нельзя использовать `reinterpret_cast`, нельзя вызывать обычные
  не-`constexpr` функции в константном вычислении. (В C++20 многое ослаблено: появились
  `constexpr`-аллокации, `std::vector`/`std::string` в `constexpr`, виртуальные вызовы.)
- **Не путайте с `const`.** `const` — «не изменю после инициализации» (рантайм-обещание).
  `constexpr` — «вычислимо компилятором» (компайл-тайм-категория). `constexpr` сильнее.
- Хотите *заставить* компайл-тайм даже там, где контекст не требует, — в C++20 есть
  `consteval` (функция обязана всегда вычисляться в компайл-тайме). В этом модуле мы её не
  трогаем, но знать полезно.

## Идея 6. Ranges и views (C++20) — конвейеры над данными

```cpp
#include <ranges>
namespace rv = std::views;

auto result = v
    | rv::filter([](int x){ return x % 2 == 0; })   // оставить чётные
    | rv::transform([](int x){ return x * x; });     // возвести в квадрат
std::vector<int> out;
for (int x : result) out.push_back(x);              // материализуем в вектор
```

### Как это работает под капотом

`rv::filter(pred)` и `rv::transform(f)` возвращают **view** — лёгкий объект, который
*не хранит* данные, а только ссылается на исходный диапазон и помнит операцию. Оператор `|`
просто «надевает» один view на другой (это range adaptor). Никакой работы при построении
конвейера не происходит — вся логика прячется в итераторе результата: когда вы делаете
`++it` и `*it`, он на лету прогоняет элемент через всю цепочку. Отсюда слово **ленивые**.

### Зачем это нужно и trade-offs

- **Нет промежуточных контейнеров.** Классический `std::transform` в новый вектор, потом
  `std::copy_if` в ещё один — это две аллокации и два прохода. Конвейер views — ноль
  промежуточных аллокаций, один проход при материализации.
- **Читается как описание намерения**, а не как механика циклов.
- **Ловушка времён жизни (висячие views).** View *ссылается* на источник. Если источник —
  временный объект, view может стать висячим:
  ```cpp
  auto bad = make_vector() | rv::filter(...);   // make_vector() умер — view висит!
  ```
  Материализуйте сразу или держите источник живым.
- **Однопроходность некоторых views.** Не все views можно обойти дважды; `filter`-view
  кэширует первый подходящий элемент. Для возврата из функции почти всегда нужно
  **материализовать** результат в `std::vector`, а не возвращать сам view.

## Идея 7. `[[nodiscard]]` — результат нельзя выбросить молча

Атрибут `[[nodiscard]]` (C++17) ставится на функцию (или на тип) и заставляет компилятор
**выдать предупреждение, если возвращённое значение проигнорировано**.
```cpp
[[nodiscard]] std::pair<int,int> divmod(int a, int b);

divmod(17, 5);            // warning: ignoring return value of 'divmod' [-Wunused-result]
auto [q, r] = divmod(17, 5);   // ок: результат использован
```

### Зачем это нужно

Возврат значения часто **и есть весь смысл** вызова. Если его проигнорировали — почти
наверняка баг:
- функции-фабрики и чистые вычисления (`divmod`, `parse`, `sqrt`) — игнор результата
  означает выброшенную работу;
- **коды ошибок.** `[[nodiscard]] Status do_thing();` — нельзя «забыть» проверить статус;
- **RAII-возвраты.** `[[nodiscard]] Lock acquire();` — если результат не сохранить, замок
  тут же освободится (временный объект умрёт), и защита исчезнет молча. `[[nodiscard]]`
  ловит именно это.

### Механика и тонкости

- Это **предупреждение, а не ошибка**. С `-Werror` станет ошибкой; в нашем курсе `-Werror`
  не включён, так что вы просто увидите warning. Это нормально для незавершённого стаба.
- Можно дать причину: `[[nodiscard("проверь код ошибки")]]` (C++20) — текст попадёт в
  диагностику.
- Можно пометить **весь тип**: `struct [[nodiscard]] Error{};` — тогда любая функция,
  возвращающая `Error`, наследует требование.
- **Явный отказ.** Если вы *сознательно* игнорируете результат, скажите это явно:
  `(void)divmod(a, b);` или `std::ignore = ...;` — предупреждение исчезнет, а намерение
  останется видимым в коде.
- Ставьте `[[nodiscard]]` экономно — на функции, где игнор почти всегда ошибка. Если
  навешать на всё подряд, предупреждения станут шумом, и их начнут глушить.

## Идея 8. `std::string_view` — строка без владения

`std::string_view` — это **невладеющее «окно»** в чужую последовательность символов: пара
`{указатель, длина}`. Он ничего не аллоцирует и не копирует — лишь смотрит на уже
существующие байты (внутри `std::string`, строкового литерала, буфера).
```cpp
#include <string_view>
void log(std::string_view msg);   // принимает и std::string, и "литерал", и char* — без копий

std::string s = "hello world";
std::string_view first = std::string_view{s}.substr(0, 5);   // "hello", БЕЗ новой строки
```

### Как это работает под капотом

Внутри `string_view` — всего два поля: `const char* data_` и `size_t size_`. Операции вроде
`substr`, `remove_prefix`, `find` **не трогают байты** — они лишь двигают указатель и/или
меняют длину. Поэтому `substr` у `string_view` — это O(1) и ноль аллокаций, в отличие от
`std::string::substr`, который копирует.

### Зачем это нужно

- **Дешёвые параметры функций.** Принимая `std::string_view` вместо `const std::string&`,
  функция одинаково принимает `std::string`, C-строку и литерал — и ни в одном случае не
  создаёт временную `std::string`.
- **Разбор без копий.** Токенайзеры, парсеры, split: нарезаем большой текст на под-окна,
  не копируя ни байта (этим вы и займётесь в задании 6).

### Trade-offs и САМЫЕ частые ошибки (опасны!)

`string_view` **не владеет** данными — отсюда все его ловушки. Помните: окно живёт ровно
столько, сколько живёт то, на что оно смотрит.

1. **Висячий view на временную строку.** Классика:
   ```cpp
   std::string make();
   std::string_view sv = make();   // make() вернул temporary, он УМЕР -> sv висит, UB
   ```
   Никогда не сохраняйте `string_view` на временный объект дольше выражения.

2. **Возврат view на локальную строку.**
   ```cpp
   std::string_view bad() { std::string local = "x"; return local; }  // local умрёт -> UB
   ```

3. **`string_view` НЕ обязан быть нуль-терминированным.** `substr` даёт окно в середину
   строки — там нет `'\0'` в конце. Передавать `sv.data()` в C-функции вроде `printf("%s")`
   или `fopen` — ошибка: они читают до `'\0'` и уедут за пределы окна.
   ```cpp
   std::string_view part = std::string_view{"abcdef"}.substr(0, 3);  // "abc", но data() -> "abcdef"
   printf("%s\n", part.data());   // НАПЕЧАТАЕТ "abcdef"! data() не учитывает длину
   ```
   Чтобы получить настоящую C-строку, постройте `std::string{part}` (вот тут копия уместна).

4. **Модификация источника инвалидирует view.** `push_back`/`reserve` у `std::string` могут
   переаллоцировать буфер — все смотрящие на него `string_view` становятся висячими.

Правило: **`string_view` — отличный тип для параметра и для короткоживущего разбора;
плохой тип для члена класса и для возврата владеющего результата.**

## Идея 9. Грамотные `move` и `forward`

`std::move` и `std::forward` — два инструмента move-семантики, которые постоянно путают.
Оба на самом деле **не двигают и не пересылают ничего** — это просто приведения типа
(`static_cast`), управляющие тем, какая перегрузка будет выбрана.

### Что такое rvalue и почему это важно

У выражения есть «категория значения». Грубо: **lvalue** — у него есть имя/адрес, оно
переживёт это выражение (`x`, `obj.field`). **rvalue** — временное, готовое умереть
(`x + 1`, `make_string()`). Move-конструктор и move-присваивание выбираются, когда источник
— rvalue: раз объект всё равно умрёт, его «потроха» (буфер, указатель) можно **украсть**, а
не копировать. Это и есть выигрыш move-семантики: O(1) перенос вместо O(n) копии.

### `std::move` — «обращайся с этим как с временным»

`std::move(x)` — это `static_cast<T&&>(x)`. Он **ничего не перемещает**, он лишь говорит:
«считай `x` rvalue, можешь забрать у него ресурсы». Реальное перемещение делает уже
вызванный move-конструктор/оператор.
```cpp
std::string a = "большая строка";
std::string b = std::move(a);   // выбран move-конструктор: b забрал буфер a за O(1)
// a теперь в "valid but unspecified" состоянии — трогать значение нельзя, можно лишь
// присвоить заново или уничтожить
```

**Частые ошибки:**
- **`return std::move(local);`** — антипаттерн. Компилятор и так применит NRVO/перемещение;
  ваш `move` лишь **мешает copy elision** и иногда делает хуже. Возвращайте просто `return local;`.
- **Использование объекта после `move`.** После `std::move(v)` значение `v` не определено —
  читать его нельзя.
- **`std::move` у `const`-объекта** молча даёт копию: из `const T` ресурсы не украсть, и
  выбирается копирующая перегрузка. `move` тут — обман зрения, переноса не будет.

### `std::forward` — «сохрани категорию ровно как пришла» (perfect forwarding)

Проблема возникает в шаблонах. Если параметр объявлен как `T&&` при **выводимом** `T` — это
не rvalue-ссылка, а **forwarding reference** (универсальная ссылка). Её хитрость: внутри
функции параметр, имеющий имя, — **всегда lvalue**, даже если в функцию пришёл rvalue.
Передадите его дальше «как есть» — и rvalue превратится в lvalue, move потеряется.
`std::forward<T>(arg)` восстанавливает исходную категорию по выведенному `T`:
```cpp
template <class T>
auto wrap(T&& x) {                 // T&& здесь — forwarding reference
    return sink(std::forward<T>(x));   // rvalue останется rvalue, lvalue — lvalue
}
```
Правило простое:
- параметр объявлен как `T&&` с **выводимым** `T` → пересылайте через **`std::forward<T>`**;
- у вас «настоящий» именованный объект, который вы отдаёте навсегда → **`std::move`**;
- в остальных случаях не трогайте — пусть копируется.

**Ошибка-близнец:** `std::move` вместо `std::forward` в шаблоне «съест» lvalue-аргумент
(украдёт ресурсы у объекта вызывающего!), а `std::forward` вместо `std::move` у обычного
параметра — лишний шум. Различайте: `forward` — для forwarding reference, `move` — для всего
остального, что отдаёте.

В задании 7 (`ScopeGuard`) вы примените `std::move` грамотно: переместите `std::function` в
поле, а не скопируете.
