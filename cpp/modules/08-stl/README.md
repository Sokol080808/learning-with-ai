# Модуль 08 — STL: контейнеры, итераторы, алгоритмы, лямбды

STL (Standard Template Library) — это огромная библиотека готовых контейнеров и
алгоритмов. Большая идея: **не пиши циклы руками там, где есть алгоритм**. Код через
алгоритмы короче, выразительнее и труднее сломать. Junior обязан свободно владеть STL.

STL держится на трёх китах, которые соединяются как кубики:

1. **Контейнеры** хранят данные (`vector`, `map`, `set`, …).
2. **Итераторы** — это «адрес внутри контейнера»; они дают единый способ обхода.
3. **Алгоритмы** работают не с конкретным контейнером, а с **парой итераторов**
   `[first, last)`. Поэтому один `std::sort` сортирует и `vector`, и кусок массива, и
   что угодно ещё, что умеет дать итераторы.

Эта развязка («алгоритмы не знают про контейнеры, а общаются только через итераторы»)
— главное архитектурное решение STL. Ниже разберём каждого кита по-настоящему глубоко.

## Идея 1. Контейнеры — как они устроены и что выбрать

Выбор контейнера — это выбор **структуры данных**, а значит выбор асимптотики операций.
«По умолчанию беру `vector`» — хорошее правило, но junior должен понимать, *почему* и
*когда* оно перестаёт работать.

### `std::vector<T>` — непрерывный массив в куче

Внутри `vector` хранит три указателя: на начало данных, на конец занятой части (`size`)
и на конец выделенной памяти (`capacity`). Элементы лежат **подряд** в одном блоке.

- Доступ по индексу `v[i]` — `O(1)`, просто арифметика адреса.
- `push_back` — амортизированно `O(1)`. Когда `size == capacity`, вектор выделяет **новый,
  больший** блок (обычно в 1.5–2 раза), **копирует/перемещает** туда все элементы и
  освобождает старый. Отдельный `push_back` может стоить `O(n)`, но в среднем по многим
  вставкам выходит `O(1)` — это и называется «амортизированная» сложность.
- Вставка/удаление **в середину** — `O(n)`: нужно сдвинуть хвост.

Главная ловушка — **инвалидация итераторов и ссылок**. Реаллокация при росте делает все
старые итераторы, указатели и ссылки на элементы недействительными:

```cpp
std::vector<int> v{1, 2, 3};
int& ref = v[0];        // ссылка на первый элемент
v.push_back(4);         // МОЖЕТ вызвать реаллокацию
// ref теперь, возможно, «висячая» — использовать НЕЛЬЗЯ
```

Если ты заранее знаешь размер — позови `v.reserve(n)`: один раз выделить память и убрать
все промежуточные реаллокации. Это часто ускоряет код в разы и стабилизирует адреса.

```cpp
std::vector<int> v;
v.reserve(1000);        // capacity >= 1000, push_back теперь без реаллокаций
for (int i = 0; i < 1000; ++i) v.push_back(i);
```

### `std::map<K, V>` — упорядоченное дерево

`map` — это сбалансированное двоичное дерево поиска (на практике красно-чёрное). Ключи
хранятся **отсортированными**; обход `for (auto& [k, v] : m)` идёт по возрастанию ключа.

- `find`, `insert`, `operator[]`, `erase` — `O(log n)`.
- Узлы — отдельные аллокации, разбросанные по памяти → хуже работает кэш процессора,
  чем у `vector`.
- Итераторы и ссылки на элементы `map` **стабильны**: вставка и удаление *других* узлов
  их не ломает (в отличие от `vector`).

Коварная деталь — `operator[]`. Он **вставляет** элемент со значением по умолчанию, если
ключа нет, и возвращает ссылку на него:

```cpp
std::map<std::string, int> m;
int x = m["нет_такого"];   // ВСТАВИЛ {"нет_такого", 0} и вернул 0!
// теперь m.size() == 1, хотя мы «просто читали»
```

Именно это свойство мы эксплуатируем в счётчиках: `++m[word]` создаёт запись со значением
`0` при первом появлении слова, затем инкрементит. Но если ты *проверяешь наличие*, не
используй `[]` — бери `m.contains(k)` (C++20) или `m.find(k) != m.end()`, иначе случайно
насоришь в словарь.

### `std::unordered_map<K, V>` — хеш-таблица

Та же роль «ключ→значение», но внутри хеш-таблица: ключ прогоняется через хеш-функцию,
получается номер «корзины» (bucket).

- `find`/`insert`/`[]` — в среднем `O(1)`, но в **худшем** случае (много коллизий) `O(n)`.
- Порядок обхода **не определён** — не рассчитывай на сортировку.
- Подходит, когда нужен быстрый поиск по ключу и порядок не важен. Если нужен
  отсортированный вывод — бери `map`.

### `std::set` / `std::unordered_set`

Множества уникальных значений. `set` — отсортирован (`O(log n)`), `unordered_set` —
хеш (`O(1)` в среднем). Классическое применение — быстро отвечать «видели ли мы уже
этот элемент?».

### `std::array<T, N>` и прочие

`std::array<T, N>` — обёртка над массивом фиксированного размера **на стеке**, без
аллокаций; размер известен на этапе компиляции. `deque` — быстрые вставки с обоих концов;
`list` — двусвязный список со стабильными итераторами, но почти всегда медленнее `vector`
на практике из-за кэша. Их берут редко и осознанно.

**Шпаргалка выбора:** нужен индекс и скорость → `vector`; нужен поиск по ключу с
сортировкой → `map`; быстрый поиск без порядка → `unordered_map`; уникальность →
`set`/`unordered_set`.

## Идея 2. Итераторы — общий «язык» обхода и его категории

Итератор — это обобщение указателя: объект, который умеет `*it` (получить значение),
`++it` (шагнуть дальше) и сравниваться (`it != end`). Любой контейнер даёт `begin()`
(первый элемент) и `end()` (позицию **за** последним). Диапазон — это всегда полуоткрытый
интервал `[begin, end)`: `end` не указывает на элемент, он маркер «конец».

```cpp
for (auto it = v.begin(); it != v.end(); ++it) std::cout << *it;
for (int x : v) std::cout << x;   // range-based for — то же самое, короче
```

Почему полуоткрытый интервал, а не «оба конца включены»? Потому что тогда «пусто»
записывается естественно как `begin == end`, и длина диапазона = `end - begin` без
поправок на единицу. Вся STL построена на этом соглашении.

### Категории итераторов — не все одинаково мощны

Алгоритмы требуют от итератора определённых возможностей. Категории (от слабых к сильным):

- **Input/Output** — однопроходные, только `++` и разыменование. Пример: чтение из потока.
- **Forward** — можно проходить много раз вперёд (`forward_list`).
- **Bidirectional** — ещё и `--it` назад (`list`, `map`, `set`).
- **Random access** — `it + n`, `it[i]`, `it2 - it1` за `O(1)` (`vector`, `array`,
  `deque`, обычный массив).

Отсюда практический вывод: `std::sort` требует **random access** итераторов — поэтому
`std::sort(list.begin(), list.end())` **не скомпилируется**, у `list` итераторы только
bidirectional. Для списка есть отдельный метод `list.sort()`. Аналогично `std::partial_sort`
и `std::nth_element` (мы их используем ниже) тоже требуют random access — то есть работают
с `vector`, но не с `map`/`set`.

### Инвалидация итераторов — главный источник UB у новичков

Менять контейнер во время обхода — классическая ошибка:

```cpp
// ОПАСНО: erase делает итератор it недействительным
for (auto it = v.begin(); it != v.end(); ++it)
    if (*it < 0) v.erase(it);   // после erase it «протух» — UB на ++it
```

Правильно — использовать то, что `erase` **возвращает** итератор на следующий элемент:

```cpp
for (auto it = v.begin(); it != v.end(); )
    if (*it < 0) it = v.erase(it);   // erase вернул валидный «следующий»
    else ++it;
```

А для `vector` фильтрацию вообще красивее делать идиомой erase-remove (Идея 6).

### Вставляющие итераторы

`std::back_inserter(c)` — это не позиция в контейнере, а «адаптер»: каждое присваивание
`*it = x` он превращает в `c.push_back(x)`. Так алгоритмы, которые пишут в выходной
диапазон (`copy_if`, `transform`), могут **наращивать** контейнер, не зная заранее размер:

```cpp
std::vector<int> out;
std::copy_if(v.begin(), v.end(), std::back_inserter(out), pred);
```

Без него пришлось бы заранее знать, сколько элементов пройдёт фильтр.

## Идея 3. Лямбды — функция «на месте», и что такое захват

Лямбда — это компактный синтаксис, который компилятор разворачивает в **анонимный класс
с перегруженным `operator()`** (так называемый замыкание-объект). Понимать это полезно:
лямбда — полноценный объект, его можно хранить, копировать, передавать.

```cpp
auto is_even = [](int x) { return x % 2 == 0; };
is_even(4);  // true
```

Это эквивалентно примерно такому:

```cpp
struct __anon { bool operator()(int x) const { return x % 2 == 0; } };
auto is_even = __anon{};
```

### Список захвата `[...]` — что лямбда «уносит с собой»

`[...]` перечисляет, какие переменные из окружающей области видимости становятся **полями**
замыкания. Захват по значению копирует переменную внутрь; захват по ссылке хранит ссылку.

```cpp
int threshold = 10;
auto big_by_val = [threshold](int x) { return x > threshold; };  // копия threshold
auto big_by_ref = [&threshold](int x) { return x > threshold; }; // ссылка на threshold
threshold = 100;
big_by_val(50);  // true  — внутри лежит старая копия (10)
big_by_ref(50);  // false — видит актуальное значение (100)
```

- `[=]` — захватить **всё** используемое по значению (копии).
- `[&]` — захватить всё по ссылке.
- `[x, &y]` — `x` копией, `y` ссылкой.
- `[]` — ничего не захватывать (тогда лямбда даже конвертируется в обычный указатель на
  функцию).

### Самая частая и опасная ошибка — висячая ссылка из `[&]`

Захват по ссылке хранит **ссылку**, а не значение. Если лямбда переживёт захваченную
переменную, получишь обращение к мёртвой памяти:

```cpp
auto make_adder() {
    int base = 42;
    return [&base](int x) { return x + base; };  // ОШИБКА: base умрёт при выходе
}                                                 // вернули лямбду с висячей ссылкой
```

Правильно — захватить по значению, тогда `base` скопируется внутрь замыкания и переживёт
функцию:

```cpp
auto make_adder() {
    int base = 42;
    return [base](int x) { return x + base; };   // base скопирован, всё живо
}
```

Правило: **передаёшь лямбду алгоритму прямо здесь и сейчас** — `[&]` дёшево и безопасно;
**лямбда уезжает наружу/живёт долго** — захватывай по значению.

### `mutable` и почему по умолчанию `operator()` константный

Захваченные по значению поля внутри лямбды по умолчанию **только для чтения** (`operator()`
помечен `const`). Если нужно их менять между вызовами — добавь `mutable`:

```cpp
int calls = 0;
auto counter = [calls]() mutable { return ++calls; };  // меняет свою копию calls
counter();  // 1
counter();  // 2  — внешний calls остался 0
```

## Идея 4. Алгоритмы `<algorithm>` / `<numeric>` — словарь, который надо знать

Алгоритм принимает диапазон `[first, last)` и (часто) функтор-предикат. Он **ничего не
знает** про тип контейнера — только про итераторы. Запомни костяк:

```cpp
#include <algorithm>
#include <numeric>

std::sort(v.begin(), v.end());                            // сортировка по возрастанию
std::sort(v.begin(), v.end(), std::greater<int>());       // по убыванию
std::sort(v.begin(), v.end(), [](int a, int b){ return a > b; }); // то же лямбдой
int n   = std::count_if(v.begin(), v.end(), is_even);     // сколько подходит предикату
auto it = std::find(v.begin(), v.end(), 42);              // поиск; it == end() если нет
int sum = std::accumulate(v.begin(), v.end(), 0);         // сумма (начальное 0)
bool a  = std::all_of(v.begin(), v.end(), is_even);       // все ли подходят
std::transform(v.begin(), v.end(), out.begin(), f);       // применить f к каждому
std::copy_if(v.begin(), v.end(), std::back_inserter(out), pred); // фильтр
```

### Компаратор — это «строго меньше», и здесь легко выстрелить себе в ногу

Компаратор `cmp(a, b)` должен возвращать `true`, **только если `a` должно идти раньше `b`**,
и моделировать *строгий* порядок (strict weak ordering). Самая частая ошибка — написать
`<=` вместо `<`:

```cpp
std::sort(v.begin(), v.end(), [](int a, int b){ return a <= b; }); // UB!
```

Для равных элементов `cmp(a, b)` и `cmp(b, a)` оба дадут `true` — это нарушает контракт
сортировки, и поведение становится неопределённым (вплоть до падения). Правило: компаратор
для «по возрастанию» — это `a < b`, для «по убыванию» — `a > b`. Никаких `=`.

### `accumulate` — следи за типом начального значения

`std::accumulate` использует тип **начального значения** как тип аккумулятора. Передашь
`int` — получишь целочисленное накопление, даже если в контейнере `double`:

```cpp
std::vector<double> d{0.5, 0.5, 0.5};
auto s1 = std::accumulate(d.begin(), d.end(), 0);    // 0 (int!) — дробные части отброшены
auto s2 = std::accumulate(d.begin(), d.end(), 0.0);  // 1.5 — правильно
```

То же со средним по большим `int`: начни с `0LL`, чтобы не переполнить `int`.

### `find` против `count` против `binary_search`

`std::find` — линейный `O(n)` поиск, работает на любом диапазоне. Если данные
**отсортированы**, используй `std::binary_search` / `std::lower_bound` — это `O(log n)`.
А для `map`/`set`/`unordered_map` поиск делается **методом контейнера** (`m.find(k)`),
а не свободным `std::find` — метод использует внутреннюю структуру и куда быстрее.

## Идея 5. `partial_sort` и `nth_element` — когда полная сортировка избыточна

Часто нужно не «отсортировать всё», а «найти k наибольших/наименьших». Полный `sort`
стоит `O(n log n)` и делает лишнюю работу. STL даёт два более точных инструмента.

### `std::partial_sort(first, middle, last, cmp)`

Переставляет элементы так, что **первые `middle - first`** оказываются отсортированными и
равны тем, что были бы там после полной сортировки; остальные — в неопределённом порядке.
Сложность примерно `O(n log k)`, где `k` — размер «головы». Идеально для «топ-k, причём
сами k нужны упорядоченными»:

```cpp
std::vector<int> v{3, 1, 4, 1, 5, 9, 2, 6};
std::partial_sort(v.begin(), v.begin() + 3, v.end(), std::greater<int>());
// v[0..2] == {9, 6, 5} (отсортированы по убыванию); хвост — как попало
```

### `std::nth_element(first, nth, last, cmp)`

Ещё дешевле — в среднем `O(n)`. Гарантирует только, что элемент на позиции `nth` встанет
**на своё итоговое место**, всё слева от него — не больше его, всё справа — не меньше
(по заданному `cmp`), но внутри этих частей порядок произвольный. Используется, когда
нужен сам k-й элемент (медиана, перцентиль) или «k наибольших без требования к их
взаимному порядку»:

```cpp
std::vector<int> v{3, 1, 4, 1, 5, 9, 2, 6};
std::nth_element(v.begin(), v.begin() + 2, v.end(), std::greater<int>());
// v[0..1] — два наибольших (в любом порядке), v[2] — третий по величине
```

**Граничные случаи**, которые надо обработать руками: `middle`/`nth` должен лежать в
`[first, last]`. Если просят больше элементов, чем есть (`k >= size`), нельзя писать
`begin() + k` — выйдешь за `end()` и получишь UB. В этом случае просто отсортируй весь
вектор. И `k <= 0` — отдельная проверка, результат пустой.

```cpp
if (k <= 0) return {};
if (static_cast<std::size_t>(k) >= v.size()) { std::sort(v.begin(), v.end(), cmp); return v; }
// иначе безопасно partial_sort на первые k
```

## Идея 6. Идиома erase-remove — как правильно удалять из `vector`

Удаление элементов по условию из `vector` — место, где новички пишут UB (см. Идею 2).
Корень проблемы: алгоритмы **не умеют менять размер контейнера** — у них есть только
итераторы, а не сам контейнер. Поэтому удаление делается в **два шага**.

### Шаг 1 — `std::remove_if` переставляет, но не удаляет

`std::remove_if(first, last, pred)` проходит диапазон и **сдвигает «выживших» (тех, для
кого `pred` == false) в начало**, сохраняя их относительный порядок. Возвращает итератор
на **новый логический конец** — позицию за последним выжившим. Элементы в «хвосте» после
этого итератора — в неопределённом состоянии (это «мусор», который ещё физически в векторе).
Сам размер вектора **не меняется**.

```cpp
std::vector<int> v{5, 1, 8, 2, 9};
auto new_end = std::remove_if(v.begin(), v.end(), [](int x){ return x < 5; });
// v стал, например, {5, 8, 9, ?, ?}; new_end указывает между 9 и хвостом
// v.size() всё ещё 5!
```

### Шаг 2 — `vector::erase` физически отрезает хвост

Теперь вызываем метод **контейнера** `erase(new_end, v.end())`, который реально уменьшает
размер:

```cpp
v.erase(new_end, v.end());   // теперь v == {5, 8, 9}, size() == 3
```

Слитно это и есть **erase-remove idiom**:

```cpp
v.erase(std::remove_if(v.begin(), v.end(), pred), v.end());
```

(В C++20 есть и свободная `std::erase_if(v, pred)`, которая делает то же самое одной
строкой — но понимать механику двух шагов обязательно, иначе не поймёшь, *почему* нужен
`erase`.)

### Зачем нам нужно ещё и **количество** удалённых

`std::remove_if` сам по себе число удалённых не возвращает — он возвращает итератор. Но
количество легко получить как **расстояние** от нового конца до старого:

```cpp
auto it = std::remove_if(v.begin(), v.end(), pred);
std::size_t removed = static_cast<std::size_t>(std::distance(it, v.end()));
v.erase(it, v.end());
```

Порядок важен: сначала вычисли `distance` (пока `it` и `v.end()` ещё указывают на старый
хвост), потом делай `erase`.

### Частая ошибка — «забыл erase»

```cpp
std::remove_if(v.begin(), v.end(), pred);   // БЕСПОЛЕЗНО как самостоятельная строка
// размер не изменился, в хвосте мусор — а человек думал, что удалил
```

`remove_if` без последующего `erase` почти всегда баг. Если компилятор предупреждает
«ignoring return value» — это оно.

## Идея 7. `std::string_view` и `std::span` — zero-copy вид на чужие данные

### Проблема, которую они решают

Когда функция принимает `const std::string&`, вызывающий код должен передать именно
`std::string`. Хочешь передать строковый литерал или срез внутри другой строки — компилятор
неявно создаст **временный** `std::string` с копированием всех байт. `const char*` дешевле,
но теряет длину и не работает с `std::string`. `std::string_view` (C++17) закрывает этот
пробел: лёгкая «двух-указательная» структура {данные, длина}, которая **не владеет** строкой
и не копирует ничего.

### `std::string_view` — строка без аллокации

```cpp
#include <string_view>

void print(std::string_view sv) {
    // sv.data(), sv.size(), sv.substr(), sv.find() — всё работает
    // sv НЕ нуль-терминирована! sv.data() нельзя передавать туда, где нужен const char*
    std::cout << sv.size() << '\n';
}

print("hello");                  // литерал — без копирования
print(some_string);              // std::string — неявная конвертация, без копии
print(some_string.substr(2, 3)); // substr возвращает std::string — здесь БУДЕТ копия
std::string_view part{some_string.data() + 2, 3}; // вид на срез — без копии
```

Механика внутри: `string_view` — просто `{const char* ptr, size_t len}`. Никакой аллокации,
никакого деструктора. Копирование `string_view` — копирование двух чисел. Идеально для
read-only параметров функций.

**Критичное правило lifetime**: `string_view` не продлевает жизнь данных, на которые
смотрит. Если источник уничтожен — `string_view` становится висячим:

```cpp
std::string_view sv;
{
    std::string s = "hello";
    sv = s;    // sv смотрит на внутренний буфер s
}              // s уничтожен, буфер освобождён
// sv.data() — висячий указатель! UB
```

Поэтому: **не храни `string_view` как поле класса или дольше, чем живёт источник**. Как
параметр локальной функции — безопасно. Как долгоживущий член — опасно.

**Нет нуль-терминатора**: `string_view` не гарантирует `\0` после последнего символа.
`sv.data()` нельзя передавать в C-функции, которые ожидают нуль-терминированную строку.
Для этого нужна `std::string{sv}` с явным копированием.

```cpp
// Типичная ошибка:
std::string_view sv = "hello";
std::fopen(sv.data(), "r");  // ОПАСНО: fopen ожидает нуль-терминатор, sv.data() его имеет
                             // только если исходная строка его имеет; с substr — нет!
```

### `std::span<T>` — вид на contiguous range любого типа

`std::span<T>` (C++20) — аналог `string_view`, но для произвольных `T` и изменяемых данных.
Тоже `{T* ptr, size_t len}`, не владеет памятью.

```cpp
#include <span>

void fill_zeros(std::span<int> s) {
    for (int& x : s) x = 0;         // span поддерживает range-based for
    // s[i], s.data(), s.size() — работают
}

std::vector<int> v{1, 2, 3, 4, 5};
fill_zeros(v);                       // автоматически — без копирования
fill_zeros(std::span<int>{v.data() + 1, 3});  // вид на срез [1..3]

int arr[5] = {1, 2, 3, 4, 5};
fill_zeros(arr);                     // и для C-массивов тоже
```

`std::span<const int>` — только для чтения, как `string_view`. Используй `span<const T>` для
read-only API, `span<T>` когда нужна запись. Важно: **не возвращай `span` как результат
функции**, если источник локален — та же проблема с lifetime, что у `string_view`.

**Сравнение трёх подходов** для «передать кусок данных в функцию»:

| Подход | Копия? | Изменяем? | Ограничения |
|---|---|---|---|
| `const std::string&` | нет (если уже `string`) | нет | только `string` |
| `std::string_view` | нет | нет | contiguous, учти lifetime |
| `std::span<const T>` | нет | нет | contiguous, любой T |
| `std::span<T>` | нет | да | contiguous, любой T |

## Идея 8. C++20 Ranges и views-конвейеры

### Проблема классического iter-pair стиля

```cpp
// Классика: фильтровать чётные и возвести в квадрат
std::vector<int> result;
std::copy_if(v.begin(), v.end(), std::back_inserter(result),
             [](int x) { return x % 2 == 0; });
std::transform(result.begin(), result.end(), result.begin(),
               [](int x) { return x * x; });
```

Этот код правильный, но:
- промежуточный `result` — лишняя аллокация;
- логика «что мы делаем» разбита на два несвязных блока;
- читается снизу вверх, а думаем мы слева направо.

C++20 Ranges решают всё это.

### Алгоритмы `std::ranges::` — ranges-версии классических алгоритмов

Пространство имён `std::ranges` содержит ranges-версии всех стандартных алгоритмов:

```cpp
#include <algorithm>
#include <ranges>

std::ranges::sort(v);                    // принимает контейнер целиком (не begin/end!)
std::ranges::sort(v, std::greater<>{}); // компаратор
std::ranges::sort(v, {}, &Person::age); // проекция: сортируем по полю .age

auto it = std::ranges::find(v, 42);     // ranges::find возвращает итератор
bool b  = std::ranges::all_of(v, pred); // стандартная семантика, ranges-интерфейс
```

Главное отличие от `std::`: алгоритмы `std::ranges::` принимают **контейнер или диапазон
целиком** (любой объект, у которого есть `begin()/end()`), а не пару итераторов. Также они
поддерживают **проекции** — `std::ranges::sort(v, {}, &T::field)` сортирует по полю без
написания лямбды.

### Views-адапторы — ленивый конвейер через `|`

`std::views` (alias `std::ranges::views`) предлагает **ленивые** адапторы, которые
соединяются через `|`:

```cpp
#include <ranges>

auto pipe = v
    | std::views::filter([](int x) { return x % 2 == 0; })  // только чётные
    | std::views::transform([](int x) { return x * x; })    // возводим в квадрат
    | std::views::take(3);                                   // первые три

// pipe — это view, вычисление ленивое: элементы обрабатываются по требованию
for (int x : pipe) std::cout << x;   // только здесь происходят вычисления
```

**Ленивость** — ключевая особенность: `views::filter | views::transform` не создаёт
промежуточных векторов. Элементы вычисляются один за одним при обходе. Это часто быстрее
(меньше аллокаций) и всегда декларативнее.

**Материализация** — когда нужен `std::vector<int>` из view, собери руками:

```cpp
// C++20: материализация вручную
std::vector<int> result;
auto pipe = v | std::views::filter(pred) | std::views::transform(f);
result.reserve(std::ranges::distance(pipe));  // опционально
std::ranges::copy(pipe, std::back_inserter(result));

// Или конструктор через итераторы (C++20):
std::vector<int> result2(pipe.begin(), pipe.end());
```

(C++23 добавил `std::ranges::to<std::vector>()`, но в C++20 его нет.)

### Словарь стандартных views

```cpp
// std::views::filter(pred)      — пропускает только те, для которых pred(x) == true
// std::views::transform(f)      — применяет f к каждому элементу
// std::views::take(n)           — первые n элементов
// std::views::drop(n)           — пропустить первые n
// std::views::reverse           — обход в обратном порядке
// std::views::iota(a, b)        — последовательность a, a+1, ..., b-1 (без вектора!)
// std::views::keys / values     — ключи/значения map-подобных диапазонов
```

Пример — «сумма квадратов нечётных чисел в диапазоне 1..100»:

```cpp
auto squares_of_odds =
    std::views::iota(1, 101)
    | std::views::filter([](int x) { return x % 2 != 0; })
    | std::views::transform([](int x) { return x * x; });

int sum = std::accumulate(squares_of_odds.begin(), squares_of_odds.end(), 0);
// Ни одного промежуточного вектора
```

**Сравнение классики и ranges:**

```cpp
// Классика: найти первое чётное > 10
auto it = std::find_if(v.begin(), v.end(),
                       [](int x) { return x > 10 && x % 2 == 0; });

// Ranges:
auto found = v | std::views::filter([](int x) { return x > 10; })
               | std::views::filter([](int x) { return x % 2 == 0; });
auto it2 = std::ranges::begin(found);  // итератор на первый подходящий
```

**Подвох — инвалидация**: view не хранит данные, он держит **ссылку** на источник. Если
источник (вектор) реаллоцировался после создания view, итерирование по view — UB. Создавай
view, используй его, не модифицируй источник между созданием и использованием.

## Идея 9. Итераторные адапторы — шире, чем `back_inserter`

`back_inserter` — самый известный адаптор, но STL предлагает целое семейство:

### `rbegin` / `rend` — обратные итераторы

Любой bidirectional-контейнер даёт `rbegin()` / `rend()` для обхода в обратном порядке.
`rbegin()` — это обёртка над `end()`, которая при `++` движется назад:

```cpp
std::vector<int> v{1, 2, 3, 4, 5};

// Печатаем задом наперёд:
for (auto it = v.rbegin(); it != v.rend(); ++it) std::cout << *it; // 5 4 3 2 1

// Алгоритм над обратным диапазоном:
auto it = std::find(v.rbegin(), v.rend(), 3);  // найти 3 с конца
// it.base() — соответствующий прямой итератор (указывает на элемент ПОСЛЕ найденного)

// Range-for с обратным диапазоном через views::reverse:
for (int x : v | std::views::reverse) std::cout << x;
```

**Применение**: `std::copy(v.rbegin(), v.rend(), ...)` — скопировать в обратном порядке.
`std::find_if(v.rbegin(), v.rend(), pred)` — найти **последнее** вхождение с конца. Сложность
всегда та же, что у прямых итераторов — никакой реверсии самих данных не происходит.

### `istream_iterator` / `ostream_iterator` — потоки как диапазоны

Позволяют применять алгоритмы напрямую к потокам ввода-вывода без явного цикла:

```cpp
#include <iterator>
#include <sstream>

std::istringstream ss("1 2 3 4 5");

// Прочитать все int из потока в вектор:
std::vector<int> v(std::istream_iterator<int>{ss},
                   std::istream_iterator<int>{});   // default-constructed = «конец»

// Или через алгоритм:
std::vector<int> v2;
std::copy(std::istream_iterator<int>{ss},
          std::istream_iterator<int>{},
          std::back_inserter(v2));

// Записать вектор в cout через пробел:
std::copy(v.begin(), v.end(),
          std::ostream_iterator<int>{std::cout, " "});
// Или с разделителем-переводом строки:
std::copy(v.begin(), v.end(),
          std::ostream_iterator<int>{std::cout, "\n"});
```

`istream_iterator<T>{}` без аргументов — «sentinel» (конечный итератор): итерирование
остановится, когда поток исчерпан или встретит ошибку чтения. Механика: `*it` читает
очередной `T` из потока; `++it` переходит к следующему. `ostream_iterator<T>` при
присваивании `*it = x` записывает `x << sep` в поток.

**Подвох**: `istream_iterator` — однопроходный (input iterator). Нельзя пройти по потоку
дважды. Алгоритмы, требующие forward-итераторов (например `std::sort`), с ним не работают —
сначала собери в вектор, потом сортируй.

### `front_inserter` — вставка в начало

`std::front_inserter(c)` превращает `*it = x` в `c.push_front(x)`. Работает только для
контейнеров с `push_front`: `deque`, `list`, `forward_list`.

```cpp
std::deque<int> dq;
std::copy(v.begin(), v.end(), std::front_inserter(dq));
// Элементы оказываются в ОБРАТНОМ порядке: last-in-first (5,4,3,2,1 если v={1,2,3,4,5})
```

Для `vector` `front_inserter` **не компилируется** — у `vector` нет `push_front` (вставка в
начало `O(n)`). Используй `back_inserter`, а потом `std::reverse`, или строй через `insert`
в нужную позицию.

### `insert_iterator` — вставка в произвольную позицию

`std::inserter(c, it)` — обобщение: вставляет перед позицией `it` и обновляет её:

```cpp
std::set<int> s;
std::copy(v.begin(), v.end(), std::inserter(s, s.begin()));
// Работает с set, map — любым контейнером с insert(iterator, value)
```

Это то, что нужно для `set` и `map`: у них нет `push_back`, поэтому `back_inserter` не
скомпилируется, а `inserter` — да.

**Шпаргалка адапторов:**

| Адаптор | Что делает `*it = x` | Контейнеры |
|---|---|---|
| `back_inserter(c)` | `c.push_back(x)` | `vector`, `deque`, `list` |
| `front_inserter(c)` | `c.push_front(x)` | `deque`, `list`, `forward_list` |
| `inserter(c, pos)` | `c.insert(pos, x)` | все с `insert` |
| `ostream_iterator<T>(os, sep)` | `os << x << sep` | потоки |

<details><summary>Задание 8 — drop_below: какой предикат</summary>
Удаляем «строго меньшие threshold» — значит предикат «оставить-в-мусор» это
<code>x &lt; threshold</code>. Элемент, равный threshold, под него НЕ попадает (это
проверяет один из тестов). Скорми этот предикат в <code>std::remove_if</code>.
</details>
<details><summary>Задание 9 — evens_squared: как начать конвейер</summary>
Подключи <code>&lt;ranges&gt;</code>. Цепочка: <code>xs | std::views::filter(предикат-чётности) |
std::views::transform(лямбда-квадрата)</code>. Результат — view-объект, не вектор.
Передай его в <code>std::ranges::copy(view, std::back_inserter(out))</code>.
</details>
<details><summary>Задание 9 — evens_squared: материализация</summary>
<code>std::vector&lt;int&gt; out;</code>, затем <code>std::ranges::copy(pipe, std::back_inserter(out))</code>.
Можно ещё <code>out.reserve(…)</code>, но не обязательно. <code>std::ranges::copy</code>
отличается от <code>std::copy</code> тем, что принимает диапазон, а не пару итераторов.
</details>
<details><summary>Задание 8 — сначала посчитать, потом стереть</summary>

<code>std::distance(it, v.end())</code> — это и есть число удалённых, — и только ПОТОМ
вызови <code>v.erase(it, v.end())</code>. Если стереть до подсчёта — считать будет нечего.

</details>
