# Модуль 09 — Обработка ошибок

Большая идея: ошибки бывают двух сортов — **исключительные ситуации** (то, чего «не
должно было случиться») и **ожидаемые отсутствия результата** (например, «такого ключа
нет»). Для первых в C++ есть исключения, для вторых — `std::optional`/`std::variant`.
Выбор правильного инструмента — признак зрелого инженера.

## Идея 1. Исключения: `throw` / `try` / `catch`

```cpp
#include <stdexcept>

int safe_divide(int a, int b) {
    if (b == 0)
        throw std::invalid_argument("division by zero");
    return a / b;
}

try {
    int r = safe_divide(10, 0);
} catch (const std::invalid_argument& e) {
    std::cout << "Ошибка: " << e.what() << "\n";
}
```
`throw` прерывает выполнение и «всплывает» вверх по стеку вызовов, пока не найдётся
подходящий `catch`. Стандартные типы (`<stdexcept>`): `std::invalid_argument`,
`std::out_of_range`, `std::runtime_error`, `std::logic_error` и их базовый
`std::exception` (у всех есть метод `.what()`).

Лови исключения по **`const`-ссылке** (`catch (const std::exception& e)`) — без копии и
с поддержкой полиморфизма.

## Идея 2. Исключения и RAII — почему это работает безопасно

Когда исключение «всплывает», все локальные объекты на пути корректно **разрушаются**
(вызываются деструкторы). Поэтому объект на RAII (умный указатель, `std::vector`,
блокировка) освободит ресурс **даже при исключении**. Это и есть **exception safety**.
Код, который вручную делает `new`/`delete` вокруг кода, способного бросить, — почти
всегда содержит утечку. Вывод: владей ресурсами через RAII, и исключения станут
безопасными почти бесплатно.

## Идея 3. `noexcept`

```cpp
int size() const noexcept;   // обещание: эта функция не бросает исключений
```
Помечай `noexcept` то, что гарантированно не бросает (особенно move-операции — помнишь
модуль 05?). Это и документация, и подсказка оптимизатору. Но не лги: если из
`noexcept`-функции вылетит исключение — программа аварийно завершится (`std::terminate`).

## Идея 4. `std::optional<T>` — «значение или ничего»

Когда «нет результата» — это **нормальный** исход, а не ошибка, бросать исключение
дорого и неуместно. `std::optional<T>` — это «может быть `T`, а может быть пусто»:

```cpp
#include <optional>

std::optional<int> to_int(const std::string& s);

auto r = to_int("42");
if (r) std::cout << *r;          // есть значение: *r или r.value()
auto x = to_int("oops").value_or(-1);  // если пусто — взять -1
```
`std::nullopt` — пустое значение. `*opt` / `opt.value()` — доступ (но `value()` бросит,
если пусто; `*` — UB если пусто, поэтому сперва проверь `if (opt)`).

## Идея 5. `std::variant<A, B, ...>` — «одно из нескольких»

Тип, который в каждый момент хранит значение **одного** из перечисленных типов
(типобезопасный union):

```cpp
#include <variant>
std::variant<int, std::string> v = 42;
if (std::holds_alternative<int>(v)) { int n = std::get<int>(v); }
// или универсально:
std::string s = std::visit([](const auto& x){ return to_text(x); }, v);
```

<details><summary>Задание 3 — корректный разбор</summary>
Можно через `std::stoi`, но он бросает исключение на плохом вводе и не проверяет
«хвост» ("12abc"). Аккуратнее — `std::from_chars` (из `<charconv>`): он сообщает,
сколько символов разобрал и где остановился. Проверь, что разобрана вся строка.
</details>
<details><summary>Когда исключение, а когда optional?</summary>
`element_at` за границей — это нарушение контракта вызывающим → исключение.
А `to_int("hello")` — ожидаемая ситуация «не число» → optional. Прочувствуй разницу,
она концептуальная. Сомневаешься — спроси, разберём на примерах.
</details>
