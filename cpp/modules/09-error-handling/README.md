# Модуль 09 — Обработка ошибок

Большая идея: ошибки бывают двух сортов — **исключительные ситуации** (то, чего «не
должно было случиться») и **ожидаемые отсутствия результата** (например, «такого ключа
нет»). Для первых в C++ есть исключения, для вторых — `std::optional`/`std::variant`.
Выбор правильного инструмента — признак зрелого инженера.

## Идея 1. Исключения: `throw` / `try` / `catch`

```cpp
#include <stdexcept>

int safe_divide(int a, int b) {
    if (b == 0)
        throw std::invalid_argument("division by zero");
    return a / b;
}

try {
    int r = safe_divide(10, 0);
} catch (const std::invalid_argument& e) {
    std::cout << "Ошибка: " << e.what() << "\n";
}
```
`throw` прерывает выполнение и «всплывает» вверх по стеку вызовов, пока не найдётся
подходящий `catch`. Стандартные типы (`<stdexcept>`): `std::invalid_argument`,
`std::out_of_range`, `std::runtime_error`, `std::logic_error` и их базовый
`std::exception` (у всех есть метод `.what()`).

Лови исключения по **`const`-ссылке** (`catch (const std::exception& e)`) — без копии и
с поддержкой полиморфизма.

## Идея 2. Исключения и RAII — почему это работает безопасно

Когда исключение «всплывает», все локальные объекты на пути корректно **разрушаются**
(вызываются деструкторы). Поэтому объект на RAII (умный указатель, `std::vector`,
блокировка) освободит ресурс **даже при исключении**. Это и есть **exception safety**.
Код, который вручную делает `new`/`delete` вокруг кода, способного бросить, — почти
всегда содержит утечку. Вывод: владей ресурсами через RAII, и исключения станут
безопасными почти бесплатно.

## Идея 3. `noexcept`

```cpp
int size() const noexcept;   // обещание: эта функция не бросает исключений
```
Помечай `noexcept` то, что гарантированно не бросает (особенно move-операции — помнишь
модуль 05?). Это и документация, и подсказка оптимизатору. Но не лги: если из
`noexcept`-функции вылетит исключение — программа аварийно завершится (`std::terminate`).

## Идея 4. `std::optional<T>` — «значение или ничего»

Когда «нет результата» — это **нормальный** исход, а не ошибка, бросать исключение
дорого и неуместно. `std::optional<T>` — это «может быть `T`, а может быть пусто»:

```cpp
#include <optional>

std::optional<int> to_int(const std::string& s);

auto r = to_int("42");
if (r) std::cout << *r;          // есть значение: *r или r.value()
auto x = to_int("oops").value_or(-1);  // если пусто — взять -1
```
`std::nullopt` — пустое значение. `*opt` / `opt.value()` — доступ (но `value()` бросит,
если пусто; `*` — UB если пусто, поэтому сперва проверь `if (opt)`).

## Идея 5. `std::variant<A, B, ...>` — «одно из нескольких»

Тип, который в каждый момент хранит значение **одного** из перечисленных типов
(типобезопасный union):

```cpp
#include <variant>
std::variant<int, std::string> v = 42;
if (std::holds_alternative<int>(v)) { int n = std::get<int>(v); }
// или универсально:
std::string s = std::visit([](const auto& x){ return to_text(x); }, v);
```

## Идея 6. Исключения против кодов ошибок: две философии

Исторически в C ошибки возвращали **кодом** (или через `errno`): функция отдаёт `int`,
а ноль/не-ноль или специальное значение сигналит о провале. C++ добавил **исключения** —
отдельный канал, по которому ошибка «всплывает» сама, минуя все промежуточные `return`.
Это не вопрос вкуса — у каждого подхода своя механика и своя цена.

**Как это работает под капотом.** Код ошибки — это обычное возвращаемое значение: вызов
стоит ровно столько же, сколько любой `return`, а проверять его обязан **сам**
вызывающий, на каждом уровне, вручную пробрасывая выше. Исключение работает иначе:
`throw` ищет ближайший подходящий `catch`, по дороге **разматывая стек** (stack
unwinding) — вызывая деструкторы всех локальных объектов на пути. Современные ABI
реализуют это по модели **«zero-cost»**: пока исключение не брошено, накладных расходов
почти нет (исполнение идёт по «горячему» пути без проверок); зато сам `throw` — операция
дорогая (поиск обработчика по таблицам, размотка).

**Когда что выбирать.**

| Признак | Скорее код/`optional` | Скорее исключение |
|---|---|---|
| Ситуация ожидаема и частая | да | нет |
| Ошибка локальна, обработка тут же | да | нет |
| Нужно «пробросить» через много слоёв | нет | да |
| Нарушен контракт (баг вызывающего) | нет | да |
| Горячий цикл, где `throw` бил бы по скорости | да | нет |

**Главная ловушка кодов ошибок — их легко молча проигнорировать:**

```cpp
FILE* f = std::fopen("data.txt", "r");
// забыли проверить f на nullptr → разыменование nullptr → UB
std::fread(buf, 1, 100, f);
```
Исключение проигнорировать нельзя: непойманное долетает до `main` и завершает программу
с `std::terminate` — шумно, но честно. Поэтому правило-ориентир: **исключения — для
того, что нарушает контракт или редко; `optional`/`Result` — для ожидаемых исходов,
которые вызывающий обязан рассмотреть.**

**Частая ошибка — исключение как поток управления.** `throw`, чтобы выйти из цикла или
вернуть «не нашёл», — антипаттерн: и медленно, и нечитаемо.

```cpp
// ПЛОХО: «нет элемента» — нормальный исход, а не катастрофа
int find(const std::vector<int>& v, int key) {
    for (int x : v) if (x == key) return x;
    throw std::runtime_error("not found");   // дорого и не к месту
}
// ЛУЧШЕ:
std::optional<int> find(const std::vector<int>& v, int key) {
    for (int x : v) if (x == key) return x;
    return std::nullopt;
}
```

## Идея 7. Уровни гарантий безопасности при исключениях

Когда функция может бросить, важно знать, **в каком состоянии останется программа**,
если это произойдёт. Эйб Кёниг и Том Каргилл формализовали четыре уровня (от слабого к
сильному); это словарь, на котором инженеры договариваются о контрактах.

1. **Никакой гарантии (no guarantee).** После исключения объект может быть в любом, в
   том числе сломанном, состоянии: утечки, висячие указатели, разорванные инварианты.
   Это всегда баг — такого кода в продакшене быть не должно.

2. **Базовая гарантия (basic).** После исключения **нет утечек** и все инварианты
   соблюдены, но конкретное значение может быть любым допустимым. Объект «жив и валиден»,
   просто, возможно, частично изменён. Это минимум, к которому надо стремиться.

3. **Строгая гарантия (strong) — «всё или ничего».** Если операция бросила, состояние
   откатывается так, **будто её не было**. Классическая реализация — идиома
   **copy-and-swap**:

   ```cpp
   class Buffer {
       std::vector<int> data_;
   public:
       void append_all(const std::vector<int>& more) {
           std::vector<int> copy = data_;        // 1. работаем на копии
           copy.insert(copy.end(), more.begin(), more.end()); // может бросить — но
                                                 //    оригинал data_ цел
           data_.swap(copy);                     // 2. swap у vector — noexcept
       }                                         //    подмена атомарна по эффекту
   };
   ```
   Ключ: вся «опасная» работа идёт на копии; финальный `swap` **не бросает**, поэтому
   точка «фиксации» неуязвима. Если бросок случится до swap — оригинал нетронут.

4. **Гарантия отсутствия исключений (nothrow / `noexcept`).** Операция не бросает
   никогда. На таких операциях держится строгая гарантия у всех остальных (тот самый
   `swap`), поэтому move-конструкторы и деструкторы стремятся делать `noexcept`.

**Почему это не абстракция.** `std::vector::push_back` даёт строгую гарантию **только
если** move-конструктор элемента — `noexcept`. Если move может бросить, vector при
реаллокации вынужден **копировать** (медленнее), чтобы при сбое не потерять данные.
Отсюда практический вывод: помечай move-операции `noexcept`, и стандартная библиотека
наградит тебя и скоростью, и строгой гарантией.

**Частая ошибка — мнимая строгая гарантия:**

```cpp
void update(std::map<int,int>& m, int k, int v) {
    m.erase(k);          // уже изменили состояние...
    m[k] = compute(v);   // ...а это может бросить → строгой гарантии НЕТ
}
```
Здесь между `erase` и присваиванием состояние уже разрушено: бросок оставит карту без
ключа. Чтобы получить строгую гарантию, сначала вычисли новое значение (то, что может
бросить), и лишь потом меняй контейнер «безопасными» шагами.

## Идея 8. `std::optional<T>` глубже: устройство, доступ, монадичность

Под капотом `std::optional<T>` — это `T` плюс булев флаг «занято/пусто», лежащие
**в одном блоке памяти без кучи** (значение хранится в выровненном буфере прямо внутри
optional). Поэтому он дёшев: ни аллокаций, ни косвенности — в отличие от
`std::unique_ptr<T>`, который тоже «либо T, либо ничего», но через указатель и кучу.

**Способы доступа и их семантика — это источник самых частых багов:**

```cpp
std::optional<int> o = get();
*o;            // НЕ проверяет: если пусто — UB (как разыменовать невалидный ptr)
o.value();     // проверяет: если пусто — бросает std::bad_optional_access
o.value_or(0); // если пусто — вернуть запасное значение, без броска
if (o) ...     // o в bool-контексте: true, если значение есть
o.has_value(); // то же явно
```
Запомни различие `*o` и `o.value()`: первое — быстрое, но небезопасное (контракт:
«я уже проверил»); второе — безопасное, но бросает. **Никогда** не пиши `*get_optional()`
без проверки — это UB, а не исключение.

**Частая ошибка — `optional<bool>` и двусмысленный `if`:**

```cpp
std::optional<bool> flag = read_flag();
if (flag) { ... }        // ВНИМАНИЕ: это «значение ЕСТЬ», а не «значение == true»!
if (flag && *flag) { ... } // правильно: и присутствует, и истинно
```

**Цепочки (C++20).** В C++20 у `optional` появились монадические методы, которые красиво
сцепляют преобразования, не утопая в `if`:

```cpp
std::optional<int> r =
    to_int(s)                       // optional<int>
      .transform([](int x){ return x * 2; })   // если есть — преобразовать
      .and_then([](int x) -> std::optional<int> {
          return x > 0 ? std::optional{x} : std::nullopt; // вернуть новый optional
      });
```
`transform` оборачивает результат обратно в `optional`; `and_then` ждёт, что лямбда уже
вернёт `optional` (для шагов, которые сами могут «обнулиться»). Пустой `optional`
проскакивает всю цепочку насквозь, не вызывая лямбд.

## Идея 9. Свой `Result` / `Expected<T, E>` — ошибка как значение

`optional` говорит «значение или **ничего**», но не объясняет, **почему** ничего. Часто
нужно вернуть либо результат, либо **описание ошибки**. В C++23 для этого есть
`std::expected<T, E>`, но в **C++20 его нет** — и это отличный повод собрать свой,
заодно поняв, как он устроен.

Идея — хранить ровно одно из двух (`T` или `E`) в `std::variant<T, E>` и дать удобный
интерфейс: `has_value()`, `value()`, `error()`.

```cpp
template <typename T, typename E>
class Expected {
    std::variant<T, E> data_;   // index 0 == значение, index 1 == ошибка
public:
    Expected(T v) : data_(std::in_place_index<0>, std::move(v)) {}
    static Expected fail(E e) {
        Expected x; x.data_.template emplace<1>(std::move(e)); return x;
    }
    bool has_value() const noexcept { return data_.index() == 0; }
    // value()/error() — с проверкой; при неверном доступе бросают.
private:
    Expected() = default;
};
```

**Тонкость №1 — почему `std::in_place_index`.** Если `T == E` (например,
`Expected<int,int>`), то `std::variant<int,int>` **нельзя** инициализировать значением:
компилятор не знает, в какую из двух одинаковых альтернатив его положить — конструктор
неоднозначен. `std::in_place_index<0>` снимает двусмысленность, явно указывая слот.

**Тонкость №2 — почему отдельная фабрика `fail()`.** При `T == E` обычный конструктор от
значения не отличит «успех» от «ошибки» — оба же `int`. Именованная фабрика делает
намерение явным в коде вызова: `Expected<int,int>::fail(5)` нельзя спутать с
`Expected<int,int>(5)`.

**Тонкость №3 — что делает `value()`/`error()` при неверном доступе.** Это нарушение
контракта вызывающим («спросил значение у ошибки»), то есть логическая ошибка → разумно
бросить `std::logic_error`. Сравни с `optional::value()`, который бросает
`std::bad_optional_access`. Главное — **проверять** `index()` перед `std::get`, иначе
`std::get` сам бросит `std::bad_variant_access` (другой тип, чем обещано в контракте).

**Зачем это вообще, если есть исключения?** `Expected` делает ошибку **частью типа
возврата**: компилятор и читатель сразу видят, что функция может не сработать, а сама
ошибка несёт данные (код, сообщение, контекст). Это удобно на границах, где исключения
нежелательны (горячий путь, API без исключений), и там, где ошибок много и они «обычны».

## Идея 10. Свои типы исключений — иерархия и `what()`

Стандартных типов хватает не всегда: иногда нужен **свой** тип, чтобы вызывающий мог
поймать именно его. Наследуй от `std::exception` или (удобнее) от `std::runtime_error` —
тогда конструктор уже принимает сообщение и реализует `what()`:

```cpp
#include <stdexcept>

class DivideByZero : public std::runtime_error {
public:
    DivideByZero() : std::runtime_error("division by zero") {}
};

int safe_div(int a, int b) {
    if (b == 0) throw DivideByZero();
    return a / b;
}
```

**Почему наследовать от стандартной базы.** Тогда твой тип ловится и «прицельно», и
обобщённо — благодаря полиморфизму:

```cpp
try {
    safe_div(1, 0);
} catch (const DivideByZero& e) {        // самый специфичный — первым
    std::cout << "ноль!\n";
} catch (const std::runtime_error& e) {  // более общий — следом
    std::cout << "runtime: " << e.what();
} catch (const std::exception& e) {      // самый общий — последним
    std::cout << "что-то: " << e.what();
}
```
Порядок `catch` важен: они проверяются **сверху вниз**, и сработает первый подходящий по
типу. Поставишь `std::exception` первым — он перехватит всё, а специфичные блоки станут
мёртвым кодом (хороший компилятор предупредит).

**Частая ошибка — `throw` указателя или временного по значению с обрезкой.** Бросай
объект (`throw DivideByZero();`), а лови по `const&` (`catch (const DivideByZero& e)`).
Если ловить по значению базового типа (`catch (std::runtime_error e)`), произойдёт
**срезка** (slicing): «хвост» производного типа отрежется, а с ним и важная информация.

**Частая ошибка — голый `throw;` против `throw e;` при пробросе.** Внутри `catch`
конструкция `throw;` (без аргумента) **перебрасывает текущее** исключение, сохраняя его
динамический тип; а `throw e;` бросает **копию** объявленного типа и срежет производный.

```cpp
catch (const std::exception& e) {
    log(e.what());
    throw;     // правильно: летит дальше тот же объект того же типа
}
```

## Идея 11. Иерархия `std::exception`: логика деления

Стандартная библиотека уже поставляет разветвлённое дерево исключений. Понимание его
структуры экономит время: вместо того чтобы придумывать собственный базовый класс, часто
достаточно выбрать правильный стандартный предок.

```
std::exception
├── std::logic_error        — нарушение контракта, предотвратимо на этапе программирования
│   ├── std::invalid_argument
│   ├── std::out_of_range
│   ├── std::length_error
│   └── std::domain_error
├── std::runtime_error      — внешние условия, проявляются в рантайме
│   ├── std::range_error
│   ├── std::overflow_error
│   └── std::underflow_error
├── std::bad_alloc          — невозможно выделить память (operator new)
├── std::bad_cast           — dynamic_cast<T&> не прошёл по типу
└── std::bad_optional_access — обращение к пустому optional через .value()
```

**Главное разделение: `logic_error` vs `runtime_error`.**

`std::logic_error` (и его потомки) — это **нарушение предусловия со стороны
вызывающего**. Пример: передал отрицательный индекс функции, которая явно задокументирована
как «индекс ≥ 0». Теоретически такая ошибка **предотвратима** — программист должен был
проверить данные до вызова. Если вы видите `logic_error` в production — это баг кода, а
не случайное стечение обстоятельств.

`std::runtime_error` — ошибка **внешней среды**, которую не удалось предвидеть заранее:
сетевой сбой, неверные данные от пользователя, нехватка памяти на хосте, файл пропал во
время чтения. Программа написана правильно, но мир не совпал с ожиданием.

**Три «особых» класса за пределами этой пары.**

- `std::bad_alloc` бросает `operator new`, когда ОС отказывает в памяти. Перехватывать
  имеет смысл только если у вас есть реальный план «B» (освободить кэш, попробовать
  меньший блок). Молча проглотить — почти всегда неправильно.
- `std::bad_cast` бросает `dynamic_cast<T&>` при несовпадении типов (вариант с указателем
  вместо ссылки возвращает `nullptr`, а не бросает). Если у вас летит `bad_cast` — скорее
  всего нарушена архитектура (разыгрываете `dynamic_cast` там, где тип должен быть
  известен заранее).
- `std::bad_optional_access` бросает `optional::value()` при обращении к пустому
  `optional`. Это сигнал того же класса, что и `logic_error`: вызывающий должен был
  проверить `if (opt)` перед `.value()`.

**Практический совет при создании своих исключений.** Наследуй от `std::runtime_error`
для ошибок среды и от `std::logic_error` — для нарушений контракта. Тогда пользователи
твоей библиотеки смогут ловить их широким `catch (const std::logic_error&)`, не зная
твои конкретные типы, — и это будет осмысленно.

## Идея 12. `assert` и `static_assert` — третий инструмент дисциплины ошибок

«Tour of C++» (гл. 3.5) прямо называет три инструмента, образующих полную систему:
исключения, коды ошибок/optional, и assert. Они не конкурируют — каждый закрывает свою
нишу.

**Когда что применять.**

| Ситуация | Инструмент |
|---|---|
| Внешняя ошибка, которую нельзя предотвратить заранее | Исключение |
| Ожидаемый «пустой» исход нормального потока | `std::optional` / код ошибки |
| Нарушен инвариант / предусловие — это **баг** кода | `assert` |
| Условие можно проверить **на этапе компиляции** | `static_assert` |

**`assert(expr)` — проверка инвариантов в рантайме.**

```cpp
#include <cassert>

int binary_search(const std::vector<int>& v, int target) {
    assert(std::is_sorted(v.begin(), v.end()));  // предусловие: массив отсортирован
    // ...
}
```

`assert` — не обработчик ошибок. Он документирует и проверяет то, что **не должно
случиться при правильном коде**: нарушенный инвариант, невозможное состояние, предусловие
функции, которое вызывающий обязан соблюдать. При срабатывании он вызывает `abort()` —
немедленное, шумное завершение (дамп ядра на Unix). В release-сборке ассерты
традиционно отключаются (`-DNDEBUG`), что важно знать: не клади в `assert` код с
побочными эффектами.

```cpp
// ПЛОХО: при NDEBUG функция не вызовется совсем
assert(initialize_subsystem());

// ХОРОШО: побочный эффект отделён от утверждения
bool ok = initialize_subsystem();
assert(ok);
```

Разница с исключением принципиальная: исключение — это путь выполнения для **ожидаемой**
ненормальной ситуации; assert — это «здесь программа сломана, продолжать нет смысла».
Смешивать их — запутывать семантику: если ассерт «поймать» нельзя, а бросать исключение
на нарушение инварианта означает предлагать продолжить работу со сломанным состоянием.

**`static_assert(expr, msg)` — проверка на этапе компиляции.**

```cpp
template <typename T>
void serialize(const T& val) {
    static_assert(std::is_trivially_copyable_v<T>,
                  "serialize требует trivially copyable тип");
    // ...
}
```

`static_assert` вычисляется компилятором: если выражение — `false`, компиляция обрывается
с человекочитаемым сообщением. Это лучший инструмент для проверки шаблонных предположений,
размеров структур, совместимости типов — всего, что можно доказать статически. Никакого
рантайм-оверхеда; сообщение об ошибке появляется сразу при сборке.

```cpp
// Убеждаемся, что размер структуры не «поплыл» после рефакторинга
struct PacketHeader { uint8_t type; uint8_t flags; uint16_t length; };
static_assert(sizeof(PacketHeader) == 4, "PacketHeader должен быть ровно 4 байта");
```

**Подвох — не используй `assert` вместо обработки ошибок.**

```cpp
// ПЛОХО: если файл не открылся — abort() в production
FILE* f = fopen(path, "r");
assert(f != nullptr);   // агрессивно и нечестно: это не «невозможно», а внешняя ошибка

// ХОРОШО: внешняя ошибка → исключение или возврат кода
if (!f) throw std::runtime_error("не удалось открыть файл: " + std::string(path));
```

`assert` уместен только для того, что доказуемо невозможно при правильном вызове —
постусловий твоего собственного алгоритма или предусловий, которые по контракту
гарантирует вызывающий.

## Идея 13. `noexcept` как оператор vs спецификатор; условный `noexcept`

В C++ слово `noexcept` встречается в двух принципиально разных ролях, которые легко
перепутать. Это не просто синтаксический каприз: зная разницу, можно писать шаблонный
код, который автоматически наследует правильную гарантию.

**`noexcept` как спецификатор — обещание функции.**

```cpp
void swap(T& a, T& b) noexcept;      // эта функция не бросает — обещание
void swap(T& a, T& b) noexcept(true); // эквивалентно
void risky(T& x) noexcept(false);     // может бросать (по умолчанию)
```

Спецификатор стоит **после списка параметров** и задаёт контракт. Нарушение контракта
(исключение из `noexcept`-функции) → `std::terminate` без размотки стека. Это
намеренно: при `noexcept` компилятор может опустить таблицы размотки стека для тела
функции, что и даёт выигрыш в размере кода и в скорости горячего пути.

**`noexcept(expr)` как **оператор** — вычисляет `bool` на этапе компиляции.**

```cpp
int x = 0;
constexpr bool b = noexcept(x + 1);  // true: сложение int не бросает
```

Оператор `noexcept(expr)` принимает **выражение** (не вычисляя его, только анализируя
типы) и возвращает `true`, если это выражение гарантированно не бросает. Это инструмент
для написания шаблонов.

**Условный `noexcept(noexcept(...))` — золотой шаблон.**

Шаблонная функция должна быть `noexcept` ровно тогда, когда `noexcept` то, что она делает
внутри. Для этого спецификатор получает аргумент, вычисляемый оператором:

```cpp
template <typename T>
void my_swap(T& a, T& b) noexcept(noexcept(T(std::move(a)))) {
    T tmp(std::move(a));
    a = std::move(b);
    b = std::move(tmp);
}
```

Здесь:
- Внешний `noexcept(...)` — **спецификатор**: «функция `noexcept` при условии...»
- Внутренний `noexcept(T(std::move(a)))` — **оператор**: «верни `true`, если
  move-конструктор `T` не бросает»

Результат: для `int` и для типов с `noexcept` move-конструктором `my_swap` автоматически
становится `noexcept`; для типов, чей move может бросить — не помечается `noexcept`. Без
этого паттерна пришлось бы либо навсегда оставить функцию «небезопасной», либо солгать
компилятору.

**Почему это важно для контейнеров стандартной библиотеки.**

`std::vector` при реаллокации выбирает между move и copy элементов на основе того, имеет
ли move-конструктор пометку `noexcept`. Если пометка есть — move (быстро, строгая
гарантия). Если нет — copy (медленно, но безопасно для строгой гарантии). Это ровно тот
же механизм, что и в `my_swap`: `std::move_if_noexcept` использует оператор `noexcept`
для выбора стратегии.

```cpp
// Напиши свой тип с правильными пометками:
class Buffer {
    std::vector<int> data_;
public:
    Buffer(Buffer&&) noexcept = default;             // move — noexcept
    Buffer& operator=(Buffer&&) noexcept = default;  // move assign — noexcept
    // vector при реаллокации теперь будет двигать Buffer, а не копировать
};
```

**Частая ошибка — чрезмерный `noexcept`.**

Помечать `noexcept` всё подряд ради «скорости» — антипаттерн. Если функция реально может
бросить (например, вызывает что-то, что выделяет память), фальшивая пометка лишь заменит
исключение на `std::terminate` — ошибка не исчезнет, а просто станет неотловимой.
Принцип: помечай `noexcept` только то, что ты доказуемо контролируешь; для шаблонов —
используй `noexcept(noexcept(...))`.

## Идея 14. `std::expected<T, E>` (C++23): стандартный Result-тип

В задании 7 ты написал собственный `Expected<T, E>` — это не просто упражнение: до C++23
именно такой подход был единственным переносимым способом вернуть «значение или ошибку»
без исключений. C++23 стандартизировал его как `std::expected<T, E>` (`<expected>`).

**Сигнатура и базовый интерфейс.**

```cpp
// C++23:
#include <expected>

std::expected<int, std::string> parse(const std::string& s);

auto r = parse("42");
if (r)                        // bool-контекст: true = успех
    use(*r);                  // *r == r.value() == значение
else
    log(r.error());           // ошибка

// Ошибочный результат создаётся через std::unexpected:
return std::unexpected{"неверный ввод"};
```

`std::unexpected<E>` — вспомогательный тип-обёртка, который сигнализирует компилятору:
«это ошибка, а не значение типа `E`». Без него при `T == E` конструктор неоднозначен —
ровно та же проблема, которую ты решил через `fail()` и `in_place_index` в задании 7.

**Монадические операции `and_then` / `transform` / `or_else`.**

C++23 добавил к `std::expected` тот же монадический стиль, что и C++20 для `optional`.
Это позволяет строить цепочки преобразований без «пирамиды if»:

```cpp
std::expected<Config, Error> result =
    read_file(path)                              // expected<string, Error>
        .and_then(parse_toml)                    // если ok — применить; ждёт expected<Config,Error>
        .transform([](Config c) {                // если ok — преобразовать значение
            c.normalize();
            return c;
        })
        .or_else([](Error e) {                   // если ошибка — попробовать восстановиться
            return std::expected<Config, Error>{default_config()};
        });
```

- `and_then(f)` — вызывает `f(value)` только при успехе; `f` должна вернуть
  `expected<U, E>`; при ошибке цепочка продолжается «насквозь» с той же ошибкой.
- `transform(f)` — вызывает `f(value)` при успехе; `f` возвращает `U`, а не `expected`
  (аналог `optional::transform`).
- `or_else(f)` — вызывает `f(error)` при ошибке; `f` должна вернуть `expected<T, F>`;
  позволяет подменять тип ошибки или «восстанавливаться» из ошибки.

**Сравнение: рукописный `Expected<T,E>` из задания 7 vs `std::expected`.**

| Аспект | Рукописный `Expected<T,E>` | `std::expected<T,E>` (C++23) |
|---|---|---|
| Хранилище | `std::variant<T,E>` с `in_place_index` | Специализированное хранилище в библиотеке |
| Создание ошибки | Именованная фабрика `fail(e)` | `std::unexpected{e}` |
| Монадические методы | Нет (только `has_value`/`value`/`error`) | `and_then` / `transform` / `or_else` |
| Совместимость при `T == E` | `in_place_index` + `fail()` | `std::unexpected<E>` |
| Доступ к значению | `value()` → `std::logic_error` если ошибка | `value()` → `std::bad_expected_access<E>` |
| Стандарт | C++20 (вручную) | C++23 (`<expected>`) |

**Ключевой вывод.** Рукописная версия из задания 7 — учебное упражнение: она наглядно
показывает, как `std::variant`, `in_place_index` и явные фабрики решают проблему
неоднозначности при `T == E`. В production C++23-коде используй стандартный
`std::expected` — он предоставляет тот же смысл, но с более богатым интерфейсом и без
«самописного колеса». До C++23 (то есть на C++20, стандарте этого курса) применяй
именно такой рукописный `Expected` или специализированные библиотеки (например,
`tl::expected`, которая предоставляет ту же семантику на C++14/17/20).

<details><summary>Задание 6 — parse_int, нюанс 3 (переполнение)</summary>

</details>

<details><summary>Задание 7 — Expected, доступ к variant</summary>
Внутри лежит `std::variant<T, E>`. «Что внутри» узнаёт `data_.index()`: 0 — значение,
1 — ошибка. `has_value()` — это просто `index() == 0`. В `value()`/`error()` сперва
проверь `has_value()`, и **только если всё согласовано** бери `std::get<0>` / `std::get<1>`.
Если доступ неверный — `throw std::logic_error(...)`. Не вызывай `std::get` без проверки:
он бросит `std::bad_variant_access`, а тесты ждут именно `std::logic_error`.
</details>

<details><summary>Задание 7 — Expected, почему operator bool помечен explicit</summary>
`explicit` не даёт `Expected` молча превратиться в `bool` там, где этого не ждёшь
(например, `if (a == b)` при сравнении двух Expected). Внутри `if (e)` явное приведение
всё равно срабатывает — это контекстное преобразование. Реализация — буквально
`return has_value();`.
</details>

<details><summary>Задание 8 — safe_div и тип исключения</summary>

сообщением). Тебе нужно лишь в нужном месте `throw DivideByZero();`. Главное — это
именно `DivideByZero`, а **не** `std::invalid_argument` из задания 1: тест ловит
конкретный тип. Поскольку он наследник `std::runtime_error`, его поймает и
`catch (const std::runtime_error&)`, и `catch (const std::exception&)` — это и проверяет
один из тестов.
</details>

или `std::visit`), обсудим плюсы каждого. И по `Expected` подумай вслух: чем твой
ручной `Expected` отличается от `std::optional` и от будущего `std::expected` — это
хороший разговор про дизайн API ошибок.
