# Модуль 12 — Инструменты и качество кода

Junior отличается от новичка не тем, что не делает ошибок, а тем, что умеет их **быстро
находить** правильными инструментами, а не «вглядыванием в код». Этот модуль — про
инструментарий. Задание здесь необычное: код уже написан, но **содержит баги**. Твоя
работа — найти и починить их, по дороге освоив инструменты.

## Идея 1. Отладчик (gdb)

Отладчик позволяет остановить программу и посмотреть, что происходит:
```bash
g++ -std=c++20 -g program.cpp -o program     # -g: добавить отладочную информацию
gdb ./program
(gdb) break main          # точка останова
(gdb) run
(gdb) next                # шаг (не заходя в функции)
(gdb) step                # шаг (заходя внутрь)
(gdb) print x             # значение переменной
(gdb) backtrace           # стек вызовов (особенно полезно при падении)
(gdb) continue
```
Когда программа падает — запусти её под gdb, выполни `run`, потом `backtrace`: увидишь,
*где именно* случился сбой. Это в разы быстрее, чем расставлять `std::cout`.

## Идея 2. Санитайзеры — детекторы ошибок памяти и UB

Это, возможно, самый ценный инструмент C++. Пересобери с флагами — и рантайм сам
поймает выход за границы, use-after-free, утечки, переполнения:
```bash
g++ -std=c++20 -fsanitize=address,undefined -g program.cpp -o program
./program     # при ошибке выведет ТОЧНОЕ место: файл, строку, вид ошибки
```
- **ASan** (`address`) — выход за границы массива, обращение к освобождённой памяти, утечки.
- **UBSan** (`undefined`) — переполнение знакового int, кривые приведения, и т.п.

Привыкай гонять тесты под санитайзером — он ловит то, что «случайно работает».

### Как ASan работает под капотом

ASan — это не магия и не виртуальная машина. Компилятор делает две вещи. Во-первых,
**инструментирует** каждое обращение к памяти: перед чтением/записью по адресу `p`
вставляется проверка «а можно ли вообще трогать `p`?». Во-вторых, рантайм-библиотека
подменяет аллокаторы (`malloc`/`free`, `new`/`delete`) и операции со стеком.

Главный механизм — **теневая память** (shadow memory). Каждые 8 байт «настоящей» памяти
описываются 1 байтом теневой: он говорит, сколько из этих 8 байт сейчас «доступны».
Адрес теневого байта вычисляется простой формулой `shadow = (addr >> 3) + offset`, то есть
проверка — это пара дешёвых инструкций, а не поход в таблицу. Когда ты выделяешь массив,
вокруг него ASan ставит **красные зоны** (redzones) — куски памяти, помеченные в тени как
«трогать нельзя». Шаг за границу массива на 1 байт попадает в redzone → проверка падает →
`heap-buffer-overflow`. Когда ты делаешь `delete`, освобождённый блок помечается как
**отравленный** (poisoned) и какое-то время не переиспользуется (карантин), поэтому
обращение к нему ловится как `heap-use-after-free`, а не «случайно работает».

Что ASan **ловит**: выход за границы кучи/стека/глобалей, use-after-free,
use-after-return (локальная переменная, адрес которой утёк из функции), двойной `free`,
утечки памяти (LeakSanitizer включён в ASan по умолчанию на Linux). Что он **не ловит**:
большинство UB, не связанного с адресами (знаковое переполнение, сдвиг на отрицательное
число, разыменование `nullptr` он ловит частично) — для этого есть UBSan. Поэтому их
почти всегда включают **вместе**: `-fsanitize=address,undefined`.

Цена: программа под ASan работает примерно в 2 раза медленнее и ест в разы больше памяти.
Это **инструмент для тестов и отладки**, а не для прода. В CI — гоняй тесты под
санитайзером; в релизной сборке — выключай.

Пример того, что ASan показывает пальцем, а «глаза» — нет:
```cpp
std::vector<int> v{1, 2, 3};
int s = 0;
for (std::size_t i = 0; i <= v.size(); ++i)  // <= вместо < : на последней итерации i == 3
    s += v[i];                                // v[3] — выход за границу!
```
Без санитайзера этот код часто «работает» и даже даёт правдоподобную сумму — потому что
по адресу `v[3]` случайно лежит что-то читаемое. Под ASan — мгновенный
`heap-buffer-overflow` с точной строкой. Именно такие баги санитайзер превращает из
«иногда падает на проде» в «падает сразу, с адресом».

### UBSan: ловит неопределённое поведение

UBSan инструментирует операции, у которых в стандарте есть запрещённые случаи. Самый
частый улов — **знаковое переполнение**: `int a = INT_MAX; a + 1;` это UB (а не «обёртка
в минус»), и компилятор вправе оптимизировать в расчёте на то, что переполнения нет.
UBSan вставит проверку и на рантайме скажет `signed integer overflow`. Другие частые
улова: сдвиг `1 << 35` для 32-битного int, деление `INT_MIN / -1`, разыменование
невыровненного указателя, выход enum за диапазон. UBSan дешевле ASan и часто оставляют
включённым даже в нагрузочных прогонах. Запомни главный вывод: **«у меня переполнилось и
стало отрицательным» — это не определённое поведение, на которое можно опираться**; это
UB, и сегодня оно «работает», а завтра оптимизатор его выкинет.

## Идея 2б. assert и предусловия — контракт функции

`assert(условие)` из `<cassert>` — это проверка-утверждение: если условие ложно, программа
немедленно падает с сообщением (файл, строка, текст условия). Это инструмент для
**ошибок программиста**, а не пользователя.

Ключевая идея — **контракт функции**. У многих функций есть *предусловия*: то, что
обязан гарантировать вызывающий. «`average(v)` определена только для непустого `v`»,
«`sqrt_int(n)` требует `n >= 0`», «индекс `< size`». Контракт можно либо
*документировать и проверять* (`assert`), либо *перекладывать в типы/исключения*. assert
говорит: «это ошибка в коде, который меня вызвал; чини вызов, а не данные».

```cpp
double average(const std::vector<int>& v) {
    assert(!v.empty() && "average: вектор не должен быть пустым");  // предусловие
    long long sum = 0;
    for (int x : v) sum += x;
    return static_cast<double>(sum) / v.size();   // постусловие держится само
}
```

Механика и тонкости:
- **assert исчезает в релизе.** Если определён макрос `NDEBUG` (его ставят релизные
  сборки, `-DNDEBUG`), `assert(expr)` разворачивается в *ничто*. Отсюда правило №1:
  **никогда не клади в `assert` выражение с побочным эффектом.** `assert(pop() == 5);` в
  релизе не вызовет `pop()` вообще — и поведение программы изменится. assert проверяет, а
  не делает.
- **assert — для инвариантов и предусловий, не для пользовательского ввода.** Если число
  пришло из файла или из сети и может быть кривым — это *ожидаемая* ситуация, её
  обрабатывают исключением или кодом ошибки, а не падением. assert ловит то, чего «не
  может быть, если код правильный».
- **Трюк с `&& "сообщение"`.** Строковый литерал всегда «истина», поэтому
  `assert(cond && "пояснение")` не меняет логику, но печатает пояснение в отчёте — видно,
  *какой* контракт нарушен.
- **assert документирует.** Читающий код сразу видит предусловие. Это исполняемая
  документация, которая не устаревает: если контракт нарушат в отладочной сборке —
  программа сама об этом крикнет.

Когда что выбирать: `assert` — баг в коде (нарушен внутренний инвариант / предусловие,
которое обязан был соблюсти вызывающий программист). Исключение/код ошибки — нештатные,
но *ожидаемые* входные данные (файл не открылся, число не распарсилось). Смешивать вредно:
если на кривой ввод пользователя падать через assert, в релизе (где assert выключен) код
молча поедет дальше с мусором.

## Идея 2в. Целочисленное переполнение — и как считать безопасно

Знаковое переполнение `int` — это **UB**, а не «оборачивание». Классическая ловушка —
«невинное» среднее двух чисел:
```cpp
int mid = (a + b) / 2;   // если a и b близки к INT_MAX, a + b переполняется ДО деления
```
При `a = b = 2'000'000'000` сумма `4'000'000'000` не влезает в 32-битный знаковый int
(предел ~2.1 млрд) → UB. UBSan это поймает, но лучше вообще так не писать. Это не
теоретическая придирка: ровно такой баг много лет жил в бинарном поиске в стандартных
библиотеках — индексы `low` и `high` большие, `(low+high)/2` переполняется.

Правильная идиома — считать **смещение от одного из концов**, оно всегда влезает:
```cpp
int mid = a + (b - a) / 2;   // (b - a) — половина расстояния, прибавляем к меньшему концу
```
Здесь нет суммы двух больших чисел. (Тонкость: если `a` и `b` разного знака и далеки друг
от друга, `b - a` тоже может переполниться; для полностью устойчивого варианта берут
беззнаковую арифметику или 64-битный промежуточный тип. Но для «обоих концов одного
знака» — а это случай индексов — `a + (b - a)/2` безопасен и идиоматичен.)

Запомни принцип шире: **не доверяй промежуточным результатам, что они влезут.** Когда
складываешь/умножаешь близкие к пределу значения, либо считай через более широкий тип
(`long long`), либо переформулируй выражение так, чтобы большого промежуточного значения
не возникало.

## Идея 3. Форматирование и статический анализ

- **clang-format** — автоформатирование по стилю. `clang-format -i file.cpp`. В команде
  единый стиль избавляет от споров о скобках и отступах.
- **clang-tidy** — статический анализатор: находит подозрительные места *без запуска*
  (висячие указатели, ненужные копии, устаревшие конструкции).
  `clang-tidy file.cpp -- -std=c++20`.

## Идея 3б. Читай предупреждения компилятора — это бесплатный ревьюер

Компилятор видит куда больше, чем кажется. `-Wall -Wextra` включают десятки проверок,
которые ловят баги **на этапе компиляции**, бесплатно и без запуска:
```bash
g++ -std=c++20 -Wall -Wextra program.cpp -o program
```
Что они находят (и что каждое реально значит):
- **`-Wsign-compare`** — сравнение знакового со беззнаковым. `for (int i = 0; i < v.size();)`
  сравнивает `int` с `size_t`; если `v.size()` огромен или `i` уходит в минус, сравнение
  даёт неожиданный результат. Лечится правильным типом счётчика (`std::size_t`).
- **`-Wunused-variable` / `-Wunused-parameter`** — заведённое и неиспользованное часто
  означает «забыл дописать логику» или опечатку в имени.
- **`-Wmaybe-uninitialized`** — чтение переменной по пути, где ей могли не присвоить
  значение. Чтение неинициализированной памяти — UB.
- **`-Wreturn-type`** — не все пути функции возвращают значение. Это тоже UB при
  достижении такого пути; компилятор предупреждает не зря.
- **`-Wshadow`** (входит в расширенный набор) — локальная переменная «затеняет» внешнюю с
  тем же именем; классический источник «почему присваивание не сработало».

Как с этим жить:
1. **Не игнорируй предупреждения.** Каждое — это компилятор, говорящий «вот тут, скорее
   всего, баг». Чистый вывод сборки — признак аккуратного кода.
2. **`-Werror` в CI** превращает предупреждения в ошибки сборки — так они не накапливаются
   (в этом учебном проекте `-Werror` намеренно выключен, чтобы стабы собирались, но в
   реальном проекте включай). Локально полезно: один раз словив `-Werror`, ты больше не
   проедешь мимо.
3. **Читай предупреждение как адрес + диагноз.** В нём есть файл, строка, и *что именно*
   не так. Это та же дисциплина, что с санитайзером: не «примерно где-то», а точная точка.

Правило junior'а: **сборка с предупреждениями — это не «зелёная» сборка.** Прежде чем
сказать «работает», убедись, что компилятор молчит.

## Идея 4. Отладка: системный метод вместо угадывания

Отладка — это не «вглядываться и думать», а **сужение пространства гипотез** измеримыми
шагами. Дисциплина важнее инструмента:

1. **Воспроизведи стабильно.** Баг, который ловится через раз, чинить почти невозможно.
   Сначала найди вход, на котором он *всегда* проявляется (часто это сам падающий тест).
2. **Локализуй.** Где именно неверно? Метод половинного деления: проверяй промежуточные
   значения в середине подозрительного участка. Не угадывай функцию — *измеряй*, в какой
   половине пути значение уже испортилось.
3. **Сформулируй гипотезу и проверь её одним изменением.** «Думаю, цикл идёт на 1 шаг
   дальше» → проверь границу. Меняй по одной вещи за раз, иначе не поймёшь, что помогло.
4. **Объясни причину прежде, чем чинить.** Если не можешь сказать, *почему* возникает баг,
   ты, скорее всего, лечишь симптом, а не причину — и он вернётся.

Инструменты под эту дисциплину:
- **gdb + `backtrace`** при падении — мгновенно даёт точку сбоя (см. Идею 1). На вопрос
  «где упало» отвечает за секунды, заменяя десяток `std::cout`.
- **Санитайзер** — для багов памяти/UB; даёт адрес и вид ошибки (Идея 2).
- **`print` / точка останова с условием** (`break file.cpp:42 if i == 3`) — остановиться
  ровно на интересной итерации, а не щёлкать `next` сотню раз.
- **Минимальный воспроизводящий пример.** Урезай вход, пока баг ещё есть: на 3 элементах
  отладить в разы легче, чем на 10 000. Часто, пока ужимаешь пример, причина становится
  очевидной сама.

## Идея 4. Минимум git для junior

```bash
git status                 # что изменилось
git add -p                 # добавить изменения по кускам (осознанно)
git commit -m "понятное сообщение в настоящем времени"
git log --oneline          # история
git diff                   # что именно поменялось
git checkout -b feature-x  # ветка под задачу
```
Коммить маленькими логичными шагами с осмысленными сообщениями — это часть качества.

### Подсказки

<details><summary>Как подступиться</summary>
Не угадывай. Запусти `./cpp/run.sh 12`, прочитай, КАКОЙ тест и КАК падает (ожидалось /
получилось). Это сразу сужает поиск до одной функции. Для «странных» падений —

</details>
<details><summary>Классы багов, которые тут спрятаны</summary>

самим собой, целочисленное деление вместо дробного. Каждый — частая реальная ошибка.
</details>

Когда найдёшь баг — прежде чем чинить, сформулируй мне, *почему* он возникает. Умение
объяснить причину важнее самой правки. Покажи — обсудим.

### Задание 6. Безопасный `at` с проверкой границ

`int checked_at(const std::vector<int>& v, std::size_t i)` — возвращает `v[i]`, но при
`i >= v.size()` **бросает `std::out_of_range`** (а не лезет за границу). Это ровно тот
выход за пределы, который в задании про баги поймал бы ASan — а здесь ты делаешь проверку
*явной частью контракта*. Это поведение `std::vector::at` против `operator[]`.

Подумай: сообщение в исключении полезно делать информативным (индекс и размер), но тесты
проверяют **тип** исключения, а не текст.

### Задание 7. Функция с предусловием через `assert`

`int int_sqrt(int n)` — целочисленный квадратный корень: наибольшее `r >= 0`, такое что
`r*r <= n`. Например `int_sqrt(0)=0`, `int_sqrt(1)=1`, `int_sqrt(8)=2`, `int_sqrt(9)=3`,
`int_sqrt(15)=3`, `int_sqrt(16)=4`.

**Предусловие:** `n >= 0`. Отрицательный `n` — это *ошибка вызывающего кода* (баг), а не
ожидаемый ввод, поэтому защищай контракт через `assert(n >= 0 && "...")`, а не через
исключение. Тесты подают только корректные (неотрицательные) входы — на них функция должна
давать правильный ответ; assert при этом не срабатывает.

Считай аккуратно: остерегайся переполнения при `r*r` (см. Идею 2в) — используй `long long`
для промежуточного произведения или цикл/двоичный поиск, который не выходит за предел.

### Задание 8. Среднее двух int без переполнения

`int midpoint_int(int a, int b)` — округлённое **вниз к меньшему концу** среднее `a` и `b`,
посчитанное так, чтобы **не было переполнения** даже у больших значений. Наивное
`(a + b) / 2` переполняется (Идея 2в) — пиши `a + (b - a) / 2`.

Контракт: считаем, что `a <= b` (тогда `b - a >= 0` и деление округляет к `a`). Тесты дают
именно такие пары, в том числе у самой границы `INT_MAX`, где наивная сумма уже UB.
Например `midpoint_int(2, 4) == 3`, `midpoint_int(1, 4) == 2` (округление вниз),
`midpoint_int(INT_MAX-1, INT_MAX) == INT_MAX-1`.

### Подсказки к новым заданиям

<details><summary>Задание 6 — checked_at</summary>
Сначала проверь условие `i >= v.size()`; если выполнено — `throw std::out_of_range(...)`.
Иначе верни `v[i]`. Не забудь `#include <stdexcept>`. Подумай, почему сравнение с
`v.size()` (тип `size_t`) и `i` типа `size_t` — без предупреждения `-Wsign-compare`.
</details>
<details><summary>Задание 7 — int_sqrt</summary>
Простейший корректный способ: перебирай `r` от 0, пока `(long long)r * r <= n`; как только
превысило — ответ предыдущий `r`. Промежуточное произведение бери в `long long`, иначе
`r*r` переполнит `int`. Бинарный поиск по `r` в `[0, n]` — то же самое, но быстрее;
сравнение `mid*mid` тоже считай в `long long`. assert ставь в самом начале функции.
</details>
<details><summary>Задание 8 — midpoint_int</summary>
Ключ — не складывать `a + b`. Запиши среднее как «меньший конец плюс половина расстояния»:
прибавляй к `a` величину `(b - a) / 2`. Поскольку `a <= b`, `b - a` неотрицательно и в
`int` влезает, а большой суммы не возникает. Проверь руками на паре у `INT_MAX`.
</details>

Перед тем как звать меня — прогони новые задания **под санитайзером**
(`-fsanitize=address,undefined`): корректная реализация должна проходить чисто, без единого
срабатывания ASan/UBSan. Если UBSan ругается на переполнение в `int_sqrt` или
`midpoint_int` — значит, промежуточное вычисление всё-таки вышло за предел; вернись к Идее 2в.
