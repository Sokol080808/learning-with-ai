# Модуль 13 — Структуры данных, алгоритмы и сложность (Big-O)

На собеседованиях junior это спрашивают почти всегда. Большая идея: для каждой задачи
есть структура данных, которая делает её быстрой, — и умение оценить **сложность**
важнее, чем знание наизусть конкретной реализации. В этом модуле мы не только считаем
Big-O, но и **строим структуры своими руками**: связный список, стек на массиве, дерево
поиска. Когда ты собрал контейнер из узлов и указателей сам, ты навсегда понимаешь,
почему `std::list::operator[]` не существует, а `std::map` держит данные отсортированными.

## Идея 1. Big-O — как растёт стоимость с размером данных

Big-O описывает, как меняется число операций при росте входа `n`:

| Обозначение | Название | Пример |
|---|---|---|
| `O(1)` | константа | доступ `v[i]`, `push_back` (в среднем) |
| `O(log n)` | логарифм | бинарный поиск в отсортированном массиве |
| `O(n)` | линейно | один проход по массиву |
| `O(n log n)` | | хорошая сортировка (`std::sort`) |
| `O(n²)` | квадрат | два вложенных цикла по `n` |

Мы оцениваем **рост**, отбрасывая константы: `3n + 10` — это `O(n)`. Вопрос не «сколько
секунд», а «во сколько раз медленнее станет, если данных станет вдвое больше».

**Что Big-O скрывает.** Big-O — это *асимптотика*: поведение при больших `n`. Он
сознательно выбрасывает константы и младшие слагаемые. Поэтому алгоритм `O(n)` с большой
константой (например, обход списка с прыжками по памяти и промахами кэша) на реальных
размерах может проигрывать `O(n log n)` с маленькой константой. Big-O говорит, *кто
победит на бесконечности*, а не *кто быстрее на твоих 1000 элементах*. Это не повод его
игнорировать — это повод **измерять**, когда разница принципиальна.

**Худший, средний и амортизированный случай — это разные вещи.**

- *Худший случай* (worst-case) — гарантия сверху. Бинарный поиск — `O(log n)` всегда.
- *Средний случай* (average-case) — ожидание по «типичным» данным. Поиск в хеш-таблице
  в среднем `O(1)`, но в худшем (все ключи в одну корзину) — `O(n)`.
- *Амортизированный* (amortised) — средняя стоимость операции по длинной серии, даже
  если отдельная операция дорогая. `vector::push_back` — амортизированно `O(1)`: иногда
  он переаллоцирует и копирует весь буфер за `O(n)`, но это случается всё реже (буфер
  растёт геометрически, обычно ×2), поэтому в среднем на один `push_back` приходится
  константа. Не путай амортизированный `O(1)` с «всегда быстро»: один конкретный
  `push_back` может оказаться очень дорогим.

**Частая ошибка — складывать вместо умножения.** Два последовательных цикла по `n` —
это `O(n) + O(n) = O(n)`. Два *вложенных* — `O(n) · O(n) = O(n²)`. Различай «друг за
другом» и «друг в друге».

```cpp
for (int x : a) { /* ... */ }   // O(n)
for (int y : a) { /* ... */ }   // ещё O(n) → суммарно O(n)

for (int x : a)                 // внешний O(n)
  for (int y : a)               // внутренний O(n)
    /* ... */;                  // → O(n²)
```

## Идея 2. Память: где живут данные и почему кэш решает

Чтобы понимать структуры, надо понимать, *где физически лежат* их элементы.

- **Непрерывная память** (`std::vector`, `std::array`, C-массив): элементы лежат подряд,
  адрес `i`-го = база + `i·sizeof(T)`. Отсюда `O(1)` доступ по индексу. И — важнее —
  процессор, читая один элемент, подтягивает в кэш соседние (cache line обычно 64 байта).
  Обход подряд лежащих данных почти бесплатен: следующий элемент уже в кэше.
- **Россыпь по куче** (узлы списка/дерева): каждый узел — отдельный `new`, узлы могут
  лежать где угодно. Переход «к следующему» — это разыменование указателя в случайный
  адрес. Если его нет в кэше — *промах кэша* (cache miss), сотни тактов простоя. Поэтому
  обход связного списка на практике в разы медленнее обхода `vector` той же длины, хотя
  оба `O(n)`.

Вывод, который удивляет новичков: **`std::vector` почти всегда — правильный выбор по
умолчанию**, даже когда «по теории» список выглядит уместнее. Список оправдан, когда ты
часто вставляешь/удаляешь в середине *по уже известному узлу* (итератору) и не ходишь по
индексам. Не выбирай список «по привычке» или «потому что вставка O(1)» — посчитай, какие
операции у тебя реально частые, и помни про кэш.

## Идея 3. Связный список — узлы, указатели, владение

Связный список — это цепочка узлов. Каждый **узел** хранит значение и ссылку на следующий
узел (`next`); у односвязного списка ссылка одна (вперёд), у двусвязного — две (`next` и
`prev`). Сам список держит указатель на **голову** (`head`), иногда ещё на **хвост**
(`tail`).

```
head → [1|•] → [2|•] → [3|nullptr]
```

**Почему вставка в начало — `O(1)`.** Чтобы добавить элемент в голову, создаём узел, его
`next` направляем на старую голову, а `head` — на новый узел. Ни одного сдвига, ни одного
обхода. Сравни с `vector`, где вставка в начало сдвигает все элементы — `O(n)`.

**Почему доступ по индексу — `O(n)`.** «Дай 5-й элемент» означает «шагни по `next` пять
раз». Случайного доступа нет, поэтому у `std::list` нет `operator[]` — язык не даёт тебе
притвориться, что это дёшево.

**Владение и время жизни.** Узлы живут в куче. Кто их освобождает? В C++ идиоматичный
ответ — **`std::unique_ptr`**: список владеет головой через `unique_ptr<Node>`, голова
владеет следующим узлом через свой `unique_ptr<Node>` и так далее. Получается цепочка
владения: когда список разрушается, разрушается голова, та разрушает следующий узел,
и так до конца — без единого ручного `delete`. (Тонкость: при *очень* длинном списке такое
рекурсивное разрушение может переполнить стек; для учебных размеров это не проблема.)

```cpp
struct Node {
    int value;
    std::unique_ptr<Node> next;   // узел ВЛАДЕЕТ следующим
    explicit Node(int v) : value(v), next(nullptr) {}
};
std::unique_ptr<Node> head;       // список владеет головой
```

**Как развернуть список (`reverse`) за `O(n)` без копий.** Идём по списку, на каждом шаге
«переворачиваем стрелку»: `next` текущего узла должен указывать на *предыдущий*. Нужны три
указателя: `prev` (куда смотреть назад, сперва `nullptr`), `curr` (текущий), `next`
(сохранённый следующий, иначе потеряем хвост). Классическая «перецепка трёх указателей».
С `unique_ptr` это делается через `std::move`, чтобы передавать владение, а не копировать
узлы.

**Самые частые ошибки со списком:**

1. *Потеря хвоста при перецепке.* Если переписать `curr->next` раньше, чем сохранить
   старое значение, остаток списка утечёт/потеряется. Всегда сначала сохрани `next`.
2. *Забыть обновить `size_`.* Если ведёшь счётчик размера — меняй его в каждой
   мутирующей операции, иначе `size()` начнёт врать.
3. *Ходить по индексам.* `for (i) list[i]` превращает `O(n)` обход в `O(n²)`: каждый
   `list[i]` сам по себе `O(i)`. По списку ходят итератором/по `next`, а не по индексам.
4. *`push_back` без `tail`.* Без указателя на хвост `push_back` каждый раз идёт от головы
   до конца — `O(n)`. С `tail_` он становится `O(1)`. Для учебной задачи допустимо и `O(n)`,
   но понимай разницу.

## Идея 4. Динамический массив — как устроен `vector` изнутри

`std::vector` — это не магия, а три поля: указатель на буфер в куче, текущий размер
(`size`) и ёмкость (`capacity`). `size ≤ capacity`. Пока есть запас (`size < capacity`),
`push_back` просто кладёт элемент и увеличивает `size` — `O(1)`. Когда запас кончился,
вектор:

1. выделяет **новый** буфер большего размера (обычно `capacity ×= 2`),
2. перемещает/копирует туда все старые элементы — `O(n)`,
3. освобождает старый буфер.

Поэтому отдельный `push_back` иногда стоит `O(n)`, но **амортизированно** — `O(1)`:
геометрический рост гарантирует, что суммарная стоимость `n` вставок есть `O(n)`, то есть
`O(1)` на штуку (см. Идею 1). Если бы буфер рос на +1 каждый раз, каждый `push_back`
копировал бы всё заново и `n` вставок стоили бы `O(n²)` — поэтому растят именно умножением.

**Практические следствия:**

- `reserve(n)` заранее выделяет ёмкость и убирает переаллокации в цикле — полезная
  оптимизация, когда финальный размер известен.
- Переаллокация **инвалидирует** все указатели/итераторы/ссылки на элементы вектора:
  буфер переехал. Это источник коварных багов — нельзя держать ссылку на `v[0]`, а потом
  делать `push_back`.
- Удаление из середины (`erase`) — `O(n)`: дырку закрывают сдвигом хвоста.

В этом модуле твой стек на массиве (Задание 6) использует `vector` как растущий буфер —
это нормально и идиоматично: ты переиспользуешь готовую амортизацию, а не изобретаешь её.

## Идея 5. Стек и очередь — поведение, а не контейнеры

Стек и очередь — это **дисциплины доступа**, а не отдельные структуры. Их строят поверх
массива или списка.

- **Стек (Stack, LIFO)** — «последним вошёл, первым вышел»: `push` кладёт наверх, `pop`
  снимает верхний, `top` смотрит верхний. Естественно строится поверх `vector`:
  добавляем/убираем с *конца* — `O(1)` (конец вектора — дешёвый край). Стек повсюду:
  стек вызовов функций, отмена операций (undo), проверка скобок, обход в глубину (DFS).
- **Очередь (Queue, FIFO)** — «первым вошёл, первым вышел»: `push` в хвост, `pop` из
  головы. Наивно поверх `vector` плохо: удаление из начала — `O(n)`. Поэтому очередь
  строят на двусвязном списке или на *кольцевом буфере*. Очередь — это обход в ширину
  (BFS), планировщики задач, буферы.

**Граничные случаи — главное в этих структурах.** Что делает `pop()` или `top()` на
*пустом* стеке? Есть две школы:

1. **UB / неопределённое поведение** — как `std::stack::top` на пустом или
   `vector::back()` на пустом: быстро, но при ошибке программа падает или читает мусор.
2. **Бросить исключение** — например `std::out_of_range`. Чуть медленнее, но ошибка
   *диагностируема*: ты видишь, что обратился к пустому стеку, а не ловишь случайный
   segfault.

В Задании 6 ты выбираешь второй путь осознанно: `pop()`/`top()` на пустом обязаны бросить
**`std::out_of_range`**. Это делает контейнер «честным»: нарушение контракта превращается
в явную, ловимую ошибку.

```cpp
const T& top() const {
    if (buf_.empty())
        throw std::out_of_range("ArrayStack::top on empty");
    return buf_.back();
}
```

**Частые ошибки со стеком/очередью:**

1. *Не проверить пустоту* перед `top`/`pop` — UB или исключение «на ровном месте».
2. *Перепутать конец, с которым работаешь.* На `vector` дёшев только *задний* край.
   Стек кладёт и снимает с зада — хорошо. Если попробуешь снимать с начала, получишь
   `O(n)` на операцию.
3. *Возвращать копию из `top()` вместо ссылки.* `top()` обычно отдаёт `const T&` — без
   лишнего копирования; копию делает уже вызывающий, если хочет.

## Идея 6. Бинарное дерево поиска (BST) — упорядоченность как инвариант

BST — это дерево, где у каждого узла есть значение и до двух детей (`left`, `right`), и
держится **инвариант поиска**: в левом поддереве — только значения *меньше* узла, в
правом — только *больше*. Этот инвариант — весь смысл структуры.

```
        5
       / \
      3   8
     / \   \
    1   4   9
```

**Поиск (`contains`) — `O(h)`, где `h` — высота.** Сравниваем искомое с корнем: равно —
нашли; меньше — уходим влево; больше — вправо. Каждый шаг отсекает целое поддерево.
Если дерево *сбалансировано*, высота `h ≈ log₂ n`, и поиск — `O(log n)`. Это как
бинарный поиск, но по «живой» структуре, в которую можно дёшево вставлять.

**Вставка (`insert`) — тоже `O(h)`.** Идём вниз по тому же правилу, пока не упрёмся в
пустое место (`nullptr`), и подвешиваем туда новый лист. Дубликаты обычно игнорируют —
тогда дерево хранит *множество* значений.

**In-order обход даёт отсортированный порядок — и это не совпадение.** Обход «лево →
корень → право» (in-order) посещает узлы в точности по возрастанию: инвариант гарантирует,
что всё левое поддерево меньше корня, а всё правое — больше. Рекурсивно: обойди левое,
выведи себя, обойди правое.

```cpp
void inorder(const Node* n, std::vector<int>& out) {
    if (!n) return;
    inorder(n->left.get(),  out);   // всё, что меньше
    out.push_back(n->value);        // сам узел
    inorder(n->right.get(), out);   // всё, что больше
}
```

**Главная ловушка BST — вырождение.** Если вставлять *уже отсортированные* данные
(1, 2, 3, 4, 5), каждый новый элемент уходит только вправо, и дерево превращается в
связный список высотой `n`. Тогда поиск деградирует до `O(n)` — вся выгода исчезает.
Поэтому в продакшене используют **самобалансирующиеся** деревья (красно-чёрные в
`std::map`/`std::set`, AVL), которые перестраиваются и держат `h = O(log n)` при любом
порядке вставок. Балансировку мы здесь не пишем (это уже не junior-уровень), но понимать,
*зачем* она нужна, обязательно: «BST в среднем `O(log n)`, в худшем `O(n)`».

**Частые ошибки с BST:**

1. *Перепутать сторону сравнения* — `<` и `>` местами: дерево «работает», но инвариант
   нарушен, и in-order перестаёт быть отсортированным. Тест на сортированность это ловит.
2. *Не обработать дубликат* — если на равенстве всё равно создавать узел, `size()` и
   in-order начнут считать одно значение дважды.
3. *Рекурсия без базового случая* — забыл `if (!n) return;`, и обход падает на `nullptr`.
4. *Владение детьми.* Как и в списке, дети держатся через `unique_ptr<Node>` — родитель
   владеет поддеревьями, и всё дерево освобождается само при разрушении корня.

## Идея 7. Хеш-таблица — обмен памяти на скорость

`std::unordered_map`/`unordered_set` дают поиск/вставку в среднем за `O(1)` благодаря
хешированию: ключ прогоняется через хеш-функцию в номер «корзины», и мы идём сразу туда.
Классический приём: заменить «двойной цикл `O(n²)`» на «один проход + хеш-множество
`O(n)`». Задание `two_sum` — ровно про это.

Цена за скорость — память (корзины) и *средний* характер гарантии: при плохой хеш-функции
или злонамеренных ключах всё может выродиться в одну корзину и `O(n)`. И данные хранятся
**неупорядоченно** — если нужен отсортированный обход, бери `std::map` (дерево, `O(log n)`),
а не `unordered_map`.

## Идея 8. Поиск и сортировка

- **Линейный поиск** — `O(n)`, работает всегда, на любых данных.
- **Бинарный поиск** — `O(log n)`, но требует **отсортированный** массив. Идея: смотрим
  середину, отбрасываем половину, повторяем. Каждый шаг вдвое сужает зону поиска — отсюда
  логарифм.
- **Сортировки**: учебные `O(n²)` (пузырьковая, вставками) полезны для понимания
  механики; в бою используй `std::sort` (`O(n log n)`, интросортировка). Реализовать
  простую сортировку один раз — обязательно, чтобы понимать, что под капотом.

### Подсказки

<details><summary>Бинарный поиск — границы</summary>
Держи два индекса `lo = 0`, `hi = size - 1`. Пока `lo <= hi`: смотри середину
`mid = lo + (hi - lo) / 2` (почему так, а не `(lo+hi)/2`? — переполнение). Если
`v[mid] == target` — нашли; если меньше — ищи справа (`lo = mid+1`), иначе слева.
</details>

<details><summary>two_sum за O(n)</summary>
Идём по массиву. Для каждого `v[i]` нужное «дополнение» — `target - v[i]`. Проверь,
встречалось ли оно раньше (хранили в `unordered_map` значение→индекс). Если да — пара
найдена. Если нет — запиши `v[i] -> i` и иди дальше.
</details>

<details><summary>Список: с чего начать</summary>
Узел уже описан за тебя в приватной части (`struct Node { T value; unique_ptr<Node> next; }`)
плюс поля `head_` и `size_`. Сначала сделай простое: `empty()` — это `size_ == 0` (или
`head_ == nullptr`), `size()` — верни `size_`. Дальше — `push_front`.
</details>

<details><summary>Список: push_front</summary>
Создай новый узел: `auto node = std::make_unique<Node>(value);`. Его `next` должен
указывать на текущую голову — но голову нельзя *скопировать* (это `unique_ptr`), её надо
**переместить**: `node->next = std::move(head_);`. Затем `head_ = std::move(node);` и
не забудь `++size_;`.
</details>

<details><summary>Список: push_back и to_vector</summary>
`to_vector`: заведи `std::vector<T> out;`, иди по узлам сырым указателем
`for (const Node* p = head_.get(); p != nullptr; p = p->next.get())` и клади `p->value`.
`push_back`: если список пуст — это то же, что `push_front`. Иначе дойди по `next` до
последнего узла (у которого `next == nullptr`) и подвесь новый туда. Идти по цепочке
владения удобно через `Node* p = head_.get();` и `p = p->next.get();`.
</details>

<details><summary>Список: reverse без копий</summary>
Не трогай *значения* — переставляй *узлы*. Идея на сырых указателях классическая («три
указателя»): держи `prev`, `curr`, заранее сохраняй `next`, разворачивай стрелку
`curr->next = prev`, сдвигай тройку. С `unique_ptr` это делается через `std::move`:
по очереди «отстёгивай» голову от старого списка и «пристёгивай» её в начало нового.
Псевдоидея: пока старый список не пуст — оторви его первый узел, поставь его перед уже
накопленным результатом. В конце результат — это и есть развёрнутый список. Подумай, как
сделать это, ни разу не вызвав `make_unique` (новых узлов не создаём!).
</details>

<details><summary>ArrayStack: бросок на пустом</summary>
`push` — это `buf_.push_back(value)`. `size`/`empty` — делегируй вектору. Ключевое:
в `pop()` и `top()` **первым делом** проверь `buf_.empty()` и, если пусто,
`throw std::out_of_range("...")`. Только после проверки трогай `buf_.back()` /
`buf_.pop_back()`. Тип исключения именно `std::out_of_range` — тест проверяет конкретный
тип.
</details>

<details><summary>BST: рекурсия и unique_ptr</summary>
Удобный приём — приватный рекурсивный хелпер, принимающий `std::unique_ptr<Node>&`
(ссылку на «слот», куда указывает родитель). Тогда вставка на пустом месте — это просто
`slot = std::make_unique<Node>(value);`, и она автоматически «подвешивается» к родителю.
Сравни `value` с `slot->value`: меньше — рекурсия в `slot->left`, больше — в `slot->right`,
равно — ничего (дубликат), и не увеличивай `size_`. `size_` увеличивай ровно там, где
создаёшь новый узел.
</details>

<details><summary>BST: contains и inorder</summary>
`contains`: спускайся от `root_.get()` сырым указателем, сравнивая `value` с
`node->value` (меньше → `node->left.get()`, больше → `node->right.get()`), пока не
`nullptr` (не нашли) или не равенство (нашли). `inorder`: рекурсивный хелпер
`(const Node*, std::vector<int>& out)` — сперва левое поддерево, потом `out.push_back`
своего значения, потом правое. Базовый случай — `if (!node) return;`. Порядок лево-себя-право
и даёт возрастание.
</details>
