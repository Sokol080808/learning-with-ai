# Модуль 04 — Классы, ООП и введение в RAII

Большая идея: класс связывает **данные** и **поведение** и устанавливает **инварианты** —
правила, которые всегда истинны для корректного объекта. Хороший класс невозможно
привести в «сломанное» состояние извне. Это и есть инкапсуляция.

## Идея 1. `struct` vs `class`

Технически разница одна: в `struct` члены по умолчанию **public**, в `class` —
**private**. По соглашению: `struct` — для простых «мешков данных» без инвариантов,
`class` — когда есть логика и что защищать.

## Идея 2. Инкапсуляция и спецификаторы доступа

```cpp
class Fraction {
public:                 // интерфейс — доступен всем
    Fraction(int num, int den);
    int numerator() const;
private:                // реализация — скрыта, видна только классу
    int num_;
    int den_;
};
```
Поля делаем `private`, доступ — через методы. Зачем? Чтобы никто не выставил, скажем,
знаменатель в 0 в обход правил. Класс сам стоит на страже своих инвариантов.

## Идея 3. Конструктор и список инициализации

Конструктор приводит объект в корректное начальное состояние. Но *как* именно члены
получают значения — вопрос неочевидный, и здесь живёт целый класс ошибок.

### Механика: инициализация против присваивания

Когда выполнение доходит до тела конструктора (открывающей `{`), **все члены уже
сконструированы**. Список инициализации после `:` управляет тем, *как* они конструируются;
присваивание в теле — это уже второй шаг поверх готового объекта.

```cpp
Fraction::Fraction(int num, int den)
    : num_(num), den_(den)   // ← члены СРАЗУ строятся из этих значений
{
    // здесь можно дополнительно нормализовать (сократить дробь и т.п.)
}
```
Сравните с антипаттерном:
```cpp
Fraction::Fraction(int num, int den) {   // плохо для нетривиальных членов
    num_ = num;   // member сначала строится по умолчанию, ПОТОМ присваивается
    den_ = den;
}
```
Для `int` разница нулевая. Но для члена типа `std::string`, `std::vector` или другого
класса это **двойная работа**: сначала вызов конструктора по умолчанию, затем
`operator=`. Список инициализации делает один шаг вместо двух.

### Почему для некоторых членов список — единственный путь

- **`const`-члены**: `const int id_;` нельзя присвоить в теле — присваивание меняет
  объект, а он `const`. Их можно только проинициализировать.
- **Ссылки**: `int& ref_;` обязаны быть привязаны при рождении и не перепривязываются.
- **Члены без конструктора по умолчанию**: если у типа члена нет `T()`, его *нельзя*
  «сначала построить по умолчанию», значит, только список.

```cpp
class Sensor {
    const int id_;
    std::string& log_;          // ссылка
public:
    Sensor(int id, std::string& log) : id_(id), log_(log) {}  // обязательно список
};
```

### Коварная ловушка: порядок инициализации

Члены инициализируются **в порядке объявления в классе**, а НЕ в порядке записи в списке
инициализации. Если один член зависит от другого, легко получить чтение мусора.

```cpp
class Range {
    int hi_;
    int lo_;
    int size_;
public:
    // ОПАСНО: size_ объявлен ПОСЛЕ hi_/lo_, но что, если переставить местами?
    Range(int lo, int hi) : lo_(lo), hi_(hi), size_(hi_ - lo_) {}  // ок: size_ объявлен последним
};
```
Если бы `size_` стоял в классе **первым**, он бы инициализировался до `lo_`/`hi_` и читал
бы их неинициализированные значения — несмотря на «правильный» порядок в списке.
Компилятор с `-Wall` обычно предупреждает (`-Wreorder`). **Правило: пишите члены в
списке в том же порядке, в каком они объявлены в классе.**

### Делегирующие конструкторы (C++11)

Один конструктор может вызвать другой, чтобы не дублировать логику:
```cpp
class Fraction {
public:
    Fraction(int num, int den);
    Fraction() : Fraction(0, 1) {}   // делегирует: «дробь по умолчанию = 0/1»
};
```

## Идея 4. Инварианты класса

**Инвариант** — это утверждение о состоянии объекта, которое истинно *всё время между
вызовами публичных методов*. Для `Fraction` инвариант такой:

> дробь всегда хранится сокращённой; знаменатель строго положителен; знак — в числителе.

Зачем это нужно? Инвариант — это **контракт, на который опирается весь остальной код**.
Если он гарантирован, `operator==` может сравнивать дроби покомпонентно (`1/2 == 1/2`),
не приводя их к общему виду каждый раз. Без инварианта `2/4` и `1/2` — «разные» объекты,
и сравнение пришлось бы делать через перекрёстное умножение всюду.

### Как инвариант устанавливается и держится

1. **Конструктор устанавливает инвариант.** После выхода из конструктора объект обязан
   быть корректным. Для `Fraction` это означает: нормализовать сразу при создании.
2. **Каждый публичный метод сохраняет инвариант.** Он вправе предполагать, что на входе
   объект корректен, и обязан оставить его корректным на выходе.
3. **`private` защищает инвариант.** Если бы `num_`/`den_` были публичными, любой код мог
   бы написать `f.den_ = 0;` и сломать всё. Приватность — это не «секретность ради
   секретности», а механизм, делающий нарушение инварианта *невозможным извне*.

```cpp
// Инвариант проверяют отладочными assert'ами — это документирует ожидания:
int Fraction::denominator() const {
    assert(den_ > 0);   // если сработает — где-то нарушили инвариант
    return den_;
}
```

### Частая ошибка: «дырявый» сеттер

```cpp
void set_denominator(int d) { den_ = d; }   // ПЛОХО: пускает 0 и отрицательные
```
Такой метод протыкает инкапсуляцию: инвариант снова можно сломать. Если сеттер нужен,
он обязан **сам восстанавливать инвариант** (проверить/нормализовать), иначе он не лучше
публичного поля.

## Идея 5. `const`-методы и const-correctness

```cpp
int numerator() const;   // const после () — «этот метод не меняет объект»
```
`const` после списка параметров делает неявный `this` указателем на константу
(`const Fraction*`). Внутри такого метода попытка изменить любой член — ошибка
компиляции. Это даёт две вещи:

- такие методы **можно вызывать у `const`-объектов и по `const`-ссылкам** — а именно так
  объекты чаще всего и передают (`const Fraction&`). Метод без `const` у `const`-объекта
  вызвать нельзя;
- компилятор ловит случайное изменение там, где вы обещали только читать.

```cpp
void print(const Fraction& f) {
    std::cout << f.numerator();   // работает ТОЛЬКО если numerator() помечен const
}
```
**Правило: помечайте `const` каждый метод, который логически только читает.** Это не
«опционально для красоты» — без `const` ваш класс почти невозможно использовать в
`const`-контексте, а его в современном C++ большинство.

## Идея 6. Деструктор и RAII (главная идея модуля!)

**Деструктор** вызывается автоматически, когда объект уничтожается: для локальной
переменной — при выходе из её области видимости (включая выход через `return`,
`break` и даже **через исключение** при раскрутке стека). Из этой гарантии рождается
ключевая идиома C++ — **RAII** (Resource Acquisition Is Initialization):

> Ресурс (память, файл, сокет, блокировка, замер времени) захватывается в
> **конструкторе** и освобождается в **деструкторе**. Тогда он освобождается *сам* —
> гарантированно, даже если посреди функции выбросили исключение.

```cpp
class FileGuard {
public:
    explicit FileGuard(const char* path) { f_ = std::fopen(path, "r"); }  // захват
    ~FileGuard() { if (f_) std::fclose(f_); }                             // освобождение
    FileGuard(const FileGuard&) = delete;             // см. ниже: страж некопируем
    FileGuard& operator=(const FileGuard&) = delete;
private:
    std::FILE* f_;
};
```

### Почему RAII надёжнее ручного освобождения

Рассмотрим «ручной» вариант:
```cpp
void process() {
    std::FILE* f = std::fopen("data.txt", "r");
    do_work(f);           // ← если бросит исключение — fclose НИКОГДА не вызовется
    std::fclose(f);       // утечка дескриптора
}
```
С RAII закрытие привязано к деструктору, а деструктор вызывается при **любом** выходе из
области видимости, включая раскрутку стека при исключении. Утечь невозможно.

### Порядок разрушения

Объекты уничтожаются в порядке, **обратном** созданию (как стопка тарелок: последняя
положенная снимается первой). Это важно, когда ресурсы зависят друг от друга: блокировку
разумно отпускать после того, как разрушены объекты, которые её использовали.

### Самая частая ошибка: копируемый страж

Если RAII-объект владеет ресурсом, его нельзя бездумно копировать: иначе два объекта
будут считать себя владельцами одного ресурса, и деструктор отработает дважды (двойное
`fclose`, двойной `delete`, двойное освобождение). Поэтому стражи обычно **запрещают
копирование** (`= delete`) или делают перемещаемыми. RAII — причина, по которой в
современном C++ почти не пишут `delete` руками: за вас это делают деструкторы контейнеров
и умных указателей. Мы вернёмся к RAII в модулях 05, 06, 09.

## Идея 7. Перегрузка операторов

Класс может определить, что для него значат `+`, `==`, `<`, `<<` и т.д. Перегрузка
оператора — это **обычный метод (или свободная функция) с особым именем** `operatorX`.
Запись `a + b` компилятор превращает либо в `a.operator+(b)` (если оператор — метод),
либо в `operator+(a, b)` (если свободная функция).

### Метод или свободная функция?

- **Бинарный оператор как метод**: левый операнд — это `*this`. Подходит, когда слева
  всегда ваш тип: `Fraction + Fraction` → `Fraction::operator+`.
- **Свободная функция**: нужна, когда левый операнд — *чужой* тип, который вы не можете
  изменить. Классический пример — вывод в поток:
  ```cpp
  std::ostream& operator<<(std::ostream& os, const Fraction& f);
  ```
  Здесь левый операнд — `std::ostream`, а не `Fraction`, поэтому методом класса
  `Fraction` это записать нельзя — пришлось бы добавлять метод в `std::ostream`.

### Канонические сигнатуры (запомните формы)

```cpp
// Симметричные бинарные операторы НЕ меняют операнды → const-метод, аргумент по const-ссылке,
// результат — новый объект по значению:
Fraction operator+(const Fraction& o) const;

// Сравнения возвращают bool и тоже ничего не меняют:
bool operator==(const Fraction& o) const;
bool operator<(const Fraction& o) const;

// Вывод в поток: ПЕРВЫЙ аргумент и возвращаемое значение — ссылка на тот же поток.
// Возврат потока обязателен, чтобы работала цепочка  os << a << b << c;
std::ostream& operator<<(std::ostream& os, const Fraction& f);
```
Почему `operator<<` возвращает `std::ostream&`? Потому что `os << a << b` разбирается как
`(os << a) << b`: результат `os << a` снова должен быть потоком, чтобы к нему применился
следующий `<< b`. Вернёте `void` — цепочка не скомпилируется.

### Согласованность операторов (важная дисциплина)

Если вы определили `==`, то `!=` должен означать «не `==`». Если определили `<`, то
порядок должен быть **строгим** (`a < a` всегда `false`) и согласованным с `==`.
Несогласованные операторы — источник коварных багов: контейнер `std::set<Fraction>` и
алгоритм `std::sort` опираются именно на то, что `<` задаёт корректный строгий порядок.

```cpp
// Типичная ошибка: нестрогое "<", где a < a даёт true → UB в std::sort/std::set.
bool operator<(const Fraction& o) const { return num_*o.den_ <= o.num_*den_; }  // ПЛОХО: <=
```
В C++20 для многих типов `<`, `<=`, `>`, `>=` можно сгенерировать одной строкой через
`operator<=>` (spaceship), но в этом задании мы пишем операторы вручную — чтобы вы поняли
механику, прежде чем доверять её компилятору.

### Когда НЕ стоит перегружать

Перегружайте оператор, только если его смысл **очевиден и привычен**. Для дроби `+` и `<`
естественны. А вот `+` для «склеивания» двух пользователей или `<` со смыслом «дешевле» —
почти всегда плохая идея: читатель кода ждёт арифметику/порядок, а не сюрприз.

## Идея 8. Правило нуля (и правило пяти)

У класса есть **пять** «специальных функций»: деструктор, конструктор копирования,
оператор копирующего присваивания, конструктор перемещения, оператор перемещающего
присваивания. Компилятор умеет генерировать их автоматически — и почти всегда делает это
правильно, **почленно** (скопировать/переместить каждый член).

### Правило нуля

> Если вашему классу не нужно вручную управлять ресурсом, **не пишите ни одной из пяти**.
> Доверьте всё компилятору.

```cpp
struct Point {       // правило нуля во всей красе
    double x, y;
};                   // копирование, перемещение, разрушение — всё сгенерировано и верно
```
`Fraction` и `Vec2` в этом модуле — тоже правило нуля: их члены (`int`, `double`)
копируются тривиально, и автоматические версии корректны. Поэтому мы не объявляем у них
ни деструктор, ни конструктор копирования — и это *правильно*, а не лень.

### Правило пяти

Как только вы **руками** написали деструктор (обычно потому, что освобождаете ресурс),
автоматически сгенерированные копирование/перемещение почти наверняка станут
**неправильными** — они скопируют сырой указатель, и два объекта попытаются освободить
один ресурс.

> Если вы определили один из пяти — продумайте **все пять** (определите или явно
> `= delete`).

```cpp
class Buffer {
    int* data_;
    std::size_t n_;
public:
    explicit Buffer(std::size_t n) : data_(new int[n]), n_(n) {}
    ~Buffer() { delete[] data_; }                  // написали деструктор...
    Buffer(const Buffer&) = delete;                // ...значит, обязаны решить про копирование
    Buffer& operator=(const Buffer&) = delete;     // тут — запретили (иначе двойной delete)
};
```
В этом модуле `Timer` владеет «логическим ресурсом» (замером, который при разрушении
записывается во внешний счётчик) и имеет нетривиальный деструктор. Поэтому мы по правилу
пяти **запрещаем его копирование** через `= delete` — копия страж-объекта не имеет смысла
и приводила бы к двойному учёту времени. Лучший способ соблюсти правило пяти — по
возможности вообще не управлять ресурсом вручную (правило нуля), а оборачивать его в уже
готовый RAII-тип (`std::unique_ptr`, `std::vector`, `std::fstream`).

4. `Fraction multiply(const Fraction& o) const` — произведение.

a/b + c/d = (a*d + c*b) / (b*d). После — снова нормализуй (это сделает конструктор,
если ты создашь новый Fraction из этих чисел). Совет: если конструктор нормализует,
то <code>add</code>/<code>multiply</code> могут просто возвращать
<code>Fraction{...}</code> с сырыми числами.
</details>

<details><summary>Задание 2 — operator+ и operator* без дублирования</summary>
Логику сложения и умножения ты уже написал в <code>add</code> и <code>multiply</code>.
Оператор может просто вызвать соответствующий метод и вернуть его результат — не копируй
формулы дважды.
</details>

<details><summary>Задание 2 — operator&lt; через целые</summary>
Сравнивать <code>a/b</code> и <code>c/d</code> через <code>double</code> — плохо
(погрешность). Лучше: поскольку знаменатели положительны, <code>a/b &lt; c/d</code>
равносильно <code>a*d &lt; c*b</code>. Убедись, что используешь строгое <code>&lt;</code>,
а не <code>&lt;=</code> — иначе равные дроби окажутся «меньше» друг друга.
</details>

<details><summary>Задание 2 — operator&lt;&lt;</summary>
Самый простой путь: у тебя уже есть <code>to_string()</code>. Выведи его в поток и верни
сам поток. Почему важно вернуть <code>os</code>, а не <code>void</code> — перечитай Идею 7
про цепочки <code>os &lt;&lt; a &lt;&lt; b</code>.
</details>

<details><summary>Задание 3 — как снять время</summary>
Возьми момент так: <code>std::chrono::steady_clock::now()</code>. Чтобы получить число
наносекунд, переведи <code>.time_since_epoch()</code> в наносекунды:
<code>std::chrono::duration_cast&lt;std::chrono::nanoseconds&gt;(...).count()</code>.
В деструкторе сними момент ещё раз и прибавь разницу к <code>*accumulator_ns_</code>.
Почему <code>steady_clock</code>, а не <code>system_clock</code>? steady-часы не «скачут»
назад при коррекции системного времени — для измерения интервалов это то, что нужно.
</details>

Скалярное произведение двух 2D-векторов — это одно число: <code>x*o.x + y*o.y</code>.
Если оно равно нулю, векторы перпендикулярны; если оба вектора одинаковы — получишь
квадрат длины.
</details>
