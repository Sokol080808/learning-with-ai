# Модуль 04 — Классы, ООП и введение в RAII

Большая идея: класс связывает **данные** и **поведение** и устанавливает **инварианты** —
правила, которые всегда истинны для корректного объекта. Хороший класс невозможно
привести в «сломанное» состояние извне. Это и есть инкапсуляция.

## Идея 1. `struct` vs `class`

Технически разница одна: в `struct` члены по умолчанию **public**, в `class` —
**private**. По соглашению: `struct` — для простых «мешков данных» без инвариантов,
`class` — когда есть логика и что защищать.

## Идея 2. Инкапсуляция и спецификаторы доступа

```cpp
class Fraction {
public:                 // интерфейс — доступен всем
    Fraction(int num, int den);
    int numerator() const;
private:                // реализация — скрыта, видна только классу
    int num_;
    int den_;
};
```
Поля делаем `private`, доступ — через методы. Зачем? Чтобы никто не выставил, скажем,
знаменатель в 0 в обход правил. Класс сам стоит на страже своих инвариантов.

## Идея 3. Конструктор и список инициализации

Конструктор приводит объект в корректное начальное состояние. Но *как* именно члены
получают значения — вопрос неочевидный, и здесь живёт целый класс ошибок.

### Механика: инициализация против присваивания

Когда выполнение доходит до тела конструктора (открывающей `{`), **все члены уже
сконструированы**. Список инициализации после `:` управляет тем, *как* они конструируются;
присваивание в теле — это уже второй шаг поверх готового объекта.

```cpp
Fraction::Fraction(int num, int den)
    : num_(num), den_(den)   // ← члены СРАЗУ строятся из этих значений
{
    // здесь можно дополнительно нормализовать (сократить дробь и т.п.)
}
```
Сравните с антипаттерном:
```cpp
Fraction::Fraction(int num, int den) {   // плохо для нетривиальных членов
    num_ = num;   // member сначала строится по умолчанию, ПОТОМ присваивается
    den_ = den;
}
```
Для `int` разница нулевая. Но для члена типа `std::string`, `std::vector` или другого
класса это **двойная работа**: сначала вызов конструктора по умолчанию, затем
`operator=`. Список инициализации делает один шаг вместо двух.

### Почему для некоторых членов список — единственный путь

- **`const`-члены**: `const int id_;` нельзя присвоить в теле — присваивание меняет
  объект, а он `const`. Их можно только проинициализировать.
- **Ссылки**: `int& ref_;` обязаны быть привязаны при рождении и не перепривязываются.
- **Члены без конструктора по умолчанию**: если у типа члена нет `T()`, его *нельзя*
  «сначала построить по умолчанию», значит, только список.

```cpp
class Sensor {
    const int id_;
    std::string& log_;          // ссылка
public:
    Sensor(int id, std::string& log) : id_(id), log_(log) {}  // обязательно список
};
```

### Коварная ловушка: порядок инициализации

Члены инициализируются **в порядке объявления в классе**, а НЕ в порядке записи в списке
инициализации. Если один член зависит от другого, легко получить чтение мусора.

```cpp
class Range {
    int hi_;
    int lo_;
    int size_;
public:
    // ОПАСНО: size_ объявлен ПОСЛЕ hi_/lo_, но что, если переставить местами?
    Range(int lo, int hi) : lo_(lo), hi_(hi), size_(hi_ - lo_) {}  // ок: size_ объявлен последним
};
```
Если бы `size_` стоял в классе **первым**, он бы инициализировался до `lo_`/`hi_` и читал
бы их неинициализированные значения — несмотря на «правильный» порядок в списке.
Компилятор с `-Wall` обычно предупреждает (`-Wreorder`). **Правило: пишите члены в
списке в том же порядке, в каком они объявлены в классе.**

### Делегирующие конструкторы (C++11)

Один конструктор может вызвать другой, чтобы не дублировать логику:
```cpp
class Fraction {
public:
    Fraction(int num, int den);
    Fraction() : Fraction(0, 1) {}   // делегирует: «дробь по умолчанию = 0/1»
};
```

## Идея 4. Инварианты класса

**Инвариант** — это утверждение о состоянии объекта, которое истинно *всё время между
вызовами публичных методов*. Для `Fraction` инвариант такой:

> дробь всегда хранится сокращённой; знаменатель строго положителен; знак — в числителе.

Зачем это нужно? Инвариант — это **контракт, на который опирается весь остальной код**.
Если он гарантирован, `operator==` может сравнивать дроби покомпонентно (`1/2 == 1/2`),
не приводя их к общему виду каждый раз. Без инварианта `2/4` и `1/2` — «разные» объекты,
и сравнение пришлось бы делать через перекрёстное умножение всюду.

### Как инвариант устанавливается и держится

1. **Конструктор устанавливает инвариант.** После выхода из конструктора объект обязан
   быть корректным. Для `Fraction` это означает: нормализовать сразу при создании.
2. **Каждый публичный метод сохраняет инвариант.** Он вправе предполагать, что на входе
   объект корректен, и обязан оставить его корректным на выходе.
3. **`private` защищает инвариант.** Если бы `num_`/`den_` были публичными, любой код мог
   бы написать `f.den_ = 0;` и сломать всё. Приватность — это не «секретность ради
   секретности», а механизм, делающий нарушение инварианта *невозможным извне*.

```cpp
// Инвариант проверяют отладочными assert'ами — это документирует ожидания:
int Fraction::denominator() const {
    assert(den_ > 0);   // если сработает — где-то нарушили инвариант
    return den_;
}
```

### Частая ошибка: «дырявый» сеттер

```cpp
void set_denominator(int d) { den_ = d; }   // ПЛОХО: пускает 0 и отрицательные
```
Такой метод протыкает инкапсуляцию: инвариант снова можно сломать. Если сеттер нужен,
он обязан **сам восстанавливать инвариант** (проверить/нормализовать), иначе он не лучше
публичного поля.

## Идея 5. `const`-методы и const-correctness

```cpp
int numerator() const;   // const после () — «этот метод не меняет объект»
```
`const` после списка параметров делает неявный `this` указателем на константу
(`const Fraction*`). Внутри такого метода попытка изменить любой член — ошибка
компиляции. Это даёт две вещи:

- такие методы **можно вызывать у `const`-объектов и по `const`-ссылкам** — а именно так
  объекты чаще всего и передают (`const Fraction&`). Метод без `const` у `const`-объекта
  вызвать нельзя;
- компилятор ловит случайное изменение там, где вы обещали только читать.

```cpp
void print(const Fraction& f) {
    std::cout << f.numerator();   // работает ТОЛЬКО если numerator() помечен const
}
```
**Правило: помечайте `const` каждый метод, который логически только читает.** Это не
«опционально для красоты» — без `const` ваш класс почти невозможно использовать в
`const`-контексте, а его в современном C++ большинство.

## Идея 6. Деструктор и RAII (главная идея модуля!)

**Деструктор** вызывается автоматически, когда объект уничтожается: для локальной
переменной — при выходе из её области видимости (включая выход через `return`,
`break` и даже **через исключение** при раскрутке стека). Из этой гарантии рождается
ключевая идиома C++ — **RAII** (Resource Acquisition Is Initialization):

> Ресурс (память, файл, сокет, блокировка, замер времени) захватывается в
> **конструкторе** и освобождается в **деструкторе**. Тогда он освобождается *сам* —
> гарантированно, даже если посреди функции выбросили исключение.

```cpp
class FileGuard {
public:
    explicit FileGuard(const char* path) { f_ = std::fopen(path, "r"); }  // захват
    ~FileGuard() { if (f_) std::fclose(f_); }                             // освобождение
    FileGuard(const FileGuard&) = delete;             // см. ниже: страж некопируем
    FileGuard& operator=(const FileGuard&) = delete;
private:
    std::FILE* f_;
};
```

### Почему RAII надёжнее ручного освобождения

Рассмотрим «ручной» вариант:
```cpp
void process() {
    std::FILE* f = std::fopen("data.txt", "r");
    do_work(f);           // ← если бросит исключение — fclose НИКОГДА не вызовется
    std::fclose(f);       // утечка дескриптора
}
```
С RAII закрытие привязано к деструктору, а деструктор вызывается при **любом** выходе из
области видимости, включая раскрутку стека при исключении. Утечь невозможно.

### Порядок разрушения

Объекты уничтожаются в порядке, **обратном** созданию (как стопка тарелок: последняя
положенная снимается первой). Это важно, когда ресурсы зависят друг от друга: блокировку
разумно отпускать после того, как разрушены объекты, которые её использовали.

### Самая частая ошибка: копируемый страж

Если RAII-объект владеет ресурсом, его нельзя бездумно копировать: иначе два объекта
будут считать себя владельцами одного ресурса, и деструктор отработает дважды (двойное
`fclose`, двойной `delete`, двойное освобождение). Поэтому стражи обычно **запрещают
копирование** (`= delete`) или делают перемещаемыми. RAII — причина, по которой в
современном C++ почти не пишут `delete` руками: за вас это делают деструкторы контейнеров
и умных указателей. Мы вернёмся к RAII в модулях 05, 06, 09.

## Идея 7. Перегрузка операторов

Класс может определить, что для него значат `+`, `==`, `<`, `<<` и т.д. Перегрузка
оператора — это **обычный метод (или свободная функция) с особым именем** `operatorX`.
Запись `a + b` компилятор превращает либо в `a.operator+(b)` (если оператор — метод),
либо в `operator+(a, b)` (если свободная функция).

### Метод или свободная функция?

- **Бинарный оператор как метод**: левый операнд — это `*this`. Подходит, когда слева
  всегда ваш тип: `Fraction + Fraction` → `Fraction::operator+`.
- **Свободная функция**: нужна, когда левый операнд — *чужой* тип, который вы не можете
  изменить. Классический пример — вывод в поток:
  ```cpp
  std::ostream& operator<<(std::ostream& os, const Fraction& f);
  ```
  Здесь левый операнд — `std::ostream`, а не `Fraction`, поэтому методом класса
  `Fraction` это записать нельзя — пришлось бы добавлять метод в `std::ostream`.

### Канонические сигнатуры (запомните формы)

```cpp
// Симметричные бинарные операторы НЕ меняют операнды → const-метод, аргумент по const-ссылке,
// результат — новый объект по значению:
Fraction operator+(const Fraction& o) const;

// Сравнения возвращают bool и тоже ничего не меняют:
bool operator==(const Fraction& o) const;
bool operator<(const Fraction& o) const;

// Вывод в поток: ПЕРВЫЙ аргумент и возвращаемое значение — ссылка на тот же поток.
// Возврат потока обязателен, чтобы работала цепочка  os << a << b << c;
std::ostream& operator<<(std::ostream& os, const Fraction& f);
```
Почему `operator<<` возвращает `std::ostream&`? Потому что `os << a << b` разбирается как
`(os << a) << b`: результат `os << a` снова должен быть потоком, чтобы к нему применился
следующий `<< b`. Вернёте `void` — цепочка не скомпилируется.

### Согласованность операторов (важная дисциплина)

Если вы определили `==`, то `!=` должен означать «не `==`». Если определили `<`, то
порядок должен быть **строгим** (`a < a` всегда `false`) и согласованным с `==`.
Несогласованные операторы — источник коварных багов: контейнер `std::set<Fraction>` и
алгоритм `std::sort` опираются именно на то, что `<` задаёт корректный строгий порядок.

```cpp
// Типичная ошибка: нестрогое "<", где a < a даёт true → UB в std::sort/std::set.
bool operator<(const Fraction& o) const { return num_*o.den_ <= o.num_*den_; }  // ПЛОХО: <=
```
В C++20 для многих типов `<`, `<=`, `>`, `>=` можно сгенерировать одной строкой через
`operator<=>` (spaceship), но в этом задании мы пишем операторы вручную — чтобы вы поняли
механику, прежде чем доверять её компилятору.

### Когда НЕ стоит перегружать

Перегружайте оператор, только если его смысл **очевиден и привычен**. Для дроби `+` и `<`
естественны. А вот `+` для «склеивания» двух пользователей или `<` со смыслом «дешевле» —
почти всегда плохая идея: читатель кода ждёт арифметику/порядок, а не сюрприз.

## Идея 8. Правило нуля (и правило пяти)

У класса есть **пять** «специальных функций»: деструктор, конструктор копирования,
оператор копирующего присваивания, конструктор перемещения, оператор перемещающего
присваивания. Компилятор умеет генерировать их автоматически — и почти всегда делает это
правильно, **почленно** (скопировать/переместить каждый член).

### Правило нуля

> Если вашему классу не нужно вручную управлять ресурсом, **не пишите ни одной из пяти**.
> Доверьте всё компилятору.

```cpp
struct Point {       // правило нуля во всей красе
    double x, y;
};                   // копирование, перемещение, разрушение — всё сгенерировано и верно
```
`Fraction` и `Vec2` в этом модуле — тоже правило нуля: их члены (`int`, `double`)
копируются тривиально, и автоматические версии корректны. Поэтому мы не объявляем у них
ни деструктор, ни конструктор копирования — и это *правильно*, а не лень.

### Правило пяти

Как только вы **руками** написали деструктор (обычно потому, что освобождаете ресурс),
автоматически сгенерированные копирование/перемещение почти наверняка станут
**неправильными** — они скопируют сырой указатель, и два объекта попытаются освободить
один ресурс.

> Если вы определили один из пяти — продумайте **все пять** (определите или явно
> `= delete`).

```cpp
class Buffer {
    int* data_;
    std::size_t n_;
public:
    explicit Buffer(std::size_t n) : data_(new int[n]), n_(n) {}
    ~Buffer() { delete[] data_; }                  // написали деструктор...
    Buffer(const Buffer&) = delete;                // ...значит, обязаны решить про копирование
    Buffer& operator=(const Buffer&) = delete;     // тут — запретили (иначе двойной delete)
};
```
В этом модуле `Timer` владеет «логическим ресурсом» (замером, который при разрушении
записывается во внешний счётчик) и имеет нетривиальный деструктор. Поэтому мы по правилу
пяти **запрещаем его копирование** через `= delete` — копия страж-объекта не имеет смысла
и приводила бы к двойному учёту времени. Лучший способ соблюсти правило пяти — по
возможности вообще не управлять ресурсом вручную (правило нуля), а оборачивать его в уже
готовый RAII-тип (`std::unique_ptr`, `std::vector`, `std::fstream`).

## Идея 9. Наследование и полиморфизм

### 9.1 Три режима наследования

```cpp
class Base { public: int x; protected: int y; private: int z; };

struct Pub  : public    Base {};   // x→public,    y→protected
struct Prot : protected Base {};   // x→protected, y→protected
struct Priv : private   Base {};   // x→private,   y→private
```

| Режим | Зачем |
|---|---|
| `public` | «является» (is-a): `Derived*` конвертируется в `Base*`. Основа полиморфизма. |
| `protected` | Деталь реализации иерархии. Редко нужен. |
| `private` | «реализован через»: внешний мир не знает о базе. Альтернатива компоновке. |

В большинстве случаев нужен `public`. `protected`/`private` — нишевые инструменты.

### 9.2 Виртуальный диспатч и механика vtable

Ключевое слово `virtual` говорит компилятору: «вызов этого метода через указатель
или ссылку на базу должен разрешаться **во время выполнения**, по реальному типу
объекта».

**Как это устроено внутри.** Компилятор добавляет в каждый полиморфный объект скрытый
указатель `__vptr`, который указывает на **vtable** (таблицу виртуальных функций) —
массив указателей на реализации виртуальных методов для данного класса. При вызове
`ptr->foo()` компилятор генерирует:

```
1. Загрузить __vptr из объекта.
2. По смещению взять указатель на foo().
3. Вызвать по нему.
```

Это **косвенный вызов** — один `load` + один `call` через регистр. Цена ≈ 1–2 нс;
главный ограничитель — не сам вызов, а то, что предсказатель ветвлений хуже работает
с косвенными переходами, и что инлайнинг через vtable в общем случае недоступен.

```cpp
class Animal {
public:
    virtual std::string sound() const { return "..."; }
    virtual ~Animal() = default;
};

class Dog : public Animal {
public:
    std::string sound() const override { return "Woof"; }
};

void print_sound(const Animal& a) {
    std::cout << a.sound() << '\n';   // вызов через vtable → Dog::sound()
}
```

Без `virtual` вызов `a.sound()` компилятор разрешил бы **статически** в `Animal::sound()`
— независимо от того, что объект на самом деле `Dog`. Это называют *статической диспетчеризацией*.

### 9.3 `override` и `final`

- **`override`** — явно сообщает компилятору: «я переопределяю виртуальный метод базы».
  Если сигнатура не совпадает (опечатка в имени, другой тип параметра) — ошибка
  компиляции. Без `override` опечатка добавляет *новый* метод в производный класс, и
  базовый по-прежнему вызывается через vtable — коварный баг.

```cpp
struct Base { virtual void process(int x); };
struct Bad  : Base { void process(long x); };      // молча новый метод, не override
struct Good : Base { void process(int x) override; }; // ошибка компиляции при несовпадении
```

- **`final`** запрещает дальнейшее переопределение метода (или наследование от класса):

```cpp
class Leaf final : public Base { ... };        // нельзя унаследоваться от Leaf
void foo() final { ... }                       // нельзя переопределить foo в потомке
```

`final` также открывает компилятору возможность девиртуализации вызовов — зная, что
наследников нет, он может заменить косвенный вызов на прямой или даже инлайнить его.

### 9.4 Чисто виртуальные методы и абстрактный класс

```cpp
class Shape {
public:
    virtual double area() const = 0;      // чисто виртуальный (pure virtual)
    virtual ~Shape() = default;
};
```

`= 0` означает: «тела нет (в общем случае), реализацию обязан дать каждый конкретный
наследник». Класс с хотя бы одним `= 0` — **абстрактный**: попытка создать его объект
напрямую — ошибка компиляции. Это идеальный инструмент для интерфейсов: нельзя
«забыть» реализовать метод — компилятор не даст собрать код.

> Технически тело чисто виртуального метода можно дать явно за пределами класса
> (`Shape::area()` в .cpp), но на практике это нужно только для `= 0` деструктора.

### 9.5 Срезка объектов (object slicing)

Срезка — одна из самых неприятных ловушек наследования:

```cpp
Dog d;
Animal a = d;       // КОПИРОВАНИЕ: из Dog скопируются ТОЛЬКО поля Animal
                    // a.sound() вернёт "...", а не "Woof" — это не ошибка компиляции!
```

При копировании `Dog` в объект типа `Animal` компилятор копирует лишь ту часть объекта,
которую «видит» базовый класс — `Dog`-специфичные поля и vtable-указатель теряются.
Результирующий объект — чистый `Animal`.

**Как избежать:** работайте с полиморфными объектами **через указатели или ссылки**:

```cpp
Animal& a_ref = d;        // нет среза — a_ref.sound() вызовет Dog::sound()
std::unique_ptr<Animal> p = std::make_unique<Dog>();  // аналогично
```

Именно поэтому полиморфные иерархии обычно хранятся через `std::unique_ptr<Base>`
или `std::shared_ptr<Base>` — ни среза, ни ручного `delete`.

### 9.6 Виртуальный деструктор — обязательное правило

```cpp
// БЕЗ виртуального деструктора:
struct Base { virtual void foo(); ~Base(); };         // НЕ виртуальный
struct Derived : Base { ~Derived() { /* ресурс */ } };

Base* p = new Derived();
delete p;   // ← UB! Вызывается ~Base(), ~Derived() НИКОГДА не вызывается.
            //   Ресурс, которым владеет Derived, никогда не освобождается.
```

**Правило:** если у класса есть хотя бы один виртуальный метод — его деструктор тоже
обязан быть виртуальным. Иначе удаление объекта производного класса через базовый
указатель — **неопределённое поведение**.

```cpp
struct Base { virtual void foo(); virtual ~Base() = default; };   // правильно
```

Практика: в этом задании `Shape` объявляет `virtual ~Shape() = default` — именно
поэтому тест `VirtualDestructorCalled` проходит: `unique_ptr<Shape>` при разрушении
вызовет деструктор правильного подкласса.

## Идея 10. Move-конструктор и move-присваивание: механика

### Что именно происходит при «перемещении»

C++11 ввёл различие между *l-value* (именованный объект, у которого есть адрес) и
*r-value* (временный, безымянный, «уже не нужен»). Для r-value компилятор вправе
вызвать *конструктор перемещения* вместо конструктора копирования.

```cpp
class Buffer {
    int* data_;
    std::size_t n_;
public:
    // Move-конструктор: берём ресурс из исходного объекта.
    Buffer(Buffer&& other) noexcept
        : data_(other.data_), n_(other.n_)
    {
        other.data_ = nullptr;   // ← исходный объект теперь «пуст»
        other.n_    = 0;
    }

    // Move-присваивание.
    Buffer& operator=(Buffer&& other) noexcept {
        if (this != &other) {
            delete[] data_;          // освобождаем то, что было у нас
            data_  = other.data_;    // перехватываем чужой ресурс
            n_     = other.n_;
            other.data_ = nullptr;
            other.n_    = 0;
        }
        return *this;
    }
    ~Buffer() { delete[] data_; }
};
```

Ключевое: исходный объект после перемещения находится в **«валидном, но неуточнённом
состоянии»** (valid but unspecified state). Стандарт гарантирует: с ним можно делать
только то, что не требует определённого состояния — присвоить новое значение или
разрушить. Читать из него (размер, данные) без предварительного переинициализации —
«UB логики», хотя и не UB С++ в строгом смысле.

Для `std::string` и `std::vector` перемещение обнуляет источник: `size()` даёт 0,
`data()` — nullptr. Это определено стандартом для стандартных типов, но для
пользовательских типов — только то, что вы явно прописали.

### Когда компилятор генерирует move-операции

Компилятор генерирует move-конструктор и move-присваивание **автоматически**, если
выполнены все три условия:

1. Вы **не объявили** никакого из: деструктора, конструктора копирования,
   оператора копирующего присваивания.
2. Вы **не объявили** конструктор перемещения (только тогда он генерируется).
3. Вы **не объявили** оператор перемещающего присваивания.

Как только вы написали деструктор (признак ручного управления ресурсом), компилятор
**перестаёт генерировать** move-операции, и вы получаете только копирование — обычно
неправильное для ресурсного класса. Это ещё одна причина для правила пяти: написал
один — пиши все пять.

```cpp
class Timer {             // у него нетривиальный деструктор
    ...
    Timer(Timer&&) = delete;              // явный запрет: Timer некопируем И непереносим
    Timer& operator=(Timer&&) = delete;
};
```

### `noexcept` у move-операций

Move-конструктор должен быть `noexcept` всякий раз, когда это возможно. Алгоритмы
STL (например, `std::vector::push_back` при росте) используют перемещение вместо
копирования **только если move помечен `noexcept`** — иначе они осторожно копируют,
чтобы в случае исключения откатиться. Правило: если move ничем не бросает — помечайте
`noexcept`, иначе теряете оптимизацию.

## Идея 11. `explicit`: защита от молчаливых конверсий

Конструктор с одним параметром по умолчанию является **конверсионным конструктором**:
компилятор разрешает неявно преобразовывать аргумент нужного типа в объект класса.

```cpp
class Duration {
public:
    Duration(int seconds);   // конверсионный конструктор
};

void wait(Duration d);

wait(5);     // компилятор молча создаёт Duration(5) — кажется удобным?
```

Проблема: такая неявная конверсия часто скрывает ошибки:

```cpp
class Timer {
public:
    Timer(std::int64_t& acc);   // аргумент — ССЫЛКА на аккумулятор
};

std::int64_t x = 42;
Timer t = x;      // без explicit: компилятор создаст Timer(x) молча — и это верно
// Но что, если написать:
Timer t2 = 42;    // Ошибка в задумке: 42 — не наш аккумулятор!
                  // Без explicit скомпилируется, сохранит ссылку на временный int — UB!
```

Ключевое слово `explicit` запрещает неявную конверсию:

```cpp
class Timer {
public:
    explicit Timer(std::int64_t& acc);
};

Timer t = 42;      // ошибка компиляции: нет неявной конверсии
Timer t(acc);      // OK: явный вызов конструктора
```

В заголовке модуля `Timer` объявлен с `explicit` именно по этой причине: смысл
конструктора — «принять внешний аккумулятор для записи результата», а не «создать
таймер из целого числа». `explicit` делает неверное использование **невозможным
на этапе компиляции**.

**Правило (Modern C++):** ставьте `explicit` на любой конструктор с одним параметром
(и на конверсионные операторы `operator T()`), если неявная конверсия не является
явным намерением дизайна. Исключение — типы, для которых конверсия семантически
естественна (например, `std::string_view` из `const char*`).

## Идея 12. `operator<=>` (spaceship, C++20)

До C++20, чтобы поддержать полный набор сравнений (`<`, `<=`, `>`, `>=`, `==`, `!=`),
нужно было писать шесть операторов. `operator<=>` генерирует их все:

```cpp
struct Point {
    int x, y;
    // Генерирует все шесть операторов сравнения + operator==
    auto operator<=>(const Point&) const = default;
};
```

Единственная строка заменяет шесть ручных реализаций, и все они **согласованы** по
определению — компилятор не даст написать противоречивые `<` и `==`.

### Типы упорядочения

Возвращаемый тип `<=>` описывает **категорию** порядка:

| Тип | Смысл | Пример |
|---|---|---|
| `std::strong_ordering` | Строгое полное (все значения сравнимы; равные неотличимы) | `int`, `std::string` |
| `std::weak_ordering` | Слабое полное (равные могут различаться внутри) | регистронезависимое сравнение строк |
| `std::partial_ordering` | Частичное (некоторые пары не сравнимы) | `float` (NaN не сравним ни с чем) |

`auto` выведет наиболее строгий тип, совместимый с типами всех членов.

### Связь `<=>` и `==`

`= default` для `<=>` автоматически генерирует и `operator==`. Но если вы пишете
`<=>` вручную, `==` нужно определить отдельно (или тоже `= default`): оптимизированное
сравнение на равенство часто эффективнее, чем проверка `a <=> b == 0`.

```cpp
struct Fraction {
    int num_, den_;
    // Canonical form → покомпонентно достаточно:
    bool operator==(const Fraction&) const = default;
    std::strong_ordering operator<=>(const Fraction& o) const {
        // Сравниваем кросс-умножением (знаменатели положительны):
        auto lhs = static_cast<long long>(num_) * o.den_;
        auto rhs = static_cast<long long>(o.num_) * den_;
        return lhs <=> rhs;
    }
};
```

В этом модуле мы пишем операторы вручную — чтобы понять механику. `<=>` в реальном
коде убирает бойлерплейт и гарантирует согласованность, но понимание основы важнее.

## Идея 13. `friend`-функции, `mutable` и логическая константность

### `friend` и оператор вывода

Когда оператор — свободная функция, но ей нужен доступ к `private`-членам, используют
`friend`:

```cpp
class Fraction {
public:
    // Объявление: operator<< имеет доступ к private-полям.
    friend std::ostream& operator<<(std::ostream& os, const Fraction& f);
private:
    int num_, den_;
};

// Определение (за пределами класса):
std::ostream& operator<<(std::ostream& os, const Fraction& f) {
    return os << f.num_ << '/' << f.den_;  // прямой доступ к private
}
```

Альтернатива — реализовать `operator<<` через публичный `to_string()` (без `friend`):

```cpp
std::ostream& operator<<(std::ostream& os, const Fraction& f) {
    return os << f.to_string();  // нет доступа к private — не нужен
}
```

Именно этот подход используется в нашем модуле: `operator<<` делегирует в `to_string()`,
и `friend` не нужен. `friend` уместен, когда прямой доступ к полям критичен для
производительности или публичный интерфейс не покрывает нужные данные.

**`friend` не нарушает инкапсуляцию** в полном смысле: это явно перечисленные
исключения. Ни один посторонний код не получит доступ к `private`; только те функции
и классы, которые автор явно указал.

### `mutable`: логическая vs физическая константность

`const`-метод обещает: «я не изменю наблюдаемое состояние объекта». Иногда
реализации нужно кешировать результат — например, дорогое вычисление:

```cpp
class Graph {
    std::vector<Edge> edges_;
    mutable bool cache_valid_ = false;     // mutable: меняется даже в const-методе
    mutable int cached_diameter_ = 0;
public:
    int diameter() const {
        if (!cache_valid_) {
            cached_diameter_ = compute_diameter();  // дорого
            cache_valid_ = true;
        }
        return cached_diameter_;
    }
};
```

Член, помеченный `mutable`, может изменяться даже внутри `const`-методов. Это
законно, пока изменение **не видно извне** (не меняет наблюдаемый контракт объекта).
Мутировать `mutable` ради мутирования — ошибка дизайна. Правильное применение:
кеши, счётчики вызовов, мьютексы для потокобезопасного чтения.

Без <code>virtual ~Shape() = default</code> удаление производного объекта через
<code>Shape*</code> вызовет только <code>~Shape()</code>, а <code>~Circle()</code>/
<code>~Rectangle()</code> — никогда. Тест <code>VirtualDestructorCalled</code> создаёт
вспомогательный класс с явным деструктором и считает вызовы — если счётчик не растёт
после <code>delete ptr</code>, значит виртуальный деструктор отсутствует.
</details>

<details><summary>Задание 5 — полиморфный вектор</summary>
<code>std::vector&lt;Shape&gt;</code> не работает — произойдёт срезка при вставке
(см. Идея 9.5). Правильно: <code>std::vector&lt;std::unique_ptr&lt;Shape&gt;&gt;</code>.
Добавление: <code>vec.push_back(std::make_unique&lt;Circle&gt;(r))</code>.
Итерация: <code>for (const auto&amp; s : vec) { total += s-&gt;area(); }</code>.
</details>

<details><summary>Задание 5 — override/final не влияют на семантику, только на безопасность</summary>
<code>override</code> заставляет компилятор проверить, что метод действительно
переопределяет виртуальный метод базы. <code>final</code> запрещает дальнейшее
наследование. Оба ключевых слова не меняют поведения программы — но ловят ошибки
<em>на этапе компиляции</em>. Их отсутствие — запах кода, а не ошибка в рантайме.
</details>
