# Модуль 21 — Реализуем контейнеры STL

Большая идея: контейнеры из `<vector>`, `<list>`, `<unordered_map>` — не магия,
а аккуратно собранные структуры данных с понятными **инвариантами** и обещаниями
по **сложности** и **безопасности исключений**. В этом модуле мы соберём пять
маленьких контейнеров своими руками. Когда ты однажды напишешь `Vector` сам, ты
навсегда перестанешь бояться `std::vector` — и начнёшь читать его поведение как
открытую книгу.

---

## Идея 1. Динамический массив: ёмкость, рост и амортизация

`std::vector` — это непрерывный блок памяти. У него два разных числа:

- **size** — сколько элементов *реально* лежит;
- **capacity** — на сколько элементов *хватает уже выделенной памяти*.

Всегда `size <= capacity`. Пока есть запас (`size < capacity`), `push_back`
просто конструирует элемент в готовой ячейке — это O(1). Когда запас кончился
(`size == capacity`), приходится:

1. выделить **новый**, больший блок;
2. перенести туда старые элементы;
3. освободить старый блок.

Это O(size) — дорого. Вопрос: **на сколько** увеличивать? Наивный ответ «+1 каждый
раз» катастрофичен: добавить N элементов будет стоить 1+2+3+…+N = O(N²) копирований.
Правильный ответ — **геометрический рост**: умножать ёмкость на константу (обычно
×2). Тогда дорогие перевыделения случаются всё реже, и суммарная стоимость N
вставок — O(N), то есть **амортизированно O(1)** на вставку.

Почему именно так считается амортизация? Представь рост 0→1→2→4→8→…→N. Суммарное
число копирований при росте — это `1+2+4+…+N ≈ 2N`. Делим на N вставок — получаем
≈2 копирования на вставку в среднем. Константа. Вот и весь фокус «амортизированно
O(1)»: редкие дорогие операции «оплачиваются» множеством дешёвых.

```cpp
// Наша политика роста (её ты реализуешь в Vector::push_back):
//   cap == 0  ->  new_cap = 1
//   иначе     ->  new_cap = cap * 2
std::size_t grow(std::size_t cap) { return cap == 0 ? 1 : cap * 2; }
```

**Инвариант**, который нельзя нарушать ни на секунду: первые `size` ячеек блока
содержат живые, сконструированные объекты; ячейки `[size, capacity)` — это сырая,
**неинициализированная** память. Поэтому нельзя просто `new T[capacity]` — это
сконструировало бы `capacity` объектов (а нам нужно ровно `size`). В «честной»
реализации память берут как сырые байты (`::operator new`) и строят объекты
**placement new** по мере надобности, а разрушают явным вызовом деструктора.

```cpp
T* raw = static_cast<T*>(::operator new(cap * sizeof(T)));  // сырьё
new (raw + i) T(value);   // placement new: построить ОДИН объект в ячейке i
raw[i].~T();              // явно разрушить
::operator delete(raw);   // вернуть сырьё
```

> В учебных целях для этого модуля можно начать с более простой реализации
> хранилища на основе `T*` и обычного `new[]/delete[]` или даже временного
> `std::vector` внутри — главное, чтобы соблюдались контракты `size/capacity` из
> тестов. Но идею «сырая память + placement new» держи в голове: именно так это
> делает настоящий `std::vector`, и именно она даёт строгую гарантию (см. Идею 2).

**Частая ошибка №1.** Брать ссылку/указатель на элемент, а потом сделать
`push_back`. После перевыделения старый блок умер — ссылка повисла (dangling).

```cpp
Vector<int> v; v.push_back(1);
int& r = v[0];
v.push_back(2);   // могло перевыделить блок — r теперь висит!
// r == ???  -- UB
```

**Частая ошибка №2.** Путать `size` и `capacity`. `capacity()` после `pop_back`
**не** уменьшается — память остаётся зарезервированной. Уменьшать ёмкость надо
осознанно (в STL — `shrink_to_fit`).

## Идея 2. Строгая гарантия исключений при `push_back`

Безопасность исключений — это контракт о том, **в каком состоянии останется
объект, если посреди операции вылетит исключение**. Стандартные уровни:

- **базовая гарантия** — объект остаётся валидным (без утечек, без UB), но
  состояние может измениться;
- **строгая гарантия** — операция либо полностью удалась, либо **ничего не
  изменила** (semantics «commit-or-rollback», как транзакция в БД);
- **noexcept** — операция вообще не бросает.

`std::vector::push_back` даёт **строгую** гарантию: если конструктор/перенос
элемента бросит, вектор останется ровно таким, каким был до вызова. Как это
достигается при росте?

1. Выделяем новый блок (если `::operator new` бросит — старый блок цел, мы ещё
   ничего не трогали → строгая гарантия соблюдена).
2. Переносим старые элементы в новый блок. **Здесь и зарыт нюанс.** Если перенос
   делается **копированием** и копирование бросит на половине пути — не беда: мы
   просто разрушаем уже скопированные, освобождаем новый блок, **старый блок
   остался нетронут**. Откат успешен.
3. Только когда все элементы благополучно в новом блоке — освобождаем старый и
   переключаем указатель. Это «точка коммита».

Опасность — в шаге 2, если переносить **перемещением** (`std::move`). Перемещение
*портит* исходные элементы. Если 3-й move-конструктор бросит, первые два исходных
элемента уже «обворованы» — откатить нельзя, старый блок испорчен → строгая
гарантия потеряна. Поэтому реальный `std::vector` применяет приём
**move-if-noexcept**: переносит перемещением *только если* move-конструктор типа
помечен `noexcept`; иначе безопасно копирует.

```cpp
// Идея move-if-noexcept (упрощённо):
if constexpr (std::is_nothrow_move_constructible_v<T>) {
    new (dst) T(std::move(src));     // быстро и безопасно: move не бросит
} else {
    new (dst) T(src);                // копируем: при исключении старое цело
}
```

Мораль, которую запомни на всю жизнь: **помечай свои move-конструкторы как
`noexcept`**, иначе контейнеры не смогут их использовать и будут молча копировать.

**Частая ошибка.** Сначала освободить/затереть старые данные, а потом
конструировать новые. Любое исключение в середине — и контейнер разрушен. Правило:
сначала построй всё новое в стороне, и лишь в самом конце, **операциями, которые
не бросают** (обмен указателей, перенос целых чисел), зафиксируй результат.

## Идея 3. Small Buffer Optimization (SBO)

Куча — это дорого: `operator new` берёт блокировку, ищет свободный блок, портит
кэш. Для маленьких контейнеров обиднее всего: создал `SmallVector` на 3 элемента —
а заплатил за поход в аллокатор. **SBO** прячет первые `N` элементов **прямо внутри
объекта** (во встроенном массиве, который живёт на стеке вместе с самим
контейнером). Куча подключается лишь когда элементов стало больше `N`.

```cpp
template <class T, std::size_t N>
class SmallVector {
    T   inline_[N];       // встроенный буфер: бесплатно, на стеке
    T*  heap_ = nullptr;  // != nullptr <=> переехали в кучу
    std::size_t size_ = 0, cap_ = N;
    // данные сейчас тут:  heap_ ? heap_ : inline_
};
```

Так устроены `llvm::SmallVector`, `boost::small_vector`, реализация `std::string`
(SSO — small string optimization: короткие строки лежат прямо в объекте, без
аллокации). Выигрыш огромен в горячих циклах, где создаётся миллион крошечных
векторов.

Цена: объект **больше** по размеру (внутри живёт массив на `N` элементов, даже
когда он пуст), и логика чуть сложнее — каждая операция должна знать, где сейчас
данные. Ключевой момент **переезда**: когда `size_ == cap_` и буфер встроенный,
надо выделить кучу, **перенести** туда `N` элементов из `inline_`, и дальше
работать с `heap_`.

**Частая ошибка.** Перепутать «где данные». Универсальный приём — единственная
функция `data()`, возвращающая `heap_ ? heap_ : inline_`, и **все** обращения
идут через неё. Тогда `operator[]`, `push_back`, деструктор не дублируют ветвление
и не разъезжаются.

## Идея 4. Интрузивные структуры

Обычный (неинтрузивный) список — `std::list<T>` — на каждый элемент **выделяет
отдельный узел** `{ T value; node* next; node* prev; }`. Это аллокация на каждую
вставку и лишний indirection.

**Интрузивный** список переворачивает идею: указатели `next/prev` встроены в **сам
объект пользователя** (или в базовый «hook»). Список не владеет узлами и **не
выделяет память** — он лишь связывает уже существующие объекты в цепочку.

```cpp
template <class T>
struct Node {
    T value;
    Node* next = nullptr;
    Node* prev = nullptr;
};
// Список хранит только head_/tail_ и size_; узлы создаёт пользователь.
```

Почему это мощно:

- **ноль аллокаций** на вставку/удаление — узел уже существует;
- `erase(node)` — **O(1)** по указателю на сам узел: перецепить соседей и всё. В
  `std::list` это тоже O(1), но там у тебя обычно итератор; тут — прямо объект;
- один объект может одновременно состоять в **нескольких** интрузивных списках
  (несколько пар hook-ов) — так делают планировщики ОС и игровые движки, держа
  сущность сразу в «списке активных» и «списке на отрисовку».

Цена и опасности:

- **владение временем жизни** теперь на пользователе. Удалил объект, забыв вынуть
  его из списка, — соседи указывают в труп (dangling), обход списка падает;
- объект «знает» о том, что он в списке (его поля заняты служебными указателями) —
  ослабленная инкапсуляция.

Механика `erase` — это аккуратная перецепка четырёх ссылок с учётом краёв (узел —
голова? хвост? единственный?):

```cpp
void erase(Node* n) {
    if (n->prev) n->prev->next = n->next; else head_ = n->next;  // n — голова?
    if (n->next) n->next->prev = n->prev; else tail_ = n->prev;  // n — хвост?
    n->next = n->prev = nullptr;   // гигиена: узел больше ни на кого не указывает
    --size_;
}
```

**Частая ошибка.** Забыть случай «узел единственный» (он и голова, и хвост) или не
обнулить `next/prev` у вынутого узла — тогда повторный `erase` или `push_back` того
же узла приведёт к каше из указателей.

## Идея 5. LRU-кэш за O(1): хэш-таблица плюс список

Кэш ограниченного размера должен при переполнении кого-то выкинуть. Политика
**LRU (Least Recently Used)** выкидывает того, к кому дольше всех не обращались.
Хочется, чтобы и поиск, и обновление «свежести», и вытеснение были **O(1)**.

Ни хэш-таблица одна, ни список один этого не дают:

- список знает порядок (свежие — в начале, кандидат на вылет — в конце), но поиск
  в нём O(n);
- хэш-таблица ищет за O(1), но не помнит порядок использования.

Комбинируем оба, и каждый закрывает слабость другого:

- **`std::list<pair<K,V>>`** хранит элементы в порядке свежести: `front` — самый
  свежий, `back` — кандидат на вытеснение;
- **`unordered_map<K, list::iterator>`** даёт за O(1) итератор на нужный узел
  списка по ключу.

```cpp
std::list<std::pair<K,V>>                       order_;   // front = свежий
std::unordered_map<K, decltype(order_)::iterator> index_;
```

Операции:

- **get(k)**: ищем в `index_` (O(1)). Нет — промах. Есть — `splice` переносим узел
  в `front` (делаем свежим, O(1), итераторы `std::list` при этом **не
  инвалидируются** — это критично!), возвращаем значение.
- **put(k,v)**: если ключ есть — обновляем значение и двигаем во `front`. Если нет
  — вставляем во `front`, кладём итератор в `index_`. Если стало больше ёмкости —
  берём `order_.back()`, удаляем его ключ из `index_` и `pop_back()` из списка.

Почему именно `std::list`, а не `std::vector`? Потому что нам нужно **перемещать
узел из середины в начало за O(1) без инвалидации итераторов**, которые лежат в
`index_`. `std::list::splice` это умеет; у `vector` любая вставка/удаление в
середине — O(n) и инвалидирует всё.

**Частая ошибка.** Обновить список, но забыть синхронно обновить `index_` (или
наоборот). Эти две структуры должны меняться **вместе** — это их общий инвариант:
для каждого ключа в `index_` итератор указывает на узел `order_` с тем же ключом, и
наоборот. Рассинхрон — и кэш начинает «терять» или «воскрешать» элементы.

## Идея 6. Кольцевой буфер и сложности операций

`RingBuffer<T, N>` — массив фиксированного размера `N`, по которому два индекса
бегают **по кругу**: `head` (откуда читать) и `tail` (куда писать). Когда индекс
доходит до конца массива, он «заворачивается» к нулю — арифметика по модулю `N`.

```cpp
tail_ = (tail_ + 1) % N;   // классический способ "завернуть" индекс
```

Зачем такая структура: это идеальная **FIFO-очередь фиксированной ёмкости без
аллокаций** — буфер выделен один раз, дальше только переиспользуем ячейки. Поэтому
кольцевые буферы повсюду: аудио/видео-потоки, очереди событий, lock-free очереди
между потоками-производителем и потребителем, логи ядра ОС.

Тонкий момент — как отличить **полный** буфер от **пустого**, если `head == tail`
в обоих случаях? Два честных решения: (а) держать отдельный счётчик `size_` (просто
и понятно — используем его), либо (б) «жертвовать» одной ячейкой, считая буфер
полным при `(tail+1)%N == head`. Мы идём путём (а).

Сложности **всех** операций — **O(1)**, причём с *гарантией*, а не амортизированно
(никаких перевыделений вообще): `push`, `pop` — это запись/чтение одной ячейки и
сдвиг индекса.

Поведение на краях — часть контракта (его пинают тесты):

- буфер полон → `try_push` возвращает `false` (мягкий отказ), а `push` бросает
  `std::out_of_range` (жёсткий);
- буфер пуст → `try_pop` возвращает `std::nullopt`.

**Частая ошибка.** Забыть `% N` и дать индексу убежать за границу массива — UB,
порча соседней памяти. Вторая — увеличить/уменьшить `size_` не синхронно со сдвигом
индекса, из-за чего `full()/empty()` начинают врать.

1. **`Vector<T>`** — динамический массив. Реализуй `push_back` (рост ёмкости:
   0→1, иначе ×2), `pop_back`, `operator[]`, `at` (бросает `std::out_of_range`
   при выходе за границы), `size`, `capacity`, `empty`, `begin`/`end` (по ним
   должен работать range-based `for`). Не забудь освобождать память в деструкторе.
2. **`SmallVector<T, N>`** — вектор с SBO. Пока элементов `<= N`, держи их во
   встроенном буфере (`on_heap() == false`, `capacity() == N`); при переполнении
   переезжай в кучу (`on_heap() == true`). Реализуй `push_back`, `pop_back`,
   `operator[]`, `size`, `capacity`, `on_heap`.
3. **`IntrusiveList<T>`** — интрузивный двусвязный список из `Node<T>`. Реализуй
   `push_back(Node*)`, `erase(Node*)` (O(1), все случаи: голова/хвост/середина/
   единственный), `size`, `empty`, `front`, `back`. Память под узлы НЕ выделяй —
   их создаёт пользователь.
4. **`LRUCache<K, V>`** — кэш на `std::list` + `unordered_map`. Конструктор с
   `capacity` (при `0` — `std::out_of_range`). Реализуй `put`, `get`
   (`std::optional<V>`), `contains`, `size`. `get` и `put` делают ключ свежим;
   при переполнении вытесняется самый давно использованный.
5. **`RingBuffer<T, N>`** — кольцевой буфер на `N` ячеек. Реализуй `try_push`
   (полон → `false`), `push` (полон → `std::out_of_range`), `try_pop`
   (`std::optional<T>`, пуст → `nullopt`), `size`, `full`, `empty`. Соблюдай FIFO
   и «заворачивание» индексов по модулю `N`.

<details><summary>Задание 3 — erase: разбор случаев</summary>
Подумай о четырёх ссылках вокруг узла `n`: `n->prev->next` и `n->next->prev`. Если
`n->prev == nullptr`, то `n` — голова (обновляй `head_`). Если `n->next ==
nullptr` — хвост (обновляй `tail_`). После перецепки обнули `n->next` и `n->prev`.
Не забудь уменьшить `size_`.
</details>
<details><summary>Задание 4 — связка list + map</summary>
`order_.push_front({key, value})` кладёт свежий элемент в начало; затем
`index_[key] = order_.begin()`. Для «сделать свежим» используй
`order_.splice(order_.begin(), order_, it)` — он переносит узел `it` в начало без
инвалидации итераторов. Вытеснение: `auto last = order_.back(); index_.erase(last.first);
order_.pop_back();`. И всегда меняй обе структуры вместе.
</details>
<details><summary>Задание 5 — заворачивание и счётчик</summary>
Держи `size_` отдельно — так `full()` это `size_ == N`, а `empty()` это `size_ ==
0`. `try_push`: если полон → `false`; иначе `buf_[tail_] = value; tail_ =
(tail_+1)%N; ++size_; return true;`. `try_pop`: если пуст → `std::nullopt`; иначе
возьми `buf_[head_]`, сдвинь `head_ = (head_+1)%N`, уменьши `size_`, верни
значение. `push` — это `try_push`, а на `false` кидай `std::out_of_range`.
</details>
