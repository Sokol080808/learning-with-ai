# Модуль 01 — Типы, переменные, `const` и ссылки

Большая идея модуля: в C++ **тип — это контракт и обещание о памяти**. От типа зависит,
сколько байт занимает значение, какие операции допустимы и как они себя ведут. Здесь
много языков «прощают» небрежность, а C++ — нет, и это его сила: ошибки видны рано.

## Идея 1. Фундаментальные типы и их размер

- Целые: `int`, `long`, `short`, `unsigned …`. Размер зависит от платформы! Когда
  важна точная ширина — бери `<cstdint>`: `int32_t`, `int64_t`, `uint8_t` и т.д.
- Дробные: `double` (по умолчанию выбирай его), `float` (меньше точность).
- `bool` (`true`/`false`), `char` (один байт, символ).

```cpp
#include <cstdint>
int32_t a = 5;     // ровно 32 бита
double  x = 2.5;   // дробное
bool    ok = true;
```

## Идея 2. Целочисленное деление — классическая ловушка

```cpp
int a = 7, b = 2;
a / b;            // == 3, а не 3.5 ! деление двух int — целочисленное
double r = a / b; // == 3.0 — деление уже произошло в int, поздно
double r2 = static_cast<double>(a) / b; // == 3.5 — приводим ДО деления
```

Запомни правило: тип результата определяется типами **операндов**, а не тем, куда ты
присваиваешь. Хочешь дробное деление — сделай хотя бы один операнд дробным.

## Идея 3. `auto` — пусть компилятор выведет тип

```cpp
auto n = 42;          // int
auto x = 3.14;        // double
auto s = std::string{"hi"};
```

`auto` не делает язык «динамическим» — тип всё равно фиксируется на этапе компиляции,
просто его выводит компилятор. Это удобно и уменьшает дублирование.

## Идея 4. `const` — «я обещаю это не менять»

```cpp
const int limit = 100;   // limit менять нельзя
limit = 200;             // ошибка компиляции — и это хорошо
```

`const` — не про оптимизацию, а про **намерение** и защиту от случайных ошибок. Чем
больше `const`, тем меньше способов сломать код по неосторожности. Это важная привычка.

## Идея 5. Ссылки (`&`) и передача в функции — value vs reference

Когда ты передаёшь аргумент **по значению**, функция получает *копию*:

```cpp
void f(int x) { x = 99; }   // меняет КОПИЮ, снаружи ничего не изменится
```

**Ссылка** (`int&`) — это псевдоним для существующей переменной. Через неё функция
меняет оригинал:

```cpp
void g(int& x) { x = 99; }  // меняет ОРИГИНАЛ
```

А `const&` — «дай доступ без копии, но менять не буду» (для больших объектов это и
быстро, и безопасно — к этому вернёмся не раз):

```cpp
void h(const std::string& s);  // не копируем строку и не портим её
```

## Идея 6. Ввод/вывод

```cpp
#include <iostream>
int x;
std::cin >> x;                       // прочитать число
std::cout << "x = " << x << "\n";    // вывести
```

В заданиях ввод/вывод не нужен (тесты вызывают функции напрямую), но это базовый
навык — поэкспериментируй отдельно.

## Идея 7. Целочисленные типы, переполнение и UB

У знакового `int` есть конечный диапазон. На типичной платформе это
`[-2147483648, 2147483647]` (см. `<limits>`: `std::numeric_limits<int>::max()`).
Что будет, если выйти за край?

```cpp
#include <limits>
int big = std::numeric_limits<int>::max();
int oops = big + 1;   // UB! переполнение ЗНАКОВОГО int — неопределённое поведение
```

Это не «просто завернётся в минус» — это **undefined behaviour**: компилятор вправе
сделать что угодно, и часто оптимизатор предполагает, что переполнения не бывает.
Поэтому проверять переполнение нужно **до** операции, по самим операндам:

```cpp
// можно ли безопасно сложить a и b?
if (b > 0 && a > std::numeric_limits<int>::max() - b) { /* переполнение вверх */ }
if (b < 0 && a < std::numeric_limits<int>::min() - b) { /* переполнение вниз  */ }
```

Тонкость: `INT_MAX - b` само должно быть безопасным. Когда `b > 0`, `max - b` всегда в
диапазоне; когда `b < 0`, `min - b` всегда в диапазоне. Поэтому условия разбиты по знаку
`b` — так ни одно промежуточное вычитание не переполняется.

Пара слов про **беззнаковые** типы: у `unsigned` переполнение *определено* — оно
заворачивается по модулю `2^N`. Это не UB, но это легко превращается в сюрприз
(`0u - 1u` == огромное число), так что осторожнее со смешиванием знаковых и беззнаковых.

## Идея 8. Неявные преобразования и сужение (narrowing)

C++ молча конвертирует один арифметический тип в другой. Иногда это удобно, иногда —
тихая потеря данных:

```cpp
double pi = 3.99;
int n = pi;        // n == 3 — дробная часть отброшена, БЕЗ ошибки
long big = 1'000'000'000'000;
int small = big;   // переполнение/усечение — значение испорчено молча
```

Такие «сужающие» преобразования компилятор пропускает в обычном присваивании. Но в
**фигурных скобках** (списочная инициализация) сужение запрещено — и это хорошо:

```cpp
int a = 3.99;      // ок (молча 3)
int b{3.99};       // ОШИБКА компиляции: narrowing — отличный предохранитель
```

Привычка: для инициализации предпочитай `{}`. Компилятор поймает то, что глаз пропустит.

## Идея 9. Ссылки vs указатели

И ссылка, и указатель дают доступ к «чужой» переменной, но ведут себя по-разному:

```cpp
int v = 10;
int& r = v;        // ссылка: ОБЯЗАНА быть привязана сразу, и навсегда к v
r = 20;            // меняем v напрямую, без разыменования

int* p = &v;       // указатель: хранит адрес, может быть null, можно перенаправить
*p = 30;           // нужно разыменование (*) чтобы добраться до значения
p = nullptr;       // указатель можно «отвязать»; ссылку — нельзя
```

Правила-ориентиры:
- Ссылку **нельзя** оставить непривязанной и нельзя перепривязать к другой переменной.
- У ссылки нет «нулевого» состояния — она всегда указывает на реальный объект.
- Указатель гибче (null, переадресация, арифметика), но за гибкость платишь
  внимательностью: разыменовать null — UB.

Правило выбора: если объект **всегда есть** и переадресация не нужна — бери ссылку,
код будет проще и безопаснее. Указатель — когда нужно «может быть, а может и нет»
(null) или нужно менять, на что смотришь.

## Идея 10. `const`-корректность

`const` мы уже видели на переменных (Идея 4). Но настоящая его сила — в *интерфейсах*:

```cpp
void print(const std::string& s);   // «прочитаю, но не испорчу» — и без копии
int  read(const int* p);            // указатель на const: через p менять нельзя
```

Различай два «const» у указателей — читай справа налево:

```cpp
const int* p;        // указатель на const int: *p менять нельзя, p — можно
int* const q = &v;   // const-указатель на int: p менять нельзя, *q — можно
const int* const z = &v; // нельзя ни то, ни другое
```

Зачем это всё: `const` в сигнатуре — это **обещание вызывающему**. Он видит, что его
данные в безопасности, а компилятор это обещание проверяет. Чем больше `const` в коде,
тем меньше способов случайно что-то сломать. Это та привычка, которая отличает
аккуратный C++ от «вроде работает».

## Идея 11. Категории значений: lvalue и rvalue

Каждое выражение в C++ имеет не только **тип**, но и **категорию значения** — это
фундамент всей move-семантики.

- **lvalue** («left-value») — выражение, у которого есть **имя и адрес**. Его можно
  поставить слева от `=`. Примеры: переменная `x`, элемент массива `arr[i]`,
  разыменованный указатель `*p`.
- **rvalue** («right-value») — **временный объект** без имени и постоянного адреса.
  Примеры: `42`, `x + 1`, `std::string("hi")`.

Почему это важно? Ссылка на неконстантный тип (`T&`) привязывается **только к lvalue**:

```cpp
int x = 5;
int& r = x;        // ок — x это lvalue
int& r2 = x + 1;  // ОШИБКА: x+1 — rvalue, у него нет адреса
```

Зато `const T&` — особый случай: компилятор **продлевает время жизни** временного объекта
до конца области видимости ссылки:

```cpp
const int& cr = 42;        // ок: временный int живёт, пока жива cr
const std::string& s = std::string("hello"); // временная строка жива здесь
```

Это не просто трюк — именно это правило позволяет передавать временные объекты в
функции по `const&` без копии:

```cpp
void print(const std::string& s);  // принимает и lvalue, и rvalue
print("hello");  // "hello" создаёт временный std::string — привязывается к const&
```

**Подвох.** Продление времени жизни работает только через **прямую** `const&`-переменную;
через поле структуры или возврат по ссылке из функции — не работает.

Это фундамент move-семантики: rvalue-ссылка `T&&` привязывается **только к rvalue** и
сигнализирует: «владелец не нужен, можно забрать ресурсы». Это тема следующих модулей.

## Идея 12. `std::optional` — значение или «ничего»

`std::optional<T>` (`<optional>`, C++17) — контейнер, который либо хранит значение типа
`T`, либо находится в состоянии «пусто» (`std::nullopt`). Это **часть типа**, а не
магическое число-сентинел.

```cpp
std::optional<int> safe_div(int a, int b) {
    if (b == 0) return std::nullopt;  // «нет значения»
    return a / b;                     // std::optional<int>{a / b}
}

auto r = safe_div(10, 2);
if (r.has_value()) std::cout << *r;  // разыменование безопасно после проверки
// или: r.value_or(-1)  — значение или дефолт
```

**Когда optional лучше исключения?**
- Ошибка *ожидаема* и нормальна для вызывающего кода (например, «ключ не найден»,
  «парсинг не удался», «переполнение»). Исключение — для *неожиданных* сбоев.
- Вызывающий *должен* проверить результат: `optional` делает это явным в типе.
- Производительность: исключения дёшевы при отсутствии ошибки, но бросок исключения
  несравнимо дороже возврата `nullopt`.

**Когда sentinel (−1, 0, NULL) хуже optional?**
Sentinel требует *договорённости вне типа* — кто-нибудь забудет проверить. С `optional`
компилятор напомнит разыменовать только после `has_value()`.

**Подвох.** Разыменование пустого `optional` (`*opt` или `opt.value()` без проверки) —
UB (`value()` бросает `std::bad_optional_access`, `*opt` — UB). Всегда проверяй
`has_value()` или используй `value_or()`.

## Идея 13. `constexpr` vs `const` — compile-time vs runtime

```cpp
const int n = 10;      // может быть вычислено в runtime (значение берётся извне)
constexpr int m = 10;  // ОБЯЗАНО быть известно на этапе компиляции
```

`constexpr` — это более сильное утверждение: «посчитай это прямо сейчас, компилятор».
Компилятор **обязан** это выполнить, или выдаст ошибку.

**Где `constexpr` обязателен, а `const` недостаточен:**

```cpp
// 1. Размер массива — константное выражение compile-time:
const int n = 5;
int arr[n];         // формально UB в стандарте (VLA), но многие компиляторы принимают
constexpr int m = 5;
int arr2[m];        // корректно и гарантированно

// 2. Аргументы шаблона:
std::array<int, n>  a;   // ошибка, если n не constexpr
std::array<int, m>  b;   // ок

// 3. case-метки в switch:
switch(x) {
    case m:   // ок — constexpr
    case n:   // может не компилироваться, если n неконстантное выражение
}
```

`constexpr` функция вычисляется во время компиляции, если все аргументы — константные
выражения; иначе — в runtime как обычная функция.

```cpp
constexpr int square(int x) { return x * x; }
constexpr int s = square(5);  // вычислено при компиляции -> 25 в .rodata
int y = 7;
int t = square(y);            // вычислено в runtime — тоже корректно
```

## Идея 14. Подвохи `auto` при выводе типа

`auto` выводит тип по образцу правил вывода шаблонных аргументов. Главная ловушка:

**`auto` отбрасывает `const` и `&` при копировании:**

```cpp
const int x = 5;
auto a = x;         // a: int (не const int!) — копия, const теряется
auto& b = x;        // b: const int& — сохраняет const (ссылка на x)
const auto& c = x;  // c: const int& — явно const, продлевает время жизни rvalue
```

Таблица:

| Объявление | Что получим | Копирует? |
|---|---|---|
| `auto v = expr` | тип expr без `const`/`&` | да |
| `auto& r = expr` | ссылка, сохраняет const | нет |
| `const auto& cr = expr` | const-ссылка, принимает rvalue | нет |
| `auto&& fr = expr` | forwarding ref: lvalue→lvalue, rvalue→rvalue | нет |

**Практическое правило:** в диапазонных for-циклах почти всегда нужна `const auto&`,
если не нужна мутация:

```cpp
std::vector<std::string> v = { "a", "b", "c" };
for (auto  s : v)         // КОПИРУЕТ каждую строку — обычно не то, что хотелось
for (auto& s : v)         // ссылка — менять можно
for (const auto& s : v)   // const-ссылка — читать без копии, правильный выбор
```

## Идея 15. `std::string_view` — const-correct просмотр строки

`std::string_view` (`<string_view>`, C++17) — это **пара (указатель, длина)** на
существующий символьный буфер. Не владеет памятью, не выделяет её.

```cpp
void greet(std::string_view name) {           // принимает что угодно
    std::cout << "Hello, " << name << "\n";
}

greet("world");                               // строковый литерал — без аллокации
greet(std::string{"Alice"});                  // std::string — без копии
greet(std::string_view(buf, 5));              // произвольный буфер
```

**Чем отличается от `const std::string&`?**

| | `const std::string&` | `std::string_view` |
|---|---|---|
| Принимает литерал (`"abc"`) | да, но создаёт временный `std::string` | да, без аллокации |
| Принимает `std::string` | да, без копии | да, без копии |
| Принимает `char*` / `char[]` | да, но с аллокацией | да, без аллокации |
| Принимает подстроку | нет, нужна копия | да — `sv.substr(1, 3)` дешевле |
| Null-terminated гарантия | да | нет — нет `\0` на конце |

**Подвох: время жизни.** `string_view` — *невладеющий* observer. Если буфер, на который
он указывает, уничтожен — view становится оборванным:

```cpp
std::string_view dangling() {
    std::string s = "hello";
    return s;    // ОШИБКА: s уничтожена при выходе, view указывает в мусор
}
```

**Правило:** `string_view` — отличный тип параметра функции, которая *читает* строку.
Не используй его как тип поля класса или возвращаемое значение из функции, создающей
строку (если только это не вид на уже существующую строку, гарантированно живущую дольше).

## Идея 16. Подвох сдвигов: UB со знаковыми и сдвиг на >= ширину

Побитовые сдвиги требуют осторожности в C++20 (и тем более до него):

```cpp
// 1. Сдвиг знакового типа влево — был UB до C++20:
int x = 1;
x << 31;   // до C++20: UB если результат не помещается в знаковый тип
           // C++20: определено для двух-дополнительного кода, но осторожнее

// 2. Сдвиг беззнакового — всегда безопасен по значению:
1u << 31;  // ok: 0x80000000u

// 3. Сдвиг на >= ширину типа — всегда UB:
uint32_t v = 1u;
v << 32;   // UB! 32 >= 32 (ширина uint32_t)
v << 33;   // UB тоже

// 4. Сдвиг на отрицательную величину — UB:
v << -1;   // UB
```

**Практическое правило:** работаешь с битами — используй **`unsigned`** типы (`1u <<
pos`, `uint32_t`, `uint64_t`). Это убирает класс UB со знаковыми. Проверяй, что `pos`
в диапазоне `[0, 31]` для 32-битного типа.

В заданиях этого модуля битовые функции принимают `unsigned int` и `pos` из `[0, 31]`,
что по умолчанию безопасно — но в реальном коде стоит это явно проверять.

<details><summary>Задание 4 — почему void и ссылка</summary>
Функция ничего не возвращает (`void`), но обязана изменить переданную переменную.
Это возможно только если параметр — ссылка (`int&`), а не копия (`int`).
</details>

<details><summary>Задание 6 — с чего начать</summary>
Тебе понадобятся `std::numeric_limits<int>::max()` и `::min()` из `<limits>`.
Ключевая мысль: сложить, а потом проверить — поздно (переполнение знакового — это уже
UB). Значит, проверяем условие на самих `a` и `b`.
</details>

<details><summary>Задание 6 — как сформулировать условие</summary>
Разбери два случая по знаку `b`. Если `b > 0`, опасно переполнение вверх: оно случится,
когда `a > max - b`. Если `b < 0`, опасно переполнение вниз: когда `a < min - b`.
Почему именно `max - b` и `min - b`, а не `max - a`? Так промежуточное вычитание само
никогда не выходит за диапазон. Если ни одно условие не сработало — складывать безопасно,
заверни сумму в `std::optional`.
</details>

<details><summary>Задание 7 — маска одного бита</summary>
Базовый кирпич — «единичка, сдвинутая на нужную позицию»: `1u << pos`. Дальше:
установить — через `|` с маской; сбросить — через `&` с инверсией маски (`~`);
переключить — через `^`; прочитать — сдвинуть value вправо и посмотреть младший бит
(или `&` с маской и сравнить с нулём). Используй именно `1u` (беззнаковый литерал),
чтобы сдвиг на 31 был корректным.
</details>

<details><summary>Задание 8 — swap без временной?</summary>
Самый понятный и правильный способ — через одну временную переменную: запомнить `a`,
положить в `a` значение `b`, затем в `b` положить запомненное. Не гонись за «хитрым»
XOR-трюком: он ломается, когда `a` и `b` — одна и та же переменная (тест это проверяет).
</details>

<details><summary>Задание 9 — zero-copy string_view</summary>
`std::string_view` умеет находить символ: `s.find(' ')` возвращает позицию первого
пробела или `std::string_view::npos` если его нет. Потом `s.substr(0, pos)` возвращает
новый `string_view` на ту же память — без копирования. Подумай, какие три случая нужно
разобрать: строка пустая, пробел найден, пробел не найден.
</details>
