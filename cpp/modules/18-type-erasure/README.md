# Модуль 18 — Стирание типов (type erasure)

Большая идея: **хранить значения разных типов за одним интерфейсом — не требуя от них
общего базового класса**. `std::function`, `std::any`, `std::shared_ptr<void>` устроены
именно так. Это «полиморфизм по требованию»: типы не обязаны заранее наследоваться от
чего-то общего — обёртка сама прячет конкретный тип внутри.

## Идея 1. Зачем это нужно

Классическое решение «разные типы за общим интерфейсом» — виртуальные функции и общий
базовый класс:

```cpp
struct Shape { virtual double area() const = 0; };
struct Circle : Shape { double area() const override { /* ... */ } };
```

Но это работает только если ты **можешь менять** типы и заставить их наследоваться. А что,
если нужно сложить в один контейнер `int`, `std::string` и чью-то чужую структуру, которые
друг о друге не знают? Они не делят базовый класс. Стирание типов даёт общий интерфейс
*снаружи*, не трогая сами типы.

Подвох: за это платят. Внутри почти всегда прячется `unique_ptr` и виртуальный вызов —
то есть аллокация и косвенность. Это не бесплатно, но часто оно того стоит.

## Идея 2. Механика: концепт + модель + указатель на базу

Рецепт type erasure состоит из трёх частей:

```cpp
class Drawable {
    struct Concept {                          // 1) «концепт» — чистый интерфейс
        virtual ~Concept() = default;
        virtual std::string draw() const = 0;
    };
    template <class T>
    struct Model : Concept {                  // 2) «модель» — знает конкретный T
        T obj_;
        explicit Model(T o) : obj_(std::move(o)) {}
        std::string draw() const override { return obj_.draw(); }
    };
    std::unique_ptr<Concept> self_;           // 3) указатель на базу — тип стёрт
public:
    template <class T>
    Drawable(T obj) : self_(std::make_unique<Model<T>>(std::move(obj))) {}
    std::string draw() const { return self_->draw(); }
};
```

`Concept` — абстрактный класс, он ничего не знает про `T`. `Model<T>` — шаблон, у него по
одному инстансу на каждый завёрнутый тип, и именно он помнит настоящий `T`. Снаружи виден
только `unique_ptr<Concept>` — тип «стёрт». Когда зовёшь `draw()`, виртуальный вызов
прыгает в нужный `Model<T>::draw`.

Подвох: `Concept` обязан иметь `virtual ~Concept()`. Без виртуального деструктора удаление
через `unique_ptr<Concept>` — это UB (не вызовется деструктор `Model<T>`).

## Идея 3. Так устроены `std::function` и `std::any`

`std::function<R(Args...)>` хранит любой вызываемый объект с такой сигнатурой — это
ровно «концепт + модель», где у концепта метод `invoke(Args...) -> R`. `std::any` хранит
значение любого типа; `any_cast<T>` проверяет, совпадает ли спрятанный тип с `T`, и при
несовпадении бросает `std::bad_cast`. Проверка обычно идёт через `typeid`:

```cpp
if (a.type() == typeid(T)) { /* можно безопасно доставать */ }
else throw std::bad_cast{};
```

Подвох: `std::any` (как и наш `Any`) **копирует** значение внутрь. Значит, обёртке нужно
уметь копировать спрятанный объект, не зная его типа — для этого в концепт добавляют
виртуальный `clone()`, который в `Model<T>` делает `make_unique<Model<T>>(value_)`.

## Идея 4. Альтернатива: `std::variant` + visitor

Если набор типов **известен заранее и закрыт**, лучше подходит `std::variant<A, B, C>` —
он не аллоцирует и хранит значение по месту. Разбирают его через `std::visit`. Удобный
приём — собрать набор лямбд в один объект-визитор с помощью `overloaded`:

```cpp
template <class... Ts> struct overloaded : Ts... { using Ts::operator()...; };
template <class... Ts> overloaded(Ts...) -> overloaded<Ts...>;

std::variant<Circle, Square> shape = Circle{2.0};
double a = std::visit(overloaded{
    [](const Circle& c) { return 3.14159 * c.r * c.r; },
    [](const Square& s) { return s.side * s.side; },
}, shape);
```

`overloaded` наследуется сразу от всех лямбд и втягивает их `operator()` через
`using ...`. `std::visit` подбирает нужную перегрузку по тому, что лежит в `variant`.

Подвох: если забыть перегрузку для одной из альтернатив, код просто не скомпилируется —
и это хорошо: компилятор гарантирует, что ты обработал все случаи.

## Идея 5. Type erasure против виртуальных функций

| | Виртуальные функции | Type erasure | `std::variant` |
|---|---|---|---|
| Общий базовый класс | обязателен | не нужен | не нужен |
| Набор типов | открытый | открытый | **закрытый** |
| Аллокация | — (если по значению) | обычно да | нет |
| Хранит чужие типы | нет | да | да (если в списке) |

Грубое правило: типы твои и их немного — `variant`. Типы чужие или набор открыт —
type erasure. Нужна максимальная скорость и закрытый набор — `variant`.

## Идея 6. Small Buffer Optimization (SBO)

Реализация на `unique_ptr<Concept>` красива, но у неё одна цена — **каждое
создание `Function` или `Any` требует аллокации в куче**. Для крошечной лямбды
`[](int x){ return x+1; }` это избыточно: она занимает несколько байт, а
`malloc` заберёт ~50 нс накладных расходов.

Настоящие `std::function` и `std::any` обходят это через **Small Buffer
Optimization**: рядом с указателем держится выровненный буфер прямо внутри
объекта, и если хранимый callable помещается в него — `placement new` кладёт
его туда без обращения к аллокатору.

```cpp
// Упрощённая схема внутри std::function (иллюстрация, не стандарт):
struct function_storage {
    // Статический буфер — обычно 16–32 байта (зависит от реализации).
    alignas(alignof(std::max_align_t))
    char buf_[24];   // <-- сюда кладут малый callable placement new-ом

    // Если callable не влез — здесь лежит указатель на кучу.
    void* heap_ptr_ = nullptr;
};
```

Два пути исполнения внутри `operator()`:
1. **Малый объект** — `buf_` содержит объект напрямую; вызов через указатель на
   функцию-диспетчер (он хранится отдельно и заменяет vtable), без аллокации.
2. **Большой объект** — `heap_ptr_` указывает на `new`-созданный `Model<F>`;
   вызов через виртуальный метод как в наивной реализации.

Почему это важно на практике:

- Большинство «горячих» лямбд в реальном коде мелкие (пустые или захватывают
  1–2 указателя). SBO делает их передачу в `std::function` почти такой же
  дешёвой, как передачу значения.
- Порог размера **не стандартизирован**: в libc++ и MSVC это обычно 24–32 байта,
  в libstdc++ — 16 байт. Если захватываешь большой массив или несколько полей
  структуры, объект «выпадет» из буфера — аллокация всё равно произойдёт.
- SBO объясняет ещё одно ограничение: объект в буфере обязан быть
  **nothrow-move-constructible**, иначе реализация не сможет безопасно
  перемещать его вместе с `function`.

Подвох: `std::any` от C++17 тоже использует SBO, но типичный порог там **ещё
меньше** — 1–2 слова. Любой нетривиальный объект немедленно уходит в кучу.

Чем наивная `unique_ptr`-реализация отличается от настоящей `std::function`:

| Аспект | Наша `Function` | `std::function` |
|---|---|---|
| Малый callable | аллокация в куче | буфер на стеке (SBO) |
| Большой callable | аллокация в куче | аллокация в куче |
| Вызов (малый) | виртуальный вызов через `Concept*` | прыжок через указатель на диспетчер |
| Move | `unique_ptr` move, дёшево | перемещение буфера (trivially, если тип позволяет) |

## Идея 7. `std::move_only_function` и ограничение CopyConstructible

`std::function` требует, чтобы сохранённый callable был **CopyConstructible**.
Это вытекает из того, что сама `std::function` копируема: при копировании она
должна скопировать и хранимый объект. Если callable содержит `unique_ptr` или
другой move-only ресурс, компилятор откажет:

```cpp
auto uptr = std::make_unique<int>(42);
// Ошибка компиляции: лямбда захватывает unique_ptr, она не копируема.
std::function<void()> f = [p = std::move(uptr)]() { /* ... */ };
```

Это болезненное ограничение: `unique_ptr`-захватывающие лямбды — типичный
паттерн в асинхронном коде (передача владения ресурсом в колбэк).

**C++23** устраняет это с помощью `std::move_only_function<R(Args...)>` из
`<functional>`. Она не копируется, зато принимает любой move-only callable:

```cpp
// C++23
std::move_only_function<void()> f = [p = std::move(uptr)]() { /* ... */ };
```

До C++23 обходной путь — хранить ресурс через `shared_ptr` (делает лямбду
копируемой ценой дополнительного разделения) или писать свою обёртку поверх
`unique_ptr<Concept>`, где копирующий конструктор удалён. Именно наша
`Function` из задания уже ближе к `move_only_function`: она не определяет
оператор копирования, поэтому компилятор неявно удалит его из-за `unique_ptr`
внутри.

## Идея 8. Стоимость vtable-вызова

Виртуальный вызов — основа type erasure, но у него есть конкретная цена, о
которой полезно знать.

**Механика косвенного вызова:**

```
obj.virtual_method()
    → загрузить vptr из объекта (1 load, обычно кэшируется)
    → загрузить адрес функции из vtable[N] (1 load)
    → call [адрес]
```

Два load плюс косвенный `call` вместо прямого — это само по себе не катастрофа
(~1–4 нс), но есть два эффекта, которые бьют сильнее:

1. **I-cache miss.** Если разные `Model<T>` вызываются в случайном порядке
   (например, гетерогенный `vector<Drawable>`), их тела расположены в разных
   местах кода. Каждый вызов может вытолкнуть предыдущий из L1-кэша инструкций.
   На горячем цикле это легко добавляет 10–20 нс на вызов.

2. **Невозможность inline.** Компилятор не знает в точке вызова, какой именно
   `Model<T>` будет за указателем, поэтому не может подставить тело функции.
   Теряются все оптимизации, которые inlining открывает: SRVO, constant
   folding, loop fusion.

**Devirtualization при `final`.**

Если компилятор **точно знает** динамический тип объекта в данной точке
(например, объект хранится по значению, а не через указатель), он может
заменить виртуальный вызов прямым — это называется devirtualization.
Пометка `final` помогает компилятору принять такое решение:

```cpp
template <class T>
struct Model final : Concept {    // final → компилятор знает: наследников нет
    // ...
};
```

Когда `Model<T>` помечен `final`, и компилятор видит объект типа `Model<T>` (а
не `Concept*`), он может devirtualize. В нашем случае объект сразу прячется за
`unique_ptr<Concept>`, поэтому выгода ограничена; `final` всё равно полезен как
документирующий сигнал «этот тип не предназначен для наследования».

Когда накладные расходы vtable критичны: горячий inner loop, обрабатывающий
миллионы элементов гетерогенного контейнера. В таких случаях рассматривай
`std::variant` (нет vtable, нет косвенности) или ручной SBO с указателем на
функцию-диспетчер (избегает виртуальной таблицы вообще).

## Идея 9. `typeid`, RTTI и `std::type_index`

`std::any` и наш `Any` используют `typeid` для проверки типа перед `cast`. Это
часть механизма **RTTI** (Run-Time Type Information), который компилятор
генерирует автоматически для полиморфных классов.

**Что делает `typeid`:**

```cpp
#include <typeinfo>

const std::type_info& ti = typeid(int);    // тип по имени
const std::type_info& tv = typeid(obj);    // тип объекта (динамический, если полиморфный)
```

`std::type_info` — непрозрачный класс со сравнением (`==`, `!=`) и методом
`name()`, возвращающим строку, **мангленное** имя которой зависит от
компилятора (g++ вернёт `"i"` для `int`, MSVC — `"int"`). Полагаться на `name()`
в portable-коде нельзя.

**Важный подвох: RTTI должен быть включён.** При компиляции с `-fno-rtti`
(иногда используется в embedded или при жёстком ограничении размера бинарника)
`typeid` для полиморфных типов ведёт себя неопределённо. `std::any` и
`std::type_index` требуют RTTI — без него не используются.

**`std::type_index` как хэшируемый ключ:**

`std::type_info` нельзя скопировать и нельзя положить в `std::map` напрямую.
`std::type_index` (заголовок `<typeindex>`) — тонкая обёртка, которая решает
обе проблемы: она копируема, сравнима (операторами `<`, `==`) и хэшируема, что
позволяет использовать её как ключ в `std::unordered_map`:

```cpp
#include <typeindex>
#include <unordered_map>
#include <string>

std::unordered_map<std::type_index, std::string> names;
names[std::type_index(typeid(int))]         = "int";
names[std::type_index(typeid(double))]      = "double";
names[std::type_index(typeid(std::string))] = "string";

// Достать имя типа хранимого в Any:
Any a = 42;
auto it = names.find(std::type_index(a.type()));
// it->second == "int"
```

Типичное применение — **фабрика по типу**: диспетчер, который по
`std::type_index` хранимого значения выбирает нужный обработчик или
сериализатор. Это позволяет регистрировать обработчики динамически, не
изменяя виртуального интерфейса.

### Подсказки

<details><summary>Задание 1 — куда положить вызываемый объект</summary>
В конструкторе `Function(F f)` создай `std::make_unique<Model<F>>(std::move(f))` и
сохрани в `self_`. В `operator()` сначала проверь `self_`: если пусто — `throw
std::bad_function_call{}`. Иначе верни `self_->invoke(std::forward<Args>(args)...)`.
</details>

<details><summary>Задание 1 — а если R == void?</summary>
`return f_(args...);` корректно компилируется и для `void`-функций: `return` с
выражением типа `void` разрешён. Так что отдельный случай для `void` писать не нужно —
один `return f_(std::forward<Args>(args)...);` работает для всех `R`.
</details>

<details><summary>Задание 2 — как сравнить типы</summary>
В `any_cast<T>` сравни `a.type() == typeid(T)`. Если совпало — нужно достать значение.
Тип спрятан, но ты *знаешь*, что это `Model<T>`, поэтому `static_cast` указателя на
базу к `Any::Model<T>*` безопасен. Если не совпало — `throw std::bad_cast{}`.
</details>

<details><summary>Задание 2 — почему копирование нетривиально</summary>
`unique_ptr` копировать нельзя. Поэтому копирующий конструктор должен сделать
*глубокую* копию: `other.self_ ? other.self_->clone() : nullptr`. А `clone()` в
`Model<T>` создаёт новую `Model<T>` с тем же `value_`. Оператор присваивания можно
выразить через copy-and-swap или просто переиспользовать конструктор копии.
</details>

<details><summary>Задание 3 — структура visit</summary>
`return std::visit(overloaded{ ... }, s);` — внутри две лямбды:
`[](const Circle& c){ ... }` и `[](const Square& s){ ... }`. Убедись, что обе
возвращают `double`, иначе типы веток разойдутся.
</details>

<details><summary>Задание 4 — одна строка делегирования</summary>
Конструктор как в Идее 2: `make_unique<Model<T>>(std::move(obj))`. `Drawable::draw()`
возвращает `self_->draw()`, а `Model<T>::draw()` — `obj_.draw()`. Общий базовый класс
у `Dot` и `Line` не нужен — в этом весь смысл.
</details>
