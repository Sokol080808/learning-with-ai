# Модуль 06 — Умные указатели и владение

В модуле 05 ты вручную писал деструктор и большую пятёрку, чтобы не текла память.
Умные указатели делают это **за тебя**. Большая идея модуля: *владение* (кто отвечает
за освобождение) должно быть выражено **в типе**.

## Идея 1. Проблема сырых указателей

`int* p = new int(5);` — глядя на `p`, нельзя понять: я владею этой памятью или просто
смотрю на чужую? Надо ли мне делать `delete`? Сырой указатель не отвечает на это.
Отсюда — утечки и двойные `delete`. Умные указатели кодируют ответ в самом типе.

## Идея 2. `std::unique_ptr` — единоличное владение

```cpp
#include <memory>
std::unique_ptr<int> p = std::make_unique<int>(5);  // владеет int на куче
*p = 10;                 // как обычный указатель
// delete не нужен — память освободится автоматически в деструкторе p
```

`unique_ptr` **нельзя копировать** (иначе двое владели бы одним ресурсом), но **можно
перемещать** — владение передаётся:

```cpp
std::unique_ptr<int> q = std::move(p);  // теперь владеет q, а p == nullptr
```

Это нулевая стоимость по сравнению с сырым указателем и почти всегда — твой выбор по
умолчанию для владения объектом в куче. Создавай через `std::make_unique<T>(...)`.

## Идея 3. `std::shared_ptr` — разделяемое владение

Когда у объекта **несколько** владельцев и неясно, кто последний:

```cpp
auto a = std::make_shared<int>(5);
auto b = a;            // КОПИЯ разрешена: теперь два владельца, счётчик == 2
// объект удалится, когда счётчик ссылок дойдёт до 0
std::cout << a.use_count();  // сколько владельцев сейчас
```

`shared_ptr` хранит счётчик ссылок (атомарный — потокобезопасный для счётчика). Он
дороже `unique_ptr`; используй его только когда владение действительно разделяемое.

## Идея 4. `std::weak_ptr` — наблюдение без владения

Два `shared_ptr`, ссылающихся друг на друга, создают **цикл**: счётчик никогда не
дойдёт до нуля → утечка. `weak_ptr` «наблюдает» за объектом, не увеличивая счётчик; он
разрывает циклы. Чтобы воспользоваться — временно повышают через `lock()`:

```cpp
std::weak_ptr<int> w = a;
if (auto sp = w.lock()) { /* объект ещё жив, sp — shared_ptr на него */ }
```

## Идея 5. Правило нуля снова

Если поля класса — умные указатели и контейнеры, тебе **не нужно** писать ни
деструктор, ни большую пятёрку: компилятор сгенерирует всё правильно. В задании ниже
ты построишь связный список на `unique_ptr` и заметишь, что деструктор писать не
требуется — а память при этом не течёт. Сравни с тем, что ты делал в модуле 05.

> Подвох: разрушение длинной цепочки `unique_ptr` рекурсивно. Для учебных размеров это
> не проблема; в проде такие структуры освобождают итеративно. Просто имей в виду.

### Подсказки

<details><summary>push_front с unique_ptr</summary>
Создай новый узел `std::make_unique<Node>(...)`. Его `next` должен забрать текущий
`head_` — но `unique_ptr` нельзя копировать, только `std::move`. Затем `head_`
должен забрать новый узел.
</details>
<details><summary>Обход списка (size/contains/to_vector)</summary>
Иди «сырым наблюдателем»: `const Node* cur = head_.get();` и пока `cur != nullptr`,
переходи `cur = cur->next.get();`. Здесь `get()` даёт сырой указатель для чтения —
мы НЕ забираем владение, только смотрим.
</details>

Это прямое следствие того, что внутри `unique_ptr`. Обсудим.
