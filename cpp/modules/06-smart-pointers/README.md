# Модуль 06 — Умные указатели и владение

В модуле 05 ты вручную писал деструктор и большую пятёрку, чтобы не текла память.
Умные указатели делают это **за тебя**. Большая идея модуля: *владение* (кто отвечает
за освобождение) должно быть выражено **в типе**.

## Идея 1. Проблема сырых указателей и понятие владения

`int* p = new int(5);` — глядя на `p`, нельзя понять: я **владею** этой памятью или
просто смотрю на чужую? Надо ли мне делать `delete`? Сырой указатель на этот вопрос не
отвечает. А ответ критичен: забыл `delete` — **утечка**; сделал `delete` дважды (или
удалил то, чем владеет кто-то ещё) — **двойное освобождение** и неопределённое
поведение; обратился после `delete` — **use-after-free**.

*Владение* — это ответственность за время жизни ресурса: кто-то должен ровно один раз
освободить его в нужный момент. Сырой указатель не кодирует, на ком эта ответственность,
поэтому её приходится держать в голове и в комментариях. Идея умных указателей: сделать
владение **частью типа** и привязать освобождение к разрушению объекта-владельца. Это
прямое применение RAII из модуля 05: ресурс захватывается в конструкторе, отпускается в
деструкторе, а раскрутка стека гарантирует вызов деструктора даже при исключении.

Ключевое разделение, которое мы будем проводить во всех заданиях:
- **владеющий** доступ — `unique_ptr` / `shared_ptr`: разрушение указателя освобождает ресурс;
- **наблюдающий** доступ — сырой `T*`, ссылка `T&`, `weak_ptr`: смотрим на чужой ресурс, ничего не освобождаем.

Правило для интерфейсов: **функция, которая забирает владение, должна принимать
`unique_ptr` по значению** (это видно по сигнатуре); функция, которая лишь читает объект,
должна принимать `const T&` или сырой `T*` — навязывать ей умный указатель незачем.

```cpp
void sink(std::unique_ptr<int> p);   // забирает владение: вызывающий это ВИДИТ
void observe(const int& x);          // только смотрит: владение остаётся снаружи
```

## Идея 2. `std::unique_ptr` — единоличное владение и семантика перемещения

`unique_ptr<T>` — ровно один владелец `T`. Внутри это всего лишь сырой указатель (плюс,
по умолчанию, пустой делитер): **никакого счётчика, никаких накладных расходов** по
сравнению с `T*`. Размер `unique_ptr<T>` со стандартным делитером равен размеру `T*`.

```cpp
#include <memory>
std::unique_ptr<int> p = std::make_unique<int>(5);  // владеет int на куче
*p = 10;                 // разыменование как у обычного указателя
// delete не нужен — память освободится в деструкторе p
```

**Механика единоличности.** Чтобы владельцем оставался ровно один, копирующие
конструктор и присваивание у `unique_ptr` `= delete`. Зато определены **перемещающие** —
они *передают* владение и зануляют источник:

```cpp
std::unique_ptr<int> q = std::move(p);  // теперь владеет q, а p == nullptr
assert(p == nullptr);                   // источник занулён — это гарантия, не случайность
```

Почему источник занулён? Если бы `move` оставил указатель и в `p`, и в `q`, то при
разрушении обоих `delete` сработал бы дважды. Перемещение `unique_ptr` — это «копируем
адрес в приёмник, в источник пишем `nullptr`»: дёшево и безопасно.

**Возврат из функции — это перемещение, а не копия:**

```cpp
std::unique_ptr<int> make() {
    auto p = std::make_unique<int>(1);
    return p;            // НЕ копия (она запрещена) — перемещение/elision. Так и нужно.
}
```

**Трейд-офф.** `unique_ptr` бесплатен, но даёт ровно одного владельца. Это почти всегда
правильный выбор по умолчанию для владения объектом в куче. Тянись к `shared_ptr` только
когда владельцев действительно несколько (см. Идею 4).

**Частые ошибки.**

```cpp
// (1) Попытка скопировать — не скомпилируется. Лечится std::move.
std::unique_ptr<int> a = make_unique_int(1);
// std::unique_ptr<int> b = a;        // ОШИБКА: копирование запрещено
std::unique_ptr<int> b = std::move(a);  // ОК: передали владение

// (2) Использование источника после move — он nullptr, разыменование = UB.
// *a = 5;   // a уже nullptr после move — так нельзя

// (3) Делать unique_ptr из стекового объекта — он попытается его delete.
int x = 0;
// std::unique_ptr<int> bad(&x);   // КАТАСТРОФА: delete нестековой памяти
```

## Идея 3. `make_unique` / `make_shared` — почему именно фабрики

Создавай умные указатели через `std::make_unique<T>(args...)` и
`std::make_shared<T>(args...)`, а не через `new`. Причины конкретные, не «стилистические».

**1. Безопасность при исключениях.** Рассмотри вызов с двумя аргументами:

```cpp
f(std::unique_ptr<A>(new A), std::unique_ptr<B>(new B));   // ОПАСНО
```

Порядок вычисления подвыражений не задан: компилятор вправе сделать `new A`, потом
`new B`, и только потом сконструировать оба `unique_ptr`. Если `new B` бросит исключение,
память от `new A` уже выделена, но ещё не отдана под защиту `unique_ptr` — **утечка**.
`make_unique` упаковывает выделение и захват в один шаг, и проблема исчезает:

```cpp
f(std::make_unique<A>(), std::make_unique<B>());           // безопасно
```

**2. Без повторения типа и без голого `new`** — меньше места для опечатки: `auto p =
std::make_unique<Widget>(...)` вместо `std::unique_ptr<Widget> p(new Widget(...))`.

**3. Одно выделение у `make_shared`.** `shared_ptr` хранит сам объект и **управляющий
блок** (со счётчиками). `shared_ptr<T>(new T)` делает **два** выделения: объект и блок.
`make_shared<T>(...)` выделяет **один** непрерывный кусок под оба — быстрее и компактнее.

**Тонкость про `make_shared` и память.** Поскольку объект и управляющий блок лежат в
одном куске, память под объект освобождается только тогда, когда исчезнет последний
`weak_ptr` (блок жив, пока живы `weak`-ссылки). Для огромных объектов с долгоживущими
`weak_ptr` это иногда повод вернуться к `shared_ptr<T>(new T)`. Для учебных задач —
всегда `make_shared`.

## Идея 4. `std::shared_ptr` — разделяемое владение и управляющий блок

Когда у объекта **несколько** равноправных владельцев и заранее неясно, кто умрёт
последним, нужен `shared_ptr`:

```cpp
auto a = std::make_shared<int>(5);   // use_count() == 1
auto b = a;                          // КОПИЯ разрешена: владельцев 2, use_count() == 2
// объект удалится, когда счётчик сильных ссылок дойдёт до 0
std::cout << a.use_count();          // сколько сильных владельцев сейчас
```

**Механика.** Рядом с объектом живёт **управляющий блок** с двумя счётчиками: число
сильных ссылок (`shared`) и число слабых (`weak`). Копирование `shared_ptr` атомарно
увеличивает сильный счётчик, разрушение — уменьшает. Когда сильный счётчик падает до 0 —
**объект уничтожается** (вызывается его деструктор). Когда и слабый падает до 0 —
освобождается сам управляющий блок.

**Атомарность — и что она НЕ покрывает.** Инкремент/декремент счётчика атомарны: два
потока могут безопасно *копировать и разрушать* свои `shared_ptr` на один объект. Но это
не делает **сам объект** потокобезопасным: одновременная запись в `*p` из двух потоков —
по-прежнему гонка. И сам один и тот же `shared_ptr`-переменную нельзя без синхронизации
менять из двух потоков (`reset`/присваивание).

**Трейд-офф.** `shared_ptr` заметно дороже `unique_ptr`: размер вдвое больше (указатель
на объект + указатель на блок), атомарные операции со счётчиком, лишнее выделение (если
не `make_shared`). Поэтому правило: **по умолчанию `unique_ptr`, `shared_ptr` — только
когда владение реально разделяемое.** «На всякий случай shared» — распространённый
антипаттерн.

**Частые ошибки.**

```cpp
// (1) Два независимых shared_ptr из одного сырого указателя => ДВА управляющих блока
//     => двойное удаление.
int* raw = new int(5);
std::shared_ptr<int> s1(raw);
// std::shared_ptr<int> s2(raw);   // КАТАСТРОФА: оба думают, что владеют, оба сделают delete
std::shared_ptr<int> s2 = s1;      // ПРАВИЛЬНО: копия делит ОДИН блок

// (2) Передавать shared_ptr туда, где владение не нужно. Если функция только читает —
//     пусть берёт const T& или T*. shared_ptr в сигнатуре навязывает лишние атомарные
//     операции и вводит в заблуждение о владении.
```

## Идея 5. `std::weak_ptr` — наблюдение без владения и разрыв циклов

Два `shared_ptr`, ссылающихся **друг на друга**, образуют **цикл владения**: каждый
держит сильную ссылку на другого, поэтому ни один счётчик никогда не дойдёт до нуля.
Даже когда снаружи ссылок не осталось — пара объектов держит сама себя. Это **утечка**,
которую RAII не спасает: деструкторы просто не вызываются.

```cpp
struct Node { std::shared_ptr<Node> other; };
auto a = std::make_shared<Node>();
auto b = std::make_shared<Node>();
a->other = b;   // b.use_count() == 2
b->other = a;   // a.use_count() == 2
// выходим из области видимости: внешние a и b исчезли, но внутренние other держат
// друг друга => use_count каждого остаётся 1 => деструкторы НЕ вызовутся => утечка
```

`weak_ptr` «наблюдает» за объектом, **не увеличивая сильный счётчик**. Он не даёт права
разыменования напрямую — ведь объект мог уже умереть. Чтобы воспользоваться, его
**временно повышают** через `lock()`, который возвращает `shared_ptr`:

```cpp
std::weak_ptr<Node> w = a;
if (auto sp = w.lock()) {     // объект ещё жив: sp — валидный shared_ptr на него
    /* пользуемся sp; пока он жив, объект гарантированно не умрёт */
} else {
    /* объект уже уничтожен — lock() вернул пустой shared_ptr */
}
```

Можно ещё спросить `w.expired()` (объект мёртв?) — но для использования всё равно нужен
`lock()`, и проверять надо именно его результат: между `expired()` и обращением объект
теоретически мог умереть, а `lock()` атомарен и потому надёжен.

**Где это спасает.** Классический паттерн «родитель ↔ дети»: родитель **владеет** детьми
(`shared_ptr`), а ребёнок лишь **ссылается** на родителя (`weak_ptr`). Тогда удаление
родителя не блокируется детьми. Именно это ты построишь в задании 3.

```cpp
struct TreeNode {
    std::vector<std::shared_ptr<TreeNode>> children;  // владеем детьми (вниз)
    std::weak_ptr<TreeNode>                parent;     // лишь смотрим на родителя (вверх)
};
```

**`enable_shared_from_this`.** Если узлу нужно отдать `shared_ptr` на самого себя
(например, чтобы записать `child->parent = (this как shared_ptr)`), сделать
`shared_ptr<TreeNode>(this)` **нельзя** — это создаст второй управляющий блок и приведёт
к двойному удалению. Правильно — унаследовать `std::enable_shared_from_this<TreeNode>` и
вызвать `shared_from_this()`; он вернёт `shared_ptr`, *разделяющий тот же* управляющий
блок. Важное условие: объект уже должен **жить под управлением `shared_ptr`** (создан
через `make_shared`); вызывать `shared_from_this()` из конструктора нельзя.

## Идея 6. Pimpl на `unique_ptr` — скрыть реализацию за указателем

`unique_ptr` — основа идиомы **Pimpl** (Pointer to IMPLementation). Идея: публичный класс
хранит только `unique_ptr<Impl>` на скрытую структуру, а все настоящие поля живут в
`Impl`, объявленной **только в `.cpp`**. Заголовок не знает устройства `Impl` — это
«незавершённый тип» (forward declaration).

```cpp
// header
class Counter {
public:
    Counter();
    ~Counter();                       // ОБЪЯВЛЕН здесь, ОПРЕДЕЛЁН в .cpp
    int value() const;
private:
    struct Impl;                      // незавершённый тип
    std::unique_ptr<Impl> impl_;
};
```

**Зачем.** (1) **Сокрытие**: клиент видит только публичный интерфейс, не приватные поля.
(2) **Развязка компиляции**: меняя приватные поля (добавил поле в `Impl`), ты не трогаешь
заголовок — значит, не нужно перекомпилировать *всех*, кто его включает. На больших
проектах это экономит часы сборки. (3) **Стабильный ABI**: размер публичного класса не
зависит от приватных полей (он всегда «один указатель»).

**Главный подвох — деструктор.** Чтобы удалить `unique_ptr<Impl>`, компилятору в точке
вызова деструктора нужен **полный тип** `Impl`. Если деструктор `Counter` неявный (или
`= default` прямо в заголовке), он инстанцируется там, где `Impl` ещё незавершён, и ты
получишь ошибку вида *«invalid application of sizeof to an incomplete type»* или
*«deletion of pointer to incomplete type»*. Лечение: **объяви** деструктор в заголовке, а
**определи** в `.cpp` (после полного определения `Impl`), пусть даже как `= default`:

```cpp
// .cpp
struct Counter::Impl { int v = 0; };   // теперь Impl — полный тип
Counter::Counter() : impl_(std::make_unique<Impl>()) {}
Counter::~Counter() = default;         // здесь Impl полный — всё корректно
int Counter::value() const { return impl_->v; }
```

То же касается **операций перемещения**: если они нужны, объяви их в заголовке и определи
в `.cpp` тем же приёмом. Копирование при Pimpl по умолчанию запрещаем (или пишем вручную
глубокое), потому что `unique_ptr` некопируем.

## Идея 7. Правило нуля снова

Если поля класса — умные указатели и контейнеры, тебе **не нужно** писать ни деструктор,
ни большую пятёрку: компилятор сгенерирует всё правильно. В задании со списком ты
заметишь, что деструктор писать не требуется — а память при этом не течёт. Сравни с тем,
что ты делал в модуле 05. (Pimpl — заметное **исключение** из правила нуля: там
деструктор приходится писать руками именно из-за незавершённого типа, см. Идею 6.)

> Подвох: разрушение длинной цепочки `unique_ptr` (как в списке) рекурсивно. Для учебных
> размеров это не проблема; в проде такие структуры освобождают итеративно. Просто имей в
> виду.

## Идея 8. `unique_ptr<T, D>` — обобщение на произвольные ресурсы (custom deleter)

По умолчанию `unique_ptr<T>` вызывает `delete` — и это работает только для памяти,
выделенной через `new`. Но владение бывает у любых ресурсов: файловых дескрипторов,
`FILE*`, мьютексов ОС, сетевых сокетов. Для них `unique_ptr` обобщён вторым параметром
шаблона — **делитером** `D`:

```cpp
// Обёртка над FILE* с лямбда-делитером: fclose вызовется автоматически.
auto file_deleter = [](FILE* f) { if (f) std::fclose(f); };
using UniqueFile = std::unique_ptr<FILE, decltype(file_deleter)>;

UniqueFile f(std::fopen("data.bin", "rb"), file_deleter);
if (!f) throw std::runtime_error("не удалось открыть файл");
// ... читаем через f.get() ...
// fclose(f.get()) вызовется в деструкторе, даже если выброшено исключение.
```

**Механика внутри.** `unique_ptr<T, D>` хранит пару `(T*, D)`. Если `D` — пустой класс
(в т.ч. лямбда без захвата или указатель на функцию через `std::function`), то благодаря
**Empty Base Optimisation** пара занимает ровно `sizeof(T*)`. Если делитер — лямбда с
захватом или `std::function`, он занимает дополнительное место.

**Зачем это важно.** Прежде чем писать RAII-обёртку вручную (с конструктором, деструктором,
запретом копирования и поддержкой перемещения), спроси себя: а не достаточно ли
`unique_ptr<T, D>`? Он уже некопируем, перемещаем, гарантирует освобождение. Самописная
обёртка нужна, когда ресурс не описывается парой «указатель + делитер» (например,
`int`-дескриптор без адреса объекта).

```cpp
// Пример с дескриптором ОС: тип ресурса — int, а не указатель.
// unique_ptr тут не подойдёт напрямую (нет T*), нужна отдельная обёртка.
// Но FILE*, дескрипторы OpenSSL, библиотечные хэндлы типа HANDLE (Windows) —
// все они «указатели», и unique_ptr отлично справляется.
```

**Почему именно лямбда, а не указатель на функцию?** Лямбда без захвата конвертируема в
указатель на функцию и не занимает лишних байтов. Лямбда с захватом — нет. Когда
освобождение требует дополнительного контекста (например, размера блока), удобнее написать
собственный класс-делитер с `operator()`.

## Идея 9. `unique_ptr` как первый movable-only тип: связь с move-семантикой

`unique_ptr` — практически первый тип в стандартной библиотеке, который **перемещаемый,
но не копируемый**. Это полезная модель для понимания move-семантики.

**`std::move` — это каст, не операция.** Имя обманчиво: `std::move(x)` ничего не
перемещает само по себе. Оно делает одно: **приводит `x` к rvalue-ссылке** (`T&&`). После
этого компилятор при разрешении перегрузки выберет конструктор или оператор `=`, принимающий
`T&&`, — и вот тогда уже произойдёт перемещение. Можно думать о `std::move` как о
`static_cast<T&&>(x)` — это буквально его определение в стандарте.

```cpp
std::unique_ptr<int> a = std::make_unique<int>(1);
// std::unique_ptr<int> b = a;          // ОШИБКА: copy-ctor = delete
std::unique_ptr<int> b = std::move(a);  // OK: std::move даёт T&&, выбирается move-ctor
// a теперь nullptr (это гарантия, а не оптимизация)
```

**Почему источник должен быть «занулён»?** Если бы `move` оставил исходный указатель
нетронутым, деструктор `a` и деструктор `b` оба вызвали бы `delete` — двойное освобождение.
Единственная безопасная семантика: перенести значение в приёмник и привести источник в
«пустое» (валидное, но неопределённое) состояние. Для `unique_ptr` это `nullptr`.

**Movable-only как паттерн.** Если твой тип олицетворяет **уникальный ресурс** (файл,
мьютекс, сетевое соединение, поток выполнения), запрети копирование (удали copy-ctor и
copy-assignment) и разрешили только перемещение. Это явно выражает семантику: ресурс
существует в одном месте, его можно **передать**, но нельзя **размножить**. `std::thread`,
`std::fstream`, `std::unique_lock` следуют тому же принципу.

```cpp
// Как выглядит минимальный movable-only RAII-тип:
class UniqueHandle {
    int fd_ = -1;
public:
    explicit UniqueHandle(int fd) : fd_(fd) {}
    ~UniqueHandle() { if (fd_ >= 0) close(fd_); }

    UniqueHandle(UniqueHandle&& o) noexcept : fd_(std::exchange(o.fd_, -1)) {}
    UniqueHandle& operator=(UniqueHandle&& o) noexcept {
        if (this != &o) { if (fd_ >= 0) close(fd_); fd_ = std::exchange(o.fd_, -1); }
        return *this;
    }
    UniqueHandle(const UniqueHandle&) = delete;
    UniqueHandle& operator=(const UniqueHandle&) = delete;
};
```

**`std::exchange`** (из `<utility>`) — полезный примитив: `std::exchange(a, b)` записывает
`b` в `a` и возвращает старое значение `a`. Это ровно то, что нужно в move-конструкторе:
«возьми старый дескриптор себе, занули источник».

## Идея 10. `std::span` — невладеющий непрерывный вид

До C++20 функция, которая хотела принять «подпоследовательность массива», вынуждена была
принимать пару `(T*, std::size_t)` или пару итераторов. Это многословно, небезопасно (нет
проверки диапазона) и несовместимо с `std::vector`, массивами, `std::array` одновременно.

`std::span<T>` — **невладеющий вид** на непрерывный участок памяти: пара `(T*, размер)`,
упакованная в типобезопасный класс. Не выделяет памяти, не копирует элементы, дёшев при
передаче по значению.

```cpp
#include <span>

void print_all(std::span<const int> s) {
    for (int x : s) std::cout << x << ' ';
}

std::vector<int> v = {1, 2, 3, 4, 5};
int arr[] = {10, 20, 30};

print_all(v);                // работает для vector
print_all(arr);              // работает для C-массива
print_all({v.data(), 3});    // первые три элемента вектора
```

**Почему не `T*` + `size`?** `span` несёт оба параметра вместе, исключая их рассинхронизацию.
`span<const T>` читается как «невладеющий вид на массив T только для чтения» — ясно по
сигнатуре. Компилятор не спутает `span<int>` с `span<const int>` (правила конверсии
аналогичны указателям).

**Bounds checking.** `span::operator[]` — без проверки границ (как у `std::vector`); для
проверки используй `span::at()` (бросает `std::out_of_range`) или пиши явно `assert(i < s.size())`.

**Срезы без копирования:**

```cpp
std::span<int> all(v);
std::span<int> first3 = all.first(3);   // span на первые три
std::span<int> last2  = all.last(2);    // span на последние два
std::span<int> mid    = all.subspan(1, 3); // три элемента с индекса 1
```

**Подвох: время жизни.** `span` хранит адрес — если владелец данных умер (вектор перераспределил
память, локальный массив вышел из области), `span` станет «висячим». Никакой умный указатель
этого не поймает. Правило: `span` должен жить **строго внутри** времени жизни владельца.

**Фиксированный vs. динамический размер.** `std::span<T>` — динамический (`extent ==
std::dynamic_extent`, размер хранится в объекте). `std::span<T, N>` — статический (размер
известен на этапе компиляции, не хранится в объекте). Используй статический, когда размер
фиксирован по смыслу и хочешь поймать несоответствие при компиляции.

## Идея 11. `std::optional<T>` — nullable-значение без кучи

Классический способ выразить «значение, которого может не быть» на C — возвращать сигнальное
число (`-1`, `nullptr`, `INT_MAX`). Это ломает типовую систему: функция, которая возвращает
«индекс или −1», обещает тип `int`, но несёт дополнительную семантику, невидимую
компилятору. `std::optional<T>` решает это правильно.

**Механика.** `optional<T>` хранит: флаг «занято/пусто» и, если занято, сам объект `T`
**инлайн** (без выделения на куче). Размер обычно `sizeof(T) + padding` для выравнивания.
Никакого `new`, никаких косвенных обращений.

```cpp
#include <optional>

std::optional<int> find_first(const std::vector<int>& v, int target) {
    for (std::size_t i = 0; i < v.size(); ++i) {
        if (v[i] == target) return i;    // неявное создание optional<int>{i}
    }
    return std::nullopt;                 // пустой optional
}

auto idx = find_first({3, 1, 4}, 1);
if (idx) {                               // operator bool: есть ли значение?
    std::cout << "индекс: " << *idx;     // operator*: значение (UB если пусто!)
}
idx.value();                             // то же, что *idx, но бросает std::bad_optional_access если пусто
idx.value_or(-1);                        // значение или замена
```

**Когда `optional`, когда `unique_ptr`?**

| Критерий | `optional<T>` | `unique_ptr<T>` |
|---|---|---|
| Где хранится `T` | инлайн, без кучи | на куче |
| Размер | `sizeof(T) + 1..8` байт | `sizeof(void*)` |
| Полиморфизм | нет | да (тип времени исполнения) |
| Стоимость разыменования | нет косвенности | один dereference |
| Перемещение «пустого» | без выделения | без выделения |
| Сценарий | «небольшое значение, которого может не быть» | «объект на куче, которого может не быть» |

Практически: **`optional<T>`** — для возврата «значения или ничего» из функции, для
необязательных полей структуры (когда `T` небольшой и не полиморфный). **`unique_ptr<T>`** —
когда объект большой, полиморфный, или надо «ленивое» выделение.

**Частые ошибки.**

```cpp
std::optional<int> opt;
// *opt — неопределённое поведение, если opt пуст. Всегда проверяй.
if (opt) {
    int x = *opt;  // безопасно
}
// opt.value() — бросает исключение, *opt — UB. Выбирай осознанно.
```

## Идея 12. Итеративное освобождение длинного `unique_ptr`-списка

В задании 2 деструктор `IntList` написан «по правилу нуля» — явного деструктора нет,
компилятор генерирует его сам. Цепочка освобождения такая:

1. Разрушается `head_` → вызывается деструктор `Node`.
2. Деструктор `Node` разрушает `next` → вызывается деструктор следующего `Node`.
3. Шаг 2 повторяется для каждого следующего узла — **рекурсивно**.

Для тысяч узлов это N рекурсивных вызовов. Каждый вызов кладёт фрейм на стек. При достаточно
большом списке — **переполнение стека** (`stack overflow`) и аварийное завершение.

**Почему так в эталоне.** Для учебных размеров это не проблема: типичный лимит стека —
1–8 МБ, кадр деструктора `Node` мал (~16 байт), влезает ~50 000 – 500 000 узлов. В
проде списки бывают длиннее.

**Как починить — итеративный деструктор:**

```cpp
IntList::~IntList() {
    // Освобождаем узлы итеративно, чтобы не переполнить стек.
    while (head_) {
        head_ = std::move(head_->next);   // берём следующий — голова умирает
        // старый head_ разрушен после перемещения, но без рекурсии
    }
}
```

**Почему это работает.** `head_ = std::move(head_->next)` — в правой части `move` вытаскивает
`next` из текущего узла (узел теперь владеет `nullptr`), в левой части старый `head_`
(уже без наследников!) разрушается. Без рекурсии: каждый узел умирает немедленно после
своего `next`.

**Подвох: «почти правило нуля».** Добавив явный деструктор, мы нарушаем правило нуля. Это
побочный эффект оптимизации. Альтернатива — хранить список как `std::vector<Node>` или
использовать нерекурсивную структуру данных изначально. Для `unique_ptr`-списков, где глубина
принципиально неограничена, итеративное освобождение — стандартная практика (так устроены
деструкторы многих реальных linked-list реализаций).
