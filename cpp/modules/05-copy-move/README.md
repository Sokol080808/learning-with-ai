# Модуль 05 — Копирование, перемещение и правило 0/3/5

Это модуль, после которого C++ «щёлкает». Микро-проект здесь — написать **свой
динамический массив** `IntVector`, управляющий памятью вручную. Сделав это, ты
поймёшь, что `std::vector` делает за тебя, и почему RAII так важна.

## Идея 1. Что значит «владеть ресурсом» и откуда берутся проблемы

Объект **владеет** ресурсом, если в его время жизни этот ресурс должен быть жив, а в
момент уничтожения объекта — освобождён. Ресурс — это не обязательно память: файловый
дескриптор, мьютекс, сокет, GPU-буфер, соединение с БД. Но самый наглядный ресурс —
блок памяти из кучи. Класс хранит `int* data_`, в конструкторе делает `new[]`, в
деструкторе — `delete[]`. Пока класс единственный владелец — всё хорошо.

Беда начинается, когда объект **копируют или возвращают из функции**. Компилятор по
умолчанию генерирует копирующий конструктор, который копирует поля **побитово**
(memberwise copy). Для `int* data_` это значит: скопировать сам указатель, а не то, на
что он указывает. Получаем два объекта, чьи `data_` указывают на **один** блок:

```cpp
struct Bad {
    int* data_;
    Bad(std::size_t n) : data_(new int[n]) {}
    ~Bad() { delete[] data_; }   // деструктор есть, а копирования нет → беда
};

void f() {
    Bad a(10);
    Bad b = a;     // компилятор скопировал указатель: a.data_ == b.data_
}                  // сначала ~b: delete[] data_; потом ~a: delete[] ТОГО ЖЕ → double free → краш
```

Это **double free**. А если бы `b` пережил `a`, то после `~a` указатель `b.data_` стал
бы **висячим** (dangling): обращение к нему — undefined behavior. Корень зла: деструктор
освобождает ресурс, но компилятор-генерируемое копирование о владении не знает. Отсюда
вытекает вся остальная теория модуля.

## Идея 2. Глубокое vs поверхностное копирование

**Поверхностная** копия (shallow) дублирует значения полей как есть — для указателя это
копия адреса. **Глубокая** копия (deep) выделяет НОВЫЙ ресурс и копирует *содержимое*,
так что у каждого объекта свой независимый блок:

```cpp
Buffer::Buffer(const Buffer& other)
    : data_(other.size_ ? new double[other.size_] : nullptr),
      size_(other.size_) {
    for (std::size_t i = 0; i < size_; ++i) data_[i] = other.data_[i];
}
```

Признак того, что копия действительно глубокая: после `Buffer b = a; b[0] = 99;`
оригинал `a[0]` **не меняется**, и `a.data() != b.data()` (разные адреса). Именно это
проверяют тесты. Тонкость: при `size_ == 0` не выделяй память вообще — храни `nullptr`.
`new double[0]` хоть и легален, но даёт ненулевой указатель, который всё равно нельзя
разыменовать; проще и честнее держать пустое состояние как `(nullptr, 0)`.

**Частая ошибка** — забыть скопировать содержимое и оставить только `size_`: тогда у
копии «правильный размер», но мусор внутри. Вторая частая ошибка — выделить буфер
размера `other.capacity_`, а скопировать `other.size_` элементов и потерять разницу;
следи, что именно ты дублируешь.

## Идея 3. Большая пятёрка и правило трёх/пяти

Если класс управляет ресурсом, у него есть пять «специальных» функций-членов:

```cpp
~T();                       // 1. деструктор          — освободить ресурс
T(const T&);                // 2. копирующий конструктор
T& operator=(const T&);     // 3. копирующее присваивание
T(T&&) noexcept;            // 4. перемещающий конструктор
T& operator=(T&&) noexcept; // 5. перемещающее присваивание
```

**Правило трёх:** если тебе понадобился хотя бы один из {деструктор, копир.
конструктор, копир. присваивание} — почти наверняка нужны все три. Они связаны единой
логикой владения: раз ты пишешь деструктор, значит, владеешь ресурсом; значит,
дефолтное (поверхностное) копирование неверно; значит, и копир. конструктор, и копир.
присваивание надо переопределить.

**Правило пяти:** добавь две move-версии — не ради корректности (без них код всё равно
правильный, просто будет копировать там, где мог бы перемещать), а ради
**производительности**. Move превращает дорогое копирование буфера в перевешивание
указателя.

Важный механический нюанс: **объявив свой деструктор или копирование, ты подавляешь
автогенерацию move-операций**. Компилятор «из осторожности» их не создаёт — он не
уверен, что побитовое перемещение корректно для класса, который явно вмешался в
управление ресурсом. Поэтому если ты написал деструктор и хочешь move — пиши его
**руками** (или явно `= default`, когда это безопасно). Молчаливое отсутствие move не
ломает программу, но молча превращает каждое «перемещение» обратно в копию.

## Идея 4. Копирование vs перемещение — в чём физическая разница

Копирование **дублирует** данные: было N байт, стало 2N, оба объекта самостоятельны.
Перемещение **перевешивает владение**: новый объект забирает указатель источника, а
источник обнуляется. Никакого `new[]`, никакого цикла копирования — три присваивания
указателя/размера, и буфер «переехал».

```cpp
Buffer::Buffer(Buffer&& other) noexcept
    : data_(other.data_), size_(other.size_) {   // забрали ресурс
    other.data_ = nullptr;                       // ОБЯЗАТЕЛЬНО обнулить источник,
    other.size_ = 0;                             // иначе его ~Buffer удалит наш буфер
}
```

Если забыть обнулить источник — деструктор источника сделает `delete[]` уже
**украденного** буфера, и получится double free. Это самая частая ошибка в move-ctor.

`std::move(x)` сам по себе ничего не двигает — это просто `static_cast` к rvalue-ссылке
`T&&`. Он лишь помечает `x` как «к нему можно отнестись как к временному, его ресурс
можно растащить». Какая именно функция отработает (копир. или перемещающий
конструктор), решается перегрузкой: rvalue выбирает `T(T&&)`, lvalue — `T(const T&)`.

```cpp
Buffer a(1000);
Buffer b = a;              // a — lvalue → копирующий конструктор (дорого)
Buffer c = std::move(a);   // выражение — rvalue → перемещающий (дёшево); a теперь пуст
```

После move источник обязан остаться в **валидном, но неопределённом** состоянии: его
можно безопасно разрушить и переприсвоить, но читать конкретные значения нельзя. В
наших задачах мы делаем сильнее — оставляем источник *пустым* `(nullptr, 0)`, и тесты
это проверяют (`a.size() == 0`, `a.data() == nullptr`).

Когда выбирать что? Копируй, если оригинал нужен дальше. Перемещай, если источник —
временный объект или ты явно сказал `std::move` и больше к нему не обращаешься.
Контейнеры STL перемещают элементы автоматически при реаллокации — но только если move
помечен `noexcept` (см. Идею 6).

## Идея 5. Идиома copy-and-swap

`operator=` коварен: нужно освободить старый ресурс, выделить новый и при этом пережить
**самоприсваивание** (`a = a`) и возможное исключение при выделении, не оставив объект
в полуразрушенном состоянии. Наивная реализация легко ошибается:

```cpp
Buffer& operator=(const Buffer& o) {   // ОПАСНО
    delete[] data_;                    // 1) освободили старое...
    data_ = new double[o.size_];       // 2) ...а если тут бросит bad_alloc — объект сломан
    size_ = o.size_;                   // а при a=a мы только что удалили СВОЙ же буфер
    for (...) data_[i] = o.data_[i];   // и читаем из освобождённой памяти — UB
    return *this;
}
```

Идиома **copy-and-swap** решает обе проблемы разом. Берём аргумент **по значению** (его
копия делается копирующим конструктором — переиспользуем уже написанный код), затем
обмениваем внутренности с `*this` через `swap`. Старое содержимое уезжает во временный
параметр и уничтожается вместе с ним в конце функции:

```cpp
void Buffer::swap(Buffer& o) noexcept {
    std::swap(data_, o.data_);
    std::swap(size_, o.size_);
}

Buffer& Buffer::operator=(Buffer other) {   // other — КОПИЯ аргумента
    swap(other);                            // забрали новое, отдали старое в other
    return *this;                           // ~other уничтожит наше прежнее содержимое
}
```

Почему это безопасно: (1) самоприсваивание не страшно — `a=a` просто скопирует и
обменяет с собой; (2) если копия не удалась (исключение в копир. конструкторе) — мы ещё
не тронули `*this`, он цел (строгая гарантия исключений); (3) один `operator=` по
значению обслуживает **и** копирование, **и** перемещение: передашь lvalue — параметр
скопируется, передашь rvalue (`std::move`) — параметр переместится.

`swap` обязан быть `noexcept` — он лишь меняет местами скаляры/указатели, бросать там
нечему, а контейнеры и алгоритмы рассчитывают на это. В заданиях `swap` вынесен в
отдельный публичный метод — реализуй его честно, и присваивание ляжет в две строки.

> Замечание: у copy-and-swap есть цена — `operator=` всегда делает полную копию, даже
> когда буферы одинакового размера и можно было бы переиспользовать память. Для
> учебного кода это правильный выбор (проще и безопаснее); в горячем коде иногда пишут
> «ручное» присваивание с переиспользованием буфера.

## Идея 6. noexcept у move — почему это не косметика

`noexcept` — обещание «эта функция не выбрасывает исключений». Для move-операций оно
имеет конкретное практическое последствие. Когда `std::vector` растёт и переносит
элементы в новый буфер, он стоит перед выбором: перемещать элементы или копировать.
Если move-конструктор элемента **может** бросить, vector обязан **копировать** — иначе
исключение на середине переноса оставит контейнер в разрушенном состоянии (часть
элементов уже «украдена» из старого буфера, откатить нельзя). Это гарантируется через
`std::move_if_noexcept`.

Итог: **move без `noexcept` молча деградирует до копирования** при реаллокации
контейнера. Ты пишешь move-конструктор, ожидаешь дешёвый перенос, а `vector<Buffer>`
всё равно копирует, потому что твой move «не обещал» не бросать.

```cpp
Buffer(Buffer&&) noexcept;             // ✓ vector будет перемещать
Buffer(Buffer&&);                      // ✗ vector на всякий случай копирует
```

Перемещение указателей и скаляров действительно не бросает — поэтому ставить `noexcept`
здесь и честно, и обязательно. Проверить можно типажом
`std::is_nothrow_move_constructible_v<Buffer>` — ровно это делает один из тестов. Если
ты случайно сделаешь внутри move что-то бросающее (например, выделение памяти), типаж
станет ложным и тест это поймает.

## Идея 7. move-only типы — запрет копирования как часть контракта

Не всякий ресурс *можно* осмысленно скопировать. Дубликат файлового дескриптора,
двойное владение мьютексом, две «уникальные» ссылки на одну сущность — это либо ошибка,
либо требует особой логики. Для таких ресурсов копирование **запрещают** на уровне
типа, оставляя только перемещение. Так устроен `std::unique_ptr`, `std::thread`,
`std::fstream`, `std::mutex` (последний вообще ни копировать, ни перемещать нельзя).

Запрет делается через `= delete`:

```cpp
class UniqueHandle {
public:
    UniqueHandle(const UniqueHandle&) = delete;             // копировать нельзя
    UniqueHandle& operator=(const UniqueHandle&) = delete;
    UniqueHandle(UniqueHandle&&) noexcept;                  // перемещать можно
    UniqueHandle& operator=(UniqueHandle&&) noexcept;
};
```

Удалённая функция **участвует в разрешении перегрузки**, но её вызов — ошибка
компиляции. Поэтому `UniqueHandle b = a;` (где `a` — lvalue) не «соскользнёт» на
move-конструктор, а честно упадёт с понятной ошибкой. А `UniqueHandle b =
std::move(a);` соберётся, потому что выберет move-версию. Move-only-ность типа можно
проверить во время компиляции типажами — тесты содержат `static_assert`, требующие
`!is_copy_constructible_v` и `is_move_constructible_v`.

Ключевые операции move-only-обёртки (по образцу `unique_ptr`):
- **release()** — отдать владение наружу: вернуть ресурс и стать пустым, **не**
  освобождая его (теперь за ресурс отвечает вызывающий код).
- **reset(r)** — освободить текущий ресурс и взять новый (или стать пустым при
  `reset()`).
- **move-assign** — освободить своё, забрать чужое, обнулить источник; и не сломаться на
  self-move (`x = std::move(x)`), который на практике прилетает из обобщённого кода.

Главный инвариант move-only-владельца: **в каждый момент за ресурс отвечает ровно один
объект**. Ни утечки (ресурс не потерян), ни двойного освобождения (ресурс не освобождён
дважды). В задании это проверяет учебный «реестр ресурсов»: он считает, сколько
ресурсов сейчас живо, и падает при попытке освободить лишнее. Если твой move случайно
оставит ресурс и у источника, и у приёмника — счётчик «живых» не сойдётся, и тест
покраснеет.

## Идея 8. Правило НУЛЯ — лучшее правило

Лучший способ соблюсти правило пяти — **не управлять ресурсом вручную**. Если поля
класса сами умеют копироваться/перемещаться/освобождаться (`std::vector`,
`std::string`, `std::unique_ptr`), то компилятор сгенерирует все пять методов
правильно, и тебе не нужно писать **ни одного**. Это и есть **правило нуля**.

> Тогда зачем мы пишем `IntVector` руками? Чтобы один раз понять, что происходит
> «под капотом». В реальном коде ты будешь применять правило нуля. Но junior обязан
> понимать механику — иначе move/copy остаются магией.

### Задание 1. Микро-проект `IntVector` — динамический массив

Интерфейс — `include/int_vector.hpp` (зафиксирован), реализация — `src/int_vector.cpp`.
Внутри храним три поля: указатель `int* data_`, `size_` (сколько элементов) и
`capacity_` (сколько влезет без перевыделения).

Нужно реализовать:
1. `IntVector()` — пустой (без выделения).
2. `explicit IntVector(std::size_t count, int value = 0)` — `count` элементов.
3. `~IntVector()` — освободить память.
4. **Копирующий** конструктор и присваивание (глубокое копирование!).
5. **Перемещающий** конструктор и присваивание (`noexcept`, источник → пустой).
6. `void push_back(int v)` — добавить в конец, при нехватке места — вырасти

7. `std::size_t size() const`, `std::size_t capacity() const`, `bool empty() const`.
8. `int& operator[](std::size_t i)` и `const int& operator[](std::size_t i) const`.

Запусти под санитайзером, чтобы поймать утечки/ошибки памяти (это уже про модуль 12,

#### Подсказки к заданию 1

<details><summary>Рост в push_back</summary>
Если `size_ == capacity_`, выдели новый буфер большего размера (например,
`capacity_ == 0 ? 1 : capacity_ * 2`), скопируй туда старые элементы, удали старый
буфер, перенаправь `data_`. Затем добавь новый элемент и увеличь `size_`.
</details>
<details><summary>Перемещающий конструктор</summary>
Забери у источника `data_`, `size_`, `capacity_`, а у источника выстави их в

</details>
<details><summary>Копирующее присваивание</summary>
Не забудь про самоприсваивание (`v = v;`) и про освобождение СТАРОЙ памяти, прежде
чем выделить новую. Или используй copy-and-swap.
</details>

### Задание 2. `Buffer` — владелец сырой памяти с полным правилом пяти

Файлы — `include/int_vector.hpp` и `src/int_vector.cpp` (класс `Buffer` уже объявлен,
тела — стабы). Это шаг вперёд от `IntVector`: размер фиксирован (нет `push_back`), зато
всё внимание на **корректности пяти методов**, проверке границ и swap-идиоме.

Контракт (его проверяют тесты `Buffer.*`):
1. `explicit Buffer(std::size_t count, double value = 0.0)` — выделить `count` элементов
   и заполнить `value`. При `count == 0` — пустое состояние `(nullptr, 0)`, **без**
   выделения.
2. `~Buffer()` — освободить буфер (`delete[]`; на `nullptr` он безопасен).
3. **Копирующий** конструктор и присваивание — **глубокая** копия: у копии свой буфер
   (`data() != other.data()`), независимое содержимое, переживает `a = a`.
4. **Перемещающие** конструктор и присваивание — забирают буфер у источника (тот же
   адрес «переезжает»), источник → `(nullptr, 0)`. Обязательно `noexcept`.
5. `size()`, `data()` (две перегрузки), `operator[]` (без проверки границ),
   `fill(value)`.
6. `at(i)` — с проверкой границ: при `i >= size()` бросить **`std::out_of_range`**
   (именно его, не `std::logic_error`).
7. `swap(other) noexcept` — обменять внутренности; на нём удобно построить `operator=`.

#### Подсказки к заданию 2

<details><summary>С чего начать</summary>
Сначала конструктор + деструктор + `size()`/`data()`/`operator[]` — этого хватит, чтобы
позеленели простые тесты на заполнение и индексацию. Копирование/перемещение —
следующим шагом.
</details>
<details><summary>at() и std::out_of_range</summary>
Нужен заголовок <code>&lt;stdexcept&gt;</code>. Если <code>i &gt;= size_</code> —
<code>throw std::out_of_range("...")</code>. Иначе верни <code>data_[i]</code>. Тест
ждёт ровно этот тип исключения — не <code>logic_error</code>.
</details>
<details><summary>swap и присваивание через него</summary>
В <code>swap</code> поменяй местами <code>data_</code> и <code>size_</code> (удобно
<code>std::swap</code> из <code>&lt;utility&gt;</code> или
<code>&lt;algorithm&gt;</code>). Тогда copy-and-swap: возьми параметр <strong>по
значению</strong> и сделай <code>swap(*this с параметром)</code> — одна реализация
обслужит и копирование, и перемещение. Хочешь — пиши копир. и перемещ. присваивание
раздельно, но не забудь самоприсваивание и self-move.
</details>
<details><summary>Почему move обязан быть noexcept</summary>
Внутри move ты только переставляешь указатель и число — это не бросает. Поставь
<code>noexcept</code> в объявлении и определении; тест с
<code>is_nothrow_move_constructible</code> это проверяет. Если внутри move ты вдруг
делаешь <code>new[]</code> — это уже не move, а копия: так нельзя.
</details>

### Задание 3. `UniqueHandle` — move-only обёртка над ресурсом

Файлы те же. `UniqueHandle` моделирует **уникальное владение** (как `std::unique_ptr`):
копировать нельзя, перемещать — можно. «Ресурс» — целочисленный `id`, который выдаёт
учебный реестр (`resource_acquire` / `resource_release` / `resource_live_count`). Реестр
**уже реализован** — трогать его не нужно; он считает «живые» ресурсы и падает при
двойном освобождении, так что любой перекос во владении тесты заметят.

Что реализовать (проверяют тесты `UniqueHandle.*`):
1. `UniqueHandle()` — пустой (`id_ == kInvalidHandle`), уже готов — посмотри, как он
   сделан, остальное по образцу.
2. `explicit UniqueHandle(int id)` — взять готовый `id` во владение.
3. `~UniqueHandle()` — если владеем (`id_ != kInvalidHandle`) — освободить через
   `resource_release`.
4. Копирование — **уже `= delete`**, ничего не пиши; `static_assert` в тестах требует,
   чтобы тип был move-only.
5. Move-конструктор / move-присваивание — украсть `id` у источника, источник обнулить;
   в присваивании сперва освободить **своё** и пережить self-move.
6. `get()`, `valid()`, `release()` (отдать наружу, **не** освобождая), `reset(id)`
   (освободить текущее, взять новое), `swap(other)`.
7. `make_handle()` — `resource_acquire()` и обернуть результат в `UniqueHandle`.

#### Подсказки к заданию 3

<details><summary>Инвариант, который надо держать</summary>
За каждый «живой» <code>id</code> отвечает ровно один <code>UniqueHandle</code>.
Захватил — кто-то один обязан освободить; передал владение — у прежнего владельца
<code>id_</code> должен стать <code>kInvalidHandle</code>, иначе он освободит ресурс,
которым уже не владеет.
</details>
<details><summary>Move-конструктор</summary>
Скопируй <code>other.id_</code> к себе, а <code>other.id_</code> выставь в
<code>kInvalidHandle</code>. Тогда деструктор источника ничего не освободит — ресурс
теперь твой, и счётчик «живых» не изменится.
</details>
<details><summary>release vs reset vs деструктор — не перепутай</summary>
<code>release()</code> — отдать <code>id</code> наружу и стать пустым, ресурс
<strong>не</strong> трогать (за него теперь отвечает вызвавший). <code>reset(x)</code> —
<strong>освободить</strong> текущий ресурс и взять <code>x</code>. Деструктор —
освободить, если владеем. Удобно выразить <code>reset</code> и move-assign через уже
написанные кусочки (например, через <code>release()</code> и <code>resource_release</code>).
</details>
<details><summary>Move-присваивание и self-move</summary>
Классический приём: <code>reset(other.release())</code> — забрать чужой <code>id</code>
(обнулив <code>other</code>) и им же заменить своё (освободив прежнее). Это естественно
переживает <code>x = std::move(x)</code>: <code>release()</code> сначала «вынет»
<code>id</code>, и <code>reset</code> вернёт его обратно, ничего не потеряв. Не забудь
<code>return *this;</code>.
</details>
