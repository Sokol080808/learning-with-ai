# Модуль 15 — Категории значений и move-семантика (глубоко)

Большая идея: у каждого выражения в C++ есть не только **тип**, но и **категория значения**.
Именно категория решает, можно ли у объекта «украсть кишки» (переместить) или его придётся
честно скопировать. Перемещение — это не магия и не «быстрое копирование»: это передача
владения ресурсом с обнулением источника. В этом модуле мы разбираем механику до винтиков:
что такое lvalue/prvalue/xvalue, что *физически* делает `std::move`, как работает
`std::forward` и сворачивание ссылок, зачем нужны forwarding-ссылки и perfect forwarding,
когда копия/мув реально происходит (RVO/NRVO), что такое move-only типы и почему
move-конструктор обязан быть `noexcept`.

---

## Идея 1. Категории значений: lvalue, prvalue, xvalue

В C++ есть **тип выражения** и отдельно — его **категория значения**. Со времён C++11
категорий три листовых:

- **lvalue** — выражение, обозначающее объект с *идентичностью*, у которого можно взять
  адрес. Имя переменной `x`, разыменование `*p`, обращение к полю `obj.field`,
  `arr[i]` — всё это lvalue. «l» исторически от «left» (может стоять слева от `=`), но
  точнее думать «locatable» — у него есть адрес/местоположение.
- **prvalue** («pure rvalue») — выражение, которое *вычисляет значение*, но само по себе
  не обозначает долгоживущий объект. Литералы `42`, `3.14`, результат `a + b`,
  вызов функции, возвращающей по значению `make()` — это prvalue. У prvalue нет адреса,
  пока он не «материализуется» во временный объект.
- **xvalue** («eXpiring value») — выражение, обозначающее объект, у которого *есть*
  идентичность, **но** его ресурсы разрешено забрать, потому что он скоро умрёт или нам
  явно сказали, что он больше не нужен. Канонический способ получить xvalue —
  `std::move(x)`. Также xvalue — вызов функции, возвращающей `T&&`.

Две объединяющие категории:

```
        expression
        /        \
   glvalue       rvalue
   /     \       /     \
lvalue   xvalue   prvalue
```

- **glvalue** (generalized lvalue) = lvalue ∪ xvalue — «есть идентичность (адрес)».
- **rvalue** = xvalue ∪ prvalue — «можно перемещать (ресурсы доступны для кражи)».

Почему это важно практически: **перегрузка по rvalue-ссылке `T&&` ловит rvalue
(prvalue и xvalue), а перегрузка по `const T&` ловит всё остальное.** Именно так
компилятор выбирает между копированием и перемещением:

```cpp
void sink(const std::string&);  // (A) копирующая ветка
void sink(std::string&&);       // (B) перемещающая ветка

std::string s = "hi";
sink(s);             // s — lvalue  -> (A)
sink(std::string{}); // prvalue     -> (B)
sink(std::move(s));  // xvalue      -> (B)
```

**Частая ошибка.** «`std::move(x)` — это rvalue, значит у `x` уже нет адреса.» Нет.
`std::move(x)` — это xvalue: у объекта **есть** идентичность (тот же `&x`), просто мы
пометили его как «разрешено перемещать». Сам объект `x` после этого жив и валиден, у него
лишь, возможно, «украли» содержимое.

## Идея 2. Что такое перемещение на самом деле

Перемещение — это **передача владения ресурсом** из одного объекта в другой так, чтобы
источник остался в *валидном, но неопределённом* состоянии (обычно — «пустом»). Никакого
встроенного «move-байткода» в процессоре нет. Всё перемещение — это просто **код в
move-конструкторе/move-присваивании**, который вы (или авторы STL) написали.

Возьмём упрощённый `std::string`, владеющий буфером:

```cpp
class MyString {
    char*  data_ = nullptr;
    size_t size_ = 0;
public:
    // Копирование: выделить НОВЫЙ буфер и продублировать байты — дорого, O(n).
    MyString(const MyString& o) : size_(o.size_) {
        data_ = new char[size_];
        std::copy(o.data_, o.data_ + size_, data_);
    }
    // Перемещение: УКРАСТЬ указатель и обнулить источник — дёшево, O(1).
    MyString(MyString&& o) noexcept : data_(o.data_), size_(o.size_) {
        o.data_ = nullptr;   // <-- без этой строки оба укажут на один буфер
        o.size_ = 0;         //     и при разрушении будет double free!
    }
};
```

Ключевые наблюдения:

1. Перемещение «быстрое» не потому, что компилятор хитрит, а потому что мы скопировали
   лишь указатель (несколько байт) вместо всего буфера.
2. **Обнуление источника обязательно.** Если его не сделать, деструктор источника
   освободит ресурс, которым теперь владеет приёмник — получите double free / use-after-free.
3. Для типов *без* владеющих ресурсов (например `int`, `double`, `std::array<int, 4>`)
   перемещение **идентично** копированию: красть нечего, копируются те же байты. Поэтому
   «move int» не быстрее «copy int» — это миф.

**Частая ошибка.** Думать, что после `auto b = std::move(a);` объект `a` «уничтожен» или
«нельзя трогать». На самом деле `a` валиден; стандарт гарантирует, что у перемещённого
объекта можно вызывать любые операции без предусловий (например, присвоить новое значение
или вызвать `.clear()`). Просто его *содержимое* неопределённо — нельзя полагаться на
конкретное значение.

## Идея 3. `std::move` — это просто приведение типа

`std::move(x)` **ничего не перемещает.** Это `static_cast` к rvalue-ссылке, который меняет
только *категорию значения* выражения (с lvalue на xvalue), чтобы при последующей
инициализации/присваивании выбралась перемещающая перегрузка. Полная реализация — одна
строка:

```cpp
template <class T>
constexpr std::remove_reference_t<T>&& move(T&& x) noexcept {
    return static_cast<std::remove_reference_t<T>&&>(x);
}
```

Разберём по частям:

- Параметр `T&&` — это **forwarding-ссылка** (см. Идею 5): она принимает и lvalue, и rvalue.
- `std::remove_reference_t<T>` снимает с `T` все ссылки, оставляя «голый» тип `U`. Это нужно,
  чтобы собрать именно `U&&`, а не случайно `U& &&`.
- `static_cast<U&&>(x)` превращает выражение в xvalue. Само значение/объект не трогается.

То есть `std::move` — это «обещание»: «я больше не рассчитываю на этот объект, можете забрать
его ресурсы». Само перемещение случится позже — в чьём-то move-конструкторе.

**Частые ошибки:**

```cpp
// (1) move на const — тихо деградирует до копии.
const std::string c = "x";
std::string d = std::move(c);  // тип -> const std::string&&; move-ctor берёт string&&,
                               // const&& туда не подходит -> выбирается copy-ctor!
```
Перемещение из `const`-объекта невозможно (нельзя обнулить const), поэтому компилятор молча
копирует. Не делайте переменные `const`, если собираетесь из них перемещать.

```cpp
// (2) move на return локальной переменной — мешает оптимизатору.
std::string f() {
    std::string s = "...";
    return std::move(s);   // ПЛОХО: отключает NRVO (см. Идею 7), а выигрыша нет
}
std::string g() {
    std::string s = "...";
    return s;              // ХОРОШО: компилятор сам сделает move/elision
}
```

Это **Задание 1**: ты напишешь свой `my_move`. Главное — вернуть аргумент как
`remove_reference_t<T>&&` через `static_cast`, ссылаясь на *тот же* объект.

## Идея 4. Сворачивание ссылок (reference collapsing)

В C++ нельзя написать «ссылку на ссылку» руками (`int& &` — ошибка). Но при **выводе
шаблонов** и через `typedef`/`using` такие комбинации возникают, и язык схлопывает их по
простому правилу:

```
T&  &   ->  T&
T&  &&  ->  T&
T&& &   ->  T&
T&& &&  ->  T&&
```

Мнемоника: **«lvalue побеждает»** — если хоть одна `&` есть, результат `&`; только
`&& && -> &&`. Это и есть фундамент, на котором работают forwarding-ссылки.

Пример, как это «всплывает»:

```cpp
template <class T>
void probe(T&& x);   // если T = int&, то параметр T&& = int& && -> int&
                     // если T = int,  то параметр T&& = int&&

int a = 0;
probe(a);   // a — lvalue, T выводится как int& -> параметр int&  (lvalue-ссылка)
probe(7);   // 7 — rvalue, T выводится как int  -> параметр int&& (rvalue-ссылка)
```

Заметь: вывод `T` *кодирует категорию аргумента в самом T*. Lvalue ⇒ `T = U&`,
rvalue ⇒ `T = U`. Именно это потом считывает `std::forward`.

**Частая ошибка** — путать `auto&&` и `const auto&`. `auto&&` тоже сворачивается:
`auto&& r = expr;` привяжется к чему угодно (lvalue → lvalue-ссылка, rvalue → rvalue-ссылка),
сохранив возможность дальше его форвардить. Это «универсальный» способ захватить выражение
без копии. Реализация трейта `is_lvalue` (**Задание 5**) опирается ровно на это различие
между `U&` и `U&&` на уровне типов.

## Идея 5. Forwarding-ссылки (`T&&`) и `std::forward`

**Forwarding-ссылка** (она же «универсальная ссылка» по Майерсу) — это параметр **ровно**
вида `T&&`, где `T` — выводимый параметр шаблона *этой же* функции (или `auto&&`). Только в
этом контексте `T&&` означает «принимаю и lvalue, и rvalue» (за счёт сворачивания из Идеи 4).

Внимание, это **не** forwarding-ссылка:

```cpp
template <class T> void a(T&& x);             // forwarding-ссылка (T выводится здесь)
template <class T> void b(std::vector<T>&& x);// НЕТ: не голый T&&, а vector<T>&& -> просто rvalue-ссылка
void c(int&& x);                              // НЕТ: int конкретен, ничего не выводится
template <class T> struct S { void d(T&& x); };// НЕТ: T у КЛАССА, в методе уже не выводится
```

Проблема, которую решает `std::forward`: внутри функции параметр `x` — это **именованная
переменная**, а значит само выражение `x` всегда **lvalue**, какой бы аргумент ни пришёл.
Если просто передать `x` дальше, перемещаемость потеряется:

```cpp
template <class T>
void wrapper(T&& x) {
    inner(x);  // ВСЕГДА передаёт как lvalue -> rvalue-аргументы будут копироваться!
}
```

`std::forward<T>(x)` восстанавливает исходную категорию, читая её из `T`:

```cpp
template <class T>
constexpr T&& forward(std::remove_reference_t<T>& x) noexcept {
    return static_cast<T&&>(x);   // T = U&  -> U& && -> U& (lvalue)
}                                 // T = U   -> U&&        (rvalue)
```

- Пришёл lvalue ⇒ `T = U&` ⇒ `static_cast<U& &&>` сворачивается в `U&` ⇒ остаётся lvalue.
- Пришёл rvalue ⇒ `T = U` ⇒ `static_cast<U&&>` ⇒ становится rvalue (xvalue).

**Разница `move` vs `forward` в одной фразе:** `std::move` *безусловно* приводит к rvalue;
`std::forward<T>` приводит к rvalue **только если** исходный аргумент был rvalue. `move` —
«всегда отдать», `forward` — «отдать, если мне отдали».

**Частые ошибки:**

1. Забыть указать `T`: `std::forward(x)` — ошибка, обязательно `std::forward<T>(x)`.
2. Использовать `std::move` там, где нужен `forward`: это превратит чужой lvalue в rvalue и
   может «украсть» объект, которым вызывающий ещё пользуется.
3. Форвардить параметр **дважды**: после первого `forward`/`move` объект, возможно, уже
   опустошён — второй проброс отдаст пустышку.

## Идея 6. Perfect forwarding

**Perfect forwarding** — приём, при котором функция-обёртка передаёт свои аргументы дальше,
*идеально* сохраняя для каждого: тип, `const`/`volatile` и категорию значения (lvalue/rvalue).
Рецепт всегда один и тот же — вариативный шаблон + forwarding-ссылки + `std::forward`:

```cpp
template <class T, class... Args>
T make(Args&&... args) {
    return T(std::forward<Args>(args)...);
}
```

Здесь `Args&&...` — пакет forwarding-ссылок, а `std::forward<Args>(args)...` разворачивается
поэлементно, восстанавливая категорию *каждого* аргумента независимо. Так устроены
`std::make_unique`, `std::make_shared`, `vector::emplace_back`, `std::make_tuple` — все они
конструируют объект «на месте», не создавая лишних временных копий.

Почему это ценно: без perfect forwarding пришлось бы писать экспоненциальное число
перегрузок (для каждого аргумента — `const T&` и `T&&`), либо принимать всё по значению
(лишние копии), либо по `const T&` (нельзя перемещать). Один шаблон с `Args&&...` покрывает
все комбинации сразу и не теряет перемещаемость.

Это **Задание 4** — твой `make_in_place<T>(args...)`. Тест проверит, что rvalue-аргумент
действительно *перемещается* внутрь `T`, а lvalue — *копируется*.

**Частая ошибка** — «полу-perfect» forwarding: объявить `Args&&...`, но передать дальше
просто `args...` без `std::forward`. Тогда всё уедет как lvalue, и для тяжёлых типов получите
ненужные копии.

## Идея 7. Copy elision, RVO и NRVO

Иногда не происходит **ни** копирования, **ни** перемещения — объект конструируется сразу в
финальном месте. Это **copy elision** (исключение копии).

- **RVO (Return Value Optimization)** — возврат **prvalue** (безымянного временного):
  ```cpp
  std::string f() { return std::string("hi"); }  // объект строится прямо в приёмнике
  std::string s = f();                            // нет ни copy, ни move
  ```
  С C++17 это **гарантировано стандартом**: возврат prvalue *обязан* конструироваться на
  месте (copy elision для prvalue — не оптимизация, а правило). Поэтому move/copy-конструктор
  тут даже не обязан существовать.

- **NRVO (Named RVO)** — возврат **именованной** локальной переменной:
  ```cpp
  std::string g() { std::string s = "hi"; return s; }
  ```
  Тут компилятору **разрешено** (но не обязано) сконструировать `s` сразу в приёмнике. NRVO —
  опциональная оптимизация; если она не применилась, `return s;` выполнит **перемещение**
  (компилятор обязан рассматривать локальную переменную в `return` как rvalue), а не копию.

Практический вывод и **частая ошибка**: писать `return std::move(local);`. Это **вредит**:
`std::move(local)` — это xvalue, а NRVO работает только для именованного объекта как такового;
явный `std::move` **запрещает NRVO** и в лучшем случае оставляет один move, в худшем — мешает
оптимизатору. Правило: **просто `return local;`** — компилятор сам выберет elision или move.

## Идея 8. Move-only типы

**Move-only тип** — тип, который можно перемещать, но нельзя копировать. Классика —
`std::unique_ptr`, `std::thread`, `std::ifstream`, `std::future`. Это естественно для
*эксклюзивного владения* ресурсом: два владельца одним сокетом/файлом/потоком — это баг,
поэтому копирование удаляют:

```cpp
class Unique {
public:
    Unique(const Unique&)            = delete;  // нельзя копировать
    Unique& operator=(const Unique&) = delete;
    Unique(Unique&&) noexcept;                  // можно перемещать
    Unique& operator=(Unique&&) noexcept;
};
```

Важные правила:

- Move-only объект всё ещё можно вернуть из функции (`return u;`), положить в `std::vector`,
  передать как аргумент — но только через rvalue (`std::move(u)` для именованных).
- При `move`-присваивании защищайтесь от **самоприсваивания** (`if (this == &other) return *this;`)
  и корректно освобождайте *собственный* старый ресурс перед захватом чужого.

Это **Задание 2** — `ScopedResource`, владеющий целочисленным «хэндлом»: корректные
move-ctor/move-assign, удалённое копирование, пустое состояние после кражи.

## Идея 9. Почему move-конструктор обязан быть `noexcept`

Это не косметика, а вопрос **корректности и производительности контейнеров**. Когда
`std::vector` растёт и переаллоцирует буфер, ему надо перенести существующие элементы в
новую память. У вектора есть гарантия *строгой безопасности исключений*: если при росте
что-то бросит, исходный вектор должен остаться нетронутым.

Дилемма:

- Если move-конструктор элемента **может** бросить, то посреди переноса исключение оставит
  часть элементов уже «украденными» в новом буфере, а часть — в старом. Восстановить
  исходное состояние нельзя (ресурсы уже перемещены). Гарантия нарушена.
- Поэтому `std::vector` использует `std::move_if_noexcept`: он перемещает элементы **только
  если** их move-конструктор `noexcept`. Иначе он **копирует** — медленнее, но копия не
  трогает источник, так что при исключении старый буфер цел.

Вывод: **не-`noexcept` move-конструктор молча превращает ваши «быстрые перемещения» в
медленные копии при росте вектора.** Поэтому правило: если move-операции действительно не
бросают (а у правильно написанных они не бросают — крадут указатели), помечайте их
`noexcept`.

```cpp
std::vector<MyType> v;
// если MyType(MyType&&) noexcept  -> при reserve/push_back элементы ПЕРЕМЕЩАЮТСЯ (O(1) каждый)
// если move может бросать          -> элементы КОПИРУЮТСЯ (O(n) на элемент)
```

**Частая ошибка** — забыть `noexcept` на move-операциях класса с ручным управлением
ресурсами и потом удивляться, почему вектор таких объектов «тормозит». В заданиях 2 и 3
move-операции помечены `noexcept`, и тесты это проверяют через `std::is_nothrow_move_*`.

### Подсказки

<details><summary>Задание 1 — из чего собрать my_move</summary>
Тип возврата уже написан за тебя: <code>std::remove_reference_t&lt;T&gt;&&</code>. Внутри
тела нужен ровно один <code>static_cast</code> к этому же типу от <code>value</code>. Ничего
не конструируй заново — иначе потеряешь тождество объекта (тест сравнивает адреса).
</details>

<details><summary>Задание 1 — почему параметр именно T&&</summary>
Это forwarding-ссылка: она ловит и lvalue, и rvalue. <code>remove_reference_t&lt;T&gt;</code>
снимает ссылку, чтобы получить «голый» <code>U</code>, а <code>&&</code> делает из него
rvalue-ссылку. Сложи их вместе в <code>static_cast</code>.
</details>

<details><summary>Задание 2 — move-конструктор ScopedResource</summary>
Скопируй <code>other.handle_</code> в свой <code>handle_</code>, затем присвой
<code>other.handle_ = kEmpty;</code>. Без второй строки источник сохранит «украденный» хэндл —
именно это проверяет тест <code>MoveConstructorTransfersAndClears</code>.
</details>

<details><summary>Задание 2 — move-присваивание и самоприсваивание</summary>
Сначала <code>if (this == &other) return *this;</code>. Потом забери чужой хэндл и обнули
<code>other</code>. Не забудь вернуть <code>*this</code>. <code>release()</code>: запомни
текущий хэндл во временную, обнули поле в <code>kEmpty</code>, верни временную.
</details>

<details><summary>Задание 3 — где инкременты</summary>
По одному <code>++</code> на операцию: <code>copies_</code> в обоих copy-* , <code>moves_</code>
в обоих move-*. Счётчики статические и общие — поэтому <code>reset()</code> вызывают перед
каждым сценарием. Подумай, почему «передача по значению» даёт ровно <em>одну</em> операцию,
а не две (подсказка: copy elision из Идеи 7 убирает лишнюю при инициализации аргумента из
prvalue, но не при передаче именованного lvalue).
</details>

<details><summary>Задание 4 — один возврат</summary>
Тело — одна строка: <code>return T(std::forward&lt;Args&gt;(args)...);</code>. Многоточие
<code>...</code> разворачивает <code>std::forward</code> по каждому аргументу пакета. Если
забудешь <code>std::forward</code>, rvalue-аргументы поедут как lvalue и скопируются — тест
<code>MovesRvalueArgument</code> это поймает.
</details>

<details><summary>Задание 5 — частичная специализация</summary>
Оставь базовый шаблон наследником <code>std::false_type</code>. Добавь рядом
<code>template &lt;class U&gt; struct is_lvalue&lt;U&&gt; : std::true_type {};</code>. Так
компилятор выберет специализацию только для lvalue-ссылочных типов. Для <code>U&&</code> и
обычных <code>U</code> подойдёт лишь базовый шаблон ⇒ <code>false</code>.
</details>

Когда тесты позеленеют — попробуй объяснить мне, почему `return std::move(local);` это
антипаттерн (Идея 7) и что произойдёт со скоростью `std::vector<ScopedResource>`, если убрать
`noexcept` у его move-конструктора (Идея 9). Это и есть «понимание поверх кода».
