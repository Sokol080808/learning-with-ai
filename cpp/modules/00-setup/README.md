ты нажимаешь «собрать». Без этого всё остальное — магия, а магию мы не любим.

## Идея 0. Минимальная программа: hello world с нуля

Прежде чем разбирать конвейер сборки, посмотрим на простейшую C++-программу целиком.
Это «on-ramp»: если ты пишешь на C++ первый день, здесь всё окажется понятным; если уже
пишешь — проверишь, что детали вросли правильно.

```cpp
#include <iostream>   // (1)

int main() {          // (2)
    std::cout << "Hello, world!\n";   // (3)
    return 0;         // (4)
}
```

Разберём каждую строку механически.

**(1) `#include <iostream>`** — директива препроцессора (подробнее в Идее 4). Она просит
препроцессор буквально вставить содержимое стандартного заголовка `<iostream>` на это
место. `iostream` объявляет `std::cout`, `std::cin`, операторы `<<` и `>>`. Без этой
строки компилятор не знает, что такое `std::cout`, и ты получишь ошибку компиляции
`'cout' was not declared`.

**(2) `int main()`** — точка входа в программу. Когда операционная система запускает
твой бинарник, она вызывает именно эту функцию. Возвращаемый тип `int` — это код
завершения: ОС получает его обратно и может проверить, успешно ли завершилась программа.

**(3) `std::cout << "Hello, world!\n";`** — выводит строку в стандартный поток вывода
(обычно терминал). `std::` — пространство имён стандартной библиотеки; оператор `<<`
«проталкивает» значение в поток. `\n` — символ новой строки.

**(4) `return 0;`** — возвращает операционной системе ноль, что означает «всё в порядке,
завершился без ошибки». Ненулевое значение сигнализирует ОС об ошибке. В C++ функция
`main` — единственная, где `return 0;` можно опустить; компилятор добавит его
автоматически. Но явно писать — хороший тон, особенно пока стандарт не стал второй
натурой.

**Почему `int`, а не `void`?** Потому что стандарт C++ требует именно `int main()`.
`void main()` — это распространённое заблуждение; формально это undefined behaviour
(неопределённое поведение), хотя некоторые компиляторы и прощают. Мы пишем по стандарту.

**Подвох: `std::endl` vs `"\n"`.** Ты встретишь `std::cout << "...\n"` и
`std::cout << "..." << std::endl`. Разница: `std::endl` сбрасывает буфер потока в
дополнение к переносу строки — это медленнее. Для большинства задач достаточно `"\n"`;
`std::endl` нужен, когда важно, чтобы вывод немедленно оказался на экране (или в файле)
до продолжения программы.

## Идея 0а. Флаги компилятора и предупреждения

Когда ты вызываешь компилятор (или CMake вызывает его за тебя), ты можешь передать ему
ключи — **флаги**. Они влияют на то, что именно компилятор проверяет и как оптимизирует
код.

### Стандарт языка

```
-std=c++20
```

Говорит компилятору: «используй C++20». Без этого флага он применит какой-то умолчальный
стандарт (обычно c++14 или c++17 в зависимости от версии компилятора) — часть
современных возможностей просто не будет работать. В этом курсе всегда включён `-std=c++20`.

### Предупреждения (warnings)

```
-Wall -Wextra -Werror
```

- **`-Wall`** — включает большой набор часто полезных предупреждений («All warnings», хотя
  на самом деле не все — название историческое).
- **`-Wextra`** — включает дополнительный набор, который `-Wall` не покрывает: например,
  предупреждение о неиспользуемых параметрах, о подозрительных сравнениях знаковых и
  беззнаковых чисел.
- **`-Werror`** — **превращает предупреждения в ошибки**. Программа перестаёт собираться,
  если компилятор нашёл хоть одно предупреждение.

**Почему предупреждения — первый диагностический инструмент с первого дня?**

Предупреждение — это когда компилятор говорит: «это синтаксически корректно, я
скомпилирую, но поведение здесь, скорее всего, не то, что ты хотел». Классические
примеры:

```cpp
int x;
std::cout << x;  // warning: 'x' is uninitialized — читаем мусор из стека
```

```cpp
unsigned int n = 10;
int i = -1;
if (i < n) { ... }  // warning: signed/unsigned comparison — i воспринимается как большое беззнаковое!
```

Без `-Werror` такие предупреждения легко проигнорировать — и получить баг, который
воспроизводится только на конкретной машине или в конкретной конфигурации. С `-Werror`
ты вынужден разобраться сразу. Профессиональные C++-проекты (Chromium, LLVM, Qt)
всегда собираются с `-Werror`.

**Подвох**: `-Werror` может быть болезненным, когда ты подключаешь чужую библиотеку с
предупреждениями в её заголовках. Решение — заголовки системных/сторонних библиотек
подключают через `<...>`, а не `"..."`, и компилятор применяет к ним другие правила
(подробнее — Идея 0б).

### Уровни оптимизации

| Флаг | Название       | Что делает                                                 | Когда использовать             |
|------|----------------|------------------------------------------------------------|--------------------------------|
| `-O0`| без оптимизации| код «честный»: переменные живут там, где написано; медленно| Debug (умолчание без флага)    |
| `-O1`| минимальная    | простые оптимизации, без агрессивного перемещения кода     | редко                          |
| `-O2`| стандартная    | большинство оптимизаций без увеличения размера бинарника   | Release (продакшн)             |
| `-O3`| агрессивная    | + векторизация, встраивание функций; может увеличить размер| HPC / числодробилки            |
| `-Os`| по размеру     | оптимизирует размер бинарника, а не скорость               | встроенные системы             |

Отдельно стоит **`-g`** — включает отладочную информацию (номера строк, имена переменных)
в бинарник. Нужна, чтобы отладчик `gdb`/`lldb` показывал «строка 42 в warmup.cpp», а не
просто адрес `0x1234abcd`.

## Идея 0б. Угловые `<заголовок>` vs кавычечные `"заголовок"` — где компилятор ищет файлы

При `#include` ты выбираешь одну из двух форм:

```cpp
#include <iostream>       // угловые скобки
#include "warmup.hpp"     // кавычки
```

Разница — в **порядке поиска файла**:

**Кавычки `"..."`** — компилятор сначала ищет файл *относительно текущего исходника*, а
только потом — в системных и указанных директориях. Используется для **твоих** заголовков.

**Угловые `<...>`** — компилятор ищет только в *системных и явно добавленных директориях*
(флагами `-I` или `-isystem`), текущую папку пропускает. Используется для **стандартной
библиотеки** и **сторонних зависимостей** (GoogleTest, Boost и т.д.).

Практические правила:
- `<iostream>`, `<vector>`, `<string>` — стандартная библиотека → угловые.
- `<gtest/gtest.h>` — GoogleTest добавлен через CMake с `-I` → угловые.
- `"warmup.hpp"` — твой заголовок рядом → кавычки.

**Почему это важно для предупреждений?** Компиляторы Clang/GCC относятся к заголовкам
из `<...>` как к «системным» и не выдают на них предупреждения `-Wall/-Wextra`. Именно
поэтому `#include <gtest/gtest.h>` пишется в угловых: заголовки GoogleTest не идеальны с
точки зрения строгих предупреждений, и мы не хотим, чтобы они захламляли диагностику
нашего кода.

**Подвох: если написать `#include "iostream"`** — компилятор поищет файл `iostream` в
папке рядом с исходником, не найдёт, и выдаст ошибку. Это редкая опечатка, но
классическая для начинающих.

## Идея 0в. Debug-сборка и Release-сборка

Одни и те же исходники можно собрать **по-разному**, и получить разные бинарники:

| Параметр              | Debug (`-g -O0`)                        | Release (`-O2 -DNDEBUG`)                     |
|-----------------------|-----------------------------------------|----------------------------------------------|
| Оптимизация           | нет (`-O0`)                             | есть (`-O2`)                                 |
| Отладочная информация | есть (`-g`)                             | обычно нет (или минимальная)                 |
| Скорость выполнения   | медленнее (10–100× для числодробилок)   | быстрее                                      |
| Размер бинарника      | больше                                  | меньше                                       |
| `assert()`            | работает                                | отключён (`NDEBUG` отключает `assert`)       |
| Читаемость в отладчике| строки, имена переменных                | оптимизатор «переставил» код, сложнее        |

**Зачем разделение?**

*Debug* — для разработки: отладчик видит все переменные, `assert()` ловит инварианты,
компиляция быстрая. Ты находишь баги здесь.

*Release* — для продакшна или финального измерения скорости. Оптимизатор может встроить
(inline) функции, переставить инструкции, убрать ветки — код летит, но отладка почти
невозможна.

**Классический баг**: программа работает правильно в Debug и падает в Release (или
наоборот). Чаще всего виновато **неопределённое поведение** (undefined behaviour): в
Debug оно «случайно» не проявляется из-за отсутствия оптимизаций, а оптимизатор в Release
переставляет код под предположением, что UB не происходит, — и логика ломается. Это одна
из причин, почему UB в C++ опасен.

В CMake тип сборки задаётся при конфигурации:
```bash
cmake -DCMAKE_BUILD_TYPE=Debug   ...   # флаги: -g -O0
cmake -DCMAKE_BUILD_TYPE=Release ...   # флаги: -O2 -DNDEBUG
```

В этом курсе по умолчанию используется Debug — чтобы тесты и отладчик работали
предсказуемо. Когда начнёшь изучать производительность (модуль 21), переключимся на
Release.

## Идея 1. Что значит «скомпилировать программу»

C++ — *компилируемый* язык. Твой `.cpp` не выполняется напрямую, как Python. Он
проходит конвейер:

1. **Препроцессор** — обрабатывает строки с `#` (`#include`, `#define`). `#include`
   буквально вставляет содержимое заголовочного файла в твой файл. Результат — один
   большой «единичный перевод» (translation unit).
2. **Компиляция** — превращает каждый `.cpp` в объектный файл (`.o`) — машинный код,
   но ещё с «дырами» там, где вызываются функции из других файлов.
3. **Линковка (компоновка)** — сшивает объектные файлы и библиотеки в одну программу,
   подставляя реальные адреса функций.

Почему это важно понимать: **ошибки компиляции и ошибки линковки — это разные вещи**.
«undefined reference to `foo`» — это не «не нашёл, что писать», а «я знаю, что есть
функция `foo`, но никто её не *реализовал*» (этап 3). А «`foo` was not declared» —
это этап 2: про функцию вообще не сказали. Ты будешь встречать обе; умение их
различать экономит часы.

## Идея 2. Зачем CMake

Вызывать компилятор руками (`g++ a.cpp b.cpp -o app`) для большого проекта невозможно.
**CMake** — это система, которая по описанию (`CMakeLists.txt`) генерирует команды
сборки. Ты говоришь «вот таргет, вот его файлы, слинкуй с такой-то библиотекой» — она
разбирается с деталями. В этом курсе CMake уже настроен, тебе его писать не нужно
(до модуля 11, где мы разберём его специально).

## Идея 3. Зачем тесты и что такое GoogleTest

**Тест** — это маленькая программа, которая вызывает твой код и проверяет, что он
ведёт себя как надо. **GoogleTest** — популярный фреймворк для этого. Выглядит так:

```cpp
TEST(ИмяГруппы, ИмяСлучая) {
    EXPECT_EQ(add(2, 3), 5);   // ожидаю, что add(2,3) вернёт 5
}
```

`EXPECT_EQ(a, b)` не прерывает тест при провале (проверит и пойдёт дальше),
`ASSERT_EQ(a, b)` — прерывает. Если проверка не прошла — тест «красный», и GoogleTest
печатает, *что* ожидалось и *что* получилось. Это твоя обратная связь.

## Идея 4. Конвейер сборки в деталях: препроцессор → компиляция → линковка

В Идее 1 мы назвали три стадии. Теперь медленнее, потому что от этого зависит, как ты
будешь читать ошибки всю оставшуюся жизнь в C++.

**Стадия 1 — препроцессор.** Чисто текстовая операция. Он не понимает C++, он просто
работает со строками, начинающимися с `#`:

```cpp
#include "warmup.hpp"   // → СЮДА буквально вставляется текст warmup.hpp
#define PI 3.14         // → дальше каждое слово PI заменяется на 3.14
```

После препроцессора `#include` исчезает, а на его месте — полное содержимое заголовка.
Хочешь увидеть результат своими глазами? `g++ -E warmup.cpp` напечатает файл *после*
препроцессора — там будет много чужого кода из подключённых заголовков.

**Стадия 2 — компиляция.** Компилятор берёт результат препроцессора и превращает его в
объектный файл (`.o`). Здесь ловятся ошибки *синтаксиса и типов*: «`;` забыл», «`foo` was
not declared», «нельзя сложить `int` и `std::string`». На этой стадии компилятор смотрит
на ОДИН файл и ничего не знает про другие.

**Стадия 3 — линковка.** Линковщик сшивает все `.o`-файлы и библиотеки в одну программу.
Когда в одном файле ты *вызвал* `abs_diff`, а *определена* она в другом, именно линковщик
соединяет вызов с телом. Если тела нигде нет — `undefined reference to abs_diff`. Если
тело есть в двух местах сразу — `multiple definition`.

Запомни таблицу — она экономит часы:

| Сообщение                          | Стадия      | Что случилось                              |
|------------------------------------|-------------|--------------------------------------------|
| `'foo' was not declared`           | компиляция  | не показали объявление (нет `#include`?)   |
| `expected ';'`, `invalid operands` | компиляция  | синтаксис / типы                           |
| `undefined reference to 'foo'`     | линковка    | объявили, но нигде не *определили*         |
| `multiple definition of 'foo'`     | линковка    | одно и то же определение попало дважды      |

## Идея 5. Единица трансляции (translation unit)

**Единица трансляции** — это один `.cpp` *вместе со всем*, что в него вставил `#include`.
Компилятор обрабатывает её целиком и независимо от других. Твой проект из трёх `.cpp` —
это три отдельные единицы трансляции, три отдельных `.o`-файла. Они «не видят» друг друга
во время компиляции и встречаются только на линковке.

Отсюда важное следствие: чтобы из `test_warmup.cpp` вызвать `add`, компилятору достаточно
*объявления* (`int add(int, int);`) — он поверит на слово, что функция где-то есть, и
оставит «дыру». А заполнит эту дыру линковщик, найдя *определение* в `warmup.o`. Поэтому
**объявления** живут в заголовках (их подключают многие), а **определения** обычных
функций — в одном `.cpp`.

## Идея 6. ODR (One Definition Rule) простыми словами

**Правило одного определения**: у каждой функции и каждой переменной во всей программе
может быть сколько угодно *объявлений*, но ровно **одно определение**.

- *Объявление* — «такая штука существует» (`int add(int, int);`). Можно повторять.
- *Определение* — «вот её тело / вот сама переменная». Должно быть одно на программу.

Если линковщик найдёт два определения `add` — это нарушение ODR, ошибка
`multiple definition`. Если ноль — `undefined reference`. ODR — это причина, по которой
обычные функции не пишут с телом прямо в заголовке: заголовок включают много `.cpp`, и
каждый получил бы своё определение → дубликаты → линковка падает.

Исключение из этого правила — слово `inline` (см. Идею 8): оно специально разрешает
одинаковое определение в нескольких единицах трансляции.

## Идея 7. Зачем include guards / `#pragma once`

Заголовки часто включают друг друга, и один и тот же `.hpp` легко попадает в файл дважды
(например, `a.hpp` включил `common.hpp`, и `b.hpp` тоже, а твой `.cpp` включил оба). Без
защиты препроцессор вставит содержимое `common.hpp` дважды — и если там есть, скажем,
определение структуры, получишь `redefinition of 'struct ...'` *внутри одной* единицы
трансляции.

Защита — «страж включения». Классический вид:

```cpp
#ifndef WARMUP_HPP   // если макрос ещё не определён —
#define WARMUP_HPP   // определяем его и идём дальше
// ... содержимое заголовка ...
#endif               // при втором включении #ifndef ложен → тело пропускается
```

Современный короткий аналог — одна строка в самом верху файла:

```cpp
#pragma once
```

Он делает то же самое (включить файл не больше раза в единицу трансляции), но без ручного
макроса. Именно поэтому он стоит первой строкой в `warmup.hpp`. `#pragma once` не входит в
стандарт формально, но поддерживается всеми реальными компиляторами — в этом курсе мы
пользуемся им.

## Идея 8. `inline`: определение в заголовке без нарушения ODR

Иногда удобно положить *тело* маленькой функции прямо в заголовок (чтобы не заводить
отдельный `.cpp` и чтобы компилятор мог встроить вызов). Но обычное определение в
заголовке = дубликаты в каждой единице трансляции = нарушение ODR. Слово `inline` снимает
этот запрет:

```cpp
// в заголовке:
inline int square(int x) { return x * x; }   // ок включать в любое число .cpp
```

`inline` говорит линковщику: «таких одинаковых определений может быть много — это
нормально, выбери любое». Важно: исторически `inline` намекал на «встроить вызов», но
сегодня это в первую очередь именно *разрешение на повторное определение*; решение о
реальной подстановке принимает компилятор сам. Шаблоны (модуль 07) живут в заголовках по
той же причине — фактически они тоже `inline`.

## Задание (разминка)

### Подсказки к заданиям 2–3

<details><summary>Задание 2 — что такое «модуль числа»</summary>

Модуль (абсолютная величина) числа — это «расстояние до нуля», без знака: `|−5| = 5`,
`|5| = 5`. Тебе нужно, чтобы результат `abs_diff` никогда не был отрицательным,
независимо от порядка `a` и `b`.
</details>

<details><summary>Задание 2 — как получить неотрицательное</summary>

Сначала посчитай разность, а потом, если она получилась отрицательной, поверни ей знак.
Подумай про `if`. (Есть и готовая `std::abs` из `<cstdlib>` — но сначала попробуй руками,
чтобы понять механику.)
</details>

<details><summary>Задание 3 — как проверить чётность</summary>

Число чётное, если делится на 2 без остатка. Остаток от деления в C++ даёт оператор `%`.
Какое значение `x % 2` у чётного числа? Сравни с ним.
</details>
