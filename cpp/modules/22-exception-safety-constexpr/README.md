# Модуль 22 — Гарантии исключений и вычисления на этапе компиляции

Две большие идеи в одном модуле. Первая: **что обещает твой код, если что-то
пошло не так** — даёшь ли ты гарантию, что объект останется в осмысленном
состоянии при исключении. Вторая: **что можно посчитать ещё до запуска
программы** — `constexpr` переносит вычисления в компилятор. Обе темы про
надёжность: одна защищает в рантайме, вторая ловит ошибки до него.

## Идея 1. Три уровня гарантий исключений

Когда операция может бросить, у неё есть «обещание» о состоянии объекта:

- **Базовая гарантия:** при броске нет утечек и объект остаётся *валидным*
  (по нему можно вызывать методы), но содержимое может измениться.
- **Строгая гарантия:** при броске состояние *как будто операция не вызывалась*
  — всё откатывается к исходному (транзакционность, commit-or-rollback).
- **Без исключений (nothrow):** операция *не бросает вообще*, помечается `noexcept`.

```cpp
void push(const T& v) noexcept;   // обещание: никогда не бросит
```

Зачем различать. Контейнеры STL дают строгую гарантию там, где могут: например,
`vector::push_back` либо добавит элемент, либо оставит вектор прежним. Это
позволяет писать код в стиле «попробовал — не вышло — состояние цело».

Подводный камень: `noexcept` — это *обещание под подписью*. Если `noexcept`-функция
всё же бросит, программа немедленно завершится через `std::terminate`. Ставь
`noexcept` только когда уверен.

## Идея 2. Как добиться строгой гарантии: «сначала подготовь, потом фиксируй»

Опасные шаги (выделение памяти, копирование, которое может бросить) делай на
*стороне*, и лишь когда всё прошло — одним безопасным движением «вклей» результат.

```cpp
void push(const T& v) {
    if (size_ == data_.capacity()) {
        data_.reserve(grow(data_.capacity()));  // выделение может бросить —
    }                                            // но стек ещё не тронут
    data_.push_back(v);  // места уже хватает; добавление не реаллоцирует
}
```

Идея: реаллокация — единственное место, где `push` мог бы испортить состояние.
Вынеси её *перед* изменением, и если она бросит, до изменения дело не дойдёт.

Подводный камень: если сначала увеличить `size_`, а потом копировать `v` (и
копия бросит) — счётчик уже соврал. Меняй наблюдаемое состояние **последним**.

## Идея 3. Идиома copy-and-swap

Как написать `operator=`, который и про копию, и про перемещение, и переживает
самоприсваивание — одной короткой функцией? Принимай аргумент **по значению** и
обменяйся с ним.

```cpp
class Buffer {
public:
    Buffer& operator=(Buffer other) noexcept {  // other — уже готовая копия
        swap(*this, other);                     // забрать её ресурс себе
        return *this;                           // старый ресурс уедет в other
    }                                           // и удалится в её деструкторе
    friend void swap(Buffer& a, Buffer& b) noexcept {
        using std::swap;
        swap(a.size_, b.size_);
        swap(a.data_, b.data_);
    }
};
```

Почему это красиво: всё опасное (выделение памяти под копию) случается *до* входа
в тело `operator=` — при копировании аргумента. Если оно бросит, `*this` ещё не
тронут (строгая гарантия бесплатно). А `swap` из указателей не бросает.

Подводный камень: `swap` должен обменивать именно *владеющие* поля (указатель и
размер), и быть `noexcept`. Не пиши `*this = ...` внутри swap — будет рекурсия.

## Идея 4. ScopeGuard — откат через деструктор

RAII работает не только для памяти. Заведи объект, который в деструкторе выполнит
«уборку», и метод `dismiss()`, который её отменяет, если всё прошло хорошо.

```cpp
ScopeGuard g([&]{ rollback(); });   // если выйдем отсюда раньше — откатимся
do_risky_step();
g.dismiss();                        // дошли до конца — отмена отката
```

При нормальном выходе ты зовёшь `dismiss()`. При раннем выходе или исключении
`dismiss()` не вызовется, и деструктор сделает уборку. Это ручная «строгая
гарантия» для кода, который иначе пришлось бы оборачивать в `try/catch`.

Подводный камень: деструктор guard'а **обязан** быть `noexcept` и не выпускать
исключений наружу. Деструктор, бросающий во время раскрутки стека от другого
исключения, валит программу.

## Идея 5a. Условный `noexcept(expr)` и оператор `noexcept`

До сих пор мы писали `noexcept` как ярлык: «функция никогда не бросает». Но есть
вторая форма — **условный noexcept**:

```cpp
template <class T>
void my_swap(T& a, T& b) noexcept(noexcept(T(std::move(a))));
```

Здесь два `noexcept`:
- **внешний** — спецификатор, говорит «эта функция noexcept, если…»
- **внутренний** — *оператор*: вычисляется в compile-time и возвращает `bool`.
  `noexcept(выражение)` не выполняет само выражение, только проверяет, помечено ли
  оно как noexcept. Никакого кода не генерируется.

Зачем это нужно: шаблонная функция сама должна давать `noexcept`-обещание только
тогда, когда операции над `T` не бросают. Иначе получился бы ложный `noexcept`
(программа упадёт в `std::terminate`) или вечная пессимизация (строгая гарантия
для `vector` потеряна).

**Стандартный шаблон для swap:**

```cpp
template <class T>
void my_swap(T& a, T& b)
    noexcept(std::is_nothrow_move_constructible_v<T> &&
             std::is_nothrow_move_assignable_v<T>)
{
    T tmp(std::move(a));
    a = std::move(b);
    b = std::move(tmp);
}
```

Точно так же устроен `std::swap` начиная с C++11.

**Неявный noexcept у деструктора и дефолтных move-операций.** Компилятор ставит
`noexcept` автоматически:
- Деструктор — всегда `noexcept` (если не объявить иначе явно).
- Дефолтный move-конструктор и move-assignment — `noexcept`, если все поля-члены
  move-операции `noexcept`.

Это важно: объявив `Foo(Foo&&) noexcept = default;` (явная пометка), мы разрешаем
`std::vector` использовать move вместо копии при перераспределении, что даёт O(n)
→ O(1) реаллокацию.

**Подвох:** `noexcept` на swap или move — это обещание. Если в теле есть что-то
бросающее, а ты пометил функцию `noexcept`, компилятор не предупредит; но при броске
сработает `std::terminate`. Всегда проверяй тело перед простановкой спецификатора.

## Идея 5b. Move-семантика и гарантии: `std::move_if_noexcept`

Когда `vector` исчерпывает ёмкость, он выделяет новый блок и **переносит** туда
элементы. Вопрос: перемещать или копировать?

- **Если move — `noexcept`**: перемещение безопасно. Если что-то пойдёт не так
  *до* начала переноса, мы ещё ничего не тронули. Перенесённые элементы уже «там»,
  непоредённые — «здесь»; промежуточного состояния нет.
- **Если move — throwing**: при броске посередине перемещения часть старого буфера
  уже обнулена, часть нового — не заполнена. Строгой гарантии нет. Поэтому `vector`
  переключается на **копирование**: копии независимы, старый буфер цел до финального
  `delete[]`.

Именно это делает `std::move_if_noexcept`:

```cpp
T tmp = std::move_if_noexcept(val);
// Если T — noexcept-move: вернёт T&&  (перемещение)
// Иначе:                   вернёт const T& (копирование)
```

Следствие: если ты пишешь `T` для хранения в контейнерах — добавь `noexcept` к
move-конструктору. Это не микрооптимизация, а условие строгой гарантии `push_back`.

**Компромисс.** noexcept-move вынуждает «опустошить» источник. Иногда это нежелательно
(например, если инвариант класса требует, чтобы объект всегда был в валидном
нетривиальном состоянии). Решение — либо убедиться, что «пустое» состояние валидно,
либо смириться с копированием при реаллокации.

## Идея 5. Почему деструкторы по умолчанию `noexcept`

В C++ деструкторы неявно `noexcept`. Причина — раскрутка стека: когда летит
исключение, для каждого объекта на пути вызывается деструктор. Если *он* тоже
бросит, окажется два исключения сразу — язык на это отвечает `std::terminate`.
Поэтому правило: **из деструктора не бросают**. Чисти ресурсы молча.

## Идея 6. `constexpr` и `consteval` — считаем в компиляторе

`constexpr`-функцию компилятор умеет выполнить *на этапе компиляции*, если
аргументы известны там же. Тогда в готовой программе вместо вызова стоит уже
посчитанная константа.

```cpp
constexpr unsigned long long factorial(unsigned n) {
    unsigned long long r = 1;
    for (unsigned i = 2; i <= n; ++i) r *= i;
    return r;
}
constexpr auto F = factorial(10);   // посчитано компилятором: 3628800
static_assert(F == 3628800);        // проверка тоже в compile-time
```

`constexpr` означает «**можно** в compile-time» (а можно и в рантайме — если
аргумент не константа). Если хочешь «**только** в compile-time» — это `consteval`:

```cpp
consteval int square(int x) { return x * x; }
constexpr int A = square(4);  // ок
// int b = square(rand());    // ошибка: consteval требует константу
```

Подводный камень: не всё годится в `constexpr`-контекст — нельзя выделять память
через `new` так, чтобы она «пережила» компиляцию, нельзя бросать пойманные
исключения и т. п. Держи `constexpr`-функции чистыми и без побочных эффектов.

## Идея 6a. `constinit` — инициализация без сюрпризов при старте

**Static Initialization Order Fiasco (SIOF)** — классическая проблема C++: порядок
инициализации статических объектов *в разных единицах трансляции* не определён
стандартом. Если глобальный объект `A` зависит от глобального `B` из другого .cpp,
`B` может оказаться ещё не инициализированным.

```cpp
// a.cpp
extern int counter;                    // определён в b.cpp
int shifted = counter + 10;            // UB: counter ещё 0 или мусор?
```

`constinit` (C++20) говорит компилятору: «эта переменная *обязана* инициализироваться
константным выражением прямо при старте программы — никакой динамической
инициализации». Если это невозможно — ошибка компиляции.

```cpp
constinit int limit = 100;             // ок: 100 — константное выражение
constinit int bad  = rand();           // ошибка: rand() не constexpr
```

**Отличие от `const` и `constexpr`:**

| Спецификатор | Иммутабельность | Гарантия const-init | Известно в compile-time |
|---|---|---|---|
| `const`      | да              | нет                 | нет (может быть динамика) |
| `constexpr`  | да              | да                  | да                       |
| `constinit`  | **нет**         | да                  | нет (значение может читаться в рантайме) |

`constinit` нужен, когда переменная должна меняться в рантайме (счётчик, флаг),
но мы хотим гарантию, что начальное значение точно вписано до `main()` без
зависимостей от порядка инициализации.

**Подвох:** `constinit` на non-`const` переменной не защищает от гонок данных —
это не `atomic`. Для глобального изменяемого состояния нужны ещё `atomic` или мьютекс.

## Идея 6b. Что нельзя делать в `constexpr` (конкретный список)

`constexpr`-функция выполняется компилятором — значит, в ней нет рантайм-механизмов.
Конкретный список запретов (C++20):

| Запрещено | Причина |
|---|---|
| `goto`                              | не поддерживается в constexpr-машине компилятора |
| `static` и `thread_local` переменные | состояние между вызовами невозможно в compile-time |
| `reinterpret_cast`                  | нарушает строгое aliasing-правило; компилятор не может проверить корректность |
| Неопределённое поведение (UB)       | UB в constexpr — ошибка компиляции (а не тихий баг!) |
| `asm`-вставки                       | не portable и непрозрачны для компилятора |
| `new`/`delete`, если объект «утекает» из constexpr | выделенная в compile-time память обязана быть освобождена там же |
| Виртуальный вызов на non-`constexpr` vtable-метод | динамическая диспетчеризация не разрешена (C++17); в C++20 — частично разрешена |
| `try`/`throw` внутри constexpr (C++17) | в C++20 `try`/`catch` *разрешены* (ветка с броском запрещена в const-eval, но синтаксис допустим) |

**Хорошая новость:** многое, что казалось запрещённым в C++14, разрешено в C++17/20.
`std::vector`, `std::string`, `std::optional` теперь `constexpr`-совместимы. Локальные
переменные, циклы, `if`, лямбды, виртуальные вызовы на `constexpr`-методах — всё это
работает в C++20.

**Практический совет:** если `constexpr`-функция ведёт себя неожиданно, попробуй
присвоить результат `constexpr`-переменной — компилятор сразу скажет, что именно
нарушено (UB вылезает как ошибка, а не тихо в рантайме).

## Идея 7. `static_assert` — проверка фактов на этапе компиляции

`static_assert(условие, "сообщение")` срабатывает в компиляторе: если условие
ложно, проект **не соберётся**, и ты увидишь сообщение. Это бесплатные тесты,
которые невозможно «забыть запустить».

```cpp
static_assert(sizeof(int) >= 4, "нужен int хотя бы 4 байта");
static_assert(factorial(5) == 120, "проверка формулы");
```

В тестах этого модуля для `factorial`/`gcd`/`is_prime`/`cstr_len`/`reversed`
заготовлены блоки `static_assert` — они **закомментированы**, чтобы стабы
собирались. Когда реализуешь функции, раскомментируй блоки: пройдёт сборка —
значит, всё считается верно ещё до запуска тестов.

## Идея 8. Массивы и строки в compile-time

`constexpr`-функция может принимать и возвращать агрегаты (например, обёртку над
`T[N]`) и обрабатывать их в цикле — всё посчитается компилятором.

```cpp
template <class T, std::size_t N>
struct Array { T data[N]{}; /* constexpr operator[] и т. п. */ };

template <class T, std::size_t N>
constexpr Array<T, N> reversed(const Array<T, N>& in) {
    Array<T, N> out{};
    for (std::size_t i = 0; i < N; ++i) out[i] = in[N - 1 - i];
    return out;
}
```

Длину C-строки тоже можно посчитать в компиляторе — просто пройди до `'\0'`:

```cpp
constexpr std::size_t cstr_len(const char* s) {
    std::size_t n = 0;
    while (s[n] != '\0') ++n;
    return n;
}
```

Подводный камень: размер `N` — часть *типа*, он известен в compile-time, поэтому
циклы по `N` спокойно разворачиваются компилятором. А вот длину C-строки нужно
честно досчитать до нуль-терминатора — она в типе не записана.

## Идея 9. Базовая vs строгая гарантия: когда строгая слишком дорога

Строгая гарантия — это «транзакция»: либо всё применено, либо ничего не изменилось.
Звучит красиво, но у неё есть цена.

**Классический пример: `push_back` для вектора своих объектов.**

Наивная строгая гарантия требует: скопировать все N элементов в новый буфер; только
если копирование прошло успешно — поменять указатель на буфер. Это **O(n)** работы
при каждой реаллокации.

Но если у элемента `T` есть **noexcept-move**, компилятор (через `std::move_if_noexcept`)
может переместить элементы. Перемещение дешевле копирования (O(1) на элемент), и
строгая гарантия сохраняется — потому что в момент переноса нет возможности броска.

**Когда строгая гарантия принципиально дорогая:**

```cpp
// Допустим, у нас есть операция append двух больших структур.
// Строгая гарантия требует: сделать полную копию состояния ЗАРАНЕЕ,
// выполнить append на копии, только потом swap.
void strong_append(BigState& dst, const BigState& src) {
    BigState tmp(dst);    // O(n) копия — это и есть цена
    tmp.do_append(src);   // может бросить — tmp умрёт, dst цел
    swap(dst, tmp);       // noexcept-swap
}
```

Если `BigState` содержит миллионы элементов, O(n)-копия — неприемлемо дорого.
**Что делают вместо:**

1. **Давать только базовую гарантию** — при броске нет утечек, объект валиден,
   но содержимое «частично применено». Документируют это явно.
2. **Откатывать вручную** — вести лог изменений и откатывать по записям. Сложно
   и чревато новыми ошибками.
3. **Проектировать типы данных так, чтобы операции были атомарными** — например,
   хранить `std::shared_ptr` на неизменяемый блок данных; «append» создаёт новый
   блок и атомарно меняет указатель (persistent data structures).

**Практический вывод:** строгая гарантия — стандарт для операций с **одним**
объектом небольшого размера (push, insert, assign). Для массовых операций над
большими структурами взвешивай стоимость. STL явно документирует уровень гарантии
для каждой операции — читай раздел *Exception safety* в cppreference.

   `is_prime`. Когда заработают — раскомментируй блок `static_assert` в тестах:
   он проверит их в compile-time.

5. **`cstr_len` + `reversed`.** Реализуй `cstr_len` (длина C-строки до `'\0'`) и

6. **`my_swap` с корректным propagation `noexcept`.** Напиши обобщённый шаблон
   `my_swap(T& a, T& b)`, который обменивает два значения через move и **правильно
   прокидывает условный `noexcept`**: функция помечается `noexcept`, только если
   `T` — noexcept-move-constructible и noexcept-move-assignable.

   Зачем: обобщённые контейнеры и алгоритмы проверяют `noexcept`-ность через
   `noexcept(my_swap(a, b))`, и ложный `noexcept` — или его отсутствие на noexcept-типах
   — либо скрывает баги, либо мешает оптимизациям (`vector` перейдёт с move на copy).

   Проверь в тестах:
   - `static_assert(noexcept(my_swap(a, b)))` для типа с `noexcept`-move.
   - `static_assert(!noexcept(my_swap(c, d)))` для типа с throwing-move.
   - Корректность обмена на конкретных значениях.

### Подсказки

<details><summary>Задание 1 — где именно «фиксировать» изменение</summary>
Сначала убедись, что места хватает (при необходимости `reserve` с запасом —
например, удвоить ёмкость). Реаллокация — единственный шаг, который может бросить
и оставить мусор; сделай её до того, как меняешь видимый размер. После того как
место гарантированно есть, добавление элемента уже не реаллоцирует.
</details>

<details><summary>Задание 1 — почему тест с «миной» (Bomb) ловит ошибку</summary>
`Bomb` бросает при копировании. Если твой `push` копирует элемент *после* того,
как уже увеличил размер или сдвинул состояние, тест увидит, что `size()` или
`top()` изменились, хотя `push` «не удался». Меняй наблюдаемое состояние последним
шагом, который заведомо не бросает.
</details>

<details><summary>Задание 2 — что хранить и что проверять</summary>
Тебе нужен флаг «активен ли guard». `dismiss()` опускает флаг. Деструктор: если
флаг ещё поднят и действие непустое — вызови действие. Подумай, нужно ли
оборачивать вызов, чтобы из деструктора точно ничего не вылетело.
</details>

<details><summary>Задание 3 — тело operator= в одну строку</summary>
Аргумент `Buffer other` приходит уже скопированным (или перемещённым) — вся
тяжёлая работа сделана за тебя. Остаётся обменяться с ним внутренностями через
`swap(*this, other)` и вернуть `*this`. Старый ресурс уедет в `other` и удалится.
</details>

<details><summary>Задание 3 — что должен делать swap</summary>
Обменять ровно поля-владельцы: указатель на массив и размер. Используй
`using std::swap;` и `swap(a.field, b.field)`. Он обязан быть `noexcept` — обмен
указателей и чисел не бросает.
</details>

<details><summary>Задание 4 — gcd по Евклиду</summary>
`gcd(a, 0) == a`; иначе `gcd(a, b) == gcd(b, a % b)`. Можно циклом: пока `b != 0`,
делай `(a, b) = (b, a % b)`. Для `is_prime`: 0 и 1 не простые; дальше проверяй
делители — достаточно до корня из `n` (то есть пока `d*d <= n`).
</details>

<details><summary>Задание 5 — почему обычный цикл годится в constexpr</summary>
В `constexpr`-функции можно использовать локальные переменные, циклы и `if` — это
разрешено с C++14. Для `reversed` заведи `Array<T,N> out{};` и заполни его:
`out[i] = in[N-1-i]`. Для `cstr_len` иди по `s[n]`, пока не встретишь `'\0'`.
</details>

<details><summary>Задание 6 — как записать условный noexcept</summary>
Сигнатура: `template &lt;class T&gt; void my_swap(T& a, T& b) noexcept(УСЛОВИЕ)`.
Условие — булево константное выражение; стандартный путь:
`std::is_nothrow_move_constructible_v<T> && std::is_nothrow_move_assignable_v<T>`.
Тело — три строки с `std::move`. Заметь: `noexcept(noexcept(T(std::move(a))))` —
это тоже рабочий синтаксис (оператор `noexcept` изнутри), но трейты читабельнее.
Для типа с throwing-move в тестах удобно сделать структуру с явным
`T(T&&) noexcept(false)` — тогда `static_assert(!noexcept(...))` сработает.
</details>
