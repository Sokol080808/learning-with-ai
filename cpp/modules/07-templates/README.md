# Модуль 07 — Шаблоны и concepts (C++20)

Большая идея: **обобщённое программирование** — писать алгоритм один раз для *любого
подходящего типа*, не теряя при этом проверок на этапе компиляции. Шаблоны — это то, на
чём построена вся STL.

## Идея 1. Шаблон функции

```cpp
template <typename T>
T my_max(const T& a, const T& b) {
    return (a < b) ? b : a;
}

my_max(3, 5);        // T = int
my_max(2.5, 1.5);    // T = double
my_max(std::string{"a"}, std::string{"b"});  // T = std::string
```
Компилятор по аргументам **выводит** `T` и генерирует отдельную версию функции для
каждого использованного типа (инстанцирование). Требование к `T` здесь неявное: для
него должен работать `operator<`.

## Идея 2. Почему шаблоны живут в заголовках

Чтобы сгенерировать код под конкретный `T`, компилятору нужно **видеть тело** шаблона в
точке использования. Поэтому реализацию шаблона кладут в `.hpp`, а не в `.cpp`. Именно
поэтому задания этого модуля ты пишешь прямо в заголовке `include/task07.hpp`.

## Идея 3. Шаблон класса

```cpp
template <typename First, typename Second>
struct Pair {
    First  first;
    Second second;
};
Pair<int, std::string> p{1, "one"};
```
`std::vector<T>`, `std::map<K,V>` — это шаблоны классов. Ты ими уже почти пользуешься.

## Идея 4. Concepts — ограничения на типы (C++20)

Раньше, если подставить «неподходящий» тип, ошибки были чудовищно длинными. **Concepts**
позволяют сказать, *каким* должен быть тип, и дают понятные сообщения:

```cpp
#include <concepts>

template <std::integral T>      // T обязан быть целочисленным
T half(T x) { return x / 2; }

half(10);     // ок
half(3.14);   // ошибка компиляции — и понятная: 3.14 не integral
```
Это можно писать и так:
```cpp
template <typename T>
    requires std::integral<T>
T half(T x) { return x / 2; }
```
Готовые концепты: `std::integral`, `std::floating_point`, `std::same_as<A,B>`,
`std::convertible_to<…>` и др. (в `<concepts>`).

## Идея 5. `auto` как обобщение «на лету»

В C++20 даже параметры функции можно объявлять через `auto` — это сокращённый шаблон:
```cpp
auto add(auto a, auto b) { return a + b; }   // эквивалент шаблона
```

## Идея 6. Нетиповые параметры шаблона (non-type template parameters)

До сих пор параметром шаблона был **тип** (`T`). Но параметром может быть и **значение**,
известное на этапе компиляции: целое, `enum`, указатель/ссылка на объект со статической
адресацией, а в C++20 — и некоторые «структурные» литеральные типы. Самый частый случай —
целочисленная константа размера:

```cpp
template <class T, std::size_t N>
struct FixedArray {
    T data[N];                       // размер зашит в ТИП
    static constexpr std::size_t size() { return N; }
};

FixedArray<int, 4> a;   // N = 4, известно компилятору
```

**Как это работает под капотом.** `N` — это не поле объекта и не аргумент функции; это
часть *идентичности типа*. `FixedArray<int,4>` и `FixedArray<int,8>` — два совершенно
разных, несовместимых типа, как `int` и `double`. Компилятор инстанцирует отдельную
структуру под каждое значение `N`, и `N` доступно в любом `constexpr`-контексте: как
размер встроенного массива, в `static_assert`, как аргумент другого шаблона. Именно так
устроен `std::array<T, N>`: размер — часть типа, поэтому `sizeof` известен, копирование
тривиально, и ничего не лежит в куче.

**Зачем это нужно (мотивация).**
- *Размер в типе* → ошибки размерности ловятся компилятором, а не падением в рантайме.
  Сложить матрицу 2×3 с матрицей 3×2 просто не получится — код не соберётся.
- *Ноль накладных расходов*: `N` не занимает память в объекте и не передаётся в рантайме.
- *Специализация под конкретные значения*: можно по-разному обрабатывать `N == 0`.

**Требования к нетиповому параметру.** Значение обязано быть константой времени компиляции
(`constexpr`). Поэтому это работает:
```cpp
constexpr std::size_t n = 3;
FixedArray<int, n> ok;          // n — constexpr
FixedArray<int, 2 + 2> ok2;     // выражение тоже считается на этапе компиляции
```
а это — нет:
```cpp
std::size_t m = read_from_user();   // значение известно только в рантайме
FixedArray<int, m> bad;             // ОШИБКА: m не constexpr
```

**Частые ошибки.**
1. *Перепутать с обычным аргументом.* `Matrix<int, 2, 3>` — это тип; `2` и `3` нельзя
   «передать в рантайме». Если нужна матрица переменного размера — это уже другой
   инструмент (`std::vector` внутри), не нетиповой параметр.
2. *Расхождение знаковости.* Литерал `5` имеет тип `int`, а параметр объявлен как
   `std::size_t`. Обычно конвертация проходит, но `Matrix<int, -1, 3>` для `std::size_t`
   превратится в гигантское число — добавляй `static_assert(Rows > 0)`, чтобы поймать это.

```cpp
template <std::size_t N>
struct Buffer {
    static_assert(N > 0, "Buffer не может быть пустым");  // защита от N == 0
    int data[N];
};
```

## Идея 7. Специализация шаблона: полная и частичная

Иногда обобщённая реализация для *одного конкретного* типа должна вести себя иначе —
быстрее, безопаснее или просто по-другому. Для этого есть **специализация**: отдельная
реализация того же шаблона под выбранные аргументы.

**Полная специализация (full specialization)** фиксирует *все* параметры:

```cpp
template <class T>
struct Serializer {                    // первичный (общий) шаблон
    static std::string to_str(const T& x) { return std::to_string(x); }
};

template <>                            // полная специализация для bool
struct Serializer<bool> {
    static std::string to_str(bool b) { return b ? "true" : "false"; }
};

Serializer<int>::to_str(42);     // "42"   — общий шаблон
Serializer<bool>::to_str(true);  // "true" — специализация
```

**Как компилятор выбирает.** При инстанцировании он ищет *самую специализированную*
подходящую версию. Полная специализация подходит точнее любого общего варианта, поэтому
для `bool` берётся именно она. Это разрешается на этапе компиляции — никакой виртуальности
и накладных расходов в рантайме нет.

**Частичная специализация (partial specialization)** фиксирует *часть* параметров или
накладывает на них шаблон-образец. Доступна только для **шаблонов классов/структур**:

```cpp
template <class T>
struct TypeInfo { static constexpr bool is_pointer = false; };

template <class T>
struct TypeInfo<T*> {                  // частичная: «любой указатель»
    static constexpr bool is_pointer = true;
    using pointee = T;                 // T выводится из T*
};

TypeInfo<int>::is_pointer;    // false
TypeInfo<int*>::is_pointer;   // true, pointee = int
```

Так устроена половина `<type_traits>`: `std::remove_pointer`, `std::is_array` и др. — это
частичные специализации, выбирающие нужную ветку по форме типа.

**Важнейшая ловушка: функции нельзя частично специализировать.** Это запрещено языком:
```cpp
template <class T> void f(T);
template <class T> void f<T*>(T*);   // ОШИБКА: частичная специализация функции
```
Вместо этого используют **перегрузку** — она решает ту же задачу и выбирается по правилам
разрешения перегрузки (более специализированная перегрузка выигрывает):
```cpp
template <class T> const char* kind(const T&)  { return "value";   }
template <class T> const char* kind(T* const&) { return "pointer"; }  // перегрузка, не спец.
```
Полная специализация *функции* (`template<> void f<int>(int)`) разрешена, но на практике
её почти всегда заменяют обычной перегрузкой — у перегрузки понятнее правила выбора.

**Частые ошибки.**
1. *Специализацию объявили после первого использования.* Специализация должна быть видна
   в точке инстанцирования. Если `Serializer<bool>` уже использовали выше, а
   специализацию написали ниже — поведение нарушено (ill-formed, no diagnostic required).
   Держи специализации рядом с первичным шаблоном.
2. *`template<>` забыли.* Полная специализация начинается с пустых угловых скобок
   `template <>`. Без них это просто переопределение.
3. *Пытаются частично специализировать функцию.* См. выше — берите перегрузку.

## Идея 9. Вывод аргументов шаблона (template argument deduction)

Когда ты пишешь `my_max(3, 5)`, компилятор **сам** определяет `T = int` — ты не пишешь
`my_max<int>(3, 5)`. Это называется вывод аргументов шаблона (template argument deduction,
TAD). Понимание механики убережёт от неочевидных ошибок.

**Как компилятор выводит T.** Компилятор сопоставляет тип аргумента вызова с типом
параметра шаблона. Правила вывода отбрасывают ряд квалификаторов:
- `const` и `volatile` с верхнего уровня отбрасываются: `const int` → `T = int`.
- Ссылки снимаются: передаёшь `int&` — выводится `T = int`.
- Массивы и функции превращаются в указатели (array-to-pointer, function-to-pointer decay).

```cpp
template <class T>
T my_max(const T& a, const T& b);

int x = 3;
const int cx = 5;
my_max(x, cx);   // T = int  (const int -> T = int, верхний const снят)
my_max(x, x);    // T = int  (int& -> T = int, ссылка снята)
```

**Когда вывод не работает — провал дедукции.** Вывод производится *независимо* для
каждого вхождения `T`. Если аргументы дают разные версии `T` — ошибка компиляции:

```cpp
my_max(1, 2.0);   // ОШИБКА: первый аргумент -> T = int,
                  //          второй аргумент -> T = double — конфликт!
```

Типичные причины провала вывода:
1. **Конфликт типов** — как в примере выше.
2. **Не-выводимый контекст** — аргумент появляется внутри другого шаблона в позиции, где
   компилятор не умеет произвести обратный вывод (например, `std::vector<T>::value_type`).
3. **Шаблонный параметр используется только в возвращаемом типе** — возврат не участвует
   в дедукции: `template <class T> T make();` требует явного `make<int>()`.

**Явное указание аргумента.** Если вывести нельзя или нужен другой тип — пиши аргументы
явно. Остальные можно оставить для вывода (справа от явно указанных):

```cpp
my_max<double>(1, 2.0);   // явно T = double; int неявно конвертируется в double
my_max<double>(1, 2);     // тоже работает: оба конвертируются в double
```

**Подвох: ссылочный параметр vs значение.** Если шаблон принимает `T` (по значению) —
`const` и `&` снимаются агрессивнее, чем если принимает `const T&`. При `T&&`
(forwarding reference) поведение совсем другое — см. Идею 15.

## Идея 10. Собственные концепты: как их писать

В Идее 4 мы использовали готовые концепты из `<concepts>`. Но можно (и нужно!)
определять **свои** — ровно под те требования, которые нужны алгоритму.

**Синтаксис концепта.** Концепт — это шаблон, чьё тело — `requires`-выражение,
проверяемое на этапе компиляции и дающее `true`/`false`:

```cpp
template <class T>
concept Addable = requires(T a, T b) {
    { a + b } -> std::convertible_to<T>;  // выражение a+b должно быть допустимо
                                           // и конвертируемо к T
};
```

Требования внутри `requires`-выражения:
- **Простое выражение** `{ expr }` — просто должно компилироваться.
- **Типовое требование** `{ expr } -> Concept<…>` — плюс тип результата должен
  удовлетворять концепту (здесь `std::convertible_to<T>`).
- **Вложенное `requires`** `requires (condition)` — `condition` должно быть `true`.
- **Type requirement** `typename T::value_type` — вложенный тип должен существовать.

**Применение.** Ограничиваем шаблонную функцию своим концептом:

```cpp
template <Addable T>
T twice(T x) { return x + x; }

twice(5);           // int — Addable? да (есть operator+)
twice(std::string("hi"));  // std::string — Addable? да (конкатенация)
// twice(some_struct{});   // ошибка — у some_struct нет operator+
```

**Почему концепт лучше SFINAE по диагностике.** До C++20 ограничения писали через
`std::enable_if` (см. Идею 11). При ошибке компилятор выдавал многостраничный трейсбэк
вглубь инстанцирования. Концепт нарушен — компилятор называет *конкретный концепт* и
показывает *какое требование* не выполнено:

```
error: no matching function for call to 'twice'
note: constraints not satisfied
note: 'Addable<SomeStruct>' evaluated to false
note: the required expression 'a + b' is ill-formed
```

Диагностика — это интерфейс библиотеки для пользователя. Концепты делают его
человекочитаемым.

**Составной концепт.** Концепты комбинируются через `&&` и `||`:

```cpp
template <class T>
concept Numeric = std::integral<T> || std::floating_point<T>;

template <Numeric T>
T square(T x) { return x * x; }
```

**Частые ошибки.**
1. *`requires` внутри тела функции* — это `requires`-выражение (возвращает `bool`),
   а не ограничение шаблона. Они похожи синтаксически, не путай.
2. *Не проверяется семантика* — концепт проверяет только синтаксическую корректность
   выражения, а не его смысл. `requires(T a, T b){ a + b; }` пройдёт, даже если `+`
   делает что-то неожиданное.
3. *Слабое требование к возврату.* Пиши `-> std::convertible_to<T>` вместо
   `-> std::same_as<T>` там, где допустимы неявные конверсии (например, `int + int`
   возвращает `int`, что `same_as<int>`, но `int + short` возвращает `int`, а `T`
   может быть `short` — тут нужен `convertible_to`).

## Идея 11. SFINAE и std::enable_if: исторический контекст

До концептов (C++20) единственным способом ограничить шаблон был механизм
**SFINAE** — «Substitution Failure Is Not An Error» («подстановка неудачна — это не
ошибка»). Понимать SFINAE важно: весь STL до C++20 построен на нём, и ты встретишь
его в любом legacy-коде.

**Суть SFINAE.** Когда компилятор пробует инстанцировать шаблон, он подставляет
аргументы в сигнатуру. Если при этом возникает ошибка *в самой сигнатуре* (не в теле),
компилятор не падает с ошибкой, а просто вычёркивает этот шаблон из кандидатов. Если
другой подходящий кандидат есть — выбирается он.

```cpp
// До C++20: enable_if<cond, T>::type существует, только если cond == true.
// При cond == false подстановка в сигнатуру «тихо» провалится (SFINAE).

template <class T>
std::enable_if_t<std::is_integral_v<T>, bool>
is_power_of_two_old(T x) {
    return x > 0 && (x & (x - 1)) == 0;
}

is_power_of_two_old(5);    // T = int, is_integral<int> = true → ок
is_power_of_two_old(3.0);  // T = double, is_integral<double> = false
                            // → SFINAE, кандидат выброшен → «no matching function»
```

**Как это работает под капотом.** `std::enable_if<cond, T>` — это шаблонная структура
с полем `type`, определённым только если `cond = true`. Если `cond = false`, `type` не
существует, и попытка взять `enable_if_t<false, T>` — это ошибка *в сигнатуре*. SFINAE
её молча поглощает и убирает кандидата.

**Проблемы SFINAE:**
- Громоздкий синтаксис: `enable_if_t<...>` разрастается в многоэтажный шаблон.
- Ошибка *в теле* шаблона — уже не SFINAE, а настоящая ошибка компиляции с километровым
  трейсбэком.
- Сообщения об ошибках трудно читаемы: компилятор говорит «нет подходящей перегрузки»,
  но не объясняет, *почему* все перегрузки отсеялись.

**Что решили концепты (C++20):**
- Читаемый декларативный синтаксис вместо `enable_if_t<…>`.
- Ошибка чётко называет нарушенный концепт и несоблюдённое требование.
- Частичное упорядочение по «более ограниченный шаблон выигрывает» — без хаков с
  `enable_if` для разруливания перегрузок.
- Концепты участвуют в `if constexpr` и `requires`-выражениях — SFINAE не мог.

Вывод: в новом коде используй концепты. Читай SFINAE, когда работаешь с C++14/17 кодом
или с заголовками STL pre-C++20.

## Идея 12. `if constexpr` — ветвление на этапе компиляции

Обычный `if` вычисляется в рантайме, и **обе** ветви должны компилироваться. В шаблоне
это проблема: если ветвишься по типу, одна ветвь может быть синтаксически допустима
только для части типов.

`if constexpr (condition)` решает это: условие вычисляется на этапе компиляции, и
**невыбранная ветвь не инстанцируется** — она может содержать код, который не
скомпилировался бы для данного `T`.

```cpp
template <class T>
std::string to_string_smart(const T& x) {
    if constexpr (std::is_arithmetic_v<T>) {
        return std::to_string(x);          // только для чисел
    } else if constexpr (std::is_same_v<T, std::string>) {
        return x;                           // строка уже есть
    } else {
        return "(unknown)";                 // для остальных
    }
}
```

**Ключевое отличие от обычного `if`.** Если бы здесь был обычный `if`, компилятор
попытался бы скомпилировать все ветви для любого `T`. `std::to_string(x)` для
`T = std::string` — это ошибка. С `if constexpr` невыбранная ветвь не трогается.

**Применение в шаблонах:**
- Разные алгоритмы для разных категорий типов (числа / строки / итерируемые).
- Рекурсивные variadic-функции: базовый случай через `if constexpr (sizeof...(Ts) == 0)`.
- «Оптимизированный путь» для типов с особыми свойствами (trivially_copyable и т.д.).

**`consteval` — ещё строже.** Если `if constexpr` позволяет смешивать compile-time и
runtime, то `consteval` (C++20) объявляет функцию *обязательно* вычисляемой на этапе
компиляции. Попытка вызвать `consteval`-функцию с runtime-значением — ошибка компиляции:

```cpp
consteval int square_ct(int x) { return x * x; }

constexpr int a = square_ct(5);   // ок — compile-time
int n = 5;
int b = square_ct(n);             // ОШИБКА: n не константа компиляции
```

Используй `consteval` для «чистых вычислителей», которым runtime-вызов не имеет смысла
(например, генерация таблиц хешей, кодовых значений). Обычные `constexpr`-функции могут
вычисляться и в compile-time, и в runtime — на ваш выбор.

**Частые ошибки.**
1. *Думают, что `if constexpr` — «быстрее».* Оба `if` и `if constexpr` в простых случаях
   дают одинаковый код (компилятор оптимизирует). Смысл `if constexpr` — не скорость,
   а то, что невыбранная ветвь *не компилируется*.
2. *Отсутствие `else`.* Без `else` обе ветви компилируются независимо. Иногда это
   замысел, но чаще — баг.

## Идея 13. Variadic templates и fold expressions (C++17)

До сих пор все наши шаблоны принимали фиксированное число параметров. **Variadic
templates** позволяют принять *произвольное* количество аргументов — именно так
устроены `std::tuple`, `std::make_unique`, `std::visit`.

**Пакет параметров (parameter pack).** `...` в объявлении означает «ноль или более
параметров»:

```cpp
template <class... Ts>      // Ts — пакет типов (type pack)
struct TypeList {};         // TypeList<int, double, char> — пример

template <class... Ts>
void print_all(Ts... args) {    // args — пакет значений (value pack)
    // (args — ...) — развёртывание пакета
}
```

**Recursion-style до C++17.** Классически variadic разворачивали рекурсией шаблонов:

```cpp
// база
void print_all() {}

template <class Head, class... Tail>
void print_all(Head h, Tail... tail) {
    std::cout << h << ' ';
    print_all(tail...);    // рекурсивный вызов с хвостом
}
```

Многословно, но работает с C++11.

**Fold expressions (C++17).** Заменяют рекурсию одной строкой. Синтаксис:
`(pack op ...)` — unary right fold; `(... op pack)` — unary left fold;
`(init op ... op pack)` — binary fold с начальным значением.

```cpp
template <class... Ts>
auto sum_args(Ts... args) {
    return (args + ...);       // unary right fold: a + (b + (c + d))
}

sum_args(1, 2, 3, 4);      // == 10
sum_args(1.5, 2.5);        // == 4.0
sum_args(std::string("hello"), std::string(" world"));  // конкатенация
```

Binary fold с начальным значением (безопаснее для пустого пакета):

```cpp
template <class... Ts>
auto sum_safe(Ts... args) {
    return (0 + ... + args);   // binary left fold: 0 + a + b + c
}
sum_safe();   // пустой пакет → 0 (без начального значения был бы UB)
```

**Поддерживаемые операторы:** `+`, `-`, `*`, `/`, `%`, `&&`, `||`, `,`, `<<`, `>>`,
`==`, и другие бинарные операторы. Особенно полезны `&&` и `||`:

```cpp
template <class... Ts>
bool all_positive(Ts... xs) {
    return ((xs > 0) && ...);   // ложь, если хоть один <= 0
}
```

**`sizeof...(pack)` — размер пакета** (доступен и в C++11):

```cpp
template <class... Ts>
constexpr std::size_t pack_size() { return sizeof...(Ts); }
pack_size<int, double, char>();  // == 3
```

**Частые ошибки.**
1. *Пустой пакет без начального значения.* `(args + ...)` при пустом `args` — UB для
   `+`. Добавь нейтральный элемент: `(0 + ... + args)`.
2. *Порядок свёртки.* Right fold: `a op (b op (c op init))`. Left fold: `((init op a) op b) op c`.
   Для `+` и `*` порядок неважен, но для `-` и `/` — критичен.
3. *Пытаются передать пакет по `const T&`.* Правильно: `template<class... Ts> void f(Ts&&... args)` —
   тогда каждый аргумент имеет свой тип и проходит идеальную пересылку.

## Идея 14. Perfect forwarding: `T&&` как forwarding reference

В C++ есть две принципиально разные вещи, написанные одинаково — `T&&`:
1. **Rvalue reference** — когда `T` — конкретный тип: `int&& r = std::move(x);`
2. **Forwarding reference** (ранее — universal reference) — когда `T` — шаблонный
   параметр именно этого шаблона и `&&` написан в объявлении параметра.

```cpp
template <class T>
void sink(T&& arg);     // arg — forwarding reference, НЕ rvalue-ref

void sink(int&& arg);   // это rvalue reference — не шаблон!
```

**Зачем это нужно — проблема perfect forwarding.** Представь функцию-обёртку, которая
должна вызвать другую с теми же аргументами, не меняя их категорию значения (lvalue
остаётся lvalue, rvalue — rvalue):

```cpp
template <class T>
void wrapper(T&& arg) {
    target(std::forward<T>(arg));   // пересылаем arg, сохраняя его value category
}
```

`std::forward<T>(arg)` — условный cast: если `T` вывелся как `int&` (переданный lvalue) —
возвращает lvalue-ref; если как `int` (переданный rvalue) — возвращает rvalue-ref. Без него
именованный параметр `arg` всегда lvalue внутри функции, и rvalue не перемещается —
происходит лишняя копия.

**Механизм: reference collapsing.** Когда `T&&` встречает:
- lvalue `int x`: `T = int&`, тип `int& &&` сворачивается в `int&` — получаем lvalue-ref.
- rvalue `int(5)`: `T = int`, тип `int&&` — rvalue-ref.

Правило сворачивания: любая комбинация с `&` даёт `&`; только `&& + &&` даёт `&&`.

**Variadic perfect forwarding** — стандартный паттерн в `std::make_unique` и др.:

```cpp
template <class T, class... Args>
std::unique_ptr<T> my_make_unique(Args&&... args) {
    return std::unique_ptr<T>(new T(std::forward<Args>(args)...));
}
```

**Частые ошибки.**
1. *Забыли `std::forward` — пишут `target(arg)`*. Именованный параметр всегда lvalue —
   rvalue-семантика теряется, происходит копия вместо перемещения.
2. *`std::move` вместо `std::forward`.* `std::move` всегда даёт rvalue-ref — это ломает
   пересылку lvalue-аргументов.
3. *Forwarding reference только в шаблоне.* `void f(auto&& x)` — тоже forwarding
   reference (сокращённый шаблон C++20). А вот `void f(const T& x)` с `const` —
   уже нет: `const T&&` всегда rvalue-ref.

## Идея 15. Variable templates и alias templates

Два небольших, но практически полезных инструмента.

**Variable template** (C++14) — шаблон, который является *переменной*, а не функцией
или классом. Вычисляется на этапе компиляции:

```cpp
template <class T>
constexpr bool is_numeric_v = std::is_arithmetic_v<T>;

is_numeric_v<int>;     // true — compile-time bool
is_numeric_v<std::string>;  // false

// Ты уже видел это в модуле: pow_v — variable template поверх структуры Pow:
template <long long Base, unsigned Exp>
inline constexpr long long pow_v = Pow<Base, Exp>::value;
```

Стандартная библиотека активно использует эту идиому: `std::is_integral_v<T>` —
это именно variable template, который короче, чем `std::is_integral<T>::value`.

**Alias template** — шаблон, который является псевдонимом типа (`using`):

```cpp
template <class T>
using Vec = std::vector<T>;   // Vec<int> == std::vector<int>

template <class K, class V>
using Map = std::map<K, V, std::greater<K>>;   // карта с обратной сортировкой

Vec<double> v{1.0, 2.0};
Map<std::string, int> m;
```

Важно: alias template — это *псевдоним*, а не новый тип. `Vec<int>` и `std::vector<int>`
— одно и то же. Поэтому частичную специализацию alias template сделать нельзя — только
у «настоящего» шаблона.

Стандартная библиотека: `std::enable_if_t<B,T>` — alias template для
`typename std::enable_if<B,T>::type`, что значительно короче.

**`pow_v` из задания 8.** Ты уже реализовал `Pow<Base,Exp>` — структуру с `::value`.
`pow_v` — это variable template, который прячет `::value` за коротким именем:
```cpp
template <long long Base, unsigned Exp>
inline constexpr long long pow_v = Pow<Base, Exp>::value;

pow_v<2, 10>;           // 1024 — короче, чем Pow<2,10>::value
static_assert(pow_v<2, 10> == 1024);
```

## Идея 8. Параметры шаблона по умолчанию

У параметра шаблона может быть значение по умолчанию — так же, как у аргумента функции.
Это работает и для типовых, и для нетиповых параметров:

```cpp
template <class T, class Compare = std::less<T>>
struct SortedBox { /* ... использует Compare для упорядочивания ... */ };

SortedBox<int>               box1;  // Compare = std::less<int>
SortedBox<int, std::greater<int>> box2;  // явно задали другой компаратор
```
Именно так `std::vector<T, Allocator = std::allocator<T>>` и `std::map<K, V, Compare, Alloc>`
позволяют не писать аллокатор в 99% случаев, но дают его поменять при необходимости.

**Правила, как у аргументов функции:** параметры по умолчанию идут *справа* — нельзя
дать дефолт левому параметру, оставив правый без него:
```cpp
template <class T = int, class U>   // ОШИБКА: U без дефолта правее T с дефолтом
struct Bad {};
```
Нетиповой параметр тоже можно задефолтить, и дефолт может *зависеть от предыдущих
параметров*:
```cpp
template <class T, std::size_t N = 16>
struct Stack { T data[N]; };

Stack<int>     s16;   // N = 16
Stack<int, 64> s64;   // N = 64
```

**Частые ошибки.**
1. *Думают, что дефолт = вывод.* Дефолт применяется, только если аргумент *не указан*; он
   не «выводится» из чего-либо. Для функций дефолт параметра не участвует в дедукции по
   аргументам — он лишь подставляется, если вывести не удалось.
2. *Дублируют дефолт в объявлении и определении.* Значение по умолчанию указывают **один
   раз** (обычно в объявлении). Повтор в определении — ошибка.
