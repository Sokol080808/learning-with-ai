# Модуль 20 — Многопоточность: atomics, future/promise, пул потоков

Большая идея: когда несколько потоков трогают **общие данные**, обычные операции
перестают быть «одним действием» и начинают **рассыпаться на части**, которые
переплетаются непредсказуемо. Этот модуль — про инструменты, которые возвращают
порядок: атомарные операции, мьютексы с условиями, `future`/`promise` и пул
потоков. Мы будем не просто называть их, а разбираться, **почему** они нужны и
**как** работают под капотом.

## Идея 1. Гонка данных — что это и почему `x++` ломается

Возьмём безобидное `++counter`, где `counter` — обычная `long`, видимая двум
потокам. Компилятор превращает инкремент в **три** машинных шага:

```
1) load   counter -> регистр
2) add    регистр, 1
3) store  регистр -> counter
```

Два потока могут «прослоиться»: оба прочитали `counter == 5`, оба прибавили,
оба записали `6`. Один инкремент **потерян**. Это и есть **гонка данных** (data
race): два потока обращаются к одной ячейке, хотя бы один пишет, и нет
синхронизации между ними.

Важно понимать радикальность ситуации в C++: гонка данных — это не «иногда
неверный результат», это **undefined behaviour**. Стандарт не обещает ничего:
компилятор имеет право предполагать, что гонок нет, и оптимизировать код так, что
наблюдаемое поведение станет вообще бессмысленным (например, выкинуть повторное
чтение переменной, решив, что «её же никто не менял»). Поэтому «да оно вроде
работает» на одном прогоне — иллюзия: вы просто не поймали плохое расписание.

**Частая ошибка №1** — «добавлю `volatile`, и станет потокобезопасно».
`volatile` запрещает компилятору кэшировать переменную в регистре, но **не**
делает `++` неделимым и **не** упорядочивает доступ между потоками. `volatile`
в C++ — про железо (memory-mapped регистры, обработчики сигналов), не про
многопоточность. Для потоков нужен `std::atomic` или мьютекс.

```cpp
std::atomic<long> counter{0};
counter.fetch_add(1);   // ВЕСЬ инкремент — одно неделимое действие
```

## Идея 2. `std::atomic` и `fetch_add`: неделимость как примитив

`std::atomic<T>` оборачивает значение так, что операции над ним **атомарны**:
ни один поток не увидит «полузаписанного» значения, и read-modify-write
(прочитать-изменить-записать) выполняется целиком или не выполняется вовсе.

Ключевые операции:

```cpp
std::atomic<long> a{0};
a.store(10);            // атомарно записать
long x = a.load();      // атомарно прочитать
long old = a.fetch_add(5);  // a += 5, ВЕРНУТЬ старое значение (было 10)
long now = a.load();        // 15
a.exchange(7);          // записать 7, вернуть прежнее
```

Обратите внимание на семантику `fetch_add`: он возвращает значение **до**
прибавления. Это не каприз — это то, что делает его полезным как «выдатчик
билетов»: каждый поток, дёрнув `fetch_add(1)`, получает свой уникальный номер,
и никакие два потока не получат один и тот же.

На современных x86/ARM атомарный инкремент компилируется в одну машинную
инструкцию с префиксом блокировки шины (`lock xadd` на x86) — это дёшево,
сильно дешевле мьютекса, потому что не уводит поток в ядро ОС.

**Частая ошибка №2** — думать, что `atomic` делает атомарными *цепочки*
операций. Атомарна каждая *отдельная* операция, но не их последовательность:

```cpp
if (a.load() < 10)   // прочитали 9...
    a.store(a.load() + 1);  // ...но между load и store кто-то уже записал 100
```

Здесь между чтением и записью образуется дыра. Чтобы «прочитать-проверить-
записать» было неделимым, нужен `compare_exchange` (Идея 3) или мьютекс.

## Идея 3. `compare_exchange` и CAS-цикл

Иногда нужно обновить атомик **в зависимости от его текущего значения** —
например, «запиши новое, только если старое всё ещё то, что я видел». Это даёт
`compare_exchange_weak(expected, desired)`:

- сравнивает атомик с `expected`;
- если равны — записывает `desired`, возвращает `true`;
- если **не** равны — записывает фактическое значение атомика **в `expected`**
  (важная деталь!) и возвращает `false`.

Это кирпичик всех неблокирующих алгоритмов. Канонический приём — **CAS-цикл**:

```cpp
void store_max(std::atomic<long>& best, long candidate) {
    long cur = best.load();
    while (candidate > cur &&
           !best.compare_exchange_weak(cur, candidate)) {
        // CAS не прошёл: значит cur уже обновлён фактическим значением best,
        // условие цикла перепроверится с актуальным cur.
    }
}
```

Логика: мы хотим best = max(best, candidate). Читаем `cur`. Пока кандидат
больше — пытаемся записать. Если кто-то опередил, `compare_exchange_weak`
сам кладёт в `cur` новое значение `best`, и мы перепроверяем: вдруг теперь
`best` уже больше кандидата — тогда цикл закончится, и мы ничего не испортим.

Почему `weak`, а не `strong`? `compare_exchange_weak` имеет право иногда вернуть
`false` **даже когда значения равны** (так называемый spurious fail — это
дешевле на некоторых архитектурах, где CAS строится на паре LL/SC). В цикле
это безвредно: лишняя итерация. `strong` гарантирует отсутствие ложных провалов,
но дороже; **в циклах всегда берут `weak`**, вне цикла (одиночная попытка) —
`strong`.

**Частая ошибка №3** — забыть, что при провале `expected` **перезаписывается**.
Если внутри цикла заново сделать `cur = best.load()`, вы потеряете значение,
которое CAS уже любезно туда положил, и добавите лишнюю гонку. Доверьтесь
`compare_exchange`: после `false` в `cur` уже свежее значение.

## Идея 4. Модель памяти и `memory_order` (на уровне интуиции)

Каждая атомарная операция принимает необязательный параметр — порядок памяти.
По умолчанию это `memory_order_seq_cst` (последовательная согласованность):
самый строгий и самый интуитивный — все потоки видят атомарные операции в одном
глобальном порядке. Это всегда **корректно**; начинайте с него.

Зачем тогда другие? Потому что процессор и компилятор **переставляют** обычные
чтения/записи ради скорости, и иногда строгий порядок — это переплата. Три
уровня, которые стоит понимать интуитивно:

- **`relaxed`** — гарантирует только атомарность самой операции, но **никакого**
  порядка относительно других обращений к памяти. Годится для счётчика, чьё
  значение важно лишь в самом конце (например, статистика), но НЕ годится, когда
  по значению атомика вы делаете вывод о состоянии **других** данных.

  ```cpp
  counter.fetch_add(1, std::memory_order_relaxed);  // просто считаем события
  ```

- **`release`** (на записи) и **`acquire`** (на чтении) образуют пару
  «публикация — подписка». Запись с `release` гарантирует: всё, что поток
  записал в обычную память **до** этой записи, станет видно другому потоку,
  который прочитает этот же атомик с `acquire`. Это механика передачи данных
  через флаг готовности:

  ```cpp
  // Поток-писатель:
  data = 42;                                   // обычная запись
  ready.store(true, std::memory_order_release); // «публикуем»

  // Поток-читатель:
  while (!ready.load(std::memory_order_acquire)) {}  // «подписались»
  assert(data == 42);  // гарантировано: data увидим как 42
  ```

  Именно так работает захват/освобождение блокировки: `lock` — это `acquire`,
  `unlock` — это `release`, и поэтому всё, что сделано под одной блокировкой,
  корректно «передаётся» следующему захватившему.

**Главная интуиция:** `acquire`/`release` создают отношение
*happens-before* — единственную в C++ гарантию того, что одна запись «видна»
другому потоку. Без неё чтение чужой обычной памяти — гонка данных, то есть UB.

**Частая ошибка №4** — `relaxed` «потому что быстрее», когда по атомику-флагу
вы читаете другие переменные. Так появляются неуловимые баги: флаг уже `true`,
а данные ещё «не доехали». Если сомневаетесь — `seq_cst` (по умолчанию).
Оптимизируйте порядок только когда доказали, что он узкое место.

## Идея 5. `condition_variable`: ждать условие, а не крутиться

Спинлок (Идея 6) хорош для очень коротких критических секций, но «ждать, пока
в очереди появится элемент» круткой — это сжигать процессор впустую. Нужно
**усыпить** поток и разбудить, когда условие может стать истинным. Это
`std::condition_variable`.

Канонический шаблон ожидания:

```cpp
std::mutex m;
std::condition_variable cv;
std::queue<int> q;

int pop() {
    std::unique_lock<std::mutex> lock(m);     // 1) захватили мьютекс
    cv.wait(lock, [&]{ return !q.empty(); });  // 2) ждём условие
    int v = q.front(); q.pop();                // 3) условие истинно — берём
    return v;
}

void push(int v) {
    {
        std::lock_guard<std::mutex> lock(m);
        q.push(v);
    }
    cv.notify_one();   // будим одного ждущего
}
```

Как это работает под капотом: `cv.wait(lock, pred)` — это, по сути,

```cpp
while (!pred())
    cv.wait(lock);   // атомарно ОСВОБОЖДАЕТ мьютекс и засыпает;
                     // при пробуждении заново захватывает мьютекс
```

Два тонких момента, оба критичны.

**Защита от ложных пробуждений (spurious wakeups).** ОС имеет право разбудить
поток на `cv.wait`, даже если никто не звал `notify`. Поэтому условие
проверяется в **цикле** (или через `wait` с предикатом, который этот цикл прячет
внутри). Если написать `if (q.empty()) cv.wait(lock);` без цикла — поток может
проснуться на пустой очереди и выполнить `q.front()` на пустом контейнере →
UB. **Всегда** используйте форму `wait(lock, predicate)` — она встроенно
зацикливает проверку.

**Почему `wait` атомарно отпускает мьютекс.** Если бы поток сначала отпустил
мьютекс, а потом заснул, между этими шагами `push` мог бы успеть положить
элемент и дёрнуть `notify` — в пустоту, потому что ждущий ещё не спит. Получили
бы **потерянное уведомление** (lost wakeup) и вечный сон. `wait` снимает
блокировку и встаёт в очередь ожидания **неделимо**, закрывая эту дыру.

**Частая ошибка №5** — звать `notify` под всё ещё держащимся мьютексом и/или
не под защитой проверять `q.empty()` снаружи `wait`. Меняйте защищённые данные
под мьютексом, а `notify` можно звать и после `unlock` — разбуженный поток всё
равно сначала захватит мьютекс. Главное — не «голый» `wait` без предиката.

## Идея 6. SpinLock на `std::atomic_flag`

`std::atomic_flag` — простейший атомик, всегда lock-free, с двумя операциями:
`test_and_set` (установить в true, **вернуть прежнее** значение) и `clear`
(сбросить в false). На них строится спинлок — мьютекс, который не спит, а
крутится:

```cpp
class SpinLock {
    std::atomic_flag flag_ = ATOMIC_FLAG_INIT;
public:
    void lock() {
        // крутимся, пока флаг был уже занят (test_and_set вернул true)
        while (flag_.test_and_set(std::memory_order_acquire)) { /* spin */ }
    }
    void unlock() {
        flag_.clear(std::memory_order_release);
    }
};
```

Почему `acquire` на захвате и `release` на освобождении? Это ровно пара из
Идеи 4: всё, что сделано под блокировкой, корректно передаётся следующему
владельцу. Без правильных порядков спинлок «как бы» работает, но защищённые им
данные могут «не доехать».

Когда спинлок уместен: критическая секция **крошечная** (несколько инструкций) и
блокировка почти всегда свободна — тогда дешевле покрутиться пару тактов, чем
уходить в ядро ОС, как делает `std::mutex`. Когда вреден: секция длинная или
потоков больше, чем ядер — крутящийся поток жжёт ядро, которое нужно тому, кто
держит лок. Тогда берите `std::mutex`.

**Частая ошибка №6** — спроектировать спинлок без правильных `memory_order`
(или с `relaxed`) и удивляться редким багам; и забыть, что спинлок **не
рекурсивный** — повторный `lock()` из того же потока навсегда повиснет.

## Идея 7. `future`/`promise`/`packaged_task`/`async`

Часто потоку нужно не «просто сделать», а **вернуть результат** — и обработать
исключение, если оно вылетело. Для этого есть пара `std::promise` (сторона
производителя) и `std::future` (сторона потребителя), соединённые общим
состоянием:

```cpp
std::promise<int> p;
std::future<int>  f = p.get_future();   // берём future ОДИН раз

std::thread t([&p]{ p.set_value(40 + 2); });  // в другом потоке кладём результат
int r = f.get();   // блокируется, пока value не положат; здесь r == 42
t.join();
```

`f.get()` блокирует вызывающий поток до готовности результата. Если производитель
вместо `set_value` сделал `p.set_exception(...)`, то `f.get()` **перебросит** это
исключение — удобный способ протащить ошибку через границу потоков.

`std::packaged_task<R(Args...)>` — это «promise, обёрнутый вокруг функции»:
вызвать его — значит выполнить функцию и автоматически положить результат (или
исключение) в связанный `future`. Это идеальный носитель задачи для пула:

```cpp
std::packaged_task<int()> task([]{ return 42; });
std::future<int> f = task.get_future();
task();              // выполнили — результат уехал в future
int r = f.get();     // 42
```

`std::async` — самый высокоуровневый вход: запускает функцию (возможно, в новом
потоке) и сразу отдаёт `future`:

```cpp
std::future<int> f = std::async(std::launch::async, []{ return 42; });
int r = f.get();
```

**Частая ошибка №7** — звать `get()` дважды. `future` одноразовый: после `get()`
общее состояние «выпито», повторный `get()` — UB/исключение. Если результат
нужен многим — есть `std::shared_future`. И ещё: `future`, возвращённый
`std::async(launch::async, ...)`, в своём деструкторе **блокирует** до конца
задачи — поэтому `std::async(...);` без сохранения результата в переменную ведёт
себя синхронно, что часто удивляет.

## Идея 8. Пул потоков (ThreadPool)

Создавать `std::thread` под каждую мелкую задачу дорого: поток — это ресурс ОС
(стек, дескриптор, переключение контекста). **Пул потоков** создаёт фиксированное
число потоков один раз, а задачи раздаёт через общую очередь. Архитектура:

- общая **очередь задач** (`std::queue<std::function<void()>>`), защищённая
  мьютексом;
- `condition_variable`, на которой воркеры спят, пока задач нет;
- флаг `stop_` для корректного завершения;
- каждый воркер крутит цикл: «дождись задачу или сигнал стоп → возьми → выполни».

```cpp
void worker_loop() {
    while (true) {
        std::function<void()> task;
        {
            std::unique_lock<std::mutex> lock(mutex_);
            cv_.wait(lock, [&]{ return stop_ || !tasks_.empty(); });
            if (stop_ && tasks_.empty()) return;   // выходим, доев очередь
            task = std::move(tasks_.front());
            tasks_.pop();
        }                  // мьютекс отпущен ДО запуска задачи!
        task();            // выполняем вне блокировки
    }
}
```

Критично: **`task()` вызывается вне мьютекса**. Иначе все воркеры
сериализуются — пул выродится в один поток, а длинная задача заблокирует
постановку новых. Захватили под локом только снятие задачи из очереди.

Корректное завершение в деструкторе — самая частая точка ошибок:

```cpp
~ThreadPool() {
    {
        std::lock_guard<std::mutex> lock(mutex_);
        stop_ = true;
    }
    cv_.notify_all();              // будим ВСЕХ, чтобы вышли
    for (auto& w : workers_) w.join();
}
```

`notify_all`, а не `notify_one` — иначе проснётся один воркер, а остальные так и
будут спать на `wait` и `join()` повиснет навсегда. Условие выхода
`stop_ && tasks_.empty()` означает «выходим, только когда сказали стоп **и**
очередь доедена» — уже принятые задачи не теряются.

`submit` связывает функцию с `packaged_task` и возвращает `future`:

```cpp
template <class F, class... Args>
auto submit(F&& f, Args&&... args)
    -> std::future<std::invoke_result_t<F, Args...>> {
    using R = std::invoke_result_t<F, Args...>;
    auto task = std::make_shared<std::packaged_task<R()>>(
        [f = std::forward<F>(f),
         ... a = std::forward<Args>(args)]() mutable {
            return f(a...);
        });
    std::future<R> fut = task->get_future();
    {
        std::lock_guard<std::mutex> lock(mutex_);
        tasks_.emplace([task]{ (*task)(); });   // стираем тип до void()
    }
    cv_.notify_one();
    return fut;
}
```

`shared_ptr` здесь потому, что `packaged_task` **не копируется** (только
move), а `std::function` требует копируемую цель — `shared_ptr` обходит это.
`invoke_result_t<F, Args...>` вычисляет тип возвращаемого значения, чтобы
`future` имел правильный параметр.

**Частая ошибка №8** — `notify_one` вместо `notify_all` в деструкторе (повисший
`join`); вызов задачи под мьютексом (нет параллелизма); забытый `join` (поток
переживёт пул → UB, обращение к разрушенным полям). Деструктор обязан дождаться
всех воркеров.

## Идея 9. Взаимные блокировки (deadlock)

Если поток A держит мьютекс M1 и ждёт M2, а поток B держит M2 и ждёт M1 — оба
застывают навсегда. Это **deadlock**. Классические рецепты против него:

- **единый порядок захвата** — все потоки берут мьютексы в одном и том же
  порядке (например, по адресу). Тогда циклического ожидания не возникает;
- **`std::scoped_lock` / `std::lock`** — захватывают несколько мьютексов
  «всё-или-ничего», без риска застрять на полпути:

  ```cpp
  std::scoped_lock lock(m1, m2);  // C++17: берёт оба атомарно, без deadlock
  ```

- **не звать чужой код под мьютексом** — колбэк может попытаться взять тот же
  лок (→ повтор-захват не-рекурсивного мьютекса = UB/повисание) или другой
  (→ скрытая инверсия порядка).

**Частая ошибка №9** — держать блокировку «на всякий случай» дольше, чем нужно,
и звать под ней длинные операции (I/O, ожидание другого потока). Держите
критическую секцию минимальной: захватили — поменяли данные — отпустили.

## Задания (пиши прямо в `include/task20.hpp`, всё в namespace `m20`)

1. **Атомарный счётчик и максимум.**
   - `long atomic_add(std::atomic<long>& cnt, long delta)` — атомарно
     прибавить `delta`, вернуть значение **до** прибавления (как `fetch_add`).
   - `void atomic_store_max(std::atomic<long>& best, long candidate)` —
     обновить `best` до `max(best, candidate)` через **CAS-цикл** на
     `compare_exchange_weak`, без мьютекса. При параллельных вызовах итог —
     глобальный максимум.

2. **`SpinLock`** — методы `lock()` и `unlock()` на `std::atomic_flag`.
   `lock` крутится на `test_and_set` с `memory_order_acquire`, `unlock` —
   `clear` с `memory_order_release`. Должен работать с `std::lock_guard`.

3. **`ThreadSafeQueue<T>`** на `std::mutex` + `std::condition_variable`:
   - `push(T)` — положить и `notify_one`;
   - `pop()` — **блокирующий**: ждать через `cv_.wait(lock, predicate)`
     (защита от ложных пробуждений!), затем снять и вернуть фронт;
   - `bool try_pop(T& out)` — неблокирующий: `false` если пусто (и `out` не
     трогать), иначе снять фронт в `out` и вернуть `true`.

4. **`ThreadPool`**:
   - конструктор запускает `worker_count` потоков;
   - `submit(f, args...)` ставит задачу в очередь и возвращает
     `std::future` с результатом (тип результата —
     `std::invoke_result_t<F, Args...>`);
   - деструктор корректно завершает все потоки, доедая очередь, без зависаний.

5. **`parallel_sum(const std::vector<long>& v, unsigned num_threads)`** —
   сумма всех элементов в `num_threads` потоках; результат **совпадает** с
   обычной суммой. Пустой вектор → 0; `num_threads >= 1`; число потоков может
   быть больше числа элементов.

### Подсказки

<details><summary>Задание 1 — atomic_store_max: почему именно цикл</summary>
Один CAS может не пройти, потому что между твоим `load` и попыткой записи кто-то
успел изменить `best`. Это нормально: `compare_exchange_weak` при провале сам
кладёт актуальное значение в твою переменную-`expected`. Зацикли попытку и
перепроверяй условие «кандидат всё ещё больше» в самом цикле — НЕ перечитывай
`best.load()` вручную внутри цикла.
</details>

<details><summary>Задание 2 — SpinLock: что возвращает test_and_set</summary>
`test_and_set` ставит флаг в true и возвращает значение, которое было ДО.
Значит, пока он возвращает `true`, лок занят кем-то другим — продолжай крутиться.
Как только вернул `false` — флаг был свободен, и ты его только что захватил.
Подумай, какой `memory_order` на `clear`, чтобы изменения под локом «доехали»
до следующего владельца.
</details>

<details><summary>Задание 3 — pop: форма wait и порядок действий</summary>
Возьми `std::unique_lock<std::mutex>` (не `lock_guard` — `wait` должен уметь
отпускать и заново брать мьютекс). Затем `cv_.wait(lock, [&]{ return
!queue_.empty(); });`. После выхода из `wait` очередь точно не пуста: снимай
`front()`, делай `pop()`, возвращай снятое значение. В `push` меняй очередь под
мьютексом, а `notify_one` можно и после.
</details>

<details><summary>Задание 4 — submit: тип результата и носитель задачи</summary>
`using R = std::invoke_result_t<F, Args...>;` даст тип возврата. Заведи
`std::packaged_task<R()>` (или его через `shared_ptr`, потому что
`packaged_task` не копируется, а `std::function` хочет копируемое). Возьми
`future` ДО того, как отдашь задачу в очередь. В очередь клади `std::function<
void()>`, которая вызывает твой `packaged_task`. Дальше — `notify_one`.
</details>

<details><summary>Задание 4 — деструктор без зависаний</summary>
Поставь `stop_ = true` под мьютексом, затем `cv_.notify_all()` (именно ALL —
иначе часть воркеров не проснётся и `join` повиснет), затем `join()` каждому.
В цикле воркера условие выхода — «`stop_` И очередь пуста», чтобы уже принятые
задачи не пропали. И помни: саму задачу вызывай ВНЕ мьютекса.
</details>

<details><summary>Задание 5 — parallel_sum: куски без общего состояния</summary>
Раздели вектор на `num_threads` примерно равных диапазонов `[begin, end)`.
Каждый поток считает сумму СВОЕГО диапазона в свою локальную переменную и
кладёт её в свою ячейку результата (например, `std::vector<long> partial(n)` —
поток i пишет только в `partial[i]`, гонок нет). После `join` всех потоков
сложи частичные суммы. Если потоков больше, чем элементов, лишние получат
пустой диапазон и вернут 0 — это нормально.
</details>

Когда тесты позеленеют — покажи мне `atomic_store_max` и цикл воркера: обсудим,
почему `weak`, а не `strong`, и почему задача вызывается вне мьютекса. Это два
самых поучительных места модуля.
