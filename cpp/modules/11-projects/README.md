# Модуль 11 — Многофайловые проекты, библиотеки и CMake

До сих пор CMake был «чёрным ящиком». Пора заглянуть внутрь. Большая идея: реальные
проекты состоят из **библиотек** (переиспользуемый код), **приложений** (точка входа) и
**тестов** — и всё это связывает система сборки. Junior должен уметь читать и писать
простой `CMakeLists.txt`.

## Идея 1. Из чего собирается C++-программа: единицы трансляции

Чтобы понять, зачем «много файлов», нужно знать, **как** компилятор C++ превращает
текст в исполняемый файл. Это конвейер из нескольких независимых стадий:

1. **Препроцессор.** Берёт один `.cpp`, рекурсивно вставляет тела всех `#include` и
   раскрывает макросы. На выходе — большой плоский текст без директив. Этот текст
   называется **единицей трансляции** (translation unit, TU). Ключевое: каждый `.cpp` —
   отдельная TU, и компилятор видит ровно её, ничего не зная про другие `.cpp`.
2. **Компиляция.** Каждая TU превращается в **объектный файл** (`.o`/`.obj`) —
   машинный код плюс таблица символов. Если в TU вызвана функция, объявленная, но не
   определённая здесь, компилятор оставляет в таблице **неразрешённую ссылку** (`U`).
3. **Линковка.** Линкер берёт все `.o` и библиотеки и **сшивает** ссылки: для каждой
   неразрешённой ссылки ищет определение в других объектниках. Нашёл ровно одно —
   хорошо; ноль — `undefined reference`; два — `multiple definition`.

Почему это важно для раскладки по файлам:

- **Скорость инкрементальной сборки.** Изменил один `.cpp` — пересобирается только его
  TU (один `.o`), затем быстрая линковка. Если бы весь код лежал в одном файле, любая
  правка пересобирала бы всё. Компиляция — самая дорогая стадия, поэтому дробление на
  TU экономит минуты на больших проектах.
- **Изоляция интерфейса от реализации.** Пользователю библиотеки нужен только `.hpp`
  (что вызывать), а не `.cpp` (как оно сделано). Можно поменять реализацию, не трогая
  заголовок, — и зависимые TU даже не нужно перекомпилировать (только перелинковать).
- **Переиспользование.** Один объектник/одну библиотеку линкуют сразу к приложению и к
  тестам — пишем реализацию один раз.

**Главная ловушка новичка:** «я подключил `#include "foo.cpp"` и всё заработало». Так
делать нельзя. `#include` — это тупая текстовая вставка. Включив `.cpp` в две TU, ты
получишь два определения одной функции → `multiple definition` при линковке. Правильно:
включаем только **заголовки** (объявления), а `.cpp` компилируем отдельно и линкуем.

```cpp
// foo.hpp  — объявление (можно включать сколько угодно раз)
int add(int a, int b);

// foo.cpp  — определение (компилируется как отдельная TU)
#include "foo.hpp"
int add(int a, int b) { return a + b; }

// main.cpp
#include "foo.hpp"        // НЕ "foo.cpp"
int main() { return add(2, 3); }
```

## Идея 2. Объявление vs определение и ODR

Два разных понятия, которые новички путают:

- **Объявление (declaration)** обещает компилятору, что такая-то сущность где-то
  существует, и описывает её *тип/сигнатуру*. Объявлений может быть **сколько угодно**.
  `double distance(Point, Point);` — объявление функции.
- **Определение (definition)** даёт сущности «тело» / выделяет место. Функция с `{...}`,
  переменная с памятью, класс с телом `{...}`.

**One Definition Rule (ODR)** — два правила в одном:

1. В пределах **одной TU** сущность нельзя определить дважды.
2. Во **всей программе** у функции/глобальной переменной должно быть ровно **одно**
   определение (для inline-сущностей — одно *на TU*, но все определения обязаны быть
   идентичны — токен в токен).

Почему это техническая, а не вкусовая проблема. Линкер видит **символы**. Обычная
функция компилируется в символ с *внешним связыванием* (external linkage): он попадает
в таблицу экспортов объектника. Если такой символ экспортируют два объектника — линкер
не знает, какой выбрать, и падает с `multiple definition`.

```cpp
// util.hpp
int square(int x) { return x * x; }   // ❌ ОПРЕДЕЛЕНИЕ в заголовке!
```
Включили `util.hpp` в `a.cpp` и `b.cpp` — оба объектника экспортируют символ `square` →
ошибка линковки. **Как чинить — три способа, выбор по ситуации:**

```cpp
// 1) Объявление в .hpp, определение в одном .cpp  — для обычных функций.
int square(int x);                 // util.hpp
int square(int x){ return x*x; }   // util.cpp

// 2) inline — определение в заголовке разрешено; линкер схлопнет копии в одну.
inline int square(int x){ return x*x; }   // util.hpp, можно включать везде

// 3) шаблон/constexpr — у них тело обязано быть в заголовке (нужно при инстанцировании).
template <class T> T square(T x){ return x*x; }
```

Почему `inline` спасает: ключевое слово `inline` (в этом контексте — *не* про
встраивание кода!) говорит линкеру «таких определений будет много копий, они
идентичны, оставь одну, остальные выброси». Шаблоны и `constexpr`-функции `inline` по
умолчанию — поэтому их тела и живут в заголовках (мы это видели в модулях 07 и 10).

**Частая ошибка:** глобальная переменная в заголовке.
```cpp
int counter = 0;            // ❌ в .hpp → multiple definition
inline int counter = 0;     // ✅ C++17 inline-переменная: одна на программу
// либо: extern int counter; в .hpp + int counter = 0; в одном .cpp
```

## Идея 3. Разделение интерфейс / реализация на практике

«Интерфейс» — это контракт: имена, сигнатуры, типы, которые видит пользователь.
«Реализация» — как именно это работает. Разделение даёт три выгоды.

**1. Время сборки.** Заголовок включается в десятки TU. Каждое изменение `.hpp`
заставляет перекомпилировать **все** включающие его TU. Изменение `.cpp` — только его
одну TU. Поэтому держи в заголовке минимум: объявления, а не реализацию.

**2. Сокрытие деталей.** Пользователю не нужно (и вредно) видеть, *как* посчитан
периметр. Если в `.hpp` утекают приватные хелперы, любой может на них завязаться, и ты
не сможешь их менять. Держи внутренние функции в `.cpp` (и/или в анонимном namespace —
см. Идею 4).

**3. Стабильный ABI/контракт.** Меняешь тело в `.cpp` — зависимые TU **не нужно
перекомпилировать**, только перелинковать. Это и есть смысл библиотеки.

Что класть куда:

| В заголовок (`.hpp`)                         | В исходник (`.cpp`)                       |
|----------------------------------------------|-------------------------------------------|
| объявления функций                           | тела обычных функций                      |
| определения классов/структур (нужен размер)  | методы класса (можно вынести из тела)     |
| `inline`/`constexpr`/шаблоны (тела)          | внутренние хелперы (анонимный namespace)  |
| публичные `using`/типы/константы (`inline`)  | глобальные переменные с определением       |

Минимизируй и сами `#include` в заголовке: что нужно только реализации — включай в
`.cpp`. Если в заголовке тип используется лишь как `T&`/`T*`, хватит **предобъявления**
`class T;` вместо `#include` — это рвёт лишние зависимости и ускоряет сборку.

```cpp
// widget.hpp
class Engine;                 // предобъявление — полного определения тут не нужно
class Widget {
    Engine* engine_;          // указатель — достаточно неполного типа
public:
    void run();               // объявление; тело в widget.cpp, где #include "engine.hpp"
};
```

## Идея 4. Namespaces — пространства имён

`namespace` группирует имена в логический «контейнер», чтобы они не сталкивались с
именами из других библиотек. Это решает реальную боль: две библиотеки объявили
`distance` — без namespace при их совместной линковке будет конфликт.

```cpp
namespace geometry {
    struct Point { double x, y; };
    double distance(Point a, Point b);   // полное имя: geometry::distance
}
geometry::Point p{0, 0};
geometry::distance(p, q);
```

**Как работает поиск имён.** Когда ты пишешь `distance(p, q)`, компилятор ищет имя по
правилам *неквалифицированного поиска*: в текущем namespace, затем во включающих, затем
в глобальном — и **останавливается на первом**, где имя нашлось. Плюс работает
**ADL (argument-dependent lookup)**: для вызова с аргументами компилятор дополнительно
заглядывает в namespace’ы, к которым принадлежат типы аргументов. Поэтому `distance(p,
q)` с `p` типа `geometry::Point` найдёт `geometry::distance` даже без квалификации —
очень удобно, но и источник сюрпризов, когда «не та» перегрузка подхватилась.

**Вложенные namespaces** строят иерархию. Имя `metric` внутри `geometry` — это
`geometry::metric`:
```cpp
namespace geometry {
    double distance(Point, Point);            // евклидово
    namespace metric { double distance(Point, Point); }  // манхэттенское
}
// теперь distance живёт в ДВУХ местах — geometry::distance и geometry::metric::distance
```

**`using`: удобство против ясности.** Три инструмента, разной «силы»:

```cpp
using geometry::Point;            // using-declaration: вносит ОДНО имя Point
using ns = geometry::metric;      // namespace-alias: короткий псевдоним для длинного пути
using namespace geometry;         // using-directive: вносит ВСЕ имена — самый опасный
```

- `using namespace ...;` в **заголовке** — почти всегда ошибка: он «протекает» на каждую
  TU, включившую заголовок, тащит за собой все имена и ломает ADL/перегрузки в чужом
  коде. В `.cpp` локально — допустимо.
- При **конфликте имён** (два видимых `distance`) неквалифицированный вызов станет
  неоднозначным (`ambiguous`) — компилятор не угадывает. Лечится **квалификацией**:
  пиши полное имя `geometry::metric::distance(...)`, и вопрос снят.

**Анонимный (безымянный) namespace** — замена устаревшему `static` для функций уровня
файла: всё внутри получает *внутреннее связывание* и видно только в своей TU. Идеально
для приватных хелперов реализации, чтобы не засорять глобальные символы и не словить
`multiple definition`:
```cpp
// shapes.cpp
namespace { double sq(double v){ return v*v; } }   // виден только в этом .cpp
```

**Частые ошибки:**
- `using namespace std;` в заголовке → конфликты вроде вашего `distance`/`std::distance`.
- Полагаться на ADL там, где видны две перегрузки → `ambiguous`; решает квалификация.
- Думать, что `namespace` влияет на время жизни/память. Нет — это **только** про имена.

## Идея 5. Связывание (linkage): кто видит символ и откуда

Каждое имя в C++ имеет **связывание** (linkage) — правило о том, сколько
объектников разделяют один и тот же символ.

### Три вида связывания

| Связывание         | Что означает                                                   | Примеры                                     |
|--------------------|----------------------------------------------------------------|---------------------------------------------|
| **External**       | Символ виден из любой TU всей программы; все TU ссылаются на одно определение | обычные функции, глобальные переменные, объявленные в `.hpp` |
| **Internal**       | Символ существует только в своей TU; другие TU его не видят   | `static` функция/переменная в TU, всё внутри анонимного `namespace` |
| **No linkage**     | Имя вообще не является символом в таблице экспортов           | локальные переменные, параметры функций, `typedef`/`using` внутри функции |

### Что имеет какое связывание по умолчанию

| Конструкция                                           | Связывание      |
|-------------------------------------------------------|-----------------|
| Обычная функция на уровне TU / глобальная переменная  | External        |
| `const`-переменная на уровне TU (без `extern`)        | Internal        |
| `constexpr` переменная / функция на уровне TU         | Internal        |
| `inline` функция/переменная                           | External (особый: каждая TU имеет копию определения, линкер схлопывает) |
| Шаблон (функция, класс, переменная)                   | External        |
| `static` функция или переменная на уровне TU          | Internal        |
| Всё внутри `namespace { }` (анонимного)               | Internal        |
| Локальная переменная внутри функции                   | No linkage      |

**Почему `const` в TU получает Internal linkage.** Стандарт делает это намеренно:
когда ты пишешь `const double PI = 3.14;` в заголовке, его включают десятки TU —
и каждая получает *собственную* копию константы. Без Internal linkage это было бы
нарушением ODR. С Internal linkage каждая TU имеет свой символ, не конкурирующий с
другими — ни конфликта, ни разделяемого адреса.

### `static` на уровне TU vs анонимный `namespace`

Оба дают Internal linkage, но механики разные:

```cpp
// Способ 1: static — C-унаследованный, до сих пор корректен
static double helper(double x) { return x * x; }

// Способ 2: анонимный namespace — предпочтительный в C++
namespace {
    double helper(double x) { return x * x; }
    struct Cache { /* ... */ };   // классы и типы тоже
}
```

Почему анонимный `namespace` лучше для C++:
- Применим к **любым** объявлениям: функциям, переменным, классам, шаблонам.
  `static` нельзя применить к определению класса или шаблонному параметру.
- Явно выражает намерение «приватный модуль»: читатель кода сразу видит границу.
- Поддерживает **вложенность**: внутри анонимного `namespace` можно открыть
  именованный подпространство, и оно тоже будет иметь Internal linkage.

`static` на уровне TU сохраняется в языке из соображений совместимости с C.

### Практический совет

Каждый «приватный хелпер» в `.cpp` — кандидат на анонимный `namespace`.
Это предотвращает «утечку» символа в глобальную таблицу и исключает
`multiple definition` при сборке нескольких TU в библиотеку.

```cpp
// shapes.cpp
#include "geometry/shapes.hpp"
#include <cmath>

namespace {
    // sq виден только в этой TU — не конфликтует ни с кем
    double sq(double v) { return v * v; }
}

namespace geometry {
    double distance(Point a, Point b) {
        return std::sqrt(sq(a.x - b.x) + sq(a.y - b.y));
    }
}
```

## Идея 6. Include guards vs `#pragma once`

Заголовочный файл часто включается в одну TU **несколько раз** транзитивно:
`main.cpp` включает `polygon.hpp`, который включает `shapes.hpp` —  и ещё `main.cpp`
сам включает `shapes.hpp`. Без защиты компилятор увидит два объявления `struct Point`,
что нарушает правило «одного определения в одной TU» и вызовет ошибку компиляции.

### Традиционный include guard

```cpp
// shapes.hpp
#ifndef GEOMETRY_SHAPES_HPP   // 1) Если макрос НЕ определён — открываемся
#define GEOMETRY_SHAPES_HPP   // 2) Определяем макрос (при первом проходе)

struct Point { double x, y; };
double distance(Point a, Point b);

#endif // GEOMETRY_SHAPES_HPP // 3) При повторном включении шаг 1 → false, тело пропускается
```

**Механика:** препроцессор — текстовый; он не знает о файлах, только о макросах.
При первом `#include "shapes.hpp"` макрос `GEOMETRY_SHAPES_HPP` ещё не определён →
всё тело обрабатывается, макрос определяется. При втором включении (из другого
заголовка в той же TU) макрос уже определён → `#ifndef` не выполняется, тело
полностью пропускается. Это **гарантированно работает** на любом компиляторе
С/С++: механизм — часть стандарта.

**Соглашение по имени:** `<ORG>_<MODULE>_<FILE>_HPP` или аналогично. Должно быть
**уникальным во всей кодовой базе** (и в сторонних библиотеках!) — иначе два разных
заголовка с одним guard-макросом будут взаимно блокировать друг друга.

### `#pragma once`

```cpp
#pragma once   // одна строка вместо трёх директив

struct Point { double x, y; };
```

`#pragma once` — нестандартная директива (не в ISO C++), но поддерживается всеми
распространёнными компиляторами (GCC, Clang, MSVC). Компилятор запоминает *путь к
файлу* и пропускает повторное включение на уровне файловой системы.

**Почему нестандартна и где ломается:**

1. **Нет гарантии по стандарту.** Экзотические или устаревшие компиляторы (keil,
   iar, некоторые embedded-цепочки) могут не поддерживать её или игнорировать.
2. **Проблема с символическими ссылками.** Если одному физическому файлу
   соответствуют два пути (hardlink или symlink), компилятор может посчитать их
   разными файлами и включить дважды — guard на основе пути не сработает.
3. **Нет переносимости по стандарту** — если проект обязан компилироваться в
   строго соответствующем режиме, `#pragma once` невалидна (хотя компилятор обычно
   лишь предупредит или молча поддержит).

| Критерий                         | `#ifndef / #define / #endif`   | `#pragma once`          |
|----------------------------------|--------------------------------|-------------------------|
| Стандарт ISO C++                 | да                             | нет (расширение)        |
| Поддержка в GCC/Clang/MSVC       | да                             | да                      |
| Риск при symlink/hardlink        | нет (макрос, не путь)          | возможен                |
| Уникальность имени макроса       | нужна, иначе конфликт          | не нужна                |
| Лаконичность                     | 3 строки                       | 1 строка                |

В этом модуле используется `#pragma once` — допустимый прагматичный выбор для
проекта под GCC/Clang/MSVC. В кросс-компиляторных или стандарт-строгих проектах —
выбирай традиционный guard.

## Идея 7. Forward declarations глубже: неполный тип и pimpl

**Предобъявление** (forward declaration) вводит имя типа или функции без полного
определения:

```cpp
class Engine;             // «Engine существует» — компилятор знает имя, не знает размер
double distance(Point, Point);   // функция без тела
```

### Когда хватает неполного типа

Компилятор требует полного определения типа только там, где ему нужен *размер* или
*члены*. Во многих позициях достаточно просто знать, что тип существует:

| Позиция                                 | Полное определение нужно? |
|-----------------------------------------|---------------------------|
| Указатель `Engine*` / ссылка `Engine&`  | Нет — размер указателя всегда 8 байт |
| Значение `Engine` (поле, локальная переменная) | Да — нужен размер для выделения памяти |
| Аргумент / возвращаемое значение функции | Нет в объявлении; да — в вызове и теле определения |
| `sizeof(Engine)`, обращение к члену `e.field` | Да |
| Наследование `class A : public Engine`  | Да |

```cpp
// widget.hpp
class Engine;            // forward declaration — не нужен #include "engine.hpp"
class Widget {
    Engine* engine_;     // ✅ указатель — неполный тип OK
public:
    void run();          // объявление — тело в widget.cpp
};

// widget.cpp
#include "engine.hpp"    // полное определение нужно лишь в реализации
void Widget::run() { engine_->start(); }  // доступ к члену — нужен полный тип
```

### Как это рвёт зависимости и ускоряет сборку

Предположим, у тебя 50 TU включают `widget.hpp`. Если `widget.hpp` содержит
`#include "engine.hpp"`, то изменение `engine.hpp` **вынуждает перекомпилировать
все 50 TU**. Если `widget.hpp` вместо этого использует `class Engine;`, зависимость
обрывается: только `widget.cpp` включает `engine.hpp` и перекомпилируется при его
изменении. 49 остальных TU — нет.

Этот принцип — **минимальные зависимости в заголовках** — важнейший рычаг ускорения
сборки в больших проектах.

### Pimpl — «указатель на реализацию»

Pimpl (pointer to implementation) — паттерн, доводящий ту же идею до предела:
весь **приватный** интерфейс класса переносится в отдельную структуру `Impl`,
которую клиент никогда не видит даже по имени.

```cpp
// widget.hpp  — видит пользователь
class Widget {
public:
    Widget();
    ~Widget();               // деструктор в .cpp (полный тип Impl там известен)
    void run();
private:
    struct Impl;             // forward declaration приватной реализации
    Impl* impl_;             // указатель — неполный тип OK
};

// widget.cpp  — виден только нам
#include "widget.hpp"
#include "engine.hpp"        // только здесь

struct Widget::Impl {        // полное определение скрыто в .cpp
    Engine engine;
    int cache = 0;
};

Widget::Widget() : impl_(new Impl{}) {}
Widget::~Widget() { delete impl_; }
void Widget::run() { impl_->engine.start(); }
```

**Плюсы pimpl:**
- Изменение приватных полей/хелперов → меняем только `.cpp`, не трогаем `.hpp` →
  перекомпилируется только одна TU; у пользователей **стабильный ABI**.
- Пользователь не может случайно завязаться на приватные детали.

**Минусы pimpl:**
- Динамическая аллокация каждого объекта (`new Impl`) → дополнительный heap-alloc.
- Двойное разыменование (`this → impl_ → engine`) при каждом вызове.
- Правило пяти: нужно вручную управлять копированием/переносом или явно удалить их.
- В современном C++ `std::unique_ptr<Impl>` вместо сырого указателя — но тогда
  деструктор **обязан** быть в `.cpp`, где `Impl` полон.

Pimpl применяют в коммерческих библиотеках (Qt, Win32 API, ряд STL-реализаций) —
именно ради стабильного ABI и быстрой сборки у пользователей.

### Ограничения предобъявлений

- Нельзя предобъявить **псевдоним** (`using MyVec = std::vector<int>;`) — нужен полный заголовок.
- Нельзя предобъявить **члены namespace std**: `std::vector`, `std::string` и т.д.
  Стандарт не позволяет добавлять объявления в `std` — всегда используй
  `#include <vector>` и т.п.
- Предобъявление типа и его полное определение обязаны согласоваться токен-в-токен;
  если ты случайно объявишь `class Foo;` там, где это `struct Foo`, на некоторых
  платформах получишь предупреждение (технически это UB по стандарту).

## Идея 8. C++20 Modules (обзор — только теория)

### Проблемы `#include`-модели

За последние тридцать лет `#include` накопил системные недостатки:

1. **Порядок и макро-протечки.** `#include` — тупая текстовая вставка. Если один
   заголовок определяет макрос (`#define min(a,b) ...`), он «загрязняет» любой
   последующий заголовок в той же TU. Классический пример — `<windows.h>` с его
   `min`/`max`, ломающий `std::min`/`std::max`.
2. **Масштаб TU.** После обработки всех `#include` одна TU может содержать сотни
   тысяч строк (стандартные библиотеки, Boost). Компилятор разбирает их заново для
   каждой TU — отсюда многоминутные сборки.
3. **Нет инкапсуляции.** Всё, что написано в заголовке, видно пользователю
   библиотеки. Нет способа «скрыть» имена внутри библиотеки на уровне языка
   (анонимный `namespace` помогает только внутри `.cpp`).
4. **Нет семантической изоляции.** Два заголовка в одной TU влияют друг на друга
   через макросы и ODR.

### Базовый синтаксис C++20 Modules

```cpp
// math.ixx  — файл модульного интерфейса (расширение varies: .ixx, .cppm, .mpp)
export module math;          // объявление имени модуля

import <cmath>;              // import вместо #include — изолирован

export double distance(double dx, double dy) {
    return std::sqrt(dx * dx + dy * dy);
}

// Внутренняя функция — НЕ экспортируется: пользователь её не видит
double sq(double v) { return v * v; }
```

```cpp
// main.cpp
import math;                 // скомпилированный модуль — не текстовая вставка

int main() {
    return static_cast<int>(distance(3.0, 4.0));   // 5.0
}
```

**Что modules решают:**

| Проблема `#include`               | Решение в modules                                      |
|-----------------------------------|--------------------------------------------------------|
| Повторная компиляция тела         | Модуль компилируется один раз → **BMI** (binary module interface), переиспользуется |
| Порядок включения + макро-протечки | `import` не распространяет макросы; порядок не важен  |
| Нет инкапсуляции                  | Только `export`-декларации видны снаружи; остальное приватно модулю |
| Медленная сборка                  | BMI + параллельная компиляция → в теории в разы быстрее |

**Текущее состояние (C++20/23):** синтаксис стандартизирован, поддержка в GCC 14,
Clang 17, MSVC 19.34 — неполная и несовместимая между компиляторами по деталям
тулчейна и CMake-интеграции. Для production — пока рано; для учебных экспериментов
с одним компилятором — уже работает. В этом курсе мы используем традиционную
`#include`-модель, но знать синтаксис modules уже нужно.

## Идея 9. Виды библиотек: статическая, динамическая, header-only

### Статическая библиотека (`.a` / `.lib`)

Архив объектных файлов. При сборке приложения линкер **копирует** нужные `.o` из
архива в итоговый исполняемый файл.

```cmake
add_library(geometry STATIC src/shapes.cpp src/polygon.cpp)
```

- **Дистрибуция:** всё внутри одного бинарника. Никаких зависимостей в рантайме.
- **Размер бинарника:** больше (код библиотеки встроен). Если сто программ линкуют
  одну статическую библиотеку — каждая несёт свою копию в памяти при запуске.
- **Пересборка:** изменение библиотеки требует перелинковки приложения.
- **Когда выбирать:** утилиты/инструменты, где важна самодостаточность; embedded.

### Разделяемая библиотека (`.so` / `.dll` / `.dylib`)

Отдельный файл, загружаемый в память при старте процесса (или по запросу). Линкер
оставляет в бинарнике лишь **ссылки** на символы; операционная система подгружает
реализацию из `.so`-файла.

```cmake
add_library(geometry SHARED src/shapes.cpp src/polygon.cpp)
```

- **Дистрибуция:** `.so` / `.dll` нужны рядом с исполняемым файлом (или в системных
  путях). Если библиотека отсутствует — приложение не запустится.
- **Размер бинарника:** меньше; одна копия библиотеки в памяти обслуживает все
  использующие её процессы (memory mapping).
- **Обновление без перелинковки:** меняешь `.so` → все зависимые приложения
  обновляются при следующем запуске, не нужно пересобирать их.
- **Скорость загрузки:** чуть медленнее старт (dynamic linker разрешает символы).
- **ABI-стабильность:** нужно следить за ABI: добавил виртуальный метод или поменял
  размер класса — старые приложения сломаются при загрузке новой `.so`.
- **Когда выбирать:** плагины, большие shared-библиотеки ОС (`libc.so`), когда много
  процессов разделяют один и тот же код.

### Header-only библиотека

Весь код — в заголовочных файлах (`.hpp`). Никаких `.cpp` и никакого `add_library`.
Пользователь просто `#include` нужный заголовок.

```cpp
// geometry.hpp  — всё определено inline/template
#pragma once
#include <cmath>
namespace geometry {
    inline double distance(double dx, double dy) {
        return std::sqrt(dx * dx + dy * dy);
    }
}
```

- **Дистрибуция:** один каталог с заголовками — скопировал и подключил. Никакой
  предварительной компиляции.
- **Плата:** определение подключается в *каждую* TU, где встречается `#include`.
  Для больших реализаций это замедляет сборку (парсим одно и то же снова и снова).
- **Когда уместно:** небольшие утилитарные библиотеки; шаблонные библиотеки (где
  всё тело и так обязано быть в заголовке); библиотеки, которым важна
  распространяемость без системы сборки (single-header, как stb_image.h).
- Примеры из экосистемы: nlohmann/json (один `.hpp`), GLM, многие STB-библиотеки.

### Сводная таблица

| Параметр                     | Static `.a`        | Shared `.so/.dll`     | Header-only        |
|------------------------------|--------------------|-----------------------|--------------------|
| Рантайм-зависимость          | нет                | да (нужен файл)       | нет                |
| Размер бинарника             | больше             | меньше                | зависит от include |
| Обновление без пересборки    | нет                | да                    | нет                |
| Время компиляции             | быстрее (1× build) | быстрее (1× build)    | медленнее (N× include) |
| ABI-стабильность             | не нужна           | критична               | не нужна           |
| Простота использования       | средняя            | средняя                | максимальная       |

**В этом модуле** мы используем `STATIC` — самый простой вариант без рантайм-зависимостей,
идеальный для учебного проекта.

## Идея 10. CMake: библиотеки, приложения, линковка

В этом модуле `CMakeLists.txt` написан явно и прокомментирован — **прочитай его**. Ключевое:

```cmake
# 1) Собрать библиотеку из исходников:
add_library(geometry STATIC src/shapes.cpp src/polygon.cpp)
target_include_directories(geometry PUBLIC include)   # где её заголовки

# 2) Приложение, которое ПОЛЬЗУЕТСЯ библиотекой:
add_executable(geometry_app app/main.cpp)
target_link_libraries(geometry_app PRIVATE geometry)  # слинковать с библиотекой

# 3) Тесты — тоже линкуются с библиотекой и с gtest:
add_executable(... tests/...)
target_link_libraries(... PRIVATE geometry GTest::gtest_main)
```
`PUBLIC`/`PRIVATE` — кому «передаётся» свойство: `PUBLIC include` у библиотеки значит,
что её заголовки видны всем, кто с ней линкуется. Это «современный CMake» — мыслим
**целями (targets)** и их свойствами, а не глобальными флагами.

### Задание 4 — разрешение имён (using vs полное имя)

Теперь `distance` живёт в **двух** местах: `geometry::distance` (евклидово) и
`geometry::metric::distance` (манхэттенское, L1). Файлы: `include/geometry/shapes.hpp`
(объявления уже есть) и `src/shapes.cpp` (пишешь тела).

1. Реализуй `geometry::metric::distance(a, b)` — манхэттенское расстояние
   `|a.x - b.x| + |a.y - b.y|`.
2. Реализуй `geometry::euclid_plus_manhattan(a, b)` — **сумму** евклидова и
   манхэттенского расстояний. Внутри тела видны **оба** имени `distance`: один лежит в
   `geometry`, другой — в `geometry::metric`. Неквалифицированный вызов `distance(a,b)`
   здесь сам по себе укажет на `geometry::distance`, но манхэттенскую версию придётся
   позвать осознанно. Сделай выбор имён явным.

Цель — почувствовать, что компилятор **не угадывает**: ты сам решаешь, какое из
одноимённых `distance` зовёшь, через полное имя или `using`.

### Задание 5 — мини-библиотека `textstats` (тестируем публичный API)

Отдельная маленькая «библиотека» в **своём** пространстве имён `textstats`: интерфейс в
`include/geometry/textstats.hpp`, реализация в `src/textstats.cpp`. Она про строки и
никак не связана с `geometry` — это и есть смысл namespace: два независимых модуля в
одной сборке не мешают друг другу. Реализуй её публичный API:

1. `int word_count(std::string_view text)` — число слов (групп непробельных символов,
   разделённых пробелами/`\t`/`\n`). Пустая строка и строка из одних пробелов → `0`.
2. `int longest_word(std::string_view text)` — длина самого длинного слова (в байтах);
   пустой ввод → `0`.
3. `std::string to_upper(std::string_view text)` — копия строки, где ASCII-буквы
   переведены в верхний регистр; цифры/знаки/не-ASCII не трогаем. Исходник не меняем.

Тесты в `tests/test_geometry.cpp` дёргают именно публичный API через
`textstats::word_count(...)` и т.д. — так и проверяют библиотеку «снаружи», как это
делал бы её пользователь.

<details><summary>perimeter использует distance</summary>
Пройди по точкам и складывай `distance(pts[i], pts[i+1])`, а в конце добавь

(`pts[(i+1) % n]`) поможет красиво замкнуть круг.
</details>

<details><summary>Задание 4: как позвать обе версии distance (нудж)</summary>
В теле `euclid_plus_manhattan` ты находишься внутри `geometry`, поэтому короткое
`distance(a, b)` уже видит евклидову версию. Манхэттенская — на уровень глубже, в
`metric`. Подумай, каким именем до неё дотянуться, чтобы компилятор не запутался.
</details>

<details><summary>Задание 4: ещё конкретнее</summary>
Самый честный способ — обе вызвать полным именем: `geometry::distance(a, b)` и
`geometry::metric::distance(a, b)`, и вернуть их сумму. Альтернатива — `using` для
одного из имён, но тогда следи, чтобы второй вызов не стал неоднозначным. Полное имя
всегда снимает вопрос.
</details>

<details><summary>Задание 5: word_count / longest_word (нудж)</summary>
Идея одна на обе функции: иди по символам и держи флаг «я сейчас внутри слова».
Граница слова — переход «пробел → не-пробел» (тут начинается новое слово) и наоборот.
`std::isspace((unsigned char)ch)` определит пробел/таб/перевод строки. Для
`longest_word` веди счётчик текущего слова и обновляй максимум на границах И в самом
конце строки (последнее слово может не заканчиваться пробелом).
</details>

<details><summary>Задание 5: to_upper (нудж)</summary>
Сделай копию `std::string r{text};` и пройди по её символам, заменяя каждый на
`std::toupper`. Приводи аргумент к `unsigned char`, иначе на символах со старшим битом
поведение неопределённое: `std::toupper((unsigned char)ch)`. Не-буквы `toupper` и так
вернёт без изменений.
</details>
