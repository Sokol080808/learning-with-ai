# Модуль 11 — Многофайловые проекты, библиотеки и CMake

До сих пор CMake был «чёрным ящиком». Пора заглянуть внутрь. Большая идея: реальные
проекты состоят из **библиотек** (переиспользуемый код), **приложений** (точка входа) и
**тестов** — и всё это связывает система сборки. Junior должен уметь читать и писать
простой `CMakeLists.txt`.

## Идея 1. Из чего собирается C++-программа: единицы трансляции

Чтобы понять, зачем «много файлов», нужно знать, **как** компилятор C++ превращает
текст в исполняемый файл. Это конвейер из нескольких независимых стадий:

1. **Препроцессор.** Берёт один `.cpp`, рекурсивно вставляет тела всех `#include` и
   раскрывает макросы. На выходе — большой плоский текст без директив. Этот текст
   называется **единицей трансляции** (translation unit, TU). Ключевое: каждый `.cpp` —
   отдельная TU, и компилятор видит ровно её, ничего не зная про другие `.cpp`.
2. **Компиляция.** Каждая TU превращается в **объектный файл** (`.o`/`.obj`) —
   машинный код плюс таблица символов. Если в TU вызвана функция, объявленная, но не
   определённая здесь, компилятор оставляет в таблице **неразрешённую ссылку** (`U`).
3. **Линковка.** Линкер берёт все `.o` и библиотеки и **сшивает** ссылки: для каждой
   неразрешённой ссылки ищет определение в других объектниках. Нашёл ровно одно —
   хорошо; ноль — `undefined reference`; два — `multiple definition`.

Почему это важно для раскладки по файлам:

- **Скорость инкрементальной сборки.** Изменил один `.cpp` — пересобирается только его
  TU (один `.o`), затем быстрая линковка. Если бы весь код лежал в одном файле, любая
  правка пересобирала бы всё. Компиляция — самая дорогая стадия, поэтому дробление на
  TU экономит минуты на больших проектах.
- **Изоляция интерфейса от реализации.** Пользователю библиотеки нужен только `.hpp`
  (что вызывать), а не `.cpp` (как оно сделано). Можно поменять реализацию, не трогая
  заголовок, — и зависимые TU даже не нужно перекомпилировать (только перелинковать).
- **Переиспользование.** Один объектник/одну библиотеку линкуют сразу к приложению и к
  тестам — пишем реализацию один раз.

**Главная ловушка новичка:** «я подключил `#include "foo.cpp"` и всё заработало». Так
делать нельзя. `#include` — это тупая текстовая вставка. Включив `.cpp` в две TU, ты
получишь два определения одной функции → `multiple definition` при линковке. Правильно:
включаем только **заголовки** (объявления), а `.cpp` компилируем отдельно и линкуем.

```cpp
// foo.hpp  — объявление (можно включать сколько угодно раз)
int add(int a, int b);

// foo.cpp  — определение (компилируется как отдельная TU)
#include "foo.hpp"
int add(int a, int b) { return a + b; }

// main.cpp
#include "foo.hpp"        // НЕ "foo.cpp"
int main() { return add(2, 3); }
```

## Идея 2. Объявление vs определение и ODR

Два разных понятия, которые новички путают:

- **Объявление (declaration)** обещает компилятору, что такая-то сущность где-то
  существует, и описывает её *тип/сигнатуру*. Объявлений может быть **сколько угодно**.
  `double distance(Point, Point);` — объявление функции.
- **Определение (definition)** даёт сущности «тело» / выделяет место. Функция с `{...}`,
  переменная с памятью, класс с телом `{...}`.

**One Definition Rule (ODR)** — два правила в одном:

1. В пределах **одной TU** сущность нельзя определить дважды.
2. Во **всей программе** у функции/глобальной переменной должно быть ровно **одно**
   определение (для inline-сущностей — одно *на TU*, но все определения обязаны быть
   идентичны — токен в токен).

Почему это техническая, а не вкусовая проблема. Линкер видит **символы**. Обычная
функция компилируется в символ с *внешним связыванием* (external linkage): он попадает
в таблицу экспортов объектника. Если такой символ экспортируют два объектника — линкер
не знает, какой выбрать, и падает с `multiple definition`.

```cpp
// util.hpp
int square(int x) { return x * x; }   // ❌ ОПРЕДЕЛЕНИЕ в заголовке!
```
Включили `util.hpp` в `a.cpp` и `b.cpp` — оба объектника экспортируют символ `square` →
ошибка линковки. **Как чинить — три способа, выбор по ситуации:**

```cpp
// 1) Объявление в .hpp, определение в одном .cpp  — для обычных функций.
int square(int x);                 // util.hpp
int square(int x){ return x*x; }   // util.cpp

// 2) inline — определение в заголовке разрешено; линкер схлопнет копии в одну.
inline int square(int x){ return x*x; }   // util.hpp, можно включать везде

// 3) шаблон/constexpr — у них тело обязано быть в заголовке (нужно при инстанцировании).
template <class T> T square(T x){ return x*x; }
```

Почему `inline` спасает: ключевое слово `inline` (в этом контексте — *не* про
встраивание кода!) говорит линкеру «таких определений будет много копий, они
идентичны, оставь одну, остальные выброси». Шаблоны и `constexpr`-функции `inline` по
умолчанию — поэтому их тела и живут в заголовках (мы это видели в модулях 07 и 10).

**Частая ошибка:** глобальная переменная в заголовке.
```cpp
int counter = 0;            // ❌ в .hpp → multiple definition
inline int counter = 0;     // ✅ C++17 inline-переменная: одна на программу
// либо: extern int counter; в .hpp + int counter = 0; в одном .cpp
```

## Идея 3. Разделение интерфейс / реализация на практике

«Интерфейс» — это контракт: имена, сигнатуры, типы, которые видит пользователь.
«Реализация» — как именно это работает. Разделение даёт три выгоды.

**1. Время сборки.** Заголовок включается в десятки TU. Каждое изменение `.hpp`
заставляет перекомпилировать **все** включающие его TU. Изменение `.cpp` — только его
одну TU. Поэтому держи в заголовке минимум: объявления, а не реализацию.

**2. Сокрытие деталей.** Пользователю не нужно (и вредно) видеть, *как* посчитан
периметр. Если в `.hpp` утекают приватные хелперы, любой может на них завязаться, и ты
не сможешь их менять. Держи внутренние функции в `.cpp` (и/или в анонимном namespace —
см. Идею 4).

**3. Стабильный ABI/контракт.** Меняешь тело в `.cpp` — зависимые TU **не нужно
перекомпилировать**, только перелинковать. Это и есть смысл библиотеки.

Что класть куда:

| В заголовок (`.hpp`)                         | В исходник (`.cpp`)                       |
|----------------------------------------------|-------------------------------------------|
| объявления функций                           | тела обычных функций                      |
| определения классов/структур (нужен размер)  | методы класса (можно вынести из тела)     |
| `inline`/`constexpr`/шаблоны (тела)          | внутренние хелперы (анонимный namespace)  |
| публичные `using`/типы/константы (`inline`)  | глобальные переменные с определением       |

Минимизируй и сами `#include` в заголовке: что нужно только реализации — включай в
`.cpp`. Если в заголовке тип используется лишь как `T&`/`T*`, хватит **предобъявления**
`class T;` вместо `#include` — это рвёт лишние зависимости и ускоряет сборку.

```cpp
// widget.hpp
class Engine;                 // предобъявление — полного определения тут не нужно
class Widget {
    Engine* engine_;          // указатель — достаточно неполного типа
public:
    void run();               // объявление; тело в widget.cpp, где #include "engine.hpp"
};
```

## Идея 4. Namespaces — пространства имён

`namespace` группирует имена в логический «контейнер», чтобы они не сталкивались с
именами из других библиотек. Это решает реальную боль: две библиотеки объявили
`distance` — без namespace при их совместной линковке будет конфликт.

```cpp
namespace geometry {
    struct Point { double x, y; };
    double distance(Point a, Point b);   // полное имя: geometry::distance
}
geometry::Point p{0, 0};
geometry::distance(p, q);
```

**Как работает поиск имён.** Когда ты пишешь `distance(p, q)`, компилятор ищет имя по
правилам *неквалифицированного поиска*: в текущем namespace, затем во включающих, затем
в глобальном — и **останавливается на первом**, где имя нашлось. Плюс работает
**ADL (argument-dependent lookup)**: для вызова с аргументами компилятор дополнительно
заглядывает в namespace’ы, к которым принадлежат типы аргументов. Поэтому `distance(p,
q)` с `p` типа `geometry::Point` найдёт `geometry::distance` даже без квалификации —
очень удобно, но и источник сюрпризов, когда «не та» перегрузка подхватилась.

**Вложенные namespaces** строят иерархию. Имя `metric` внутри `geometry` — это
`geometry::metric`:
```cpp
namespace geometry {
    double distance(Point, Point);            // евклидово
    namespace metric { double distance(Point, Point); }  // манхэттенское
}
// теперь distance живёт в ДВУХ местах — geometry::distance и geometry::metric::distance
```

**`using`: удобство против ясности.** Три инструмента, разной «силы»:

```cpp
using geometry::Point;            // using-declaration: вносит ОДНО имя Point
using ns = geometry::metric;      // namespace-alias: короткий псевдоним для длинного пути
using namespace geometry;         // using-directive: вносит ВСЕ имена — самый опасный
```

- `using namespace ...;` в **заголовке** — почти всегда ошибка: он «протекает» на каждую
  TU, включившую заголовок, тащит за собой все имена и ломает ADL/перегрузки в чужом
  коде. В `.cpp` локально — допустимо.
- При **конфликте имён** (два видимых `distance`) неквалифицированный вызов станет
  неоднозначным (`ambiguous`) — компилятор не угадывает. Лечится **квалификацией**:
  пиши полное имя `geometry::metric::distance(...)`, и вопрос снят.

**Анонимный (безымянный) namespace** — замена устаревшему `static` для функций уровня
файла: всё внутри получает *внутреннее связывание* и видно только в своей TU. Идеально
для приватных хелперов реализации, чтобы не засорять глобальные символы и не словить
`multiple definition`:
```cpp
// shapes.cpp
namespace { double sq(double v){ return v*v; } }   // виден только в этом .cpp
```

**Частые ошибки:**
- `using namespace std;` в заголовке → конфликты вроде вашего `distance`/`std::distance`.
- Полагаться на ADL там, где видны две перегрузки → `ambiguous`; решает квалификация.
- Думать, что `namespace` влияет на время жизни/память. Нет — это **только** про имена.

## Идея 5. CMake: библиотеки, приложения, линковка

В этом модуле `CMakeLists.txt` написан явно и прокомментирован — **прочитай его**. Ключевое:

```cmake
# 1) Собрать библиотеку из исходников:
add_library(geometry STATIC src/shapes.cpp src/polygon.cpp)
target_include_directories(geometry PUBLIC include)   # где её заголовки

# 2) Приложение, которое ПОЛЬЗУЕТСЯ библиотекой:
add_executable(geometry_app app/main.cpp)
target_link_libraries(geometry_app PRIVATE geometry)  # слинковать с библиотекой

# 3) Тесты — тоже линкуются с библиотекой и с gtest:
add_executable(... tests/...)
target_link_libraries(... PRIVATE geometry GTest::gtest_main)
```
`PUBLIC`/`PRIVATE` — кому «передаётся» свойство: `PUBLIC include` у библиотеки значит,
что её заголовки видны всем, кто с ней линкуется. Это «современный CMake» — мыслим
**целями (targets)** и их свойствами, а не глобальными флагами.

## Микро-проект: библиотека `geometry`

Структура (обрати внимание, как раскладывается код):
```
include/geometry/shapes.hpp     объявления: Point, distance, metric::distance, euclid_plus_manhattan
include/geometry/polygon.hpp    объявление: perimeter
include/geometry/textstats.hpp  интерфейс мини-«библиотеки» textstats (свой namespace)
src/shapes.cpp                  реализация distance, metric::distance, …  ← пишешь тут
src/polygon.cpp                 реализация perimeter                      ← пишешь тут
src/textstats.cpp               реализация textstats::*                   ← пишешь тут
app/main.cpp                    готовое приложение
tests/test_geometry.cpp         тесты
```

Реализуй (всё в пространстве имён `geometry`):
1. `double distance(Point a, Point b)` — евклидово расстояние
   `sqrt((ax-bx)^2 + (ay-by)^2)`.
2. `double perimeter(const std::vector<Point>& pts)` — сумма длин сторон замкнутого
   многоугольника (последняя точка соединяется с первой). Для < 2 точек — `0`.
   Используй `distance` из соседнего файла — это и есть переиспользование между
   единицами трансляции.

### Задание 4 — разрешение имён (using vs полное имя)

Теперь `distance` живёт в **двух** местах: `geometry::distance` (евклидово) и
`geometry::metric::distance` (манхэттенское, L1). Файлы: `include/geometry/shapes.hpp`
(объявления уже есть) и `src/shapes.cpp` (пишешь тела).

1. Реализуй `geometry::metric::distance(a, b)` — манхэттенское расстояние
   `|a.x - b.x| + |a.y - b.y|`.
2. Реализуй `geometry::euclid_plus_manhattan(a, b)` — **сумму** евклидова и
   манхэттенского расстояний. Внутри тела видны **оба** имени `distance`: один лежит в
   `geometry`, другой — в `geometry::metric`. Неквалифицированный вызов `distance(a,b)`
   здесь сам по себе укажет на `geometry::distance`, но манхэттенскую версию придётся
   позвать осознанно. Сделай выбор имён явным.

Цель — почувствовать, что компилятор **не угадывает**: ты сам решаешь, какое из
одноимённых `distance` зовёшь, через полное имя или `using`.

### Задание 5 — мини-библиотека `textstats` (тестируем публичный API)

Отдельная маленькая «библиотека» в **своём** пространстве имён `textstats`: интерфейс в
`include/geometry/textstats.hpp`, реализация в `src/textstats.cpp`. Она про строки и
никак не связана с `geometry` — это и есть смысл namespace: два независимых модуля в
одной сборке не мешают друг другу. Реализуй её публичный API:

1. `int word_count(std::string_view text)` — число слов (групп непробельных символов,
   разделённых пробелами/`\t`/`\n`). Пустая строка и строка из одних пробелов → `0`.
2. `int longest_word(std::string_view text)` — длина самого длинного слова (в байтах);
   пустой ввод → `0`.
3. `std::string to_upper(std::string_view text)` — копия строки, где ASCII-буквы
   переведены в верхний регистр; цифры/знаки/не-ASCII не трогаем. Исходник не меняем.

Тесты в `tests/test_geometry.cpp` дёргают именно публичный API через
`textstats::word_count(...)` и т.д. — так и проверяют библиотеку «снаружи», как это
делал бы её пользователь.

```bash
./cpp/run.sh 11

```

### Подсказки

<details><summary>perimeter использует distance</summary>
Пройди по точкам и складывай `distance(pts[i], pts[i+1])`, а в конце добавь
`distance(pts.back(), pts.front())`, чтобы замкнуть контур. Оператор `%` по индексу
(`pts[(i+1) % n]`) поможет красиво замкнуть круг.
</details>

<details><summary>Задание 4: как позвать обе версии distance (нудж)</summary>
В теле `euclid_plus_manhattan` ты находишься внутри `geometry`, поэтому короткое
`distance(a, b)` уже видит евклидову версию. Манхэттенская — на уровень глубже, в
`metric`. Подумай, каким именем до неё дотянуться, чтобы компилятор не запутался.
</details>

<details><summary>Задание 4: ещё конкретнее</summary>
Самый честный способ — обе вызвать полным именем: `geometry::distance(a, b)` и
`geometry::metric::distance(a, b)`, и вернуть их сумму. Альтернатива — `using` для
одного из имён, но тогда следи, чтобы второй вызов не стал неоднозначным. Полное имя
всегда снимает вопрос.
</details>

<details><summary>Задание 5: word_count / longest_word (нудж)</summary>
Идея одна на обе функции: иди по символам и держи флаг «я сейчас внутри слова».
Граница слова — переход «пробел → не-пробел» (тут начинается новое слово) и наоборот.
`std::isspace((unsigned char)ch)` определит пробел/таб/перевод строки. Для
`longest_word` веди счётчик текущего слова и обновляй максимум на границах И в самом
конце строки (последнее слово может не заканчиваться пробелом).
</details>

<details><summary>Задание 5: to_upper (нудж)</summary>
Сделай копию `std::string r{text};` и пройди по её символам, заменяя каждый на
`std::toupper`. Приводи аргумент к `unsigned char`, иначе на символах со старшим битом
поведение неопределённое: `std::toupper((unsigned char)ch)`. Не-буквы `toupper` и так
вернёт без изменений.
</details>

После того как заработает — открой `CMakeLists.txt` этого модуля и пройдись по нему
строка за строкой. Обрати внимание: когда добавился `src/textstats.cpp`, его пришлось
дописать в `add_library` — иначе линкер не нашёл бы определений `textstats::*`. Если
что-то непонятно в `add_library`/`target_link_libraries` — спрашивай, это важный навык
для junior.
