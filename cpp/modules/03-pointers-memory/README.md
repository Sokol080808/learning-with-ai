# Модуль 03 — Указатели, ссылки и модель памяти

Это сердце C++ и то, чем он отличается от Python/JS. Здесь ты впервые работаешь с
**памятью напрямую**. Будет непривычно — это нормально. Поняв этот модуль, ты поймёшь
половину «странностей» C++.

## Идея 1. Стек и куча

- **Стек (stack)** — где живут локальные переменные. Память выделяется автоматически
  при входе в функцию и освобождается при выходе. Быстро, но время жизни ограничено
  областью видимости `{ ... }`.
- **Куча (heap)** — память «по запросу», живёт пока ты её не освободишь. Гибко, но
  ответственность на тебе. В современном C++ кучу почти всегда трогают через
  контейнеры и умные указатели (модули 05–06), а не вручную.

```cpp
int x = 5;          // x на стеке, исчезнет в конце блока
int* p = new int(5); // значение на куче; p (сам указатель) — на стеке
delete p;            // обязательно освободить, иначе утечка
```
> В этом модуле мы НЕ пишем `new`/`delete` в заданиях — сначала надо понять сами
> указатели. К ручному управлению и почему его избегать вернёмся в модулях 05–06.

## Идея 2. Что такое указатель

Указатель — это переменная, которая хранит **адрес** другой переменной.

```cpp
int x = 42;
int* p = &x;   // & — «взять адрес». p указывает на x
std::cout << *p;   // * — «разыменование»: значение по адресу == 42
*p = 100;          // меняем x ЧЕРЕЗ указатель; теперь x == 100
```
Два разных `*`: в объявлении `int* p` звёздочка — часть типа («указатель на int»);
в выражении `*p` — операция разыменования. И `&` тоже двулик: в типе `int&` — ссылка,
в выражении `&x` — взятие адреса.

## Идея 3. `nullptr` — указатель «в никуда»

```cpp
int* p = nullptr;   // ни на что не указывает
if (p != nullptr) { /* безопасно разыменовывать */ }
```
**Разыменование `nullptr` — это краш (UB).** Всегда думай: может ли указатель быть
нулевым? Если да — проверяй перед `*p`.

## Идея 4. Массивы и арифметика указателей

Имя массива легко «распадается» в указатель на первый элемент:

```cpp
int a[3] = {10, 20, 30};
int* p = a;        // p указывает на a[0]
*(p + 1);          // == a[1] == 20   (p+1 сдвигает на ОДИН int, а не на 1 байт)
p[2];              // == a[2] == 30   (индексация — это и есть *(p+2))
```
Важно: «сырой» массив **не знает свой размер**. Поэтому в функцию его передают вместе
с длиной: `f(int* arr, int n)`. (Позже `std::vector`/`std::array` решат это за нас.)

## Идея 5. Указатель vs ссылка

| | Указатель `int*` | Ссылка `int&` |
|---|---|---|
| может быть «пустым» | да (`nullptr`) | нет, всегда привязана |
| можно перенаправить | да | нет, навсегда к одному объекту |
| синтаксис доступа | `*p`, `p->` | как обычная переменная |

Правило большого пальца: если значение **обязано** существовать и не меняет «адресата» —
бери ссылку (проще и безопаснее). Если может отсутствовать — указатель.

## Идея 6. const-correctness с указателями

```cpp
const int* p;        // нельзя менять *p (значение), можно менять p (куда смотрит)
int* const p = &x;   // можно менять *p, нельзя менять p
const int* const p;  // нельзя ни то, ни другое
```
Читай справа налево: `const int*` → «pointer to const int». `const&`-параметр —
твой друг: даёт доступ без копии и обещает не портить данные.

## Идея 7. Висячие указатели/ссылки (dangling)

```cpp
int* bad() {
    int local = 5;
    return &local;   // ОПАСНО: local умрёт, адрес станет мусором
}
```
Никогда не возвращай адрес/ссылку на локальную переменную. Это источник коварных багов.

## Идея 8. Стек vs куча — кто за что отвечает

До сих пор мы только читали указатели. Теперь — куда они могут смотреть.

```cpp
void demo() {
    int s = 5;             // СТЕК: память освобождается сама в конце demo()
    int* h = new int(5);   // КУЧА: живёт, пока ты сам не сделаешь delete
    delete h;              // забыл — утечка памяти
}
```
- **Стек** — дисциплина «последним пришёл, первым ушёл». Объект живёт ровно до конца
  своего `{ ... }`. Ничего освобождать руками не надо. Зато размер должен быть известен
  и нельзя «пережить» свою область видимости.
- **Куча** — выделяешь сам (`new`), освобождаешь сам (`delete`). Гибко: размер можно
  узнать в рантайме, объект живёт сколько нужно. Цена — **ответственность**: забудешь
  `delete` — утечка; освободишь дважды — краш.

Подводный камень: возврат указателя на стековую переменную (Идея 7) — баг именно потому,
что стековый объект исчезает, а куча — нет. Если данные должны «пережить» функцию, им место
на куче (или, по-взрослому, в контейнере/умном указателе).

## Идея 9. Арифметика указателей и распад массива

Имя массива в большинстве выражений **распадается** (decay) в указатель на первый элемент.
Прибавление к указателю считается **в элементах**, а не в байтах:

```cpp
int a[4] = {10, 20, 30, 40};
int* p = a;          // распад: p == &a[0]
p + 2;               // адрес a[2]; сдвиг на 2*sizeof(int) байт под капотом
*(p + 2);            // == 30
p[2];                // ровно то же самое: p[i] это *(p + i)
a + 4;               // адрес "за концом" — на него можно указывать, но НЕ разыменовывать
```
Разность двух указателей в один массив даёт **число элементов** между ними:
```cpp
int* q = &a[3];
q - p;               // == 3
```
Именно так устроен обход «через указатели»: держишь `begin` и `end` и идёшь `++p`, пока
`p != end`. Подводный камень: выход за границы (`p[4]` у массива из 4) — это UB, даже если
«вроде работает». Распад теряет размер: `sizeof(a)` это 16 байт, а `sizeof(p)` — размер
указателя, не массива.

## Идея 10. Парность `new`/`delete` и почему RAII лучше

Каждому выделению — ровно одно парное освобождение, и **формы должны совпадать**:

```cpp
int*  one  = new int(5);     // ↔ delete one;
int*  many = new int[5];     // ↔ delete[] many;   (квадратные скобки обязательны!)
```
Грубые ошибки:
- `new[]`, а освободил `delete` (без скобок) — UB.
- освободил, а потом ещё раз `delete` — **double free**, краш.
- забыл `delete` — утечка (в долгоживущей программе растёт бесконечно).
- после `delete p` оставил `p` и разыменовал — **висячий указатель**.

Ручное `new/delete` хрупко: любой `return` или исключение на полпути между `new` и `delete`
утечёт. Поэтому в современном C++ владение оборачивают в объект, чей **деструктор** сам
зовёт `delete` — это и есть **RAII** (Resource Acquisition Is Initialization):

```cpp
struct Buf {
    int* p;
    explicit Buf(int n) : p(new int[n]) {}
    ~Buf() { delete[] p; }          // освобождение привязано к концу жизни объекта
};
// уйдёт ли управление из функции через return или throw — деструктор всё равно сработает.
```
В заданиях ниже ты один раз напишешь такой класс руками (`DynArray`), чтобы прочувствовать
парность `new[]`/`delete[]`. А в модулях 05–06 увидишь, как `std::vector` и умные указатели
делают это за тебя — и почему «голый» `new` в нормальном коде почти не встречается.

## Идея 11. Глубокое vs поверхностное копирование

Если класс хранит **указатель** на кучу, копирование по умолчанию (побитовое) скопирует
сам указатель — и два объекта станут владеть **одним** буфером:

```cpp
struct Bad {
    int* p;
    explicit Bad(int v) : p(new int(v)) {}
    ~Bad() { delete p; }
    // компилятор сгенерит копирование, которое копирует p (адрес!) — это ПОВЕРХНОСТНАЯ копия
};
Bad a(1);
Bad b = a;     // b.p == a.p — один буфер на двоих
// в конце: ~b делает delete p, ~a делает delete того же p → DOUBLE FREE, краш
```
**Поверхностная (shallow)** копия делит указатель. **Глубокая (deep)** — выделяет копии
свой буфер и копирует *значение*:

```cpp
Good(const Good& o) : p(new int(*o.p)) {}   // свой new, своё значение — глубокая копия
```
Если класс владеет ресурсом и ты пишешь деструктор — почти наверняка нужны ещё и
копирующий конструктор, и копирующее присваивание (**правило трёх**). В присваивании не
забудь про self-assignment (`a = a`) и про освобождение старого буфера. Подводный камень
присваивания — утечь старым буфером или поделить новый. В задании 8 ты сделаешь копирование
`OwnedInt` глубоким; тесты проверяют, что у копии **другой** указатель и независимое значение.

<details><summary>Задание 6 — reverse через указатели</summary>
То же, что задание 5, но вместо индексов держи два указателя: <code>int* left = arr;</code>
и <code>int* right = arr + n - 1;</code>. Пока <code>left &lt; right</code> — меняй
<code>*left</code> и <code>*right</code> местами, потом <code>++left; --right;</code>.
Подумай, почему при <code>n == 0</code> и <code>n == 1</code> цикл просто не выполнится —
тогда отдельные проверки не нужны.
</details>
<details><summary>Задание 7 — DynArray: с чего начать</summary>
В конструкторе: если <code>n &gt; 0</code> — <code>data_ = new int[n];</code> и сохрани
<code>size_ = n</code>. Потом цикл по всем элементам, присваивая <code>fill</code>.
В деструкторе — ровно одно <code>delete[] data_;</code> (для <code>nullptr</code> это
безопасно). Сколько <code>new[]</code> — столько и <code>delete[]</code>.
</details>
<details><summary>Задание 7 — at и проверка границ</summary>
Сначала проверь <code>i &lt; 0 || i &gt;= size_</code> и в этом случае
<code>throw std::out_of_range("...");</code> (нужен <code>&lt;stdexcept&gt;</code>).
Иначе верни <code>data_[i]</code>. Неконстантная версия возвращает <code>int&amp;</code>
(чтобы можно было писать <code>a.at(0) = 7</code>), константная — <code>int</code> по
значению. Тела почти одинаковые — это нормально.
</details>
<details><summary>Задание 8 — глубокое копирование</summary>
Поверхностная копия просто скопировала бы указатель <code>ptr_</code> — и два объекта
делили бы один буфер (а в деструкторах — double free). Тебе нужен <b>свой</b> буфер:
в копирующем конструкторе выдели <code>new int(...)</code> и положи туда <i>значение</i>
оригинала, а не его адрес.
</details>
<details><summary>Задание 8 — operator= (если застрял на присваивании)</summary>
Шаги присваивания при владении сырым указателем: (1) защита от самоприсваивания —
если <code>this == &amp;other</code>, верни <code>*this</code> сразу; (2) перепиши
значение в СВОЙ уже выделенный буфер (здесь буфер всего один <code>int</code>, так что
переаллокация не нужна — достаточно скопировать значение); (3) <code>return *this;</code>.
Проверь на тесте <code>SelfAssignmentSafe</code>, что <code>a = a;</code> не ломает объект.
</details>
