# Модуль 02 — Управление потоком и функции

Большая идея: программа — это **решения** (что выполнять) и **повторения** (сколько
раз), упакованные в **функции** (переиспользуемые блоки с именем). Хорошая программа
читается как текст: маленькие функции с понятными именами, каждая делает одно дело.

## Идея 1. Ветвление: `if`, `else`, тернарный оператор, `switch`

```cpp
if (n > 0)        std::cout << "положительное";
else if (n < 0)   std::cout << "отрицательное";
else              std::cout << "ноль";

int sign = (n >= 0) ? 1 : -1;   // тернарный оператор: условие ? a : b

switch (op) {                   // switch — выбор по значению
    case '+': return a + b;
    case '-': return a - b;
    default:  return 0;         // если ничего не подошло
}
```
Важно про `switch`: после `case` без `break`/`return` выполнение «проваливается» в
следующий `case`. В этом курсе мы обычно `return` прямо из `case`.

## Идея 2. Циклы: `while`, `for`, range-based for

```cpp
for (int i = 0; i < n; ++i) { /* i от 0 до n-1 */ }

int i = 0;
while (i < n) { ++i; }

for (int x : {10, 20, 30}) { /* x по очереди = 10,20,30 */ }  // range-based for
```
`++i` (преинкремент) и `i++` (постинкремент) оба увеличивают `i`; в заголовке цикла
разницы по смыслу нет, привыкай писать `++i`.

## Идея 3. Функции: объявление и определение

```cpp
int square(int n);            // объявление (прототип): «такая функция есть»
int square(int n) { return n * n; }   // определение: тело
```
В заголовочном файле (`.hpp`) — **объявления**, в `.cpp` — **определения**. Так
несколько файлов могут пользоваться функцией, видя только её прототип. Это и есть
разделение интерфейса и реализации, на котором держатся большие проекты.

## Идея 4. Рекурсия

Функция может вызывать саму себя. Главное — **база** (когда остановиться):

```cpp
int factorial(int n) {
    if (n <= 1) return 1;          // база
    return n * factorial(n - 1);   // шаг к базе
}
```
Рекурсию почти всегда можно переписать циклом, и наоборот. Выбирай то, что **читается
понятнее** для конкретной задачи.

## Идея 5. Перегрузка и значения по умолчанию

Можно иметь несколько функций с одним именем, если различаются параметры —
компилятор выберет подходящую (**перегрузка**):

```cpp
int area(int side);          // квадрат
int area(int w, int h);      // прямоугольник
```
И задавать значения по умолчанию:
```cpp
int power(int base, int exp = 2);   // power(5) == 25
```

## Идея 6. `switch` и сознательное «проваливание»

`switch` сравнивает значение с метками `case` (только целые/`enum`/`char`) и прыгает к
совпавшей. Дальше выполнение идёт **подряд вниз**, пока не встретит `break` — это и есть
«проваливание» (fall-through):

```cpp
switch (day) {
    case 6:
    case 7:                       // обе метки ведут в один блок — это намеренно
        std::cout << "выходной";
        break;                    // без break ушли бы в default
    default:
        std::cout << "будни";
}
```
Иногда проваливание нужно (как выше — объединить случаи), но **случайно** забытый
`break` — классическая ошибка: число попадёт сразу в несколько веток. Когда хочешь
провалиться нарочно, оставь комментарий `// fall-through`, чтобы это читалось как
замысел, а не как баг. Подводный камень: внутри `case` нельзя без `{}` объявлять
переменную с инициализацией — заверни тело `case` в свои фигурные скобки.

## Идея 7. Тернарный оператор как *выражение*

`if` — это инструкция (что-то делает), а тернарный оператор `?:` — **выражение**
(имеет значение), поэтому его можно сразу присвоить или вернуть:

```cpp
int max2 = (a > b) ? a : b;            // одно из двух
int sign = (n == 0) ? 0 : (n > 0 ? 1 : -1);  // вложенный, но уже на грани читаемости
```
Сила в краткости, но не злоупотребляй: длинные вложенные `?:` читаются хуже честного
`if/else`. Подводный камень: обе ветки должны давать **совместимый тип** — `cond ? 1 :
"нет"` не скомпилируется, потому что `int` и `const char*` не сводятся к одному типу.

## Идея 8. Structured bindings в циклах

Если контейнер хранит пары (например `std::map` отдаёт `std::pair<const K, V>`),
**структурное связывание** `auto [k, v]` распаковывает элемент прямо в заголовке цикла —
не нужно писать `.first`/`.second`:

```cpp
std::map<std::string, int> ages{{"Аня", 20}, {"Боря", 19}};
for (const auto& [name, age] : ages) {
    std::cout << name << ": " << age << '\n';   // читается как обычные имена
}
```
Берём `const auto&`, чтобы не копировать каждый элемент. Подводный камень: число имён в
скобках должно **точно** совпадать с числом полей — для пары ровно два, для своей
структуры — сколько у неё членов.

## Идея 9. Инвариант цикла — главный инструмент мышления

**Инвариант** — это утверждение, истинное перед каждой итерацией. Если оно верно в
начале, не ломается за один проход тела и в конце даёт нужный ответ — цикл корректен.
Пример для суммы:

```cpp
int s = 0;
for (int i = 0; i < n; ++i)   // инвариант: s == сумма a[0..i-1]
    s += a[i];
// после цикла i == n, значит s == сумма a[0..n-1] — то, что нужно
```
Этот же приём — ключ к бинарному поиску: держи в голове «искомое, если оно есть, лежит
внутри `[lo, hi)`» и каждый шаг сужай отрезок, не нарушая это. Подводный камень для
бинарного поиска — границы: путаница `<` / `<=` и `mid ± 1` приводит к зацикливанию или
к промаху на один элемент. Сначала чётко выпиши инвариант, потом код.

## Идея 10. Оценка сложности (big-O) на пальцах

Big-O говорит, как растёт **число операций** с ростом размера входа `n` — грубо, без
констант. Линейный проход по `n` элементам — это `O(n)`; вложенные циклы по `n` — уже
`O(n²)`; деление пополам на каждом шаге — `O(log n)`.

```cpp
// O(n): один проход
for (int i = 0; i < n; ++i) use(i);

// O(n²): проход внутри прохода
for (int i = 0; i < n; ++i)
    for (int j = 0; j < n; ++j) use(i, j);
```
Почему это важно: при `n = 1 000 000` алгоритм `O(n)` сделает ~миллион шагов, а `O(n²)` —
триллион; разница между «мгновенно» и «не дождёшься». Поэтому линейный поиск (`O(n)`) по
большому отсортированному массиву проигрывает бинарному (`O(log n)`), а решето Эратосфена
(`O(n log log n)`) быстро отдаёт все простые, тогда как «проверять каждое число делением»
заметно медленнее. Подводный камень: big-O игнорирует константы, поэтому на **маленьких**
`n` «асимптотически худший» алгоритм бывает быстрее — это про поведение на росте.

### Подсказки

<details><summary>gcd — алгоритм Евклида</summary>
Пока `b != 0`: одновременно `a, b = b, a % b`. Когда `b` стал нулём — ответ в `a`.
</details>
<details><summary>fizzbuzz — порядок проверок</summary>
Сначала проверяй делимость на 15 (или одновременно на 3 и на 5), иначе «FizzBuzz»
никогда не сработает.
</details>

<details><summary>lcm — связь с gcd</summary>
Есть тождество: <code>a * b == gcd(a, b) * lcm(a, b)</code>. Отсюда `lcm` выражается
через уже написанный тобой `gcd`. Только сначала отдельно обработай случай, когда `a`
или `b` равны нулю (там и НОД, и деление ведут себя не так, как хочется).
</details>
<details><summary>lcm — почему делить, а не умножать в лоб</summary>
`a * b` может переполниться раньше времени. Лучше сначала разделить одно из чисел на
`gcd`, и только потом умножить: `(a / gcd(a, b)) * b`. Деление на НОД всегда без
остатка — подумай почему.
</details>

<details><summary>sieve — с чего начать</summary>
Заведи `std::vector<bool> is_prime(n + 1, true)` (индексы 0..n). Числа 0 и 1 простыми не
считаются — пометь их `false` сразу. Потом иди по `i` и, встретив ещё не вычеркнутое
`i`, вычёркивай его кратные.
</details>
<details><summary>sieve — какие кратные вычёркивать</summary>
Для простого `i` вычёркивать можно начиная с `i*i` (меньшие кратные уже вычеркнуты
меньшими простыми), шагом `i`: `j = i*i, i*i+i, ...` пока `j <= n`. В конце пройди по
массиву и собери все `i`, у которых `is_prime[i]` осталось `true`, в результат.
</details>

<details><summary>binary_search — каркас</summary>
Держи два индекса-границы и инвариант из «Идеи 9»: искомое, если есть, лежит внутри
полуинтервала `[lo, hi)`. Старт: `lo = 0`, `hi = v.size()`. Пока `lo < hi`, смотри
середину `mid = lo + (hi - lo) / 2`.
</details>
<details><summary>binary_search — как сужать и когда -1</summary>
Сравни `v[mid]` с `target`: если равно — нашёл, верни `mid`. Если `v[mid] < target` —
ответ правее, сдвинь `lo = mid + 1`. Иначе ответ левее (или это и есть mid слева),
сдвинь `hi = mid`. Если цикл закончился, ничего не найдя — верни `-1`. Записывай
`mid = lo + (hi - lo) / 2`, а не `(lo + hi) / 2`, чтобы не переполниться на больших
индексах.
</details>

Покажи мне свою `calc` или любую из функций — обсудим, что можно улучшить.
