# Модуль 19 — Память, выравнивание, аллокаторы и устройство умных указателей

Большая идея: научиться смотреть на память **снизу** — не как на «`new`/`delete`, которые
кто-то написал за нас», а как на сырой массив байт, который мы сами размечаем, выравниваем,
заселяем объектами и убираем за собой. Поняв это, ты перестанешь бояться `unique_ptr` и
`shared_ptr` — потому что напишешь их сам, и они окажутся на удивление маленькими.

## Идея 1. Выравнивание (alignment): почему адрес объекта не любой

Любой тип `T` имеет два числа: `sizeof(T)` — сколько байт он занимает, и `alignof(T)` —
**выравнивание**, то есть требование «адрес объекта обязан делиться на это число нацело».
Для `int` обычно `alignof(int) == 4`, для `double` — `8`, для `char` — `1`.

Откуда берётся это требование? Из железа. Процессор читает память не побайтно, а словами
(на x86-64 — обычно 8-байтными блоками с естественных границ). Если 8-байтный `double`
лёг по адресу, кратному 8, его можно прочитать одной транзакцией шины. Если он «съехал»
через границу слова — на одних архитектурах это два чтения и склейка (медленнее), на
других (ARM в строгом режиме, SIMD-инструкции) — **аппаратное исключение** и падение.
Поэтому компилятор обязан класть объекты только на разрешённые адреса.

Из выравнивания полей растёт **padding** (набивка) внутри структур:

```cpp
struct S {
    char  a;   // смещение 0
    // 3 байта набивки, чтобы int лёг по кратному 4 адресу
    int   b;   // смещение 4
    char  c;   // смещение 8
    // 3 байта набивки в хвосте, чтобы sizeof(S) был кратен alignof(S)==4
};              // sizeof(S) == 12, alignof(S) == 4
```

`alignof(S)` всегда равен максимальному выравниванию среди полей. Хвостовая набивка нужна,
чтобы в массиве `S arr[2]` второй элемент тоже лёг по выровненному адресу.

Управлять выравниванием можно ключевым словом `alignas`:

```cpp
alignas(16) std::byte buffer[64];   // буфер начинается с адреса, кратного 16
alignas(std::max_align_t) char raw[256]; // максимально «безопасное» выравнивание
```

`std::max_align_t` — тип с самым строгим выравниванием, которое гарантирует обычный
`new`/`malloc`. Когда мы делаем свой аллокатор и не знаем заранее тип, мы по умолчанию
выравниваем под `alignof(std::max_align_t)` — тогда туда влезет что угодно.

**Ключевой трюк — выровнять адрес вверх.** Дан адрес `p` (как число) и `align` —
**степень двойки**. Ближайший не-меньший адрес, кратный `align`:

```cpp
std::uintptr_t up = (p + (align - 1)) & ~(align - 1);
```

Почему работает: `align - 1` — это маска младших битов (для `align == 8` это `0b111`).
Прибавив её, мы «перепрыгиваем» через границу, если уже не на ней; затем `& ~(align-1)`
обнуляет младшие биты — отбрасывает остаток. Для адресов всегда считай в `std::uintptr_t`
(целочисленный тип под указатель из `<cstdint>`), потом приводи обратно.

**Самая частая ошибка** — арифметика выравнивания над `void*`: к `void*` нельзя прибавлять,
по нему нельзя делать `&`. Приводи к `std::uintptr_t` для вычислений или к `std::byte*` для
побайтового сдвига. И помни: формула с маской верна **только для степеней двойки** — для
произвольного `align` пришлось бы делать `((p + align - 1) / align) * align`.

## Идея 2. Placement new и явный вызов деструктора

Обычный `new T(args)` делает **две** вещи: (1) выделяет память под `T` и (2) строит в ней
объект. Обычный `delete p` — тоже две: (1) вызывает деструктор и (2) освобождает память.
Иногда нам нужно разделить эти шаги — например, память уже есть (наш буфер), и мы хотим
лишь *построить* в ней объект. Для этого есть **placement new**:

```cpp
#include <new>
alignas(T) std::byte buf[sizeof(T)];     // сырые байты, объекта ещё НЕТ
T* p = new (buf) T(args...);             // построить T прямо в buf, НЕ выделяя память
```

Форма `new (адрес) T(...)` не выделяет память — она принимает готовый адрес и вызывает в нём
конструктор. Буфер обязан быть **достаточного размера** (`>= sizeof(T)`) и **достаточного
выравнивания** (`alignof(buf) >= alignof(T)` — отсюда `alignas(T)` выше).

Раз `new (buf)` не выделял память, то и `delete p` делать **нельзя** — он попытается её
освободить и сломает кучу. Вместо этого мы вызываем деструктор **вручную**:

```cpp
p->~T();   // разрушить объект; память buf остаётся нашей и не освобождается
```

После этого в `buf` снова просто байты, и буфером можно распоряжаться дальше.

**Время жизни объекта в сыром буфере.** Это тонкий момент. С точки зрения языка, объект
*начинает* жить с момента возврата placement new и *перестаёт* — с момента вызова
деструктора (или начала нового placement new в той же памяти). Обращаться к объекту вне
этого окна — UB, даже если байты «выглядят правильно». Поэтому правило: на каждый успешный
placement new ровно один явный вызов деструктора, и ни обращения раньше, ни после.

**Частые ошибки:**

```cpp
// ОШИБКА: delete после placement new — двойное/чужое освобождение
T* p = new (buf) T();
delete p;                 // ✗ buf не из кучи!

// ОШИБКА: забыли ~T() — деструктор не вызван, ресурсы (файлы, память) утекли
new (buf) std::string("...");   // ✗ строка внутри держит кучу, её надо разрушить

// ПРАВИЛЬНО
T* p = new (buf) T();
/* ... пользуемся p ... */
p->~T();                  // ✓ разрушили объект, память buf оставили себе
```

Именно на этой паре «placement new + явный деструктор» построены `std::vector` (он держит
сырой буфер ёмкости больше размера), `std::optional`, `std::variant` и любой аллокатор.

## Идея 3. Bump / arena аллокатор

Самый простой аллокатор из всех. У нас есть большой сырой буфер и **курсор** — указатель на
первый свободный байт. Чтобы выделить `n` байт: выровняли курсор вверх, запомнили его как
результат, сдвинули курсор на `n` вперёд. Всё. Никаких списков, никаких заголовков блоков.

```
buffer: [■■■■□□□□□□□□□□□□]
              ^cursor   ^end
         занято   свободно
```

Цена за простоту: **освобождать по одному блоку нельзя**. Курсор только растёт. Зато есть
`reset()` — он мгновенно «отматывает» курсор в начало и объявляет весь буфер свободным
(O(1), освобождение тысяч объектов одной строкой). Отсюда второе имя — *arena*: ты
аллоцируешь много мелочи за один кадр/запрос, а в конце одним движением сбрасываешь арену.

Почему это быстро и зачем нужно: обычный `new` — это поиск подходящей дыры в куче,
синхронизация между потоками, обновление метаданных; десятки-сотни тактов. Bump-аллокация —
это сложение и сравнение, единицы тактов. Для парсеров, игровых кадров, обработки одного
HTTP-запроса (выделил всё нужное → ответил → сбросил арену) это идеальный паттерн.

Контракт `allocate(n, align)`:
1. выровнять курсор вверх до кратного `align`;
2. проверить, что `выровненный_курсор + n` не вышел за конец буфера;
3. если вышел — это **out-of-memory**, бросаем `std::bad_alloc`;
4. иначе запомнить выровненный адрес, сдвинуть курсор на `n`, вернуть запомненный адрес.

**Частые ошибки.** (1) Проверять вместимость **до** выравнивания — выравнивание само
съедает байты, считать остаток надо после него. (2) Считать `used()` как «сумму запрошенных
`n`» — на самом деле занятое = `cursor - begin`, и в него входят байты, потраченные на
выравнивание. (3) Переполнение при `cursor + n`: на гигантских `n` сложение указателей
переполнится; аккуратный код сравнивает `остаток_буфера >= n`, а не `cursor + n <= end`.
(4) Bump-аллокатор раздаёт **сырую память** — конструкторы объектов он не зовёт; если в
арене лежат не-тривиальные объекты, их деструкторы при `reset()` не вызовутся (это твоя
ответственность — арена для этого и существует, чтобы про деструкторы не думать на POD-данных).

## Идея 4. Пул-аллокатор фиксированного размера (free-list)

Bump хорош, когда не нужно освобождать по одному. Но если объекты создаются и удаляются
вперемешку — нужен **пул**. Идея: заранее нарезаем буфер на блоки **одинакового** размера и
держим список свободных блоков — **free-list**.

Гениальный трюк: отдельной памяти под список не нужно. Пока блок свободен, он всё равно
никем не используется — значит, в его первых байтах можно хранить указатель на *следующий*
свободный блок. Свободные блоки сами образуют односвязный список, «вложенный» в буфер:

```
free_head ──▶ [блок2] ──▶ [блок0] ──▶ [блок3] ──▶ nullptr
              (в первых sizeof(void*) байтах каждого — адрес следующего)
```

`allocate()` — снять голову списка: запомнить `free_head`, прочитать из него адрес
следующего, сделать его новым `free_head`, вернуть снятый блок. O(1).

`deallocate(p)` — вернуть блок в голову: записать в первые байты `p` текущий `free_head`,
затем `free_head = p`. Тоже O(1). Заметь: освобождение — это просто «вставка в начало
списка», порядок блоков перемешивается, и это нормально.

Условия корректности: размер блока `>= sizeof(void*)` (иначе указатель «следующего» не
влезет) и блоки выровнены так, чтобы хранить указатель. Когда блоки кончились
(`free_head == nullptr`), `allocate()` возвращает `nullptr` — сигнал «пул исчерпан».

Зачем это: пулы дают **постоянную** стоимость и **нулевую фрагментацию** для большого числа
одинаковых объектов (узлы списка/дерева, частицы, сетевые пакеты, игровые сущности). Куча
общего назначения такого не гарантирует.

**Частые ошибки.** (1) Хранить указатель «следующего» где-то снаружи блока вместо самих
первых байт блока — теряется весь смысл, нужна лишняя память. Правильно — *type-punning*
через `std::memcpy`/`reinterpret_cast` прямо в начало блока. (2) Забыть, что после
`deallocate` порядок свободных блоков меняется — не полагайся на «адреса идут подряд».
(3) Дважды освободить один блок (double-free) — он попадёт в список дважды, и пул выдаст
один и тот же адрес двум владельцам. (4) Так же, как bump, пул раздаёт **сырую** память —
конструировать/разрушать объекты placement-new'ом обязан пользователь.

Простой и переносимый способ записать/прочитать указатель в первых байтах блока:

```cpp
void* next;
std::memcpy(&next, block, sizeof(void*));   // прочитать «следующего» из блока
std::memcpy(block, &next, sizeof(void*));   // записать «следующего» в блок
```

## Идея 5. Как изнутри устроен `unique_ptr`

`unique_ptr<T>` — это всего лишь **обёртка над сырым указателем** с двумя обещаниями:
в деструкторе он делает `delete`, и его **нельзя копировать**, только перемещать. Размер —
ровно один указатель (для пустого удалителя), накладных расходов в рантайме ноль.

Почему move-only? Потому что владелец должен быть **единственным**: если бы два
`unique_ptr` указывали на один объект, оба бы вызвали `delete` — двойное освобождение.
Поэтому копирующие конструктор и оператор `= delete`, а перемещение **передаёт** владение:
источник после `std::move` становится пустым (`get() == nullptr`).

Ключевые операции и идиома их реализации:

```cpp
// move-конструктор: украсть указатель, оставив источник пустым
UniquePtr(UniquePtr&& o) noexcept : ptr_(std::exchange(o.ptr_, nullptr)) {}

// release(): отдать владение наружу — вернуть указатель, внутри обнулиться
T* release() noexcept { return std::exchange(ptr_, nullptr); }

// reset(p): удалить старый объект, принять новый
void reset(T* p = nullptr) noexcept { delete ptr_; ptr_ = p; }
```

`std::exchange(a, b)` — «положить в `a` значение `b`, а вернуть то, что было в `a`»; идеален
для передачи владения в одну строку.

**Частые ошибки.** (1) В move-присваивании забыть удалить **свой старый** объект перед тем,
как забрать чужой указатель — утечка. (2) Не обнулить источник при перемещении — потом оба
позовут `delete`. (3) Не обработать самоприсваивание `p = std::move(p)` — наивная
реализация удалит объект и тут же запишет висячий указатель. (4) Сделать конструктор из
`T*` неявным — тогда `UniquePtr<T> p = some_raw_ptr;` молча «усыновит» чужой указатель;
поэтому он **`explicit`**.

## Идея 6. `shared_ptr`, `weak_ptr` и контрол-блок

`shared_ptr<T>` позволяет **нескольким** владельцам совместно держать объект; объект живёт,
пока есть хоть один владелец. Чтобы это работало, нужен **контрол-блок** — отдельный кусочек
памяти, *общий* для всех копий одного `shared_ptr`. В нём минимум два счётчика:

* **strong** — сколько `shared_ptr` владеют объектом. Когда strong падает до 0 — **объект**
  разрушается (`delete`).
* **weak** — сколько `weak_ptr` наблюдают. Когда И strong, И weak станут 0 — разрушается
  **сам контрол-блок** (память счётчиков освобождается).

```
shared_ptr a ──┐
shared_ptr b ──┼──▶ [ControlBlock: ptr→Object, strong=2, weak=1]
weak_ptr   w ──┘
```

Копирование `shared_ptr` → `++strong`; разрушение → `--strong` (и если стало 0 — удалить
объект). `use_count()` — это текущее значение strong. Перемещение **не** меняет strong:
источник просто отдаёт свой указатель на контрол-блок, не трогая счётчик.

Почему контрол-блок и объект — это (логически) **две разные** сущности с **двумя** условиями
удаления? Из-за `weak_ptr`. Слабая ссылка должна уметь спросить «жив ли объект?» даже после
того, как все сильные ссылки ушли и объект уже разрушен. Значит, счётчики должны пережить
объект. Поэтому объект убивают при `strong == 0`, а память счётчиков — позже, при
`strong == 0 && weak == 0`. (В настоящем `std::make_shared` объект и контрол-блок лежат в
одном выделении ради скорости, но логика двух счётчиков та же.)

**Зачем `weak_ptr` — он рвёт циклы.** Представь два узла, ссылающихся друг на друга через
`shared_ptr`:

```cpp
struct Node { SharedPtr<Node> other; };
auto a = SharedPtr<Node>(new Node);
auto b = SharedPtr<Node>(new Node);
a->other = b;   // strong(b) == 2
b->other = a;   // strong(a) == 2
// выходим из области видимости: локальные a,b ушли → strong каждого стал 1,
// но они держат друг друга → ни один не дойдёт до 0 → УТЕЧКА памяти
```

Счётчик ссылок не умеет распознавать **циклы** — это его фундаментальное ограничение.
Лекарство: одну из ссылок цикла сделать `weak_ptr`. Слабая ссылка **не увеличивает strong**,
поэтому не держит объект на плаву; когда внешние сильные ссылки уходят, strong доходит до 0
и цикл распадается. Чтобы воспользоваться объектом через `weak_ptr`, его «повышают» до
`shared_ptr` методом `lock()`: если объект ещё жив (`strong > 0`) — `lock()` возвращает
полноценный `shared_ptr` (с `++strong`); если уже мёртв — пустой. `expired()` — быстрый
вопрос «strong уже 0?».

**Частые ошибки.** (1) Сделать два `shared_ptr` из **одного** сырого указателя
(`SharedPtr<T> a(p); SharedPtr<T> b(p);`) — получится **два** контрол-блока и **двойное**
удаление; правильно — копировать `shared_ptr`, а не плодить их из `T*`. (2) Хранить
`weak_ptr` и обращаться к объекту без `lock()` — между проверкой `expired()` и доступом
объект мог умереть; всегда работай через результат `lock()`. (3) Забыть, что контрол-блок
переживает объект — после `strong==0` нельзя трогать `ptr`, но `weak`-счётчик ещё жив.
(4) В `release_` удалить контрол-блок при `strong==0`, не дождавшись `weak==0` — повисшие
`weak_ptr` будут читать освобождённую память.

## Идея 7. `std::allocator`, `allocator_traits` и allocator-aware контейнеры

Свой bump/pool — это хорошо, но в реальном коде аллокатор не вызывают напрямую: его
**подставляют в контейнер**. `std::vector`, `std::map`, `std::list` — все они *allocator-aware*:
вторым (для ассоциативных — последним) шаблонным параметром принимают тип аллокатора и всю
работу с памятью ведут через него. По умолчанию это `std::allocator<T>` — тончайшая обёртка
над `::operator new`/`::operator delete`. Заменив его своим типом, ты перенаправляешь *всю*
динамическую память контейнера в свою арену/пул, не трогая ни строчки внутри `vector`.

**Что такое тип-аллокатор.** Это не объект с состоянием «куча», а **дескриптор политики
выделения**: лёгкое значение (часто просто указатель на настоящий ресурс), которое можно
копировать. Минимальный аллокатор `A` для типа `T` обязан предоставить ровно немного:

```cpp
template <class T>
struct MyAlloc {
    using value_type = T;                       // обязателен
    MyAlloc() = default;
    template <class U> MyAlloc(const MyAlloc<U>&) {} // конвертирующий (rebind) ctor
    T*   allocate(std::size_t n);               // n объектов T → сырые байты
    void deallocate(T* p, std::size_t n);       // вернуть их
};
template <class T, class U>
bool operator==(const MyAlloc<T>&, const MyAlloc<U>&) noexcept; // + operator!=
```

Всё остальное достраивает **`std::allocator_traits<A>`** — это слой косвенности между
контейнером и аллокатором. Контейнер **никогда** не зовёт `a.allocate()` напрямую; он зовёт
`std::allocator_traits<A>::allocate(a, n)`. Зачем лишний слой? Чтобы аллокатор мог задавать
*минимум*, а traits подставит разумные значения по умолчанию для всего, что не задано:

| что нужно контейнеру | если аллокатор задал — берётся его | иначе traits даёт по умолчанию |
|---|---|---|
| `allocate(a,n)` | `a.allocate(n)` | (обязателен, дефолта нет) |
| `deallocate(a,p,n)` | `a.deallocate(p,n)` | (обязателен) |
| `construct(a,p,args...)` | `a.construct(...)` | **placement new**: `::new (p) T(args...)` |
| `destroy(a,p)` | `a.destroy(p)` | явный деструктор: `p->~T()` |
| `pointer`, `size_type`, … | свои typedef'ы | `T*`, `std::size_t`, … |
| `rebind_alloc<U>` | свой `rebind` | переподстановка: `A` → `A` с `U` вместо `T` |
| `max_size(a)` | `a.max_size()` | `SIZE_MAX / sizeof(T)` |

Видишь Идеи 2 и 5 насквозь: `construct`/`destroy` по умолчанию — это ровно наши
`construct_at19`/`destroy_at19`. `allocate` отвечает только за **сырые байты**; объекты в них
строит уже `construct`. Поэтому `vector::push_back` — это **два** шага: `allocate` ёмкости (раз
на реаллокацию) и `construct` каждого элемента placement-new'ом.

**`rebind` — зачем он.** Ты пишешь `std::list<int, A>` с `A = MyAlloc<int>`. Но `list` хранит
не голые `int`, а **узлы** `Node{int value; Node* next; Node* prev;}`. Узлы тоже надо
выделять — через аллокатор, но уже для `Node`, не для `int`. Контейнер берёт
`allocator_traits<A>::rebind_alloc<Node>` — это «тот же аллокатор, но для другого типа». По
умолчанию для шаблона `MyAlloc<T>` ребайнд даёт `MyAlloc<Node>` (переподставляет параметр).
Чтобы из `MyAlloc<int>` получить *рабочий* `MyAlloc<Node>`, нужен **конвертирующий
конструктор** `template<class U> MyAlloc(const MyAlloc<U>&)`: он переносит состояние (у нас —
указатель на арену) в ребайнднутый аллокатор. Без него `std::list`/`std::map` с твоим
аллокатором просто не скомпилируются — а `std::vector` (он хранит сами `T`) ещё мог бы.

**Зачем `operator==`.** Два аллокатора равны, если память, выданную одним, можно вернуть через
другой. От этого зависит, может ли контейнер при move/swap **просто переставить указатель** на
буфер вместо поэлементного копирования: если аллокаторы равны — да; если нет — придётся
аллоцировать новый буфер чужим аллокатором и переносить элементы. Для аллокатора без
состояния (`std::allocator`) все экземпляры равны всегда. Для аллокатора *с* состоянием (наш —
с указателем на арену) равенство = «та же арена».

**Подстановка своего аллокатора:**

```cpp
alignas(std::max_align_t) std::byte storage[1 << 16];
MyArena arena(storage, sizeof storage);
std::vector<int, ArenaAllocator<int>> v{ ArenaAllocator<int>(arena) };
v.reserve(100);          // один allocate из арены под 100 int
v.push_back(42);         // construct на месте, без обращения к куче
```

**Подвох с состоянием.** `std::allocator` *без состояния*, поэтому его экземпляр контейнеру
можно не хранить и не передавать. Аллокатор *с состоянием* (ссылка на арену) контейнер обязан
скопировать и таскать с собой — поэтому его передают в конструктор контейнера, а сам
аллокатор должен быть дёшево-копируемым. И помни: аллокатор **не владеет** ресурсом —
арена/буфер должны пережить контейнер, иначе висячая ссылка.

**Частые ошибки.** (1) Думать, что `allocate` строит объекты — нет, только сырые байты;
конструирование делает `construct` (placement new). (2) Забыть конвертирующий конструктор —
`vector` стерпит, а `list`/`map` сломаются на ребайнде узлов. (3) Сделать `operator==`
несогласованным с реальной взаимозаменяемостью памяти — UB при move контейнеров. (4) Считать,
что аллокатор владеет памятью — он лишь политика поверх чужого ресурса.

## Идея 8. Over-aligned типы: когда `alignas` на стеке недостаточно

`alignof(std::max_align_t)` (обычно 16) — это потолок, который гарантируют `new`/`malloc`.
Но бывают типы с **бо́льшим** требованием: SIMD-векторы (`__m256` хочет 32, `__m512` — 64),
буферы под выровненный к кэш-линии (64 байта) или к странице (4096) объект. Их называют
*over-aligned*.

На стеке всё просто — `alignas` работает:

```cpp
alignas(64) std::array<float, 16> v;   // локальная переменная, выровнена под 64
```

Но из **кучи** обычный `new T` до C++17 мог вернуть память, выровненную лишь под
`max_align_t` — и положить туда `alignas(64)`-тип было UB. С C++17 это починили двумя
способами.

**1. Aligned-перегрузки `operator new`.** Если `alignof(T) > __STDCPP_DEFAULT_NEW_ALIGNMENT__`,
обычный `new T` сам вызывает *выровненную* версию `::operator new(size, std::align_val_t)`.
Её можно дёрнуть и вручную для сырой памяти:

```cpp
#include <new>
void* p = ::operator new(n, std::align_val_t{64});   // n байт, выровнены под 64
// ... используем ...
::operator delete(p, std::align_val_t{64});           // ПАРНО с тем же align!
```

Ключевое: освобождать over-aligned память надо **парной** `operator delete` с тем же
`align_val_t` — обычный `::operator delete(p)` для неё некорректен.

**2. `std::aligned_alloc(align, size)`** (C-функция из `<cstdlib>`): возвращает указатель,
выровненный под `align`. Жёсткое требование стандарта C: `align` — степень двойки, а `size`
**кратен** `align`. Освобождается обычным `std::free`. (На Windows её нет в MSVC CRT — там
`_aligned_malloc`/`_aligned_free`; это типичный портабельный подвох.)

```cpp
void* p = std::aligned_alloc(64, 64 * 4);  // size кратен align — обязательно
std::free(p);
```

**Когда что.** `alignas` на стеке — для локальных и членов. Для динамики в C++ предпочитай
`::operator new(n, align_val_t{...})` (он в системе исключений C++ и парен своему delete) либо
просто `new T`, если у `T` уже стоит `alignas` — компилятор сам выберет выровненную форму.
`std::aligned_alloc` — когда нужен C-style сырой блок с явным размером, кратным выравниванию.

**Подвох.** В *своём* аллокаторе над фиксированным буфером гарантировать произвольное большое
выравнивание нельзя, если сам буфер выровнен слабее: `align_up` подвинет курсор, но если
`alignof(T) > alignof(buffer)`, начало буфера может уже не давать нужную кратность. Поэтому
буфер арены выравнивают под максимум ожидаемых типов (`alignas(64)` и т.п.), а не под
`max_align_t`.

## Идея 9. `std::launder` и provenance после placement new

Тонкий, но честный угол стандарта. Рассмотри:

```cpp
struct X { const int n; };
alignas(X) std::byte buf[sizeof(X)];
X* p1 = new (buf) X{1};      // построили первый X
p1->~X();
X* p2 = new (buf) X{2};      // в той же памяти — НОВЫЙ объект
```

Здесь `p2` указывает на свежий объект и им пользоваться можно. А вот `p1` после второго
placement new — **протухший** указатель: формально это адрес уже не существующего объекта.
Если у `X` есть `const`-член (или ссылка), компилятор имеет право считать, что значение
`p1->n` не меняется (оно `const`!) и закэшировать его — хотя в памяти давно лежит другой `X`.
Обращение через `p1` после переписывания памяти — UB, и «байты те же» тебя не спасут.

`std::launder(p)` (из `<new>`) — это барьер для оптимизатора: «дай мне указатель на объект,
который *реально* живёт по этому адресу сейчас, забудь, что ты знал про прежний». Он не
меняет битов адреса — он чинит **provenance** (происхождение указателя), то есть привязку
указателя к конкретному живому объекту:

```cpp
X* p = std::launder(reinterpret_cast<X*>(buf));   // легально читаем текущий X
```

**Когда launder реально нужен** (узко): ты переиспользуешь память под объект с
`const`/ссылочными членами и хочешь обращаться к нему через *старый* указатель; либо ты
получил `T*` из `reinterpret_cast` по сырым байтам, в которых построил объект, и хочешь
гарантий. **Когда НЕ нужен:** если ты пользуешься указателем, который вернул *сам* placement
new (`p2` выше) — он уже «правильного происхождения», laundering не требуется. Поэтому в нашем
`construct_at19` мы возвращаем результат `new (raw) T(...)` — и вызывающий работает именно с
ним, без launder.

`std::vector`/`std::optional` устроены так, что отдают наружу указатель из самого placement
new, и в типичном коде launder не всплывает. Знать про него надо, чтобы (а) не словить
коварный UB на reused storage с `const`-полями и (б) понимать, почему «тот же адрес» в C++ ≠
«тот же объект».

## Идея 10. `make_shared`: одно выделение и exception-safety

Сравни два способа создать `shared_ptr`:

```cpp
std::shared_ptr<T> a(new T(args));      // (1)
auto b = std::make_shared<T>(args);     // (2)
```

**Память.** (1) делает **два** выделения: `new T` (объект) и внутри конструктора `shared_ptr` —
ещё одно под контрол-блок (счётчики). (2) делает **одно**: `make_shared` выделяет единый блок,
в котором лежат *и* контрол-блок, *и* сам объект, и строит объект placement-new'ом в его
хвосте. Меньше обращений к куче, лучше локальность (счётчики и объект рядом в кэше),
вдвое меньше накладных байт на заголовки аллокатора. Это та самая «оптимизация одного
выделения», о которой упомянуто в Идее 6.

**Exception-safety — главный довод.** Рассмотри вызов:

```cpp
foo(std::shared_ptr<T>(new T), bar());   // ОПАСНО
```

Порядок вычисления аргументов между собой **не определён**. Компилятор вправе: (а) выполнить
`new T`, (б) вызвать `bar()`, (в) построить `shared_ptr`. Если `bar()` на шаге (б) бросит
исключение — `new T` уже выделил объект, но `shared_ptr` ещё не успел его «усыновить»:
указатель потерян, **утечка**. С `make_shared` такого окна нет — нет «голого» `new`, объект и
владелец рождаются неразделимо:

```cpp
foo(std::make_shared<T>(), bar());       // утечки нет при любом порядке
```

(В C++17 порядок вычисления аргументов частично ужесточили, но `make_shared` остаётся
рекомендованной идиомой — он чище и быстрее.)

**Подвох make_shared.** Раз объект и контрол-блок в одном блоке, память объекта **не вернётся
ОС, пока жив хоть один `weak_ptr`**: контрол-блок переживает объект (Идея 6), а он сидит в том
же выделении, что и «тело» объекта. Для огромных объектов с долгоживущими `weak_ptr` это может
держать память дольше, чем `shared_ptr(new T)`, где тело освобождается сразу при `strong==0`.
Редкий случай, но знать стоит. Ещё: `make_shared` не умеет принять *custom deleter* и
использует `::new`/`::delete` — если нужен свой удалитель, придётся идти через
`shared_ptr(ptr, deleter)`.

## Идея 11. Custom deleter у `unique_ptr<T, D>`

Наш `UniquePtr` (Задание 2) всегда зовёт `delete`. Но владеть можно не только тем, что создано
`new`: файлами (`FILE*` → `fclose`), дескрипторами ОС (`close`), C-ресурсами (`free`),
хэндлами библиотек. `std::unique_ptr` это закрывает **вторым** шаблонным параметром —
*типом удалителя* `D`:

```cpp
std::unique_ptr<T, D> p;   // в деструкторе зовёт d(ptr), а не delete ptr
```

Удалитель — это вызываемая сущность (функтор, лямбда, указатель на функцию), принимающая
хранимый указатель. Классический пример — `FILE*`:

```cpp
struct FileCloser {
    void operator()(std::FILE* f) const noexcept { if (f) std::fclose(f); }
};
std::unique_ptr<std::FILE, FileCloser> file(std::fopen("data.txt", "r"));
// при выходе из области видимости — автоматический fclose, никаких утечек дескриптора
```

**Размер — важный нюанс.** Если `D` — пустой тип (функтор без полей, как `FileCloser`),
`unique_ptr` благодаря *empty base optimization* остаётся размером **в один указатель**: ноль
накладных расходов. Если же `D` — **указатель на функцию** (`void(*)(FILE*)`) или
захватывающая лямбда, его надо где-то хранить — и `unique_ptr` распухает до двух указателей:

```cpp
// D — указатель на функцию: sizeof(unique_ptr) == 2 указателя
std::unique_ptr<std::FILE, int(*)(std::FILE*)> f(std::fopen("x","r"), &std::fclose);
```

Поэтому для минимального размера предпочитают **тип-функтор** (или лямбду через
`decltype`), а не указатель на функцию. `std::shared_ptr`, наоборот, хранит удалитель в
контрол-блоке и его тип **не** входит в тип `shared_ptr<T>` (type erasure) — гибче, но это
стоит того самого контрол-блока.

Идиома «фабрика ресурса»:

```cpp
auto make_file = [](const char* path) {
    return std::unique_ptr<std::FILE, decltype(&std::fclose)>(
        std::fopen(path, "r"), &std::fclose);
};
```

**Частые ошибки.** (1) Хранить ресурс, не созданный `new`, в `unique_ptr<T>` без удалителя —
тогда деструктор сделает `delete` над тем, что нельзя `delete`'ить (UB). (2) Удивляться, что
`unique_ptr` с лямбда-удалителем «вдруг» в два указателя — это плата за хранение состояния
удалителя. (3) Забыть `noexcept` у удалителя — деструкторы не должны бросать.

## Идея 12. Thread-safety счётчика `shared_ptr`: цена `std::atomic`

Идея 6 рисовала счётчики как обычные `long`. В настоящем `std::shared_ptr` они —
**атомарные** (`std::atomic<long>` или эквивалент). Почему это обязано быть так и сколько
стоит?

**Что именно потокобезопасно.** Стандарт гарантирует: одновременное копирование/уничтожение
*разных* `shared_ptr`-объектов, указывающих на *один* контрол-блок, из *разных* потоков —
безопасно. То есть `++strong`/`--strong` на общем счётчике обязаны быть атомарны: иначе два
потока, одновременно делающие `--strong`, могут оба прочитать `2`, оба записать `1` — потерянное
уменьшение, объект не удалится (утечка) или удалится дважды. Поэтому декремент — это
атомарный `fetch_sub` с правильным барьером памяти (release при декременте, acquire-fence
перед удалением — чтобы все записи в объект «увиделись» до его разрушения).

**Чего НЕ гарантируют.** Безопасны *счётчики*, но не *сам объект* и не *один и тот же*
`shared_ptr`-объект, разделяемый потоками без синхронизации: если два потока пишут через
`*sp` или один делает `sp = ...`, пока другой копирует `sp`, — это гонка, нужна внешняя
блокировка (или `std::atomic<std::shared_ptr<T>>` из C++20).

**Цена (Busato).** Атомарный инкремент/декремент **дороже** обычного: это `lock`-префикс
(x86) / LL-SC цикл (ARM), сериализация по строке кэша, межъядерный трафик
когерентности. На горячем пути копирование `shared_ptr` в цикле — заметная статья расходов,
особенно когда много ядер дёргают один счётчик (cache-line bouncing). Отсюда практические
выводы:

* передавай `shared_ptr` в функции по `const&`, а не по значению, если не забираешь владение —
  не платишь за лишний атомарный ++/--;
* `unique_ptr` **не** имеет счётчика вообще → перемещение бесплатно (один указатель), никаких
  атомиков. Если владелец логически один — бери `unique_ptr`, а не `shared_ptr` «на всякий
  случай»;
* `weak_ptr::lock()` — это атомарный *compare-and-increment* (поднять strong, только если он
  ещё > 0), он чуть дороже простого `++`.

Правило-итог: `shared_ptr` — это не «умный указатель по умолчанию», а инструмент **разделяемого
владения**, за который ты платишь атомарной арифметикой и контрол-блоком. Дефолт — `unique_ptr`.

   Идею 7). Это самый сложный пункт. Цель — чтобы работало вот это:

   ```cpp
   m19::BumpAllocator arena(buffer, capacity);
   std::vector<int, m19::ArenaAllocator<int>> v{ m19::ArenaAllocator<int>(arena) };
   v.push_back(42);   // память идёт из arena, а не из глобальной кучи
   ```

   Реализуй по *Allocator named requirement* (минимум для C++20):
   * `using value_type = T;`
   * `explicit ArenaAllocator(BumpAllocator& arena)` — привязка к **общей** арене (аллокатор
     не владеет ею, лишь ссылается);
   * `template<class U> ArenaAllocator(const ArenaAllocator<U>&)` — конвертирующий (rebind)
     конструктор: разделяет ту же арену (нужен, чтобы `allocator_traits::rebind_alloc<U>`
     дал рабочий аллокатор);
   * `T* allocate(std::size_t n)` — взять `n * sizeof(T)` байт из арены, выравнивание
     `alignof(T)` (под капотом — `arena.allocate(...)`); при нехватке полетит `std::bad_alloc`
     прямо из арены;
   * `void deallocate(T*, std::size_t) noexcept` — **no-op** (bump-арена не освобождает по
     одному; вся память живёт до `arena.reset()`);
   * `operator==` / `operator!=` — равны ⇔ одна и та же арена.

   `construct`/`destroy`/`rebind_alloc` писать **не** нужно — их даёт
   `std::allocator_traits<ArenaAllocator<T>>` по умолчанию (placement new / явный деструктор /
   переподстановка шаблонного параметра).

### Подсказки

<details><summary>Задание 1 — выравнивание курсора</summary>
Считай в целых. Возьми `auto cur = reinterpret_cast<std::uintptr_t>(cursor_);`, выровняй
вверх формулой с маской из Идеи 1 (она верна, потому что `align` — степень двойки), затем
проверь, что от выровненного адреса до конца буфера осталось не меньше `n` байт. Только
после этого двигай курсор и возвращай адрес. `used()` — это не сумма `n`, а `cursor_ - begin_`.
</details>

<details><summary>Задание 1 — как правильно бросить «нет памяти»</summary>
Тест ждёт ровно `std::bad_alloc`, не `std::logic_error`. Бросай `throw std::bad_alloc{};`
когда блок не помещается. Сравнивай остаток (`конец - выровненный_курсор`) с `n`, чтобы не
переполнить указатель на больших `n`.
</details>

<details><summary>Задание 2 — передача владения без лишних строк</summary>
`std::exchange(o.ptr_, nullptr)` забирает указатель у источника и оставляет его пустым за
один приём — удобно и в move-конструкторе, и в `release()`. В move-присваивании сначала
разберись со своим старым объектом, потом забери чужой; не забудь случай `this == &other`.
</details>

<details><summary>Задание 3 — кто за что отвечает</summary>
Заведи один приватный `release_()` и зови его из деструктора и из обоих присваиваний — так
логика счётчиков не размножится. Порядок в `SharedPtr::release_`: если `ctrl_` пуст — выйти;
`--strong`; если `strong == 0` — удалить *объект*; если вдобавок `weak == 0` — удалить
*контрол-блок*. В `WeakPtr::release_`: `--weak`; удалить контрол-блок только если и
`strong == 0`, и `weak == 0`.
</details>

<details><summary>Задание 3 — lock()</summary>
`lock()` должен отдать `SharedPtr`, разделяющий тот же `ctrl_`. Раз `SharedPtr(T*)` создаёт
*новый* контрол-блок, он тут не подойдёт. Удобно: проверить `!expired()`, сделать пустой
`SharedPtr`, вручную присвоить ему `ctrl_` и сделать `++strong`. `WeakPtr` — `friend`
`SharedPtr` (объявлено), так что к приватному полю доступ есть.
</details>

<details><summary>Задание 4 — связать блоки и снимать голову</summary>
Адрес i-го блока — `static_cast<std::byte*>(buffer) + i * block_size`. Чтобы записать/прочитать
указатель «следующего» в первых байтах блока, удобнее всего `std::memcpy` (см. Идею 4) — это
переносимо и без UB. `allocate` снимает `free_head_`, `deallocate` вставляет блок в начало.
Не забудь поддерживать `free_count_`.
</details>

<details><summary>Задание 5 — две строки, но с тонкостью</summary>
`construct_at19`: `return new (raw) T(std::forward<Args>(args)...);` — это placement new,
память не выделяется. `destroy_at19`: `p->~T();` — память буфера при этом НЕ освобождаем,
ей владеет вызывающий. Никакого `delete` здесь быть не должно.
</details>

<details><summary>Задание 6 — что хранить в аллокаторе</summary>
Аллокатор сам не держит память — он лишь *ссылается* на арену. Храни внутри
`BumpAllocator*` (один указатель, дёшево копируется). Конструктор из `BumpAllocator&`
сохраняет `&arena`. Не делай поле ссылкой `BumpAllocator&` — тогда аллокатор станет
не присваиваемым, а контейнеру это иногда нужно.
</details>

<details><summary>Задание 6 — rebind-конструктор и доступ к чужому полю</summary>
Конвертирующий конструктор `template<class U> ArenaAllocator(const ArenaAllocator<U>& o)`
должен прочитать `o.arena_`. Но `ArenaAllocator<U>` — это *другой* тип, и его приватное поле
снаружи не видно. Объяви `template<class U> friend class ArenaAllocator;` — тогда все
инстанцирования шаблона дружат между собой и видят приватные поля друг друга. Тело
конструктора — просто `arena_(o.arena_)`.
</details>

<details><summary>Задание 6 — allocate, deallocate, сравнение</summary>
`allocate(n)`: верни `static_cast<T*>(arena_->allocate(n * sizeof(T), alignof(T)))`. Сырые
байты — конструировать объекты будет `allocator_traits::construct` (placement new), не ты.
`deallocate` — пустое тело: bump-арена не освобождает поштучно. `operator==` сравнивает
указатели на арену; делай его свободной функцией с двумя параметрами `<T>` и `<U>`, чтобы
сравнивались и ребайнднутые аллокаторы (`ArenaAllocator<int>` vs `ArenaAllocator<Node>`).
</details>

Когда всё зелёное — попробуй сам собрать цикл из двух узлов через `SharedPtr` и убедись
(через `use_count`), что без `WeakPtr` он не разрушается, а с `WeakPtr` — разрушается.
Покажешь — разберём, почему счётчик ссылок в принципе слеп к циклам.
