## Что строим

**MiniDB** — потокобезопасная in-memory база данных в духе Redis. Это апгрейд бывшего `kvstore`
(который умел только строки). Теперь в ней:

- **три типа значений** — строка, список строк, хеш (поле→строка) — через `std::variant`;
- **TTL** — ключам можно ставить срок жизни, после которого они «протухают»;
- **потокобезопасность** — несколько потоков работают с одной базой без гонок данных;
- **персистентность** — база сохраняется в файл и загружается обратно;
- **текстовый протокол** — с базой можно «разговаривать» командами, как через `redis-cli`.

Проект состоит из трёх слоёв (классический расклад из модуля 11):

```
include/value.hpp     Value — одно значение (variant из трёх форм)
include/database.hpp  Database — потокобезопасное хранилище ключ→Value с TTL и save/load
include/command.hpp   execute() — разбор текстовой команды в вызов Database
src/*.cpp             РЕАЛИЗАЦИИ этих трёх (сейчас стабы — пишешь ты)
app/main.cpp          REPL — готовое приложение поверх Database+execute (НЕ трогать)
tests/*.cpp           приёмочные тесты (НЕ трогать — это твоя «спецификация в коде»)
CMakeLists.txt        явная сборка: lib + app + tests (прочитай, тут нет магии)
```

## Рекомендуемая архитектура

Это **рекомендация**, а не приказ — лишь бы сигнатуры из заголовков и поведение из тестов
сошлись. Но если не знаешь, с чего начать, иди так.

### 1. `Value` (модули 07/09 — шаблоны, variant, ошибки)
Тонкая обёртка над `std::variant<std::string, vector<string>, map<string,string>>`.
- `type()` — порядок альтернатив variant совпадает с `enum Type`, так что `data_.index()`
  отображается в `Type` напрямую.
- `as_string()/as_list()/as_hash()` — через `std::get_if<T>(&data_)`: не тот тип → `nullptr` →
  `throw std::runtime_error`. Это «бросающие аксессоры» из модуля 09.
- Пара `const`/неconst (модуль 04): неconst-версия даёт менять контейнер на месте (нужно для
  `lpush`/`hset`), const — только читать.

### 2. `Database` (модули 04/05/06/08/09/**14**)
Сердце проекта. Внутри — `std::map<std::string, Entry>`, где `Entry { Value value; optional<int64> expire_at; }`.
- **Потокобезопасность.** Один `std::mutex mutex_`. КАЖДЫЙ публичный метод первой строкой берёт
  `std::lock_guard<std::mutex> lock(mutex_);` (RAII из модуля 04 — освобождение в деструкторе).
  Осторожно с **дедлоком**: публичный метод НЕ должен звать другой публичный (тот снова захватит
  тот же мьютекс). Выноси общую логику в приватные хелперы без блокировки.
- **Ленивое протухание.** Не нужен фоновый поток-уборщик. Перед работой с записью проверяй
  `expired(e)` по `now()`; если протухла — считай, что ключа нет (можешь сразу `erase`).
- **Инъекция часов.** `now()` зовёт `clock_()`. В тестах clock — лямбда с управляемым «сейчас»;
  в приложении (пустой clock) — реальные `std::chrono::system_clock` в секундах.
- **Персистентность.** `save`/`load` — простой текстовый формат (по строке на ключ: тип, имя,
  TTL, данные). Сеть не нужна; тесты пишут во временный файл.

### 3. `execute()` (модули 08/09 — строки, парсинг, ошибки)
Разбор строки в токены (`std::istringstream`), имя команды к верхнему регистру, диспетчеризация
по имени, вызов нужного метода `Database`, форматирование Redis-подобного ответа. Любая ошибка
разбора → строка `"ERR ..."`; исключения наружу НЕ выпускать (оберни в try/catch).

## Глубже: четыре «сеньорских» вопроса проектирования

Капстоун — повод не просто «сделать зелёным», а понять, ПОЧЕМУ контракт выглядит именно так.
Ниже — четыре решения, которые в коде почти невидимы, но отличают junior-код от продуманного.

### 1. Почему `Value` следует rule of 0 (и причём тут `move` и `noexcept`)

**Rule of 0.** `Value` не объявляет ни деструктор, ни копирующие/перемещающие операции —
ровно ничего из «большой пятёрки». Его единственное поле — `std::variant<...>`, а variant сам
правильно копируется, перемещается и разрушается (он распространяет эти операции на активную
альтернативу). Раз управлять вручную нечем, лучший спецкод — отсутствие спецкода: компилятор
сгенерирует все пять операций корректно. Это и есть **rule of 0** — «не пишешь руками то, что
тип-член уже делает за тебя». Стоит добавить хоть один из спецчленов (например, пустой
`~Value(){}`), и ты, во-первых, подавишь неявную генерацию `move` (он «откатится» к копии —
тихая потеря производительности), во-вторых, возьмёшь на себя обязательство поддерживать
остальные четыре. Правило **3/5/0**: пишешь один спецчлен — продумай все пять; не пишешь ни
одного — это идеал.

**Почему важен `move`.** Значения в MiniDB передаются *по значению* (`set(std::string value)`)
и затем «утекают» внутрь через `std::move` (см. `make_string`). Без перемещения каждый `set`
большой строки копировал бы её содержимое в кучу заново. Move крадёт внутренний буфер у
источника за указатели — O(1) вместо O(n).

**Почему важен `noexcept` у `now()`/`expired()`.** Перемещение особенно критично внутри
контейнеров. Когда `std::vector` растёт и переселяет элементы, он выбирает между copy и move по
одному признаку: помечена ли move-операция типа как `noexcept`. Если move *может* бросить, vector
ради строгой гарантии исключений вынужден **копировать** (чтобы при сбое откатиться) — и весь
выигрыш move исчезает. Поэтому дешёвые наблюдатели вроде `now()`/`expired()` стоит делать
`noexcept`: они не выделяют память и не бросают, а компилятор использует это знание, чтобы
выбирать move-путь и не платить за copy. Привычка «помечай не-бросающее как `noexcept`» —
часть того, почему move-семантика реально ускоряет, а не только числится в сигнатуре.

> Подвох: `noexcept`-функция, которая всё же бросит, вызовет `std::terminate` (не обычную
> раскрутку). Поэтому `noexcept` — это *обещание*, а не пожелание: ставь его только там, где
> действительно гарантируешь отсутствие исключений (чтение поля, арифметика — да; работа с
> аллокатором — осторожно).

### 2. Почему интерфейс берёт `const std::string&`, а не `std::string_view`

Методы `Database` принимают ключ как `const std::string& key`. Альтернатива — `std::string_view`
(C++17): невладеющая пара «указатель + длина», которая умеет ссылаться и на `std::string`, и на
строковый литерал, и на кусок буфера — БЕЗ копии и БЕЗ выделения памяти. Для read-only
параметра, по которому мы только *ищем* в `std::map`, `string_view` обычно идеален: вызов
`db.get("name")` со строковым литералом не создаёт временный `std::string`.

**Почему здесь всё же `const std::string&`.** Две причины. (1) **Гетерогенный поиск.** `string_view`
блистает, когда по ключу можно искать без материализации `std::string`. Но `std::map<std::string,…>`
по умолчанию ищет только по `std::string` — гетерогенный поиск требует прозрачного компаратора
(`std::less<>`), которого в контракте нет; то есть из `string_view` внутри всё равно пришлось бы
строить `std::string` для `find`. (2) **Срок жизни.** `string_view` ничего не *владеет*: сохранить
его в долгоживущем поле (как ключ записи) — почти гарантированный висячий указатель, когда исходный
буфер умрёт. А `set` обязан *сохранить* ключ в `data_`. `const std::string&` здесь честнее: на входе
мы либо уже имеем `std::string` (типичный случай в тестах), либо платим одну копию при сохранении —
которая всё равно неизбежна, ведь map *владеет* своими ключами.

Правило-памятка: `string_view` — для параметров, которые ты только *читаешь* и не переживаешь
дольше вызова. Как только нужно *сохранить* строку — нужен владелец (`std::string`), и `view`
лишь отсрочит копию, не отменит её. (То же про `std::span` для не-владеющих «окон» над массивами.)

### 3. Дисциплина ошибок: `optional` vs исключения vs `std::expected`

MiniDB сознательно смешивает два механизма — и это не небрежность, а выбор «по природе ошибки».

- **`std::optional<T>` — для ожидаемого «ничего нет».** `get`/`hget`/`type` возвращают
  `optional`: отсутствие ключа — это **штатный** результат, не авария. Вызывающий обязан
  проверить `has_value()`; забыть проверку и дёрнуть `*opt` на пустом — UB, зато тип в сигнатуре
  кричит «меня может не быть». Дёшево (без раскрутки стека), локально, самодокументируемо.
- **Исключения (`std::runtime_error`) — для нарушения инварианта.** Бросающие аксессоры `Value`
  (`as_list()` у строки) кидают: вызвать «не тот» аксессор — это **ошибка программиста**, а не
  обычная ветка данных. Исключение здесь правильно: оно не даёт «случайно поехать дальше» с
  мусором и заставляет либо ловить, либо падать громко. А `execute()` ставит **барьер**: оборачивает
  всё в `try/catch` и превращает любое исключение в `"ERR …"` — наружу, в REPL, исключения не текут.
- **`std::expected<T, E>` (C++23) — «или значение, или ошибка» в одном типе.** Это то, чего не
  хватает связке optional+exceptions: `optional` говорит *есть/нет*, но не *почему* нет;
  исключения несут причину, но дороги и невидимы в сигнатуре. `expected<std::string, Error>` вернул
  бы либо значение, либо *типизированную причину провала*, без раскрутки стека и с явным контрактом
  в типе — фактически «Result» из Rust. **В этом треке (C++20) мы `std::expected` НЕ используем в
  коде** (нужен C++23 и свежий libc++); упоминаем как ориентир: если будешь проектировать API,
  где провал — норма, но причина важна, `expected` часто чище, чем «optional + отдельный код ошибки».

> Памятка-эвристика: «нет данных» — `optional`; «нарушен инвариант / ошибка вызывающего» —
> исключение (с барьером на границе слоя); «провал — штатный исход, и причина важна» — `expected`
> (когда доступен C++23).

### 4. `std::shared_mutex` — компромисс для read-heavy базы

В ядре стоит один `std::mutex`: каждый публичный метод берёт его эксклюзивно. Это **корректно**
и просто, но грубо: два потока, которые лишь *читают* (`get`, `keys`), всё равно выстраиваются в
очередь, хотя одновременное чтение гонки не создаёт. Для базы, где читают намного чаще, чем пишут
(типичный профиль кэша/БД), это узкое место.

**`std::shared_mutex` (C++17)** даёт схему «многие читатели / один писатель»:
- читатели берут **разделяемый** замок — `std::shared_lock<std::shared_mutex>` — и пускаются
  парал­лельно; одновременных читателей может быть сколько угодно;
- писатель берёт **эксклюзивный** замок — `std::unique_lock<std::shared_mutex>` — и на это время
  не пускает ни читателей, ни других писателей.

В MiniDB это означало бы: `get/llen/lrange/hget/hkeys/keys/size/ttl/type` — под `shared_lock`,
а `set/lpush/hset/del/expire/persist/save/load` — под `unique_lock`. Чтения «горячих» ключей
перестали бы толкаться.

**Но компромисс не бесплатен.** (1) `shared_mutex` *тяжелее* обычного: внутри он считает читателей
и согласует их с писателем, поэтому при коротких критических секциях и низкой конкуренции простой
`mutex` нередко **быстрее**. (2) Возможна **starvation писателя**: при непрерывном потоке
читателей писатель может долго не получать эксклюзив (политика зависит от реализации). (3) Ловушка
**ленивого протухания**: в MiniDB даже `get` *модифицирует* `data_` (стирает протухший ключ под
`erase`) — а это уже запись, и под `shared_lock` её делать нельзя (гонка!). Значит, чистый
read/write-разбор сломался бы: либо «чтение с возможной чисткой» придётся всё же брать
эксклюзивно, либо разносить «наблюдение» и «уборку» (например, не стирать под чтением, а помечать).
Вот почему в эталоне оставлен один `mutex`: он заведомо корректен. `shared_mutex` — честная
оптимизация, но требует пересмотра инвариантов; обсуди со мной, если захочешь её внедрить.

> Замечание про C++20/23-инструменты синхронизации (`std::jthread`, `std::latch`, `std::barrier`,
> `std::counting_semaphore`): они существуют и местами удобнее, но в этом треке мы их **в коде не
> применяем** (libc++ поддерживает их неровно) — только как теорию. В коде придерживаемся надёжной
> классики: `mutex`/`lock_guard`/`unique_lock`/`shared_lock`, `atomic`, `thread`, `async`/`future`.

### Стретч-цель (необязательно): `Cache<K, V>` — обобщённый LRU

Ядро MiniDB опирается на шаблоны и умные указатели «по краям» (variant, `lock_guard`-RAII), но не
показывает их *в полный рост*. Если хочется добить эти темы — в `include/cache.hpp` лежит
**header-only LRU-кэш** `Cache<K, V>` (тесты — `tests/test_cache.cpp`, входят в `minidb_tests`).
Это самодостаточный заголовок, НЕ связанный с ядром: он демонстрирует (1) **шаблоны** (модуль 07) —
кэш параметризован типами ключа и значения; (2) **владение через `std::unique_ptr`** (модули 04/06) —
двусвязный список узлов, где `next` — владеющий `unique_ptr`, а `prev` — *наблюдающий* сырой
указатель (так нет цикла владения и не нужен ни один `delete`); (3) политику вытеснения **LRU**
(`get`/`put` за O(1) через список порядка использования + `unordered_map`). Разбери реализацию как
упражнение на ownership-дисциплину: где владеет `unique_ptr`, почему `prev` обязан быть сырым, и
что сломается, если сделать оба направления владеющими. Полноценный вариант этой задачи как
*отдельного* капстоуна описан ниже (пункт 2).

Сейчас, до единой строки реализации, всё должно **собираться**, а тесты — **падать**
(стабы намеренно возвращают неверные результаты). Это нормальный старт капстоуна: красные тесты —
твой список задач.

## Как работать со мной

Здесь нет подсказок-«раскладушек», как в обычных модулях, — капстоун проектируешь ты. Но я рядом:

- **Заклинило на старте?** Скажи «разбери со мной майлстоун 1» — обсудим устройство `Value`,
  не давая готовый код.
- **Не сходится тест?** Покажи имя теста и текущую реализацию метода — разберём, что именно
  не совпало с контрактом.
- **Многопоточность пугает?** Спроси «почему мой `test_concurrency` иногда зелёный без мьютекса» —
  это важный разговор про коварство гонок (и про ThreadSanitizer, см. README модуля 14).
- **Дедлок / зависает?** Почти наверняка публичный метод зовёт другой публичный под тем же
  мьютексом. Опиши, кто кого зовёт, — найдём.
- **Хочу глубже.** Спрашивай про `std::shared_mutex` (читатели/писатели), про `std::scoped_lock`
  для нескольких мьютексов, про формат сериализации — это уже «сеньорские» темы поверх задания.

Проси **объяснение и направление**, а не готовое решение: смысл капстоуна — что финальную
архитектуру собрал ты.

## Альтернативные капстоуны (если хочется другого домена)

Тот же набор навыков (variant/шаблоны, RAII/умные указатели, move, STL, ошибки и **обязательно
многопоточность**) можно отработать на другом проекте. Если MiniDB не зажигает — обсуди со мной
один из них, я помогу составить спецификацию и тесты в таком же формате:

1. **ThreadPool + параллельный конвейер задач.** Пул рабочих потоков, потокобезопасная очередь
   задач (`std::mutex` + `std::condition_variable`), `submit()` возвращает `std::future`
   (модуль 14 в полный рост). Поверх — построй параллельный `map`/`reduce` над большим массивом
   и сравни время с однопоточным. Здесь сильнее акцент на синхронизацию и `future`/`promise`.

2. **Потокобезопасный кэш с TTL и политикой вытеснения (LRU).** `Cache<K,V>` на шаблонах
   (модуль 07), хранит значения с временем жизни (инъекция часов, как тут), вытесняет давно
   неиспользуемые при переполнении (LRU через `list` + `unordered_map`), и обслуживает
   конкурентные `get/put` под мьютексом. Меньше про парсинг, больше про структуры данных
   (модуль 13) и обобщённость.

Оба используют те же модули и ту же дисциплину «спецификация + красные тесты». MiniDB —
рекомендуемый путь, потому что задействует максимум пройденного сразу.
