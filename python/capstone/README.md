# Капстоун — CLI-менеджер задач (todo) с хранением в JSON

Поздравляю: ты дошёл до финала. Все предыдущие модули были «тренажёром» — здесь
ты впервые получаешь задачу так, как её получает junior на работе: **спецификация
плюс критерии готовности (приёмочные тесты), но без пошаговой инструкции**. Архитектуру
ты проектируешь сам. Это нормально, если поначалу непонятно, с чего начать — именно
ради этого ощущения капстоун и существует.

Большая идея капстоуна: *собрать из разрозненных навыков одну работающую программу.*
Класс-хранилище (модули 03, 06), разбор пользовательского ввода (02, 04), сохранение
состояния на диск в JSON (12), цикл «прочитал — выполнил — напечатал» (REPL). По
отдельности ты это уже умеешь. Здесь надо соединить.

## Что строим

Маленькую программу в терминале, которая хранит список задач и понимает команды. Ты
печатаешь строку — программа отвечает строкой. Это называется **REPL** (Read–Eval–Print
Loop): читаем ввод, вычисляем результат, печатаем ответ, повторяем.

Команды (регистр команды не важен: `add` и `ADD` — одно и то же):

| Команда            | Что делает                                        | Ответ                       |
|--------------------|---------------------------------------------------|-----------------------------|
| `ADD <title>`      | добавить задачу с текстом `<title>`               | `<id>` новой задачи         |
| `DONE <id>`        | пометить задачу `<id>` выполненной                | `OK`                        |
| `RM <id>`          | удалить задачу `<id>`                              | `OK`                        |
| `LIST`             | показать ВСЕ задачи (по возрастанию id)           | многострочный список        |
| `PENDING`          | показать только невыполненные задачи              | многострочный список        |
| `SAVE <path>`      | сохранить состояние в JSON-файл `<path>`          | `OK`                        |
| `LOAD <path>`      | загрузить состояние из JSON-файла `<path>`        | `OK`                        |
| `QUIT`             | выйти из программы                                 | (REPL завершается)          |

Ошибки не роняют программу, а возвращают строку, начинающуюся с `ERR `. Например,
`DONE 999` для несуществующей задачи → `ERR no such task: 999`.

### Пример сессии

```
> ADD купить молоко
1
> ADD написать резюме
2
> ADD позвонить маме
3
> DONE 2
OK
> LIST
[ ] 1 купить молоко
[x] 2 написать резюме
[ ] 3 позвонить маме
> PENDING
[ ] 1 купить молоко
[ ] 3 позвонить маме
> RM 1
OK
> SAVE todo.json
OK
> DONE 999
ERR no such task: 999
> QUIT
```

Формат строки задачи в `LIST`/`PENDING`: `[x] <id> <title>` для выполненной и
`[ ] <id> <title>` для невыполненной (между скобкой, id и заголовком — по одному пробелу).
Если задач нет — отдельная строка `(нет задач)` (для `PENDING` — `(нет невыполненных задач)`).

## Рекомендуемая архитектура

Раздели «что делает программа» и «как с ней говорят». Это классическое разделение
ответственности (separation of concerns) — каждый кусок проще писать и тестировать
по отдельности.

```
capstone/
├── app.py               # REPL: ввод/вывод, главный цикл. УЖЕ написан — читай, не трогай.
├── tasks/
│   ├── __init__.py      # реэкспорт TaskStore и execute (готов)
│   ├── store.py         # ← ТЫ: класс TaskStore — данные и операции над ними
│   └── commands.py      # ← ТЫ: execute(store, line) — разбор строки в вызов метода
└── tests/
    ├── test_store.py    # приёмочные тесты ядра и персистентности
    └── test_commands.py # приёмочные тесты разбора команд
```

- **`TaskStore` (store.py)** — «модель». Хранит задачи и ничего не знает про строки
  команд. Задача — это `dict {"id": int, "title": str, "done": bool}`. Методы:
  `add`, `complete`, `remove`, `all`, `pending`, `count`, `save`, `load`. Полные
  сигнатуры и контракт — в docstring'ах стаба и в тестах `test_store.py`.
- **`execute` (commands.py)** — «контроллер». Берёт сырую строку, понимает, какая это
  команда, дёргает нужный метод `TaskStore` и форматирует ответ. Никакого `print` и
  `input` здесь нет — только строка на входе, строка на выходе. Поэтому это легко
  тестировать (см. `test_commands.py`).
- **`app.py`** — «вид» (тонкий слой ввода/вывода). Он создаёт `TaskStore`, в цикле
  читает строку, зовёт `execute`, печатает ответ. Уже написан и работает — твоя
  реализация `TaskStore` и `execute` «оживят» его.

Почему именно так: тесты гоняют `TaskStore` и `execute` напрямую, без терминала. Это
и есть главный приём — **выносить логику из I/O-цикла туда, где её можно проверить
тестами.** `app.py` остаётся настолько тупым, что его можно не тестировать вовсе.

## Майлстоуны

Иди по порядку — каждый следующий опирается на предыдущий. Запускай тесты часто:

```bash
./python/run.sh
```

(капстоун подхватывается тем же `run.sh`, что и модули; можно сузить до
`./python/.venv/bin/pytest python/capstone`, когда окружение уже создано).

### Майлстоун 1 — хранилище задач (модули 03, 06, 08)

Реализуй `TaskStore` в `store.py` (кроме `save`/`load` — они в майлстоуне 3).
Опирайся на: списки/словари (модуль 03), классы и методы (06), исключения (08 —
понадобятся для краевых случаев в майлстоуне 2). Подумай заранее:

- Где хранить задачи? Список словарей или словарь `id → задача`? У каждого варианта
  свой компромисс: список проще выводить по порядку, словарь — быстрее искать по id.
- Откуда брать новый id? Он должен **расти и не переиспользоваться** даже после
  удаления (удалил задачу 2 — следующая всё равно получит больший id, а не займёт 2).
  Подумай, какое поле-счётчик за это отвечает.
- `all()` обязан возвращать задачи **по возрастанию id**. `pending()` — только те, у
  которых `done == False`. Цель — чтобы тесты `test_store.py` стали зелёными.

Зелёные тесты `test_store.py` (кроме блока про save/load) — майлстоун закрыт.

### Майлстоун 2 — разбор команд (модули 02, 04)

Реализуй `execute(store, line)` в `commands.py`. Опирайся на: ветвление (модуль 02),
работу со строками — `split`, `strip`, `lower`, срезы (04). Подумай:

- Как отделить **имя команды** от **аргумента**? Подсказка: `line.split(maxsplit=1)`
  даёт ровно две части — слово и «весь остаток как есть» (важно: заголовок задачи
  может содержать пробелы, его дробить нельзя).
- Регистр имени команды игнорируем (`add` == `ADD`), но **заголовок задачи трогать
  нельзя** — его регистр и пробелы сохраняем как есть.
- Числовой аргумент (`<id>`) надо превратить в `int`. Что если там не число? Это
  ошибка ввода → верни `ERR ...`, а не дай программе упасть.
- Пустая строка и неизвестная команда — тоже `ERR ...`, не падение.

Точные тексты ответов и ошибок — в `test_commands.py`. Не угадывай, а смотри тесты:
они и есть спецификация.

### Майлстоун 3 — персистентность JSON (модуль 12)

Реализуй `save(path)` и `load(path)` в `TaskStore`. Опирайся на модуль 12: `json.dump`
/`json.load`, `open` через `with`, `pathlib.Path`. Подумай:

- Что именно сохранять? Не только список задач, но и **счётчик id** — иначе после
  `LOAD` новый `ADD` может выдать уже занятый id. Сохраняй всё состояние целиком.
- `load` должен **заменить** текущее содержимое хранилища, а не дописать к нему.
- После `save` в файл и `load` обратно состояние обязано совпасть (round-trip). Тесты
  `test_store.py` проверяют это через `tmp_path`.

### Майлстоун 4 — REPL и стретч-цели

`app.py` уже готов. Запусти его и поиграй:

```bash
./python/.venv/bin/python python/capstone/app.py
```

Когда базовый функционал зелёный — попробуй расширить (по желанию, тестов на это нет —
ты сам себе заказчик):

- команда `CLEAR` — удалить все выполненные задачи;
- приоритеты или дедлайны у задачи (новое поле + команда `PRI <id> <n>`);
- автосохранение в файл по умолчанию при `QUIT` и автозагрузка при старте;
- цветной вывод; команда `HELP` со списком команд;
- поиск: `FIND <подстрока>` — задачи, чей заголовок содержит подстроку.

Сначала **напиши тест** на свою фичу, потом код — ты уже умеешь и то, и другое.

## Как работать над капстоуном со мной

1. Сначала прочитай `test_store.py` и `test_commands.py` целиком. Тесты — это
   ТЗ заказчика: они показывают желаемое поведение точнее любого текста.
2. Набросай на бумаге/в комментариях, как хранишь задачи и как разбираешь строку.
   Принеси набросок — обсудим развилки ДО кода (это дешевле, чем переписывать потом).
3. Реализуй майлстоун, гоняй тесты, доводи красное до зелёного.
4. Застрял — покажи код и текст падающего теста. Я подскажу ступенчато, готового
   решения не дам: цель — чтобы ты понимал каждую строку, которую написал.

## Альтернативные капстоуны

Если todo-менеджер не цепляет — возьми что-то из этого (та же идея: спецификация +
свои тесты, архитектуру проектируешь сам). Скажи мне, и я подготовлю приёмочные тесты.

- **Анализатор логов (CLI).** На входе текстовый лог веб-сервера; на выходе сводка:
  сколько запросов, топ-N URL, сколько ошибок 4xx/5xx, разбивка по часам. Тренирует
  строки (04), словари-счётчики и `collections.Counter` (03, 11), файлы (12),
  аргументы командной строки (`argparse`). Уже есть заготовка в `modules/12-files-data/
  loganalyzer.py` — её можно развить до полноценного инструмента.
- **Упрощённый конвертер Markdown → HTML.** Берёт текст с заголовками (`# …`),
  списками (`- …`), жирным (`**…**`) и абзацами, выдаёт HTML. Тренирует разбор строк
  построчно (02, 04), функции (05), аккуратную работу с краевыми случаями. Намеренно
  упрощённый (без вложенности и таблиц) — иначе утонешь в спецификации Markdown.
