# Капстоун — мини-СУБД с SQL-подобным языком запросов

Это **финальный проект курса**. Он крупнее всего, что было раньше, и сознательно собирает
вместе почти всё пройденное: разбор текста (токенизатор + парсер), ООП, генераторы,
исключения и аккуратную обработку ошибок, аннотации типов и стандартную библиотеку.

Важно понять формат. До этого модули водили тебя за руку: маленькая функция, подсказки,
зелёный тест. **Капстоун устроен иначе.** Здесь нет пошаговой инструкции «сделай 1, потом 2».
Есть **спецификация** (этот README + docstring'и в стабах) и **приёмочные тесты** — они и есть
точное определение «готово». Архитектуру внутри ты проектируешь сам. Я рядом — но как
ревьюер и собеседник, а не как пошаговый рецепт.

Тесты стартуют **красными**: все стабы кидают `NotImplementedError`. Твоя работа — сделать
их зелёными, не трогая тесты.

## Что мы строим

Маленькую базу данных, которая живёт в оперативной памяти и понимает урезанный SQL. Никаких
файлов и сети — только текст на входе и структуры данных внутри. Поддерживаемый язык:

```
CREATE TABLE users (id, name, age)
INSERT INTO users VALUES (1, Alice, 30)
SELECT * FROM users
SELECT name, age FROM users WHERE age > 25
SELECT * FROM users WHERE age >= 18 AND name = Bob
SELECT * FROM users ORDER BY age DESC
DELETE FROM users WHERE id = 1
```

Правила языка (намеренно простые, чтобы фокус был на архитектуре, а не на синтаксисе SQL):

- Значения — это **целые числа** (`30`, `-7`) или **слова без кавычек** (`Alice`, `Bob`).
  Число превращается в `int`, всё остальное остаётся `str`.
- Операторы сравнения: `=  !=  <  >  <=  >=`.
- Несколько условий в `WHERE` склеиваются только через `AND` (логическое И).
- `ORDER BY <колонка>` сортирует по возрастанию; добавь `DESC` — по убыванию.
- Ключевые слова можно писать в любом регистре (`SELECT` == `select`), а вот имена таблиц,
  колонок и значения сохраняются как есть.

### Пример живой сессии (как это выглядит в REPL)

Когда движок будет реализован, `python app.py` даст вот такое:

```
$ python app.py
мини-СУБД. Введите запрос или .quit для выхода.
db> CREATE TABLE users (id, name, age)
OK
db> INSERT INTO users VALUES (1, Alice, 30)
OK
db> INSERT INTO users VALUES (2, Bob, 20)
OK
db> INSERT INTO users VALUES (3, Carol, 25)
OK
db> SELECT name, age FROM users WHERE age >= 25 ORDER BY age DESC
name  | age
------+----
Alice | 30
Carol | 25
db> DELETE FROM users WHERE id = 1
удалено строк: 1
db> SELECT * FROM users WHERE age >= 18 AND name = Bob
id | name | age
```

`app.py` уже написан и работает — его трогать не нужно. Пока движок в стабах, он будет
печатать `(не реализовано: ...)` вместо результата. Это удобный «градусник» прогресса.

## Рекомендуемая архитектура

Конвейер из трёх слоёв. Каждый слой тупее и независимее предыдущего — так его легче
тестировать и легче держать в голове.

```
  строка SQL
      │
      ▼   tokenize()                  db/tokenizer.py   — слой 1: ЛЕКСЕР
  список токенов ["SELECT","*","FROM","users"]
      │
      ▼   parse()                     db/parser.py      — слой 2: ПАРСЕР
  объект запроса  Select(table="users", columns=["*"], ...)
      │
      ▼   Database.execute()          db/database.py    — слой 3: ДВИЖОК
  результат: None | list[dict] | int
```

1. **Токенизатор** (`db/tokenizer.py`). Режет строку на атомы-токены. Ничего не знает про
   смысл. `(`, `)`, `,` и операторы `= != < > <= >=` — отдельные токены; пробелы —
   разделители.
2. **Парсер** (`db/parser.py`). Из плоского списка токенов собирает **объект запроса** —
   один из dataclass'ов `Create / Insert / Select / Delete` (они уже описаны в файле; их
   форма зафиксирована тестами, менять не нужно). Любой **синтаксический** мусор →
   `QueryError`. Условия `WHERE` хранятся как список `Condition(column, op, value)`.
3. **Движок** (`db/database.py`). Хранит таблицы в памяти и исполняет запрос. Ловит
   **семантические** ошибки: таблица уже есть / её ещё нет / не то число значений /
   неизвестная колонка. Здесь же единая точка типизации значений — функция `coerce`
   (`"30" → 30`, `"Alice" → "Alice"`).

Почему именно так: разделение «синтаксис vs. семантика» — это классика. Парсер отвечает на
вопрос «правильно ли НАПИСАНО?», движок — «имеет ли это СМЫСЛ на текущих данных?». Когда эти
две заботы не смешаны, и код, и сообщения об ошибках получаются чище.

`QueryError` — единственный тип ошибки, который видит пользователь. Он определён в
`database.py` и реэкспортируется пакетом (`from db import Database, QueryError`). Парсер тоже
им пользуется — импортируй его внутри функции `parse`, чтобы не словить циклический импорт
(подсказка есть прямо в `parser.py`).

## Майлстоуны (привязаны к модулям курса)

Двигайся снизу вверх по конвейеру — каждый слой опирается на готовый предыдущий. После
каждого шага запускай ровно его тесты (команды ниже).

**Майлстоун 0 — старт.** Проверь, что всё импортируется и тесты честно красные:
`pytest capstone -q` из каталога `python/`. Увидишь `NotImplementedError` — это правильно.

**Майлстоун 1 — токенизатор.** Реализуй `tokenize`. Зелёным должен стать `test_tokenizer.py`.
Опирается на: работу со строками (модуль 04), `list`/срезы (модуль 03), при желании `re`
из стандартной библиотеки (модуль 11).

**Майлстоун 2 — парсер.** Реализуй `parse`: распознай четыре команды, собери dataclass'ы,
поднимай `QueryError` на синтаксис. Зелёным станет `test_parser.py`. Опирается на:
dataclasses (модуль 07), исключения (модуль 08), аннотации типов (модуль 13), функции и
декомпозиция (модуль 05).

**Майлстоун 3 — движок: CREATE / INSERT.** Реализуй `Database.__init__`, `coerce` и ветки
CREATE/INSERT в `execute`. Зеленеют тесты на создание/вставку и типы значений. Опирается на:
ООП и инкапсуляцию состояния (модуль 06), исключения (модуль 08).

**Майлстоун 4 — движок: SELECT.** Проекция колонок, фильтр `WHERE` (включая `AND`),
`ORDER BY ASC/DESC`. Зеленеет основная масса `test_database.py`. Опирается на: генераторы
для ленивой фильтрации строк (модуль 09), `sorted`/`operator` из стандартной библиотеки
(модуль 11).

**Майлстоун 5 — движок: DELETE + шлифовка.** `DELETE` возвращает число удалённых строк;
дочини оставшиеся ошибочные сценарии. Когда `pytest capstone` полностью зелёный — запусти
`python app.py` и проиграй пример сессии руками. Опирается на: всё вышеперечисленное.

Не обязательно идти строго по порядку внутри слоя, но **слой за слоем** — самый спокойный
путь: красный тест парсера почти всегда чинится в парсере, а не в движке.

## Майлстоуны 6–7 — необязательные, но тестируемые (выходные навыки)

Базовый конвейер (майлстоуны 0–5) делает мини-СУБД законченной: текст на входе → таблицы в
памяти → результат. Но настоящая СУБД умеет ещё две вещи, которых нам пока не хватает, —
**пережить перезапуск** (данные на диске) и **применять пачку операций атомарно** (всё или
ничего). Это ровно те «выходные» навыки курса, что иначе остались бы без капстоуна:
работа с файлами и JSON (модуль 12) и контекст-менеджеры (модуль 08).

Эти майлстоуны **необязательны для зачёта базового капстоуна**, но у них есть свои приёмочные
тесты (`tests/test_persistence_props.py`) — то есть «готово» для них определено так же строго,
как и для движка. Берись за них, когда основной конвейер уже зелёный.

**Майлстоун 6 — персистентность (`save` / `load`).** Научи базу выгружаться на диск и
читаться обратно. Контракт:

```
db.save(path)            -> записывает базу в JSON-файл, возвращает Path
Database.load(path)      -> classmethod: читает файл и собирает НОВЫЙ Database
```

Главный инвариант — **round-trip**: `Database.load(db.save(path)) == db`. Чтобы это
равенство имело смысл, базе нужен `__eq__` (две базы равны, когда совпадает их содержимое).
Опирается на: файлы и JSON (модуль 12 — `json.dump`/`json.load`), `pathlib.Path`, оператор
`with` для гарантированного закрытия файла (модуль 08), classmethod как альтернативный
конструктор (модуль 06).

**Почему JSON, а не `pickle`.** JSON — текстовый, человекочитаемый, переносимый между языками
и безопасный при чтении (в отличие от `pickle`, который исполняет произвольный код). Наши
значения — это `int` и `str`, ключи строки, строки словари: всё это `json` сериализует без
единой настройки. Цена — JSON не умеет произвольные объекты, поэтому таблицу нужно превратить
в «чистые данные» (списки и словари) перед записью.

**Подвох.** Не сериализуй объект `_Table` напрямую — у `json` нет правила, как превратить его
в текст. Сначала разложи базу в простой `dict` (`{имя: {"columns": [...], "rows": [...]}}`),
потом отдай его `json.dump`. На чтении — обратный путь: из словаря собери `_Table` заново.

**Майлстоун 7 — транзакция как контекст-менеджер.** Дай возможность выполнить пачку операций
атомарно: если посреди блока вылетает исключение, состояние откатывается к тому, что было
**до входа** в блок.

```
with db.transaction():
    db.execute("INSERT INTO users VALUES (2, Bob, 20)")
    db.execute("DELETE FROM ghosts")   # QueryError -> откат: INSERT выше тоже отменён
```

При нормальном выходе изменения фиксируются; при выходе по исключению — откатываются, а само
исключение пробрасывается дальше (мы его **не глотаем**). Опирается на: контекст-менеджеры
(модуль 08) — `@contextmanager` или пара `__enter__`/`__exit__`; копирование состояния
(`copy.deepcopy`) для снимка.

**Почему снимок — глубокая копия.** Строки таблиц — изменяемые словари. Если запомнить просто
ссылку на `self._tables`, то мутация строки «на месте» испортит и снимок. `copy.deepcopy`
делает независимую копию всего дерева, и откат честно возвращает дотранзакционное состояние.

**Подвох.** Ловить надо `BaseException`, а не только `Exception`: тогда откат сработает даже
на `KeyboardInterrupt`/`GeneratorExit`. И обязательно `raise` после отката — проглоченное
исключение спрячет ошибку от вызывающего кода и сломает тест на «исключение видно снаружи».

### Подсказки (майлстоуны 6–7)

<details>
<summary>Как разложить базу в JSON-совместимый вид?</summary>

Таблица — это `columns` (список строк) и `rows` (список словарей `col -> int|str`). И то,
и другое уже «чистый» JSON. Соедини всё в один словарь верхнего уровня: ключ — имя таблицы,
значение — `{"columns": [...], "rows": [...]}`. Этот словарь и отдавай в `json.dump`.
</details>

<details>
<summary>Почему load — это classmethod, а не обычный метод?</summary>

`load` не меняет существующую базу, а **создаёт новую** из файла. Альтернативный конструктор —
классический повод для `@classmethod`: внутри пишешь `db = cls()` и наполняешь его. Снаружи
читается как `Database.load(path)` — симметрично `Database()`.
</details>

<details>
<summary>Минимальный каркас контекст-менеджера транзакции</summary>

С `@contextmanager` это функция-генератор с одним `yield`: код до `yield` — «вход» (сделай
снимок), `yield` — тело блока, а `try/except` вокруг `yield` ловит вылетевшее исключение,
восстанавливает снимок и делает `raise`. На нормальном выходе снимок просто не используется.
Альтернатива — класс с `__enter__`/`__exit__`, где `__exit__(exc_type, ...)` при ненулевом
`exc_type` откатывает состояние и возвращает «не подавлять исключение».
</details>

## Как запускать

Все команды — из каталога `python/` (там лежит `pytest.ini`):

```
pytest capstone -q                                  # весь капстоун
pytest capstone/tests/test_tokenizer.py -q          # один слой
pytest capstone/tests/test_parser.py -q
pytest capstone/tests/test_database.py -q
pytest capstone/tests/test_persistence_props.py -q  # майлстоуны 6–7 (save/load + транзакция)
pytest capstone -q -k where                          # точечно по имени теста
python capstone/app.py                               # ручной REPL
```

`conftest.py` сам добавляет каталог капстоуна в `sys.path`, так что импорт `from db ...`
работает из любой директории.

## Как работать со мной

- **Сначала прочитай тест, потом пиши код.** Тест — это и есть точная постановка задачи.
  Не уверен, чего хочет тест, — спроси меня, что он проверяет и почему.
- **Проси разбор, а не готовый код.** «Объясни, как устроить курсор по токенам», «как
  отличить синтаксическую ошибку от семантической», «почему здесь циклический импорт и как
  его аккуратно разорвать» — на такие вопросы я отвечу с удовольствием.
- **Приходи с красным тестом и трассировкой.** Покажи команду, упавший тест и стектрейс —
  вместе быстро найдём причину.
- **Защити свои решения.** Где хранить таблицы? Приводить тип в парсере или в движке? Делать
  `Condition` неизменяемым? Готовых «единственно верных» ответов тут нет — есть обоснованные.
  Я буду спрашивать «почему именно так».

## Если хочется другого капстоуна

Тот же набор навыков (разбор + ООП + генераторы + ошибки + типы + stdlib) можно прокачать на
другом сюжете. Два равноценных варианта, если этот не зажёг:

1. **Калькулятор выражений с переменными.** Токенизатор + парсер по грамматике с приоритетом
   операторов (`2 + 3 * (4 - 1)`), вычислитель по дереву, присваивания `x = 5` и
   использование переменных. Те же три слоя, но в центре — рекурсивный спуск и приоритеты.
2. **Движок задач из лога команд (mini task tracker).** Язык вроде `ADD buy milk #home`,
   `DONE 3`, `LIST #home`, `LIST DONE`; парсинг команд, фильтры по тегам/статусу, сортировки,
   `QueryError` на кривые команды. Ближе к «реальному CLI», та же архитектура парсер→движок.

Выбери один и доведи до конца — ценность капстоуна в законченности, а не в количестве начатого.
