# Модуль 10 — Модули, пакеты и импорты

До сих пор твой код жил в одном файле. Реальные программы так не пишут: код разбивают
на части и собирают обратно через **импорты**. Большая идея модуля: *модуль — это просто
.py-файл, а пакет — каталог с файлом `__init__.py`; импорт — это способ дать имени из
одного файла жить в другом.* Поняв это, ты перестанешь бояться `ModuleNotFoundError` и
поймёшь, почему `import` иногда «не видит» твой код.

## Идея 1. Модуль = .py-файл, пакет = каталог с `__init__.py`

**Модуль** — любой файл `foo.py`. Импортируя его, ты получаешь доступ к его именам:

```python
# в файле greet.py
def hello(name):
    return f"Привет, {name}"

# в другом файле
import greet
greet.hello("Аня")        # обращаемся через имя модуля
```

**Пакет** — это каталог, который Python считает «модулем из нескольких файлов». Раньше
обязательным признаком был файл `__init__.py` внутри каталога. Структура нашего задания:

```
mathx/                  ← это ПАКЕТ (каталог)
├── __init__.py         ← «лицо» пакета; выполняется при import mathx
├── arithmetic.py       ← подмодуль mathx.arithmetic
└── sequences.py        ← подмодуль mathx.sequences
```

ПОЧЕМУ так: пакет позволяет сгруппировать связанные модули под одним именем
(`mathx.arithmetic`, `mathx.sequences`) и спрятать внутреннюю кухню за `__init__.py`.
Снаружи человек пишет `from mathx import add` и ему не важно, в каком именно файле эта
`add` физически лежит.

## Идея 2. `import`, `from … import` и как Python ищет модули (sys.path)

Есть несколько форм импорта — и они дают разные имена:

```python
import mathx                       # имя mathx; зовёшь mathx.add(...)
import mathx.arithmetic as ar      # длинный путь под коротким псевдонимом: ar.add(...)
from mathx import add, mul         # имена add, mul прямо в текущем файле
from mathx.sequences import fib    # одно конкретное имя из конкретного подмодуля
```

ПОЧЕМУ важно понимать, **где** Python ищет модуль. Когда ты пишешь `import mathx`,
интерпретатор перебирает каталоги из списка `sys.path` и в первом подходящем находит
`mathx`. Если каталога с `mathx` нет ни в одном из путей — получишь `ModuleNotFoundError`.

```python
import sys
print(sys.path)   # список каталогов, где ищутся модули
```

В нашем курсе тест `test_mathx.py` лежит **в том же каталоге**, что и пакет `mathx/`.
pytest автоматически добавляет каталог теста в `sys.path`, поэтому `from mathx import …`
просто работает — никаких ручных манипуляций с путями не нужно. Это типичная причина
ошибок у новичков: «код есть, а импорт не находит» почти всегда означает «нужного
каталога нет в `sys.path`».

> Подводный камень: не называй свои файлы именами стандартных модулей (`json.py`,
> `random.py`, `string.py`). Свой файл может «перекрыть» стандартный, и потом `import json`
> подтянет твой файл вместо настоящего — будут загадочные ошибки.

## Идея 3. `if __name__ == "__main__":` — запуск vs импорт

У каждого модуля есть скрытая переменная `__name__`. Её значение зависит от того, **как**
модуль попал в программу:

- запустили файл напрямую (`python foo.py`) → внутри `foo.py` будет `__name__ == "__main__"`;
- этот же файл кто-то импортировал (`import foo`) → внутри будет `__name__ == "foo"`.

```python
# в файле foo.py
def main():
    print("работаю как программа")

if __name__ == "__main__":   # выполнится ТОЛЬКО при прямом запуске
    main()
```

ПОЧЕМУ это нужно: при `import foo` Python **исполняет весь файл сверху вниз**, чтобы
создать его функции. Если бы код запуска не был спрятан под этим `if`, он бы срабатывал
и при простом импорте — а ты обычно хочешь просто переиспользовать функции, а не
запускать чужую программу. Эта строчка отделяет «я — библиотека» от «я — точка входа».

## Идея 4. Зачем вообще разбивать код на модули

Можно ведь всё свалить в один файл — почему не стоит:

- **Понятность.** Имя файла подсказывает, что внутри: `arithmetic.py` — про числа,
  `sequences.py` — про последовательности. Не нужно листать тысячу строк.
- **Переиспользование.** Готовый модуль импортируется в любой другой проект как есть.
- **Изоляция.** Поломал что-то в `sequences.py` — `arithmetic.py` это не задело.
- **Командная работа.** Разные люди правят разные файлы, меньше конфликтов.

`__init__.py` здесь играет роль «витрины»: внутри пакета может быть десяток файлов, но
наружу через `from .arithmetic import add` выставляется только то, что считается публичным
API. Пользователю пакета не нужно знать его внутреннюю раскладку.

<details><summary>Задание 4 — fib без рекурсии</summary>

</details>

<code>mathx.arithmetic.add</code>, и просто <code>mathx.add</code>. Это и есть реэкспорт.
</details>

## Задание (существенное): подмодуль `mathx.statistics` — описательная статистика

Ты уже видел, как `mathx.__init__.py` собирает публичное API из нескольких подмодулей.
Теперь ты *расширяешь пакет* сам: добавляешь третий подмодуль `mathx/statistics.py`
и четыре функции описательной статистики. Идея задания — не выучить формулы (они простые),
а почувствовать, как пакет *проектируется* снаружи внутрь: сначала «какой API я хочу
видеть в тесте», потом — «что лежит в каком файле».

Стаб уже создан в `mathx/statistics.py`, реэкспорт в `__init__.py` — тоже готов.
Твоя задача: реализовать тела четырёх функций.

### Спецификация API

```python
from mathx import mean, median, mode, variance
```

**`mean(data: Sequence[float]) -> float`**
Среднее арифметическое: сумма элементов, делённая на их количество.
- Пустой список → `ValueError`.

**`median(data: Sequence[float]) -> float`**
Медиана отсортированного ряда.
- Нечётное количество — средний элемент.
- Чётное количество — среднее двух центральных.
- Исходный `data` изменять нельзя (работай с отсортированной копией).
- Пустой список → `ValueError`.

**`mode(data: Sequence[float]) -> float`**
Наиболее частый элемент. При нескольких одинаково частых — наименьший.
- Пустой список → `ValueError`.

**`variance(data: Sequence[float], ddof: int = 0) -> float`**
Дисперсия: среднее квадратов отклонений от среднего.
- `ddof=0` — делим на `N` (генеральная).
- `ddof=1` — делим на `N - 1` (выборочная, исправленная по Бесселю).
- Пустой список → `ValueError`.
- `len(data) < 2` при `ddof=1` → `ValueError` (деление на 0).
- `ddof` не 0 и не 1 → `ValueError`.

> Нельзя делать `import statistics` (стандартная библиотека): реализуй алгоритмы сам.
> `math` и `collections` — разрешены.

### Подсказки

<details><summary>Подсказка 1 — с чего начать (mean)</summary>
Проверь <code>len(data) == 0</code> в самом начале и подними <code>ValueError</code>.
Среднее — это <code>sum(data) / len(data)</code>. Один вызов <code>sum()</code> —
никаких циклов.
</details>

<details><summary>Подсказка 2 — median: как работать с копией и найти центр</summary>
<code>sorted(data)</code> возвращает новый список, не меняя исходный.
Дальше:
<pre>
n = len(sorted_data)
mid = n // 2
if n % 2 == 1:
    return float(sorted_data[mid])
else:
    return (sorted_data[mid - 1] + sorted_data[mid]) / 2.0
</pre>
Почему <code>mid - 1</code> и <code>mid</code>? При чётном n два центральных элемента
стоят на позициях <code>n//2 - 1</code> и <code>n//2</code>.
</details>

<details><summary>Подсказка 3 — mode: подсчёт частот без лишнего кода</summary>
<code>collections.Counter(data)</code> даёт словарь <code>{значение: частота}</code>.
Найди максимальную частоту: <code>max_freq = max(counter.values())</code>.
Собери все элементы с такой частотой: <code>[k for k, v in counter.items() if v == max_freq]</code>.
Верни <code>min(...)</code> из них.
</details>

<details><summary>Подсказка 4 — variance: связь с mean и правильный знаменатель</summary>
Сначала посчитай среднее через уже готовую <code>mean(data)</code>.
Затем квадраты отклонений: <code>[(x - m) ** 2 for x in data]</code>.
Поделить на <code>n - ddof</code>, где <code>n = len(data)</code>.
Не забудь: при <code>ddof=1</code> и <code>n == 1</code> получится деление на 0 —
это и есть краевой случай с <code>ValueError</code>.
</details>
