# Модуль 10 — Модули, пакеты и импорты

До сих пор твой код жил в одном файле. Реальные программы так не пишут: код разбивают
на части и собирают обратно через **импорты**. Большая идея модуля: *модуль — это просто
.py-файл, а пакет — каталог с файлом `__init__.py`; импорт — это способ дать имени из
одного файла жить в другом.* Поняв это, ты перестанешь бояться `ModuleNotFoundError` и
поймёшь, почему `import` иногда «не видит» твой код.

## Идея 1. Модуль = .py-файл, пакет = каталог с `__init__.py`

**Модуль** — любой файл `foo.py`. Импортируя его, ты получаешь доступ к его именам:

```python
# в файле greet.py
def hello(name):
    return f"Привет, {name}"

# в другом файле
import greet
greet.hello("Аня")        # обращаемся через имя модуля
```

**Пакет** — это каталог, который Python считает «модулем из нескольких файлов». Раньше
обязательным признаком был файл `__init__.py` внутри каталога. Структура нашего задания:

```
mathx/                  ← это ПАКЕТ (каталог)
├── __init__.py         ← «лицо» пакета; выполняется при import mathx
├── arithmetic.py       ← подмодуль mathx.arithmetic
└── sequences.py        ← подмодуль mathx.sequences
```

ПОЧЕМУ так: пакет позволяет сгруппировать связанные модули под одним именем
(`mathx.arithmetic`, `mathx.sequences`) и спрятать внутреннюю кухню за `__init__.py`.
Снаружи человек пишет `from mathx import add` и ему не важно, в каком именно файле эта
`add` физически лежит.

## Идея 2. `import`, `from … import` и как Python ищет модули (sys.path)

Есть несколько форм импорта — и они дают разные имена:

```python
import mathx                       # имя mathx; зовёшь mathx.add(...)
import mathx.arithmetic as ar      # длинный путь под коротким псевдонимом: ar.add(...)
from mathx import add, mul         # имена add, mul прямо в текущем файле
from mathx.sequences import fib    # одно конкретное имя из конкретного подмодуля
```

ПОЧЕМУ важно понимать, **где** Python ищет модуль. Когда ты пишешь `import mathx`,
интерпретатор перебирает каталоги из списка `sys.path` и в первом подходящем находит
`mathx`. Если каталога с `mathx` нет ни в одном из путей — получишь `ModuleNotFoundError`.

```python
import sys
print(sys.path)   # список каталогов, где ищутся модули
```

В нашем курсе тест `test_mathx.py` лежит **в том же каталоге**, что и пакет `mathx/`.
pytest автоматически добавляет каталог теста в `sys.path`, поэтому `from mathx import …`
просто работает — никаких ручных манипуляций с путями не нужно. Это типичная причина
ошибок у новичков: «код есть, а импорт не находит» почти всегда означает «нужного
каталога нет в `sys.path`».

> Подводный камень: не называй свои файлы именами стандартных модулей (`json.py`,
> `random.py`, `string.py`). Свой файл может «перекрыть» стандартный, и потом `import json`
> подтянет твой файл вместо настоящего — будут загадочные ошибки.

## Идея 3. `if __name__ == "__main__":` — запуск vs импорт

У каждого модуля есть скрытая переменная `__name__`. Её значение зависит от того, **как**
модуль попал в программу:

- запустили файл напрямую (`python foo.py`) → внутри `foo.py` будет `__name__ == "__main__"`;
- этот же файл кто-то импортировал (`import foo`) → внутри будет `__name__ == "foo"`.

```python
# в файле foo.py
def main():
    print("работаю как программа")

if __name__ == "__main__":   # выполнится ТОЛЬКО при прямом запуске
    main()
```

ПОЧЕМУ это нужно: при `import foo` Python **исполняет весь файл сверху вниз**, чтобы
создать его функции. Если бы код запуска не был спрятан под этим `if`, он бы срабатывал
и при простом импорте — а ты обычно хочешь просто переиспользовать функции, а не
запускать чужую программу. Эта строчка отделяет «я — библиотека» от «я — точка входа».

## Идея 4. Зачем вообще разбивать код на модули

Можно ведь всё свалить в один файл — почему не стоит:

- **Понятность.** Имя файла подсказывает, что внутри: `arithmetic.py` — про числа,
  `sequences.py` — про последовательности. Не нужно листать тысячу строк.
- **Переиспользование.** Готовый модуль импортируется в любой другой проект как есть.
- **Изоляция.** Поломал что-то в `sequences.py` — `arithmetic.py` это не задело.
- **Командная работа.** Разные люди правят разные файлы, меньше конфликтов.

`__init__.py` здесь играет роль «витрины»: внутри пакета может быть десяток файлов, но
наружу через `from .arithmetic import add` выставляется только то, что считается публичным
API. Пользователю пакета не нужно знать его внутреннюю раскладку.

## Идея 5. `__all__`: публичный контракт пакета

В `mathx/__init__.py` есть строка:

```python
__all__ = ["add", "mul", "factorial", "fib", "mean", "median", "mode", "variance"]
```

Это список строк — явное объявление того, что считается **публичным API** пакета.

**Как он работает.** `__all__` управляет ровно одним случаем: `from mathx import *`. Когда
пишешь звёздочку, Python берёт только те имена, которые перечислены в `__all__`. Без него
`import *` захватил бы вообще всё, что есть в пространстве имён модуля (включая
импортированные внутренности — `Counter`, `math`, вспомогательные функции и т.д.).

```python
# Допустим, __all__ = ["add", "mul"]
from mathx import *
# В текущем пространстве имён — только add и mul.
# Если бы __all__ не было — попало бы всё.
```

**Что `__all__` НЕ ограничивает.** Это тонкое место: `__all__` влияет только на
«звёздочный» импорт. Явный импорт (`from mathx import factorial`, `import mathx.arithmetic`)
работает независимо от `__all__`. Даже если имя не перечислено в списке, его всё равно
можно импортировать напрямую — `__all__` лишь намекает на «предназначенное для людей».

```python
# Пусть в __all__ нет имени "variance":
from mathx import variance   # работает! явный импорт игнорирует __all__
from mathx import *          # variance сюда не попадёт
```

ПОЧЕМУ это полезно: `__all__` — это документация через код. Пишешь библиотеку и хочешь
скрыть внутренние детали? Перечисли публичный API в `__all__`. Пользователь смотрит на него
и сразу видит, что предназначено для использования снаружи. Инструменты вроде `pydoc` и IDE
тоже ориентируются на `__all__` при показе автодополнения и справки.

> Подводный камень: `__all__` не делает имена «приватными» в смысле защиты. Python — язык
> «взрослых людей»: всё можно импортировать явно, если очень захотеть. `__all__` — это
> соглашение, а не барьер.

## Идея 6. Относительные импорты: `from .mod import x` и `from .. import y`

Внутри пакета можно импортировать соседние модули двумя способами.

**Абсолютный импорт** — указываешь полный путь от корня `sys.path`:

```python
from mathx.arithmetic import add   # полный путь: пакет.модуль
```

**Относительный импорт** — указываешь путь относительно текущего файла:

```python
# внутри mathx/__init__.py
from .arithmetic import add     # . = этот же пакет (mathx)
from .sequences import fib      # . = mathx, берём sequences

# внутри гипотетического mathx/stats/helpers.py
from ..arithmetic import add    # .. = родительский пакет (mathx)
```

Точка (`.`) означает «текущий пакет», две точки (`..`) — «на уровень выше». Это аналог
`./` и `../` в путях файловой системы.

**Когда использовать что.** Внутри пакета принято использовать относительные импорты —
они работают независимо от того, как пакет называется и куда он установлен. Абсолютные
удобны в «точке входа» или когда хочется явности.

**Почему относительный модуль нельзя запустить как `python mod.py`.**
Вот главная ловушка. Если в `arithmetic.py` написать `from .sequences import fib` и
запустить `python arithmetic.py` напрямую, получишь:

```
ImportError: attempted relative import with no known parent package
```

Причина: при прямом запуске Python не знает, что этот файл — часть пакета `mathx`. Переменная
`__package__` пустая, и «относительно чего» считать точку — непонятно. Файл работает только
как модуль внутри пакета (`python -m mathx.arithmetic` или через импорт из другого кода).

```
Правило: файлы с относительными импортами — не точки входа.
Точка входа всегда использует абсолютные импорты.
```

## Идея 7. Ловушка: циклические импорты (A ↔ B)

Циклический импорт — это когда модуль A импортирует из B, а B импортирует из A.

```python
# a.py
from b import greet   # ← импортирует b

# b.py
from a import name    # ← импортирует a
```

**Что происходит при `import a`:** Python начинает исполнять `a.py`. Встречает
`from b import greet` — начинает исполнять `b.py`. `b.py` встречает `from a import name`.
Но `a.py` ещё не дочитан до конца! Python смотрит в `sys.modules` — там уже есть объект
модуля `a`, но он «полупустой» (функция `name` ещё не определена, потому что до неё не
дошло выполнение). Получаем `ImportError: cannot import name 'name' from partially initialized
module 'a'` — или ещё хуже, молча получаем `None`.

**Минимальный пример, который ломается:**

```python
# a.py
from b import B_VALUE

A_VALUE = 10

# b.py
from a import A_VALUE   # A_VALUE ещё не существует в момент импорта!

B_VALUE = 20
```

**Как лечить.**

1. **Поздний импорт (внутри функции).** Перенести `import` туда, где он нужен:

```python
# b.py — исправлено
B_VALUE = 20

def get_a():
    from a import A_VALUE   # импортируется только когда вызвана функция
    return A_VALUE
```

2. **Переразбиение на третий модуль.** Если A и B используют общее имя — вынести его
в отдельный `c.py`, чтобы и A, и B импортировали из C, но не друг из друга.

```
a.py ──→ c.py ←── b.py      # вместо a.py ←→ b.py
```

ПОЧЕМУ это важно понимать: циклические импорты — одна из самых частых ошибок при росте
проекта. Симптом: `ImportError` или `AttributeError` с «None» вместо функции. Лекарство
«поздний импорт» работает всегда, но лучше сразу проектировать зависимости без циклов.

## Идея 8. Пакет как pip-библиотека: `pyproject.toml`, `pip install -e .` и `sys.modules`

**Как пакет становится устанавливаемым.** Когда пакет готов к распространению, ему нужен
файл `pyproject.toml` в корне проекта:

```toml
# pyproject.toml — минимальная конфигурация
[build-system]
requires = ["hatchling"]
build-backend = "hatchling.build"

[project]
name = "mathx"
version = "0.1.0"
```

После этого команда `pip install -e .` устанавливает пакет в «режиме разработки» (editable):
Python добавляет текущий каталог в `sys.path` через специальную ссылку. Теперь
`import mathx` работает из любого места, а изменения в коде видны сразу без переустановки.
Для продакшн-установки: `pip install .` (копирует файлы) или публикация на PyPI через
`python -m build` + `twine upload`.

**`sys.modules` — кэш импортов.** Когда Python первый раз видит `import mathx`, он:

1. Находит пакет в `sys.path`.
2. Исполняет `mathx/__init__.py` — весь файл, сверху вниз.
3. Кладёт результирующий объект модуля в словарь `sys.modules["mathx"]`.

При следующем `import mathx` — где угодно в программе — Python смотрит в `sys.modules` и
возвращает уже готовый объект. **Код модуля выполняется ровно один раз.**

```python
import sys
import mathx

print(sys.modules["mathx"])        # <module 'mathx' from '.../__init__.py'>
print(sys.modules["mathx.arithmetic"])  # подмодуль тоже кешируется
```

ПОЧЕМУ это важно: если ты меняешь код модуля и делаешь `import` снова — Python вернёт
старый объект из кэша. В интерактивной сессии (`python`, Jupyter) это сбивает с толку:
«я сохранил файл, а изменений нет». Решение: `importlib.reload(module)` или перезапуск
интерпретатора.

> Подводный камень с изменяемым состоянием: если модуль содержит список или счётчик
> на уровне модуля (`count = 0`), он инициализируется один раз и живёт до конца процесса.
> Это и мощь (дешёвое глобальное состояние), и опасность (неожиданные побочные эффекты при
> тестировании).

## Задание (существенное): подмодуль `mathx.statistics` — описательная статистика

Ты уже видел, как `mathx.__init__.py` собирает публичное API из нескольких подмодулей.
Теперь ты *расширяешь пакет* сам: добавляешь третий подмодуль `mathx/statistics.py`
и четыре функции описательной статистики. Идея задания — не выучить формулы (они простые),
а почувствовать, как пакет *проектируется* снаружи внутрь: сначала «какой API я хочу
видеть в тесте», потом — «что лежит в каком файле».

Стаб уже создан в `mathx/statistics.py`, реэкспорт в `__init__.py` — тоже готов.
Твоя задача: реализовать тела четырёх функций.

### Спецификация API

```python
from mathx import mean, median, mode, variance
```

**`mean(data: Sequence[float]) -> float`**
Среднее арифметическое: сумма элементов, делённая на их количество.
- Пустой список → `ValueError`.

**`median(data: Sequence[float]) -> float`**
Медиана отсортированного ряда.
- Нечётное количество — средний элемент.
- Чётное количество — среднее двух центральных.
- Исходный `data` изменять нельзя (работай с отсортированной копией).
- Пустой список → `ValueError`.

**`mode(data: Sequence[float]) -> float`**
Наиболее частый элемент. При нескольких одинаково частых — наименьший.
- Пустой список → `ValueError`.

**`variance(data: Sequence[float], ddof: int = 0) -> float`**
Дисперсия: среднее квадратов отклонений от среднего.
- `ddof=0` — делим на `N` (генеральная).
- `ddof=1` — делим на `N - 1` (выборочная, исправленная по Бесселю).
- Пустой список → `ValueError`.
- `len(data) < 2` при `ddof=1` → `ValueError` (деление на 0).
- `ddof` не 0 и не 1 → `ValueError`.

> Нельзя делать `import statistics` (стандартная библиотека): реализуй алгоритмы сам.
> `math` и `collections` — разрешены.

### Подсказки

<details><summary>Подсказка 1 — с чего начать (mean)</summary>
Проверь <code>len(data) == 0</code> в самом начале и подними <code>ValueError</code>.
Среднее — это <code>sum(data) / len(data)</code>. Один вызов <code>sum()</code> —
никаких циклов.
</details>

<details><summary>Подсказка 2 — median: как работать с копией и найти центр</summary>
<code>sorted(data)</code> возвращает новый список, не меняя исходный.
Дальше:
<pre>
n = len(sorted_data)
mid = n // 2
if n % 2 == 1:
    return float(sorted_data[mid])
else:
    return (sorted_data[mid - 1] + sorted_data[mid]) / 2.0
</pre>
Почему <code>mid - 1</code> и <code>mid</code>? При чётном n два центральных элемента
стоят на позициях <code>n//2 - 1</code> и <code>n//2</code>.
</details>

<details><summary>Подсказка 3 — mode: подсчёт частот без лишнего кода</summary>
<code>collections.Counter(data)</code> даёт словарь <code>{значение: частота}</code>.
Найди максимальную частоту: <code>max_freq = max(counter.values())</code>.
Собери все элементы с такой частотой: <code>[k for k, v in counter.items() if v == max_freq]</code>.
Верни <code>min(...)</code> из них.
</details>

<details><summary>Подсказка 4 — variance: связь с mean и правильный знаменатель</summary>
Сначала посчитай среднее через уже готовую <code>mean(data)</code>.
Затем квадраты отклонений: <code>[(x - m) ** 2 for x in data]</code>.
Поделить на <code>n - ddof</code>, где <code>n = len(data)</code>.
Не забудь: при <code>ddof=1</code> и <code>n == 1</code> получится деление на 0 —
это и есть краевой случай с <code>ValueError</code>.
</details>
