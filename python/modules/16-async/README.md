# Модуль 16 — Асинхронность (`asyncio`)

Представь программу-краулер: она скачивает 100 страниц. Каждое скачивание — это «отправил
запрос и ждёшь ответа сети» секунду-другую. Если делать по очереди — 100 секунд, причём
почти всё это время процессор **простаивает**, тупо ожидая сеть. Хочется, пока ждём один
ответ, отправить остальные 99 запросов. Не «считать в 100 потоков», а **переключаться на
другую работу, пока текущая ждёт ввода-вывода**.

**Большая идея модуля:** `asyncio` даёт *кооперативную многозадачность в одном потоке*.
Одна корутина, дойдя до точки ожидания (`await`), добровольно отдаёт управление обратно
— и событийный цикл (event loop) пускает поработать другую. Никакой параллельности на
ядрах нет: всё крутится в одном потоке, но за счёт того, что «ожидание» отдаётся циклу,
сотни I/O-операций летят как бы одновременно. Это не про скорость вычислений — это про
**не простаивать во время ожидания**.

> Этот модуль про *чистый* `asyncio` из стандартной библиотеки: ни `aiohttp`, ни прочих
> сторонних либ. Все «долгие операции» мы имитируем через `asyncio.sleep` — он честно
> отдаёт управление циклу, как настоящий сетевой вызов, только без сети.

## Идея 1. Корутина: `async def` и `await`

Функция, объявленная через `async def`, — это **корутина-функция**. Главная неожиданность:
её вызов **ничего не выполняет**.

```python
import asyncio

async def greet(name):
    return f"hello, {name}"

c = greet("world")     # тело НЕ выполнилось! c — это coroutine-объект
print(c)               # <coroutine object greet at 0x...>
print(asyncio.run(c))  # вот теперь выполнилось: 'hello, world'
```

`greet("world")` возвращает «рецепт работы», а не результат. Чтобы рецепт ожил, его надо
*запустить* (через `asyncio.run` снаружи или `await` изнутри другой корутины).

Внутри корутины `await` означает «приостанови меня здесь, дождись вот этого awaitable,
и продолжи с результатом, когда он будет готов»:

```python
async def main():
    msg = await greet("world")   # ждём корутину, получаем её результат
    print(msg)
```

**Почему так, а не как обычная функция:** `await` — это *точка, где можно переключиться*.
Пока эта корутина ждёт, поток не блокируется — событийный цикл уводит его на другую
готовую работу. В этом вся суть: `await` = «мне тут нечего делать, займись пока другими».

**Подвох — «забытый `await`».** Если написать `greet("world")` без `await`, ты получишь
coroutine-объект, а не строку, и Python предупредит `RuntimeWarning: coroutine '...' was
never awaited`. Корутина так и не запустилась. Это ошибка №1 у новичков: код «молча
ничего не делает», потому что результат — это неисполненный рецепт.

## Идея 2. Событийный цикл и `asyncio.run`

Кто, собственно, *исполняет* корутины и переключает их? **Событийный цикл** (event loop) —
планировщик в одном потоке. Он держит набор задач, запускает одну до ближайшего `await`,
там она «засыпает», цикл будит следующую готовую, и так по кругу.

Раньше цикл создавали руками; сейчас канонический вход — `asyncio.run`:

```python
async def main():
    ...

asyncio.run(main())   # создаёт цикл, выполняет main() до конца, закрывает цикл
```

`asyncio.run(coro)` — это **мост между синхронным и асинхронным миром**: его зовут *один*
раз из обычного (синхронного) кода как точку входа. Он сам поднимает event loop, гоняет
корутину до завершения и аккуратно всё закрывает.

**Почему именно так:** `await` можно писать только *внутри* `async def`. А `main` нужно
как-то запустить из синхронного `if __name__ == "__main__":`. `asyncio.run` и есть этот
стартер. **Именно поэтому наши тесты — обычные `def`, а внутри `asyncio.run(coro())`:**
тест — синхронный код, который запускает корутину и проверяет её результат.

**Подвох:** нельзя вызвать `asyncio.run` *внутри* уже работающего цикла (например, внутри
другой корутины) — получишь `RuntimeError: asyncio.run() cannot be called from a running
event loop`. Внутри корутины переключаются через `await`, а не через ещё один `run`.

## Идея 3. Конкурентный запуск: `asyncio.gather` (и порядок результатов)

`await coro()` запускает *одну* корутину и ждёт её. Если так делать в цикле — получится
**последовательно**, никакого выигрыша:

```python
async def slow_sequential():
    a = await delayed("a", 0.3)   # ждём 0.3с
    b = await delayed("b", 0.3)   # потом ещё 0.3с
    return [a, b]                 # итого ~0.6с — никакой конкурентности
```

Чтобы запустить корутины **конкурентно**, их отдают в `asyncio.gather`:

```python
async def fast_concurrent():
    a, b = await asyncio.gather(
        delayed("a", 0.3),
        delayed("b", 0.3),
    )
    return [a, b]                 # ~0.3с — обе ждали ОДНОВРЕМЕННО
```

`gather` принимает несколько awaitable'ов, запускает их все «сразу» и ждёт, пока
завершатся **все**. Возвращает список результатов.

**Ключевое свойство — порядок.** Результаты идут в **том порядке, в каком переданы
аргументы**, а *не* в порядке завершения. Даже если второй аргумент досчитался первым:

```python
async def main():
    # второй спит 0с (готов сразу), первый — 0.05с (готов позже)
    out = await asyncio.gather(delayed("first", 0.05), delayed("second", 0.0))
    return out                    # ['first', 'second'] — по позициям, не по времени!
```

**Почему это важно:** ты можешь конкурентно запросить 100 URL и спокойно сопоставить
`results[i]` с `urls[i]` — порядок гарантирован. Это и проверяют тесты: дай корутинам
разные задержки и убедись, что список вышел в исходном порядке.

**Подвох — CPU-bound.** `gather` НЕ ускоряет вычисления. Если корутина — тяжёлый цикл
без единого `await`, она не отдаёт управление, и «конкурентности» не будет: остальные
будут ждать, пока она досчитает (см. Идею 7). `asyncio` помогает там, где есть *ожидание*
(I/O), а не там, где есть *счёт*.

## Идея 4. Задачи: `asyncio.create_task`

`gather` хорош, когда корутины известны заранее и нужны все результаты. Но иногда хочется
**запустить корутину в фоне прямо сейчас** и продолжить, а результат забрать позже. Для
этого корутину «оживляют» в **Task**:

```python
async def main():
    task = asyncio.create_task(delayed("done", 0.1))  # СРАЗУ запланирована в цикле
    # ... здесь можно делать другую работу, пока task крутится в фоне ...
    result = await task                                # заберём результат, когда нужен
    return result
```

Разница тонкая, но важная: `coro()` — это только рецепт, он *не запущен*, пока его не
заавейтят. `asyncio.create_task(coro())` сразу **отдаёт корутину циклу на исполнение** —
она начинает крутиться в фоне немедленно, ещё до `await task`.

**Почему так:** задачи — это способ *запараллелить* (в смысле конкурентности) куски
работы, не дожидаясь каждого по очереди. Кстати, `asyncio.gather` под капотом и
оборачивает переданные корутины в задачи.

**Подвох:** не теряй ссылку на задачу. Если создал `create_task(...)`, нигде не сохранил
и не заавейтил — её могут не доделать (а в новых версиях её ещё и может «съесть» сборщик
мусора). Создал фоновую задачу — где-то её дождись (`await task`, или собери в список и
`await asyncio.gather(*tasks)`).

## Идея 5. Очередь `asyncio.Queue`: producer/consumer

Когда несколько корутин-производителей (**producers**) поставляют данные, а несколько
потребителей (**consumers**) их разбирают, удобна **очередь** — `asyncio.Queue`. Это
безопасная для конкурентного доступа труба: один кладёт `put`, другой берёт `get`.

```python
queue = asyncio.Queue()

async def consumer():
    while True:
        item = await queue.get()   # ждёт, пока что-то появится
        if item is None:           # договорённый «стоп-сигнал»
            queue.task_done()
            break
        process(item)
        queue.task_done()          # отметили: этот элемент обработан
```

Producer кладёт элементы, а в конце — по одному **стоп-сигналу** (например `None`) на
каждого consumer:

```python
async def main(items, n_consumers):
    consumers = [asyncio.create_task(consumer()) for _ in range(n_consumers)]
    for item in items:
        await queue.put(item)
    for _ in range(n_consumers):
        await queue.put(None)       # каждому consumer'у — по сигналу «всё, выходи»
    await queue.join()              # ждём, пока очередь полностью разобрана
    await asyncio.gather(*consumers)
```

**Почему стоп-сигналы:** `await queue.get()` на пустой очереди **ждёт вечно**. Если
producer кончился, а consumer'ы про это не знают — они навсегда зависнут на `get()`. Чтобы
завершить их корректно, producer явно говорит «данных больше не будет» — кладёт по
сентинелу (`None`) на каждого. Сколько consumer'ов — столько сигналов.

**`task_done` / `join`:** на каждый `get()` consumer вызывает `queue.task_done()`, а
`await queue.join()` блокируется, пока число `task_done` не сравняется с числом `put`.
Это аккуратный способ дождаться, что *вся* работа разобрана, не угадывая тайминги.

**Подвох:** забыл стоп-сигналы → `gather(*consumers)` зависнет навсегда (consumer'ы в
вечном `get`). Положил сигналов меньше, чем consumer'ов → часть так и не выйдет. Сигналов
ровно столько, сколько потребителей.

## Идея 6. Таймауты: `asyncio.wait_for`

Что, если операция может зависнуть надолго, а мы не готовы ждать бесконечно?
`asyncio.wait_for(awaitable, timeout)` даёт корутине лимит времени:

```python
try:
    result = await asyncio.wait_for(delayed("late", 1.0), timeout=0.1)
except asyncio.TimeoutError:
    result = "too slow"            # не дождались за 0.1с — корутину отменили
```

Если awaitable не успел за `timeout` секунд, `wait_for`:
1. **отменяет** внутреннюю корутину (бросает в неё `CancelledError`),
2. бросает наружу `asyncio.TimeoutError`.

**Почему это важно:** в реальном I/O «зависшее навсегда» — норма (сеть отвалилась, сервер
молчит). Таймаут превращает «висим вечно» в обрабатываемую ошибку. Это твоя страховка от
бесконечного ожидания.

**Подвох:** отмена работает только в *точках `await`*. Если внутри корутины идёт долгий
синхронный счёт без `await`, отменить её посреди счёта нельзя — `CancelledError` доставят
лишь на ближайшем `await`, которого может и не быть.

## Идея 7. Когда `asyncio` помогает — и чем отличается от потоков

Это, возможно, самый важный раздел. `asyncio` — **не** «сделать всё быстрее».

**`asyncio` выигрывает на I/O-bound:** задача почти всё время **ждёт** ввода-вывода —
сеть, диск, БД. Процессор простаивает в ожидании. Пока одна корутина ждёт ответ, цикл
запускает другую. Сотни ожиданий перекрываются — общее время резко падает.

**`asyncio` НЕ помогает на CPU-bound:** задача *считает* (перемножает матрицы, жмёт
архив). Тут нечего «ждать» — процессор занят. Корутина без `await` не отдаёт управление,
цикл стоит, конкурентности нет. Хуже: один тяжёлый счёт **заблокирует весь цикл**, и все
остальные корутины замрут (см. подвох ниже). Для CPU-нагрузки нужны процессы
(`multiprocessing`) — реальные ядра.

**Чем отличается от потоков (`threading`):**

| | `asyncio` | потоки (`threading`) |
|---|---|---|
| Параллелизм | один поток, кооперативная многозадачность | несколько потоков (но GIL мешает CPU) |
| Переключение | **добровольно**, в точках `await` | **вытесняюще**, в любой момент (планировщик ОС) |
| Гонки данных | почти нет (переключение предсказуемо) | есть (нужны блокировки) |
| Цена «единицы» | дёшево: корутина — это объект | дороже: поток ОС |
| Масштаб I/O | десятки тысяч соединений легко | упираешься в число потоков |

Главное отличие в одной фразе: в `asyncio` переключение происходит **только там, где ты
написал `await`** — ты видишь все точки переключения в коде. В потоках ОС может прервать
тебя *в любой* момент, поэтому нужны замки. asyncio проще рассуждать, но требует, чтобы
**весь** код был неблокирующим.

**Главный подвох — блокирующий вызов в цикле.** Если в корутине вызвать обычную
*блокирующую* функцию — `time.sleep(1)`, синхронный сетевой запрос, тяжёлый счёт — она
**не отдаст управление циклу**, и встанет ВСЁ: ни одна другая корутина не шевельнётся, пока
блокировка не кончится. Правила:
- паузы — только `await asyncio.sleep(...)`, **никогда** `time.sleep(...)`;
- I/O — только асинхронные библиотеки/вызовы;
- неизбежный блокирующий/CPU код — выноси через `await asyncio.to_thread(func, ...)`
  (отдельный поток), чтобы не заморозить цикл.

</details>

## Задание (существенное): `bounded_map(func, items, limit)`

Конкурентность — это прекрасно, но «запустить 10 000 запросов разом» обычно нельзя:
сервер забанит, файловых дескрипторов не хватит, память кончится. Нужен **потолок
конкурентности**: применяй корутину к каждому элементу конкурентно, но чтобы
**одновременно** работало не больше `limit` штук. Это типовой, очень практичный паттерн.

**Задача** — реализовать `bounded_map`: конкурентный `map` с ограничителем через
`asyncio.Semaphore`.

```python
async def double(x):
    await asyncio.sleep(0.01)
    return x * 2

asyncio.run(bounded_map(double, [1, 2, 3, 4, 5], limit=2))
# [2, 4, 6, 8, 10]  — но в каждый момент работало не больше 2 вызовов double
```

### API

```python
async def bounded_map(
    func: Callable[[Any], Awaitable[Any]],
    items: Iterable[Any],
    limit: int,
) -> list: ...
```

**Контракт:**

- `func` — это **корутина-функция**: `func(x)` возвращает awaitable, его надо `await`.
- Результаты — в **исходном порядке** `items` (i-й результат соответствует i-му элементу),
  независимо от того, какой вызов завершился раньше.
- В каждый момент времени **активно не более `limit`** вызовов `func` (между захватом и
  освобождением слота семафора).
- `limit >= 1`. При `limit < 1` → `ValueError`.
- Пустой `items` → `[]` (функция `func` не вызывается ни разу).
- Не полагайся на конкретное *время* — полагайся на семафор: он сам гарантирует потолок.

### Подсказки

<details><summary>Чем ограничивать число одновременных — Semaphore</summary>
<code>asyncio.Semaphore(limit)</code> — это «счётчик слотов». <code>async with sem:</code>
захватывает один слот на входе и освобождает на выходе из блока. Если все
<code>limit</code> слотов заняты, очередной <code>async with sem</code> <b>ждёт</b> (через
внутренний await), пока кто-то не освободит слот. Так число одновременных и держится
&lt;= limit, причём ждущие не блокируют цикл.
</details>

<details><summary>Как сохранить исходный порядок результатов</summary>
Сделай внутреннюю корутину-«воркер», которая под <code>async with sem</code> вызывает
<code>await func(x)</code> и возвращает результат. Затем построй по одному воркеру на
каждый элемент и отдай их в <code>asyncio.gather(*workers)</code>. Помни из теории:
<code>gather</code> возвращает результаты В ПОРЯДКЕ АРГУМЕНТОВ — порядок получишь
бесплатно, специально пересортировывать ничего не надо.
</details>

<details><summary>Скелет решения</summary>

```python
if limit < 1:
    raise ValueError(...)
sem = asyncio.Semaphore(limit)

async def worker(x):
    async with sem:
        return await func(x)

return list(await asyncio.gather(*(worker(x) for x in items)))
```

Разберись, ПОЧЕМУ <code>async with sem</code> стоит вокруг <code>await func(x)</code>, а не
снаружи генератора воркеров: слот должен удерживаться ровно на время РАБОТЫ одного вызова.
</details>

<details><summary>Как тест проверяет «не больше limit одновременно» без таймингов</summary>
Хитрость без зависимости от времени: пусть <code>func</code> на входе увеличивает счётчик
«сейчас активно», обновляет максимум и уменьшает на выходе. После
<code>bounded_map(..., limit=k)</code> зафиксированный максимум обязан быть
<code>&lt;= k</code>. Это проверяет реальный потолок конкурентности, а не «быстрее
последовательного» — никаких <code>assert</code> на конкретные секунды.
</details>
