# Модуль 13 — Аннотации типов и тестирование

Большая идея модуля: Python — язык с **динамической** типизацией (тип проверяется во время
выполнения, а не заранее), но это не значит, что про типы можно молчать. *Аннотации типов* —
это способ **сказать вслух**, чего функция ждёт и что возвращает. Сам интерпретатор их не
проверяет — но их читают люди, IDE и отдельные инструменты вроде `mypy`. А чтобы убедиться, что
код реально делает то, что обещает, есть вторая половина модуля — **тесты**. Junior пишет код,
который не только работает, но и *проверяемо* работает: с аннотациями, объясняющими намерение, и
с тестами, которые ловят регрессии. В этом модуле ты сам напишешь функции под уже готовые тесты —
и заодно разберёшь, как такие тесты устроены.

## Идея 1. Аннотации типов: `int`, `list[str]`, `dict[str, int]`, `X | None`

Аннотация — это приписка к параметру или к возвращаемому значению, говорящая о его типе:

```python
def greet(name: str) -> str:        # принимает str, возвращает str
    return "Привет, " + name

age: int = 36                       # переменную тоже можно аннотировать
```

Для контейнеров тип уточняется в квадратных скобках — что *внутри*:

```python
names: list[str] = ["Ada", "Linus"]        # список строк
scores: dict[str, int] = {"Ada": 10}       # словарь: ключ-строка -> значение-int
pair: tuple[int, int] = (1, 2)             # кортеж из двух int
```

Когда значение может быть «либо то, либо ничего», пишут `X | None` (читается «X или None»):

```python
def find(d: dict[str, int], key: str) -> int | None:
    # вернёт int, если ключ есть, и None, если нет
    ...
```

ПОЧЕМУ это важно: аннотация — это **документация, которую видит инструмент**. Глядя на
`def safe_get(d: dict[str, int], key: str) -> int | None`, и человек, и IDE сразу понимают:
сюда подаём словарь строка→число, а назад можем получить число *или* `None` — значит, результат
надо проверять перед использованием. Без аннотации эту же мысль пришлось бы держать в голове или
писать в комментарии, который легко устаревает. Запись `X | None` — современная (Python 3.10+);
раньше для того же писали `Optional[X]` из модуля `typing`.

> Мелочь, но важная: `list[str]` (нижний регистр, встроенный тип) работает «из коробки» с
> Python 3.9+. В старых учебниках встретишь `List[str]` из `typing` — это то же самое, просто
> устаревший синтаксис. Пиши строчными: `list`, `dict`, `tuple`, `set`.

## Идея 2. Union и Optional: несколько возможных типов

Иногда функция принимает (или возвращает) значение, которое может быть одного из нескольких
типов. Для этого есть **`Union`** — объединение типов.

Современный синтаксис (Python 3.10+) использует вертикальную черту:

```python
def stringify(x: int | str) -> str:
    return str(x)   # принимает int ИЛИ str
```

До Python 3.10 то же писали через `Union` из модуля `typing`:

```python
from typing import Union

def stringify(x: Union[int, str]) -> str:
    return str(x)
```

Обе записи означают одно и то же; современный `int | str` читается удобнее.

**`Optional[X]` — это синоним `X | None`:**

```python
from typing import Optional

def find_user(uid: int) -> Optional[str]:   # то же, что str | None
    if uid == 0:
        return "admin"
    return None
```

`Optional[X]` часто встречается в чужом коде и стандартной библиотеке — важно уметь его читать.
В новом коде предпочитай `X | None` (короче, без импорта).

**Сужение типа через `isinstance`** — когда в теле функции хочешь работать с конкретным типом
из Union, используй проверку:

```python
def process(value: int | str) -> str:
    if isinstance(value, int):
        # здесь mypy знает: value: int
        return f"число {value * 2}"
    # здесь mypy знает: value: str
    return value.upper()
```

Такое «сужение» (type narrowing) — основной способ безопасно работать с объединёнными типами.

ПОЧЕМУ это важно: реальный код полон ситуаций «либо есть значение, либо нет» (результат поиска,
ответ API, необязательный аргумент). `Optional[X]` / `X | None` честно говорит об этом в
сигнатуре, и mypy проверит, что ты не забыл обработать `None` перед использованием значения.

## Идея 3. `mypy` — статическая проверка типов

Аннотации сами по себе пассивны: Python их не проверяет. Чтобы они *работали*, нужен внешний
инструмент — **`mypy`**. Это статический анализатор: он читает твой код (не запуская его) и
сверяет, согласуются ли типы.

```python
def double(x: int) -> int:
    return x * 2

double("привет")      # Python это ЗАПУСТИТ (и вернёт "приветпривет")
```

`mypy` на такой код заругается ещё до запуска:

```text
$ mypy myfile.py
error: Argument 1 to "double" has incompatible type "str"; expected "int"
```

ПОЧЕМУ это важно: `mypy` ловит целый класс ошибок **до того**, как они дойдут до прода, — там,
где компилируемые языки (вроде C++) ловят их компилятором. В Python это опциональный, но мощный
слой защиты: чем точнее аннотации, тем больше ошибок `mypy` поймает за тебя. В этом модуле
запускать `mypy` мы не требуем (тесты его не зовут) — но знай команду и идею: `mypy файл.py`
читает аннотации и говорит, где типы не сходятся.

> `mypy` не входит в стандартную библиотеку — это отдельный пакет (`pip install mypy`). Он
> *ничего не меняет* в поведении программы: только смотрит и сообщает. Запускать его или нет —
> решаешь ты; Python и без него выполнит код.

## Идея 4. Как устроен хороший тест: arrange-act-assert

Тест — это маленькая функция, которая проверяет один факт о коде. У хорошего теста видна
структура из трёх шагов, **arrange-act-assert** (подготовь — выполни — проверь):

```python
def test_clamp_above_range():
    # arrange — готовим входные данные
    x, lo, hi = 10, 0, 5
    # act — вызываем то, что проверяем
    result = clamp(x, lo, hi)
    # assert — сверяем результат с ожиданием
    assert result == 5
```

`assert` — встроенный оператор: если выражение истинно, тест молча проходит; если ложно — падает.
`pytest` находит все функции, начинающиеся с `test_`, запускает их и красиво показывает, какие
`assert` не сошлись.

Когда функция *должна* бросать исключение, проверяют это через `pytest.raises` — менеджер
контекста, который ждёт ошибку и падает, если её не случилось:

```python
import pytest

def test_clamp_rejects_bad_range():
    with pytest.raises(ValueError):
        clamp(1, hi=0, lo=5)        # lo > hi — функция обязана возмутиться
```

ПОЧЕМУ это важно: `arrange-act-assert` держит тест читаемым — сразу видно, что подали, что
вызвали и чего ждут. А `pytest.raises` — единственный честный способ проверить «эта ситуация
должна привести к ошибке»: если просто вызвать `clamp(...)` и оно бросит исключение, упадёт сам
тест; `with pytest.raises(...)` превращает ожидаемую ошибку в **успех**.

## Идея 5. Параметризация: один тест — много случаев

Часто хочется проверить функцию на десятке пар «вход → ожидание». Копировать тело теста десять
раз — скучно и легко ошибиться. `@pytest.mark.parametrize` прогоняет один тест по списку
наборов аргументов:

```python
@pytest.mark.parametrize("x, lo, hi, expected", [
    (3, 0, 5, 3),     # внутри диапазона — без изменений
    (-1, 0, 5, 0),    # ниже нижней границы — прижали к lo
    (10, 0, 5, 5),    # выше верхней границы — прижали к hi
    (0, 0, 5, 0),     # ровно на границе
])
def test_clamp(x, lo, hi, expected):
    assert clamp(x, lo, hi) == expected
```

`pytest` запустит это как **четыре отдельных теста**: если упадёт случай `(10, 0, 5, 5)`, ты
увидишь именно его, а не «тест clamp сломался где-то».

ПОЧЕМУ это важно: и снова про аннотации — **они НЕ проверяются в рантайме**. Это стоит
прочувствовать на примере. Сигнатура `clamp(x: int, ...) -> int` — лишь подсказка; Python не
мешает позвать `clamp(3.5, 0, 5)` с `float`. Поэтому правильность гарантируют не аннотации, а
тесты: аннотация *объясняет намерение*, тест *проверяет поведение*. Это две разные опоры, и
junior опирается на обе.

> Запомни границу ответственности: аннотация говорит «я ожидаю `int`», `mypy` проверяет это
> *до запуска*, а во время выполнения никто не следит — передашь не тот тип, Python спокойно
> выполнит код (и, может, сломается позже и непонятно где). Поэтому тесты и аннотации дополняют
> друг друга, а не заменяют.

## Идея 6. `Callable` — аннотация функций как аргументов

В Python функции — объекты первого класса: их можно передавать в другие функции как аргументы
(callbacks, ключи сортировки, декораторы). Тип такого аргумента описывается через `Callable`:

```python
from typing import Callable

# Callable[[тип_арг1, тип_арг2, ...], тип_результата]
def apply_twice(f: Callable[[int], int], x: int) -> int:
    return f(f(x))
```

Запись `Callable[[int], int]` читается: «функция, принимающая один `int` и возвращающая `int`».

Реальные примеры аннотирования callbacks:

```python
# ключ сортировки: принимает элемент, возвращает comparable
def sort_by(items: list[str], key: Callable[[str], int]) -> list[str]:
    return sorted(items, key=key)

# обработчик событий без аргументов и без возвращаемого значения
def on_click(handler: Callable[[], None]) -> None:
    handler()
```

ПОЧЕМУ это важно: «функция высшего порядка» (higher-order function) — базовый паттерн Python.
Аннотация `Callable` сразу говорит, какой именно вид функции ожидается — с каким числом и типами
аргументов, с каким результатом. IDE может подсвечнуть, что ты передал функцию не той «формы».

> Подвох: `Callable` не различает позиционные и именованные аргументы — `Callable[[int], str]`
> это просто «функция с одним int-аргументом и str-результатом», независимо от имени параметра.
> Для более точных описаний в Python 3.10+ есть `ParamSpec`, но для большинства задач `Callable`
> с конкретными типами аргументов достаточно.

## Идея 7. `Protocol` — структурная (duck-typing) типизация

Python традиционно использует «утиную типизацию»: если объект умеет всё, что нужно — он подходит,
неважно, от какого класса унаследован. `Protocol` (из `typing`) позволяет **описать это
формально** так, чтобы `mypy` мог это проверить.

```python
from typing import Protocol

class Drawable(Protocol):
    def draw(self) -> None: ...   # любой объект с методом draw() подходит

def render(item: Drawable) -> None:
    item.draw()
```

Теперь `mypy` проверит, что всё, что передаётся в `render`, имеет метод `draw()` нужной
сигнатуры — **не через наследование**, а через наличие нужных атрибутов. Это называется
*структурной типизацией* (structural subtyping), в отличие от *номинальной* (nominal) — «должен
быть подклассом `Drawable`».

ПОЧЕМУ это важно: протоколы — мост между динамической утиной типизацией Python и строгой
статической проверкой. Они позволяют писать «принимаю что угодно с методом `__len__`» и при этом
получать поддержку инструментов. Подробнее протоколы разобраны в **модуле 07** (ООП, абстрактные
базовые классы): `Protocol` и `ABC` решают похожую задачу, но разными средствами — `ABC`
требует явного наследования, `Protocol` нет.

> В стандартной библиотеке протоколы уже есть: `Iterable`, `Sized`, `Hashable` и другие из
> `collections.abc`. Когда аннотируешь «принимаю любой итерируемый объект», пиши
> `Iterable[int]`, а не `list[int]` — это честнее и гибче.

## Идея 8. pytest-фикстуры: подготовка данных без копипасты

Когда нескольким тестам нужен один и тот же «готовый объект» (заполненный стек, подключённая БД,
набор функций), не стоит его создавать в каждом тесте вручную. `@pytest.fixture` — способ
**вынести подготовку** в одно место и переиспользовать:

```python
import pytest
from typingtest import Stack

@pytest.fixture
def filled_stack() -> Stack[int]:
    """Стек с тремя элементами [1, 2, 3], вершина — 3."""
    s: Stack[int] = Stack()
    for x in [1, 2, 3]:
        s.push(x)
    return s

def test_peek_on_filled(filled_stack: Stack[int]) -> None:
    assert filled_stack.peek() == 3   # фикстура уже готова

def test_len_on_filled(filled_stack: Stack[int]) -> None:
    assert len(filled_stack) == 3
```

`pytest` автоматически находит фикстуры по имени аргумента функции-теста. Перед каждым тестом
фикстура вызывается заново — стек в `test_peek_on_filled` и `test_len_on_filled` — разные объекты
(если только не указать `scope="module"` или `scope="session"`).

ПОЧЕМУ это важно: фикстуры уменьшают дублирование (DRY) и делают тест читаемым — видно только
то, что проверяется, без ритуала «создай-заполни-подготовь». А когда логика подготовки меняется —
меняешь только фикстуру, не трогая каждый тест отдельно.

> Фикстуры поддерживают `yield` для «teardown» — очистки после теста:
> ```python
> @pytest.fixture
> def temp_file(tmp_path):
>     f = tmp_path / "data.txt"
>     f.write_text("hello")
>     yield f                  # тест получает файл
>     f.unlink(missing_ok=True)  # очистка после теста
> ```
> В этом модуле teardown не нужен, но знать паттерн полезно.

## Задание (существенное): Stack[T] — типизированный дженерик-контейнер

Три задачи выше — функции с аннотациями. Теперь шаг крупнее: реализуй **целый класс**, у
которого типизация устроена иначе — через `TypeVar` и `Generic`. Именно это позволяет
`Stack[int]` и `Stack[str]` быть разными специализациями одного класса, и IDE понимает, что
`pop()` вернёт `int` в первом случае и `str` во втором.

Стек — структура данных LIFO (last-in, first-out): последний вошедший элемент выходит первым.
Как стопка тарелок: кладёшь сверху, берёшь тоже сверху.

### API класса `Stack(Generic[T])`

```python
from typing import Generic, TypeVar, Iterator
T = TypeVar("T")

class Stack(Generic[T]):
    def push(self, item: T) -> None:   ...  # положить на вершину
    def pop(self) -> T:                ...  # снять и вернуть вершину; IndexError если пуст
    def peek(self) -> T:               ...  # посмотреть на вершину без удаления; IndexError если пуст
    def __len__(self) -> int:          ...  # количество элементов
    def is_empty(self) -> bool:        ...  # True ↔ len == 0
    def __iter__(self) -> Iterator[T]: ...  # обход от вершины к основанию, стек не меняется
    def __repr__(self) -> str:         ...  # читаемое представление
```

**Контракт и краевые случаи:**

- `pop()` и `peek()` на пустом стеке обязаны бросать `IndexError`.
- `push(x)` увеличивает `len` на 1, и после него `peek() == x`.
- `pop()` уменьшает `len` на 1 и возвращает элемент, добавленный последним.
- `__iter__` не меняет содержимое: `len` до и после одинаков; элементы идут от вершины к
  основанию.
- Два разных стека с одинаковым содержимым независимы: `push` в один не трогает другой.

### Подсказки

<details><summary>Подсказка 1 — с чего начать: хранилище</summary>
Стеку нужен внутренний список. Добавь в <code>__init__</code> поле
<code>self._data: list[T] = []</code>. Конструктор не нужно объявлять отдельно, если нет
логики кроме инициализации хранилища — но объявить явно чище. Все методы работают с
<code>self._data</code>.
</details>

<details><summary>Подсказка 2 — push и pop</summary>
<code>push</code> — это <code>list.append</code>. <code>pop</code> — это <code>list.pop()</code>,
но сначала проверь, что список непуст: <code>if not self._data: raise IndexError(...)</code>.
<code>peek</code> похож на <code>pop</code>, но не удаляет: возвращает <code>self._data[-1]</code>.
</details>

<details><summary>Подсказка 3 — __len__ и is_empty</summary>
<code>__len__</code> возвращает <code>len(self._data)</code>. <code>is_empty</code> может просто
<code>return len(self) == 0</code> — делегировать к уже написанному <code>__len__</code>, а не
дублировать логику.
</details>

<details><summary>Подсказка 4 — __iter__ без изменений</summary>
Нужно обойти от вершины к основанию, не трогая сам список. Самый простой способ:
<code>return iter(reversed(self._data))</code>. <code>reversed</code> создаёт итератор в обратном
порядке, не меняя список. Альтернатива — <code>yield</code>-генератор:
<code>for item in reversed(self._data): yield item</code>.
</details>

4. `apply_all(funcs: list[Callable[[int], int]], x: int) -> list[int]` — применить каждую
   функцию из `funcs` к числу `x` и вернуть список результатов в том же порядке. Если `funcs`
   пустой — вернуть пустой список. Аннотация `Callable[[int], int]` говорит: каждый элемент
   списка — функция, принимающая `int` и возвращающая `int`.

   Пример:
   ```python
   apply_all([lambda x: x + 1, lambda x: x * 2, abs], -3)
   # → [-2, -6, 3]
   ```

<details><summary>Задание 4 — apply_all (подсказка)</summary>
Функция — обход списка <code>funcs</code> с вызовом каждой на <code>x</code>. Классический
способ: <code>return [f(x) for f in funcs]</code>. Это list comprehension; альтернатива —
<code>list(map(lambda f: f(x), funcs))</code>, но comprehension читается чище. Обрати внимание:
результаты собираются в <em>новый</em> список — <code>funcs</code> не меняется.
</details>
