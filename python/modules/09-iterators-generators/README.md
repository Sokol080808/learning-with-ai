# Модуль 09 — Итераторы и генераторы

В прошлых модулях ты много раз писал `for x in something`. Но что вообще делает этот
`for`? Откуда `for` знает, как достать следующий элемент? Здесь мы заглянем под капот
цикла и научимся **сами производить значения** — по одному, по требованию.

**Большая идея модуля:** в Python перебор — это *протокол*, а не магия. Любой объект,
умеющий выдавать «следующий элемент», можно перебирать в `for`. А генераторы — это
способ описать такую последовательность простой функцией с `yield`, не выдавая все
значения сразу, а лениво — когда их попросят.

## Идея 1. iterable vs iterator: `__iter__`, `__next__`, `StopIteration`

Здесь два разных понятия, которые легко спутать:

- **iterable** (итерируемое) — то, *по чему можно пройтись*: список, строка, словарь,
  `range`. У него есть метод `__iter__`, возвращающий итератор.
- **iterator** (итератор) — то, что *помнит, где мы остановились*, и умеет выдавать
  следующий элемент методом `__next__`.

`for` под капотом делает примерно так:

```python
nums = [10, 20, 30]
it = iter(nums)        # вызывает nums.__iter__() — получили итератор
print(next(it))        # 10  — вызывает it.__next__()
print(next(it))        # 20
print(next(it))        # 30
print(next(it))        # StopIteration! элементы кончились
```

Когда элементы заканчиваются, `__next__` бросает исключение `StopIteration`. Цикл `for`
ловит его молча и просто завершается — поэтому ты этого никогда не видишь.

**ПОЧЕМУ так:** список и итератор разделены не случайно. Список можно перебирать
много раз — каждый `for` берёт **новый** итератор с начала. А сам итератор одноразовый:
дошёл до конца — всё, он «выгорел». Сравни:

```python
nums = [1, 2, 3]
list(nums)   # [1, 2, 3]
list(nums)   # [1, 2, 3]  — снова, список переиспользуем

it = iter(nums)
list(it)     # [1, 2, 3]
list(it)     # []  — итератор уже пуст, второй раз ничего не даст
```

## Идея 2. Генераторы: `yield` и ленивость

Писать класс с `__iter__`/`__next__` руками — многословно. Python даёт короткий путь:
**функция-генератор**. Достаточно поставить `yield` вместо `return` — и функция
превращается в фабрику итераторов.

```python
def first_three():
    yield 1
    yield 2
    yield 3

g = first_three()      # тело функции ЕЩЁ НЕ выполнялось!
print(next(g))         # 1  — выполнилось до первого yield и замерло
print(next(g))         # 2  — продолжило с того места
print(next(g))         # 3
# следующий next(g) -> StopIteration
```

Ключевое отличие от обычной функции: `yield` **не завершает** функцию, а
*приостанавливает* её, запоминая всё состояние (значения переменных, место в коде).
Следующий `next()` продолжает ровно оттуда.

**ПОЧЕМУ это ленивость:** значения не вычисляются заранее и не лежат в памяти все
сразу. Каждое следующее рождается только в момент, когда его попросили через `next()`
(или очередную итерацию `for`). Генератор сам по себе — это **iterator**: у него уже
есть и `__iter__`, и `__next__`, бесплатно.

```python
for x in first_three():   # for прекрасно работает с генератором
    print(x)              # 1, 2, 3
```

## Идея 3. Генераторные выражения `(x*x for x in …)`

Ты уже знаешь list comprehension: `[x*x for x in nums]` — он **сразу** строит список.
Если поменять квадратные скобки на круглые — получится **генераторное выражение**: то
же самое, но ленивое.

```python
squares_list = [x*x for x in range(5)]   # список целиком: [0, 1, 4, 9, 16]
squares_gen  = (x*x for x in range(5))   # генератор: пока ничего не вычислено

print(squares_gen)            # <generator object ...> — не список!
print(next(squares_gen))      # 0
print(sum(x*x for x in range(5)))  # 30 — скобки можно опустить внутри вызова
```

**ПОЧЕМУ это удобно:** когда результат нужно тут же «прожевать» (передать в `sum`,
`max`, `any`, `for`), список целиком собирать незачем — это лишняя память. Генераторное
выражение отдаёт элементы по одному. Правило большого пальца: нужен **готовый список**
(будешь индексировать, перебирать дважды) — бери `[...]`; нужно **один раз пройтись** —
бери `(...)`.

## Идея 4. Зачем ленивость: память и бесконечные последовательности

Два больших выигрыша:

**1) Память.** Чтобы просуммировать квадраты миллиона чисел, список `[x*x for x in
range(1_000_000)]` создаст миллион чисел в памяти. Генератор `(x*x for x in
range(1_000_000))` держит в каждый момент **одно** число. Результат тот же — память
несравнима.

**2) Бесконечность.** Список обязан быть конечным — его надо «достроить до конца».
Генератор может производить значения вечно, а потребитель сам решит, сколько взять:

```python
def naturals():
    n = 1
    while True:        # бесконечный цикл — но это НЕ зависание
        yield n
        n += 1

g = naturals()
print(next(g), next(g), next(g))   # 1 2 3 — взяли сколько надо и остановились
```

`naturals()` не зависнет: тело крутится не само по себе, а *шагает* по одному `yield`
на каждый `next()`. Никто не просит — никто не считает. Именно поэтому в задании ниже
функция `take` так полезна: она берёт первые `k` значений даже из **бесконечного**
источника.

### Подсказки

<details><summary>Задание 1 — countdown</summary>
Это генератор, значит внутри будет <code>yield</code>, а не <code>return</code> со
списком. Пройдись по нужным числам и на каждом сделай <code>yield</code>. Удобно:
<code>range(n, 0, -1)</code> идёт от <code>n</code> до <code>1</code> включительно
(шаг −1). Если <code>n &lt;= 0</code>, такой range пуст — генератор просто ничего не
выдаст, отдельную проверку писать не нужно.
</details>

<details><summary>Задание 2 — take, почему не срез <code>it[:k]</code></summary>
Срез <code>it[:k]</code> работает только для списков/строк, но <b>не</b> для генератора
(у него нет индексов) и тем более не для бесконечного источника. Нужно перебрать
<code>it</code> в цикле <code>for</code> и набирать элементы в список, пока их не станет
<code>k</code> — тогда прервись (<code>break</code>). Так ты не упрёшься в бесконечность.
</details>

<details><summary>Задание 2 — крайний случай k &lt;= 0</summary>
Если <code>k</code> не положительное, брать нечего — результат пустой список. Проверь,
что твой цикл и условие выхода это уже покрывают (например, цикл вообще не должен
ничего добавить).
</details>

<details><summary>Задание 3 — running_total</summary>
Заведи переменную-накопитель (например <code>total = 0</code>). Иди по <code>xs</code>,
на каждом шаге прибавляй элемент к накопителю и <code>yield</code> текущее значение
накопителя. Первое выданное значение для <code>[1,2,3]</code> — это <code>1</code>,
потом <code>3</code>, потом <code>6</code>.
</details>

<details><summary>Задание 4 — chunks</summary>
Шагай по индексам не по одному, а сразу на <code>size</code>:
<code>range(0, len(xs), size)</code> даёт начала кусков <code>0, size, 2*size, …</code>.
На каждом таком <code>i</code> сделай <code>yield xs[i:i+size]</code> — срез сам аккуратно
обрежется в конце, если элементов осталось меньше <code>size</code>.
</details>
