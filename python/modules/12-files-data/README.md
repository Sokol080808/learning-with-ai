# Модуль 12 — Файлы и данные (текст, JSON, CSV)

Большая идея модуля: программа становится полезной, когда умеет **сохранять** результат
и **читать** чужие данные. До сих пор всё жило в памяти и исчезало с концом процесса.
Теперь мы выходим наружу — к файлам на диске и к двум форматам, на которых держится половина
реального мира данных: **JSON** (обмен между программами) и **CSV** (таблицы, экспорт из
Excel/баз). Junior отличается тем, что не склеивает файлы руками, а знает: открыл — обязательно
закрой, путь — это не строка, а `Path`, а готовый парсер почти всегда лучше самописного.

## Идея 1. `open` + `with`; режимы чтения/записи; почему `with`

Файл — это внешний ресурс операционной системы. Его надо **открыть**, поработать и
**закрыть**. Открывает `open(path, mode)`:

```python
f = open("notes.txt", "w", encoding="utf-8")   # "w" — запись (перезаписывает файл с нуля)
f.write("привет\n")
f.close()                                       # ОБЯЗАТЕЛЬНО закрыть
```

Режимы, которые нужны нам сейчас:

- `"r"` — чтение (по умолчанию). Файла нет → `FileNotFoundError`.
- `"w"` — запись: создаёт новый файл или **обнуляет** существующий.
- `"a"` — дозапись в конец (append), старое содержимое сохраняется.

Проблема `close()` в том, что его легко забыть — а если между `open` и `close` вылетит
исключение, файл вообще не закроется (утечка дескриптора, потерянные данные). Поэтому в Python
файлы открывают через `with`:

```python
with open("notes.txt", "w", encoding="utf-8") as f:
    f.write("привет\n")
# здесь файл уже гарантированно закрыт — даже если внутри было исключение
```

ПОЧЕМУ это важно: `with` — это **контекстный менеджер** (ты видел их в модуле 08). На выходе
из блока он сам вызовет `f.close()`, что бы ни случилось внутри. Это не «удобная мелочь», а
правильный способ работать с ресурсами: закрытие привязано к области видимости, а не к твоей
памяти. Запомни правило: **открываешь файл — открывай через `with`.**

> Про `encoding="utf-8"`: всегда указывай его явно. По умолчанию кодировка зависит от системы,
> и один и тот же код может читать кириллицу по-разному на разных машинах. `utf-8` —
> универсальный выбор, на нём говорит весь современный текст.

Читать целиком умеют `f.read()` (вся строка) и `f.readlines()` (список строк **с `\n` на
конце**), а можно просто итерировать файл по строкам в цикле `for line in f:`.

## Идея 2. JSON-файлы: `dump` / `load`

В модуле 11 ты переводил объект ↔ строку через `json.dumps`/`json.loads`. У `json` есть и
«файловые» близнецы — **без буквы `s`**: они пишут и читают сразу в файловый объект.

```python
import json

data = {"name": "Ada", "langs": ["Python", "C++"], "level": 7}

with open("data.json", "w", encoding="utf-8") as f:
    json.dump(data, f)              # объект -> ФАЙЛ (dump = сбросить наружу)

with open("data.json", "r", encoding="utf-8") as f:
    back = json.load(f)            # ФАЙЛ -> объект (load = загрузить внутрь)

back == data                       # True
```

ПОЧЕМУ это важно: запомни мнемонику с прошлого модуля — **`s` на конце = string**.
`dumps`/`loads` работают со **строкой**, а `dump`/`load` (без `s`) — с **файлом**. Это та же
сериализация, просто целевой «приёмник» другой. Внутри по-прежнему действуют ограничения JSON:
только `dict`/`list`/`str`/число/`bool`/`None`, ключи объекта — всегда строки, кортеж вернётся
списком.

> Для читаемого глазами файла удобно `json.dump(data, f, indent=2, ensure_ascii=False)`:
> отступы делают структуру наглядной, а `ensure_ascii=False` сохраняет кириллицу как есть,
> а не как `\uXXXX`. На равенство объектов при обратном чтении это не влияет.

## Идея 3. CSV: разбор строк с заголовком

CSV (*comma-separated values*) — простейший табличный формат: строки таблицы разделены
переводами строк, ячейки внутри строки — запятыми. **Первая строка** обычно содержит
**заголовок** — имена колонок:

```
name,age,city
Ada,36,London
Bjarne,70,Aarhus
```

Идея разбора: взять заголовок, а каждую следующую строку «сшить» с ним в словарь
`имя_колонки → значение`. Так таблица превращается в список словарей — структуру, с которой
в Python работать удобнее всего:

```python
[
    {"name": "Ada",    "age": "36", "city": "London"},
    {"name": "Bjarne", "age": "70", "city": "Aarhus"},
]
```

ПОЧЕМУ это важно: словарь по имени колонки читается куда понятнее, чем `row[1]` — не надо
помнить, что во второй колонке возраст. Обрати внимание: значения остаются **строками**
(`"36"`, не `36`) — CSV не знает типов, всё в нём текст; приведение к числу — отдельный шаг
на твоей совести.

В этом модуле мы пишем **учебный** парсер вручную (`split` по запятой), чтобы понять механику.

> Подвох настоящего CSV: значение само может содержать запятую — тогда его берут в кавычки
> (`"London, UK"`), и наивный `split(",")` ломается. Для таких данных в стандартной библиотеке
> есть модуль `csv` (`csv.DictReader`) — он и кавычки, и экранирование разбирает правильно.
> В заданиях у нас данные **без кавычек и без запятых внутри ячеек**, поэтому простого `split`
> достаточно — но знай, где граница и куда тянуться в реальном коде.

## Идея 4. Стандартный модуль `csv`: `DictReader`, `DictWriter` и критичный подвох `newline=""`

Ручной `split(",")` ломается, как только в ячейке встречается запятая — тогда значение берётся
в кавычки: `"London, UK"`. Поэтому **не пишут CSV-парсер руками** — для этого есть стандартный
модуль `csv`, который правильно обрабатывает кавычки, экранирование и разные диалекты.

```python
import csv

# Чтение: csv.DictReader делает ровно то, что наш parse_csv, но надёжно
with open("people.csv", newline="", encoding="utf-8") as f:
    reader = csv.DictReader(f)    # первая строка — заголовок автоматически
    rows = list(reader)           # список dict, значения — строки

# Запись: csv.DictWriter пишет строки-словари в файл
fieldnames = ["name", "city"]
with open("out.csv", "w", newline="", encoding="utf-8") as f:
    writer = csv.DictWriter(f, fieldnames=fieldnames)
    writer.writeheader()                              # записать строку заголовка
    writer.writerows([{"name": "Ada", "city": "London, UK"}])
    # результат: name,city\r\nAda,"London, UK"\r\n  ← кавычки добавлены автоматически
```

Почему `newline=""` КРИТИЧЕН при открытии: модуль `csv` сам управляет переводами строк и ожидает
**нетронутый** поток байт. Если Python откроет файл в обычном текстовом режиме (без `newline=""`),
на **Windows** он заменит `\r\n` на `\n` при чтении *и* добавит лишний `\r` при записи — в итоге
в файл попадёт `\r\r\n` вместо `\r\n`, что другие программы не поймут. `newline=""` отключает эту
автоматическую нормализацию и отдаёт управление модулю `csv`. **Правило: открывай CSV-файл всегда
с `newline=""`.**

> Подвох: `csv.DictWriter` по умолчанию завершает строки `\r\n` (диалект `excel`). Это стандарт
> RFC 4180 для CSV. Поэтому `read_text()` на записанном файле будет содержать `\r\n`, а не `\n` —
> это нормально. Для round-trip теста читай обратно через `csv.DictReader`, а не сравнивай сырой текст.

Чем `DictReader` лучше ручного разбора:
- **Кавычки**: `"London, UK"` → одна ячейка `London, UK` (без кавычек в значении)
- **Экранирование**: `"He said ""hi"""` → `He said "hi"`
- **Диалекты**: `csv.reader(..., delimiter=';')` — для точки с запятой вместо запятой
- **Пустые поля**: `,, ` правильно превращается в три значения `["", "", " "]`

## Идея 5. Режим `"x"` и обработка `FileNotFoundError`

Помимо `r / w / a`, у `open()` есть режим `"x"` — **эксклюзивное создание**:

```python
try:
    with open("config.json", "x", encoding="utf-8") as f:
        json.dump({"version": 1}, f)
except FileExistsError:
    print("файл уже есть, не перезаписываю")
```

Режим `"x"` создаёт файл, если его **нет**, а если он **уже существует** — поднимает
`FileExistsError`. Это полезно, когда нужно гарантировать, что ты не затрёшь чужие данные
случайно: «создай новый или откажись».

Таблица режимов целиком:

| Режим | Файл существует | Файла нет |
|-------|----------------|-----------|
| `"r"` | читает | `FileNotFoundError` |
| `"w"` | **перезаписывает** | создаёт |
| `"a"` | дозаписывает в конец | создаёт |
| `"x"` | `FileExistsError` | создаёт |

Обработка `FileNotFoundError` при чтении — стандартный паттерн:

```python
try:
    with open("settings.json", "r", encoding="utf-8") as f:
        cfg = json.load(f)
except FileNotFoundError:
    cfg = {}   # файла нет — используем дефолт
```

ПОЧЕМУ это важно: программы, которые читают конфигурацию или кэш, должны корректно стартовать
и при отсутствии файла. Молчаливый `except FileNotFoundError` + дефолт — правильный паттерн;
«поймать всё» (`except Exception`) — антипаттерн, прячущий настоящие ошибки.

## Идея 6. Ограничения JSON (не-строковые ключи и кортежи)

JSON — не Python. При сериализации объектов через `json.dump` / `json.dumps` действуют правила,
которые ломают наивные ожидания:

**Ключи словаря — только строки.** В Python ключом может быть любой хэшируемый объект:

```python
import json
d = {1: "один", (2, 3): "пара"}
json.dumps(d)  # → '{"1": "один"}'  ← кортеж-ключ вообще выброшен (TypeError в строгом режиме)
# точнее: TypeError: keys must be strings
```

При обратном чтении из JSON ключ `"1"` (строка) ≠ ключу `1` (число):

```python
data = {1: "один"}
back = json.loads(json.dumps(data))
back[1]     # KeyError! ключа-числа нет
back["1"]   # "один"  ← ключ стал строкой
```

**Кортеж превращается в список.** JSON не знает `tuple`, поэтому `(1, 2, 3)` сериализуется
как `[1, 2, 3]`, и после `load` это уже `list`:

```python
obj = {"pair": (1, 2)}
back = json.loads(json.dumps(obj))
back["pair"]        # [1, 2]  ← list, не tuple
type(back["pair"])  # <class 'list'>
```

ПОЧЕМУ это важно: если сохраняешь словарь с числовыми ключами или tuple-значениями в JSON и
потом читаешь обратно — тихо получаешь другой объект. Это не ошибка JSON, а ограничение формата.
Для Python-специфических структур используй `pickle` (но только в trusted-окружении) или
`json` с пользовательским `JSONEncoder`/`object_hook`. В курсе мы придерживаемся чистого JSON.

## Идея 4 (продолжение). `pathlib.Path` для путей

Путь к файлу — это не просто строка. Склейка строками (`folder + "/" + name`) хрупкая:
забыл слэш — сломалось, на Windows разделитель вообще `\`. `pathlib.Path` даёт объект, который
знает про файловую систему и собирается оператором `/`:

```python
from pathlib import Path

p = Path("data") / "logs" / "app.txt"   # путь собран по-человечески, без ручных слэшей
p.name        # 'app.txt'   — имя файла
p.suffix      # '.txt'      — расширение
p.parent      # Path('data/logs')
p.exists()    # есть ли такой файл/папка
```

А ещё `Path` умеет читать и писать текст **одной строкой**, сам открывая и закрывая файл:

```python
p.write_text("привет\n", encoding="utf-8")   # создаст/перезапишет файл
text = p.read_text(encoding="utf-8")          # прочитает целиком
```

ПОЧЕМУ это важно: `Path` принимают почти все функции, ожидающие путь — в том числе `open()`.
Поэтому в заданиях параметр `path` может прийти и строкой, и `Path`; твой код не должен этого
замечать — `open(path, ...)` и `Path(path)` работают с обоими. В тестах путь приходит от
фикстуры `tmp_path` — это `Path` на временную папку, которую pytest сам создаёт и потом
удаляет. Так тесты не мусорят в репозитории и не зависят друг от друга.

## Задание 6 — `read_csv_typed` и `write_csv` через стандартный модуль `csv`

### `write_csv(path, rows: list[dict], fieldnames: list[str]) -> None`

Записать список словарей в CSV-файл с заголовком.

- Потом пишет строки данных через `writer.writerows(rows)`.

Прочитать CSV-файл и вернуть список словарей с приведёнными типами.

Потом — строки: <code>writer.writerows(rows)</code>.
Не забудь <code>import csv</code> в начале файла. Главное: <code>open(..., newline="")</code>
без этого на Windows в файл полезут двойные <code>\r\r\n</code>.
</details>
<details><summary>read_csv_typed — как применить schema</summary>
После <code>reader = csv.DictReader(f)</code> итерируй: <code>for row in reader:</code>.
Каждый <code>row</code> — это <code>dict[str, str]</code>. Пройди по <code>schema.items()</code>
и замени: <code>row[col] = schema[col](row[col])</code>. Не забудь, что <code>DictReader</code>
возвращает <code>OrderedDict</code> (Python 3.8+: обычный <code>dict</code>), конвертация
работает на месте. Собирай строки в список и верни.
</details>
<details><summary>round-trip — почему не сравнивать сырой текст</summary>
<code>csv.DictWriter</code> по умолчанию завершает строки <code>\r\n</code> (стандарт RFC 4180).
Поэтому правильный round-trip тест: записал через <code>write_csv</code> → прочитал через
<code>read_csv_typed</code> → сравнил словари, а не <code>read_text()</code>.
</details>

## Задание (существенное): TypedCSVReader

До сих пор `parse_csv` принимает уже-прочитанный текст и возвращает строки строками.
В реальных задачах CSV-файл читается с диска, а значения нужны уже готовыми к работе:
возраст — числом, флаг — булевым. Ты реализуешь `TypedCSVReader` — класс, который
**сам управляет файлом** (открывает / закрывает через контекстный менеджер) и
**приводит типы** согласно схеме, переданной при создании. Это шаг от «читаю текст» к
«работаю с данными».

### Спецификация API

```python
class TypedCSVReader:
    def __init__(self, path: StrPath, schema: dict[str, Callable[[str], Any]]) -> None: ...
    def __enter__(self) -> "TypedCSVReader": ...
    def __exit__(self, exc_type, exc_val, exc_tb) -> bool: ...
    def __iter__(self) -> Iterator[dict[str, Any]]: ...
```

**Параметры конструктора:**
- `path` — путь к CSV-файлу (строка или `Path`). Файл ещё **не открывается** в `__init__`.
- `schema` — словарь `{имя_колонки: конвертер}`. Конвертер — любой вызываемый:
  `int`, `float`, `str`, лямбда. Колонки, не упомянутые в `schema`, остаются строками.

**Контракт:**
- `__enter__` открывает файл на чтение (`encoding="utf-8"`), читает первую строку как
  заголовок. Если `schema` содержит имя, которого нет в заголовке, поднимает
  `KeyError(имя_колонки)`. Возвращает `self`.
- `__exit__` закрывает файл. Исключения **не подавляет** (возвращает `False`).
- `__iter__` итерирует строки данных (всё после первой). Каждая строка →
  `dict {имя: значение}`. Для колонок из `schema` вызывается конвертер; остальные —
  строки. Порядок ключей совпадает с заголовком. Пустой файл или только заголовок →
  итерация пустая.
- После выхода из `with`-блока итерация по уже-закрытому объекту должна поднимать
  `ValueError` («файл закрыт»).

**Краевые случаи:**
- `schema={}` — все значения строками (эквивалент `parse_csv`, но из файла).
- Конвертер поднял `ValueError` (например, `int("abc")`) — исключение прокидывается
  наружу без изменений.
- Несуществующий файл — `FileNotFoundError` от `open()` прокидывается из `__enter__`.

**Пример использования:**

```python
# grades.csv содержит: name,score,passed / Ada,95,true / Bjarne,72,false
with TypedCSVReader(path, schema={"score": int, "passed": lambda s: s == "true"}) as reader:
    rows = list(reader)
# rows[0] == {"name": "Ada", "score": 95, "passed": True}
# rows[1] == {"name": "Bjarne", "score": 72, "passed": False}
```

### Подсказки

<details><summary>Шаг 1 — где хранить состояние</summary>
В <code>__init__</code> просто сохрани <code>self._path = path</code> и
<code>self._schema = schema</code>. Добавь <code>self._file = None</code> и
<code>self._header: list[str] = []</code> — они заполнятся в <code>__enter__</code>.
Файл на этом этапе не открывай: класс должен позволять создать объект, не имея файла
рядом, и открыть его только внутри <code>with</code>.
</details>

<details><summary>Шаг 2 — __enter__: открытие и заголовок</summary>
Открой файл: <code>self._file = open(self._path, "r", encoding="utf-8")</code>. Прочитай
первую строку: <code>first = self._file.readline()</code>. Если файл пустой,
<code>readline()</code> вернёт <code>""</code> — сохрани пустой заголовок и вернись.
Иначе: <code>self._header = first.rstrip("\n").split(",")</code>. Затем проверь схему:
<code>for col in self._schema: if col not in self._header: raise KeyError(col)</code>.
Верни <code>self</code>.
</details>

<details><summary>Шаг 3 — __exit__: закрытие файла</summary>
Достаточно двух строк: <code>self._file.close()</code> и <code>return False</code>.
Возврат <code>False</code> означает: «не подавляю исключение» — если внутри
<code>with</code>-блока что-то упало, оно полетит дальше. Можно дополнительно
записать <code>self._file = None</code>, чтобы <code>__iter__</code> мог
проверить: файл закрыт?
</details>

<details><summary>Шаг 4 — __iter__: строки данных с конвертерами</summary>
Итерируй по открытому файлу: <code>for raw in self._file:</code> — каждый
<code>raw</code> — строка с <code>\n</code> на конце. Разбей:
<code>values = raw.rstrip("\n").split(",")</code>. Сшей с заголовком:
<code>row = dict(zip(self._header, values))</code>. Пройди по <code>row</code> и
для колонок из <code>self._schema</code> замени строку на
<code>self._schema[col](row[col])</code>. Сделай <code>yield row</code>.
Чтобы ловить закрытый файл: в начале метода проверь
<code>if self._file is None or self._file.closed: raise ValueError("файл закрыт")</code>.
</details>

## Микро-проект: Анализатор логов

Когда базовые функции зелёные — собери из навыков чтения данных маленький, но настоящий
инструмент: разбор лог-файла. Логи — это строки вида `УРОВЕНЬ: сообщение`, например
`ERROR: disk full`. Инженеру в первую очередь нужно понять: сколько чего случилось и какие
именно ошибки. Этим занимаются и сложные системы мониторинга — в основе та же идея.

Файл `loganalyzer.py`. На вход обеим функциям приходит список строк-логов (уже разбитый по
строкам — как раз результат твоего `read_lines`). Уровни: `INFO`, `WARN`, `ERROR`.

1. `count_levels(lines: list[str]) -> dict[str, int]` — посчитать, сколько строк каждого
   уровня. Строка имеет вид `"LEVEL: message"` — уровень это часть **до первого `:`**. В ответе
   только реально встретившиеся уровни (на пустом вводе — пустой словарь `{}`). Строки, не
   подходящие под формат (нет `:` или неизвестный уровень), **игнорируй**.

2. `errors_only(lines: list[str]) -> list[str]` — вернуть только строки уровня `ERROR`,
   **целиком и в исходном порядке**.

```bash
./python/run.sh 12
```

### Подсказки

<details><summary>count_levels — как достать уровень</summary>
Уровень — это часть строки до первого двоеточия. Удобно <code>line.split(":", 1)</code> —
второй аргумент <code>1</code> делит максимум один раз, так что двоеточия в самом сообщении не
помешают. Возьми <code>parts[0]</code> и сними пробелы <code>.strip()</code>. Считать можно
обычным <code>dict</code> через <code>d.get(level, 0) + 1</code> или взять
<code>collections.Counter</code> из модуля 11. Не забудь отбросить уровни вне набора
<code>{"INFO", "WARN", "ERROR"}</code> и строки без двоеточия.
</details>
<details><summary>errors_only — фильтр</summary>
Пройди по строкам и оставь те, у которых уровень (часть до первого <code>:</code>) равен
<code>"ERROR"</code>. Тот же приём <code>split(":", 1)[0].strip()</code>. Возвращай строку
<strong>целиком</strong> (как пришла), а не только сообщение. Списковое включение
<code>[line for line in lines if ...]</code> читается лучше всего и само сохраняет порядок.
</details>

Реализуешь — покажи код, скажу, в верном ли направлении. Особенно любопытно глянуть на
<code>count_levels</code>: обработка «мусорных» строк (без <code>:</code> или с чужим уровнем) —
частое место, где «вроде работает», но на смешанном вводе спотыкается.
