# Модуль 00 — Setup и как запускается Python

Это самый первый шаг. Здесь мы не учим «программировать на Python» — мы учимся
**запускать** Python и **видеть результат** тестов. Если ты пришёл из C++ курса рядом
(папка `cpp/`), держи в голове одно отличие: там был долгий цикл «собрать → запустить»,
здесь его почти нет. Поняв этот модуль, ты будешь спокойно гонять `red → green` и не
бояться вывода pytest.

**Большая идея модуля:** Python исполняет твой `.py` построчно, прямо как есть; чтобы это
было предсказуемо и повторяемо, мы изолируем зависимости в виртуальном окружении, а
проверяем себя тестами, которые точно знают, как должен вести себя код.

## Идея 1. Python интерпретируется, а не компилируется

В C++ исходник сначала **компилируется** в бинарник, и только потом ты его запускаешь:
ошибка в строке 50 всплывёт ещё до старта программы. В Python отдельного шага сборки нет —
интерпретатор читает файл **сверху вниз** и выполняет строку за строкой.

```python
print("строка 1 выполнилась")
print("строка 2 выполнилась")
foo()   # ошибка проявится ТОЛЬКО когда дойдём до этой строки
```

**Почему это важно.** Код «до» ошибки успеет отработать — это и плюс (быстрый цикл правок:
сохранил → запустил, без сборки), и подвох (ошибка в неиспользованной ветке может тихо
ждать своего часа). Поэтому в Python так ценят тесты: они **проходят по строкам за тебя** и
ловят то, что компилятор поймал бы в C++.

> Уточнение для дотошных: CPython всё же компилирует исходник в промежуточный байткод
> (`.pyc` в `__pycache__`), но делает это автоматически и незаметно. Отдельной команды
> «собери», как `cmake`/`g++`, тебе вызывать не нужно.

## Идея 2. Первая программа: print, input и запуск файла

Прежде чем идти дальше — убедимся, что ты умеешь запустить простейшую программу и
поговорить с ней.

### Вывод и ввод

```python
print("Привет, мир!")          # вывод строки в терминал (stdout)
name = input("Как тебя зовут? ")  # читает строку, введённую пользователем, и кладёт в name
print("Привет,", name)         # print с запятой — печатает с пробелом между аргументами
```

`print(...)` принимает любое количество аргументов и по умолчанию разделяет их пробелом.
`input(...)` всегда возвращает **строку** (`str`) — даже если пользователь набрал «42».
Чтобы получить число, нужно явное преобразование: `int(input("Введи число: "))`.

### Запуск файла напрямую

Сохрани код в файл `hello.py` и запусти:

```bash
python3 hello.py
```

Интерпретатор читает файл сверху вниз и выполняет его. Никакой компиляции — сохранил,
запустил, увидел результат.

### if \_\_name\_\_ == "\_\_main\_\_"

Когда Python запускает файл **напрямую** (`python3 hello.py`), он устанавливает
специальную переменную `__name__` в строку `"__main__"`. Когда тот же файл
**импортируют** (`import hello` из другого модуля), `__name__` будет равна `"hello"`.

```python
def greet(name: str) -> str:
    return f"Привет, {name}!"

if __name__ == "__main__":
    # этот блок выполнится только при прямом запуске файла,
    # но НЕ при импорте — поэтому тесты его не трогают
    user = input("Имя: ")
    print(greet(user))
```

**Почему это важно.** Без этой защиты `input(...)` в корне модуля сработал бы при
каждом импорте — pytest завис бы, ожидая ввода. Весь «запускаемый» код принято прятать
за `if __name__ == "__main__"`: функции и классы объявляй свободно, а побочные эффекты
(ввод, вывод, сетевые запросы) изолируй в этот блок.

**Подвох.** `input()` всегда возвращает строку — классическая ошибка новичка:

```python
x = input("Число: ")   # пользователь ввёл 5
y = x + 1              # TypeError: can only concatenate str (not "int") to str
```

Решение: `x = int(input("Число: "))`.

## Идея 3. Виртуальное окружение (venv) и pip

`pip` ставит сторонние пакеты (например, `pytest`). Если ставить их «глобально», все проекты
делят один набор версий — и рано или поздно проект A хочет `pytest 7`, а проект B `pytest 8`,
и они конфликтуют. **Виртуальное окружение** (`venv`) — это отдельная папка с собственным
Python и собственными пакетами, привязанная к одному проекту.

```bash
python3 -m venv .venv          # создать окружение в папке .venv
source .venv/bin/activate      # «войти» в него (в нашем run.sh это делается за тебя)
pip install pytest             # пакет ставится в .venv, а не на всю систему
```

**Почему это важно.** Изоляция = повторяемость: у тебя и у любого другого человека один и
тот же набор версий, «у меня работает» перестаёт быть отговоркой. Наш скрипт `run.sh`
создаёт `.venv` и ставит `pytest` **сам, один раз** — тебе про venv достаточно понимать, что
он есть и зачем.

## Идея 4. pytest и цикл red → green

`pytest` — это программа, которая находит файлы `test_*.py`, запускает функции `test_*` и
проверяет твои `assert`. Каждое задание в курсе устроено одинаково: **тесты уже написаны** и
знают правильное поведение, а ты дописываешь функцию, пока красное не станет зелёным.

```python
def test_add():
    assert add(2, 3) == 5     # тест ОЖИДАЕТ, что add(2,3) даст 5
```

Пока функция не реализована, тест **падает** (red). Реализовал верно — **проходит** (green).
Учись читать вывод. Упавший тест выглядит так:

```text
    def test_add():
>       assert add(2, 3) == 5
E       assert 4 == 5          # слева — что ПОЛУЧИЛОСЬ, справа — что ОЖИДАЛОСЬ
```

Строка с `>` — где сломалось; строка с `E` — суть расхождения. На старте этого модуля тесты
красные не из-за неверного ответа, а потому что функция ещё кидает `NotImplementedError` —
это нормально и ожидаемо.

**Почему это важно.** Тест — это исполняемая спецификация. Тебе не нужно гадать, «а правильно
ли я понял задачу»: запусти тесты, и они скажут. Это и есть твой главный инструмент обратной
связи на весь курс.

## Идея 5. Динамическая типизация

В C++ тип переменной фиксируется при объявлении (`int x = 5;` — `x` навсегда `int`). В Python
переменная — это просто **имя**, привязанное к значению; имя не имеет типа, а вот **значение**
типизировано всегда.

```python
x = 5        # сейчас x ссылается на int
x = "пять"   # теперь то же имя ссылается на str — Python не против
type(x)      # <class 'str'>
```

При этом сами значения строго типизированы: `1 + 2 == 3`, но `1 + "2"` — **ошибка**
(`TypeError`), Python не станет молча складывать число со строкой.

```python
def add(a: int, b: int) -> int:   # это type hints — ПОДСКАЗКА читателю и инструментам
    return a + b
```

**Почему это важно.** Аннотации `a: int` в сигнатуре — **не** проверка во время выполнения:
Python их не навязывает, передать туда строку технически можно. Они нужны людям и линтерам,
чтобы понимать намерение. А раз компилятор за типами не следит — снова выручают тесты.
