# Модуль 04 — Строки и обработка текста

Строка — это, наверное, самый частый тип данных в реальном коде: имена, ввод
пользователя, строки лог-файла, ответы из сети. **Большая идея модуля:** в Python строка —
это *неизменяемая* последовательность символов, и почти вся работа с текстом сводится к
тому, чтобы из одной строки аккуратно собрать другую. Поняв это, ты перестанешь искать
несуществующий «метод, который меняет строку на месте», и начнёшь мыслить в стиле
«взять → преобразовать → вернуть новое».

## Идея 1. Методы строк: split / join / strip / lower / replace

Это рабочая лошадка обработки текста. Запомни пять движений — и 80% задач закрыты.

```python
s = "  Привет, Мир  "
s.strip()              # "Привет, Мир"   — срезает пробелы по краям
s.lower()              # "  привет, мир  " — в нижний регистр
s.replace(",", "")     # "  Привет Мир  " — заменить подстроку
"a b  c".split()       # ["a", "b", "c"]  — split() без аргумента бьёт по ЛЮБЫМ пробелам
"a,b,c".split(",")     # ["a", "b", "c"]  — с аргументом бьёт строго по нему
"-".join(["a", "b"])   # "a-b"            — собрать список обратно в строку
```

**ПОЧЕМУ это важно.** `split()` без аргумента — особый герой: он не только режет по
пробелам, но и *схлопывает* несколько подряд идущих пробелов и игнорирует края. То есть
`"  a   b ".split()` даёт ровно `["a", "b"]`, без пустых элементов. Это прямой путь к
заданию `normalize_spaces`. А `join` — зеркало `split`: строка, на которой ты вызываешь
`join`, становится «клеем» между элементами. Частая ошибка новичка — писать
`["a","b"].join("-")`; правильно наоборот: `"-".join([...])`.

## Идея 2. Срезы и разворот `s[::-1]`

Строка индексируется как список, и к ней применимы *срезы* `s[начало:конец:шаг]`.

```python
s = "Python"
s[0]        # "P"      — первый символ
s[-1]       # "n"      — последний (отрицательный индекс = с конца)
s[0:3]      # "Pyt"    — символы с 0 по 2 (конец НЕ включается)
s[::2]      # "Pto"    — каждый второй
s[::-1]     # "nohtyP" — шаг -1 идёт справа налево: это и есть разворот
```

**ПОЧЕМУ `s[::-1]` разворачивает.** Шаг `-1` означает «двигаться к началу», а
пропущенные `начало`/`конец` при отрицательном шаге берутся как «с самого конца до самого
начала». Получается перебор всех символов в обратном порядке. Это короче и быстрее ручного
цикла. Пригодится в `is_palindrome`: палиндром — это строка, равная самой себе
перевёрнутой.

> Подводный камень: срез никогда не «вылетает» за границы. `"abc"[1:99]` спокойно даёт
> `"bc"`, а не ошибку. Это удобно, но иногда маскирует баг — будь внимателен.

## Идея 3. f-строки и форматирование

f-строка — это литерал с `f` перед кавычкой, где в `{...}` можно вставить выражение.

```python
name, n = "Аня", 3
f"{name} прислала {n} файла"     # "Аня прислала 3 файла"
f"{n * 2}"                        # "6"   — внутри {} обычный Python
f"{3.14159:.2f}"                  # "3.14" — :.2f = два знака после точки
f"{42:>5}"                        # "   42" — выравнивание по правому краю в поле ширины 5
```

**ПОЧЕМУ f-строки лучше склейки через `+`.** Сравни: `name + " прислала " + str(n)` —
длинно, легко забыть `str(...)` и получить `TypeError`. f-строка читается как обычный
текст с «окошками», сама приводит значения к строке и поддерживает мини-язык форматирования
после двоеточия. В этом модуле они понадобятся для `title_case` и в выводе частот.

## Идея 4. Строки неизменяемы — методы возвращают новую строку

Это ключ ко всему модулю. У строки **нельзя** изменить символ на месте.

```python
s = "кот"
s[0] = "р"          # TypeError! строки неизменяемы
s = "р" + s[1:]     # "рот" — мы НЕ изменили старую строку, а собрали новую
up = s.upper()      # up == "РОТ", но s по-прежнему "рот"
s.upper()           # вызвали — и выбросили результат: исходная s не меняется!
```

**ПОЧЕМУ так и почему это важно.** Методы вроде `lower`, `strip`, `replace` *ничего не
портят* в исходной строке — они **возвращают новую**. Поэтому строку всегда нужно
*присвоить*: `s = s.strip()`, а не просто `s.strip()`. Самый частый баг новичка —
«вызвал метод, а строка не поменялась». Не поменялась, потому что и не должна:
неизменяемость — это гарантия, что переданную тобой строку никто втихаря не испортит.
Зато одинаковые строки можно безопасно переиспользовать в памяти.

## Идея 5. Текст vs байты: `str` в памяти и `bytes` на диске

В Python строка `str` — это **последовательность кодовых точек Unicode** (числовых
идентификаторов символов). В памяти Python хранит их в собственном компактном формате;
ты никогда не видишь «сырых байтов». Но при записи в файл, отправке по сети или
передаче в C-библиотеку строку нужно **закодировать** — превратить в `bytes`.

```python
s = "Привет"          # str: последовательность 6 кодовых точек
b = b"hello"          # bytes: литерал b"..." — именно байты (0..255)
type(s)               # <class 'str'>
type(b)               # <class 'bytes'>
```

```python
# bytes — неизменяемая последовательность целых 0..255
b = bytes([72, 101, 108, 108, 111])   # b'Hello'
b[0]                  # 72  — не символ, а число
```

**ПОЧЕМУ разделение важно.** Когда ты открываешь файл без `encoding=`, Python на Windows
может выбрать CP1251, на Linux — UTF-8, и текст придёт другим. Это источник
«кракозябр», которые преследуют junior-разработчиков. Правило: **всегда явно указывай
`encoding="utf-8"` при `open()`**, если не знаешь, что делаешь.

```python
with open("file.txt", "r", encoding="utf-8") as f:
    text = f.read()   # гарантированно UTF-8, независимо от ОС
```

## Идея 6. encode / decode: туда и обратно

**Кодирование** (`str → bytes`) — выбираешь кодировку, Python выдаёт байтовое
представление.

```python
s = "Кот"
s.encode("utf-8")        # b'\xd0\x9a\xd0\xbe\xd1\x82'  — 6 байт (каждый символ 2 байта)
s.encode("cp1251")       # b'\xca\xee\xf2'              — 3 байта
```

**Декодирование** (`bytes → str`) — обратная операция:

```python
b = b'\xd0\x9a\xd0\xbe\xd1\x82'
b.decode("utf-8")        # "Кот"
b.decode("cp1251")       # "╨Я╨╛╤В"  — моджибейк: байты от UTF-8 читаем как CP1251
```

**Откуда `UnicodeDecodeError`.**

```python
b'\xff\xfe'.decode("utf-8")
# UnicodeDecodeError: 'utf-8' codec can't decode byte 0xff in position 0
```

Ошибка возникает, когда байты не соответствуют выбранной кодировке. Типичный сценарий:
файл написан в CP1251, а ты читаешь `open(..., encoding="utf-8")`. Решение — либо
указать правильную кодировку, либо использовать `errors="replace"` / `errors="ignore"`.

**Моджибейк** (文字化け) — японский термин для «кракозябр»: текст, декодированный не той
кодировкой. Исправить потом сложно; проще предотвратить правильным `encoding=`.

```python
# Безопасный fallback-декодер:
def safe_decode(data: bytes, encodings: list[str]) -> str:
    for enc in encodings:
        try:
            return data.decode(enc)
        except (UnicodeDecodeError, LookupError):
            continue
    raise ValueError(f"Cannot decode with any of {encodings}")
```

## Идея 7. Юникод честно: кодовые точки, эмодзи, нормализация

**`len(s)` считает кодовые точки, не визуальные символы.**

```python
s = "é"               # ЛАТИНСКАЯ СТРОЧНАЯ E + ЗНАК УДАРЕНИЯ (2 кодовые точки)
len(s)                # 2  — две кодовые точки!
```

Но `"é"` может быть записана и одной кодовой точкой (предсобранный символ U+00E9). Две
строки выглядят одинаково, но не равны друг другу:

```python
e_composed   = "é"              # é — 1 кодовая точка (NFC)
e_decomposed = "é"             # e + ◌́ — 2 кодовые точки (NFD)
e_composed == e_decomposed           # False!
len(e_composed), len(e_decomposed)   # (1, 2)
```

**Эмодзи** — тоже кодовые точки, но часто несколько:

```python
flag = "🇷🇺"          # флаг России = 2 региональных индикатора
len(flag)             # 2
```

**Почему `[::-1]` небезопасен на «сложном» юникоде.**

```python
s = "é"         # e + знак ударения
s[::-1]               # '́e' — знак ударения оказался перед e: визуально неверно
```

**Решение — нормализация перед сравнением или срезами:**

```python
import unicodedata
unicodedata.normalize("NFC", "é")   # "é" — 1 кодовая точка
unicodedata.normalize("NFD", "é")    # "é" — 2 кодовые точки
```

NFC («составная форма») — стандарт для хранения в базах данных, HTTP, современных ОС.
При сравнении строк из разных источников всегда нормализуй обе стороны в NFC.

## Идея 8. f-строки глубже: выравнивание, разделители, debug

f-строки поддерживают мини-язык форматирования после двоеточия: `{значение:спецификация}`.

**Выравнивание и заливка:**

```python
x = 42
f"{x:<10}"     # "42        "  — по левому краю в поле шириной 10
f"{x:>10}"     # "        42"  — по правому краю
f"{x:^10}"     # "    42    "  — по центру
f"{x:*^10}"    # "****42****"  — по центру, заполнитель *
f"{'hi':_<8}"  # "hi______"    — по левому, заполнитель _
```

**Разделители разрядов:**

```python
n = 1_000_000
f"{n:,}"       # "1,000,000"   — запятая-разделитель (английский стиль)
f"{n:_}"       # "1_000_000"   — подчёркивание-разделитель
```

**Ширина и точность для вещественных чисел:**

```python
pi = 3.14159
f"{pi:.2f}"        # "3.14"          — 2 знака после точки
f"{pi:>10.2f}"     # "      3.14"    — поле 10, выравнивание вправо
f"{pi:0>10.2f}"    # "0000003.14"    — заполнитель 0
```

**Debug-спецификатор `=`** (Python 3.8+):

```python
x = 42
f"{x=}"        # "x=42"          — имя переменной + значение, отлично для отладки
f"{x * 2 = }"  # "x * 2 = 84"   — работает и с выражениями
```

**Подвох:** символ `{` или `}` в f-строке нужно удвоить: `f"{{literal}}"` → `"{literal}"`.

## Микро-проект: Частотный словарь

Когда базовые функции зелёные — собери из них маленький, но настоящий инструмент:
посчитать, какие слова в тексте встречаются чаще всего. Это классическая задача обработки
текста (так работают облака тегов, простые анти-спам фильтры, аналитика отзывов).

Файл `wordfreq.py`. Две функции:

1. `word_count(text: str) -> dict[str, int]` — словарь «слово → сколько раз встретилось».
   Слова разделяются **пробелами** (используй `split()`), сравнение **без учёта регистра**
   (приведи к нижнему). Для `"a A a b"` ответ — `{"a": 3, "b": 1}`.
2. `top_n(text: str, n: int) -> list[tuple[str, int]]` — список из `n` самых частых слов,
   как пары `(слово, частота)`. Сортировка: сначала по **убыванию частоты**, при равной
   частоте — по **алфавиту** (по возрастанию). Если `n` больше числа уникальных слов —
   вернуть сколько есть.

### Подсказки

<details><summary>word_count — как считать</summary>
Заведи пустой `dict`. Пройди по `text.lower().split()` и для каждого слова делай
`d[w] = d.get(w, 0) + 1`. Метод `get` со вторым аргументом возвращает 0, если слова ещё
нет, — так не нужен лишний `if`. (Когда дойдёшь до модуля 11, узнаешь про
`collections.Counter` — он делает это в одну строку. Здесь же важно понять механику.)
</details>

<details><summary>top_n — двойная сортировка и tie-break</summary>
Возьми `word_count(text).items()` — это пары `(слово, частота)`. Сложность в том, что
частоту надо сортировать по убыванию, а слово при равенстве — по возрастанию. Если просто
взять `key=lambda p: (-p[1], p[0])`, то минус развернёт частоту (большие — первыми), а
слово останется по алфавиту. После `sorted(...)` возьми срез `[:n]`.
</details>

<details><summary>top_n — почему именно (-частота, слово)</summary>
`sorted` сравнивает кортежи покомпонентно: сначала первый элемент, при равенстве — второй.
Нам нужен главный ключ «частота по убыванию» → берём `-частота` (тогда обычная сортировка
по возрастанию даст убывание частоты). Вторичный ключ «слово по возрастанию» оставляем как
есть. Кортеж `(-freq, word)` кодирует ровно это правило.
</details>
