# Модуль 14 — Инструменты и качество кода

Большая идея модуля: код пишут не в вакууме. Вокруг junior-разработчика стоит набор
**инструментов**, которые ловят ошибки раньше, держат стиль единым и помогают разбираться,
*почему* код ведёт себя не так. В этом модуле мы не учим новую конструкцию языка — мы учим
**ремесло**: изолировать окружение, форматировать и проверять код, ходить по нему отладчиком и
сохранять историю в git. И сразу применяем главное умение инженера — **находить и чинить баги**.

Этот модуль особенный. В файле `buggy.py` функции уже написаны и **запускаются** — никаких
`NotImplementedError`. Но в каждой спрятан баг, и из-за этого тесты красные. Твоя задача — не
писать с нуля, а **починить**. Это ровно то, чем junior занимается в первый месяц на работе:
читает чужой код, воспроизводит проблему, находит строку-виновницу, правит.

## Идея 1. `venv` и `pip` — изоляция и `requirements`

Разные проекты хотят разные версии библиотек. Если ставить всё «глобально», рано или поздно
проект A потребует `requests==2.0`, а проект B — `requests==3.0`, и они подерутся. Решение —
**виртуальное окружение**: отдельная песочница с собственным набором пакетов под каждый проект.

```bash
python3 -m venv .venv          # создать окружение в папке .venv
source .venv/bin/activate      # «войти» в него (Windows: .venv\Scripts\activate)
pip install pytest             # ставится ТОЛЬКО внутрь .venv, не в систему
pip freeze > requirements.txt  # записать точные версии в файл
deactivate                     # выйти обратно
```

ПОЧЕМУ это важно: `requirements.txt` — это список зависимостей проекта с версиями. Любой другой
человек (или ты на новой машине) делает `python3 -m venv .venv` и `pip install -r requirements.txt`
— и получает **тот же** набор пакетов. Воспроизводимость окружения это половина успеха: «у меня
работает, а у тебя нет» почти всегда про разные версии.

> Кстати, скрипт `./python/run.sh` этого курса сам создаёт `.venv` и ставит туда `pytest` —
> ты уже пользуешься виртуальным окружением, просто оно спрятано за одной командой.

## Идея 2. `pyproject.toml` — единый дом проекта

Раньше каждый инструмент нёс свой конфиг-файл: `setup.cfg` для пакета, `.flake8` для линтера,
`mypy.ini` для проверки типов, `tox.ini` для тестов. В большом проекте это **россыпь файлов в
корне**, и смотреть, «где что настроено», неудобно. Стандарт **PEP 621** собрал всё в один файл —
`pyproject.toml`.

```toml
# pyproject.toml
[project]
name = "my-package"
version = "0.1.0"
requires-python = ">=3.11"
dependencies = ["requests>=2.28"]

[project.optional-dependencies]
dev = ["pytest", "hypothesis", "ruff", "mypy"]

[tool.ruff]
line-length = 88
select = ["E", "F", "I"]   # pycodestyle, pyflakes, isort

[tool.ruff.lint.per-file-ignores]
"tests/*" = ["S101"]       # в тестах assert — это норма

[tool.mypy]
strict = true
ignore_missing_imports = true
```

ПОЧЕМУ это важно: `[project]` описывает **что** это за пакет и **от чего** зависит — полная замена
`requirements.txt` плюс метаданные для `pip install`. Разделы `[tool.*]` — настройки инструментов
**в одном месте**: `ruff`, `mypy`, `black`, `pytest` — каждый читает свою секцию из того же файла.
Один файл — один источник правды для всего проекта.

> Подвох: `pyproject.toml` стал стандартом в Python ≥ 3.11 / pip ≥ 21.3. В старых проектах
> ты ещё встретишь `setup.py` и `setup.cfg` — это не ошибка, просто другая эпоха.

## Идея 3. Форматтер `black` и линтер `ruff`

Два разных инструмента про «чистоту», их легко перепутать:

- **Форматтер** (`black`) — переписывает **внешний вид** кода по единому стилю: отступы, кавычки,
  переносы, пробелы. Он не думает про смысл, он причёсывает. Запускается и просто меняет файлы.
- **Линтер** (`ruff`) — **проверяет** код и сообщает о подозрительном: неиспользованный импорт,
  переменная объявлена и не используется, сравнение с `None` через `==` вместо `is`, и так далее.
  Он не меняет смысл — он предупреждает (и часть мелочей умеет чинить сам).

```bash
black buggy.py        # переформатировать файл по стандарту
ruff check buggy.py   # показать предупреждения (warnings/ошибки стиля и логики)
ruff check --fix .    # починить то, что ruff умеет чинить автоматически
```

ПОЧЕМУ это важно: единый стиль убирает споры «где ставить пробел» из код-ревью — за это отвечает
`black`, и обсуждать нечего. А `ruff` ловит целый класс ошибок **до запуска**: например,
неиспользованный импорт часто значит, что ты что-то не дописал, а `== None` — почти всегда баг.
Линтер — это бесплатная пара дополнительных глаз, которая смотрит на код раньше тестов.

> `black` и `ruff` не входят в стандартную библиотеку — их ставят через `pip` (см. Идею 1) и
> обычно прописывают в `requirements.txt`. Запоминалка: **black** делает код *одинаковым*,
> **ruff** делает код *честным*.

### Современный `ruff format` и `functools.lru_cache`

Два дополнения, которые стоит знать сразу:

**`ruff format`** — с версии 0.1 `ruff` умеет не только проверять, но и **форматировать** код,
полностью заменяя `black`. Одна утилита вместо двух:

```bash
ruff format .          # переформатировать весь проект (как black)
ruff check --fix .     # починить то, что можно автоматически
```

Скорость `ruff format` в десятки раз выше `black` на больших кодовых базах — именно поэтому
его всё чаще выбирают вместо `black`.

**`functools.lru_cache`** — встроенный в стандартную библиотеку кэш результатов, stdlib-аналог
нашего `memoize`:

```python
from functools import lru_cache

@lru_cache(maxsize=128)
def fib(n):
    if n < 2:
        return n
    return fib(n - 1) + fib(n - 2)
```

`lru_cache` реализует **LRU-политику** (Least Recently Used): когда кэш заполнен до `maxsize`,
вытесняет самые давно использованные записи. При `maxsize=None` кэш неограничен — это аналог
нашего простого `memoize`. Наш `memoize` из задания — это учебная основа; в реальном коде
обычно берут `lru_cache`.

## Идея 4. `mypy` — статическая проверка типов

`mypy` — инструмент, который читает **аннотации типов** в Python-коде и проверяет, что они
не противоречат друг другу. Баги находятся **до запуска**, без тестов:

```python
# example.py
def greet(name: str) -> str:
    return "Hello, " + name

greet(42)   # mypy поймает: Argument 1 to "greet" has incompatible type "int"; expected "str"
```

Запуск прямой:

```bash
mypy example.py
mypy decorators.py   # на файле из этого модуля: типы аннотированы, ошибок не будет
mypy --strict .      # строгий режим: требует аннотации везде, запрещает Any
```

ПОЧЕМУ это важно: в динамическом Python ошибки типов — распространённый источник багов, которые
проявляются только в рантайме на редких входных данных. `mypy` находит их **статически**:
передал `int` вместо `str`, вернул `None` там, где ожидался `int`, перепутал `Optional[str]` и
`str` — всё это ловится при анализе, а не при запуске. Посмотри на `decorators.py`: там все
функции аннотированы (`times: int`, `-> Callable`). Запусти `mypy decorators.py` — увидишь
чистый отчёт. Теперь намеренно сломай аннотацию и посмотри, что именно напишет `mypy`.

> Подвох: `mypy` не гарантирует отсутствие ошибок — он проверяет то, что аннотировано.
> Неаннотированный код (или код с `Any`) он молча пропускает. Именно поэтому `--strict`
> принуждает аннотировать всё — и именно поэтому в реальных проектах добавляют `mypy`
> в CI постепенно, модуль за модулем.

## Идея 5. `pre-commit` — автоматический контроль качества перед коммитом

Отдельно запускать `ruff`, `mypy`, `black` перед каждым `git commit` — легко забыть. **`pre-commit`**
решает это: он регистрирует **git-хук** (`pre-commit`), который автоматически запускается
перед каждым коммитом и не даёт закоммитить код, не прошедший проверки.

Конфиг — файл `.pre-commit-config.yaml` в корне проекта:

```yaml
# .pre-commit-config.yaml
repos:
  - repo: https://github.com/astral-sh/ruff-pre-commit
    rev: v0.4.0
    hooks:
      - id: ruff          # линтер
      - id: ruff-format   # форматтер

  - repo: https://github.com/pre-commit/mirrors-mypy
    rev: v1.10.0
    hooks:
      - id: mypy
        args: [--ignore-missing-imports]
```

Установка и активация:

```bash
pip install pre-commit
pre-commit install          # один раз: регистрирует хук в .git/hooks/pre-commit
pre-commit run --all-files  # запустить вручную на всех файлах (без коммита)
```

Теперь при каждом `git commit`:
1. `ruff` проверяет стиль и логику — если есть ошибки, коммит отменяется.
2. `ruff-format` проверяет форматирование (или исправляет его).
3. `mypy` проверяет типы.

Коммит проходит, только если все хуки зелёные.

ПОЧЕМУ это важно: `pre-commit` **сшивает** инструменты в единый барьер качества прямо
в рабочий цикл git. Команде не нужно договариваться «помни запускать линтер» — это
происходит автоматически. Плохой код не попадает в историю проекта.

> Подвох: `pre-commit` проверяет только **staged** файлы (те, что в `git add`).
> Если ты сделал `git add -p` и закоммитил часть изменений, хуки видят только
> её — остаток в рабочем дереве не проверяется.

## Идея 6. Отладчик `pdb`: `breakpoint()` и команды `n/s/p/c`

`print` — нормальный способ заглянуть в код, но он быстро превращается в свалку. **Отладчик**
позволяет **остановить** программу в нужной точке и ходить по ней пошагово, разглядывая значения.
В Python встроенная команда `breakpoint()` ставит точку останова прямо в коде:

```python
def average(xs):
    breakpoint()          # тут выполнение остановится, откроется интерактивный pdb
    return (sum(xs) + 1) / len(xs)
```

Запускаешь как обычно — на строке `breakpoint()` появится приглашение `(Pdb)`. Базовые команды:

| команда | что делает |
|---|---|
| `n` (next) | выполнить текущую строку и встать на следующую (НЕ заходя внутрь вызовов) |
| `s` (step) | то же, но **зайти внутрь** вызванной функции |
| `p выражение` (print) | напечатать значение: `p xs`, `p sum(xs)`, `p len(xs)` |
| `c` (continue) | продолжить выполнение до конца (или до следующего `breakpoint`) |
| `l` (list) | показать код вокруг текущей строки |
| `q` (quit) | выйти из отладчика |

ПОЧЕМУ это важно: баг — это всегда расхождение между «что я думаю происходит» и «что происходит
на самом деле». `p` показывает реальные значения в реальный момент, и расхождение мгновенно
становится видно. Для багов этого модуля приём один и тот же: поставь `breakpoint()` перед
`return`, прогони падающий пример, и через `p` сравни то, что функция собирается вернуть, с тем,
что ждёт тест.

> Не забудь убрать `breakpoint()` перед тем, как считать задачу решённой — иначе тесты «зависнут»
> на приглашении `(Pdb)`. Кстати, `ruff` умеет ругаться на забытый `breakpoint()` в коде.

## Идея 7. `git` для junior: `status/add/commit/log/diff/branch`

`git` — система контроля версий: он хранит **историю** твоего кода снимками («коммитами»), чтобы
можно было откатиться, сравнить и понять, *что и когда* изменилось. Минимальный набор команд,
которого junior хватает на первые месяцы:

```bash
git status            # что изменено / что готово к коммиту — твоя «карта местности»
git diff              # ПОСТРОЧНО показать, что именно изменилось (ещё не закоммиченное)
git add buggy.py      # отметить файл к следующему коммиту (положить в «корзину»)
git commit -m "fix: почини off-by-one в sum_first_n"   # сохранить снимок с описанием
git log --oneline     # история коммитов кратко, по одной строке на коммит
git branch fix-bugs   # создать ветку (отдельную линию истории под задачу)
```

ПОЧЕМУ это важно: история даёт **бесстрашие**. Можно смело менять код — если сломал, `git diff`
покажет что, а откат вернёт рабочую версию. Хороший рабочий ритм: перед правкой — `git status`
(понять, где стоишь), после правки — `git diff` (перечитать собственные изменения, часто баг
видно прямо тут), затем `git add` + `git commit` с осмысленным сообщением. **Ветки** позволяют
вести задачу в стороне от основной линии и влить её, когда всё готово и тесты зелёные.

> Сообщение коммита — это записка будущему себе и команде. «fix» или «изменения» бесполезны;
> «fix: average больше не прибавляет лишнюю 1» — сразу понятно, что и зачем.

которого падает её тест. Почини так, чтобы тесты позеленели. **Сигнатуры менять нельзя.**

3. `average(xs: list[int]) -> float` — **среднее** непустого списка как `float`.

4. `has_duplicate(xs: list[int]) -> bool` — есть ли в списке **повтор** значения.
   `has_duplicate([1, 2, 3]) is False`, `has_duplicate([1, 2, 2]) is True`, на `[5]` — `False`.
   *(подвох: элемент сравнивается сам с собой)*

5. `repeat(s: str, times: int) -> str` — строка `s`, повторённая **ровно** `times` раз.
   `repeat("ab", 3) == "ababab"`, `repeat("ab", 0) == ""`. *(подвох: `times - 1`)*

## Задание (существенное): Декораторы — retry, memoize, timed

Декоратор — это функция, которая **оборачивает** другую функцию и добавляет ей новое поведение,
не меняя её исходный код. Именно из декораторов состоят реальные инструменты: логирование,
кэши, повторные попытки при ошибках, замер времени — всё это работает одной строкой `@decorator`
над любой функцией. В этом задании ты реализуешь три таких инструмента сам.

Код — в файле `decorators.py`. Стабы уже там, сигнатуры и docstring'и описывают контракт.
Тесты — в `test_decorators.py` и `test_decorators_props.py` — сейчас красные.

### API и контракт

**`retry(times: int) -> Callable`** — декоратор-фабрика.

```python
@retry(3)
def flaky():
    ...
```

- Вызывает оригинальную функцию до `times` раз, пока она бросает исключение.
- Если на какой-то попытке функция вернула значение — сразу вернуть его, больше не пытаться.
- Если все `times` попыток завершились исключением — пробросить исключение **последней** попытки.
- `times >= 1`. При `times == 1` — ровно одна попытка, как без декоратора.
- `functools.wraps` обязателен: `__name__`, `__doc__`, `__wrapped__` сохраняются.

**`memoize(func: Callable) -> Callable`** — декоратор напрямую.

```python
@memoize
def expensive(x, y):
    ...
```

- Кэширует результаты по ключу `(args, frozenset(kwargs.items()))`.
- Повторный вызов с теми же аргументами возвращает значение из кэша, функция не вызывается.
- Кэш доступен как `func.cache` (обычный `dict`). Очистка: `func.cache.clear()`.
- `functools.wraps` обязателен.

**`timed(func: Callable) -> Callable`** — декоратор напрямую.

```python
@timed
def slow():
    ...
```

- После каждого вызова сохраняет время выполнения в `func.last_elapsed` (секунды, `float`).
- До первого вызова `last_elapsed == None`.
- Если функция бросает исключение — время всё равно фиксируется, исключение пробрасывается.
- `functools.wraps` обязателен.

### Подсказки

<details><summary>Шаг 0 — как вообще устроен декоратор</summary>

Декоратор — это функция, которая принимает функцию и возвращает функцию.
<code>@retry(3)</code> — сначала вызывается <code>retry(3)</code> (возвращает обёртку),
затем эта обёртка применяется к исходной функции.

```python
def retry(times):
    def decorator(func):
        @functools.wraps(func)
        def wrapper(*args, **kwargs):
            ...  # логика повторов
        return wrapper
    return decorator
```

Обязательно вызывай <code>functools.wraps(func)</code> внутри: без него <code>wrapper.__name__</code>
будет <code>"wrapper"</code>, а не именем исходной функции.
</details>

<details><summary>Подсказка 1 — retry: как считать попытки</summary>

Заведи счётчик и крути цикл <code>for attempt in range(times)</code>.
Внутри — <code>try/except</code>: при успехе сразу <code>return</code>,
при исключении — запомни его в переменную (<code>exc = e</code>).
После цикла — <code>raise exc</code>, чтобы пробросить исключение последней попытки.
</details>

<details><summary>Подсказка 2 — memoize: что использовать как ключ кэша</summary>

Ключ должен быть хешируемым. Позиционные аргументы — <code>args</code> (уже кортеж, хешируем).
Именованные — <code>kwargs</code>, это <code>dict</code>, он не хешируем.
Преврати его в <code>frozenset(kwargs.items())</code> и объедини: <code>key = (args, frozenset(kwargs.items()))</code>.
Кэш — обычный <code>dict</code>: <code>if key in cache: return cache[key]</code>,
иначе — вычисли, положи в <code>cache[key]</code>, верни.
</details>

<details><summary>Подсказка 3 — timed: как измерить время</summary>

В стандартной библиотеке есть <code>time.perf_counter()</code> — монотонные часы высокого разрешения.
До вызова функции: <code>start = time.perf_counter()</code>.
После: <code>wrapper.last_elapsed = time.perf_counter() - start</code>.
Чтобы время фиксировалось даже при исключении, оберни вызов в <code>try/finally</code>:
в блоке <code>finally</code> вычисли elapsed, в блоке <code>try</code> — вызов и <code>return</code>.
</details>
