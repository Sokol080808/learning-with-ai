# Модуль 14 — Инструменты и качество кода

Большая идея модуля: код пишут не в вакууме. Вокруг junior-разработчика стоит набор
**инструментов**, которые ловят ошибки раньше, держат стиль единым и помогают разбираться,
*почему* код ведёт себя не так. В этом модуле мы не учим новую конструкцию языка — мы учим
**ремесло**: изолировать окружение, форматировать и проверять код, ходить по нему отладчиком и
сохранять историю в git. И сразу применяем главное умение инженера — **находить и чинить баги**.

Этот модуль особенный. В файле `buggy.py` функции уже написаны и **запускаются** — никаких
`NotImplementedError`. Но в каждой спрятан баг, и из-за этого тесты красные. Твоя задача — не
писать с нуля, а **починить**. Это ровно то, чем junior занимается в первый месяц на работе:
читает чужой код, воспроизводит проблему, находит строку-виновницу, правит.

## Идея 1. `venv` и `pip` — изоляция и `requirements`

Разные проекты хотят разные версии библиотек. Если ставить всё «глобально», рано или поздно
проект A потребует `requests==2.0`, а проект B — `requests==3.0`, и они подерутся. Решение —
**виртуальное окружение**: отдельная песочница с собственным набором пакетов под каждый проект.

```bash
python3 -m venv .venv          # создать окружение в папке .venv
source .venv/bin/activate      # «войти» в него (Windows: .venv\Scripts\activate)
pip install pytest             # ставится ТОЛЬКО внутрь .venv, не в систему
pip freeze > requirements.txt  # записать точные версии в файл
deactivate                     # выйти обратно
```

ПОЧЕМУ это важно: `requirements.txt` — это список зависимостей проекта с версиями. Любой другой
человек (или ты на новой машине) делает `python3 -m venv .venv` и `pip install -r requirements.txt`
— и получает **тот же** набор пакетов. Воспроизводимость окружения это половина успеха: «у меня
работает, а у тебя нет» почти всегда про разные версии.

> Кстати, скрипт `./python/run.sh` этого курса сам создаёт `.venv` и ставит туда `pytest` —
> ты уже пользуешься виртуальным окружением, просто оно спрятано за одной командой.

## Идея 2. Форматтер `black` и линтер `ruff`

Два разных инструмента про «чистоту», их легко перепутать:

- **Форматтер** (`black`) — переписывает **внешний вид** кода по единому стилю: отступы, кавычки,
  переносы, пробелы. Он не думает про смысл, он причёсывает. Запускается и просто меняет файлы.
- **Линтер** (`ruff`) — **проверяет** код и сообщает о подозрительном: неиспользованный импорт,
  переменная объявлена и не используется, сравнение с `None` через `==` вместо `is`, и так далее.
  Он не меняет смысл — он предупреждает (и часть мелочей умеет чинить сам).

```bash
black buggy.py        # переформатировать файл по стандарту
ruff check buggy.py   # показать предупреждения (warnings/ошибки стиля и логики)
ruff check --fix .    # починить то, что ruff умеет чинить автоматически
```

ПОЧЕМУ это важно: единый стиль убирает споры «где ставить пробел» из код-ревью — за это отвечает
`black`, и обсуждать нечего. А `ruff` ловит целый класс ошибок **до запуска**: например,
неиспользованный импорт часто значит, что ты что-то не дописал, а `== None` — почти всегда баг.
Линтер — это бесплатная пара дополнительных глаз, которая смотрит на код раньше тестов.

> `black` и `ruff` не входят в стандартную библиотеку — их ставят через `pip` (см. Идею 1) и
> обычно прописывают в `requirements.txt`. Запоминалка: **black** делает код *одинаковым*,
> **ruff** делает код *честным*.

## Идея 3. Отладчик `pdb`: `breakpoint()` и команды `n/s/p/c`

`print` — нормальный способ заглянуть в код, но он быстро превращается в свалку. **Отладчик**
позволяет **остановить** программу в нужной точке и ходить по ней пошагово, разглядывая значения.
В Python встроенная команда `breakpoint()` ставит точку останова прямо в коде:

```python
def average(xs):
    breakpoint()          # тут выполнение остановится, откроется интерактивный pdb
    return (sum(xs) + 1) / len(xs)
```

Запускаешь как обычно — на строке `breakpoint()` появится приглашение `(Pdb)`. Базовые команды:

| команда | что делает |
|---|---|
| `n` (next) | выполнить текущую строку и встать на следующую (НЕ заходя внутрь вызовов) |
| `s` (step) | то же, но **зайти внутрь** вызванной функции |
| `p выражение` (print) | напечатать значение: `p xs`, `p sum(xs)`, `p len(xs)` |
| `c` (continue) | продолжить выполнение до конца (или до следующего `breakpoint`) |
| `l` (list) | показать код вокруг текущей строки |
| `q` (quit) | выйти из отладчика |

ПОЧЕМУ это важно: баг — это всегда расхождение между «что я думаю происходит» и «что происходит
на самом деле». `p` показывает реальные значения в реальный момент, и расхождение мгновенно
становится видно. Для багов этого модуля приём один и тот же: поставь `breakpoint()` перед
`return`, прогони падающий пример, и через `p` сравни то, что функция собирается вернуть, с тем,
что ждёт тест.

> Не забудь убрать `breakpoint()` перед тем, как считать задачу решённой — иначе тесты «зависнут»
> на приглашении `(Pdb)`. Кстати, `ruff` умеет ругаться на забытый `breakpoint()` в коде.

## Идея 4. `git` для junior: `status/add/commit/log/diff/branch`

`git` — система контроля версий: он хранит **историю** твоего кода снимками («коммитами»), чтобы
можно было откатиться, сравнить и понять, *что и когда* изменилось. Минимальный набор команд,
которого junior хватает на первые месяцы:

```bash
git status            # что изменено / что готово к коммиту — твоя «карта местности»
git diff              # ПОСТРОЧНО показать, что именно изменилось (ещё не закоммиченное)
git add buggy.py      # отметить файл к следующему коммиту (положить в «корзину»)
git commit -m "fix: почини off-by-one в sum_first_n"   # сохранить снимок с описанием
git log --oneline     # история коммитов кратко, по одной строке на коммит
git branch fix-bugs   # создать ветку (отдельную линию истории под задачу)
```

ПОЧЕМУ это важно: история даёт **бесстрашие**. Можно смело менять код — если сломал, `git diff`
покажет что, а откат вернёт рабочую версию. Хороший рабочий ритм: перед правкой — `git status`
(понять, где стоишь), после правки — `git diff` (перечитать собственные изменения, часто баг
видно прямо тут), затем `git add` + `git commit` с осмысленным сообщением. **Ветки** позволяют
вести задачу в стороне от основной линии и влить её, когда всё готово и тесты зелёные.

> Сообщение коммита — это записка будущему себе и команде. «fix» или «изменения» бесполезны;
> «fix: average больше не прибавляет лишнюю 1» — сразу понятно, что и зачем.

## Задания

Файл `buggy.py`. Все функции **уже написаны и запускаются**, но в каждой ровно один баг, из-за
которого падает её тест. Почини так, чтобы тесты позеленели. **Сигнатуры менять нельзя.**

1. `sum_first_n(xs: list[int], n: int) -> int` — сумма **первых n** элементов списка.
   Контракт: `sum_first_n([10, 20, 30, 40], 2) == 30`. При `n == 0` — `0`. *(подвох: off-by-one)*

2. `max_in(xs: list[int]) -> int` — **максимум** непустого списка. Обязан работать и когда **все**
   числа отрицательные: `max_in([-5, -2, -9]) == -2`. *(подвох: неверная инициализация)*

3. `average(xs: list[int]) -> float` — **среднее** непустого списка как `float`.
   Контракт: `average([2, 4]) == 3.0`. *(подвох: лишняя единица в сумме / делитель)*

4. `has_duplicate(xs: list[int]) -> bool` — есть ли в списке **повтор** значения.
   `has_duplicate([1, 2, 3]) is False`, `has_duplicate([1, 2, 2]) is True`, на `[5]` — `False`.
   *(подвох: элемент сравнивается сам с собой)*

5. `repeat(s: str, times: int) -> str` — строка `s`, повторённая **ровно** `times` раз.
   `repeat("ab", 3) == "ababab"`, `repeat("ab", 0) == ""`. *(подвох: `times - 1`)*

<details><summary>Общий метод — как чинить баг (применимо ко всем пяти)</summary>
Запусти <code>./python/run.sh 14</code> и прочитай <b>первый</b> упавший тест: pytest печатает,
что было получено и что ожидалось. Открой соответствующую функцию, поставь перед <code>return</code>
строку <code>breakpoint()</code> и прогони падающий пример руками (или повтори вызов в Python).
В приглашении <code>(Pdb)</code> через <code>p</code> посмотри ключевые значения — и сравни с тем,
что ждёт тест. Расхождение укажет на строку. Почини, убери <code>breakpoint()</code>, перезапусти.
</details>
<details><summary>Задание 1 — sum_first_n</summary>
«Первые n элементов» — это срез <code>xs[:n]</code> (от начала до индекса n, не включая его).
Сколько элементов в <code>xs[:n]</code>? А сколько в <code>xs[:n-1]</code>? Подставь <code>n = 2</code>
и сравни. Граница, на которой чаще всего ловят off-by-one, — конец среза.
</details>
<details><summary>Задание 2 — max_in</summary>
С чего стартует «текущий максимум»? Если начать с <code>0</code>, то для списка из одних
отрицательных чисел ноль так и останется «лучшим» — а его в списке нет. Чем безопаснее
инициализировать: каким-то <b>реальным</b> элементом списка, например самым первым (<code>xs[0]</code>).
</details>
<details><summary>Задание 3 — average</summary>
Среднее — это <code>сумма / количество</code>. Перечитай числитель: не прибавляется ли там что-то
лишнее к <code>sum(xs)</code>? Проверь на <code>[0, 0, 0]</code>: каким <b>обязано</b> быть среднее
и что выдаёт текущий код?
</details>
<details><summary>Задание 4 — has_duplicate</summary>
Дубликат — это когда <b>два РАЗНЫХ</b> элемента равны. Посмотри на индексы внутреннего цикла: с
какого значения <code>j</code> он стартует? Если <code>j</code> может совпасть с <code>i</code>,
элемент сравнивается сам с собой и всегда «равен» — это ложное срабатывание. Пусть внутренний
индекс идёт <b>строго после</b> внешнего. (Есть и более короткий путь — через <code>set</code>:
если длина множества меньше длины списка, значит был повтор.)
</details>
<details><summary>Задание 5 — repeat</summary>
В Python <code>"ab" * 3</code> уже даёт <code>"ababab"</code>. Сколько раз сейчас умножается
строка? Подставь <code>times = 3</code> в текущее выражение и посчитай длину результата — сходится
ли с ожидаемой?
</details>
