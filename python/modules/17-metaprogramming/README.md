# Модуль 17 — Метапрограммирование: атрибуты, дескрипторы, хуки классов

До сих пор ты писал классы «вручную»: поля, методы, наследование. Теперь — про то,
как **классы и доступ к атрибутам можно настраивать программно**. Большая идея модуля:
в Python *обращение к атрибуту — это не просто чтение из словаря, а вызов протокола*,
который ты можешь перехватить. На этом держатся `property`, ORM-поля, валидация,
авто-реестры плагинов и почти вся «магия» популярных библиотек.

Метапрограммирование звучит страшно, но в 90% случаев это три инструмента:
**дескрипторы**, **`__init_subclass__`** и **декораторы классов**. Метаклассы —
тяжёлая артиллерия, которую почти никогда не нужно доставать; в конце мы посмотрим на
них обзорно, чтобы ты узнавал их в чужом коде.

> Канон модуля — *Fluent Python* (2-е изд.), часть V «Metaprogramming». Главы про
> доступ к атрибутам, дескрипторы и хуки создания классов. Мы идём тем же путём:
> от перехвата атрибутов → к дескрипторам → к хукам подклассов → к метаклассам.

## Идея 1. Доступ к атрибуту — это протокол, а не просто словарь

Когда ты пишешь `obj.x`, Питон не «лезет в словарь». Он выполняет цепочку:

1. ищет дескриптор данных `x` в классе (о них — Идея 2);
2. смотрит в `obj.__dict__` (поля самого инстанса);
3. смотрит в классе и его предках (по MRO);
4. если ничего не нашёл — зовёт `obj.__getattr__("x")`, если такой метод есть;
5. если и его нет — `AttributeError`.

Три «крючка», которые ты можешь переопределить:

```python
class Spy:
    def __getattr__(self, name):        # ТОЛЬКО для отсутствующих атрибутов
        return f"<no attr {name}>"

    def __setattr__(self, name, value): # для ЛЮБОГО присваивания obj.name = value
        print(f"set {name} = {value}")
        super().__setattr__(name, value)  # обязательно делегируем — иначе ничего не запишется

    def __getattribute__(self, name):   # для ЛЮБОГО чтения (опасный, см. ниже)
        return super().__getattribute__(name)
```

**`__getattr__` vs `__getattribute__` — ключевое различие.**

- `__getattr__(self, name)` зовётся **только когда обычный поиск провалился**. Это
  «запасной выход»: его удобно использовать для ленивых/дефолтных атрибутов. Дёшево и
  безопасно — обычные поля сюда даже не доходят.
- `__getattribute__(self, name)` зовётся **на каждое** чтение атрибута, включая методы и
  служебные поля. Это «главный вход». Мощно, но опасно.

> Подвох (бесконечная рекурсия). Внутри `__getattribute__` нельзя писать `self.foo` или
> `self.__dict__[...]` напрямую — это снова вызовет `__getattribute__`, и так до
> переполнения стека. Всегда делегируй базовому: `object.__getattribute__(self, name)`
> (или `super().__getattribute__(name)`). Та же ловушка в `__setattr__`: внутри него
> `self.x = ...` снова дёрнет `__setattr__`. Пиши `super().__setattr__(name, value)` или
> `self.__dict__[name] = value`.

> Подвох (`__getattr__` и приватные имена). Если внутри `__getattr__` ты обращаешься к
> `self._store`, а `_store` ещё не создан (например, в момент `__init__` или при
> распаковке через `pickle`/`copy`), получишь рекурсию или маскировку реальной ошибки.
> Практика: для имён, начинающихся с `_`, в `__getattr__` поднимай `AttributeError(name)`
> — пусть служебный доступ идёт обычным путём.

## Идея 2. Дескрипторы — переиспользуемая логика доступа к полю

Дескриптор — это **объект, который, будучи атрибутом КЛАССА, перехватывает доступ к
одноимённому атрибуту ИНСТАНСА**. Он реализует один или несколько методов протокола:

```python
class Descriptor:
    def __get__(self, instance, owner): ...     # чтение obj.attr
    def __set__(self, instance, value): ...     # запись  obj.attr = value
    def __set_name__(self, owner, name): ...     # Питон зовёт при создании класса-владельца
```

Зачем это нужно? Чтобы **один раз** написать логику «как читать/писать поле» (валидация,
ленивое вычисление, логирование, преобразование) и **переиспользовать** её на множестве
полей и классов — без копипасты `@property` для каждого.

```python
class Account:
    balance = Positive()    # дескриптор как атрибут класса
    limit = Positive()

a = Account()
a.balance = 100             # вызывает Positive.__set__(self, a, 100) → валидация
a.balance                   # вызывает Positive.__get__(self, a, Account)
a.balance = -5              # ValueError: must be > 0
```

**`__set_name__` — автоимя.** Питон вызывает `__set_name__(owner, name)` автоматически в
момент создания класса `Account` и сообщает дескриптору, под каким именем тот лежит
(`"balance"`, `"limit"`). Поэтому дескриптор знает своё имя без ручного дублирования
`Positive("balance")` — пишешь просто `Positive()`. До Python 3.6 имя приходилось
передавать руками; `__set_name__` (PEP 487) убрал эту боль.

### Data-дескриптор vs non-data дескриптор — и почему это важно

| Что реализует | Тип | Приоритет |
|---|---|---|
| `__get__` **и** (`__set__` или `__delete__`) | **data descriptor** | **ВЫШЕ** `instance.__dict__` |
| только `__get__` | **non-data descriptor** | **НИЖЕ** `instance.__dict__` |

Это решает, кто «победит» при конфликте имён между дескриптором (в классе) и полем (в
`instance.__dict__`):

- **data-дескриптор** перехватывает доступ всегда, даже если в `instance.__dict__` есть
  одноимённый ключ. Поэтому валидация надёжна: пользователь не «перезапишет» поле в обход
  `__set__`. `property` — это data-дескриптор.
- **non-data дескриптор** (только `__get__`) **уступает** полю инстанса. Именно так
  работают обычные методы и `functools.cached_property`: первое обращение вычисляет
  значение и кладёт его прямо в `instance.__dict__` — дальше дескриптор больше не зовётся
  (поле его «затеняет»).

> **`property` — это просто дескриптор.** `@property` создаёт объект с `__get__`/`__set__`/
> `__delete__` — то есть data-дескриптор, привязанный к одному классу. Дескриптор, который
> ты пишешь сам, — это «многоразовый property», который можно навесить на любое количество
> полей и классов.

### Где дескриптор хранит значение — главная ловушка

Сам дескриптор **один на класс** (это атрибут класса!). Если хранить значение в
`self` дескриптора, все инстансы будут делить одно поле:

```python
class Bad:
    def __set__(self, instance, value):
        self.value = value      # ❌ ОДНО на всех — Account() и Account() перетрут друг друга
```

Правильно — хранить **per-instance**, в `instance.__dict__`:

```python
def __set__(self, instance, value):
    instance.__dict__[self.private_name] = value   # ✅ у каждого инстанса своё
```

> Подвох (имя ключа). Если хранить под тем же именем `instance.__dict__[self.name]`, а
> дескриптор — data-дескриптор, то всё корректно (data-дескриптор всё равно главнее
> словаря при доступе). Но во многих реализациях для ясности берут «приватный» ключ
> `f"_{name}"`, чтобы случайно не пересечься. Так делает и наш `Field`.

## Идея 3. `__init_subclass__` — хук при создании ПОДКЛАССА

`__init_subclass__` — это classmethod-хук базового класса, который Питон вызывает
**автоматически каждый раз, когда определяют его подкласс** (PEP 487). Идеально для:

- **регистрации** (авто-реестр плагинов/команд/сериализаторов без ручного списка);
- **валидации** подкласса (проверить, что он задал нужные атрибуты);
- передачи параметров через заголовок класса.

```python
class Plugin:
    registry: dict[str, type] = {}

    def __init_subclass__(cls, /, name=None, **kwargs):
        super().__init_subclass__(**kwargs)        # всегда зови super — кооперативность
        key = name or cls.__name__
        cls.name = key
        Plugin.registry[key] = cls                 # авто-регистрация

class Json(Plugin, name="json"):   # параметр name летит прямо в __init_subclass__
    ...

Plugin.registry            # {'json': <class Json>}
```

**Почему `name="json"` в заголовке класса работает?** Ключевые аргументы в списке баз
(`class Json(Plugin, name="json")`) Питон передаёт в `__init_subclass__` как именованные
параметры. Это легальный канал «параметризовать создание класса» без метакласса.

> Подвох. `__init_subclass__` неявно становится `classmethod` (его не надо помечать
> декоратором). Первый параметр — `cls` (создаваемый подкласс), **не** инстанс. И всегда
> вызывай `super().__init_subclass__(**kwargs)`, иначе сломаешь кооперацию, если в
> иерархии несколько таких хуков.

## Идея 4. Декораторы классов — «доработать класс после создания»

Декоратор класса — это функция `type -> type`: получает готовый класс и возвращает его
(обычно тот же, доработанный). `@dataclass` — самый известный пример: он смотрит на
аннотации и **дописывает** в класс `__init__`, `__repr__`, `__eq__`.

```python
def auto_repr(cls):
    fields = [n for n in cls.__init__.__annotations__ if n != "return"]
    def __repr__(self):
        body = ", ".join(f"{n}={getattr(self, n)!r}" for n in fields)
        return f"{cls.__name__}({body})"
    cls.__repr__ = __repr__
    return cls

@auto_repr
class P:
    def __init__(self, x: int, y: int) -> None:
        self.x, self.y = x, y

repr(P(1, 2))    # 'P(x=1, y=2)'
```

**Декоратор класса vs `__init_subclass__` vs метакласс — что выбрать?**

- Нужно доработать **один конкретный класс** → **декоратор класса** (просто и локально).
- Нужно что-то делать с **каждым подклассом** базового → **`__init_subclass__`**.
- Нужно вмешаться в **сам процесс построения класса** (имя, базы, тело namespace) до того,
  как класс существует → **метакласс** (Идея 5). Это редко.

## Идея 5. Метаклассы — обзорно (обычно не нужны)

Класс — это тоже объект. А раз объект, у него есть тип. Тип класса называется
**метаклассом**, и по умолчанию это `type`:

```python
type(42)        # <class 'int'>     — тип объекта
type(int)       # <class 'type'>    — тип класса (метакласс)
```

`type` — это **фабрика классов**. Эти два определения эквивалентны:

```python
class A:
    x = 1

A = type("A", (), {"x": 1})    # имя, кортеж баз, словарь-namespace
```

Свой метакласс — это подкласс `type`, который перехватывает **создание класса**:

```python
class Meta(type):
    def __new__(mcs, name, bases, namespace, **kw):
        # namespace — тело класса до того, как класс создан; можно его менять
        cls = super().__new__(mcs, name, bases, namespace)
        return cls
    def __init__(cls, name, bases, namespace, **kw):
        super().__init__(name, bases, namespace)

class B(metaclass=Meta):   # B строится метаклассом Meta, а не type
    ...
```

`__new__` метакласса создаёт объект класса (может править имя/базы/namespace **до**
рождения класса); `__init__` донастраивает уже созданный класс.

> **Когда метакласс оправдан — почти никогда.** Канон (Fluent Python, а до него — известный
> совет Тима Питерса): *«Метаклассы — это глубокая магия, о которой 99% пользователей не
> должны даже задумываться. Если ты сомневаешься, нужны ли они тебе, — они тебе не нужны».*
> Почти всё, ради чего раньше писали метаклассы (регистрация, валидация, инъекция методов),
> теперь делается проще через `__init_subclass__`, `__set_name__` и декораторы классов
> (PEP 487). Метаклассы остаются нужны разве что фреймворкам (Django ORM, enum, ABCMeta),
> и даже там их прячут. **Узнавай метаклассы в чужом коде — но в своём предпочитай
> дескрипторы / `__init_subclass__` / декораторы.**

### Разминка A — `DefaultDict.__getattr__` (ленивый дефолт)

`DefaultDict(default=...)` — объект, где обращение к **отсутствующему** атрибуту даёт
`default`, а присвоенный атрибут читается как обычно.

- `__init__(self, default=None)` — сохрани `self._default`.
- `__getattr__(self, name)` — для отсутствующих имён верни `self._default`; для имён,
  начинающихся с `_`, подними `AttributeError(name)` (служебный доступ не маскируем).

```python
d = DefaultDict(default=0)
d.x          # 0
d.x = 5
d.x          # 5
```

### Существенное задание — валидирующие дескрипторы

Семейство **data-дескрипторов**, хранящих значение per-instance, с автоименем через
`__set_name__`. Базовый `Field` + три специализации:

1. **`Field`** — базовый дескриптор:
   - `__set_name__(owner, name)` — запомни `self.name = name` и `self.private_name = f"_{name}"`.
   - `__get__(instance, owner=None)` — если `instance is None` (доступ через класс),
     верни сам дескриптор; иначе верни значение из `instance.__dict__[self.private_name]`
     (а если его там ещё нет — `AttributeError(self.name)`).
   - `__set__(instance, value)` — позови `self.validate(value)`, затем положи значение в
     `instance.__dict__[self.private_name]`.
   - `validate(value)` — в базовом классе ничего не проверяет.

2. **`Typed(expected_type)`** — переопредели `validate`: если значение не
   `isinstance(value, expected_type)` → `TypeError`. Особый случай: при `expected_type is int`
   значение типа `bool` тоже должно давать `TypeError` (`bool` — подкласс `int`).

3. **`Positive()`** — только числа `> 0`. Неверный тип (включая `bool`) → `TypeError`;
   `value <= 0` → `ValueError`.

4. **`Ranged(maximum)`** — расширяет `Positive` верхней границей: значение в `(0, maximum]`.
   Сначала те же проверки, что у `Positive`; затем `value > maximum` → `ValueError`.

Дескрипторы должны хранить значение **независимо для каждого инстанса** (per-instance).

### Вторая задача — авто-реестр через `__init_subclass__`

База **`Plugin`** с реестром потомков:

- атрибут класса `registry: dict[str, type]` (общий, на базовом классе);
- `__init_subclass__(cls, /, name=None, **kwargs)` — позови `super().__init_subclass__(**kwargs)`,
  возьми `key = name or cls.__name__`, запиши `cls.name = key`, зарегистрируй
  `Plugin.registry[key] = cls`. Если `key` уже занят → `ValueError`.
- classmethod `get(name)` — верни класс по имени (`KeyError`, если нет).

```python
class Json(Plugin, name="json"): ...
Plugin.registry["json"]      # <class Json>
Plugin.get("json")           # <class Json>
```

(без `'return'`) в виде `ИмяКласса(поле=значение, ...)`. Возвращает тот же класс.

### Подсказки

<details><summary>Разминка A — почему нет рекурсии в __getattr__</summary>

`__getattr__` зовётся **только когда обычный поиск провалился**. `self._default`
существует (положили в `__init__`), значит обычный поиск его находит и до `__getattr__`
дело не доходит — рекурсии нет. А вот если бы ты в `__init__` написал
`self._default = ...` при наличии своего `__setattr__`, было бы сложнее; здесь
`__setattr__` мы не трогаем, так что всё просто. Имена на `_` отбивай через
`raise AttributeError(name)`, чтобы случайный служебный доступ не возвращал тихо default.
</details>

<details><summary>Существенное — структура __get__/__set__</summary>

`__set_name__` — это где ты узнаёшь имя поля; сохрани и публичное `name`, и приватный
ключ `_name`. В `__set__` порядок такой: **сначала** `self.validate(value)` (пусть
кинет исключение, если значение плохое — и тогда в `__dict__` ничего не попадёт),
**потом** запись в `instance.__dict__[self.private_name]`. В `__get__` не забудь про
`instance is None` (доступ `Account.balance` через класс) — верни `self`.
</details>

<details><summary>Существенное — почему bool отдельно</summary>

`isinstance(True, int)` → `True`, потому что `bool` наследует `int`. Если ты этого не
учтёшь, `Positive()` молча примет `True` как «число 1 > 0». Канонически булевы значения в
числовых полях считают ошибкой типа. Поэтому: до проверки `isinstance(value, (int, float))`
явно отсекай `isinstance(value, bool)` через `TypeError`. То же в `Typed` при
`expected_type is int`.
</details>

<details><summary>Существенное — per-instance хранение (самая частая ошибка)</summary>

Дескриптор — атрибут **класса**, он один на все инстансы. Если положить значение в
`self.value` (в сам дескриптор), `a.balance` и `b.balance` будут делить одно поле. Храни
в `instance.__dict__[self.private_name]` — у каждого инстанса собственный словарь, поэтому
состояние независимо. Это и проверяет property-тест про независимость инстансов.
</details>

<details><summary>Вторая задача — наследование validate в Ranged</summary>

`Ranged.validate` не должен дублировать проверки типа и положительности — позови
`super().validate(value)` (это `Positive.validate`), а потом добавь только проверку
верхней границы. Так работает кооперативное наследование валидаторов.
</details>

<details><summary>Вторая задача — __init_subclass__ как classmethod</summary>

`__init_subclass__` не помечай `@classmethod` — Python делает это сам. Первый параметр —
`cls` (создаваемый подкласс). Реестр держи на базовом `Plugin` (`Plugin.registry[...]`),
а не на `cls`, иначе у каждого подкласса будет свой словарь. Зови
`super().__init_subclass__(**kwargs)` первым делом.
</details>

<details><summary>Разминка B — где взять имена полей</summary>

Аннотации `__init__` лежат в `cls.__init__.__annotations__` — это dict, где есть и
параметры, и (возможно) `'return'`. Отфильтруй `'return'`. Значения бери через
`getattr(self, name)`. `__repr__` — обычная функция, которую ты присваиваешь
`cls.__repr__ = ...` и возвращаешь `cls`.
</details>

Реализуешь — покажи код. Особенно интересно обсудить: почему `Positive` обязан быть
**data-дескриптором** (с `__set__`), чтобы валидацию нельзя было обойти, и что было бы,
если бы он реализовывал только `__get__`.
