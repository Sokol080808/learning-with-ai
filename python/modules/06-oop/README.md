# Модуль 06 — ООП: классы, dunder-методы, свойства

В модуле 05 ты увидел, что функции — это объекты, и научился навешивать на них поведение.
Теперь сделаем следующий шаг: будем создавать **собственные типы данных**. Класс — это
способ сказать интерпретатору «вот новый тип; так он выглядит, так себя ведёт, так
сравнивается».

Большая идея модуля: *данные и поведение, которое над ними работает, живут вместе.*
Вместо «словарь с числителем и знаменателем + куча отдельных функций над ним» ты опишешь
**один тип `Fraction`**, который сам знает, как складываться, умножаться и печататься.
А «магические» (dunder) методы — это то, как Python подключает твой тип к своему синтаксису:
`==`, `repr()`, доступ к атрибуту через точку.

## Идея 1. `class`, `__init__`, `self`, атрибуты и методы

Класс — это чертёж. По нему создают **экземпляры** (объекты). Метод `__init__` —
*конструктор*: он вызывается в момент создания и заполняет объект данными.

```python
class Point:
    def __init__(self, x: int, y: int) -> None:
        self.x = x          # атрибут экземпляра: «прибиваем» данные к self
        self.y = y

    def manhattan(self) -> int:    # метод: функция, привязанная к объекту
        return abs(self.x) + abs(self.y)

p = Point(3, -4)     # Python сам вызовет __init__(p, 3, -4)
p.x                  # 3       — читаем атрибут через точку
p.manhattan()        # 7       — self подставляется автоматически
```

**Почему `self`?** Когда ты пишешь `p.manhattan()`, Python превращает это в
`Point.manhattan(p)` — то есть передаёт сам объект первым аргументом. Имя `self` —
лишь соглашение, но нарушать его не стоит: его ждут все, кто читает твой код. `self` —
это «вот этот конкретный экземпляр», к которому ты обращаешься изнутри методов.

Важно различать: `self.x = x` создаёт атрибут *экземпляра* (у каждого Point свой `x`).
Это и есть состояние объекта.

## Идея 2. `__repr__` и `__eq__` — представление и сравнение

По умолчанию `print(obj)` выдаёт что-то вроде `<__main__.Point object at 0x10a...>` —
бесполезно. А `obj1 == obj2` по умолчанию сравнивает **по идентичности** (это один и тот
же объект в памяти?), а не по содержимому. Оба поведения чинятся dunder-методами.

```python
class Point:
    def __init__(self, x: int, y: int) -> None:
        self.x, self.y = x, y

    def __repr__(self) -> str:
        return f"Point({self.x}, {self.y})"   # как объект выглядит в отладке/REPL

    def __eq__(self, other: object) -> bool:
        if not isinstance(other, Point):
            return NotImplemented            # «я не знаю, как сравнивать с этим типом»
        return (self.x, self.y) == (other.x, other.y)

Point(1, 2)                 # в REPL покажет: Point(1, 2)
Point(1, 2) == Point(1, 2)  # True — теперь по содержимому, а не по адресу
```

**Почему `__repr__` важен.** Это «честное» текстовое представление объекта, в первую
очередь для тебя-разработчика: его показывает REPL, отладчик и `pytest`, когда тест
падает. Хорошее правило: `repr` должен по возможности выглядеть так, чтобы по нему было
понятно, что за объект и с какими данными.

**Почему `NotImplemented`, а не `False`.** Если ты вернёшь `NotImplemented` (не путать с
исключением `NotImplementedError`!), Python даст шанс правому операнду сравнить себя с
тобой, и только если и тот спасует — решит, что объекты не равны. Вернёшь жёсткое `False` —
сломаешь это вежливое поведение. Поэтому при «чужом» типе возвращают `NotImplemented`.

## Идея 3. `@property` — вычисляемый атрибут

Иногда значение не нужно хранить — его можно **вычислить** из других полей. Но заставлять
писать `obj.area()` со скобками некрасиво, если по смыслу это «атрибут». `@property`
позволяет обращаться к методу как к полю, без скобок:

```python
class Circle:
    def __init__(self, r: float) -> None:
        self.r = r

    @property
    def area(self) -> float:          # объявляем как метод…
        return 3.14159 * self.r ** 2

c = Circle(2)
c.area        # 12.566...  — вызываем БЕЗ скобок, как обычный атрибут
```

**Почему это удобно.** Снаружи `c.area` выглядит как простое поле — пользователю класса не
важно, хранится оно или считается на лету. А ты внутри волен сменить реализацию (хранить
→ вычислять и наоборот), не ломая чужой код. Это и есть инкапсуляция: интерфейс стабилен,
внутренности скрыты. Property без сеттера ещё и делает атрибут фактически «только для
чтения» — присвоить `c.area = 5` снаружи не выйдет.

## Идея 4. Инкапсуляция и инварианты (нормализация в `__init__`)

**Инвариант** — это свойство объекта, которое обязано выполняться всегда, на протяжении
всей его жизни. Лучшее место его установить — конструктор: на входе пользователь даёт
«сырые» данные, а `__init__` приводит их к **каноничной форме**.

```python
class Money:
    def __init__(self, cents: int) -> None:
        if cents < 0:
            raise ValueError("отрицательная сумма недопустима")  # отсекаем мусор сразу
        self.cents = cents
```

Для дроби `2/4` и `1/2` — это «одно и то же» число. Если хранить их как есть, то
`Fraction(2, 4) == Fraction(1, 2)` окажется ложью, арифметика начнёт плодить
несокращённые дроби, а знак может оказаться где попало (`1/-2` против `-1/2`). Решение —
**нормализовать в конструкторе**: сократить на наибольший общий делитель (НОД) и держать
знак только в числителе. Тогда у одного и того же числа всегда одно представление, и
сравнение/печать становятся тривиальными.

```python
from math import gcd
gcd(8, 12)   # 4 — на это можно сократить 8/12 → 2/3
```

**Почему проверки в `__init__` — это хорошо.** Знаменатель `0` — это не дробь, а ошибка.
Поймать её в момент создания (бросив `ValueError`) гораздо лучше, чем получить загадочный
сбой позже при делении. Объект, который удалось создать, всегда валиден — на это можно
опираться во всех остальных методах.

<details><summary>Подсказка 1 — нормализация в __init__</summary>
Сначала отсекай ошибку: <code>if den == 0: raise ValueError(...)</code>. Затем приведи
знак: если знаменатель отрицательный, домножь и числитель, и знаменатель на <code>-1</code>

</details>

<details><summary>Подсказка 2 — какой НОД у нуля</summary>
<code>Fraction(0, 5)</code> должна нормализоваться в <code>0/1</code>. В Python
<code>gcd(0, 5) == 5</code>, так что <code>0/5 → 0/1</code> получается само собой —
отдельный случай для нуля писать не нужно. Но <code>gcd</code> может вернуть отрицательное?
Нет: в стандартной библиотеке он всегда неотрицателен. Главное — сначала разберись со
знаком, потом сокращай.
</details>

<details><summary>Подсказка 3 — add и mul возвращают новый объект</summary>
Не меняй <code>self</code> и <code>other</code> — дроби должны вести себя как числа
(неизменяемо). Посчитай числитель и знаменатель результата по формуле и верни
<code>Fraction(новый_num, новый_den)</code>. Нормализацию делать руками не надо — её сделает
сам конструктор новой дроби.
</details>

<details><summary>Подсказка 4 — __eq__ и сравнение с «чужим»</summary>
Раз обе дроби уже нормализованы, равенство — это просто совпадение пар
<code>(self.num, self.den) == (other.num, other.den)</code>. Но сначала проверь
<code>isinstance(other, Fraction)</code>; если это не дробь — верни <code>NotImplemented</code>
(не <code>False</code> и не исключение).
</details>

<details><summary>Подсказка 5 — repr и value</summary>
<code>__repr__</code> — это f-строка <code>f"{self.num}/{self.den}"</code>. Свойство
<code>value</code> помечается декоратором <code>@property</code> над методом, а внутри —
<code>return self.num / self.den</code> (деление <code>/</code> даёт <code>float</code>).
Обращаться будешь как <code>frac.value</code>, без скобок.
</details>
