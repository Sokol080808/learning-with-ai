# Модуль 06 — ООП: классы, dunder-методы, свойства

В модуле 05 ты увидел, что функции — это объекты, и научился навешивать на них поведение.
Теперь сделаем следующий шаг: будем создавать **собственные типы данных**. Класс — это
способ сказать интерпретатору «вот новый тип; так он выглядит, так себя ведёт, так
сравнивается».

Большая идея модуля: *данные и поведение, которое над ними работает, живут вместе.*
Вместо «словарь с числителем и знаменателем + куча отдельных функций над ним» ты опишешь
**один тип `Fraction`**, который сам знает, как складываться, умножаться и печататься.
А «магические» (dunder) методы — это то, как Python подключает твой тип к своему синтаксису:
`==`, `repr()`, доступ к атрибуту через точку.

## Идея 1. `class`, `__init__`, `self`, атрибуты и методы

Класс — это чертёж. По нему создают **экземпляры** (объекты). Метод `__init__` —
*конструктор*: он вызывается в момент создания и заполняет объект данными.

```python
class Point:
    def __init__(self, x: int, y: int) -> None:
        self.x = x          # атрибут экземпляра: «прибиваем» данные к self
        self.y = y

    def manhattan(self) -> int:    # метод: функция, привязанная к объекту
        return abs(self.x) + abs(self.y)

p = Point(3, -4)     # Python сам вызовет __init__(p, 3, -4)
p.x                  # 3       — читаем атрибут через точку
p.manhattan()        # 7       — self подставляется автоматически
```

**Почему `self`?** Когда ты пишешь `p.manhattan()`, Python превращает это в
`Point.manhattan(p)` — то есть передаёт сам объект первым аргументом. Имя `self` —
лишь соглашение, но нарушать его не стоит: его ждут все, кто читает твой код. `self` —
это «вот этот конкретный экземпляр», к которому ты обращаешься изнутри методов.

Важно различать: `self.x = x` создаёт атрибут *экземпляра* (у каждого Point свой `x`).
Это и есть состояние объекта.

## Идея 2. `__repr__` и `__eq__` — представление и сравнение

По умолчанию `print(obj)` выдаёт что-то вроде `<__main__.Point object at 0x10a...>` —
бесполезно. А `obj1 == obj2` по умолчанию сравнивает **по идентичности** (это один и тот
же объект в памяти?), а не по содержимому. Оба поведения чинятся dunder-методами.

```python
class Point:
    def __init__(self, x: int, y: int) -> None:
        self.x, self.y = x, y

    def __repr__(self) -> str:
        return f"Point({self.x}, {self.y})"   # как объект выглядит в отладке/REPL

    def __eq__(self, other: object) -> bool:
        if not isinstance(other, Point):
            return NotImplemented            # «я не знаю, как сравнивать с этим типом»
        return (self.x, self.y) == (other.x, other.y)

Point(1, 2)                 # в REPL покажет: Point(1, 2)
Point(1, 2) == Point(1, 2)  # True — теперь по содержимому, а не по адресу
```

**Почему `__repr__` важен.** Это «честное» текстовое представление объекта, в первую
очередь для тебя-разработчика: его показывает REPL, отладчик и `pytest`, когда тест
падает. Хорошее правило: `repr` должен по возможности выглядеть так, чтобы по нему было
понятно, что за объект и с какими данными.

**Почему `NotImplemented`, а не `False`.** Если ты вернёшь `NotImplemented` (не путать с
исключением `NotImplementedError`!), Python даст шанс правому операнду сравнить себя с
тобой, и только если и тот спасует — решит, что объекты не равны. Вернёшь жёсткое `False` —
сломаешь это вежливое поведение. Поэтому при «чужом» типе возвращают `NotImplemented`.

## Идея 3. `@property` — вычисляемый атрибут

Иногда значение не нужно хранить — его можно **вычислить** из других полей. Но заставлять
писать `obj.area()` со скобками некрасиво, если по смыслу это «атрибут». `@property`
позволяет обращаться к методу как к полю, без скобок:

```python
class Circle:
    def __init__(self, r: float) -> None:
        self.r = r

    @property
    def area(self) -> float:          # объявляем как метод…
        return 3.14159 * self.r ** 2

c = Circle(2)
c.area        # 12.566...  — вызываем БЕЗ скобок, как обычный атрибут
```

**Почему это удобно.** Снаружи `c.area` выглядит как простое поле — пользователю класса не
важно, хранится оно или считается на лету. А ты внутри волен сменить реализацию (хранить
→ вычислять и наоборот), не ломая чужой код. Это и есть инкапсуляция: интерфейс стабилен,
внутренности скрыты. Property без сеттера ещё и делает атрибут фактически «только для
чтения» — присвоить `c.area = 5` снаружи не выйдет.

## Идея 3б. `__str__` vs `__repr__` — для пользователя и для разработчика

Python различает два вида строкового представления объекта:

| Метод | Когда вызывается | Для кого |
|-------|-----------------|----------|
| `__repr__` | `repr(obj)`, REPL, pytest при падении теста | Разработчик |
| `__str__` | `str(obj)`, `print(obj)`, f-строки | Конечный пользователь |

```python
class Fraction:
    def __repr__(self) -> str:
        return f"Fraction({self.num}, {self.den})"   # понятно разработчику: что за тип и данные

    def __str__(self) -> str:
        return f"{self.num}/{self.den}"               # понятно пользователю: просто дробь

f = Fraction(1, 2)
repr(f)   # "Fraction(1, 2)" — технический вид
str(f)    # "1/2"            — читаемый вид
print(f)  # "1/2"            — print использует __str__
```

**Правило fallback.** Если `__str__` не определён, Python использует `__repr__` как запасной
вариант. Но обратного нет: если не определён `__repr__`, `str(obj)` всё равно вернёт
`<ClassName object at 0x...>`. Поэтому **всегда начинай с `__repr__`** — он важнее и
полезнее при отладке.

**Когда разделять?** Если тип простой (число, дробь, точка), `__repr__` часто один на оба
случая. Разделяй, когда технический вид (с именем класса и всеми полями) мешает
пользователю, а «красивый» вид не воспроизводит объект однозначно.

## Идея 3в. `__hash__` и хешируемость

Определил `__eq__`? Python немедленно ставит `__hash__ = None` — объект становится
**нехешируемым**. Это не баг, а логика: если два объекта равны (`a == b`), они **обязаны**
иметь одинаковый хеш (`hash(a) == hash(b)`). Python просто страхует тебя от нарушения
этого контракта.

```python
class Bad:
    def __eq__(self, other):
        return ...
    # __hash__ не определён → Python ставит __hash__ = None

Bad()  # нехешируемый! использовать как ключ dict/set нельзя
```

Для **неизменяемого** типа (как наша `Fraction`) правильно явно вернуть хеш:

```python
class Fraction:
    def __eq__(self, other):
        ...

    def __hash__(self) -> int:
        # хешируем кортеж из нормализованных полей — равные дроби дадут одинаковый хеш
        return hash((self.num, self.den))
```

Почему `hash((self.num, self.den))`? Потому что равенство дробей определяется парой
`(num, den)` после нормализации. Делегируем хеширование кортежу — встроенному хешируемому
типу.

**Подвох.** Никогда не хеши изменяемые объекты — если объект изменился, хеш должен был бы
измениться, но словарь/множество уже положил его по старому хешу → потеряешь объект. Вот
почему `list` нехешируем, а `tuple` — хешируем.

## Идея 3г. Перегрузка операторов через dunder

Операторы `+`, `*`, `==` — это синтаксический сахар. `a + b` Python раскрывает в вызов
`type(a).__add__(a, b)`. Определив `__add__` в своём классе, ты «учишь» Python складывать
объекты твоего типа:

```python
class Fraction:
    def __add__(self, other: "Fraction") -> "Fraction":
        if not isinstance(other, Fraction):
            return NotImplemented          # не знаем, как складывать с чужим типом
        new_num = self.num * other.den + other.num * self.den
        new_den = self.den * other.den
        return Fraction(new_num, new_den)

Fraction(1, 2) + Fraction(1, 3)   # вызовет __add__ → 5/6
```

**`NotImplemented` для чужого операнда.** Когда `__add__` не знает тип `other`, верни
`NotImplemented` (не исключение!). Тогда Python попробует **рефлективный** метод правого
операнда: `other.__radd__(self)`. Только если и он вернёт `NotImplemented` — бросит
`TypeError`.

```
a + b
  → a.__add__(b)         # сначала левый операнд
  → если NotImplemented:
      b.__radd__(a)      # потом правый (рефлективный)
  → если NotImplemented: TypeError
```

**Рефлективные методы** — это двойники с префиксом `r`: `__radd__`, `__rmul__`, `__rsub__`
и т.д. Они нужны, чтобы поддержать `3 + Fraction(1, 2)`: `int.__add__(Fraction)` вернёт
`NotImplemented`, и Python попробует `Fraction.__radd__(3)`.

Аналогично умножение: `__mul__` / `__rmul__`. Таблица самых частых пар:

| Оператор | Прямой | Рефлективный |
|----------|--------|-------------|
| `+` | `__add__` | `__radd__` |
| `*` | `__mul__` | `__rmul__` |
| `-` | `__sub__` | `__rsub__` |
| `/` | `__truediv__` | `__rtruediv__` |

## Идея 3д. `@classmethod`, `@staticmethod` и соглашения о видимости

**`@classmethod`** — метод, который получает **класс** (не экземпляр) первым аргументом.
Главный сценарий — альтернативный конструктор: когда удобнее создать объект не через
`__init__`, а через другой интерфейс.

```python
class Fraction:
    @classmethod
    def from_int(cls, n: int) -> "Fraction":
        """Создать дробь из целого числа: n → n/1."""
        return cls(n, 1)

Fraction.from_int(3)   # Fraction(3, 1) → хранится как 3/1
```

Почему `cls`, а не `Fraction`? Если кто-то наследует класс, `cls` будет ссылаться на
**подкласс**. Жёсткая ссылка на `Fraction` «сломает» наследников.

**`@staticmethod`** — обычная функция, «прикреплённая» к пространству имён класса.
Не получает ни `self`, ни `cls`. Используется для утилит, логически связанных с классом,
но не нуждающихся в его состоянии.

```python
class Fraction:
    @staticmethod
    def _parse(s: str) -> tuple[int, int]:
        """Вспомогательная: парсит строку '3/4' → (3, 4)."""
        num, _, den = s.partition("/")
        return int(num), int(den)
```

**Соглашения о видимости.** Python не имеет настоящей «приватности», но есть соглашения:

| Форма | Смысл |
|-------|-------|
| `name` | публичный атрибут/метод |
| `_name` | «защищённый» — для внутреннего использования; извне можно, но не стоит |
| `__name` | name mangling: Python переименовывает в `_ClassName__name`; затрудняет случайное переопределение в подклассах |
| `__name__` | зарезервировано для dunder-методов Python |

```python
class BankAccount:
    def __init__(self, balance: int) -> None:
        self._balance = balance      # «защищённый» — не трогай снаружи
        self.__pin = 1234            # name mangling: извне будет AttributeError

acc = BankAccount(100)
acc._balance         # 100   — работает, но нарушает соглашение
acc.__pin            # AttributeError!
acc._BankAccount__pin  # 1234  — всё же можно, но явно «лезешь куда не звали»
```

## Идея 4. Инкапсуляция и инварианты (нормализация в `__init__`)

**Инвариант** — это свойство объекта, которое обязано выполняться всегда, на протяжении
всей его жизни. Лучшее место его установить — конструктор: на входе пользователь даёт
«сырые» данные, а `__init__` приводит их к **каноничной форме**.

```python
class Money:
    def __init__(self, cents: int) -> None:
        if cents < 0:
            raise ValueError("отрицательная сумма недопустима")  # отсекаем мусор сразу
        self.cents = cents
```

Для дроби `2/4` и `1/2` — это «одно и то же» число. Если хранить их как есть, то
`Fraction(2, 4) == Fraction(1, 2)` окажется ложью, арифметика начнёт плодить
несокращённые дроби, а знак может оказаться где попало (`1/-2` против `-1/2`). Решение —
**нормализовать в конструкторе**: сократить на наибольший общий делитель (НОД) и держать
знак только в числителе. Тогда у одного и того же числа всегда одно представление, и
сравнение/печать становятся тривиальными.

```python
from math import gcd
gcd(8, 12)   # 4 — на это можно сократить 8/12 → 2/3
```

**Почему проверки в `__init__` — это хорошо.** Знаменатель `0` — это не дробь, а ошибка.
Поймать её в момент создания (бросив `ValueError`) гораздо лучше, чем получить загадочный
сбой позже при делении. Объект, который удалось создать, всегда валиден — на это можно
опираться во всех остальных методах.
