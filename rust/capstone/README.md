# Капстоун — `kvstore`: key-value хранилище с REPL и файлом

Это финальный проект курса. Здесь нет пошаговых стабов «допиши тело функции» — есть
**спецификация** и **приёмочные тесты**. Ты собираешь маленькую, но настоящую программу
целиком: in-memory хранилище «ключ → значение», текстовый язык команд поверх него,
сохранение/загрузку в файл и интерактивный REPL.

Большая идея капстоуна: всё, что ты учил по кусочкам, сходится в одном месте. Владение
строками внутри `HashMap` (модули 03–05, 08), `Option`/`Result` как честные типы вместо
исключений (06, 07), методы и инкапсуляция (05), работа с файлами и оператор `?` (07),
разбор строк и итераторы (08, 11). Твоя задача — **спроектировать**, как они складываются.

Готовых решений в `src/lib.rs` нет: только сигнатуры и контракты. Тесты в
`tests/acceptance.rs` — это эталон поведения; они трогать не нужно.

```bash
# из корня воркспейса:
cargo test -p kvstore       # прогнать приёмочные тесты капстоуна
cargo run  -p kvstore       # запустить REPL (после того как реализуешь lib.rs)
./rust/run.sh               # прогнать ВЕСЬ воркспейс (все модули + капстоун)
```

## Что должно получиться: язык команд REPL

REPL читает stdin построчно, на каждую строку печатает ответ. Команды (имя — В ВЕРХНЕМ
регистре):

| Команда                 | Что делает                                   | Ответ                                  |
|-------------------------|----------------------------------------------|----------------------------------------|
| `SET <key> <value...>`  | положить/перезаписать пару                   | `OK`                                   |
| `GET <key>`             | прочитать значение                           | `значение` или `(nil)`, если нет ключа |
| `DEL <key>`             | удалить ключ                                  | `OK`, если был; `(nil)`, если не было  |
| `COUNT`                 | сколько пар в хранилище                      | число, напр. `3`                       |
| `KEYS`                  | все ключи, отсортированы                      | через пробел; пусто → `` (пустая строка) |
| `SAVE <path>`           | сохранить хранилище в файл                   | `OK` или `ERR <текст>`                 |
| `LOAD <path>`           | загрузить хранилище из файла (заменяет всё)  | `OK` или `ERR <текст>`                 |
| `QUIT`                  | выйти из REPL                                 | (обрабатывает сам REPL, не `execute`)  |

Важная тонкость: в `SET` значение — это **весь остаток строки** после ключа. То есть
`SET msg hello big world` кладёт значение `hello big world` (с пробелами). Это самая
каверзная команда для разбора — продумай её отдельно.

Любая нераспознанная строка, неизвестная команда или нехватка аргументов → ответ,
начинающийся с `ERR ` (точный текст после `ERR ` — на твой вкус, тесты его не проверяют).

### Пример сессии

```text
kvstore REPL. Команды: SET GET DEL COUNT KEYS SAVE LOAD QUIT.
SET name Alice
OK
SET city Berlin
OK
SET note has spaces inside
OK
GET name
Alice
GET note
has spaces inside
GET missing
(nil)
COUNT
3
KEYS
city name note
DEL city
OK
DEL city
(nil)
SAVE /tmp/my.db
OK
LOAD /tmp/my.db
OK
QUIT
bye
```

## Рекомендуемая архитектура

«Тонкий бинарь над толстой библиотекой» — вся логика в `src/lib.rs`, а `src/main.rs`
занят только вводом/выводом. Это уже сделано за тебя:

- **`src/main.rs`** (ГОТОВ, менять не нужно) — REPL: читает строки, зовёт `execute`,
  печатает ответ, ловит `QUIT`. Он не знает про команды ничего — просто крутит цикл.
- **`src/lib.rs`** (пишешь ТЫ):
  - `struct KvStore` — обёртка над `HashMap<String, String>` с методами `new`, `set`,
    `get`, `remove`, `len`, `is_empty`, `keys`, `save`, `load`. Поле `map` приватное —
    снаружи к нему не дотянуться, только через методы (инкапсуляция).
  - `fn execute(store, line) -> String` — единственное место, где текст команды
    превращается в действие над `store` и в строку-ответ.

Почему так: `execute` возвращает `String`, а не печатает сам. Тогда логику можно
тестировать, сравнивая строки, вообще не запуская REPL, — что и делают приёмочные тесты.
Это та же мысль, что в модуле 07: результат/ошибка — это **значение**, а не побочный эффект.

### Формат файла (save/load)

Формат — твой выбор, но `save` и `load` обязаны быть **симметричны**: что записал один,
то другой прочитает обратно в идентичное хранилище. Простой надёжный вариант — по строке
на пару, `ключ<TAB>значение\n`. Табуляция как разделитель удобна тем, что в значениях
она почти не встречается, а `line.split_once('\t')` аккуратно делит строку на ключ и
значение по первому табу (всё после — часть значения, даже если там пробелы).

Совет по детерминизму: пиши пары в отсортированном порядке ключей (`self.keys()`) — тогда
файл одинаков при одинаковом содержимом, и его приятно сравнивать через `diff`.

> Подвох на будущее (для стретч-целей): если значение само может содержать `\t` или `\n`,
> построчный формат сломается. Учебные тесты такого не присылают, но это честная причина
> позже взять экранирование или нормальный формат (JSON и т.п.).

## Майлстоуны

Двигайся снизу вверх — каждый шаг включает свою группу тестов. Запускай
`cargo test -p kvstore` после каждого и смотри, как красное становится зелёным.

### Майлстоун 1 — ядро `KvStore`  ⟶ модули 05, 06, 08
Реализуй `new`, `set`, `get`, `remove`, `len`, `is_empty`, `keys`.
- `get` возвращает `Option<String>` (клон значения), а не `Option<&String>` — подумай
  почему (подсказка: чтобы вызывающий не держал заимствование хранилища дольше, чем нужно).
- `keys` ВСЕГДА отсортирован — это инвариант интерфейса, живёт в одном месте.
- Зелёными станут тесты группы «МАЙЛСТОУН 1».

### Майлстоун 2 — разбор команд  ⟶ модули 02, 07
Реализуй `execute`: разбор строки на команду и аргументы, аккуратные `ERR ...` на
кривой ввод. Самое каверзное — `SET`, где значение это весь остаток строки. Подумай над
`splitn`/`split_once`, а не просто `split_whitespace` (иначе потеряешь пробелы в значении).
- Зелёными станут тесты группы «МАЙЛСТОУН 2».

### Майлстоун 3 — персистентность  ⟶ модули 07, 08
Реализуй `save` и `load` (и команды `SAVE`/`LOAD` внутри `execute`). Используй `?` для
проброса `io::Error`. Помни: `load` ЗАМЕНЯЕТ текущее содержимое целиком.
- Зелёными станут тесты группы «МАЙЛСТОУН 3».

### Майлстоун 4 — REPL и стретч-цели
`src/main.rs` уже готов — после майлстоунов 1–3 запусти `cargo run -p kvstore` и
поиграй вживую. Дальше — по желанию, и тут ты уже сам себе автор тестов:
- регистронезависимые команды (`set` == `SET`);
- команда `CLEAR` (очистить всё) и `EXISTS <key>` (`true`/`false`);
- автозагрузка из файла при старте и автосохранение при `QUIT`;
- экранирование `\t`/`\n` в значениях, чтобы формат файла стал по-настоящему надёжным;
- `TTL` (время жизни ключа) — серьёзный вызов на подумать про время и структуры данных.

## Как работать со мной

Это проектный, а не «допиши тело» формат — поэтому проси меня не решения, а разбор:
- **Прежде чем писать** — расскажи мне свой план разбора `SET` или формат файла; я укажу
  на грабли, пока они дешёвые.
- **Когда компилятор ругается на владение/время жизни** — присылай сообщение целиком.
  В капстоуне почти все «ошибки» будут про `&` / `&mut` / клонирование строк — это и есть
  главная тема Rust, и обычно компилятор прямо подсказывает решение. Разберём ПОЧЕМУ.
- **Когда тест красный** — покажи свою функцию и текст ассерта; будем читать контракт
  вместе, а не подгонять под зелёное наугад.
- **Перед стретч-целью** — сначала придумай, как ты её ПРОТЕСТИРУЕШЬ. Здесь ты впервые
  пишешь тесты сам; это половина дела.

## Альтернативные капстоуны

Если key-value стор не зажёг — возьми один из этих. Дух тот же: спецификация + тесты,
проектируешь сам. Скажи мне, и я соберу для него такой же каркас (стаб + приёмочные тесты).

- **`grep-lite`** — поиск подстроки/паттерна в файле. Команды/флаги: номера строк,
  `-i` (без учёта регистра), `-c` (счётчик совпадений), `-v` (инверсия). Тренирует работу
  с файлами, итераторы по строкам, разбор аргументов. Модули: 07, 08, 11.
- **Калькулятор выражений** — вычисление арифметики со скобками и приоритетом операций
  (`2 + 3 * (4 - 1)`). Тренирует разбор (токенизация → вычисление), `enum` для токенов,
  `Result` для ошибок («деление на ноль», «незакрытая скобка»). Модули: 06, 07, 11.

Удачи. Помни про главную идею Rust: когда компилятор спорит с тобой про владение — он не
мешает, он показывает дизайн. Спроектируй так, чтобы спорить было не о чем.
