# Капстоун — `minidb`: конкурентное in-memory хранилище

Это **финальный проект курса**. Он крупнее всего, что было: ты соберёшь маленькую, но
настоящую базу данных в духе Redis — с типами значений (строки, списки, хеши), текстовым
протоколом команд, сохранением на диск и — главное — **безопасной работой из многих
потоков** через `Arc<Mutex<T>>`. Здесь нет пошаговых инструкций. Есть спецификация (этот
README + документация над каждой сигнатурой в `src/`) и **приёмочные тесты**
(`tests/acceptance.rs`), которые описывают требуемое поведение. Архитектуру ты выбираешь
сам; тесты — твой эталон и твой судья.

> Состояние на старте: крейт **компилируется**, но все тела помечены `todo!()`, поэтому
> тесты **красные**. Твоя работа — реализовать тела так, чтобы каждый тест позеленел, не
> меняя ни сигнатуры в `src/`, ни вызовы в тестах.

---

## Что мы строим

`minidb` — это библиотека (`src/lib.rs`) **и** приложение-REPL (`src/main.rs`) в одном
крейте. Под одним ключом может лежать значение трёх видов:

- **строка** — `SET` / `GET`;
- **список строк** — `LPUSH` / `LRANGE` / `LLEN`;
- **хеш** (словарь поле→значение) — `HSET` / `HGET` / `HKEYS`.

Плюс общие команды: `DEL`, `KEYS`, `TYPE`, `SAVE`, `LOAD`. Ответы — в Redis-подобном
текстовом стиле (`OK`, `(nil)`, `(integer) 3`, `(error) ...`).

### Пример сессии (как это выглядит, когда всё реализовано)

```text
$ cargo run -p minidb
minidb REPL — введите команду (QUIT для выхода).
minidb> SET name alice
OK
minidb> GET name
alice
minidb> LPUSH tasks buy-milk
(integer) 1
minidb> LPUSH tasks call-bob
(integer) 2
minidb> LRANGE tasks 0 -1
call-bob buy-milk
minidb> HSET user city berlin
(integer) 1
minidb> HGET user city
berlin
minidb> TYPE tasks
list
minidb> LPUSH name oops          # name — строка, не список
(error) WRONGTYPE key holds a string, expected list
minidb> KEYS
name tasks user
minidb> SAVE /tmp/dump.minidb
OK
minidb> QUIT
bye
```

Та же логика, но из **многих потоков**, работает через `SharedDb` (обёртка над
`Arc<Mutex<Database>>`): восемь потоков параллельно делают по 500 `LPUSH` в один список —
и в итоге длина ровно `8 * 500 = 4000`, ни одна вставка не теряется. Это и есть
«бесстрашная конкурентность»: мьютекс выстраивает потоки в очередь, а компилятор через
`Send`/`Sync` доказывает, что гонки данных невозможны.

## Что капстоун задействует из курса

| Тема курса | Где работает в `minidb` |
|---|---|
| Владение и заимствование (03–05) | `&self` против `&mut self`; `String` (владеем) против `&str` (заимствуем) во всех методах |
| `enum` + `match` (06) | тип значения `Value::{Str, List, Hash}`; разбор имени команды |
| `Option` (06) | `get`, `hget`, `type_of` — «значения может не быть» |
| `Result` и `?` (07) | `lpush`/`hset` (`Err` при неверном типе); `save`/`load` (I/O-ошибки) |
| `Vec` / `String` / `HashMap` (08) | вся структура хранилища и значений |
| Обобщения и трейты (09) | границы `Send + Sync`; дженерик `print_prompt<W: Write>` в REPL |
| Умные указатели (13) | `Arc` — разделяемое владение, как `Rc`, но для потоков |
| **Потоки + `Arc<Mutex<T>>`** (новая тема капстоуна) | `SharedDb`: `handle()` клонирует `Arc`, `exec()` берёт `lock()` |

## Рекомендуемая архитектура

Слои разделены намеренно: сначала корректная логика, потом обёртка для конкурентности.
Ты можешь устроить иначе — главное, чтобы публичные сигнатуры и поведение совпали с
тестами.

```text
src/
├── value.rs    enum Value { Str, List, Hash }  + Value::type_name()
├── db.rs       struct Database (HashMap<String, Value>) — однопоточная логика
│                 set/get, lpush/lrange/llen, hset/hget/hkeys,
│                 del/keys/type_of, save/load
├── command.rs  fn execute(&mut Database, line) -> String  — текстовый протокол
├── shared.rs   struct SharedDb (Arc<Mutex<Database>>) — new/handle/exec, Send+Sync
├── lib.rs      склейка модулей + pub use (публичный API крейта)
└── main.rs     РАБОЧИЙ REPL (уже написан; опирается только на публичный API)
```

Почему именно так:
- `Database` ничего не знает о потоках — её легко тестировать в одном потоке;
- `execute` — чистая функция «строка → строка», без состояния, поэтому переиспользуется
  и в REPL, и в `SharedDb::exec`, и в тестах;
- потокобезопасность живёт ровно в одном месте (`SharedDb`) — её просто рассуждать и
  проверять.

## Майлстоуны

Иди по порядку — каждый следующий опирается на предыдущий. Прогоняй
`cargo test -p minidb`; на старте всё красное, твоя цель — зеленеть группами.

### Майлстоун 1 — типизированное хранилище (модули 06, 07, 08)
Реализуй `Value::type_name` и все методы `Database`, кроме `save`/`load`: строки,
списки (включая отрицательные индексы в `lrange`), хеши, `del`/`keys`/`type_of`. Ключевая
идея — **типизированность**: `lpush` по строке и `hset` по списку возвращают
`Err`, начинающийся с `WRONGTYPE`. Зеленеют тесты из раздела «Майлстоун 1».

### Майлстоун 2 — текстовый протокол (модули 06, 07)
Реализуй `execute`: разбор строки (`split_whitespace`), `match` по имени команды
(регистронезависимо), форматирование Redis-подобных ответов, проброс `Err(...)` базы в
`(error) ...`, обработка неизвестных команд и нехватки аргументов. После этого REPL
`cargo run -p minidb` оживает. Зеленеют тесты из раздела «Майлстоун 2».

### Майлстоун 3 — персистентность (модуль 07, I/O)
Реализуй `save`/`load` так, чтобы round-trip (`save` → `load`) точно восстанавливал
данные всех трёх типов. Формат сериализации выбираешь сам (подсказка — простой строчный
формат с табами в `db.rs`). `load` **заменяет** текущее содержимое. Зеленеют тесты из
раздела «Майлстоун 3».

### Майлстоун 4 — конкурентность (умные указатели + потоки)
Реализуй `SharedDb`: `new` (`Arc::new(Mutex::new(...))`), `handle` (`Arc::clone`), `exec`
(`lock()` → `execute`). Тест `concurrent_lpush_counts_every_operation` — главный экзамен
капстоуна: 8 потоков × 500 `LPUSH`, итог обязан быть 4000. Тест `shared_db_is_send_and_sync`
— компиляционная гарантия, что обёртка пригодна для потоков. Зеленеют тесты из раздела
«Майлстоун 4».

### Финал и стретч-цели
Когда всё зелёное, выбери 1–2 расширения (тесты для них напишешь сам):
- `RPUSH` (в конец списка) и `LPOP`/`RPOP`;
- `EXPIRE key seconds` + ленивое удаление по времени (внедри **инъекцию времени** через
  замыкание/трейт-часы, чтобы тесты остались детерминированными — без `Instant::now()`
  напрямую);
- значения с пробелами через кавычки в `execute`;
- флаг пути к файлу для REPL и авто-`LOAD` при старте;
- `RwLock` вместо `Mutex` (много читателей, один писатель) — измерь, изменилось ли что-то.

## Как работать со мной

- Идём по майлстоунам. Перед кодом — расскажи мне план словами: какие варианты `match`,
  где `&self`, а где `&mut self`, что возвращаем — `Option` или `Result` и почему.
- Когда тест красный — **не угадывай**. Прочитай его (это спецификация), покажи мне свою
  гипотезу, что именно расходится.
- Споткнулся о борроу-чекер (а на `lpush`/`hset` это вероятно — мы и читаем, и меняем
  одну запись `HashMap`) — покажи ошибку компилятора целиком, разберём, почему он прав, и
  как выразить намерение (часто помогает `entry(...).or_insert_with(...)`).
- На майлстоуне 4 будь готов объяснить мне: что именно даёт `Arc`, а что — `Mutex`;
  почему `Rc` тут не годится; что такое «отравленный мьютекс»; почему `exec(&self)`, а не
  `&mut self`.

Самопроверка перед тем, как звать меня: `cargo build -p minidb` собирается; падающие
тесты падают **с паникой `not yet implemented`** (а не с ошибкой компиляции) — значит,
сигнатуры на месте, осталась только логика.

```bash
cargo build -p minidb        # из каталога rust/ — должно собираться всегда
cargo test  -p minidb        # запустить приёмочные тесты капстоуна
./rust/run.sh                # прогнать весь воркспейс (все модули + капстоун)
cargo run   -p minidb        # запустить REPL вручную
```

## Альтернативные капстоуны (если хочется другого домена)

Те же темы курса (владение, `enum`/`match`, `Result`/`?`, коллекции, трейты, `Arc<Mutex>`,
потоки), другой сюжет:

1. **Конкурентный пул задач (thread pool + очередь job'ов).** `JobQueue` на
   `Arc<Mutex<VecDeque<Job>>>`; N воркер-потоков забирают задачи и кладут результаты в
   общий `Arc<Mutex<HashMap<JobId, Outcome>>>`. `enum JobStatus { Pending, Running, Done(T),
   Failed(String) }`. Приёмочный тест: поставили K задач, дождались `join`, у каждой статус
   `Done`, и сумма результатов сходится — проверка, что ни одна задача не потерялась и не
   выполнилась дважды.

2. **Потокобезопасный кэш с метриками (`Arc<Mutex<HashMap>>` + статистика hit/miss).**
   `Cache<K, V>` — дженерик с трейт-границами (`K: Eq + Hash + Clone`), методы
   `get_or_compute(key, f)` где `f` — замыкание-вычислитель; счётчики попаданий/промахов
   защищены тем же замком. Тест: много потоков спрашивают пересекающиеся ключи; ключ
   вычисляется ровно столько раз, сколько было уникальных промахов (а не на каждый
   `get`) — это проверяет корректность синхронизации.
