# Модуль 00 — Setup, cargo и цикл сборки

Большая идея: прежде чем учить язык, нужно научиться **жать на кнопку**. В этом
модуле ты не столько пишешь Rust, сколько настраиваешь рабочий цикл, в котором
будешь жить весь курс: правишь код → запускаешь `cargo test` → читаешь, что
сказал компилятор → правишь снова. Этот цикл — твой главный инструмент. Две
крошечные функции здесь — лишь повод его прокрутить.

Правило курса простое: **тесты знают, как должно быть**. Они уже написаны и их
трогать не нужно. Твоя задача — дописать стаб (`src/lib.rs`), где сейчас стоят
заглушки `todo!()`, и превратить красные тесты в зелёные. Не «написать код», а
**пройти тест с пониманием почему**.

## Идея 1. Rust компилируется в нативный код; cargo делает всё остальное

Rust — компилируемый язык: исходник превращается в **нативный машинный код**
(как C/C++), без виртуальной машины и сборщика мусора. Поэтому Rust быстрый, а
ещё компилятор успевает поймать кучу ошибок **до** запуска программы.

Сам компилятор называется `rustc`, но руками ты его почти никогда не вызываешь.
Всем дирижирует **cargo** — это сборщик, менеджер зависимостей и запускатель
тестов в одном флаконе:

```bash
cargo build    # собрать
cargo test     # собрать и прогнать тесты
cargo run      # собрать и запустить (для бинарных крейтов)
```

**Почему через cargo, а не `rustc` напрямую?** Потому что cargo знает структуру
проекта, тянет зависимости нужных версий, кеширует сборку и единообразно
запускает тесты. В реальном Rust-проекте `rustc` руками не зовут вообще никогда.

## Идея 2. Крейт, воркспейс и цикл red → green

**Крейт (crate)** — единица компиляции в Rust. Грубо: «одна библиотека или одна
программа». Наш модуль — это библиотечный крейт `m00_setup`: файл `src/lib.rs`
плюс описание в `Cargo.toml`:

```toml
[package]
name = "m00_setup"      # имя крейта — под ним он подключается в тестах
version = "0.1.0"
edition = "2021"        # «диалект» языка; весь курс на edition 2021
```

**Воркспейс (workspace)** — это набор связанных крейтов под одной крышей. Весь
курс — один воркспейс (`rust/Cargo.toml`), а каждый модуль — отдельный
крейт-член. Благодаря этому `cargo test` можно звать как для всего курса, так и
для одного модуля.

Файлы модуля раскладываются так:

```
modules/00-setup/
├── Cargo.toml          # паспорт крейта
├── src/lib.rs          # СТАБ — здесь пишешь ты
├── tests/warmup.rs     # готовые тесты — их не трогаем
├── tests/props.rs      # property-тесты — тоже не трогаем
└── tests/convert.rs    # тесты к заданию 3 — не трогаем
```

Тесты лежат в `tests/` и подключают наш крейт строкой `use m00_setup::*;` —
то есть «возьми всё публичное (`pub`) из крейта `m00_setup`». Поэтому функции в
`src/lib.rs` обязаны быть `pub`, иначе тест их не увидит.

**Цикл red → green** — главный ритм курса:

1. Запускаешь тесты → они **красные** (стоит `todo!()`, который паникует).
2. Дописываешь тело функции в `src/lib.rs`.
3. Запускаешь снова → **зелёные**. Готово, идёшь дальше.

`todo!()` — это удобный маркер «здесь будет код». Он позволяет файлу
**скомпилироваться** (тип возврата формально совпадает), но при запуске
**паникует** с сообщением `not yet implemented`. Идеально для стаба: сборка
проходит, а тест честно красный, пока ты не написал настоящее тело.

## Идея 2a. `pub` и граница видимости: почему без него тест не компилируется

Когда ты пишешь функцию без `pub`, она **приватная** — видна только внутри того
же файла/модуля. Тесты в папке `tests/` — это отдельные бинарные крейты: они
подключают `m00_setup` как внешнюю зависимость и видят только то, что явно
помечено `pub`. Именно поэтому каждая функция в `src/lib.rs` начинается со
слова `pub`.

```rust
fn secret(x: i64) -> i64 { x * 2 }   // приватная: тест её НЕ увидит
pub fn visible(x: i64) -> i64 { x }   // pub: тест может вызвать
```

То же самое касается **перечислений (enum)** и их **вариантов**. Мало объявить
`pub enum Unit` — если варианты не `pub`, их нельзя назвать снаружи:

```rust
pub enum Unit {
    Seconds,   // pub по умолчанию, если сам enum pub
    Minutes,
}
// внешний код: Unit::Seconds — ок
```

Правило простое: **pub перед именем = «видно снаружи крейта»**. Всё, что без
`pub`, — строго внутреннее. Если тест жалуется на «not found» или «private item»,
в первую очередь проверяй, есть ли `pub`.

## Идея 2b. Типы целых чисел: i64, i32, u64, usize — и почему важен выбор

Rust не приводит числовые типы автоматически — надо явно указывать правильный.
Вот краткая шпаргалка:

| Тип     | Знак      | Размер   | Диапазон (примерно)           |
|---------|-----------|----------|-------------------------------|
| `i32`   | знаковый  | 32 бита  | ±2.1 млрд                     |
| `i64`   | знаковый  | 64 бита  | ±9.2 × 10¹⁸                  |
| `u32`   | беззнаковый | 32 бита | 0 … 4.3 млрд                 |
| `u64`   | беззнаковый | 64 бита | 0 … 1.8 × 10¹⁹               |
| `usize` | беззнаковый | 32/64 бита* | индексы и размеры          |

\* `usize` совпадает с адресным пространством платформы.

**Зачем в курсе `i64`, а не `i32`?** Посмотри на `seconds_in(24)` = 86 400 или на
конвертер, где `Days → Seconds` может легко вырасти в миллионы. `i32` бы
переполнился при 596 часах (2 147 483 647 / 3600 ≈ 596). `i64` держит числа до
миллиарда лет в секундах — для учебного примера с запасом.

**Частая ошибка новичка** — смешать типы в выражении:

```rust
pub fn bad(hours: i64) -> i64 {
    3600 * hours as i32  // ← ОШИБКА КОМПИЛЯЦИИ:
    // нельзя умножить i64 на i32 без явного приведения
}
```

Компилятор скажет что-то вроде `expected i64, found i32`. Реши либо литералом
нужного типа (`3600_i64 * hours`), либо просто пиши `3600 * hours` — тогда
Rust выведет тип `3600` как `i64` из контекста.

**`usize` — специальный тип для индексов.** Массивы и срезы индексируются только
по `usize`. Это гарантирует, что индекс не может быть отрицательным и не может
выйти за адресное пространство.

## Идея 2c. Паника vs ошибка компиляции — принципиальная разница

Rust различает два класса «что-то пошло не так»:

**Ошибка компиляции** — компилятор отказывается создавать программу. Никакого
исполняемого файла нет вообще. Это хорошо: программа с логическим изъяном
просто не запустится и не навредит.

```rust
pub fn add(a: i64, b: i64) -> i64 {
    a + b as i32   // ← компилятор НЕ даст это собрать
}
```

**Паника** — ошибка во время выполнения. Программа успешно компилируется,
стартует, но в какой-то момент падает с сообщением и завершается. По умолчанию
Rust раскручивает стек (unwind): вызывает деструкторы всех живых значений
по цепочке, а затем завершает поток. Можно настроить режим `abort` — тогда
процесс останавливается немедленно без раскрутки (быстрее, но без cleanup).

```rust
fn main() {
    let v: Vec<i32> = vec![];
    println!("{}", v[0]); // паника: index out of bounds
    // программа сюда не доходит
}
```

`todo!()` — это тоже паника (макрос, который всегда паникует с текстом
`"not yet implemented"`). Поэтому стаб **компилируется**, но тест **красный**:
при вызове функции программа паникует и тест-раннер cargo фиксирует провал.

Когда ты заменяешь `todo!()` на настоящее тело — паники больше нет, тест
зеленеет. Простое правило: **если cargo test показывает `FAILED` с `panicked at`
— это рантаймовая паника, а не ошибка компилятора**.

## Идея 2d. Cargo.toml vs Cargo.lock — два файла, два назначения

В каждом Rust-проекте живут два файла с похожими именами, но разными ролями:

**`Cargo.toml`** — это «рецепт»: ты описываешь зависимости в виде диапазонов
версий (`"^1.0"`, `">=0.5"`). Этот файл ты редактируешь руками и
**коммитишь в git**.

```toml
[dependencies]
serde = "1.0"        # любая совместимая 1.x
rand  = ">=0.8"      # 0.8 или новее
```

**`Cargo.lock`** — это «снимок»: cargo записывает сюда точные версии всего, что
выбрал (включая транзитивные зависимости). Файл генерируется автоматически и
нужен для воспроизводимых сборок: два разных разработчика с одним `Cargo.lock`
получат побайтово одинаковый результат.

Правило из официального гайда:
- **Для бинарных крейтов (программ)** — `Cargo.lock` коммитить в git. Ты хочешь,
  чтобы CI и продакшн строились по тем же точным версиям.
- **Для библиотечных крейтов** — обычно *не* коммитить: потребители библиотеки
  сами выбирают, какие версии зависимостей использовать в своей программе.

В нашем учебном курсе — библиотечные крейты, поэтому `Cargo.lock`
игнорируется через `.gitignore`. Если у тебя нет внешних зависимостей вовсе
(как в этом модуле) — файл и не возникнет.

## Идея 2e. `cargo new` — как рождается проект с нуля

Прежде чем работать с готовым модулем, полезно знать, как проект создаётся.
Одна команда — и скелет готов:

```bash
cargo new my_project          # создать бинарный крейт (src/main.rs)
cargo new my_lib --lib        # создать библиотечный крейт (src/lib.rs)
```

**Что генерирует `cargo new`:**

```
my_lib/
├── Cargo.toml   # паспорт пакета: имя, версия, edition, зависимости
└── src/
    └── lib.rs   # точка входа библиотеки; по умолчанию — один пустой тест
```

В `lib.rs` сразу лежит заготовка с тестом:

```rust
pub fn add(left: u64, right: u64) -> u64 {
    left + right
}

#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    fn it_works() {
        let result = add(2, 2);
        assert_eq!(result, 4);
    }
}
```

`cargo new` так же инициализирует git-репозиторий (если ты не внутри уже
существующего git-дерева) — чтобы ты сразу мог коммитить. В нашем курсе
воркспейс уже существует, поэтому новые модули просто добавляются как члены
в `rust/Cargo.toml`, а `cargo new` не нужен. Но в собственных проектах ты
будешь начинать именно с него.

**ПОЧЕМУ `--lib` для курса, а не умолчание?** Библиотечный крейт не имеет
точки входа `main()` — его предназначение быть импортированным тестами или
другим кодом. Это нас устраивает: тесты в `tests/` — отдельные бинарные
крейты, которые подключают нашу библиотеку через `use m00_setup::*`.

## Идея 2f. Воркспейс, пакет, крейт — три уровня иерархии

Rust-экосистема оперирует тремя понятиями, которые легко спутать. Разберём
каждое точно:

**Крейт (crate)** — единица **компиляции**. Это то, что `rustc` компилирует
за один раз. Бывает двух видов:
- *библиотечный крейт* (`lib`) — нет `fn main`, компилируется в `.rlib` для
  импорта другими.
- *бинарный крейт* (`bin`) — содержит `fn main`, компилируется в исполняемый
  файл.

```
Один крейт = один файл-корень (src/lib.rs ИЛИ src/main.rs).
```

**Пакет (package)** — единица **сборки** и **публикации**. Это директория
с файлом `Cargo.toml`. Один пакет может содержать:
- ровно один библиотечный крейт (`src/lib.rs`),
- один или несколько бинарных крейтов (`src/main.rs`, `src/bin/*.rs`).

```
Один пакет = один Cargo.toml = один или несколько крейтов.
```

Команда `cargo new my_lib --lib` создаёт **пакет** с именем `my_lib`,
внутри которого — **один библиотечный крейт** (`src/lib.rs`). Часто слова
«пакет» и «крейт» используют как синонимы в неформальной речи — это
допустимо, но технически они разные.

**Воркспейс (workspace)** — набор связанных пакетов под одним корневым
`Cargo.toml`. Цель: единый `Cargo.lock` и общий `target/` (кеш сборки),
чтобы не пересобирать одни и те же зависимости для каждого пакета отдельно.

```
Один воркспейс = один корневой Cargo.toml + список [[members]] = много пакетов.
```

В нашем курсе:

| Уровень | Файл / директория | Пример |
|---------|-------------------|--------|
| Воркспейс | `rust/Cargo.toml` | весь курс Rust |
| Пакет | `rust/modules/00-setup/Cargo.toml` | модуль 00 |
| Крейт | `rust/modules/00-setup/src/lib.rs` | библиотека `m00_setup` |

**Pitfall**: люди часто говорят «крейт» там, где имеют в виду «пакет» —
особенно когда публикуют на `crates.io` (реестр хранит **пакеты**, но его
называют «реестром крейтов»). В документации Rust Book оба слова встречаются
почти взаимозаменяемо. Запомни ключевое: `Cargo.toml` = граница пакета;
`src/lib.rs` или `src/main.rs` = корень крейта.

## Идея 2g. Editions — версии языка без ломающих изменений

В конце `Cargo.toml` ты видишь строчку `edition = "2021"`. Что это значит
и зачем оно вообще существует?

**Что такое edition.** Rust периодически хочет ввести синтаксические или
семантические изменения, которые **технически ломают** обратную
совместимость (новое ключевое слово, изменённые правила захвата в замыканиях
и т.д.). Вместо того чтобы сломать весь существующий код, Rust вводит
**редакции (editions)**: каждая редакция — это «диалект» языка, который крейт
выбирает в своём `Cargo.toml`. Сейчас есть три: `2015`, `2018`, `2021`.

**Гарантия совместимости.** Крейты разных редакций **могут зависеть друг от
друга** без каких-либо проблем — cargo компилирует каждый крейт в его
собственной редакции. Старый крейт на `edition = "2015"` продолжает
собираться вечно, а ты пишешь новый код на `2021`. Нет «ломающего перехода»
для экосистемы в целом.

**Что принесла edition 2021.** Три основных изменения:
1. *Prelude additions* — типы `TryInto`, `TryFrom`, `array::IntoIter` добавлены
   в стандартный `prelude` (не нужно `use`).
2. *Улучшенный захват в замыканиях* — замыкание теперь захватывает только те
   поля структуры, которые реально использует, а не всю структуру целиком.
   Это снимает ряд ошибок borrow-checker-а.
3. *`or`-паттерны в `let`, `if let`, `while let`* — `if let A | B = expr`
   теперь работает без скобок.

**Pitfall.** Не путай редакцию с версией компилятора (`rustc`). `rustc`
обновляется каждые 6 недель; редакции выходят раз в несколько лет (2015,
2018, 2021). Работая с `edition = "2021"`, ты по-прежнему можешь собирать
старый код `edition = "2015"` тем же `rustc`. Для нашего курса всегда
используем `2021` — это современный идиоматичный Rust.

## Идея 3. Предпросмотр: владение (ownership) — то, чем Rust отличается от всех

Забегаю вперёд, но это слишком важно, чтобы молчать до середины курса. Главная
идея Rust — **владение (ownership)**. Оно решает проблему, которую C/C++
передают программисту вручную, а Java/Python перекладывают на сборщик мусора:
**кто и когда освобождает память?**

### Проблема, которую решает владение

В языках без сборщика мусора есть два классических источника багов:

1. **Двойное освобождение (double-free)** — `free(p)` вызывается дважды:
   неопределённое поведение, иногда уязвимость.
2. **Обращение к освобождённой памяти (use-after-free)** — указатель используется
   после того, как объект уничтожен: данные могут быть перезаписаны чем угодно.

Эти ошибки молчаливы в рантайме и порождают трудно-воспроизводимые баги.

### Как Rust решает это через владение

У каждого значения в любой момент времени есть ровно **один владелец**.
Когда владелец выходит из области видимости — значение автоматически
освобождается. Никакого `free` вручную, никакого сборщика мусора.

```rust
let s = String::from("привет"); // s — владелец строки
let t = s;                       // владение ПЕРЕЕХАЛО в t; s больше недействителен
// println!("{}", s);            // ← это НЕ скомпилируется: s уже не владеет строкой
println!("{}", t);               // ok: t — владелец
```

Компилятор отследит это статически и не даст программе запуститься с такой
ошибкой. double-free физически невозможен: освобождает только один владелец,
ровно один раз.

Из владения растут **заимствование (borrowing)** ссылками `&` и `&mut` и
**времена жизни (lifetimes)**. Звучит непривычно — и это нормально. Запомни
сейчас одно: то, что в других языках ловится в рантайме (use-after-free, гонки
данных), в Rust ловит компилятор **до запуска**. Мы будем возвращаться к этой
идее весь курс (модули 03, 04, 12). В сегодняшних функциях работаем с обычными
`i64`, которые просто **копируются**, поэтому про владение можно пока не думать —
но знай, что оно ждёт за углом и оно тут самое интересное.

## Идея 4. Как читать ошибки компилятора Rust (они подробные — это помощь)

У Rust, пожалуй, самые дружелюбные сообщения об ошибках среди всех языков. Когда
компилятор «ругается» — это не препятствие, а **подсказка**. Типичное сообщение
показывает: код ошибки, что именно не так, точное место в коде стрелочкой `^^^`
и часто — как починить (`help:`).

```text
error[E0308]: mismatched types
  --> src/lib.rs:20:5
   |
20 |     a + b as i32
   |     ^^^^^^^^^^^^ expected `i64`, found `i32`
   |
help: ...
```

Алгоритм чтения: (1) посмотри **код** ошибки (`E0308`) и одну строку описания;
(2) найди **стрелочки** — они указывают на проблемное место; (3) прочитай блок
`help:`, если он есть. Не пугайся объёма: Rust многословен нарочно, чтобы тебе
**не пришлось** гуглить. Зелёный тест и понятная ошибка — два твоих лучших
друга в этом курсе.

### Подсказки к заданиям 1–2

<details><summary>Подсказка 1 — как вообще вернуть значение из функции</summary>

В Rust последнее **выражение без точки с запятой** становится возвращаемым
значением. То есть тело может быть буквально одной строкой:

```rust
pub fn add(a: i64, b: i64) -> i64 {
    a + b
}
```

Обрати внимание: после `a + b` НЕТ `;`. Если поставить точку с запятой, строка
станет инструкцией со значением «ничего» (`()`), и компилятор скажет, что ждал
`i64`, а получил `()`. Можно написать и явно: `return a + b;` — но идиоматичнее
без `return`.
</details>

<details><summary>Подсказка 2 — seconds_in</summary>

Тут вообще не на чем спотыкаться: один час = 3600 секунд, значит ответ —
произведение `3600 * hours`. Снова: последнее выражение без `;`. Сверься с
контрактом в `///`-комментарии над функцией: `seconds_in(24)` должно дать
`86_400` (подчёркивания в числе — это просто разделители разрядов, для
читаемости; на значение они не влияют).
</details>

## Задание 3 — Конвертер единиц времени

Это первое «настоящее» задание модуля. Ты реализуешь маленький, но полноценный
конвертер: секунды ↔ минуты ↔ часы ↔ сутки. По пути освоишь два ключевых
инструмента Rust — **перечисление `enum`** и **сопоставление с образцом `match`**.

### Что нужно добавить в `src/lib.rs`

**Перечисление:**

```rust
pub enum Unit { Seconds, Minutes, Hours, Days }
```

**Функцию:**

```rust
pub fn convert(value: i64, from: Unit, to: Unit) -> i64
```

Функция переводит `value` из единицы `from` в единицу `to`.

### Зачем нужен `enum`?

Можно было бы передавать строку `"hours"` или число `2` — но тогда никто не
помешает написать `convert(5, "hoUrs", "seconds")` или `convert(5, 99, 0)`, и
ошибка выяснится только в рантайме. Вариант с `enum` делает **недопустимое
состояние непредставимым**: компилятор проверит, что ты передаёшь только `Hours`,
`Seconds` и т.д. Это один из базовых приёмов Rust-дизайна — «make illegal
states unrepresentable».

### Ключевая идея: канонический формат

Самый надёжный способ конвертировать N единиц в M единиц — **не** писать
N×M формул (Hours→Minutes, Hours→Seconds, Minutes→Days…). Вместо этого:

1. Переводим всё в **одну каноническую единицу** (обычно — самую маленькую,
   здесь это секунды).
2. Переводим из канонической в нужную.

Итого 2 шага и N + M ветвей `match` вместо N×M. Запомни этот паттерн — он
встречается повсюду: конвертеры валют, единиц измерения, систем координат.

### Как устроен `match` в Rust

`match` — это как `switch` в других языках, но строже: компилятор требует
**исчерпывающего** перечисления вариантов. Если ты забыл один вариант `enum` —
не скомпилируется. Это отличие принципиальное: ошибка не прячется до рантайма.

```rust
let unit = Unit::Hours;
let seconds_per_unit = match unit {
    Unit::Seconds => 1,
    Unit::Minutes => 60,
    Unit::Hours   => 3_600,
    Unit::Days    => 86_400,
    // если убрать любую строку — компилятор откажет: "non-exhaustive patterns"
};
```

Обрати внимание: после `=>` идёт выражение, а строки разделяются запятыми
(не точками с запятой). Весь `match` — тоже **выражение**, его результат можно
присвоить переменной или вернуть из функции.

### Подсказки к заданию 3

<details><summary>Подсказка 1 — с чего начать</summary>

Напиши сначала перечисление `Unit` с четырьмя вариантами. Добавь перед ним
`#[derive(Debug, Clone, Copy)]` — это позволит печатать значения в тестах и
копировать их (мы разберём `Copy` в модуле 03, пока воспринимай как магическую
аннотацию для простых enum).

Потом напиши заглушку функции `convert` с `todo!()` и убедись, что
`cargo build` проходит.
</details>

<details><summary>Подсказка 2 — алгоритм двух шагов</summary>

Шаг 1: match по `from`, чтобы получить количество секунд.
Шаг 2: match по `to`, чтобы поделить секунды на нужный коэффициент.

Коэффициенты: секунда = 1, минута = 60, час = 3600, сутки = 86 400.

Помни: деление целочисленное — `/`. Дробная часть отбрасывается.
Например, 90 секунд / 60 = 1 минута (не 1.5).
</details>

<details><summary>Подсказка 3 — что если from == to?</summary>

Никакого специального случая не нужно: алгоритм двух шагов сам справится.
Например, Hours → Hours: умножаем на 3600, потом делим на 3600 — получаем
исходное значение. Проверь это в голове и не дублируй код.
</details>
