# Модуль 11 — Замыкания и итераторы

Большая идея: в Rust привычные «ручные циклы» почти всегда заменяются **цепочкой
итераторов** — декларативным описанием *что* мы делаем с данными, а не *как* мы их
перебираем. Двигатель этого стиля — **замыкания**: маленькие функции, которые можно
создать на месте и которые умеют захватывать переменные из окружения. Вместе они дают
код, который читается как фраза («возьми все элементы, возведи в квадрат, сложи») и при
этом компилируется в такой же быстрый машинный код, как написанный руками `for`.

Эти две вещи — обычные значения. Замыкание можно положить в переменную, передать в
функцию аргументом и вернуть из функции наружу. Именно вокруг этого — весь модуль.

## Идея 1. Замыкания и три типажа: `Fn` / `FnMut` / `FnOnce`

Замыкание — это анонимная функция, записанная прямо в выражении:

```rust
let k = 3;
let triple = |x: i64| x * k;   // захватили k из окружения
assert_eq!(triple(10), 30);
```

Когда замыкание передают в функцию, его тип описывают одним из трёх **типажей** —
по тому, *как именно* оно обращается с захваченным окружением:

- **`Fn`** — только читает окружение (или не трогает его). Можно вызывать сколько
  угодно раз. Это самый «спокойный» вариант.
- **`FnMut`** — *меняет* захваченное окружение между вызовами (например, копит сумму
  в захваченной переменной). Тоже многократно, но требует `mut`.
- **`FnOnce`** — *забирает во владение* (перемещает) что-то из окружения, поэтому
  его корректно вызвать только **один раз**.

```rust
fn apply<F: Fn(i64) -> i64>(f: F, x: i64) -> i64 {
    f(x)   // граница Fn => можно звать, и не один раз
}
```

**ПОЧЕМУ так:** Rust не прячет от тебя, что делает функция с данными — это видно прямо
в типе. Если ты пишешь `F: Fn`, ты обещаешь компилятору и читателю «вызов безопасен и
повторяем». Если бы тебе понадобилось менять окружение — пришлось бы написать `FnMut`,
и компилятор бы это проверил. Это та же идея владения, что и везде в Rust, просто
применённая к функциям.

> Подводный камень: эти три типажа вложены — каждое `Fn` является и `FnMut`, и
> `FnOnce`. Поэтому бери **самую слабую** границу, какую позволяет твоя логика: функция
> с `F: Fn` примет больше замыканий, чем с `F: FnOnce`? — наоборот: `FnOnce` —
> самая общая граница для *приёма*, `Fn` — самая строгая. Для учебных заданий держись
> `Fn`: оно интуитивно и работает для «чистых» преобразований.

## Идея 2. Адаптеры итераторов: `map` / `filter` / `sum` / `collect` и ленивость

Итератор — это объект, который по запросу отдаёт элементы по одному (метод `next`).
Срез умеет дать итератор по ссылкам: `v.iter()` отдаёт `&i64`.

**Адаптеры** — это методы, которые из одного итератора делают другой, описывая
преобразование, но **ничего пока не считая**:

```rust
let v = [1, 2, 3, 4];
let it = v.iter()            // итератор по &i64
          .filter(|x| **x % 2 == 0)  // оставить чётные  (ленивый адаптер)
          .map(|x| x * 10);          // умножить на 10    (ленивый адаптер)
// до этой строки НИ ОДНОГО элемента ещё не обработано!
let result: Vec<i64> = it.collect();   // вот ТЕПЕРЬ всё считается
assert_eq!(result, vec![20, 40]);
```

**Ленивость** — ключевое слово. `map` и `filter` не выполняются сразу; они лишь
«накручивают» обёртки. Реальная работа начинается только когда подключается
**потребитель** (consumer): `collect()`, `sum()`, `count()`, `for`-цикл и т.п.

```rust
let s: i64 = v.iter().map(|x| x * x).sum();   // sum() — потребитель, тянет элементы
```

**ПОЧЕМУ это хорошо:**
- Цепочку читаешь сверху вниз как список операций — намерение видно сразу.
- Ленивость означает «ни одного лишнего прохода»: всё сливается в один проход по
  данным, без промежуточных векторов на каждом шаге.
- Никакого изменяемого `mut sum = 0;` и риска опечататься в индексе.

> Подводный камень №1: если построить цепочку адаптеров и **не** подключить
> потребителя, компилятор предупредит «iterator adaptor is not used» — потому что
> буквально *ничего не произошло*.
>
> Подводный камень №2: `iter()` даёт **ссылки**. В `filter` элемент — это `&&i64`
> (ссылка на `&i64`), отсюда `**x`. А `collect::<Vec<i64>>()` хочет *значения*, не
> ссылки — поэтому пригодится `.copied()` (для `Copy`-типов) или `.cloned()`.

## Идея 3. `impl Fn` в возвращаемом типе — функция, которая делает функции

Замыкание можно не только принять, но и **вернуть**. Только вот точный тип замыкания
безымянный (его придумывает компилятор), назвать его в сигнатуре нельзя. Решение —
`impl Trait` в позиции возврата:

```rust
fn make_adder(n: i64) -> impl Fn(i64) -> i64 {
    move |x| x + n   // move => замыкание ЗАБИРАЕТ n себе
}

let add5 = make_adder(5);
assert_eq!(add5(10), 15);
```

`impl Fn(i64) -> i64` читается как: «я возвращаю *какое-то конкретное* значение,
которое можно вызывать как `i64 -> i64`; имя его типа — моя внутренняя кухня».

**ПОЧЕМУ нужен `move` (а часто и нет):** возвращаемое замыкание переживёт вызов
`make_adder` и его локальные переменные. Если замыкание ссылается на `n` по ссылке —
ссылка повиснет, и компилятор это запретит. `move` заставляет замыкание **взять `n`
во владение** (для `i64` это просто копия, ведь он `Copy`), и проблема исчезает. Здесь
компилятор «ругается» не назло — он спасает тебя от висячей ссылки.

> Тонкость: для `i64` (тип `Copy`) код «прибавлятора» часто компилируется и без `move`,
> потому что компилятор сам сообразит скопировать. Но привычка ставить `move` при
> *возврате* замыкания наружу — хорошая и безопасная. Попробуй оба варианта и подумай,
> почему так.

## Идея 4. Функциональный стиль вместо ручных циклов

Сравни две версии одной задачи — «сумма квадратов»:

```rust
// Императивно: как именно перебирать.
let mut total = 0;
for x in v {
    total += x * x;
}

// Декларативно: что делаем с данными.
let total: i64 = v.iter().map(|x| x * x).sum();
```

Вторая версия короче, в ней нет изменяемого состояния (`mut`), нет ручного индекса и
негде ошибиться с границами. Она читается как предложение.

**ПОЧЕМУ это не «магия ценой скорости»:** благодаря ленивости и тому, что компилятор
видит всю цепочку насквозь, она оптимизируется в тот же цикл, что ты написал бы руками
(это называют *zero-cost abstractions* — абстракции без накладных расходов). Ты пишешь
выразительно — машина исполняет эффективно.

> Когда обычный `for` всё же лучше? Когда тело шага делает что-то «грязное» — печатает,
> пишет в файл, рано прерывается со сложным условием. Итераторы блестят на *чистых
> преобразованиях* данных. Выбирай по задаче, не из догмы.

### Ступенчатые подсказки

<details><summary>apply — с чего начать</summary>

`f` — это значение, которое можно вызывать как функцию. Просто вызови его: `f(x)`.
Граница `Fn` гарантирует, что вызов разрешён. Тело — одна строка.
</details>

<details><summary>make_adder — почему «висячая ссылка» и как её избежать</summary>

Верни замыкание `|x| x + n`. Замыкание захватывает `n`. Поскольку оно «уезжает» из
функции наружу, заставь его взять `n` во владение: добавь `move` перед `|x|`. Для
`i64` это просто копия. Подумай: что было бы, если бы `n` был, скажем, большим `String`?
</details>

<details><summary>sum_of_squares — форма цепочки</summary>

`v.iter()` даёт итератор по `&i64`. Дальше `.map(|x| x * x)` — каждый элемент в квадрат.
Завершает потребитель `.sum()`. Тип результата `i64` компилятор выведет из сигнатуры.
Для пустого среза `sum()` честно вернёт `0` — отдельный случай писать не нужно.
</details>

<details><summary>evens — ловушка ссылок против значений</summary>

`v.iter()` даёт `&i64`, поэтому в `.filter(|x| ...)` элемент — это `&&i64`. Чётность:
`**x % 2 == 0` (или сначала разыменуй). Но в `Vec<i64>` нужны **значения**, а не
ссылки — поставь `.copied()` где-нибудь до `.collect()`, чтобы превратить `&i64` в `i64`.
Итог: `v.iter().filter(...).copied().collect()`.
</details>

<details><summary>word_lengths — самый прямой map</summary>

`words.iter()` отдаёт `&&str`, но `.len()` на нём работает (авто-разыменование), отдавая
`usize`. Значит `words.iter().map(|w| w.len()).collect()`. Тип `Vec<usize>` берётся из
сигнатуры — `collect()` сам поймёт, что собирать.
</details>
