# Модуль 09 — Обобщения и трейты

Большая идея: один и тот же код хочется применять к **разным типам** — найти максимум
из двух `i32`, из двух `f64`, из двух строк. Копировать функцию под каждый тип — глупо.
Обобщения (`generics`) позволяют написать функцию **один раз** для «любого подходящего
типа `T`», а трейты задают, что значит «подходящий». При этом — и это ключ — **типобезопасность
не теряется**: всё проверяется на этапе компиляции, никакого приведения типов в рантайме.

В C++ это были шаблоны (`template<class T>`); в Rust та же идея, но ограничения на `T`
выражаются явно через трейты, и компилятор проверяет их заранее, а не «когда что-то
сломается при инстанцировании».

## Идея 1. Обобщённые функции и мономорфизация

Параметр типа объявляется в угловых скобках после имени функции:

```rust
fn first<T>(pair: (T, T)) -> T {
    pair.0
}

let a = first((1, 2));        // здесь T = i32
let b = first(("x", "y"));    // здесь T = &str
```

`T` — это «дырка» для типа, которую компилятор заполнит в момент вызова. Под капотом
Rust делает **мономорфизацию**: для каждого реально использованного типа генерирует
отдельную, специализированную копию функции. То есть `first::<i32>` и `first::<&str>` —
это две разные функции в итоговом бинарнике.

**ПОЧЕМУ это важно:** обобщения в Rust — это **нулевая стоимость в рантайме**. Ты пишешь
обобщённый код для удобства, а машинный код получается такой же быстрый, как если бы ты
вручную написал версию под каждый тип. Никакой динамической диспетчеризации, никаких
проверок типа на лету. (Платишь временем компиляции и размером бинарника — но не скоростью.)

## Идея 2. Трейт — общий интерфейс; ограничения вроде `T: PartialOrd`

Чистый `T` почти бесполезен: про «совсем любой тип» компилятор ничего не знает, поэтому
ты не можешь даже сравнить два значения. Ограничения (`trait bounds`) говорят: «`T` — не
любой, а только такой, который умеет вот это».

```rust
fn biggest<T: PartialOrd>(a: T, b: T) -> T {
    if a > b { a } else { b }   // > доступен, потому что T: PartialOrd
}
```

`PartialOrd` — это трейт из стандартной библиотеки, который даёт операции `<`, `>`,
`<=`, `>=`. Записав `T: PartialOrd`, ты получил право сравнивать `a` и `b`.

**ПОЧЕМУ компилятор требует ограничение, а не «доверяет»:** если бы ты попытался написать
`a > b` без `T: PartialOrd`, Rust остановил бы тебя **прямо в определении функции**, не
дожидаясь вызова. Это огромная разница с шаблонами C++, где ошибка «тип не умеет `>`»
вылезает позже, при инстанцировании, простынёй непонятных сообщений. В Rust контракт
проверяется один раз, в одном месте — и для автора функции, и для всех её пользователей.

Несколько ограничений соединяются плюсом: `T: PartialOrd + Copy` значит «`T` умеет
сравниваться **и** копироваться побитово».

## Идея 3. `impl Trait` для типа; методы по умолчанию

Трейт можно не только использовать как ограничение, но и **объявить свой** и реализовать
для своих типов:

```rust
trait Greet {
    fn name(&self) -> String;            // обязательный метод

    fn hello(&self) -> String {          // метод ПО УМОЛЧАНИЮ
        format!("Привет, {}!", self.name())
    }
}

struct Person { who: String }

impl Greet for Person {
    fn name(&self) -> String {
        self.who.clone()
    }
    // hello НЕ переопределяем — берём реализацию по умолчанию
}
```

`impl Greet for Person { ... }` — это «реализация трейта `Greet` для типа `Person`».
Метод `name` обязателен, его нужно написать. А `hello` имеет **тело по умолчанию** прямо
в трейте: оно вызывает `self.name()` и строит приветствие. `Person` его не переопределяет
и получает бесплатно.

**ПОЧЕМУ методы по умолчанию полезны:** они дают общее поведение всем реализациям, не
заставляя каждый тип копировать одно и то же. Тип обязан определить только «уникальную»
часть (`name`), а «общая» (`hello`) приходит из трейта. Хочешь особое поведение — переопредели
метод в своём `impl`; не хочешь — пользуйся умолчанием.

## Идея 4. ПОЧЕМУ обобщения дают переиспользование без потери типобезопасности

Соберём всё вместе. Альтернативы обобщениям обычно плохи:

- **Дублировать код** под каждый тип — копипаста, которая расходится при правках.
- **Использовать «динамический» тип** (как `Object` в некоторых языках) — теряешь
  проверки на компиляции, ошибки всплывают в рантайме, и платишь за упаковку/проверки.

Обобщения с трейтами дают третий путь: **один** кусок кода, который работает для всех
типов, удовлетворяющих контракту, и при этом:

- если ты вызовешь `my_max(3, "строка")` (разные типы) — **ошибка компиляции**;
- если вызовешь с типом, который не реализует нужный трейт — **ошибка компиляции** с
  понятным сообщением «трейт `X` не реализован для `Y`»;
- в рантайме всё работает на полной скорости (мономорфизация, Идея 1).

Это и есть девиз: «переиспользование **без** потери типобезопасности и **без** платы за
скорость». Ругань компилятора здесь — твой союзник: он ловит несовместимость типов до
того, как программа вообще запустится.

## Идея 5. `impl Trait` в позиции аргумента и возвращаемого значения

Помимо синтаксиса `fn f<T: Trait>(x: T)`, Rust позволяет писать ограничение прямо
в сигнатуре, без явного параметра `T`:

```rust
// «принять что угодно, что умеет summarize»
fn print_summary(item: &impl Summary) {
    println!("{}", item.summarize());
}
```

`impl Summary` в позиции аргумента — это синтаксический сахар над `<T: Summary>`.
Компилятор мономорфизирует функцию точно так же. Разница лишь в читаемости: когда
`T` нужен только в одном месте и нигде больше не упоминается, `impl Trait` короче.

Второй вариант использования — **в позиции возврата**:

```rust
fn make_greeter() -> impl Greet {
    Person { who: String::from("Мир") }
}
```

Здесь функция обещает «вернуть что-то, что умеет `Greet`», не называя конкретный тип.
Это удобно, когда хочешь скрыть внутренний тип от вызывающей стороны. Ограничение:
все пути возврата должны возвращать **один и тот же** конкретный тип — иначе компилятор
откажет (это не динамический полиморфизм!).

**Типичная ошибка:** написать `-> impl Trait` и попробовать вернуть разные типы в
`if`/`else` — Rust не разрешает: оба плеча должны совпадать.

```rust
// ЭТО НЕ СКОМПИЛИРУЕТСЯ — два разных типа в разных ветках:
fn bad(flag: bool) -> impl Greet {
    if flag { Person { who: "A".into() } } else { Robot { id: 1 } }
    //                                             ^^^^^^ другой тип — ошибка
}
```

## Идея 6. `where`-клаузы: читаемость при сложных ограничениях

Когда ограничений становится много, сигнатура с `<T: A + B, U: C + D>` превращается в
нечитаемую стену. Синтаксис `where` переносит ограничения в отдельный блок:

```rust
// Трудночитаемо:
fn combine<T: PartialOrd + Copy + std::fmt::Debug, U: Clone + std::fmt::Display>(
    a: T, b: U
) -> T { todo!() }

// То же самое, но аккуратно:
fn combine<T, U>(a: T, b: U) -> T
where
    T: PartialOrd + Copy + std::fmt::Debug,
    U: Clone + std::fmt::Display,
{ todo!() }
```

Оба варианта **эквивалентны** для компилятора — это чисто стилистический выбор. В
реальных проектах `where` практически обязателен, как только параметров типа больше
двух или у каждого по несколько ограничений.

`where` также позволяет накладывать ограничения на **ассоциированные типы** и
сложные выражения, которые нельзя записать в обычных угловых скобках:

```rust
fn foo<I>(iter: I)
where
    I: Iterator,
    I::Item: std::fmt::Display,  // ограничение на ассоциированный тип
{ todo!() }
```

## Идея 7. Обобщённые структуры и `impl<T>` блоки

Обобщения применяются не только к функциям, но и к **структурам**. Параметр типа
объявляется при определении структуры и повторяется в блоке `impl`:

```rust
struct Pair<T> {
    first: T,
    second: T,
}

impl<T: std::fmt::Display + PartialOrd> Pair<T> {
    fn larger(&self) -> &T {
        if self.first >= self.second { &self.first } else { &self.second }
    }
}
```

Обрати внимание: ограничения (`Display + PartialOrd`) ставятся **не** на саму
структуру, а на конкретный блок `impl`. Это означает, что `Pair<T>` можно создать для
любого `T`, но метод `larger` доступен только если `T` умеет сравниваться и выводиться.
Это гибче, чем ограничивать структуру целиком — разным методам могут нужны разные
ограничения.

**Частая ошибка:** написать `impl Pair<T>` без `<T>` после `impl` — компилятор не поймёт,
что `T` это параметр типа, а не конкретный тип с таким именем.

```rust
impl Pair<T> { ... }     // ОШИБКА: T неизвестен
impl<T> Pair<T> { ... }  // правильно: объявляем T как параметр
```

## Идея 8. Несколько параметров типа

Функция (или структура) может иметь несколько независимых параметров типа:

```rust
fn zip_with<A, B, F>(a: A, b: B, f: F) -> String
where
    A: std::fmt::Display,
    B: std::fmt::Display,
    F: Fn(String, String) -> String,
{
    f(a.to_string(), b.to_string())
}

let result = zip_with(42_i32, 3.14_f64, |a, b| format!("{a} и {b}"));
// result == "42 и 3.14"
```

`A` и `B` — независимые параметры: их могут заполнить разные типы. При вызове
компилятор выведет `A = i32`, `B = f64` из аргументов. Параметры именуют по
соглашению одной заглавной буквой, но можно и словами: `Key`, `Value`, `Item`.

## Идея 9. Правило сиротства (orphan rule) — границы реализаций трейтов

В Rust нельзя реализовать **чужой трейт** для **чужого типа**. Это называется
**правилом сиротства** (orphan rule), и оно — фундамент целостности системы типов.

Пример того, что **нельзя**:

```rust
// Файл в твоём крейте:
impl std::fmt::Display for Vec<i32> { ... }  // ОШИБКА!
// Display — из std, Vec — тоже из std. Ни то, ни другое не твоё.
```

Что **можно**:

```rust
struct Wrapper(Vec<i32>);  // твой тип

impl std::fmt::Display for Wrapper { ... }  // OK: тип твой
```

**ПОЧЕМУ это правило существует:** без него два независимых крейта могли бы оба
реализовать `Display for Vec<i32>` по-разному, и компилятор не знал бы, какую выбрать.
Правило сиротства гарантирует, что у каждой пары `(тип, трейт)` есть ровно один
«ответственный» крейт. Обходной путь — паттерн «обёртка» (`newtype`): создать свой тип
`struct Wrapper(Vec<i32>)` и реализовать трейт уже для него.

## Идея 10. Статическая и динамическая диспетчеризация: `dyn Trait`

До сих пор все функции с обобщениями использовали **статическую диспетчеризацию**:
компилятор заранее знает конкретный тип и генерирует специализированный код для каждого.
Это быстро, но иногда хочется **одного списка** объектов разных типов, которые все
реализуют один трейт. Тут на сцену выходит `dyn Trait`:

```rust
fn announce(items: &[&dyn Summary]) {  // срез объектов РАЗНЫХ типов
    for item in items {
        println!("{}", item.summarize());  // вызов через vtable
    }
}
```

`dyn Summary` — это **трейт-объект** (trait object). За кулисами он хранит два
указателя: на данные и на таблицу виртуальных методов (vtable). Это похоже на
виртуальные методы в C++ — вызов чуть медленнее (косвенный переход), но тип конкретного
объекта может быть неизвестен на этапе компиляции.

Сравним подходы:

| Критерий                  | `impl Trait` / `<T: Trait>` (статика)   | `dyn Trait` (динамика)            |
|---------------------------|-----------------------------------------|-----------------------------------|
| Скорость вызова           | как обычная функция (встраивается)      | чуть медленнее (vtable)           |
| Размер бинарника          | растёт (копии под каждый тип)           | одна копия функции                |
| Гетерогенная коллекция    | невозможно                              | `Vec<Box<dyn Trait>>`             |
| Тип известен в компиляции | да                                      | нет                               |

**Правило большого пальца:** пока не знаешь, что тебе нужна гетерогенная коллекция или
«объект неизвестного типа», используй статику. `dyn` — это осознанный выбор, а не
умолчание.

```rust
// Статика: два отдельных монорфных вызова
fn print_static<T: Summary>(item: &T) { println!("{}", item.summarize()); }

// Динамика: один вызов, тип разрешается в рантайме через vtable
fn print_dynamic(item: &dyn Summary) { println!("{}", item.summarize()); }
```

## Идея 11. Супертрейты: `trait Foo: Bar`

Иногда один трейт имеет смысл только вместе с другим. Например, нельзя красиво
отобразить что-то, что нельзя описать словами. Запись `trait Foo: Bar` означает:
«чтобы реализовать `Foo`, тип обязан также реализовать `Bar`».

```rust
trait Printable: std::fmt::Display {
    fn print(&self) {
        println!("{}", self);  // можно — Display гарантирован
    }
}

struct Point { x: i32, y: i32 }

impl std::fmt::Display for Point {
    fn fmt(&self, f: &mut std::fmt::Formatter<'_>) -> std::fmt::Result {
        write!(f, "({}, {})", self.x, self.y)
    }
}

impl Printable for Point {}  // OK: Display реализован выше
```

Если попробовать реализовать `Printable` для типа без `Display` — ошибка компиляции.
Супертрейты — это способ выразить «у меня есть зависимость от другого контракта» прямо
в системе типов.

## Идея 12. `Sized` и `?Sized` — знание о размере типа

Каждый параметр типа `T` по умолчанию неявно включает ограничение `T: Sized` — то есть
компилятор предполагает, что размер `T` известен в момент компиляции. Для большинства
типов (`i32`, `String`, пользовательские структуры) это верно.

Но есть типы, размер которых **не** известен на этапе компиляции — например, срезы `[T]`
или сами трейт-объекты `dyn Trait`. Их называют **DST** (dynamically sized types). Напрямую
хранить DST в переменной нельзя — только за указателем (`&[T]`, `Box<dyn Trait>`).

Если хочешь функцию, которая принимает и `Sized`, и DST-типы, снимай ограничение через `?Sized`:

```rust
fn print_it<T: std::fmt::Display + ?Sized>(val: &T) {
    println!("{val}");
}

print_it("обычный &str");   // &str — это &[u8] под капотом, DST
print_it(&42_i32);           // i32 — Sized, тоже работает
```

Знак `?` здесь читается как «может быть не Sized» — ослабление ограничения вместо
его усиления. В повседневном коде `?Sized` встречается редко, но в стандартной
библиотеке его много (например, `Box<T>` принимает `?Sized`, чтобы поддерживать
`Box<dyn Trait>`).

## Идея 13. Ассоциированные типы: трейт сам выбирает «свой» тип

До сих пор тип-параметры писали мы, снаружи: `Pair<T>`, `my_max::<i32>`. Но трейт может
держать тип **внутри себя** — это **ассоциированный тип**. Канонический пример — итератор:

```rust
trait Iterator {
    type Item;                       // ← ассоциированный тип: «что я выдаю»
    fn next(&mut self) -> Option<Self::Item>;
}
```

`Item` — это «дырка для типа», но заполняет её **не вызывающий, а реализация**. Когда ты
пишешь `impl Iterator for Counter { type Item = u32; ... }`, ты раз и навсегда фиксируешь:
«мой итератор выдаёт `u32`». То же у сложения: `trait Add { type Output; ... }` — `Output`
отвечает на вопрос «а что вернёт `+`?».

**ПОЧЕМУ ассоциированный тип, а не второй параметр `trait Iterator<Item>`?** Сравни:

```rust
trait IteratorBad<Item> { fn next(&mut self) -> Option<Item>; }  // параметр
trait Iterator        { type Item; ... }                          // ассоциированный тип
```

С параметром один тип мог бы реализовать `IteratorBad<u32>` И `IteratorBad<String>` —
и тогда при вызове `c.next()` компилятор не знал бы, какую из реализаций брать, пришлось
бы всюду писать `Iterator::<u32>::next(...)`. Ассоциированный тип говорит: «на данный тип —
**ровно одна** реализация, и `Item` у неё **один**». Это убирает неоднозначность и делает
сигнатуры короче: `fn collect<I: Iterator>(it: I)` вместо `fn collect<I, T>(it: I) where I: Iterator<T>`.

Правило-памятка: **параметр типа** — когда один тип логично реализует трейт для МНОГИХ
вариантов (`From<u8>`, `From<u16>` для одного типа — норм). **Ассоциированный тип** — когда
вариант ровно один и однозначно определяется реализующим типом (`Item`, `Output`).

Ограничивать ассоциированный тип можно прямо в bound: `T: Iterator<Item = u8>` или
`T: Add<Output = T>` — «складывается сам в себя, результат того же типа».

## Идея 14. Перегрузка операторов через `std::ops`

Операторы `+`, `-`, `*`, `[]`, `==` в Rust — это **синтаксический сахар над трейтами** из
модуля `std::ops` (и `std::cmp`). Компилятор переписывает `a + b` в `Add::add(a, b)`,
`a * b` — в `Mul::mul(a, b)`, `v[i]` — в `Index::index(&v, i)`. Чтобы научить `+` работать
с твоим типом, ты реализуешь соответствующий трейт:

```rust
use std::ops::Add;

#[derive(Clone, Copy, PartialEq, Debug)]
struct Vec2 { x: f64, y: f64 }

impl Add for Vec2 {
    type Output = Vec2;                               // ассоциированный тип (Идея 13)
    fn add(self, rhs: Vec2) -> Vec2 {
        Vec2 { x: self.x + rhs.x, y: self.y + rhs.y } // покоординатно
    }
}

let s = Vec2 { x: 1.0, y: 2.0 } + Vec2 { x: 3.0, y: 4.0 };  // s == Vec2 { x: 4.0, y: 6.0 }
```

Заметь связку: `Add` несёт ассоциированный тип `Output` — это тот самый тип результата.
Для `Vec2 + Vec2` результат — снова `Vec2`, поэтому `type Output = Vec2`. (Output может
отличаться от операндов: у `Mul<f64> for Vec2` логично `Output = Vec2`, а у скалярного
произведения был бы `Output = f64`.)

**ПОЧЕМУ это полезно, а не «красивость»:** реализовав `Add<Output = T>`, твой тип
становится **взаимозаменяем с числами в обобщённом коде**. Функция `fn sum<T>(items: &[T]) -> T
where T: Add<Output = T> + Copy + Default` будет суммировать и `i32`, и твой `Vec2` — потому
что оба честно реализуют один контракт. Это прямое продолжение «переиспользования без
потери типобезопасности» (Идея 4), но уже на уровне операторов.

**Подвохи.** (1) `+` забирает операнды по значению (`fn add(self, rhs)`), поэтому для не-`Copy`
типов либо клонируй, либо реализуй `Add` ещё и для ссылок (`impl Add for &Vec2`). (2) Перегрузка
ограничена правилом сиротства (Идея 9): `impl Add<MyType> for i32` нельзя, если `i32` чужой —
поможет newtype-обёртка. (3) Не перегружай оператор так, чтобы он значил неожиданное (`+`,
который вычитает) — читателю кода это сломает интуицию.

## Идея 15. Блэнкет-реализации (blanket impls): один `impl` на бесконечное множество типов

Обычный `impl` пишут для одного типа: `impl Summary for Article`. Но можно реализовать трейт
**сразу для всех типов, удовлетворяющих условию** — это **блэнкет-реализация**:

```rust
trait Loud {
    fn shout(&self) -> String;
}

// Реализация для ВСЕХ T, которые умеют Display:
impl<T: std::fmt::Display> Loud for T {
    fn shout(&self) -> String {
        format!("{}!!!", self)
    }
}

// Теперь shout() есть у всего, что Display: i32, String, f64, char, ...
let s = 42.shout();        // "42!!!"
let t = "hi".shout();      // "hi!!!"
```

Голова `impl<T: Display> Loud for T` читается так: «для любого `T`, у которого есть `Display`,
вот реализация `Loud`». Это покрывает **неограниченное, заранее неизвестное** множество типов
одним блоком кода.

Самый знаменитый пример — в стандартной библиотеке:

```rust
impl<T: Display> ToString for T { ... }
```

Именно поэтому `.to_string()` есть у каждого `Display`-типа: никто не писал `impl ToString`
отдельно для `i32`, `f64`, `char` — одна блэнкет-реализация покрыла их все разом.

**ПОЧЕМУ это мощно:** ты задаёшь поведение через *свойство* (`: Display`), а не перечисляя
типы. Добавится новый `Display`-тип в чужом крейте — он автоматически получит и `Loud`,
и `to_string()`, без единой строчки дополнительного кода.

**Подвох:** блэнкет-impl тоже подчиняется правилу сиротства — `impl<T> Loud for T` законен
только если трейт `Loud` **твой**. И две перекрывающиеся блэнкет-реализации запрещены
(нельзя `impl Loud for T` и одновременно `impl Loud for i32` — компилятор увидит конфликт
«coherence»).

## Идея 16. `#[derive]` — трейты «из коробки», без ручного `impl`

Многие базовые трейты настолько механистичны, что писать их `impl` руками — рутина. Для них
есть автовывод через атрибут `#[derive(...)]`:

```rust
#[derive(Debug, Clone, Copy, PartialEq, Eq, Hash, PartialOrd, Ord, Default)]
struct Point { x: i32, y: i32 }
```

Эта строка автоматически генерирует реализации сразу нескольких трейтов:

| derive       | что даёт                                        |
|--------------|-------------------------------------------------|
| `Debug`      | печать через `{:?}` / `{:#?}`                    |
| `Clone`      | явное `.clone()`                                |
| `Copy`       | побитовое копирование (требует, чтобы все поля были `Copy`) |
| `PartialEq`  | сравнение `==` (поле за полем)                   |
| `Hash`       | использование как ключа в `HashMap`/`HashSet`    |
| `PartialOrd` | сравнения `<`, `>` (лексикографически по полям)  |
| `Default`    | `Point::default()` — все поля в значение по умолчанию |

**ПОЧЕМУ derive, а не руками:** для структуры из 5 полей ручной `impl PartialEq` — это пять
сравнений, которые легко рассинхронить при добавлении поля. `derive` строит реализацию из
полей механически и обновляется сам. Правило: **сначала пробуй `derive`, руки — только когда
нужно нестандартное поведение** (например, сравнивать по одному ключевому полю, а не по всем).

**Подвохи.** (1) `derive(Copy)` требует `derive(Clone)` и чтобы ВСЕ поля были `Copy` —
`String` внутри это сломает. (2) Derive накладывает bound на параметры: `#[derive(Clone)]
struct W<T>(T)` сгенерирует `impl<T: Clone> Clone for W<T>` — клон доступен, только если `T: Clone`.
(3) Перегрузку операторов (`Add`, Идея 14) derive **не** делает — её всегда пишут руками.

## Идея 17. Маркерные трейты `Send` и `Sync` — трейты вообще без методов

Бывают трейты **без единого метода** — они ничего не «умеют», а просто **помечают** тип
неким свойством. Их называют **маркерными**. Самые важные — `Send` и `Sync`:

- `Send` — «значение этого типа можно **переместить** в другой поток»;
- `Sync` — «на значение этого типа можно **разделять ссылку** `&T` между потоками».

У них нет тел, нечего реализовывать — присутствие самого трейта и есть вся информация.
Более того, они **автоматические**: компилятор сам выводит `Send`/`Sync` для типа, если из
него выводятся все его поля. Своими руками их обычно не пишут — они «прилипают» сами.

```rust
fn assert_send_sync<T: Send + Sync>() {}

assert_send_sync::<i32>();             // OK: примитивы — Send + Sync
assert_send_sync::<String>();          // OK
// assert_send_sync::<std::rc::Rc<i32>>();  // ОШИБКА: Rc НЕ Send и НЕ Sync
```

**ПОЧЕМУ это важно (и при чём тут безопасность потоков):** именно `Send`/`Sync` — то, чем
Rust на этапе компиляции доказывает отсутствие гонок данных. `thread::spawn` требует, чтобы
переданное в поток замыкание было `Send`; если ты попробуешь утащить в поток `Rc` (неатомарный
счётчик ссылок), компилятор откажет — потому что `Rc` не `Send`. Это спасает от целого класса
багов **до запуска программы**. (Глубоко эту тему разбирает модуль про конкурентность; здесь
важно усвоить идею: трейт может не нести методов, а лишь *маркировать* свойство, и компилятор
умеет на такие маркеры опираться.)

Другие маркеры, которые ты уже видел: `Copy` (Идея 16) — тоже почти маркер (наследует от
`Clone`, но сам без методов, лишь меняет семантику присваивания), и `Sized` (Идея 12).

## Существенное задание: `MinMaxTracker<T>`

Реализуй обобщённую структуру `MinMaxTracker<T>` — **трекер экстремумов потока данных**.
Она принимает значения по одному через метод `push` и в любой момент может сказать:
какой минимум, какой максимум и сколько значений было подано.

Публичный контракт:

```rust
pub struct MinMaxTracker<T> { /* поля — приватные */ }

impl<T: PartialOrd + Copy> MinMaxTracker<T> {
    pub fn new() -> Self;               // создать пустой трекер
    pub fn push(&mut self, v: T);       // подать следующее значение
    pub fn min(&self) -> Option<T>;     // None если ещё не было push
    pub fn max(&self) -> Option<T>;     // None если ещё не было push
    pub fn count(&self) -> usize;       // число вызовов push
}
```

Задание учит:
- объявлять обобщённую структуру с приватными полями;
- писать `impl<T: PartialOrd + Copy>` блок с несколькими методами;
- хранить состояние между вызовами (`&mut self`);
- использовать `Option<T>` до первого `push` — как способ выразить «ещё ничего нет»;
- работать с `PartialOrd` (сравнение) и `Copy` (хранение значения без Move).

Примеры ожидаемого поведения:

```rust
let mut t: MinMaxTracker<i32> = MinMaxTracker::new();
assert_eq!(t.min(), None);      // пусто
assert_eq!(t.count(), 0);

t.push(5);
t.push(2);
t.push(8);
assert_eq!(t.min(), Some(2));
assert_eq!(t.max(), Some(8));
assert_eq!(t.count(), 3);

// Работает для f64:
let mut tf: MinMaxTracker<f64> = MinMaxTracker::new();
tf.push(3.14);
tf.push(-1.0);
assert_eq!(tf.min(), Some(-1.0));
assert_eq!(tf.max(), Some(3.14));

// И для char:
let mut tc: MinMaxTracker<char> = MinMaxTracker::new();
tc.push('z'); tc.push('a'); tc.push('m');
assert_eq!(tc.min(), Some('a'));
assert_eq!(tc.max(), Some('z'));
```

### Подсказки к MinMaxTracker

<details><summary>Шаг 1: какие поля нужны структуре?</summary>

Подумай: что трекер должен помнить после каждого `push`? Ему нужно хранить текущий
минимум и максимум (оба `Option<T>`, чтобы до первого `push` вернуть `None`), и счётчик.
Попробуй объявить структуру с тремя приватными полями и написать `new()`, который
создаёт её «пустой».
</details>

<details><summary>Шаг 2: как объявить impl<T: ...> блок?</summary>

Синтаксис: `impl<T: PartialOrd + Copy> MinMaxTracker<T> { ... }`. Ограничения
(`PartialOrd + Copy`) ставятся на `T` в этом блоке. Обрати внимание: структуру
объявляют как `struct MinMaxTracker<T>` без ограничений — ограничения появляются
только в конкретных `impl`-блоках, там где они реально нужны.
</details>

<details><summary>Шаг 3: логика push — сравнить и обновить</summary>

Внутри `push` тебе нужно обновить оба `Option`-поля. Разбери случай для минимума:
- если поле `min` равно `None` — сохрани `Some(v)` (это первый элемент);
- если `Some(cur)` — сравни `v < cur`; если `v` меньше, замени на `Some(v)`.

Удобно через `match self.min { None => v, Some(cur) => if v < cur { v } else { cur } }`.
</details>

<details><summary>Шаг 4: зачем именно Copy, а не Clone?</summary>

Когда ты делаешь `self.min = Some(v)`, ты сохраняешь значение `v`. Но `push` уже
получил `v` по значению — почему здесь нужен `Copy`? Потому что внутри `match` ты
сравниваешь `v` с `cur` (PartialOrd), а потом хочешь вернуть одно из них. Если бы тип
не был `Copy`, сравнение `v < cur` могло бы переместить значение, и потом вернуть его
было бы уже нельзя. `Copy` гарантирует «бесплатное» клонирование по битам — всё остаётся
доступным.
</details>

<details><summary>Подсказка 5.2: какой трейт реализовать и где взять Output</summary>

Оператор `+` — это сахар над трейтом `std::ops::Add` (Идея 14). Каркас реализации:

```rust
impl std::ops::Add for Point {
    type Output = /* что вернёт + ? */;
    fn add(self, other: Point) -> /* ? */ {
        // ...
    }
}
```

`type Output` — это **ассоциированный тип** (Идея 13): он отвечает на вопрос «какой тип
у результата `Point + Point`?». Сложив две точки, мы получаем снова точку — подставь сюда
подходящий тип. Тип, который ты напишешь в `Output`, и тип возврата `add` должны совпадать.
</details>

<details><summary>Подсказка 5.3: тело add — покоординатно</summary>

`add` забирает `self` и `other` по значению (для `Copy`-типа это просто копии — оригиналы
не пропадают). Нужно собрать **новую** `Point`, сложив одноимённые координаты:

- `x` результата = `self.x + other.x`;
- `y` результата = `self.y + other.y`.

Верни `Point { x: ..., y: ... }`. Заметь: коммутативность (`a + b == b + a`) и нейтральность
нуля получатся **сами собой** из того, что сложение `f64` коммутативно — отдельно их
программировать не надо. (Полное тело не привожу — собери его из этих двух строк.)
</details>
