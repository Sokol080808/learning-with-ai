# Модуль 09 — Обобщения и трейты

Большая идея: один и тот же код хочется применять к **разным типам** — найти максимум
из двух `i32`, из двух `f64`, из двух строк. Копировать функцию под каждый тип — глупо.
Обобщения (`generics`) позволяют написать функцию **один раз** для «любого подходящего
типа `T`», а трейты задают, что значит «подходящий». При этом — и это ключ — **типобезопасность
не теряется**: всё проверяется на этапе компиляции, никакого приведения типов в рантайме.

В C++ это были шаблоны (`template<class T>`); в Rust та же идея, но ограничения на `T`
выражаются явно через трейты, и компилятор проверяет их заранее, а не «когда что-то
сломается при инстанцировании».

## Идея 1. Обобщённые функции и мономорфизация

Параметр типа объявляется в угловых скобках после имени функции:

```rust
fn first<T>(pair: (T, T)) -> T {
    pair.0
}

let a = first((1, 2));        // здесь T = i32
let b = first(("x", "y"));    // здесь T = &str
```

`T` — это «дырка» для типа, которую компилятор заполнит в момент вызова. Под капотом
Rust делает **мономорфизацию**: для каждого реально использованного типа генерирует
отдельную, специализированную копию функции. То есть `first::<i32>` и `first::<&str>` —
это две разные функции в итоговом бинарнике.

**ПОЧЕМУ это важно:** обобщения в Rust — это **нулевая стоимость в рантайме**. Ты пишешь
обобщённый код для удобства, а машинный код получается такой же быстрый, как если бы ты
вручную написал версию под каждый тип. Никакой динамической диспетчеризации, никаких
проверок типа на лету. (Платишь временем компиляции и размером бинарника — но не скоростью.)

## Идея 2. Трейт — общий интерфейс; ограничения вроде `T: PartialOrd`

Чистый `T` почти бесполезен: про «совсем любой тип» компилятор ничего не знает, поэтому
ты не можешь даже сравнить два значения. Ограничения (`trait bounds`) говорят: «`T` — не
любой, а только такой, который умеет вот это».

```rust
fn biggest<T: PartialOrd>(a: T, b: T) -> T {
    if a > b { a } else { b }   // > доступен, потому что T: PartialOrd
}
```

`PartialOrd` — это трейт из стандартной библиотеки, который даёт операции `<`, `>`,
`<=`, `>=`. Записав `T: PartialOrd`, ты получил право сравнивать `a` и `b`.

**ПОЧЕМУ компилятор требует ограничение, а не «доверяет»:** если бы ты попытался написать
`a > b` без `T: PartialOrd`, Rust остановил бы тебя **прямо в определении функции**, не
дожидаясь вызова. Это огромная разница с шаблонами C++, где ошибка «тип не умеет `>`»
вылезает позже, при инстанцировании, простынёй непонятных сообщений. В Rust контракт
проверяется один раз, в одном месте — и для автора функции, и для всех её пользователей.

Несколько ограничений соединяются плюсом: `T: PartialOrd + Copy` значит «`T` умеет
сравниваться **и** копироваться побитово».

## Идея 3. `impl Trait` для типа; методы по умолчанию

Трейт можно не только использовать как ограничение, но и **объявить свой** и реализовать
для своих типов:

```rust
trait Greet {
    fn name(&self) -> String;            // обязательный метод

    fn hello(&self) -> String {          // метод ПО УМОЛЧАНИЮ
        format!("Привет, {}!", self.name())
    }
}

struct Person { who: String }

impl Greet for Person {
    fn name(&self) -> String {
        self.who.clone()
    }
    // hello НЕ переопределяем — берём реализацию по умолчанию
}
```

`impl Greet for Person { ... }` — это «реализация трейта `Greet` для типа `Person`».
Метод `name` обязателен, его нужно написать. А `hello` имеет **тело по умолчанию** прямо
в трейте: оно вызывает `self.name()` и строит приветствие. `Person` его не переопределяет
и получает бесплатно.

**ПОЧЕМУ методы по умолчанию полезны:** они дают общее поведение всем реализациям, не
заставляя каждый тип копировать одно и то же. Тип обязан определить только «уникальную»
часть (`name`), а «общая» (`hello`) приходит из трейта. Хочешь особое поведение — переопредели
метод в своём `impl`; не хочешь — пользуйся умолчанием.

## Идея 4. ПОЧЕМУ обобщения дают переиспользование без потери типобезопасности

Соберём всё вместе. Альтернативы обобщениям обычно плохи:

- **Дублировать код** под каждый тип — копипаста, которая расходится при правках.
- **Использовать «динамический» тип** (как `Object` в некоторых языках) — теряешь
  проверки на компиляции, ошибки всплывают в рантайме, и платишь за упаковку/проверки.

Обобщения с трейтами дают третий путь: **один** кусок кода, который работает для всех
типов, удовлетворяющих контракту, и при этом:

- если ты вызовешь `my_max(3, "строка")` (разные типы) — **ошибка компиляции**;
- если вызовешь с типом, который не реализует нужный трейт — **ошибка компиляции** с
  понятным сообщением «трейт `X` не реализован для `Y`»;
- в рантайме всё работает на полной скорости (мономорфизация, Идея 1).

Это и есть девиз: «переиспользование **без** потери типобезопасности и **без** платы за
скорость». Ругань компилятора здесь — твой союзник: он ловит несовместимость типов до
того, как программа вообще запустится.

### Подсказки

<details><summary>my_max: сравнение и владение</summary>

Ограничение `T: PartialOrd` даёт тебе операторы сравнения. Простое `if a >= b { ... } else
{ ... }` вернёт нужное значение. Аргументы пришли по значению (ты ими владеешь), так что
можешь спокойно вернуть один из них наружу.
</details>

<details><summary>largest: обход среза</summary>

Заведи «текущий максимум» — например, возьми первый элемент: `let mut best = v[0];`
(копия, ведь `T: Copy`). Затем пройди по остальным элементам и обновляй `best`, если
встретился больший. Итерироваться можно так:

```rust
for &item in v {       // &item: распаковываем ссылку из &[T] в значение (T: Copy)
    if item > best { best = item; }
}
```

В конце верни `best`. Срез ты только читал — владение никуда не уходило.
</details>

<details><summary>largest: почему именно T: Copy</summary>

`v: &[T]` — это заимствование: ты не владеешь элементами и не можешь «вынести» элемент
из среза, забрав владение (это оставило бы дыру в чужих данных). Но если `T: Copy`,
элемент можно дёшево **скопировать** — и вернуть копию, не трогая срез. Для типов вроде
`String`, которые не `Copy`, понадобилась бы другая сигнатура (например, возврат ссылки
`&T`). Здесь мы намеренно ограничились `Copy`, чтобы вернуть само значение.
</details>

<details><summary>Summary: метод по умолчанию и impl</summary>

Тело `preview` пиши **внутри трейта**, рядом с сигнатурой — это и делает его методом по
умолчанию. Оно может звать `self.summarize()`, даже несмотря на то что сам `summarize`
ещё не определён конкретным типом: к моменту вызова тип его уже реализует.

```rust
fn preview(&self) -> String {
    format!("Читать дальше: {}", self.summarize())
}
```

В `impl Summary for Article` реализуй только `summarize` (верни копию заголовка —
`self.title.clone()`). `preview` **не** пиши в `impl` — пусть подхватится из трейта.
</details>
