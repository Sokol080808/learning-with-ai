# Модуль 13 — Умные указатели: Box, Rc, RefCell

Большая идея: до сих пор владение было простым — у каждого значения **ровно один**
владелец, и менять значение можно было, только имея `&mut`. Умные указатели снимают эти
ограничения там, где они мешают, **не отказываясь от безопасности**. `Box<T>` кладёт
данные в кучу (и делает возможными рекурсивные типы). `Rc<T>` разрешает **нескольким**
владельцам делить одно значение. `RefCell<T>` разрешает менять данные через общую ссылку
`&self`, перенося проверку правил заимствования из компиляции в рантайм. Каждый из них —
это компромисс, и весь смысл модуля — понять, **какой инструмент когда брать**.

Умный указатель — это тип, который ведёт себя как указатель (через него можно добраться до
данных), но дополнительно владеет ими и сам освобождает память, когда выходит из области
видимости (тот же RAII, что и у `String`/`Vec`).

## Идея 1. `Box<T>` — данные в куче и рекурсивные типы

`Box<T>` — самый простой умный указатель: он кладёт значение **в кучу**, а на стеке
оставляет указатель фиксированного размера на него.

```rust
let b = Box::new(5);   // i64 живёт в куче, b — указатель на него на стеке
println!("{}", *b);    // *b разыменовывает Box, получаем 5
// память освободится автоматически, когда b выйдет из области видимости
```

Само по себе «положить число в кучу» — редкость. Настоящая причина существования `Box` —
**рекурсивные типы**. Попробуй описать cons-список «в лоб»:

```rust
enum List {
    Cons(i64, List),   // ❌ ошибка: "recursive type `List` has infinite size"
    Nil,
}
```

ПОЧЕМУ ошибка: чтобы разложить значение по памяти, компилятору нужно знать его **размер**.
Размер `List` = размер `i64` + размер `List` + … — рекурсия без дна, бесконечность. Решение
— спрятать хвост за указателем:

```rust
enum List {
    Cons(i64, Box<List>),   // ✅ Box — указатель фиксированного размера
    Nil,
}
```

Теперь размер `List` конечен: `i64` плюс один указатель. Сами данные хвоста лежат в куче, и
к ним ведёт `Box`. Так в Rust строят списки, деревья и другие самоссылающиеся структуры.

### Box как стёртый тип: `Box<dyn Trait>`

Есть ещё одна повседневная роль `Box` — **хранить трейт-объект**, когда конкретный тип
неизвестен на этапе компиляции:

```rust
trait Shape {
    fn area(&self) -> f64;
}

struct Circle { radius: f64 }
struct Rect   { w: f64, h: f64 }

impl Shape for Circle { fn area(&self) -> f64 { std::f64::consts::PI * self.radius * self.radius } }
impl Shape for Rect   { fn area(&self) -> f64 { self.w * self.h } }

// Вектор разных фигур — без Box не обойтись: Vec<Shape> не работает,
// потому что компилятор не знает размер каждого элемента.
let shapes: Vec<Box<dyn Shape>> = vec![
    Box::new(Circle { radius: 1.0 }),
    Box::new(Rect   { w: 2.0, h: 3.0 }),
];

for s in &shapes {
    println!("{:.2}", s.area()); // динамическая диспетчеризация
}
```

ПОЧЕМУ именно `Box<dyn Shape>`: трейт-объект `dyn Shape` не имеет фиксированного размера
(у каждой структуры он свой), а `Box` — это указатель, его размер всегда `usize`.
Под капотом `Box<dyn Shape>` — два слова: указатель на данные и указатель на таблицу
виртуальных методов (vtable). Вызов метода идёт через vtable — это **динамическая
диспетчеризация**, чуть медленнее статической, но без неё гетерогенный список был бы
невозможен.

### Трейт `Deref` и deref-коэрция

`Box<T>` реализует трейт `Deref`. Именно он позволяет написать `*b` для разыменования.
Компилятор автоматически строит **цепочки deref-коэрций**: если функция ожидает `&T`, а
ты передаёшь `&Box<T>` — Rust незаметно вставит нужное число `*`:

```rust
fn print_val(v: &i64) { println!("{v}"); }

let b = Box::new(42i64);
print_val(&b);   // &Box<i64> → автоматически коэрцируется в &i64
```

Эта цепочка может быть длиннее: `&Box<String>` → `&String` → `&str` — каждый шаг делает
один `Deref`. Благодаря этому функции, принимающие `&str`, спокойно работают с `&String`,
`&Box<String>` и т.д. без явного разыменования.

ПОЧЕМУ это работает безопасно: `Deref` возвращает именно `&T` (ссылку, а не значение),
никакое перемещение данных не происходит. Вся цепочка — просто несколько обращений по
адресу.

### Трейт `Drop` и порядок освобождения

Когда переменная выходит из области видимости, Rust вызывает метод `drop` из трейта
`Drop`. Для умных указателей это и есть момент освобождения памяти. Ты можешь реализовать
`Drop` для собственного типа, чтобы выполнить произвольную «уборку»:

```rust
struct Loud(i64);

impl Drop for Loud {
    fn drop(&mut self) {
        println!("drop {}", self.0);
    }
}

{
    let a = Loud(1);
    let b = Loud(2);
    let c = Loud(3);
    // При выходе из блока: drop 3, drop 2, drop 1 — ОБРАТНЫЙ порядок объявления!
}
```

ПОЧЕМУ обратный порядок: это то же правило, что и для стека вызовов — «последним вошёл,
первым вышел». Переменная, объявленная позже, могла получить ссылки на более ранние, и
их нельзя уничтожать первыми.

Поля структуры тоже дропаются в **обратном** порядке объявления. А между разными
переменными одного блока — они дропаются в порядке, обратном объявлению.

Принудительно дропнуть раньше срока позволяет `std::mem::drop(value)` (свободная
функция). Написать `value.drop()` нельзя — компилятор это запрещает, чтобы не допустить
двойного дропа.

## Идея 2. `Rc<T>` — разделяемое владение через счётчик ссылок

Правило «один владелец» иногда слишком строгое. Представь граф, где на один узел указывают
несколько рёбер: кто из них «владелец»? Ответ — **все сразу**. Для этого есть `Rc<T>`
(Reference Counted, счётчик ссылок):

```rust
use std::rc::Rc;

let a = Rc::new(5);          // одна сильная ссылка, счётчик == 1
let b = Rc::clone(&a);       // ещё одна, счётчик == 2 (данные НЕ копируются!)
let c = Rc::clone(&a);       // счётчик == 3

println!("{}", Rc::strong_count(&a));   // 3
```

`Rc` хранит рядом с данными счётчик. `Rc::clone` лишь **увеличивает счётчик** и отдаёт ещё
один указатель на те же данные — это дёшево, глубокого копирования не происходит. Когда
очередной `Rc` уничтожается, счётчик уменьшается; данные освобождаются, **когда счётчик
дойдёт до нуля**, то есть когда исчезнет последний владелец.

ПОЧЕМУ пишут `Rc::clone(&a)`, а не `a.clone()` (хотя работает и то, и другое): это
соглашение. `Rc::clone` визуально кричит «я делаю дешёвый клон указателя, а не копирую
данные» — читателю кода сразу ясно, что происходит разделяемое владение.

> Подвох: `Rc` рассчитан на **один поток** (счётчик неатомарный — поэтому он быстрый).
> Для нескольких потоков есть его «брат» `Arc`. И ещё: `Rc` даёт только **общий** доступ
> (`&T`), менять данные через него нельзя — для этого его сочетают с `RefCell` (см. ниже).

### Циклические ссылки и `Weak<T>`

Главная ловушка `Rc` — **циклические ссылки**. Если узел A держит `Rc` на B, а B держит
`Rc` на A, то счётчики никогда не упадут до нуля — это утечка памяти:

```rust
use std::rc::Rc;
use std::cell::RefCell;

struct Node {
    next: Option<Rc<RefCell<Node>>>,
}

let a = Rc::new(RefCell::new(Node { next: None }));
let b = Rc::new(RefCell::new(Node { next: Some(Rc::clone(&a)) }));
a.borrow_mut().next = Some(Rc::clone(&b));
// Цикл: a → b → a. Когда блок завершится, счётчики a и b будут равны 1, не 0.
// Память НЕ освободится — тихая утечка.
```

Решение — **слабые ссылки** `Weak<T>`. Слабая ссылка не увеличивает счётчик владельцев и
не препятствует удалению данных. Чтобы обратиться к данным через слабую ссылку, нужно
«обновить» её до сильной (`upgrade()`) — и это может вернуть `None`, если данные уже
уничтожены:

```rust
use std::rc::{Rc, Weak};
use std::cell::RefCell;

struct Node {
    value: i64,
    parent: Option<Weak<RefCell<Node>>>,  // слабая — не создаёт цикл
    child:  Option<Rc<RefCell<Node>>>,    // сильная — удерживает ребёнка
}

let parent = Rc::new(RefCell::new(Node { value: 1, parent: None, child: None }));
let child  = Rc::new(RefCell::new(Node {
    value: 2,
    parent: Some(Rc::downgrade(&parent)),  // слабая ссылка вверх
    child: None,
}));
parent.borrow_mut().child = Some(Rc::clone(&child));

// Теперь child → parent через Weak; цикла нет; память освободится корректно.
if let Some(p) = child.borrow().parent.as_ref().and_then(|w| w.upgrade()) {
    println!("parent value = {}", p.borrow().value);  // 1
}
```

Правило большого пальца: **сильная ссылка идёт вниз по дереву** (от родителя к детям),
**слабая — вверх** (от детей к родителю). Это канонический Rust-паттерн для деревьев с
родительскими ссылками.

### Почему `Rc` не `Send` и не `Sync` (Rc против Arc)

`Rc<T>` нельзя отправить в другой поток (`Send` не реализован) и нельзя разделить между
потоками (`Sync` не реализован). Причина — счётчик ссылок в `Rc` обновляется **неатомарно**.
Если два потока одновременно клонируют или дропают один `Rc`, счётчик испортится — это гонка
данных. Rust просто не позволяет компилировать такой код.

Для многопоточного разделяемого владения используй `Arc<T>` (Atomically Reference Counted):

```rust
use std::sync::{Arc, Mutex};
use std::thread;

let data = Arc::new(Mutex::new(vec![1, 2, 3]));

let data2 = Arc::clone(&data);
let handle = thread::spawn(move || {
    data2.lock().unwrap().push(4);
});
handle.join().unwrap();

println!("{:?}", data.lock().unwrap()); // [1, 2, 3, 4]
```

`Arc` — полный аналог `Rc`, но с атомарными операциями над счётчиком. Атомарность стоит
дороже, поэтому в однопоточном коде используй `Rc`, в многопоточном — `Arc`.
Аналогично, вместо `RefCell` (не `Sync`) в многопоточном коде берут `Mutex` или `RwLock`.

## Идея 3. `RefCell<T>` — внутренняя изменяемость

Обычное правило Rust: чтобы изменить значение, нужна изменяемая ссылка `&mut`. Компилятор
проверяет это статически и гарантирует, что в один момент либо много читателей, либо ровно
один писатель. Но иногда тебе нужно изменить данные, имея на руках лишь **общую** ссылку
`&self` — например, чтобы счётчик внутри объекта рос при «читающем» на вид вызове.

`RefCell<T>` даёт **внутреннюю изменяемость** (interior mutability): он позволяет это, но
сдвигает проверку правил заимствования с компиляции в **рантайм**:

```rust
use std::cell::RefCell;

let cell = RefCell::new(0);

*cell.borrow_mut() += 1;          // взяли изменяемый доступ, поменяли
println!("{}", *cell.borrow());   // взяли доступ на чтение: 1
```

- `borrow()` — доступ на чтение (можно много одновременно);
- `borrow_mut()` — изменяемый доступ (только один, и без читателей в этот момент).

ПОЧЕМУ это безопасно: те же правила, что обычно проверяет компилятор, теперь проверяет сам
`RefCell` во время работы. Нарушишь (например, попросишь `borrow_mut`, пока жив другой
заём) — будет **паника**, а не тихая порча данных. Ты не теряешь безопасность, ты лишь
меняешь «ошибка компиляции» на «паника в рантайме» — там, где статически выразить правило
неудобно.

```rust
struct Counter { count: RefCell<i64> }

impl Counter {
    fn increment(&self) {              // &self, не &mut self!
        *self.count.borrow_mut() += 1; // и всё равно меняем
    }
}
```

### `Cell<T>` для Copy-типов

Если внутри лежит тип, реализующий `Copy` (целые, булевы, числа с плавающей точкой),
есть более простая альтернатива — `Cell<T>`. Она не требует `borrow()`/`borrow_mut()`:
значение просто **копируется** внутрь и наружу:

```rust
use std::cell::Cell;

let x = Cell::new(0i64);
x.set(42);
println!("{}", x.get()); // 42
```

`Cell` не хранит живых ссылок на содержимое — только целые «снимки» значения. Поэтому
вопрос «а есть ли сейчас заём?» вообще не возникает, никакой проверки в рантайме нет, и
паниковать тут нечему. Используй `Cell` вместо `RefCell`, когда содержимое `Copy` — это
проще и быстрее.

| | `Cell<T>` | `RefCell<T>` |
|---|---|---|
| Работает с типами | только `Copy` | любой `T` |
| Доступ | только `get/set` (копирование) | `borrow/borrow_mut` (ссылки) |
| Рантайм-проверка | нет | да |
| Риск паники | нет | да (при нарушении правил) |

### Паттерн `Rc<RefCell<T>>` и двойное разыменование

Комбинация `Rc<RefCell<T>>` — стандартный способ иметь **несколько владельцев и при этом
изменять** общее значение:

```rust
use std::rc::Rc;
use std::cell::RefCell;

let shared = Rc::new(RefCell::new(vec![1, 2, 3]));

let owner_a = Rc::clone(&shared);
let owner_b = Rc::clone(&shared);

owner_a.borrow_mut().push(4);
owner_b.borrow_mut().push(5);

println!("{:?}", shared.borrow()); // [1, 2, 3, 4, 5]
```

Конструкция `*rc.borrow_mut()` выглядит непривычно — там два уровня разыменования:

1. `rc.borrow_mut()` — вызываем `borrow_mut` через `Deref` на `Rc`, получаем `RefMut<Vec<i64>>`;
2. `*...` — разыменовываем `RefMut`, получаем `Vec<i64>`.

Или ещё явнее: `(*rc.borrow_mut()).push(4)`. Оба варианта эквивалентны — Rust применяет
deref-коэрцию автоматически, поэтому на практике пишут просто `rc.borrow_mut().push(4)`.

## Идея 4. Когда что выбирать

Короткая шпаргалка — держи её в голове, выбирая инструмент:

| Нужно | Бери |
|---|---|
| Положить значение в кучу / рекурсивный тип, **один** владелец | `Box<T>` |
| Трейт-объект (гетерогенный список, динамическая диспетчеризация) | `Box<dyn Trait>` |
| **Несколько** владельцев одних данных (только чтение) | `Rc<T>` |
| Менять данные через `&self` (Copy-тип) | `Cell<T>` |
| Менять данные через `&self` (любой тип) | `RefCell<T>` |
| Несколько владельцев **и** возможность менять | `Rc<RefCell<T>>` |
| То же, но между потоками | `Arc<T>` / `Arc<Mutex<T>>` |
| Обратная ссылка в дереве (без цикла) | `Weak<T>` |

ПОЧЕМУ не брать `Rc<RefCell<T>>` всегда «на всякий случай»: каждый уровень — это цена
(счётчик, проверки в рантайме, риск паники) и потерянная статическая гарантия. Правило
простое: бери **самый простой** инструмент, которого хватает. Чаще всего хватает обычного
владения и `&mut`; умные указатели — для случаев, где их по-настоящему не хватает.

### Подсказки к заданиям 1–4

<details><summary>Идея 1 — <code>len</code> и <code>sum</code> через рекурсию</summary>

Метод объявлен в `impl List { ... }`, где `&self` — ссылка на текущий узел. Сделай
`match self { ... }` с двумя ветками:

- `List::Nil => 0` — дно рекурсии;
- `List::Cons(value, tail) => ...` — здесь `tail` имеет тип `&Box<List>`. Box разыменуется
  автоматически, поэтому ты можешь прямо вызвать `tail.len()` (или `tail.sum()`) — это и
  есть рекурсивный спуск по хвосту. Для `len` верни `1 + tail.len()`, для `sum` —
  `value + tail.sum()`.
</details>

<details><summary>Идея 1 — <code>build_list</code> с правильным порядком</summary>

Cons-список строится **изнутри наружу**: сначала самый хвост (`Nil`), потом каждый элемент
оборачивается вокруг уже построенного остатка. Значит, идти удобно **с конца** среза.

Начни с `let mut acc = List::Nil;`. Затем перебирай элементы в обратном порядке
(`for &x in items.iter().rev()`) и на каждом шаге делай
`acc = List::Cons(x, Box::new(acc));`. Так последний элемент окажется глубже всех, а
первый — головой. Не забудь `Box::new(...)` вокруг хвоста.
</details>

<details><summary>Идея 3 — <code>Counter</code> и <code>RefCell</code></summary>

`new()` создаёт `Counter { count: RefCell::new(0) }`. В `increment` возьми изменяемый
доступ и прибавь единицу: `*self.count.borrow_mut() += 1;`. В `get` возьми доступ на
чтение и верни значение: `*self.count.borrow()`. Звёздочка `*` нужна, чтобы из обёртки
(`Ref`/`RefMut`) добраться до самого `i64`.
</details>

<details><summary>Идея 2 — <code>rc_clone_count</code></summary>

```rust
let a = Rc::new(42);
let _b = Rc::clone(&a);   // второй владелец; счётчик стал 2
Rc::strong_count(&a)      // вернуть текущее число сильных ссылок
```

Важно: клон должен быть **жив** в момент вызова `strong_count`. Если ты создашь клон во
вложенном блоке `{ ... }`, по выходе из блока он уничтожится и счётчик снова упадёт. Держи
клон в переменной, которая живёт до `return`.
</details>

## Задание 5 — Разделяемое изменяемое дерево

Настоящий Rust-проект для этого паттерна — дерево, где несколько частей кода держат
ссылки на одни и те же узлы и могут добавлять детей. Это классический случай
`Rc<RefCell<T>>`.

В `src/lib.rs` объявлена структура `TreeNode` с полями `value: i64` и
`children: Vec<Rc<RefCell<TreeNode>>>`. Тебе нужно реализовать четыре функции:

- `TreeNode::new(value: i64) -> Rc<RefCell<Self>>` — создать новый узел;
- `TreeNode::add_child(parent: &Rc<RefCell<Self>>, child: Rc<RefCell<Self>>)` — добавить
  ребёнка к родителю;
- `TreeNode::sum_tree(node: &Rc<RefCell<Self>>) -> i64` — рекурсивная сумма;
- `TreeNode::depth(node: &Rc<RefCell<Self>>) -> usize` — максимальная глубина.

Тесты лежат в `tests/tree.rs`.

<details><summary>Шаг 1 — почему тип узла именно <code>Rc&lt;RefCell&lt;TreeNode&gt;&gt;</code>?</summary>

Подумай: какую проблему решает каждый слой?

- Без `Rc`: у каждого узла ровно один владелец. Но ты хочешь хранить ребёнка в `Vec`
  родителя и одновременно держать `Rc` на него снаружи (например, чтобы потом добавить
  к нему своих детей). Значит, нужно разделяемое владение → `Rc`.
- Без `RefCell`: `Rc` даёт только `&T` (общий доступ), но не `&mut T`. Добавление ребёнка
  — это мутация `children`, которая нужна, имея лишь `&Rc<TreeNode>`. Значит, нужна
  внутренняя изменяемость → `RefCell`.
</details>

<details><summary>Шаг 2 — как реализовать <code>new</code> и <code>add_child</code></summary>

`new` просто оборачивает значение в две обёртки:
```rust
Rc::new(RefCell::new(TreeNode { value, children: Vec::new() }))
```

`add_child` берёт изменяемый заём родителя и пушит ребёнка:
```rust
parent.borrow_mut().children.push(child);
```

Важно: `borrow_mut()` живёт только на время этой строки. После того как строка выполнена,
заём отпускается — следующий вызов `add_child` (или `borrow` в любом месте) пройдёт
без паники.
</details>

<details><summary>Шаг 3 — как рекурсировать, не нарушая правила займов RefCell</summary>

Главная ловушка в `sum_tree` и `depth`: если взять `borrow()` и удерживать его во время
рекурсивного вызова, можно поймать панику (или не поймать — смотря по топологии). Безопасный
подход — **взять заём, скопировать нужное, отпустить заём** до рекурсии:

```rust
// НЕ делай так — borrow живёт весь цикл рекурсии:
// let b = node.borrow();
// for child in &b.children { sum += sum_tree(child); }

// Делай так — заём живёт только в блоке:
let (value, children) = {
    let b = node.borrow();
    (b.value, b.children.clone())   // clone дёшев: клонируем Rc-указатели, не данные
};
// Теперь b отпущен. Рекурсируем безопасно:
children.iter().fold(value, |acc, child| acc + sum_tree(child))
```

`b.children.clone()` клонирует `Vec<Rc<...>>` — это только клон указателей, не глубокое
копирование узлов. Каждый `Rc` просто увеличивает счётчик на 1.
</details>
