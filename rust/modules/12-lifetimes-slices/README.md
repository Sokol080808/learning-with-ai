# Модуль 12 — Времена жизни и срезы

В модулях 03–04 ты разобрался с владением и заимствованием: у каждого значения
один владелец, а ссылки (`&`, `&mut`) дают доступ, не забирая владение. Теперь —
следующий уровень той же идеи. Большая идея модуля: **ссылка не имеет права
пережить данные, на которые смотрит**, и Rust доказывает это на этапе компиляции
с помощью *времён жизни* (lifetimes). А *срез* — это и есть типичная такая
ссылка: заимствованное «окно» в кусок данных.

Если компилятор ругается на время жизни — это не препятствие, а сработавшая
защита: он только что предотвратил *висячую ссылку*, то есть чтение уже
освобождённой памяти. В C++ такой баг ловят отладчиком в три часа ночи; в Rust
его ловит компилятор за тебя.

## Идея 1. Срезы `&[T]` и `&str` — заимствованное окно в данные

Срез — это ссылка не на одно значение, а на *непрерывный участок* чужих данных:
указатель на начало + длина. Он **ничего не копирует и ничем не владеет**.

```rust
let v = vec![10, 20, 30, 40];
let window: &[i64] = &v[1..3];   // окно в элементы 20 и 30
assert_eq!(window, &[20, 30]);

let s = String::from("hello world");
let hello: &str = &s[0..5];      // &str — это срез байтов строки (валидный UTF-8)
assert_eq!(hello, "hello");
```

ПОЧЕМУ это важно: функция, принимающая `&[i64]`, работает и с `Vec`, и с массивом,
и с под-куском любого из них — потому что смотрит на данные, а не владеет ими.
Это дешевле, чем передавать копию, и честнее, чем передавать владение.

> Подвох: `&str` режется по **байтам**, а граница должна попадать на стык символов
> UTF-8. `&"ёж"[0..1]` паникует — 'ё' занимает 2 байта. Поэтому для посимвольной
> работы берут `.chars()`, а не индексацию по байтам.

## Идея 2. Аннотации времён жизни связывают время жизни ссылок

Когда функция и принимает, и **возвращает** ссылки, компилятору иногда нужна
подсказка: к какому из входов привязан выход? Это и есть аннотация `'a`.

```rust
fn longest<'a>(a: &'a str, b: &'a str) -> &'a str {
    if a.len() >= b.len() { a } else { b }
}
```

Читается так: «существует некоторое время жизни `'a`; оба входа живут хотя бы
столько, и результат живёт ровно столько же». Аннотация **ничего не меняет в
рантайме** — это подпись под контрактом, которую компилятор проверяет.

```rust
let s1 = String::from("long string");
let result;
{
    let s2 = String::from("xyz");
    result = longest(&s1, &s2);   // result привязан к МИН. из времён жизни s1 и s2
    println!("{result}");          // ок: s2 ещё жива
}
// println!("{result}");           // НЕ скомпилируется: s2 уже мертва, а result мог
                                    // указывать на неё
```

## Идея 3. ПОЧЕМУ компилятору нужны времена жизни — предотвратить висячие ссылки

Представь, что аннотаций бы не было и компилятор разрешал такое:

```rust
fn dangling() -> &str {        // НЕ компилируется
    let s = String::from("oops");
    &s                          // вернуть ссылку на локальную строку...
}                               // ...которая прямо здесь уничтожается!
```

Возвращённая ссылка указывала бы на освобождённую память — классический
use-after-free. Времена жизни — это машина доказательств, которая делает такой
код невыразимым. Когда ты пишешь `-> &'a str`, ты обещаешь, что результат
заимствован **из чего-то, что живёт как минимум `'a`** — а локальная переменная
не живёт дольше функции, поэтому вернуть ссылку на неё нельзя.

Часто аннотации можно **опустить** — действует *элизия* (lifetime elision):
если у функции ровно один входной-ссылочный параметр, его время жизни
автоматически отдаётся всем выходным ссылкам. Поэтому `fn first_word(s: &str)
-> &str` пишется без `'a`, хотя связь там та же.

## Идея 4. Возврат ссылок из функций (и структуры со ссылками)

Возвращать ссылку безопасно ровно тогда, когда она заимствована из **входа**, а
не из локальной переменной. Та же логика — для структур, которые ХРАНЯТ ссылку:

```rust
struct Wrapper<'a> {
    text: &'a str,     // поле — заимствованное окно; владелец строки снаружи
}
```

Параметр `'a` у структуры значит: «экземпляр `Wrapper` не может пережить строку,
на которую он ссылается». Положить сюда ссылку на временное значение и
использовать обёртку позже — не выйдет, и это правильно: иначе поле `text`
указывало бы в никуда.

### Подсказки

<details><summary>1. longest — сравнение и возврат ссылки</summary>
Сравни `a.len()` и `b.len()`. Тело — одно выражение `if`/`else`, возвращающее
`a` или `b`. Никакого `.clone()` и никаких `String` — ты просто возвращаешь ту же
ссылку, что пришла. Аккуратно с равенством: контракт требует вернуть `a`, когда
длины равны (значит, условие «вернуть b» — это строгое `b.len() > a.len()`).
</details>

<details><summary>2. first_word — где заканчивается слово</summary>
Найди индекс первого пробела. Можно пройтись по байтам:
`s.as_bytes().iter().position(|&b| b == b' ')`, либо проще — `s.find(' ')`. Оба
дают `Option<usize>`. Если `Some(i)` — верни `&s[..i]`; если `None` — верни весь
`s` (пробела нет). Срез `&s[..i]` не копирует — он смотрит внутрь `s`.
</details>

<details><summary>3. first_char — символ, а не байт</summary>
Строку нельзя индексировать как `s[0]` (Rust это запрещает именно из-за UTF-8).
Возьми итератор по символам и первый из него: `self.text.chars().next()`. Он сам
вернёт `Option<char>` — ровно то, что нужно: `Some(символ)` или `None` для пустой
строки.
</details>

<details><summary>4. max_slice — максимум как ссылка</summary>
`v.iter()` даёт итератор по `&i64`. У него есть готовый потребитель `.max()`,
который возвращает `Option<&i64>` — `None` для пустого итератора и ссылку на
наибольший элемент иначе. То есть тело — это буквально `v.iter().max()`. Заметь:
тебе не пришлось писать ни цикла, ни проверки на пустоту — тип `Option` уже
выражает «может не быть результата».
</details>
