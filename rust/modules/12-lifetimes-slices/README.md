# Модуль 12 — Времена жизни и срезы

В модулях 03–04 ты разобрался с владением и заимствованием: у каждого значения
один владелец, а ссылки (`&`, `&mut`) дают доступ, не забирая владение. Теперь —
следующий уровень той же идеи. Большая идея модуля: **ссылка не имеет права
пережить данные, на которые смотрит**, и Rust доказывает это на этапе компиляции
с помощью *времён жизни* (lifetimes). А *срез* — это и есть типичная такая
ссылка: заимствованное «окно» в кусок данных.

Если компилятор ругается на время жизни — это не препятствие, а сработавшая
защита: он только что предотвратил *висячую ссылку*, то есть чтение уже
освобождённой памяти. В C++ такой баг ловят отладчиком в три часа ночи; в Rust
его ловит компилятор за тебя.

---

## Идея 1. Срезы `&[T]` и `&str` — заимствованное окно в данные

Срез — это ссылка не на одно значение, а на *непрерывный участок* чужих данных.
Под капотом срез — это **«толстый указатель» (fat pointer)**: два слова размером
с машинное слово:

```
┌─────────────────────┬───────────────┐
│ указатель на начало │ длина (байты) │
│     (ptr: *T)       │  (len: usize) │
└─────────────────────┴───────────────┘
```

Два слова — 16 байт на 64-битной машине. Именно поэтому `&[T]` и `&str`
передаются по значению без проблем: это пара (ptr, len), а не копия данных.
Он **ничего не копирует и ничем не владеет**.

```rust
let v = vec![10, 20, 30, 40];
let window: &[i64] = &v[1..3];   // окно в элементы 20 и 30
assert_eq!(window, &[20, 30]);

let s = String::from("hello world");
let hello: &str = &s[0..5];      // &str — это срез байтов строки (валидный UTF-8)
assert_eq!(hello, "hello");
```

**Контракт `as_ptr()`:** метод `as_ptr()` у среза возвращает указатель на первый
байт. Если срез `sub` получен из строки `s` операцией `&s[i..j]`, то
`sub.as_ptr() == unsafe { s.as_ptr().add(i) }` — они указывают в одну и ту же
память. Именно это проверяют тесты с `assert_eq!(r.as_ptr(), a.as_ptr())`: они
удостоверяются, что функция вернула не копию, а настоящий суб-срез.

ПОЧЕМУ это важно: функция, принимающая `&[i64]`, работает и с `Vec`, и с массивом,
и с под-куском любого из них — потому что смотрит на данные, а не владеет ими.
Это дешевле, чем передавать копию, и честнее, чем передавать владение.

> Подвох: `&str` режется по **байтам**, а граница должна попадать на стык символов
> UTF-8. `&"ёж"[0..1]` паникует — 'ё' занимает 2 байта. Поэтому для посимвольной
> работы берут `.chars()`, а не индексацию по байтам.

## Идея 2. Аннотации времён жизни связывают время жизни ссылок

Когда функция и принимает, и **возвращает** ссылки, компилятору иногда нужна
подсказка: к какому из входов привязан выход? Это и есть аннотация `'a`.

```rust
fn longest<'a>(a: &'a str, b: &'a str) -> &'a str {
    if a.len() >= b.len() { a } else { b }
}
```

Читается так: «существует некоторое время жизни `'a`; оба входа живут хотя бы
столько, и результат живёт ровно столько же». Аннотация **ничего не меняет в
рантайме** — это подпись под контрактом, которую компилятор проверяет.

```rust
let s1 = String::from("long string");
let result;
{
    let s2 = String::from("xyz");
    result = longest(&s1, &s2);   // result привязан к МИН. из времён жизни s1 и s2
    println!("{result}");          // ок: s2 ещё жива
}
// println!("{result}");           // НЕ скомпилируется: s2 уже мертва, а result мог
                                    // указывать на неё
```

### Несколько разных времён жизни: `'a, 'b`

Иногда два параметра связаны с **разными** временами жизни, никак не
пересекающимися. Тогда пишут два имени:

```rust
// Функция принимает две строки с разными временами жизни и возвращает
// ссылку, привязанную только к ПЕРВОМУ аргументу.
fn first_of<'a, 'b>(x: &'a str, _y: &'b str) -> &'a str {
    x
}
```

Здесь `'b` — время жизни `_y`. Компилятор знает: результат не может указывать на
`_y`, потому что тип `-> &'a str` не упоминает `'b`. Это значит, что `_y` может
умереть раньше результата — и это безопасно.

Когда нужно два разных имени?
- когда функция возвращает ссылку, зависящую только от одного из параметров;
- когда нужно явно показать, что два параметра не связаны между собой.

```rust
fn prefix_len<'a, 'b>(haystack: &'a str, needle: &'b str) -> &'a str {
    if haystack.starts_with(needle) {
        &haystack[..needle.len()]
    } else {
        &haystack[..0]
    }
}
// needle может умереть раньше результата — и это ок, потому что результат
// указывает внутрь haystack, а не needle.
```

## Идея 3. ПОЧЕМУ компилятору нужны времена жизни — предотвратить висячие ссылки

Представь, что аннотаций бы не было и компилятор разрешал такое:

```rust
fn dangling() -> &str {        // НЕ компилируется
    let s = String::from("oops");
    &s                          // вернуть ссылку на локальную строку...
}                               // ...которая прямо здесь уничтожается!
```

Возвращённая ссылка указывала бы на освобождённую память — классический
use-after-free. Времена жизни — это машина доказательств, которая делает такой
код невыразимым. Когда ты пишешь `-> &'a str`, ты обещаешь, что результат
заимствован **из чего-то, что живёт как минимум `'a`** — а локальная переменная
не живёт дольше функции, поэтому вернуть ссылку на неё нельзя.

### Типичные ошибки компилятора

**E0106 — отсутствует аннотация времени жизни:**

```rust
struct Bad {
    text: &str,  // ошибка E0106: у &str нет времени жизни
}
// Исправление: struct Good<'a> { text: &'a str }
```

Компилятор не знает, как долго живёт строка, на которую ссылается `text` — нужно
связать её через параметр структуры.

**E0597 — ссылка не живёт достаточно долго:**

```rust
fn demo() -> &str {
    let s = String::from("hello");
    &s              // E0597: `s` не живёт достаточно долго
}                   // s уничтожается здесь, но мы пытаемся вернуть ссылку на неё
```

Или в более тонком варианте:

```rust
let r;
{
    let s = String::from("inner");
    r = &s;         // E0597: `s` не живёт достаточно долго
}
println!("{r}");    // s уже мертва — висячая ссылка!
```

Компилятор покажет именно это место и объяснит, почему время жизни `s` короче,
чем нужно. Это не баг компилятора — это он делает свою работу.

## Идея 4. Элизия времён жизни — три правила

Часто аннотации можно **опустить** — действует *элизия* (lifetime elision).
Компилятор применяет три правила по порядку:

**Правило 1.** Каждому входному параметру-ссылке назначается своё уникальное время
жизни.

```rust
// Написано:
fn f(x: &str, y: &str) -> &str { ... }
// Компилятор раскрывает до:
fn f<'a, 'b>(x: &'a str, y: &'b str) -> &str { ... }
```

**Правило 2.** Если после правила 1 входной параметр ровно один — его время жизни
присваивается всем выходным ссылкам.

```rust
// Написано:
fn first_word(s: &str) -> &str { ... }
// После правил 1+2 компилятор видит:
fn first_word<'a>(s: &'a str) -> &'a str { ... }
// Явная 'a не нужна — и связь та же.
```

**Правило 3.** Если хотя бы один входной параметр — `&self` или `&mut self`,
то его время жизни присваивается всем выходным ссылкам.

```rust
impl Wrapper<'_> {
    // Написано:
    fn get(&self) -> &str { self.text }
    // После правил 1+3:
    fn get<'a>(&'a self) -> &'a str { self.text }
}
```

Если после применения всех трёх правил у выходной ссылки так и нет времени жизни
— это ошибка компиляции E0106, и нужно указать его явно.

> Практическое правило: если у функции **один ссылочный вход**, аннотации не нужны.
> Если **два и больше** — и ты возвращаешь ссылку — нужно явно указать, из какого
> входа она берётся.

## Идея 5. `'static` — время жизни, живущее вечно

Особое время жизни `'static` означает: ссылка действительна на протяжении **всей
программы**. Бывает двух видов:

```rust
// 1. Строковые литералы — всегда 'static (зашиты в бинарник)
let s: &'static str = "hello, world";

// 2. Явно объявленные статические данные
static GREETING: &str = "hi";  // GREETING: &'static str
```

`'static` — это не «лучше» или «безопаснее»; это просто самый длинный
возможный срок жизни. В generic-коде иногда требуют `T: 'static` — это значит
«тип T не содержит коротких ссылок»; его значения можно хранить вечно.

```rust
fn store<T: 'static>(val: T) {
    // val не содержит ссылок, которые могли бы устареть — его можно
    // безопасно положить в глобальную структуру, поток и т.д.
}
```

Когда ты видишь `'static` в сообщении об ошибке, компилятор обычно говорит:
«это значение живёт только часть программы, а ты просишь что-то, что живёт вечно».

## Идея 6. Времена жизни на `impl`-блоках

Когда у структуры есть параметр времени жизни, он повторяется в `impl`:

```rust
struct Wrapper<'a> {
    text: &'a str,
}

impl<'a> Wrapper<'a> {
    // Здесь 'a доступна для всего блока
    pub fn get_text(&self) -> &'a str {
        self.text
    }
}
```

Читается так: «для любого времени жизни `'a`: реализуем методы для
`Wrapper<'a>`». Без `impl<'a>` нельзя написать `Wrapper<'a>` в заголовке.

Тонкость: `&self` в методе — своё, анонимное время жизни. Когда метод возвращает
`&'a str` (не `&str`), он явно обещает: «результат живёт столько же, сколько поле
`text`, а не сколько ссылка `self`». Это важно, если `self` — временная ссылка:

```rust
fn example<'a>(w: &Wrapper<'a>) -> &'a str {
    w.get_text()   // живёт 'a, даже если &w умрёт раньше
}
```

## Идея 7. NLL — неэксклюзивные лексические времена жизни

До Rust 2018 времена жизни были *лексическими*: ссылка «жила» до конца блока
`{}`, в котором она объявлена. Это приводило к ложным ошибкам:

```rust
// Rust 2015 — НЕ компилировалось:
let mut v = vec![1, 2, 3];
let r = &v[0];   // ссылка начинается
println!("{r}"); // ссылка используется
v.push(4);       // ошибка: v уже заимствован!
```

С **Non-Lexical Lifetimes (NLL)** — начиная с Rust 2018 — компилятор отслеживает
время жизни ссылки точнее: до **последнего использования**, а не до конца блока:

```rust
// Rust 2018+: ок
let mut v = vec![1, 2, 3];
let r = &v[0];
println!("{r}");  // последнее использование r — здесь
v.push(4);        // r уже не живёт, мутировать можно
```

NLL — это чисто компиляторная оптимизация анализа. В рантайме ничего не меняется:
никаких счётчиков ссылок, никаких флагов. Компилятор просто умеет точнее строить
«граф потока» и определять, где ссылка реально нужна.

Когда компилятор ругается на конфликт между `&` и `&mut` в одном блоке, стоит
спросить: «моё последнее использование ссылки действительно пересекается с
мутацией, или компилятор ошибается?» Чаще всего перестановка строк — или вынос
использования в отдельный `let` — решает проблему.

## Идея 8. Возврат ссылок из функций (и структуры со ссылками)

Возвращать ссылку безопасно ровно тогда, когда она заимствована из **входа**, а
не из локальной переменной. Та же логика — для структур, которые ХРАНЯТ ссылку:

```rust
struct Wrapper<'a> {
    text: &'a str,     // поле — заимствованное окно; владелец строки снаружи
}
```

Параметр `'a` у структуры значит: «экземпляр `Wrapper` не может пережить строку,
на которую он ссылается». Положить сюда ссылку на временное значение и
использовать обёртку позже — не выйдет, и это правильно: иначе поле `text`
указывало бы в никуда.

### Задание 5 — `words` и `longest_word`: срезы без копирования

Когда функция возвращает не одну, а **несколько** ссылок сразу, их время жизни
тоже нужно связать с входом. Здесь тебе предстоит реализовать:

- `fn words(s: &str) -> Vec<&str>` — разбить строку по пробелам; каждый элемент
  вектора — это `&str`, смотрящий **внутрь** исходной строки `s` (ноль
  копирований, ноль выделений строк).

- `fn longest_word<'a>(s: &'a str) -> Option<&'a str>` — вернуть самое длинное
  слово. Если слов нет — `None`. При равной длине — первое из найденных.

Обрати внимание на сигнатуру `longest_word`: явная `'a` здесь **необходима** —
у функции два смысловых «выхода» (`Some(&str)` и `None`), и компилятор должен
знать, что `Some`-вариант ссылается именно на `s`.

После реализации запусти тесты из `tests/words.rs` — они проверят, что каждый
элемент `words(s)` является настоящим суб-срезом `s` (проверка через `as_ptr()`
и диапазон адресов).

### Подсказки к заданиям 1–4

<details><summary>1. longest — сравнение и возврат ссылки</summary>
Сравни `a.len()` и `b.len()`. Тело — одно выражение `if`/`else`, возвращающее
`a` или `b`. Никакого `.clone()` и никаких `String` — ты просто возвращаешь ту же
ссылку, что пришла. Аккуратно с равенством: контракт требует вернуть `a`, когда
длины равны (значит, условие «вернуть b» — это строгое `b.len() > a.len()`).
</details>

<details><summary>2. first_word — где заканчивается слово</summary>
Найди индекс первого пробела. Можно пройтись по байтам:
`s.as_bytes().iter().position(|&b| b == b' ')`, либо проще — `s.find(' ')`. Оба
дают `Option<usize>`. Если `Some(i)` — верни `&s[..i]`; если `None` — верни весь
`s` (пробела нет). Срез `&s[..i]` не копирует — он смотрит внутрь `s`.
</details>

<details><summary>3. first_char — символ, а не байт</summary>
Строку нельзя индексировать как `s[0]` (Rust это запрещает именно из-за UTF-8).
Возьми итератор по символам и первый из него: `self.text.chars().next()`. Он сам
вернёт `Option<char>` — ровно то, что нужно: `Some(символ)` или `None` для пустой
строки.
</details>

<details><summary>4. max_slice — максимум как ссылка</summary>
`v.iter()` даёт итератор по `&i64`. У него есть готовый потребитель `.max()`,
который возвращает `Option<&i64>` — `None` для пустого итератора и ссылку на
наибольший элемент иначе. То есть тело — это буквально `v.iter().max()`. Заметь:
тебе не пришлось писать ни цикла, ни проверки на пустоту — тип `Option` уже
выражает «может не быть результата».
</details>

### Подсказки к заданию 5

<details><summary>5а. words — как разбить без копирования</summary>
Пробел — это символ, отличный от буквы. Стандартный итератор
`s.split_whitespace()` уже делает именно то, что нужно — только собери его в
`Vec`. Каждый элемент итератора уже является `&str` — суб-срезом `s`. Убедись
мысленно: тип элемента — `&'_ str`, где `'_` привязано к `s`. Именно это гарантирует,
что вектор не переживёт строку, из которой он взят.
</details>

<details><summary>5б. longest_word — почему нужна явная 'a</summary>
Попробуй написать `fn longest_word(s: &str) -> Option<&str>` без явной `'a`.
Применим правила элизии: правило 1 даёт входу анонимное время жизни; правил 2 и 3
нет (входов несколько? нет — один; self нет). Итого правило 2 сработает: анонимное
время жизни входа достанется выходу. Значит элизия РАБОТАЕТ для такой сигнатуры!
Можешь попробовать оба варианта — с явной `'a` и без. Затем подумай: зачем тогда
здесь написана явная аннотация? Ответ — читаемость и явность контракта.
</details>

<details><summary>5в. longest_word — как найти самое длинное слово</summary>
Получи итератор слов (через `words()` или `split_whitespace()`), затем найди
максимум по длине байт. `Iterator::max_by_key` принимает замыкание, возвращающее
ключ сравнения. Помни контракт: при равной длине — первое слово. Как это соотносится
с тем, что `max_by_key` возвращает при равных ключах?
