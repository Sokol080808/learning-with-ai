# Модуль 01 — Переменные, типы, функции

Большая идея: в Rust значение по умолчанию **неизменяемо**, типы статичны и проверяются
на этапе компиляции, а функция — это **выражение, которое что-то возвращает**. Эти три
вещи звучат как мелочи, но именно из них растёт ощущение, что «если скомпилировалось —
скорее всего, работает». В этом модуле мы пишем пять функций, но по дороге встречаем все
базовые кирпичики, на которые потом будет опираться весь курс.

Здесь ещё нет владения и заимствования (это модули 03–04). Все наши типы — простые
**скаляры** (числа, `bool`) и коллекция `Vec`, которые работают без лишней магии.
Поэтому пока ни одного `&` — наслаждайся: дальше будет интереснее.

## Идея 1. `let`, неизменяемость по умолчанию, `mut` и затенение

```rust
let x = 5;          // неизменяемая переменная
// x = 6;           // ОШИБКА компиляции: cannot assign twice to immutable variable

let mut y = 5;      // mut — «разрешаю менять»
y = 6;              // ок

let x = x + 1;      // затенение (shadowing): новая переменная с тем же именем
let x = x * 2;      // снова shadowing — теперь x == 12
```

**ПОЧЕМУ так.** Неизменяемость по умолчанию — это не вредность компилятора, а защита:
большинство переменных менять не нужно, и `let` без `mut` фиксирует это намерение. Если
кто-то (включая тебя через месяц) случайно присвоит заново — компилятор остановит.

**Не путай `mut` и затенение.** `mut` меняет *то же самое* значение в той же ячейке.
Затенение (`let x = ...` поверх старого `x`) создаёт **новую** переменную — можно даже
сменить тип:

```rust
let spaces = "   ";          // &str
let spaces = spaces.len();   // теперь usize — старое имя «затенено»
```

С `mut` так нельзя: тип переменной менять запрещено. Затенение — это про «переиспользовать
удобное имя», а не «мутировать».

### Затенение в блочных областях видимости

Важный нюанс: затенение работает не только в «плоском» коде, но и во вложенных блоках
`{ ... }`. При этом внешняя переменная не уничтожается — она снова становится видна, как
только блок закрывается:

```rust
let x = 5;
{
    let x = x * 10;  // внутри блока: x == 50
    println!("{x}"); // 50
}
println!("{x}");     // снова 5 — внешний x «вернулся»
```

Это отличается от `mut`: при `mut` нет никакого «старого значения», которое вернётся.
Блочное затенение полезно, когда нужно временно переименовать или преобразовать значение,
не затрагивая остальной код функции.

## Идея 2. `const` vs `let`

`const` — это не переменная, а **именованная константа**, вычисленная на этапе компиляции.
Её нельзя объявить без явного типа, нельзя задать через `mut`, и она живёт всю программу
(не привязана к блоку).

```rust
const MAX_POINTS: u32 = 100_000;  // тип обязателен, значение — константное выражение
let score = 42;                    // тип выводится, значение — рантайм
```

**Когда использовать `const`.** Магические числа в бизнес-логике (лимиты, константы
физических формул) лучше называть через `const` — тогда компилятор может подставить их
прямо в машинный код без выделения памяти. `let` нужен, когда значение вычисляется во
время работы программы.

```rust
const FREEZING_F: f64 = 32.0;

fn to_fahrenheit(c: f64) -> f64 {
    c * 9.0 / 5.0 + FREEZING_F  // читаемо и без «магической 32»
}
```

## Идея 3. Скалярные типы и вывод типов

Базовые скаляры Rust: целые `i32`, `i64`, `u32`, `u64`, … ; числа с плавающей точкой
`f64` (по умолчанию) и `f32`; логический `bool`; символ `char` (целый Unicode-скаляр, в
одинарных кавычках).

```rust
let a = 42;        // вывод типа -> i32 (целое «по умолчанию»)
let b = 3.14;      // -> f64
let ok = true;     // -> bool
let letter = 'R';  // -> char  (НЕ строка! у строки двойные кавычки)

let big: i64 = 10_000_000_000;  // явная аннотация, когда i32 мало (подчёркивания — просто для глаз)
```

**ПОЧЕМУ важен тип.** Компилятор обычно **выводит** тип сам, но иногда его надо подсказать
(`let big: i64 = ...`). В этом модуле сигнатуры функций уже заданы — `i64`, `f64`, `bool` —
и это сразу фиксирует типы аргументов и результата. Внутри тела вывод типов сделает
остальное.

### `usize` и `isize` — типы для индексов и размеров

Помимо `i32`, `i64` и компании, есть два специальных типа: `usize` (беззнаковый) и `isize`
(знаковый). Их размер совпадает с разрядностью платформы: 8 байт на 64-битной машине,
4 байта на 32-битной.

```rust
let v = vec![10, 20, 30];
let i: usize = 1;
println!("{}", v[i]);  // обращение по индексу требует именно usize
```

**Почему это важно.** Индексы массивов и размеры коллекций в Rust всегда `usize`. Если
попробовать проиндексировать вектор через `i32` или `i64`, компилятор откажет. Это
источник частых ошибок для новичков — нужно писать явное `as usize`:

```rust
let n: i64 = 2;
let v = vec![1, 2, 3];
println!("{}", v[n as usize]);  // ок
```

`isize` встречается реже — в основном в низкоуровневом коде с указателями и смещениями.

### Целочисленные литералы: формы записи и суффиксы типов

Rust позволяет записывать числа в разных системах счисления и явно указывать тип прямо
в литерале:

```rust
let decimal  = 255;          // десятичный
let hex      = 0xFF;         // шестнадцатеричный (0xff тоже ок)
let octal    = 0o377;        // восьмеричный
let binary   = 0b1111_1111;  // двоичный (подчёркивания — для читаемости)
let byte     = b'A';         // байт (u8) из ASCII-символа

// суффиксы типов:
let a = 42u8;    // u8, значение 42
let b = 42i64;   // i64, значение 42
let c = 42_u32;  // можно писать с подчёркиванием перед суффиксом
```

Это пригодится, когда компилятор жалуется на «mismatched types» при передаче числа
в функцию с конкретным типом — достаточно добавить суффикс к литералу.

### Целочисленное переполнение: debug vs release

Rust защищает от тихих переполнений, но поведение зависит от режима сборки:

- **debug** (по умолчанию): переполнение вызывает **панику** во время выполнения.
  Программа падает с сообщением `attempt to add with overflow`.
- **release** (`cargo build --release`): переполнение **оборачивается** по модулю типа
  (как в C), без паники. `255u8 + 1 == 0`.

```rust
let x: u8 = 255;
let y = x + 1;  // panic в debug! В release: y == 0
```

```
thread 'main' panicked at 'attempt to add with overflow'
```

**Что делать правильно.** Если переполнение — штатная ситуация, используй явные
«оборачивающие» методы:

```rust
let z = x.wrapping_add(1);   // всегда оборачивается: 255 -> 0
let z = x.saturating_add(1); // насыщается: 255 -> 255
let z = x.checked_add(1);    // возвращает Option<u8>: None при переполнении
```

Никогда не полагайся на «release оборачивает» в бизнес-логике — это источник
трудноуловимых багов. Если переполнение возможно — обработай его явно.

## Идея 4. Выражения против инструкций (главная привычка Rust!)

В Rust почти всё — **выражение**, у которого есть значение. Это меняет то, как пишут функции.

```rust
fn double(n: i64) -> i64 {
    n * 2          // НЕТ точки с запятой -> это возвращаемое выражение функции
}

fn double_v2(n: i64) -> i64 {
    return n * 2;  // можно и так, но в Rust обычно пишут без return
}
```

Даже `if` — выражение и возвращает значение:

```rust
let sign = if n >= 0 { 1 } else { -1 };   // ветви дают значение, оно ложится в sign
```

**ПОЧЕМУ это ловушка для новичка.** Последняя строка тела **без `;`** — это результат
функции. Стоит поставить `;` — и строка превращается в **инструкцию**, её значение
выбрасывается, тело «возвращает» пустоту `()`, и компилятор ругается на несовпадение типов:

```rust
fn double(n: i64) -> i64 {
    n * 2;         // ОШИБКА: с точкой с запятой это инструкция -> функция возвращает (), а обещали i64
}
```

Запомни это сейчас — половина «странных» ошибок в первых модулях именно про лишнюю или
забытую `;`.

### Тип `()` — «пустое» значение

`()` (читается «unit» или «пустой кортеж») — это **настоящий тип** в Rust, а не отсутствие
значения. Функция без `-> ...` неявно возвращает `()`. Инструкция (строка с `;`) тоже
«имеет значение» типа `()`.

Ошибка, которую ты почти наверняка встретишь:

```
error[E0308]: mismatched types
  --> src/main.rs:2:5
   |
   | fn double(n: i64) -> i64 {
   |                       --- expected `i64` because of return type
   |     n * 2;
   |          ^ expected `i64`, found `()`
```

Компилятор говорит: «тело вернуло `()`, а сигнатура обещает `i64`». Решение — убрать `;`
с последней строки.

`()` появляется и в других местах: методы типа `println!()` возвращают `()`, `for`-цикл
возвращает `()`, `push` на `Vec` тоже возвращает `()`. Это нормально — просто «нет
полезного результата».

## Идея 5. Нет неявных числовых преобразований — нужен `as`

```rust
let n: i64 = 5;
let x: f64 = n;          // ОШИБКА: i64 и f64 — разные типы, Rust сам НЕ преобразует
let x: f64 = n as f64;   // ок: явное преобразование через `as`
```

**ПОЧЕМУ так строго.** В C число молча превратится из одного типа в другой — иногда с
потерей данных, о которой ты не узнаешь. Rust требует написать `as` явно: преобразование
есть, но оно **видно в коде**. Это пригодится в `to_fahrenheit`, если ты вдруг захочешь
смешать целые и дробные — но проще сразу считать во `f64` (см. подсказку к заданию).

И помни про целочисленное деление: `9 / 5 == 1`, а `9.0 / 5.0 == 1.8`. Тип литералов
решает всё.

### Подсказки к базовым функциям

<details><summary>to_fahrenheit — про целочисленное деление</summary>

Главная ловушка — не сам алгоритм, а типы. Если напишешь `9 / 5`, Rust посчитает это
целыми и получит `1`, а не `1.8`. Считай во `f64`: например `c * 9.0 / 5.0 + 32.0`.
Все входные/выходные значения уже `f64`, так что `as` тебе тут не понадобится.
</details>

<details><summary>is_even — оператор остатка</summary>

Остаток от деления — оператор `%`. Число чётное, когда `n % 2 == 0`. Это выражение типа
`bool` — его и верни (последней строкой, без `;`). Для отрицательных это тоже верно:
`-2 % 2 == 0`.
</details>

<details><summary>min_of_three — два пути</summary>

Путь первый, «руками»: сравни через `if`/`else`, помня, что `if` — выражение.
Путь второй, короткий: у `i64` есть метод `.min(...)`, и его можно сцепить —
`a.min(b).min(c)`. Оба решения проходят тесты; выбери тот, что тебе понятнее, и подумай,
почему `.min` читается лучше при росте числа аргументов.
</details>

<details><summary>abs_diff — где взять модуль</summary>

Вычти одно из другого и возьми модуль разницы: у `i64` есть метод `.abs()`. Подумай, что
будет без `.abs()`, если `a < b`: разница окажется отрицательной, а контракт обещает
неотрицательный результат. Поэтому модуль обязателен.
</details>
