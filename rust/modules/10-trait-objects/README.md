# Модуль 10 — Трейт-объекты и динамическая диспетчеризация

Большая идея: трейт описывает **поведение** («что объект умеет»), не привязываясь к
конкретному типу. В модуле 09 мы раздавали это поведение через обобщения — там тип
известен на этапе компиляции. Здесь мы делаем шаг дальше: складываем в один список
**разные** типы, которые объединяет лишь общий трейт, и решаем, чью реализацию вызвать,
уже **в рантайме**. Это называется *трейт-объект* (`dyn Trait`), а механизм — *динамическая
диспетчеризация*. Заодно подружимся с двумя трейтами из стандартной библиотеки — `Display`
(как тип печатается) и `Iterator` (как по типу итерируются).

## Идея 1. `dyn Trait` и `Box<dyn Trait>` — динамическая диспетчеризация

Представь, что у тебя круги и квадраты, и ты хочешь сложить их в один `Vec`. Беда: это
разные типы, у них разный размер в памяти. `Vec<T>` хранит элементы одного типа подряд —
сюда «и круг, и квадрат» не положишь.

Решение — спрятать конкретный тип за общим трейтом:

```rust
trait Shape { fn area(&self) -> f64; }

let shapes: Vec<Box<dyn Shape>> = vec![
    Box::new(Circle { r: 1.0 }),
    Box::new(Square { side: 2.0 }),
];
```

`dyn Shape` — это **трейт-объект**: «что-то, что умеет `Shape`, а что именно — неважно».
У трейт-объекта нет фиксированного размера (круг и квадрат занимают разное место),
поэтому его нельзя положить на стек напрямую — его кладут за указатель. Самый частый
указатель для владения — `Box`, отсюда `Box<dyn Shape>`.

ПОЧЕМУ это работает: каждый `Box<dyn Shape>` несёт с собой не только данные, но и
скрытую таблицу методов (*vtable*) — указатели на нужные реализации. Когда ты пишешь
`shape.area()`, программа в рантайме смотрит в эту таблицу и прыгает в правильный `area`.
Отсюда название — *динамическая* (рантайм) диспетчеризация.

> Подвох: за гибкость платишь крохой производительности (лишний прыжок по указателю) и
> теряешь возможность инлайнинга. Для большинства задач это незаметно. Выбирай `dyn`,
> когда тебе действительно нужна разнородная коллекция или единый тип возврата.

## Идея 2. Статическая (обобщения) vs динамическая (`dyn`) диспетчеризация

Оба варианта дают «полиморфизм», но решают, *какой код звать*, в разные моменты:

```rust
fn print_area_static<T: Shape>(s: &T)   { /* ... s.area() ... */ } // обобщение
fn print_area_dynamic(s: &dyn Shape)     { /* ... s.area() ... */ } // трейт-объект
```

- **Статическая** (`<T: Shape>`): компилятор на этапе сборки генерирует ОТДЕЛЬНУЮ копию
  функции под каждый реальный тип (это называется *мономорфизация*). Быстро как ручной
  код, всё можно заинлайнить — но один вызов работает строго с одним типом, и бинарник
  чуть пухнет.
- **Динамическая** (`dyn Shape`): одна копия функции на всех; нужная реализация ищется в
  рантайме. Чуть медленнее, зато одна функция/коллекция обслуживает РАЗНЫЕ типы сразу.

ПОЧЕМУ это важно: «положить круги и квадраты в один `Vec`» обобщениями не сделать —
`Vec<T>` всё ещё один тип. Тут нужен именно `Vec<Box<dyn Shape>>`. Правило большого
пальца: бери обобщения по умолчанию, переходи на `dyn`, когда нужна разнородность.

## Идея 3. Стандартный трейт `Display` — как твой тип печатается

`Display` отвечает за «человеческий» вывод через `{}` в `println!`/`format!`. Реализуешь
один метод `fmt` — и твой тип печатается как родной:

```rust
use std::fmt;

impl fmt::Display for Point {
    fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result {
        write!(f, "({}, {})", self.x, self.y)
    }
}

let p = Point { x: 1.0, y: 2.0 };
println!("{}", p);          // (1, 2)
let s = format!("{}", p);   // s == "(1, 2)"
```

ПОЧЕМУ через `write!`, а не `return String`: `f` — это «приёмник», в который понемногу
пишется итоговый текст; так формат работает без лишних аллокаций. Возвращай ровно то, что
вернул `write!` — это `fmt::Result`.

Приятный бонус: как только есть `impl Display`, у типа **автоматически** появляется метод
`.to_string()`. Бесплатно, из самого факта реализации трейта.

> Замечание про числа: `f64` через `Display` печатает `1.0` как `1`, а `2.5` как `2.5`.
> Поэтому `Point { x: 1.0, y: 2.0 }` даёт `"(1, 2)"`, а не `"(1.0, 2.0)"`. Учитывай это.

## Идея 4. Стандартный трейт `Iterator` — реализуй для своего типа

`Iterator` — сердце циклов и комбинаторов в Rust. Чтобы сделать свой тип итерируемым,
достаточно реализовать ОДИН метод — `next`:

```rust
struct Counter { count: u32 }

impl Iterator for Counter {
    type Item = u32;                       // что выдаём
    fn next(&mut self) -> Option<u32> {    // следующий элемент или None
        if self.count < 5 {
            self.count += 1;
            Some(self.count)               // 1, 2, 3, 4, 5
        } else {
            None                           // последовательность кончилась
        }
    }
}
```

ПОЧЕМУ это так мощно: реализовав только `next`, ты бесплатно получаешь ВЕСЬ арсенал —
`for x in counter`, `.map`, `.filter`, `.sum`, `.collect` и десятки других. Все они
построены поверх `next`. Это та же идея, что в C++ с диапазонами, только аккуратнее:
`None` явно сигналит конец, перепутать границы сложнее.

```rust
let total: u32 = Counter::new().sum();     // 1+2+3+4+5 = 15
```

### Подсказки

<details><summary>1. area() для Circle и Square</summary>
Площадь круга — π·r². Число π бери из стандартной библиотеки:
<code>std::f64::consts::PI</code>. Площадь квадрата — <code>self.side * self.side</code>.
</details>

<details><summary>1. total_area по срезу боксов</summary>
Заведи сумму <code>let mut sum = 0.0;</code>, пройди <code>for s in shapes</code> и
добавляй <code>s.area()</code>. Здесь <code>s</code> — это <code>&Box&lt;dyn Shape&gt;</code>,
но метод <code>area()</code> вызывается напрямую: Rust сам разыменует бокс. Когда станет
привычно — то же одной строкой: <code>shapes.iter().map(|s| s.area()).sum()</code>.
</details>

<details><summary>2. Display::fmt</summary>
Тело — ровно одна строка:
<code>write!(f, "({}, {})", self.x, self.y)</code>.
Возвращай результат <code>write!</code> как есть (не ставь <code>;</code> в конце, чтобы
он стал возвращаемым значением). Не забудь <code>use std::fmt;</code> в шапке — он уже там.
</details>

<details><summary>3. Iterator::next и new</summary>
В <code>new</code> верни <code>Counter { count: 0 }</code>. В <code>next</code>: если
<code>self.count &lt; 5</code> — увеличь <code>self.count</code> на 1 и верни
<code>Some(self.count)</code>; иначе верни <code>None</code>. Так выйдет именно
1, 2, 3, 4, 5, затем бесконечный <code>None</code>.
</details>

<details><summary>3. counter_sum</summary>
<code>Counter::new().sum()</code>. Тип результата <code>u32</code> Rust выведет из
сигнатуры функции. Метод <code>.sum()</code> доступен только потому, что ты реализовал
<code>next</code> — почувствуй связь.
</details>
