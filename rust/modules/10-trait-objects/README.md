# Модуль 10 — Трейт-объекты и динамическая диспетчеризация

Большая идея: трейт описывает **поведение** («что объект умеет»), не привязываясь к
конкретному типу. В модуле 09 мы раздавали это поведение через обобщения — там тип
известен на этапе компиляции. Здесь мы делаем шаг дальше: складываем в один список
**разные** типы, которые объединяет лишь общий трейт, и решаем, чью реализацию вызвать,
уже **в рантайме**. Это называется *трейт-объект* (`dyn Trait`), а механизм — *динамическая
диспетчеризация*. Заодно подружимся с двумя трейтами из стандартной библиотеки — `Display`
(как тип печатается) и `Iterator` (как по типу итерируются).

## Идея 1. `dyn Trait` и `Box<dyn Trait>` — динамическая диспетчеризация

Представь, что у тебя круги и квадраты, и ты хочешь сложить их в один `Vec`. Беда: это
разные типы, у них разный размер в памяти. `Vec<T>` хранит элементы одного типа подряд —
сюда «и круг, и квадрат» не положишь.

Решение — спрятать конкретный тип за общим трейтом:

```rust
trait Shape { fn area(&self) -> f64; }

let shapes: Vec<Box<dyn Shape>> = vec![
    Box::new(Circle { r: 1.0 }),
    Box::new(Square { side: 2.0 }),
];
```

`dyn Shape` — это **трейт-объект**: «что-то, что умеет `Shape`, а что именно — неважно».
У трейт-объекта нет фиксированного размера (круг и квадрат занимают разное место),
поэтому его нельзя положить на стек напрямую — его кладут за указатель. Самый частый
указатель для владения — `Box`, отсюда `Box<dyn Shape>`.

ПОЧЕМУ это работает: каждый `Box<dyn Shape>` несёт с собой не только данные, но и
скрытую таблицу методов (*vtable*) — указатели на нужные реализации. Когда ты пишешь
`shape.area()`, программа в рантайме смотрит в эту таблицу и прыгает в правильный `area`.
Отсюда название — *динамическая* (рантайм) диспетчеризация.

> Подвох: за гибкость платишь крохой производительности (лишний прыжок по указателю) и
> теряешь возможность инлайнинга. Для большинства задач это незаметно. Выбирай `dyn`,
> когда тебе действительно нужна разнородная коллекция или единый тип возврата.

### Под капотом: жирный указатель (fat pointer)

`Box<dyn Trait>` — это не обычный восьмибайтный указатель. Это **жирный указатель**
(fat pointer): два слова вместо одного.

```
Box<dyn Shape>
┌────────────────┬────────────────┐
│  data ptr      │  vtable ptr    │
│  → Circle/...  │  → {area, ...} │
└────────────────┴────────────────┘
  8 байт            8 байт
```

- **data ptr** указывает на сами данные объекта (аллоцированные на куче).
- **vtable ptr** указывает на таблицу виртуальных функций — статически выделенный блок
  с указателями на методы трейта плюс размер/выравнивание типа (для `drop`).

Когда вызывается `shape.area()`, компилятор вставляет примерно такой код:
```
let area_fn = shape.vtable_ptr->area;  // читаем из vtable
area_fn(shape.data_ptr)                // вызываем через указатель
```

То же самое справедливо для `&dyn Trait` — это тоже жирный указатель (два слова на
стеке), только без владения.

```rust
fn print_area(s: &dyn Shape) {
    println!("{}", s.area());
}
// &dyn Shape = (data ptr, vtable ptr) — два слова
```

### Три формы трейт-объектов: `&dyn`, `Box<dyn>`, `Rc<dyn>`

Все три — жирные указатели, но с разной семантикой владения:

| Форма           | Семантика          | Когда использовать                             |
|-----------------|--------------------|------------------------------------------------|
| `&dyn Trait`    | заимствование      | Краткосрочный доступ, не нужно хранить         |
| `Box<dyn Trait>`| единственное владение | Коллекции, возврат из функции, долгая жизнь |
| `Rc<dyn Trait>` | разделённое владение | Несколько мест держат одного логгера/обработчика |

```rust
// &dyn — просто смотрим
fn describe(s: &dyn Shape) -> String {
    format!("area={}", s.area())
}

// Box<dyn> — владеем и храним
let owned: Box<dyn Shape> = Box::new(Circle { r: 1.0 });

// Rc<dyn> — несколько мест разделяют один объект
use std::rc::Rc;
let shared: Rc<dyn Shape> = Rc::new(Square { side: 2.0 });
let shared2 = Rc::clone(&shared); // второй владелец
```

Правило выбора: начинай с `&dyn`, если достаточно заимствования; переходи на `Box<dyn>`,
когда нужно хранить или возвращать; используй `Rc<dyn>` (или `Arc<dyn>` в потоках),
когда несколько мест должны держать один и тот же объект.

### Объектная безопасность (object safety)

Не каждый трейт можно превратить в `dyn Trait`. Трейт должен быть **объектно-безопасным**
(object-safe). Правило простое: компилятор должен уметь построить vtable — таблицу
указателей на методы. Это невозможно в двух случаях:

**1. Методы, возвращающие `Self`:**

```rust
trait Clone {
    fn clone(&self) -> Self; // Self = неизвестный конкретный тип
}
// Box<dyn Clone> — ОШИБКА КОМПИЛЯЦИИ: not object safe
```

Почему: vtable хранит указатель на `clone`, но не знает размер возвращаемого `Self`.
Разные типы (`Circle`, `Square`) возвращают значения разного размера — как положить их
в один указатель?

**2. Методы с обобщёнными параметрами (generic methods):**

```rust
trait Converter {
    fn convert<T>(&self) -> T; // T неизвестен при построении vtable
}
// Box<dyn Converter> — ОШИБКА: не объектно-безопасен
```

Почему: для каждого `T` нужна отдельная запись в vtable. Но vtable строится в момент
компиляции, а `T` может быть любым — бесконечная таблица не помещается.

**Правило: пиши трейты для `dyn` без `Self` в возвращаемых позициях и без generics.**

```rust
// Объектно-безопасный трейт:
trait Logger {
    fn log(&self, message: &str); // нет Self, нет generics — всё хорошо
}
// Box<dyn Logger> — работает!
```

### Эвакюационный люк: `where Self: Sized`

Иногда трейт задуман в основном для `dyn`-использования, но тебе хочется добавить в него
один «удобный» метод с обобщённым параметром или возвращающий `Self` — и не ломать
объектную безопасность всего трейта. Выход: пометить такой метод ограничением
`where Self: Sized`. Это говорит компилятору: «метод доступен только тогда, когда
конкретный тип известен статически, — в vtable ему не место».

```rust
trait Shape {
    fn area(&self) -> f64; // обычный метод — идёт в vtable

    // Метод с generic-параметром: в vtable не попадёт,
    // но трейт всё равно остаётся объектно-безопасным.
    fn cast_to<T: From<f64>>(&self) -> T
    where
        Self: Sized, // ← эвакюационный люк
    {
        T::from(self.area())
    }
}

let shapes: Vec<Box<dyn Shape>> = vec![Box::new(Circle { r: 1.0 })];
shapes[0].area(); // ✅ работает через vtable
// shapes[0].cast_to::<f32>(); // ❌ — Sized не выполнено для dyn Shape (и не нужно)
```

ПОЧЕМУ это безопасно: `dyn Shape` — тип без фиксированного размера (`!Sized`), поэтому
ограничение `where Self: Sized` автоматически исключает его из множества вызывающих. Компилятор
просто не разрешит вызывать помеченный метод через трейт-объект — и именно поэтому его
не нужно класть в vtable.

> Подвох: если ты пометил метод `where Self: Sized`, через `Box<dyn Shape>` он недоступен.
> Не жди, что это «срежет путь» к нему при динамической диспетчеризации — метода там нет.

## Идея 2. Статическая (обобщения) vs динамическая (`dyn`) диспетчеризация

Оба варианта дают «полиморфизм», но решают, *какой код звать*, в разные моменты:

```rust
fn print_area_static<T: Shape>(s: &T)   { /* ... s.area() ... */ } // обобщение
fn print_area_dynamic(s: &dyn Shape)     { /* ... s.area() ... */ } // трейт-объект
```

- **Статическая** (`<T: Shape>`): компилятор на этапе сборки генерирует ОТДЕЛЬНУЮ копию
  функции под каждый реальный тип (это называется *мономорфизация*). Быстро как ручной
  код, всё можно заинлайнить — но один вызов работает строго с одним типом, и бинарник
  чуть пухнет.
- **Динамическая** (`dyn Shape`): одна копия функции на всех; нужная реализация ищется в
  рантайме. Чуть медленнее, зато одна функция/коллекция обслуживает РАЗНЫЕ типы сразу.

ПОЧЕМУ это важно: «положить круги и квадраты в один `Vec`» обобщениями не сделать —
`Vec<T>` всё ещё один тип. Тут нужен именно `Vec<Box<dyn Shape>>`. Правило большого
пальца: бери обобщения по умолчанию, переходи на `dyn`, когда нужна разнородность.

### Цена динамической диспетчеризации: LLVM не может заинлайнить вызов

Вызов через vtable — это **косвенный вызов через указатель функции**. LLVM видит нечто вроде:

```
call [rax]   ; rax = адрес из vtable — значение неизвестно до рантайма
```

Оптимизатор не знает заранее, какая именно функция там окажется, поэтому не может:

- **заинлайнить** тело метода в вызывающий код;
- убрать ненужные ветки (`if x.area() == 0.0`), зная, что конкретная реализация всегда
  возвращает ненулевое значение;
- применить авто-векторизацию (SIMD) к циклу, обходящему `Vec<Box<dyn Shape>>`.

На практике для большинства задач эта цена **незначительна** — одно чтение из памяти и
непрямой прыжок. Но если у тебя **горячий тесный цикл** (tight loop), обрабатывающий
миллионы элементов, отсутствие инлайнинга может быть ощутимым. В таких местах стоит
предпочесть статическую диспетчеризацию (`impl Trait` или обобщения) или конкретный тип.

```rust
// Статическая: LLVM заинлайнит area() и может убрать операцию умножения
fn sum_static<T: Shape>(shapes: &[T]) -> f64 {
    shapes.iter().map(|s| s.area()).sum()
}

// Динамическая: LLVM не заинлайнит — каждый вызов area() идёт через vtable
fn sum_dynamic(shapes: &[Box<dyn Shape>]) -> f64 {
    shapes.iter().map(|s| s.area()).sum()
}
```

> Подвох: не оптимизируй преждевременно. Профилируй сначала. В 99% кода разница
> незаметна, зато `dyn` даёт гибкость, которую статика не может выразить.

### Обратная сторона мономорфизации: раздувание бинарника

Статическая диспетчеризация порождает **отдельную копию кода** для каждого конкретного
типа — это называется *раздуванием бинарника* (binary bloat). Чем больше разных типов
используют обобщённую функцию, тем больше кода компилятор генерирует:

```rust
fn process<T: Shape>(s: &T) -> f64 { s.area() * 2.0 }

// Если вызываешь для Circle, Square, Triangle, Pentagon —
// компилятор создаёт ЧЕТЫРЕ копии process: process::<Circle>, process::<Square>, ...
```

В крупных проектах это накапливается: компиляция замедляется, итоговый бинарник
становится больше, и кэш инструкций процессора заполняется дублирующим кодом.

Именно поэтому **в плагинных и коллбэк-паттернах** `dyn Trait` — канонический выбор:

```rust
// ✅ Одна функция на всех — плагин-регистрация, обработчики событий, логгеры:
fn register(handler: Box<dyn EventHandler>) { ... }

// Вместо: fn register<H: EventHandler>(handler: H) — N копий под N типов обработчиков.
```

Правило: если тип будет известен в рантайме (плагин, коллбэк, список разнородных
объектов) — `dyn`; если тип известен на этапе компиляции и важна скорость в горячем пути —
обобщения.

## Идея 3. Стандартный трейт `Display` — как твой тип печатается

`Display` отвечает за «человеческий» вывод через `{}` в `println!`/`format!`. Реализуешь
один метод `fmt` — и твой тип печатается как родной:

```rust
use std::fmt;

impl fmt::Display for Point {
    fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result {
        write!(f, "({}, {})", self.x, self.y)
    }
}

let p = Point { x: 1.0, y: 2.0 };
println!("{}", p);          // (1, 2)
let s = format!("{}", p);   // s == "(1, 2)"
```

ПОЧЕМУ через `write!`, а не `return String`: `f` — это «приёмник», в который понемногу
пишется итоговый текст; так формат работает без лишних аллокаций. Возвращай ровно то, что
вернул `write!` — это `fmt::Result`.

Приятный бонус: как только есть `impl Display`, у типа **автоматически** появляется метод
`.to_string()`. Бесплатно, из самого факта реализации трейта.

> Замечание про числа: `f64` через `Display` печатает `1.0` как `1`, а `2.5` как `2.5`.
> Поэтому `Point { x: 1.0, y: 2.0 }` даёт `"(1, 2)"`, а не `"(1.0, 2.0)"`. Учитывай это.

## Идея 4. Стандартный трейт `Iterator` — реализуй для своего типа

`Iterator` — сердце циклов и комбинаторов в Rust. Чтобы сделать свой тип итерируемым,
достаточно реализовать ОДИН метод — `next`:

```rust
struct Counter { count: u32 }

impl Iterator for Counter {
    type Item = u32;                       // что выдаём
    fn next(&mut self) -> Option<u32> {    // следующий элемент или None
        if self.count < 5 {
            self.count += 1;
            Some(self.count)               // 1, 2, 3, 4, 5
        } else {
            None                           // последовательность кончилась
        }
    }
}
```

ПОЧЕМУ это так мощно: реализовав только `next`, ты бесплатно получаешь ВЕСЬ арсенал —
`for x in counter`, `.map`, `.filter`, `.sum`, `.collect` и десятки других. Все они
построены поверх `next`. Это та же идея, что в C++ с диапазонами, только аккуратнее:
`None` явно сигналит конец, перепутать границы сложнее.

```rust
let total: u32 = Counter::new().sum();     // 1+2+3+4+5 = 15
```

### `IntoIterator` и как `for` на самом деле работает

Когда ты пишешь `for x in collection`, компилятор **разворачивает** это в явный цикл
через трейт `IntoIterator`. Примерно так:

```rust
// Это:
for x in my_vec {
    println!("{}", x);
}

// Становится примерно вот этим:
let mut iter = my_vec.into_iter();   // вызов IntoIterator::into_iter
loop {
    match iter.next() {
        Some(x) => println!("{}", x),
        None => break,
    }
}
```

`IntoIterator` — трейт с одним методом `into_iter(self) -> impl Iterator`. Если твой тип
реализует `Iterator`, то Rust автоматически реализует для него и `IntoIterator` (просто
возвращая `self`). Поэтому `Counter` сразу работает в `for`:

```rust
for n in Counter::new() {          // работает, потому что Counter: Iterator
    println!("{}", n);             // выводит 1, 2, 3, 4, 5
}
```

Если хочешь, чтобы твой тип был *итерируемым* (но не сам является итератором), реализуй
`IntoIterator` явно — он вернёт отдельную структуру-итератор:

```rust
struct Points(Vec<Point>);

impl IntoIterator for Points {
    type Item = Point;
    type IntoIter = std::vec::IntoIter<Point>;
    fn into_iter(self) -> Self::IntoIter {
        self.0.into_iter()
    }
}
// Теперь: for p in points { ... }
```

### Ассоциированные типы vs параметры обобщений

Почему `Iterator` объявлен с `type Item`, а не как `Iterator<T>`? Это принципиальный
выбор дизайна трейтов в Rust.

**Параметр обобщений** (`trait Foo<T>`) позволяет реализовать трейт *несколько раз* для
одного типа с разными параметрами:

```rust
trait Convert<T> {
    fn convert(&self) -> T;
}

impl Convert<f64> for i32 { fn convert(&self) -> f64 { *self as f64 } }
impl Convert<String> for i32 { fn convert(&self) -> String { self.to_string() } }
// У i32 теперь два impl Convert — это нормально
```

**Ассоциированный тип** (`type Item` внутри трейта) говорит: «для данного типа есть
ровно ОДНА реализация этого трейта». Например, `Vec<u32>` выдаёт только `u32`, а не
одновременно `u32` и `String`:

```rust
trait Iterator {
    type Item;                          // строго один Item для каждого типа
    fn next(&mut self) -> Option<Self::Item>;
}

impl Iterator for Counter {
    type Item = u32;                    // Counter выдаёт только u32
    fn next(&mut self) -> Option<u32> { ... }
}
// Нельзя написать второй impl Iterator for Counter — компилятор запретит
```

**Практическое правило:**
- Если смысл трейта предполагает *однозначную* связь («итератор имеет один тип элементов»,
  «тип имеет один тип ошибки») — используй ассоциированный тип.
- Если тип может «конвертироваться» в несколько разных вещей или реализовывать трейт
  несколько раз с разными параметрами — используй параметр обобщений.

     метод `log` принимает `&self` (разделяемое заимствование), но нам нужно мутировать
     буфер — это требует *внутренней мутабельности*.
   - **`Dispatcher`** — хранит `Vec<Box<dyn Logger>>` и рассылает каждое сообщение
     всем зарегистрированным логгерам.

   Запуск тестов модуля:

   ```bash
   ./rust/run.sh 10
   ```

### Подсказки

<details><summary>1. area() для Circle и Square</summary>
Площадь круга — π·r². Число π бери из стандартной библиотеки:
<code>std::f64::consts::PI</code>. Площадь квадрата — <code>self.side * self.side</code>.
</details>

<details><summary>1. total_area по срезу боксов</summary>
Заведи сумму <code>let mut sum = 0.0;</code>, пройди <code>for s in shapes</code> и
добавляй <code>s.area()</code>. Здесь <code>s</code> — это <code>&Box&lt;dyn Shape&gt;</code>,
но метод <code>area()</code> вызывается напрямую: Rust сам разыменует бокс. Когда станет
привычно — то же одной строкой: <code>shapes.iter().map(|s| s.area()).sum()</code>.
</details>

<details><summary>2. Display::fmt</summary>
Тело — ровно одна строка:
<code>write!(f, "({}, {})", self.x, self.y)</code>.
Возвращай результат <code>write!</code> как есть (не ставь <code>;</code> в конце, чтобы
он стал возвращаемым значением). Не забудь <code>use std::fmt;</code> в шапке — он уже там.
</details>

<details><summary>3. Iterator::next и new</summary>
В <code>new</code> верни <code>Counter { count: 0 }</code>. В <code>next</code>: если
<code>self.count &lt; 5</code> — увеличь <code>self.count</code> на 1 и верни
<code>Some(self.count)</code>; иначе верни <code>None</code>. Так выйдет именно
1, 2, 3, 4, 5, затем бесконечный <code>None</code>.
</details>

<details><summary>3. counter_sum</summary>
<code>Counter::new().sum()</code>. Тип результата <code>u32</code> Rust выведет из
сигнатуры функции. Метод <code>.sum()</code> доступен только потому, что ты реализовал
<code>next</code> — почувствуй связь.
</details>

<details><summary>4. Трейт Logger и объектная безопасность</summary>
Убедись, что метод <code>log</code> не возвращает <code>Self</code> и не имеет обобщённых
параметров. Тогда <code>Box&lt;dyn Logger&gt;</code> скомпилируется без проблем.
</details>

<details><summary>4. BufferedLogger и внутренняя мутабельность</summary>
Метод <code>log</code> берёт <code>&self</code>, но тебе надо добавить строку в вектор.
Это противоречие решает <code>RefCell&lt;Vec&lt;String&gt;&gt;</code>: снаружи это
неизменяемое поле, но <code>borrow_mut()</code> даёт изменяемый доступ в рантайме.
Какое поле нужно добавить в структуру <code>BufferedLogger</code>? Как создать его пустым?
</details>

<details><summary>4. Dispatcher::dispatch и разнородная коллекция</summary>
Перебери все логгеры: <code>for logger in &self.loggers</code>. Каждый <code>logger</code>
здесь — <code>&Box&lt;dyn Logger&gt;</code>, но метод <code>log</code> вызовется через
динамическую диспетчеризацию. Что будет, если <code>loggers</code> пуст — это ошибка
или просто ничего?
</details>
