# Модуль 03 — Владение (ownership)

**Большая идея:** в Rust у каждого значения есть ровно **один владелец**. Когда владелец
выходит из области видимости, значение автоматически уничтожается (`drop`) — память
освобождается без сборщика мусора и без ручного `free`. Из этого правила вырастает вся
безопасность Rust: компилятор статически (на этапе сборки!) гарантирует, что ты не
обратишься к уже освобождённой памяти и не освободишь её дважды.

Если в C++ за этим следил RAII и твоя дисциплина, то здесь за этим следит **компилятор**.
Поначалу он будет «ругаться» — и это не препятствие, а фича: каждая такая ошибка — это
пойманный за руку баг, который в C мог бы стать `use-after-free`.

## Идея 1. Один владелец, конец области видимости → drop

```rust
{
    let s = String::from("hello"); // s — владелец строки в куче
    // ... пользуемся s ...
} // <- здесь s выходит из области видимости: Rust сам вызывает drop, память свободна
```

**ПОЧЕМУ так:** привязав время жизни ресурса к области видимости переменной, Rust делает
освобождение **предсказуемым** и **автоматическим**. Нет момента «а кто же это удалит?» —
ответ всегда один: владелец, и ровно один раз.

## Идея 2. Семантика перемещения (move)

Присваивание или передача значения в функцию **перемещает** владение. Старая переменная
после этого недействительна:

```rust
let a = String::from("hi");
let b = a;          // владение ПЕРЕМЕСТИЛОСЬ из a в b
// println!("{a}"); // ОШИБКА компиляции: value used after move (a больше не владелец)
println!("{b}");    // ок: теперь владелец — b
```

То же при передаче в функцию: `fn eat(s: String)` — забирает владение; после `eat(b)`
переменной `b` тоже пользоваться нельзя.

**ПОЧЕМУ так:** если бы `a` и `b` оба «владели» одной строкой, при выходе из области
видимости `drop` сработал бы дважды по одному буферу — двойное освобождение. Move решает
это в корне: владелец всегда один, поэтому и `drop` ровно один.

## Идея 3. `Copy`-типы копируются, а не перемещаются

Маленькие значения фиксированного размера (`i32`, `i64`, `f64`, `bool`, `char`, кортежи
из таких типов) реализуют трейт `Copy`. Они при присваивании **копируются** целиком:

```rust
let x = 5;
let y = x;        // копия, а не move
println!("{x}");  // ок! x всё ещё доступен — копировать дёшево, обоим хватит
```

**ПОЧЕМУ так:** копировать пару байт на стеке так же дёшево, как перемещать, а у таких
значений нет ресурса в куче, который можно было бы «освободить дважды». Поэтому для них
move-семантика была бы лишним неудобством — Rust делает их `Copy`. Строки и вектора
владеют буфером в куче, поэтому они НЕ `Copy` (копия была бы дорогой и неявной).

## Идея 4. Хочешь оставить оригинал — клонируй

Если значение нужно и тебе, и вызывающему коду, есть два честных пути: **одолжить**
ссылку (`&T`, это следующий модуль) или сделать **явную глубокую копию** через `.clone()`:

```rust
let v = vec![1, 2, 3];
let w = v.clone();  // отдельный буфер; теперь два независимых владельца двух буферов
// v и w живут сами по себе, drop каждого освободит свой буфер
```

**ПОЧЕМУ так:** `clone()` всегда **виден в коде** — никакого скрытого дорогого
копирования за спиной. Дорого? Значит, ты это написал явно и осознанно. Это намеренное
дизайнерское решение Rust: «дорогое — заметным».

## Идея 5. Стек и куча: почему `String` и `Vec` не `Copy`

Чтобы понять, почему `String` не реализует `Copy`, нужно заглянуть под капот — посмотреть,
как значение устроено в памяти.

**`i64` на стеке.** Переменная `let x: i64 = 42` — это ровно 8 байт на стеке. Всё
значение целиком находится прямо в стековом фрейме. Скопировать его — значит просто
скопировать эти 8 байт. Дёшево, безопасно, никакой кучи.

**`String` — «широкий» указатель.** `let s = String::from("hello")` создаёт на стеке
структуру из трёх полей:

```
стек:                      куча:
┌───────────┐             ┌───────────────────┐
│ ptr  ─────┼────────────►│ h  e  l  l  o     │
│ len = 5   │             └───────────────────┘
│ cap = 5   │
└───────────┘
```

- **ptr** — указатель на буфер в куче (где лежат сами байты);
- **len** — сколько байт сейчас занято;
- **cap** (capacity) — сколько байт выделено.

Тоже самое — `Vec<T>`: на стеке `(ptr, len, cap)`, в куче — элементы.

**Почему нельзя `Copy`?** Если бы `String` был `Copy`, то при `let b = a` обе переменные
имели бы одинаковый `ptr`, указывающий на один и тот же буфер в куче. При выходе из
области видимости `drop` вызвался бы для обеих — **двойное освобождение одного буфера**,
это неопределённое поведение. Move-семантика устраняет проблему в корне: после `let b = a`
компилятор объявляет `a` недействительной — один `ptr`, один `drop`.

Вывод: **`Copy` несовместим с владением ресурсом в куче**.

## Идея 6. Трейт `Drop` и его несовместимость с `Copy`

Когда владелец выходит из области видимости, Rust автоматически вызывает **деструктор** —
метод `drop`. Это поведение определяется трейтом `std::ops::Drop`:

```rust
struct FileHandle {
    name: String,
}

impl Drop for FileHandle {
    fn drop(&mut self) {
        println!("Закрываю файл: {}", self.name);
    }
}

{
    let f = FileHandle { name: String::from("log.txt") };
    // ... работаем с f ...
} // <- здесь автоматически вызывается f.drop() → "Закрываю файл: log.txt"
```

Ты можешь реализовать `Drop` для собственного типа, чтобы освобождать ресурсы (файлы,
сокеты, блокировки) — точно как деструктор в C++.

**`Drop` и `Copy` исключают друг друга.** Rust запрещает реализовывать оба трейта для
одного типа. Причина логична: `Copy` означает «это значение можно тупо скопировать
побитово», а `Drop` означает «при уничтожении нужно выполнить особую логику». Если бы тип
был и `Copy`, и `Drop`, то у каждой копии вызывался бы собственный деструктор — и ресурс
освобождался бы многократно. Компилятор отлавливает такое противоречие статически:

```rust
#[derive(Copy, Clone)]
struct Bad {
    x: i32,
}

impl Drop for Bad {         // ОШИБКА: cannot implement both Copy and Drop
    fn drop(&mut self) {}
}
```

## Идея 7. `mem::drop` — ранний сброс

По умолчанию `drop` происходит в конце блока. Но иногда нужно освободить ресурс
**раньше** — например, снять блокировку (`MutexGuard`) до конца функции, чтобы другой
поток не ждал лишнего. Для этого есть `std::mem::drop`:

```rust
use std::mem;

let heavy = vec![0u8; 1_000_000]; // выделили мегабайт
// ... сделали с ним что нужно ...
mem::drop(heavy);                  // освобождаем немедленно; `heavy` больше недоступен
// дальше в функции этот мегабайт уже свободен
```

`mem::drop(x)` — это просто функция, которая принимает `x` **по значению** (забирает
владение) и тут же выходит из области видимости, вызывая деструктор. Ничего магического.
Важно: нельзя написать `heavy.drop()` напрямую — компилятор запрещает явный вызов метода
`drop` из трейта `Drop`, чтобы не допустить двойного освобождения. Только через
`mem::drop`.

## Идея 8. Частичные перемещения (partial move)

Можно переместить **отдельное поле** структуры — но тогда вся структура становится
«частично перемещённой» и использовать её целиком уже нельзя:

```rust
struct Person {
    name: String,
    age: u32,
}

let p = Person { name: String::from("Алиса"), age: 30 };

let name = p.name;   // переместили поле name из p
// println!("{}", p.name); // ОШИБКА: поле уже перемещено
println!("{}", p.age);     // ок: age — Copy-тип, он не перемещался
// println!("{:?}", p);    // ОШИБКА: p частично перемещён, целиком использовать нельзя
```

`p.age` по-прежнему доступен, потому что `u32` реализует `Copy` — оно было скопировано,
а не перемещено. `p.name` — `String`, не `Copy` — поэтому именно оно «ушло» при `let name = p.name`.

**Частичные перемещения на практике встречаются редко** — чаще задачу решают через
заимствование (`&p.name`) или метод, потребляющий структуру целиком. Но понять их важно,
чтобы читать сообщения компилятора «use of partially moved value».

## Идея 9. Замыкания с `move`

Обычное замыкание **захватывает** переменные из окружения по ссылке (если это возможно).
Ключевое слово `move` заставляет замыкание **взять переменные во владение**:

```rust
let s = String::from("hello");

// Без move: замыкание хранит &s — s должна жить дольше замыкания
let greet = || println!("{s}");
greet(); // ок, пока s живёт

// С move: замыкание забирает s себе — s больше не доступна снаружи
let greet_owned = move || println!("{s}");
// println!("{s}"); // ОШИБКА: s перемещена в замыкание
greet_owned();
```

`move`-замыкания незаменимы, когда замыкание **переживает** свою область видимости —
например, передаётся в поток (`thread::spawn`) или возвращается из функции:

```rust
use std::thread;

let data = vec![1, 2, 3];
let handle = thread::spawn(move || {
    // data принадлежит этому потоку — безопасно, нет гонок
    println!("data: {:?}", data);
});
handle.join().unwrap();
// println!("{:?}", data); // ОШИБКА: data ушла в поток
```

Без `move` компилятор бы отказал: ссылка на `data` может пережить стек, где `data` живёт.

<details><summary>1. sum_vec — складываем элементы</summary>

</details>

<details><summary>2. concat_strings — переиспользуй буфер a</summary>

`a` уже твоя (move). `String` умеет дописывать: `a.push_str(&b)` добавит содержимое `b`
в конец `a` (обрати внимание на `&b` — `push_str` берёт строковый срез, а не владение).
После этого просто верни `a`. Так ты не выделяешь третий буфер зря.
</details>

<details><summary>3. repeat_string — есть готовый метод</summary>

У строкового среза есть метод `.repeat(n)`, который возвращает новую `String`,
повторённую `n` раз. Можешь вызвать его на `&s` (или на `s.as_str()`). Случай `n == 0`
он обрабатывает сам — вернёт пустую строку.
</details>

<details><summary>4. clone_and_push — сначала клон, потом push</summary>

`v` пришёл как `&Vec<i64>` — это ссылка, менять её ты не можешь и не должен. Сделай
владеемую копию: `let mut out = v.clone();`. Вот теперь `out` — твой собственный вектор,
объяви его `mut` и сделай `out.push(x)`. Верни `out`. Оригинал у вызывающего не
пострадал — мы его лишь прочитали для клонирования.
</details>

## Задание — часть 2: `ValueStore` (владение в структуре)

Реализуй структуру `ValueStore` — простое хранилище строк, которое учит тому, как
**структура сама владеет** своими данными, и как от выбора получателя (`self` / `&self` /
`&mut self`) зависит, что происходит с владением.

Тебе нужно написать:

- `ValueStore::new() -> Self` — создаёт пустое хранилище.
- `push(&mut self, s: String)` — добавляет строку в конец (`&mut self`: мутируем хранилище, но не потребляем).
- `pop(&mut self) -> Option<String>` — забирает последнюю строку (LIFO); если пусто — `None`.
- `drain_into(self) -> Vec<String>` — **потребляет** хранилище (`self` по значению!) и возвращает все строки как `Vec`. После этого вызова `ValueStore` уничтожен.
- `len(&self) -> usize` — количество строк (только читаем, `&self`).
- `is_empty(&self) -> bool` — `true`, если строк нет.
