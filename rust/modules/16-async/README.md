# Модуль 16 — Асинхронность: Future, `.await`, executor

Большая идея: асинхронность — это **не про «быстрее»**. Это про то, как **ждать многое,
не блокируя поток**. Синхронный код, дойдя до «подожди ответа от сети», встаёт колом:
поток просто стоит и ничего не делает, пока байты не придут. Async позволяет в этот
момент **отдать управление** и заняться другой ждущей задачей — на том же одном потоке.
Тысячи «ожидающих» соединений живут рядом, и поток крутится только тогда, когда есть
реальная работа.

В Rust строительный материал — `async fn`, которая возвращает **`Future`**: ленивое
описание работы. Само future ничего не делает; его двигает **executor** (среда
выполнения, runtime), поллингом доводя до результата. В этом модуле executor **виден
явно** — мы зовём `futures::executor::block_on(future)` руками, без `#[tokio::main]` и
прочей магии. Так понятнее, кто на самом деле крутит шестерёнки.

> **Особенность модуля.** Весь курс Rust — на стандартной библиотеке, без внешних
> зависимостей. Этот модуль — **единственное исключение**: он зависит от крейта
> `futures` (учебный, прозрачный executor). В проде стандарт — **tokio**, но его магия
> прячет двигатель; для обучения нам важнее увидеть `block_on` своими глазами.
> Этот модуль не входит в общий воркспейс курса — у него свой `Cargo.toml` с пустой
> секцией `[workspace]`, поэтому тесты запускаются **из каталога модуля**:
> `cd rust/modules/16-async && cargo test`.

Модуль продолжает две предыдущие темы. Из модуля 11 (итераторы): **`Future` — ленивое,
как итератор**; ничего не считается, пока его не «прокрутят». **`Stream` — это
async-итератор**. Из модуля 15 (потоки): здесь мы противопоставим async и
`thread::spawn` — два разных ответа на вопрос «как делать несколько дел сразу».

## Идея 1. Что такое async и почему «ленивое описание работы»

`async fn` — это функция, помеченная словом `async`. Её особенность: вызвать её — **ещё
не значит выполнить**. Вызов лишь создаёт **future** — объект-описание «вот что надо
сделать, когда дойдут руки».

```rust
async fn greet() -> String {
    String::from("привет")
}

let fut = greet();   // НИЧЕГО не выполнилось! fut — это future, описание работы.
// строка "привет" ещё не создана; greet() лишь вернула future.
```

Чтобы работа реально произошла, future надо **довести до результата** — этим занимается
executor:

```rust
use futures::executor::block_on;

let result = block_on(greet());   // ВОТ ТЕПЕРЬ greet выполнилась, result == "привет"
```

**ПОЧЕМУ так, а не как в JavaScript.** В JS промис **eager**: как только ты вызвал
`fetch(...)`, запрос уже полетел. В Rust future **lazy**: «**a future does nothing until
polled**» — пока future никто не поллит (не двигает), оно спит. Это позволяет дёшево
конструировать future, комбинировать их (`join!`, `select!`), отменять — всё до того,
как хоть что-то начнёт исполняться.

> Полезный вопрос на самопроверку: что напечатает программа, в которой есть
> `async fn main_work() { println!("работаю"); }`, но `main_work()` нигде не передан в
> `block_on`? — **Ничего.** Future создалось и было отброшено, не прокрученное ни разу.
> `println!` внутри так и не выполнился.

## Идея 2. Конкурентность ≠ параллелизм

Это два разных слова, которые часто путают.

- **Параллелизм** — несколько дел исполняются **физически одновременно**, на разных
  ядрах. Нужно несколько потоков/ядер.
- **Конкурентность** — несколько дел **в работе одновременно**, но не обязательно
  исполняются в один и тот же момент. Их можно чередовать на одном исполнителе.

Аналогия — один повар на кухне. Он **конкурентно** готовит три блюда: поставил суп
вариться (ожидание!), пока он кипит — режет салат, заглянул в духовку, вернулся к супу.
Один повар, одно «ядро» — но три блюда движутся вперёд, потому что повар переключается
**в моменты ожидания**. Это и есть async: **конкурентность на одном потоке за счёт
перекрытия пауз ожидания**.

Параллелизм — это уже **несколько поваров**. Каждый занят своим блюдом по-настоящему
одновременно. Это потоки (модуль 15).

**Связь с `thread::spawn` (модуль 15).** Поток — это параллелизм ценой ресурсов ОС:
каждый поток забирает мегабайты под стек, его создаёт и переключает ядро. Async-задача —
это **сотни байт** (стейт-машина future) на куче; «переключение» — это возврат из
`poll`, дешёвая операция в пользовательском коде. Поэтому тысяч потоков ты не поднимешь,
а **тысячи async-задач** — легко.

| | потоки (м. 15) | async (этот модуль) |
|---|---|---|
| Что даёт | параллелизм (несколько ядер) | конкурентность (перекрытие ожиданий) |
| Цена одной единицы | ~МБ стека, ОС-ресурс | ~сотни байт, объект на куче |
| Переключает | ядро ОС (дорого) | executor в user-space (дёшево) |
| Где силён | CPU-bound работа | I/O-bound, много ожиданий |

## Идея 3. `async fn` и трейт `Future`

Запись `async fn` — это сахар. Вот что под ним:

```rust
async fn foo() -> i64 { 42 }
// в точности эквивалентно:
fn foo() -> impl std::future::Future<Output = i64> { /* стейт-машина */ }
```

То есть `async fn foo() -> T` — это «функция, возвращающая **что-то, реализующее
`Future<Output = T>`**». Ассоциированный тип `Output` — это тип финального результата
(тот, что ты получишь после `block_on`).

Сам трейт `Future` (упрощённо) выглядит так:

```rust
trait Future {
    type Output;
    fn poll(self: Pin<&mut Self>, cx: &mut Context) -> Poll<Self::Output>;
}

enum Poll<T> {
    Ready(T),    // готово, вот результат
    Pending,     // ещё не готово, позови меня позже
}
```

Руками `poll` мы **не пишем** (компилятор генерирует стейт-машину из `async`/`.await`
автоматически) — но важно понимать модель. Executor зовёт `poll`:
- если future возвращает `Poll::Ready(value)` — всё, результат получен;
- если `Poll::Pending` — future ещё ждёт (например, данных из сети). Executor отложит
  его и займётся другими, а вернётся, когда future **разбудят** (см. Идею 5, Waker).

Это и есть «ленивость» из Идеи 1, выраженная в типе: future продвигается **только** при
вызове `poll`, а зовёт `poll` только executor.

## Идея 4. `.await` — точка приостановки

`.await` — постфиксный оператор: пишется после future, говорит «**дождись результата
этого future, отдав управление, пока он не готов; потом продолжи отсюда**».

```rust
async fn pipeline() -> i64 {
    let a = step_one().await;   // дождались step_one, получили i64
    let b = step_two(a).await;  // дождались step_two
    a + b
}
```

**ПОЧЕМУ `.await` не блокирует поток.** Когда future внутри `pipeline` отвечает
`Pending`, `.await` не крутит поток вхолостую — он **возвращает управление наверх**, к
executor-у. Тот, пока наш future ждёт, поллит **другие** future. Когда наш future
разбудят, executor вернётся ровно в эту точку `.await` и продолжит. Поток всё это время
не простаивал — он работал над чем-то ещё.

Полезная параллель — оператор `?` из модуля 07. Оба **постфиксные** (`x.await`, `x?`),
оба — **точки управления потоком исполнения**: `?` — точка раннего выхода при ошибке,
`.await` — точка приостановки до готовности.

> **Подвох (цвет функций).** `.await` законен **только внутри async-контекста** —
> внутри `async fn` или `async { }` блока. В обычной синхронной функции написать
> `.await` нельзя: компилятор скажет `` `await` is only allowed inside `async` functions
> and blocks``. Это значит, что async «заражает» вызывающих: чтобы сделать `.await`,
> функция сама должна стать `async`. Отсюда метафора «**цвет функций**»: async-функции и
> обычные — как два цвета, и async-цвет распространяется вверх по цепочке вызовов.

> **Подвох (последовательный `.await`).** `a().await; b().await;` запускает их
> **по очереди**: пока не закончится `a`, `b` даже не начнётся. Если оба только ждут
> (I/O), это медленнее, чем надо — паузы не перекрылись. Лекарство — Идея 6 (`join!`).

## Идея 5. Зачем нужен executor (runtime) и что делает `block_on`

Future само себя не запускает (Идея 1). Нужен тот, кто будет **поллить его в цикле** до
`Ready` — это и есть **executor** (часть runtime). Его работа:
1. вызвать `poll` у future;
2. если `Pending` — отложить future и не трогать его, пока не придёт сигнал «можно
   двигать дальше»;
3. этот сигнал приносит **Waker** — объект, который future (точнее, источник события:
   сокет, таймер) получает в `Context` и дёргает, когда событие случилось. Waker говорит
   executor-у «разбуди вот эту задачу». Так executor не крутит `poll` вхолостую, а спит,
   пока есть что ждать.

`futures::executor::block_on(fut)` — простейший executor: он **блокирует текущий поток**,
гоняет ровно один future до `Ready` и возвращает его `Output`. Это **мост из синхронного
мира** (обычная `fn main`, `#[test]`) в async-мир:

```rust
use futures::executor::block_on;

fn main() {
    let answer = block_on(async {
        compute().await + 1
    });
    println!("{answer}");
}
```

**ПОЧЕМУ в этом курсе тесты используют `block_on`, а не `#[tokio::test]`.** Атрибут
`#[tokio::test]` (или `#[futures_test]`) — это proc-макрос: он сам разворачивает тело в
синхронное и прячет создание runtime. Удобно в проде, но **executor становится
невидимым**. В учебном модуле нам важно обратное — видеть мост явно. Поэтому каждый тест
здесь — обычная `#[test] fn`, внутри которой стоит `block_on(...)`. Executor на виду.

## Идея 6. Конкурентный запуск: `join!` и `join_all`

Вернёмся к подвоху из Идеи 4: `a.await; b.await;` — это **последовательно**. Чтобы
запустить future **конкурентно** (executor двигает оба, перекрывая их паузы), есть макрос
`join!` и функция `join_all`.

```rust
use futures::join;

async fn fetch_both() -> (i64, i64) {
    // ПОСЛЕДОВАТЕЛЬНО: сначала весь a, потом весь b
    // let a = task_a().await;
    // let b = task_b().await;

    // КОНКУРЕНТНО: оба двигаются вперёд, ждём, пока готовы ОБА
    let (a, b) = join!(task_a(), task_b());
    (a, b)
}
```

`join!(a, b)` возвращает **кортеж** результатов, когда готовы все аргументы. Для
коллекции однотипных future есть `futures::future::join_all`:

```rust
use futures::future::join_all;

async fn fetch_many(futs: Vec<impl std::future::Future<Output = i64>>) -> Vec<i64> {
    join_all(futs).await   // Vec результатов, в исходном порядке
}
```

**ПОЧЕМУ это быстрее (при ожиданиях).** Если `task_a` и `task_b` по секунде ждут сети, то
последовательно это ~2 секунды (ждём a, потом b), а `join!` — ~1 секунду: пока a ждёт,
executor двигает b, их паузы **перекрылись**.

> **Подвох.** `join!` — это **не параллелизм**. Всё крутится на одном потоке. Если
> `task_a` и `task_b` — тяжёлые CPU-вычисления (а не ожидания), `join!` **не ускорит**:
> один поток всё равно считает их по очереди. Выигрыш `join!` — только на **перекрытии
> ожиданий**. Для CPU-bound нужен параллелизм — потоки (модуль 15).

## Идея 7. Async-каналы: `mpsc` и `oneshot`

В модуле 15 каналы (`std::sync::mpsc`) связывали **потоки**. В async-мире нужны каналы,
которые при пустоте **не блокируют поток**, а отдают управление executor-у. Их даёт
`futures::channel`.

**`mpsc::unbounded()`** — **m**ultiple **p**roducer, **s**ingle **c**onsumer: много
отправителей, один получатель, неограниченный буфер.

```rust
use futures::channel::mpsc;
use futures::stream::StreamExt;   // нужен для .next()/.fold() на приёмнике

async fn collect_sum() -> i64 {
    let (tx, rx) = mpsc::unbounded::<i64>();

    tx.unbounded_send(1).unwrap();
    tx.unbounded_send(2).unwrap();
    tx.unbounded_send(3).unwrap();
    drop(tx);   // ← закрываем отправитель: «значений больше не будет»

    // rx — это Stream («async-итератор»). Свернём его в сумму.
    rx.fold(0, |acc, v| async move { acc + v }).await
}
```

Приёмник `rx` реализует трейт **`Stream`** — это **async-итератор**: вместо `next() ->
Option<T>` у него `next().await -> Option<T>`, элементы приходят по мере появления.
Методы `next`, `fold`, `collect`, `map` для Stream живут в трейте `StreamExt` — его надо
импортировать (`use futures::stream::StreamExt;`).

> **Подвох (зависание).** Stream-приёмник завершается (`next()` отдаёт `None`), только
> когда **все** отправители закрыты (дропнуты). Забудешь `drop(tx)` — приёмник будет
> ждать «а вдруг придёт ещё», `.fold(..).await` никогда не вернётся, и `block_on`
> **зависнет навсегда**. Это классическая async-ловушка: всегда закрывай отправитель,
> когда отправил всё.

**`oneshot::channel()`** — один-единственный результат «туда-обратно»: отправитель шлёт
**одно** значение, приёмник его `await`-ит.

```rust
use futures::channel::oneshot;

let (tx, rx) = oneshot::channel::<i64>();
tx.send(42).unwrap();                 // одно значение
let v: i64 = rx.await.unwrap();       // receiver.await -> Result<i64, Canceled>
```

`oneshot` идеален, чтобы фоновая задача вернула **один** результат туда, где её ждут.

**ПОЧЕМУ async-канал, а не `std::sync::mpsc`.** Синхронный `recv()` **блокирует поток**,
пока нет данных — в async это катастрофа (заблокируешь весь executor). Async-канал при
пустоте отдаёт `Pending`, executor двигает другие задачи, а к приёмнику вернётся, когда
данные придут.

## Идея 8. Спавн задач: `LocalPool` + `spawn_local`

`block_on(fut)` гоняет **ровно один** корневой future. А если нужны **фоновые задачи** —
producer-ы, которые работают сами по себе и шлют результаты в канал? Их надо **заспавнить**
— отдать executor-у, не дожидаясь каждой тут же по очереди.

`futures` даёт однопоточный исполнитель `LocalPool` с возможностью спавна:

```rust
use futures::executor::LocalPool;
use futures::task::LocalSpawnExt;   // даёт метод spawn_local

let mut pool = LocalPool::new();
let spawner = pool.spawner();

// Спавним фоновую задачу — она НЕ выполняется тут же, попадает в очередь executor-а.
spawner.spawn_local(async {
    println!("я фоновая задача");
}).unwrap();

// pool.run() крутит все заспавненные задачи до завершения.
// pool.run_until(fut) крутит, пока не завершится конкретный future (попутно двигая фон).
pool.run();
```

**ПОЧЕМУ спавн, а не просто `.await`.** `.await` встраивает работу **в текущий** future и
ждёт её здесь же. Спавн **отпускает** задачу жить параллельно (конкурентно) с другими.
Типичный паттерн: заспавнить несколько producer-ов, пишущих в канал, а в корневом future
сидеть consumer-ом и собирать результаты — как в задании 5.

> Заметь разницу с `thread::spawn` из модуля 15: там задача уезжала на **отдельный
> поток ОС**. Здесь `spawn_local` оставляет задачу **на том же потоке**, просто в очереди
> того же executor-а — это конкурентность, не параллелизм.

## Идея 9. Async или потоки — когда что (и какой runtime брать)

Итоговая шпаргалка выбора.

| Ситуация | Бери |
|---|---|
| I/O-bound, тысячи ожиданий (сеть, файлы), мало CPU | **async** |
| CPU-bound (тяжёлый счёт, перемалывание данных) | **потоки / thread-pool** (м. 15) |
| Простая блокирующая задача, пара воркеров | **потоки** (проще, без runtime) |
| Много дешёвых «ждущих» сущностей (10k соединений) | **async** (задача ~сотни байт vs ~МБ на поток) |

**ПОЧЕМУ не «async везде».** За async платишь сложностью:
- **цвет функций** — async заражает всех вызывающих вверх по цепочке;
- **отладка сложнее** — стек-трейсы рвутся на точках `.await`, стейт-машины менее
  очевидны;
- **нужен runtime** — без executor-а async-код просто не запустится.

Для чисто CPU-bound или простых случаев честный поток короче и понятнее.

**Какой runtime выбрать.**
- **`futures`** (этот модуль) — учебный, прозрачный: `block_on`, `join!`, каналы. Двигатель
  на виду. Для реальных сетевых приложений минимален (нет встроенного асинхронного I/O,
  таймеров).
- **`tokio`** — **де-факто индустриальный стандарт**. `#[tokio::main]`, асинхронные сокеты,
  таймеры, тайм-ауты, многопоточный планировщик. Production default — почти любой
  серьёзный async-проект на Rust сегодня на tokio.
- **`async-std`** — исторически второй большой runtime с API «как у std»; сейчас
  уступил tokio по популярности.

Хорошая новость: освоив `futures::block_on` / `.await` / `join!` / каналы, переход на
tokio — почти **переименование импортов**. Сам язык (`async`/`.await`/`Future`) один и
тот же; меняется лишь конкретный executor и набор асинхронных примитивов вокруг.

Запуск **из каталога модуля** (он не в общем воркспейсе):

```bash
cd rust/modules/16-async && cargo test
```

1. `pub async fn double(x: i64) -> i64` (= `2 * x`); `pub async fn add(a: i64, b: i64) ->
   i64` (= `a + b`); `pub async fn double_then_add(x: i64, y: i64) -> i64` — должна
   вызвать `double(x).await` и прибавить `y`.

2. `pub async fn sum_two<FA, FB>(fa: FA, fb: FB) -> i64` где
   `FA: Future<Output = i64>, FB: Future<Output = i64>` — дождаться **обоих
   конкурентно** через `futures::join!` и вернуть сумму.

3. `pub async fn sum_all<F>(futs: Vec<F>) -> i64` где `F: Future<Output = i64>` —
   `join_all(futs).await`, затем сумма результатов. Пустой вектор → `0`.

4. `pub async fn channel_sum(values: Vec<i64>) -> i64` — создать `mpsc::unbounded`,
   отправить все значения, **закрыть отправитель**, собрать приёмник-Stream в сумму.
   Пустой вход → `0`. Функция обязана завершаться (не зависать).

5. `pub fn run_workers(chunks: Vec<Vec<i64>>) -> i64` (**синхронная** обёртка) — на
   каждый чанк заспавнить задачу, считающую частичную сумму и шлющую её через `oneshot`;
   корневой future собирает частичные суммы в общий итог. Маленький конкурентный
   map-reduce. Пустой список чанков → `0`; пустые чанки вносят `0`; порядок не важен.

> **Две частые ошибки async — держи в голове, пока пишешь:**
> 1. **`.await` вне async-контекста** → ошибка компиляции `` `await` is only allowed
>    inside `async`...``. Решение: функция (или блок) должна быть `async`.
> 2. **Последовательный `await` вместо `join!`** → код работает, но при ожиданиях
>    медленнее, чем мог бы. Для задания 2 нужен именно `join!`, не `fa.await + fb.await`.
> 3. **Забыл `drop(tx)`** в задании 4 → приёмник ждёт вечно, тест зависает.

### Подсказки к заданию 1

<details><summary>Подсказка 1.1 — это самое простое, не усложняй</summary>

Тело `async fn double` — это просто `2 * x`. Слово `async` уже стоит в сигнатуре;
компилятор сам обернёт результат в future. Внутри пиши обычное синхронное выражение —
никакого `.await` тут не нужно (мы ничего не ждём).

То же для `add`: тело — `a + b`.
</details>

<details><summary>Подсказка 1.2 — где здесь нужен <code>.await</code></summary>

`.await` нужен только в `double_then_add`, потому что там ты **используешь результат**
другого future. `double(x)` возвращает future `i64`; чтобы получить из него число,
допиши `.await`:

```text
double(x).await   // это i64
```

Дальше прибавь `y`. Подумай: можно ли написать `double(x) + y` без `.await`? (Нет —
`double(x)` это future, а не число; складывать future с `i64` нельзя.)
</details>

### Подсказки к заданию 2

<details><summary>Подсказка 2.1 — какой инструмент даёт «конкурентно»</summary>

Контракт требует запустить `fa` и `fb` **конкурентно**, а не по очереди. Последовательное
`fa.await + fb.await` сюда не годится по духу задания (хотя и даст верную сумму). Нужен
макрос `futures::join!` — он импортирован в начале файла как `use futures::join;`.
</details>

<details><summary>Подсказка 2.2 — что возвращает <code>join!</code></summary>

`join!(fa, fb)` возвращает **кортеж** результатов, когда готовы оба:

```text
let (a, b) = join!(fa, fb);
```

Теперь `a` и `b` — это два `i64`. Верни их сумму. Обрати внимание: `join!` сам внутри
ждёт оба future, тебе не нужно отдельно писать `.await` на `fa`/`fb`.
</details>

### Подсказки к заданию 3

<details><summary>Подсказка 3.1 — join_all и его результат</summary>

`futures::future::join_all(futs)` (импортирован как `join_all`) берёт `Vec` future и
сам возвращает одно future. Дождись его: `join_all(futs).await`. Результат — `Vec<i64>`
(результаты в исходном порядке).
</details>

<details><summary>Подсказка 3.2 — свернуть Vec в сумму</summary>

После `.await` у тебя `Vec<i64>`. Это уже не async — обычный вектор. Сложи его привычным
способом из модуля 11: `.into_iter().sum()` (или `.iter().sum()`). Пустой вектор даст
`0` сам собой — отдельный случай прописывать не нужно.
</details>

### Подсказки к заданию 4

<details><summary>Подсказка 4.1 — создать канал и отправить</summary>

`let (tx, rx) = mpsc::unbounded::<i64>();` — `tx` отправитель, `rx` приёмник. Пройди по
`values` обычным `for` и на каждом шаге `tx.unbounded_send(v)`. Этот метод возвращает
`Result` — отправка не пройдёт, только если приёмник уже мёртв (у нас он жив), так что
`.unwrap()` или `.expect(...)` здесь безопасны.
</details>

<details><summary>Подсказка 4.2 — почему обязателен <code>drop(tx)</code></summary>

После цикла отправки **закрой отправитель**: `drop(tx);`. Без этого приёмник-Stream
никогда не завершится — он будет ждать, не придёт ли ещё значение, — и весь `block_on`
зависнет. Закрытие отправителя — это сигнал приёмнику «поток значений окончен».
</details>

<details><summary>Подсказка 4.3 — собрать приёмник-Stream в сумму</summary>

`rx` — это `Stream`. Методы на нём приходят из `StreamExt` (уже импортирован). Свернуть в
сумму удобно через `fold`:

```text
rx.fold(0_i64, |acc, v| async move { acc + v }).await
```

Обрати внимание: замыкание в `fold` для Stream — **асинхронное** (возвращает future),
поэтому тело обёрнуто в `async move { ... }`. Альтернатива: `rx.collect::<Vec<_>>().await`,
затем `.iter().sum()` — но `fold` короче.
</details>

### Подсказки к заданию 5

<details><summary>Подсказка 5.1 — общая схема (не код, а план)</summary>

Это синхронная функция (`fn`, не `async fn`) — внутри сама поднимает executor. План:

1. создать `LocalPool` и взять у него `spawner` (`pool.spawner()`);
2. для каждого чанка: создать `oneshot::channel`, **заспавнить** задачу
   (`spawner.spawn_local(async move { ... })`), которая посчитает сумму чанка и пошлёт
   её в `tx`; приёмник `rx` сложить в вектор приёмников;
3. корневым future (`pool.run_until(async move { ... })`) дождаться все приёмники и
   сложить частичные суммы.
</details>

<details><summary>Подсказка 5.2 — задача-воркер и oneshot</summary>

Для одного чанка:

```text
let (tx, rx) = oneshot::channel::<i64>();
spawner.spawn_local(async move {
    let partial: i64 = chunk.iter().sum();
    let _ = tx.send(partial);   // отправляем одно значение обратно
}).unwrap();
```

`chunk` нужно **переместить** в задачу (`async move`), иначе он не доживёт. `rx` сложи в
`Vec<oneshot::Receiver<i64>>` — его будешь собирать в корне.
</details>

<details><summary>Подсказка 5.3 — собрать частичные суммы в корне</summary>

Когда все воркеры заспавнены, заверши работу через `pool.run_until(...)` с async-блоком,
который собирает приёмники. `oneshot::Receiver` сам по себе future
(`Output = Result<i64, Canceled>`), поэтому к ним применим `join_all`:

```text
pool.run_until(async move {
    let partials = join_all(receivers).await;   // Vec<Result<i64, Canceled>>
    partials.into_iter().map(|r| r.unwrap()).sum()
})
```

`run_until` крутит executor (двигая попутно все заспавненные задачи), пока не завершится
этот корневой future, и возвращает его результат — он и есть ответ функции. Пустой список
чанков? Тогда `receivers` пуст, `join_all` мгновенно даёт пустой Vec, сумма — `0`.
</details>
