# Модуль 02 — Поток управления и match

Большая идея: в Rust поток управления — это не просто «прыжки» по коду, а **выражения,
которые возвращают значения**. `if`, `match`, `loop` — всё это можно присвоить
переменной. А `match` вдобавок **исчерпывающий**: компилятор не даст забыть ветку.
Это превращает разбор вариантов из источника багов в надёжный инструмент.

## Идея 1. `if` — это выражение

В большинстве языков `if` — инструкция: «сделай то или это». В Rust `if` ещё и
**возвращает значение**, поэтому его можно положить прямо в `let`:

```rust
let n = 7;
let parity = if n % 2 == 0 { "even" } else { "odd" };
//           ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^ всё это — одно выражение
```

ПОЧЕМУ это удобно: не нужна лишняя `let mut`, которую ты потом присваиваешь в каждой
ветке. Переменная сразу инициализирована и может быть неизменяемой.

Подвох: обе ветки обязаны иметь **один тип**, и без `else` тут не обойтись (иначе при
ложном условии значения бы не было). Также заметь: ветка возвращает значение
**последним выражением без `;`** — точка с запятой превратила бы его в `()`.

## Идея 2. `match` — исчерпывающее сопоставление

`match` сравнивает значение с набором образцов сверху вниз и выполняет первую
подходящую ветку. Главная сила — **исчерпываемость**: компилятор требует покрыть все
возможные случаи (для остальных есть `_`).

```rust
let sign = match n.cmp(&0) {
    std::cmp::Ordering::Less => "negative",
    std::cmp::Ordering::Equal => "zero",
    std::cmp::Ordering::Greater => "positive",
};
```

Можно сопоставлять и кортежи — это пригодится в FizzBuzz:

```rust
let label = match (i % 3, i % 5) {
    (0, 0) => "FizzBuzz",   // делится и на 3, и на 5
    (0, _) => "Fizz",       // только на 3
    (_, 0) => "Buzz",       // только на 5
    _ => "число",           // ни на что — обязательная «всё остальное» ветка
};
```

ПОЧЕМУ это важно: если завтра ты добавишь новый вариант в `enum`, а где-то забудешь
его обработать — код **не скомпилируется**. Компилятор работает твоим напарником по
ревью. «Ругань» компилятора здесь — не препятствие, а защита.

### 2а. Сторожевые условия в `match` (match guards)

Иногда образца недостаточно — нужно дополнительное условие. Для этого есть
**сторожевое условие** (`guard`): `образец if условие =>`.

```rust
let описание = match x {
    n if n < 0   => "отрицательное",
    0            => "ноль",
    n if n % 2 == 0 => "положительное чётное",
    _            => "положительное нечётное",
};
```

Сторож — это обычное булево выражение. Имя, связанное в образце (`n`), доступно
внутри него. Ветка срабатывает только если **и** образец совпадает, **и** сторож
истинен; иначе проверяется следующая ветка.

Подвох: компилятор не умеет доказать, что сторожи покрывают все случаи — поэтому
ветка `_` или иной «тотальный» образец по-прежнему нужна, если предыдущие ветки
не покрывают всё.

### 2б. Диапазонные образцы (`1..=10 =>`)

В образцах `match` можно использовать **включающие диапазоны** для чисел и `char`:

```rust
let категория = match возраст {
    0..=17  => "несовершеннолетний",
    18..=64 => "взрослый",
    65..=   => "пенсионер",    // начиная с 65 — до конца типа
};
```

Заметь: **только включающий** диапазон (`..=`) разрешён в образцах. Полуоткрытый
`0..17` там не работает — он допустим в выражениях (циклах), но не в `match`.

### 2в. Опровержимые и неопровержимые образцы (refutable vs irrefutable)

Образцы в Rust делятся на два класса:

- **Неопровержимый** (irrefutable) — всегда совпадает с любым значением. Примеры:
  `let x = ...`, `let (a, b) = ...`, аргументы функции. Здесь допускаются только
  неопровержимые образцы — иначе компилятор ошибку.

  ```rust
  let (x, y) = (1, 2);   // всегда успешно — irrefutable ✓
  ```

- **Опровержимый** (refutable) — может не совпасть. Примеры: `Some(x)`, `Ok(v)`,
  конкретное число `42`. Такие образцы используются в `match`, `if let`, `while let`.

  ```rust
  if let Some(v) = maybe_value {   // Some(x) — опровержимый: может быть None ✓
      println!("{v}");
  }
  ```

  Если попытаться использовать опровержимый образец в `let` — компилятор откажет:

  ```rust
  // ОШИБКА: refutable pattern in local binding
  let Some(v) = maybe_value;   // ← что делать, если None?
  ```

Практическое правило: `let` и параметры функций — только irrefutable; всё остальное
(match-ветки, `if let`, `while let`) — refutable тоже можно.

### 2г. `if let ... else` — первоклассный идиом для двух вариантов

Когда у тебя есть ровно два случая — «образец совпал» и «нет» — полный `match`
становится громоздким. Идиом `if let ... else` делает это компактно:

```rust
let config: Option<String> = get_config("timeout");

// Вместо двух ветвей match:
if let Some(value) = config {
    println!("таймаут: {value}");
} else {
    println!("таймаут не задан, используем умолчание");
}
```

ПОЧЕМУ это удобнее `match` в данном случае: читается как обычный `if`, но при этом
**связывает** переменную внутри тела — `value` доступна только в ветке `Some`, а не
снаружи. Это важно: компилятор не позволит использовать `value` в ветке `else`.

Подвох: `if let` без `else` молча ничего не делает при несовпадении. Если у тебя
есть «путь ошибки» — всегда пиши `else` явно, иначе легко пропустить случай. Если
вариантов больше двух — возвращайся к `match`.

```rust
// Типичный паттерн: «извлечь или выйти из функции»
fn process(input: Option<i64>) -> i64 {
    if let Some(n) = input {
        n * 2
    } else {
        return 0;   // явный запасной путь
    }
}
```

## Идея 3. `loop` / `while` / `for`; диапазоны и возврат из `loop`

Три цикла под разные задачи:

```rust
// for по диапазону: 0, 1, 2, 3, 4  (верхняя граница НЕ включается)
for i in 0..5 { /* ... */ }

// 1..=n — включающий диапазон: 1, 2, ..., n
for k in 1..=n { /* ... */ }

// while — пока условие истинно
while b != 0 { /* ... */ }
```

`loop` — бесконечный цикл, но он умеет **возвращать значение** через `break`:

```rust
let mut x = 1;
let result = loop {
    x *= 2;
    if x > 100 {
        break x;        // выходим И отдаём x наружу
    }
};
```

ПОЧЕМУ `loop`-с-`break`, а не `while` с внешней переменной: значение «выпадает» из
цикла сразу в `let`, не нужно заводить `mut`-переменную над циклом и надеяться, что
после цикла она в правильном состоянии.

Подвох с диапазонами: `0..n` — это **полуинтервал** (без `n`), а `0..=n` —
**включающий** (с `n`). Перепутаешь — получишь off-by-one. Для факториала и нужен
`1..=n`.

### 3а. `continue` — перейти к следующей итерации

`continue` немедленно переходит к следующей итерации цикла, пропуская оставшийся
код тела:

```rust
for i in 0..10 {
    if i % 2 == 0 {
        continue;          // пропустить чётные
    }
    println!("{i}");       // напечатает 1, 3, 5, 7, 9
}
```

В `while` и `loop` работает точно так же. В Rust `continue` — инструкция, а не
выражение (не возвращает значения).

### 3б. Именованные метки (`'label`) и выход из вложенных циклов

Когда циклы вложены, `break` и `continue` по умолчанию действуют на **ближайший**
цикл. Чтобы выйти из внешнего цикла напрямую, используют **именованные метки**:

```rust
'outer: for i in 0..5 {
    for j in 0..5 {
        if i + j == 6 {
            break 'outer;   // выходим из ВНЕШНЕГО цикла
        }
        println!("{i},{j}");
    }
}
// Продолжение программы здесь
```

Метка записывается как `'имя` перед ключевым словом цикла. `continue 'outer` тоже
работает — перейти к следующей итерации внешнего цикла, пропустив весь внутренний.

### 3в. Метка + возврат значения из `loop`

`loop` может **одновременно** иметь метку и возвращать значение через `break`.
Это сочетание особенно полезно, когда нужно вернуть результат поиска из
вложенного контекста — без дополнительных флагов и изменяемых переменных:

```rust
let matrix = vec![
    vec![1, 2, 3],
    vec![4, 5, 6],
    vec![7, 8, 9],
];
let target = 5;

let position: Option<(usize, usize)> = 'search: loop {
    for (i, row) in matrix.iter().enumerate() {
        for (j, &val) in row.iter().enumerate() {
            if val == target {
                break 'search Some((i, j));  // выходим из loop С значением
            }
        }
    }
    break 'search None;   // не нашли — тоже через break с явным значением
};

println!("{:?}", position);   // Some((1, 1))
```

ПОЧЕМУ это удобнее `mut`-флага: `position` — неизменяемая переменная, её тип и
значение определены в одном месте. Компилятор проверяет, что все пути `break`
возвращают один тип.

Подвох: метку `'search:` ставь **перед `loop`**, не перед `for`. Именно `loop`
здесь является владельцем возвращаемого значения — `break 'search` выходит из него.

### 3г. `while let` — цикл до несовпадения образца

`while let` — это `loop` + `if let` в одном: выполняет тело пока образец совпадает,
а при несовпадении выходит:

```rust
let mut stack = vec![1, 2, 3];

while let Some(top) = stack.pop() {
    println!("{top}");   // выведет 3, 2, 1
}
// здесь stack пуст, pop() вернул None — цикл завершился
```

ПОЧЕМУ это лучше явного `loop { match { ... break } }`: конструкция читается как
«пока есть значение — обрабатывай» — намерение сразу видно.

Подвох 1 — **бесконечный цикл на вечно-`Some`**: если источник никогда не возвращает
`None`, цикл не завершится. Убедись, что на каждой итерации прогресс есть:

```rust
// ОПАСНО: если always_some() всегда возвращает Some — зависнешь навсегда
while let Some(x) = always_some() {
    process(x);
    // нет break, нет изменения состояния → бесконечный цикл
}
```

Подвох 2 — **забытый `break`**: внутри тела `while let` нет запрета на бесконечный
цикл. Если ты вручную хочешь выйти раньше, чем образец не совпадёт — используй
`break` явно.

## Идея 4. `for` по итератору — потребление и заимствование

`for` в Rust ходит не по индексам, а по **итератору**. Диапазон `0..n` — это уже
итератор; коллекция тоже даёт итератор. Поэтому индекс почти никогда не нужен.

Ключевой вопрос: **передаёт ли `for` владение** коллекцией или **заимствует** её?

### Потребляющий итератор: `for x in v`

```rust
let v = vec!["hello", "world"];
for s in v {              // v перемещается в цикл — ownership transferred
    println!("{s}");
}
// println!("{:?}", v);   // ОШИБКА: v уже перемещён, нельзя использовать
```

После цикла `v` недоступен — он «съеден». Это полезно, когда коллекция больше не
нужна и ты хочешь избежать лишнего клонирования.

### Заимствующий итератор: `for x in &v`

```rust
let v = vec!["hello", "world"];
for s in &v {             // &v — создаём итератор по ссылкам
    println!("{s}");      // s имеет тип &&str
}
println!("{:?}", v);      // OK: v всё ещё наш
```

Здесь `v` остаётся доступным после цикла. Используй `&v` всегда, когда нужно
продолжить работать с коллекцией после прохода.

ПОЧЕМУ так лучше «классического» `for i in 0..v.len()`: нет ручной индексации —
значит, нет выхода за границы массива. Ты получаешь сами элементы, а не их номера.
(Если номер всё же нужен — есть `.enumerate()`, но это уже модуль про итераторы.)

### Как `for` работает под капотом

`for x in expr` — синтаксический сахар:

```rust
// for x in expr { тело }
// разворачивается примерно в:
let mut iter = expr.into_iter();
loop {
    match iter.next() {
        Some(x) => { тело }
        None    => break,
    }
}
```

`into_iter()` потребляет `expr`; для `&v` вызывается `(&v).into_iter()`, который
возвращает итератор по ссылкам, не трогая владение. Вот почему `for x in v` и
`for x in &v` ведут себя по-разному.

### Тройка `iter()` / `iter_mut()` / `into_iter()`

На коллекциях (Vec, HashMap, срезы) есть три метода для получения итератора — выбор
определяется тем, **что** ты хочешь делать с элементами:

| Вызов               | Тип элемента | Коллекция после цикла | Когда использовать |
|---------------------|--------------|-----------------------|--------------------|
| `v.iter()`          | `&T`         | доступна (заимствована) | читать без изменений |
| `v.iter_mut()`      | `&mut T`     | доступна (изменяемо заимствована) | изменять на месте |
| `v.into_iter()`     | `T`          | **перемещена**, недоступна | потреблять / преобразовывать |

```rust
let mut nums = vec![1, 2, 3];

// iter() — получаем &i32, коллекция остаётся
for n in nums.iter() {
    println!("{n}");        // n: &i32
}
println!("{:?}", nums);     // OK — nums жив

// iter_mut() — изменяем каждый элемент in-place
for n in nums.iter_mut() {
    *n *= 2;                // n: &mut i32, нужно разыменовать
}
println!("{:?}", nums);     // [2, 4, 6]

// into_iter() — коллекция перемещается в цикл
for n in nums.into_iter() {
    println!("{n}");        // n: i32 — владение перешло
}
// println!("{:?}", nums);  // ОШИБКА: nums перемещён
```

ПОЧЕМУ это важно: `for x in &v` — это сокращение для `for x in v.iter()`;
`for x in &mut v` — для `v.iter_mut()`; `for x in v` — для `v.into_iter()`.
Зная явные методы, ты можешь также передавать итератор в функции, цеплять адаптеры
(`.map()`, `.filter()`) — это тема модуля 11. Здесь главное: знай, что за каждым
`for` стоит один из этих трёх видов.

### Задание 6 — кодирование серий (run-length encoding)

**Что такое RLE.** Кодирование серий — один из простейших алгоритмов сжатия: вместо
строки символов хранишь пары «символ + сколько раз подряд». Например:
`"aabccc"` → `[('a', 2), ('b', 1), ('c', 3)]`. Если развернуть эти пары обратно —
получится исходная строка.

**Контракт функции:**
- Принимает строковый срез `s`.
- Возвращает `Vec<(char, usize)>` — вектор пар в порядке следования серий.
- Для пустой строки — пустой вектор.
- Каждый `char` в Rust — **Copy**, что делает его удобным для хранения в кортежах
  и передачи по значению без явного клонирования.

**Зачем это задание.** Здесь в одном месте встречается несколько идей модуля:
- `for c in s.chars()` — итерация по символам строки;
- `match` по `Option<(char, usize)>` — одновременно проверяем, есть ли текущая серия,
  и совпадает ли символ;
- сброс серии в вектор при смене символа;
- финальный `if let` — не забыть последнюю серию.

### Ступенчатые подсказки

<details><summary>fizzbuzz — с чего начать</summary>

Заведи `let mut out: Vec&lt;String&gt; = Vec::new();`, пройди `for i in 1..=n` и на
каждом шаге вычисли строку, затем `out.push(...)`. Число превращается в строку через
`i.to_string()`. В конце верни `out` (последним выражением без `;`).
</details>

<details><summary>fizzbuzz — как выбрать строку без лишних if</summary>

`match (i % 3, i % 5)` с образцами `(0, 0)`, `(0, _)`, `(_, 0)`, `_`. Обрати внимание
на **порядок**: `(0, 0)` обязан стоять первым, иначе `(0, _)` перехватит кратные 15 и
`"FizzBuzz"` никогда не появится. Ветки `match` должны давать один тип — здесь это
`String`, так что для числа используй `i.to_string()`, а для литералов — `.to_string()`
или `String::from(...)`.
</details>

<details><summary>factorial — накопитель</summary>

`let mut acc = 1u64; for k in 1..=n { acc *= k; } acc`. При `n == 0` диапазон `1..=0`
пуст, цикл не выполнится ни разу, и `acc` останется `1` — ровно то, что нужно.
</details>

<details><summary>fib — без рекурсии</summary>

Держи пару `(a, b) = (0, 1)`. Повтори `n` раз: `let next = a + b; a = b; b = next;`.
После цикла ответ — в `a`. Проверь на `fib(0)` и `fib(1)` руками, прежде чем запускать
тесты. Рекурсию не пиши: `fib(50)` рекурсией считался бы заметно дольше.
</details>

<details><summary>gcd — алгоритм Евклида</summary>

```text
пока b != 0:
    (a, b) = (b, a % b)
ответ = a
```

В Rust присвоение кортежем работает: `let (na, nb) = (b, a % b); a = na; b = nb;` или
сразу `(a, b) = (b, a % b);`. Когда `b` дойдёт до нуля, в `a` лежит НОД. Случай
`gcd(a, 0)` обрабатывается сам: цикл не запустится, вернётся `a`.
</details>

<details><summary>classify — почему &'static str</summary>

Все три ответа — строковые **литералы**, они вшиты в бинарник и живут всю программу,
поэтому тип `&'static str` (ссылка, живущая «вечно»), а не `String` — ничего
аллоцировать и кем-то владеть не нужно. Используй `match` по `n.cmp(&0)` (даёт
`Ordering::Less/Equal/Greater`) или по сторожевым условиям `match n { x if x < 0 => ... }`.
</details>

<details><summary>run_length_encode — как хранить «текущую серию»</summary>

Тебе нужно помнить текущий символ и сколько раз он уже встречался. Это пара
`(char, usize)`, которая то существует, то нет — идеально для `Option`. Заведи
`let mut current: Option&lt;(char, usize)&gt; = None;`.
</details>

<details><summary>run_length_encode — что делать с каждым символом</summary>

На каждом шаге у тебя есть `current` и очередной символ `c`. Напиши `match current`:
- `None` → первый символ, установи `current = Some((c, 1))`;
- `Some((prev, count))` если `prev == c` → та же серия, увеличь счётчик;
- `Some((prev, count))` если `prev != c` → серия закончилась, положи `(prev, count)` в
  результат и начни новую серию: `current = Some((c, 1))`.

Для сторожевого условия как раз пригодится `if prev == c` в ветке `match`.
</details>

<details><summary>run_length_encode — не забудь последнюю серию</summary>

Цикл заканчивается — а последняя незавершённая серия осталась в `current`. После
цикла сделай `if let Some((c, count)) = current { result.push((c, count)); }`. Без
этого последняя группа будет потеряна.
</details>

---

Когда станет зелено — остановись и спроси себя: где ты применил `match`, а где `if`, и
**почему** именно так? В FizzBuzz `match` по кортежу читается яснее лесенки `if/else`,
а в `classify` — заменяет три сравнения одной исчерпывающей таблицей. В `run_length_encode`
`match` по `Option` явно указывает, что оба случая (`Some` и `None`) обработаны.
Умение выбрать правильную конструкцию важнее, чем знание их синтаксиса. Покажи мне свой
`lib.rs` — обсудим, где код можно сделать ещё выразительнее.
