# Модуль 05 — Структуры и методы

Большая идея: `struct` связывает разрозненные данные в **один именованный тип**, а
`impl`-блок навешивает на него **поведение**. Точка — это не «две `f64`, которые мы
договорились считать координатами», а именно `Point`. Тип несёт смысл: компилятор не
даст случайно передать `Rectangle` туда, где ждут `Point`, даже если внутри те же два
числа. Это первый шаг к тому, чтобы «невозможные состояния» становились
непредставимыми — главную мысль мы будем углублять в модулях 06 (enum) и дальше.

В отличие от C++, в Rust данные (`struct`) и поведение (`impl`) пишут **раздельными
блоками**. Полей с логикой внутри объявления нет — сначала описываем форму, потом
отдельно методы. Привыкни к этому разделению, оно здесь повсюду.

## Идея 1. `struct`, `impl`, метод `&self` и ассоциированная функция `::new`

```rust
pub struct Point {
    pub x: f64,
    pub y: f64,
}

impl Point {
    // Ассоциированная функция: НЕТ self. Зовут через тип: Point::new(1.0, 2.0)
    pub fn new(x: f64, y: f64) -> Point {
        Point { x, y }      // сокращение: поле x = переменная x
    }

    // Метод: ПЕРВЫЙ параметр — self (здесь &self). Зовут через значение: p.length()
    pub fn length(&self) -> f64 {
        (self.x * self.x + self.y * self.y).sqrt()
    }
}
```

Разница, которую важно прочувствовать:

- `Point::new(...)` — **ассоциированная функция**. У неё нет `self`, она не привязана к
  конкретному объекту; это «фабрика». Двойное двоеточие `::` — обращение к чему-то,
  что принадлежит типу, а не экземпляру.
- `p.length()` — **метод**. Его первый параметр — `self` (в той или иной форме). Точка
  `.` означает «вызвать у этого конкретного значения».

ПОЧЕМУ `&self`, а не `self`: `length` только читает поля. Брать `self` по значению
значило бы **забрать владение** точкой и уничтожить её после вызова — расточительно и
неудобно. `&self` одалживает объект для чтения и возвращает обратно. Это ровно то
заимствование (`&`), что ты изучал в модуле 04, просто теперь сахарно записанное как
первый параметр метода.

## Идея 2. `#[derive(...)]` — компилятор пишет рутину за тебя

```rust
#[derive(Debug, Clone, Copy, PartialEq)]
pub struct Point {
    pub x: f64,
    pub y: f64,
}
```

`derive` просит компилятор **автоматически** сгенерировать реализацию трейта, выводя её
из структуры поля за полем. Что мы просим здесь:

- **`Debug`** — печать в отладочном виде через `{:?}`. Без неё `assert_eq!` не сможет
  показать тебе значения при провале теста — это обязательный «налог» на удобную диагностику.
- **`Clone`** — явное копирование: `let b = a.clone();`.
- **`Copy`** — а вот это меняет саму семантику присваивания. У `Copy`-типа `let b = a;`
  делает **копию**, и `a` остаётся доступной. Без `Copy` (вспомни модуль 03) присваивание
  было бы **перемещением** — `a` стало бы недействительным. `Copy` уместен только для
  маленьких, чисто-значимых типов вроде двух `f64`.
- **`PartialEq`** — сравнение `==` и `!=`. Сгенерированная версия сравнивает **все поля**
  попарно.

ПОЧЕМУ это хорошо: поведение «сравни два поля» или «скопируй два поля» механически
выводится из формы типа. Писать его руками — шум и риск опечатки. `derive` —
декларативно: ты говоришь *что* нужно, а не *как*.

> Подвох: `derive(Copy)` требует, чтобы все поля тоже были `Copy`. Для `f64` и `u8` это
> так. Но как только в структуре появится, скажем, `String` (она владеет кучей и НЕ
> `Copy`), `derive(Copy)` перестанет компилироваться — и это правильно: «дешёвое
> побитовое копирование» там было бы ложью.

### `PartialEq` vs `Eq`: почему `f64` не бывает `Eq`

Это тонкий момент, который путает многих. В Rust есть **два** трейта равенства:

- **`PartialEq`** — *частичное* равенство: для любых `a` и `b` определена операция `a == b`, но она **не обязана быть рефлексивной**. Иными словами, не гарантируется, что `a == a`.
- **`Eq`** — *полное* равенство: гарантирует, что `a == a` всегда (рефлексивность). Это подтрейт `PartialEq` — всё, что `Eq`, автоматически и `PartialEq`.

Зачем делить? Из-за `NaN` (Not a Number) в IEEE 754. По стандарту `NaN != NaN` — это не баг, а часть спецификации плавающей арифметики:

```rust
let nan = f64::NAN;
println!("{}", nan == nan);   // false — NaN не равен самому себе!
println!("{}", nan != nan);   // true
```

Поэтому `f64` реализует `PartialEq`, но **не** `Eq`. Структура `Point { x: f64, y: f64 }` с `derive(PartialEq)` тоже не может автоматически получить `Eq` — компилятор не будет делать за тебя ложное утверждение. Если захочешь сравнивать точки как ключи `HashMap`, понадобится другой подход (например, хранить целочисленные координаты).

На практике: `derive(PartialEq)` достаточно почти всегда. `Eq` нужен, только когда тип будет ключом `HashMap`/`BTreeMap` или обязан участвовать в чём-то, требующем полного равенства.

### `{:?}` vs `{:#?}`: компактная и «красивая» отладочная печать

`Debug` даёт сразу два формата вывода — и переключаются они одним символом `#` в
спецификаторе:

- `{:?}` — **компактно, в одну строку**: `Config { name: "db", timeout: 30, retries: 3, verbose: false }`.
- `{:#?}` — **«pretty», с переносами и отступами**: каждое поле на своей строке.

```rust
#[derive(Debug)]
struct Config { name: String, timeout: u32, retries: u8 }

let c = Config { name: "db".to_string(), timeout: 30, retries: 3 };

println!("{c:?}");
// Config { name: "db", timeout: 30, retries: 3 }

println!("{c:#?}");
// Config {
//     name: "db",
//     timeout: 30,
//     retries: 3,
// }
```

ПОЧЕМУ два формата, а не один: `{:?}` удобен в логах и однострочных сообщениях, где
важна плотность. `{:#?}` незаменим, когда структура большая или вложенная (структура
внутри структуры внутри `Vec`) — без отступов такой вывод нечитаем. Оба берутся из
**одной и той же** `derive(Debug)`, отдельно реализовывать ничего не нужно.

> Подвох: `#` — это часть спецификатора формата, а не отдельный аргумент. Пишется
> `{:#?}`, а не `{#:?}` или `{:?#}`. Тот же `#` работает и с числами: `{:#x}` печатает
> `0xff` (с префиксом `0x`) вместо голого `ff`. `#` всюду означает «альтернативная,
> более развёрнутая форма».

## Идея 3. Методы, заимствующие vs потребляющие `self`

Первый параметр метода бывает трёх видов — и это не формальность, а заявление о
намерениях прямо в сигнатуре:

```rust
fn read(&self)        { /* только читаю поля */ }
fn modify(&mut self)  { /* меняю поля; нужен &mut, как изменяемое заимствование */ }
fn consume(self)      { /* забираю владение; после вызова объекта больше нет */ }
```

- `&self` — заимствую для **чтения**. Самый частый случай (наши `area`, `distance_to`).
- `&mut self` — заимствую для **изменения**. Объект остаётся у хозяина, но я его правлю.
- `self` — **потребляю**. Вызывающий теряет объект (если тип не `Copy`). Используется,
  когда метод логически «израсходовал» значение или превращает его в нечто другое.

ПОЧЕМУ это важно: сигнатура — контракт. Видя `fn area(&self)`, читатель сразу знает, что
площадь не портит прямоугольник. Видя `fn consume(self)`, знает, что после вызова
переменной пользоваться нельзя. Компилятор это проверяет — забыть невозможно.

### Пример: `&mut self` в действии

Счётчик — типичный пример мутирующего метода: он должен *изменить* состояние объекта:

```rust
pub struct Counter {
    value: u32,
}

impl Counter {
    pub fn new() -> Counter {
        Counter { value: 0 }
    }

    /// &mut self: метод меняет поле value — без mut компилятор запретит.
    pub fn increment(&mut self) {
        self.value += 1;
    }

    /// &self: только читаем — не нужен mut.
    pub fn get(&self) -> u32 {
        self.value
    }
}

let mut c = Counter::new();  // переменная тоже должна быть mut!
c.increment();
c.increment();
assert_eq!(c.get(), 2);
```

Два правила, которые легко забыть:
1. Чтобы вызвать `&mut self`-метод, переменная-владелец должна быть объявлена `let mut`.
2. Во время вызова `&mut self`-метода нельзя иметь других активных ссылок на объект — как и с обычным `&mut` (правило уникального заимствования из модуля 04 в полной силе).

### Автоматическое получение ссылки (auto-ref / auto-deref)

Метод объявлен как `fn area(&self)` — то есть формально первым аргументом он ждёт
**ссылку** `&Rectangle`. Но вызываешь ты его просто `r.area()`, без `&r`. Откуда тогда
берётся ссылка?

Это делает компилятор сам. При вызове `r.method()` он смотрит на сигнатуру метода и
**автоматически** добавляет столько `&`, `&mut` или разыменований `*`, сколько нужно,
чтобы тип получателя совпал с тем, что требует метод. Это называется *automatic
referencing / dereferencing*:

```rust
let mut r = Rectangle::new(3.0, 4.0);

r.area();        // метод хочет &self  → компилятор подставляет (&r).area()
r.scale(2.0);    // метод хочет &mut self → компилятор подставляет (&mut r).scale(2.0)

// Работает и в обратную сторону — через ссылку/умный указатель:
let rr: &Rectangle = &r;
rr.area();       // метод хочет &self, а rr уже &Rectangle → берётся как есть

let boxed = Box::new(Rectangle::new(1.0, 1.0));
boxed.area();    // Box<Rectangle> авто-разыменуется до &Rectangle
```

ПОЧЕМУ это важно знать, а не просто «оно как-то работает»:

- В отличие от C++, где ты вручную выбираешь `obj.method()` (для значения) против
  `ptr->method()` (для указателя), в Rust оператор **всегда `.`** — компилятор сам
  добавит нужные `&`/`*`. Стрелки `->` в языке нет вообще.
- Поэтому `p.distance_to(&q)` пишут так: **получатель** `p` авто-реферится сам (метод
  взял бы `&p`), а вот `q` идёт обычным аргументом — для него правило auto-ref **не
  работает**, ссылку `&` ставишь руками. Auto-ref касается только получателя слева от
  точки, не остальных параметров.

> Подвох: auto-ref выбирает `&` или `&mut` по сигнатуре метода. Если метод требует
> `&mut self`, а сам объект лежит за `&` (неизменяемой ссылкой) — компилятор не сможет
> «придумать» `&mut` из `&` и выдаст ошибку про невозможность изменяемого
> заимствования. Это не каприз: правило уникального изменяемого доступа из модуля 04
> здесь в полной силе, auto-ref его не обходит.

## Идея 4. Кортежные структуры

Иногда имена полей — лишний шум, но **отдельный тип** всё равно нужен:

```rust
pub struct Color(pub u8, pub u8, pub u8);   // поля без имён

let red = Color(255, 0, 0);
let r = red.0;     // доступ по индексу: .0, .1, .2
```

ПОЧЕМУ не просто кортеж `(u8, u8, u8)`: тогда любой `(u8, u8, u8)` подошёл бы туда, где
ждут цвет. Кортежная структура `Color` — **именованный, отличимый** тип: координаты,
цвет и «просто тройку чисел» компилятор уже не перепутает. Это та же защита смыслом, что
и у обычной `struct`, только без накладных расходов на имена полей.

## Идея 5. Видимость полей: `pub` vs приватное

По умолчанию поля структуры **приватны** — доступны только внутри того модуля, где
объявлена структура. `pub` перед полем открывает его наружу.

```rust
pub struct Point {
    pub x: f64,   // доступно везде
    pub y: f64,   // доступно везде
}

pub struct BankAccount {
    owner: String,    // приватное: снаружи нельзя прочитать или изменить
    balance: f64,     // приватное: изменяется только через методы
}

impl BankAccount {
    pub fn new(owner: String) -> BankAccount {
        BankAccount { owner, balance: 0.0 }
    }

    pub fn deposit(&mut self, amount: f64) {
        // Проверка инварианта — нельзя занести отрицательную сумму.
        // Снаружи баланс изменить напрямую невозможно — только через этот метод.
        if amount > 0.0 {
            self.balance += amount;
        }
    }

    pub fn balance(&self) -> f64 {
        self.balance
    }
}
```

**Зачем делать поля приватными?** Это и есть *инкапсуляция*: внутреннее представление
скрыто, и публичный API — единственный способ взаимодействовать со структурой. Выгоды:

1. **Инварианты защищены**: `deposit` может проверить, что сумма положительная. Если
   поле `balance` было бы `pub`, любой мог бы написать `acc.balance = -1000.0` в обход
   логики.
2. **Свобода рефакторинга**: хочешь изменить внутреннее представление (например, хранить
   баланс в копейках как `u64`)? Меняешь только тело методов — внешний код не ломается,
   он пользуется только публичным API.

Правило большого пальца: **делай поля приватными по умолчанию**, открывай только то, что
действительно должно быть частью контракта типа.

## Идея 6. Синтаксис обновления структуры (`..base`)

Когда нужно создать новое значение, отличающееся от существующего лишь несколькими
полями, Rust предлагает компактный синтаксис:

```rust
#[derive(Debug, Clone)]
pub struct Config {
    pub debug: bool,
    pub timeout: u32,
    pub retries: u8,
    pub verbose: bool,
}

let default = Config {
    debug: false,
    timeout: 30,
    retries: 3,
    verbose: false,
};

// Создаём вариант с другим timeout — остальные поля берём из default.
let fast = Config {
    timeout: 5,
    ..default.clone()   // ..base должен идти последним
};

println!("{:?}", fast);
// Config { debug: false, timeout: 5, retries: 3, verbose: false }
```

`..base` буквально говорит: «все поля, которые я не перечислил явно, возьми из `base`».
Важные детали:

- `..base` всегда **последний** в списке полей — иначе ошибка компиляции.
- Если тип не реализует `Copy`, поля **перемещаются** из `base` (он частично
  потребляется). Если тип `Copy` — копируются. Вот почему выше пришлось написать
  `..default.clone()`: `Config` содержит `String` — нет, в примере не содержит, но если
  бы содержал, клонирование было бы обязательным.
- Перемещение из `base` происходит **побито**: если ты явно указал поле `timeout`, то
  `timeout` из `base` не перемещается, а остальные — да. После `..base` (без `clone()`)
  можно использовать только явно указанные поля `base`, но не те, что были взяты через `..`.

Этот синтаксис часто встречается в builder-паттернах и конфигурациях.

## Идея 7. Юнит-структуры

Структуру можно объявить вообще без полей:

```rust
pub struct Marker;          // юнит-структура
pub struct AlwaysTrue;

impl AlwaysTrue {
    pub fn check(&self) -> bool { true }
}

let t = AlwaysTrue;
assert!(t.check());
```

Зачем нужна структура без данных?

- **Маркер типа**: передать в обобщённую функцию «какую стратегию использовать», не
  неся никаких данных (нулевой размер в памяти!).
- **Реализация трейта**: «это нечто умеет делать X», но хранить ничего не нужно.
- **Временный заглушка-тип**: пока не решил, что будет хранить структура.

Юнит-структура занимает **0 байт** — Rust гарантирует, что `std::mem::size_of::<Marker>() == 0`.

## Идея 8. `impl Display` — красивый вывод для пользователя

`Debug` (`{:?}`) — для разработчика. `Display` (`{}`) — для пользователя. Его нельзя
`derive`'нуть: компилятор не знает, как ты хочешь отобразить свой тип для человека.
Реализуется вручную через трейт `std::fmt::Display`:

```rust
use std::fmt;

pub struct Point {
    pub x: f64,
    pub y: f64,
}

impl fmt::Display for Point {
    fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result {
        write!(f, "({}, {})", self.x, self.y)
    }
}

let p = Point { x: 3.0, y: -4.0 };
println!("{p}");           // (3, -4)
println!("Точка: {p}");   // Точка: (3, -4)
let s = p.to_string();    // реализация Display автоматически даёт .to_string()
assert_eq!(s, "(3, -4)");
```

`fmt::Result` — это `Result<(), fmt::Error>`. Макрос `write!` возвращает именно его, поэтому `write!(f, ...)` как последнее выражение функции — типичный паттерн.

Дополнительный бонус: любой тип с `Display` автоматически получает метод `.to_string()` через бланкетную имплементацию стандартной библиотеки.

### Подсказки к заданиям 1–3

<details><summary>Создание Point в new</summary>

оператором тут нет — просто умножь разность саму на себя. Корень — метод у `f64`:
`some_f64.sqrt()`. Заметь, что благодаря `&self` и `&other` после вызова обе точки
остаются доступны (есть тест ровно на это).
</details>

<details><summary>area и is_square</summary>

`area` — это `self.width * self.height`. `is_square` — это сравнение `self.width ==
self.height`, которое само по себе уже `bool`; `return` и `if` не нужны, просто верни
выражение.
</details>

## Задание 4 — `Stats`: аккумулятор статистики

Это главное задание модуля. Здесь тебе нужно самостоятельно **придумать форму структуры** (поля — приватные!) и реализовать публичное API.

**Контракт:**

```rust
pub struct Stats { /* твои приватные поля */ }

impl Stats {
    pub fn new() -> Stats { todo!() }
    pub fn push(&mut self, v: f64) { todo!() }
    pub fn count(&self) -> usize { todo!() }
    pub fn mean(&self) -> Option<f64> { todo!() }
    pub fn min(&self) -> Option<f64> { todo!() }
    pub fn max(&self) -> Option<f64> { todo!() }
}
```

`push` принимает числа по одному; остальные методы отвечают на вопросы о накопленной
выборке. Если ни одного числа ещё не было, `mean`/`min`/`max` возвращают `None`.

**Что здесь нового по сравнению с заданиями 1–3:**

- Поля — **приватные**: снаружи никто не может залезть в счётчик или сумму напрямую.
- `push` — первый `&mut self`-метод, который ты пишешь с нуля: он должен *изменять* состояние.
- `Option<f64>` как возвращаемый тип — честный способ сказать «ответа может не быть».

### Подсказки к Stats

<details><summary>Какие поля нужны?</summary>

Тебе нужно хранить достаточно, чтобы ответить на все четыре вопроса без второго прохода
по данным. Подумай: для среднего нужны сумма и количество; для минимума и максимума — по
одному числу каждого. Итого — четыре поля.
</details>

<details><summary>Как инициализировать min и max в new()?</summary>

В `new()` ни одного числа ещё нет. Чему должны равняться «пустые» min и max? Подумай:
когда придёт первый элемент, он должен автоматически стать и минимумом, и максимумом.
Какое начальное значение у «min», чтобы любое пришедшее число оказалось меньше него?
Подсказка: в `f64` есть константы вроде `f64::INFINITY` и `f64::NEG_INFINITY`.
</details>

<details><summary>push: структура метода</summary>

`push` должен обновить все четыре поля. Порядок: сначала увеличь счётчик и прибавь к
сумме; потом сравни пришедший элемент с текущим min и max и при необходимости замени.
Можно использовать обычный `if`, а можно — конструкцию `if v < self.min { self.min = v }`.
</details>

<details><summary>mean, min, max: когда возвращать None?</summary>

Все три метода должны вернуть `None`, если `self.count == 0`. Деление на нулевой счётчик
дало бы бессмысленное значение (или NaN), а min/max при «магических» начальных значениях
INFINITY были бы ложью. `if self.count == 0 { None } else { Some(...) }` — прямолинейное
решение.
</details>

## Задание 5 — `Config` + `Sentinel`: обновление структуры и юнит-тип

Маленькое задание, закрепляющее Идею 6 (синтаксис обновления `..base`) и Идею 7
(юнит-структуры) на конкретном коде.

**Контракт:**

```rust
#[derive(Debug, Clone, PartialEq)]
pub struct Config {
    pub name: String,
    pub timeout: u32,
    pub retries: u8,
    pub verbose: bool,
}

impl Config {
    pub fn new(name: &str) -> Config { todo!() }
    /// Новый Config, отличающийся ТОЛЬКО полем timeout.
    pub fn with_timeout(&self, secs: u32) -> Config { todo!() }
}

pub struct Sentinel;

impl Sentinel {
    pub fn is_active(&self) -> bool { todo!() }
}
```

`with_timeout` — это мини-«builder»: он не меняет исходный конфиг, а возвращает **новое**
значение, где переписано лишь поле `timeout`, а всё остальное скопировано из `self`.
Именно для этого и существует синтаксис обновления `Config { timeout: secs, ..self.clone() }`.

`Sentinel` — юнит-структура нулевого размера. Метод `is_active` всегда возвращает `true`:
смысл задания не в логике, а в том, чтобы написать `impl` для типа **без полей** и вызвать
у него метод через auto-ref (`Sentinel.is_active()` — компилятор сам подставит `&`).

**Что здесь нового:**

- **Struct update syntax** на не-`Copy` типе: `Config` владеет `String`, поэтому
  `..self.clone()` обязателен — без клона поля пришлось бы переместить из `self`, а из-за
  `&self` (ссылки) перемещать нельзя.
- **Юнит-структура с `impl`**: тип без данных, у которого всё равно есть поведение.

### Подсказки к заданию 5

<details><summary>Подсказка 5.1 — Config::new со значениями по умолчанию</summary>

`new` принимает `name: &str`, а поле `name` имеет тип `String` — нужно превратить
строковый срез в владеющую строку: `name.to_string()`. Остальные поля задай разумными
дефолтами (например, `timeout: 30`, `retries: 3`, `verbose: false`) — конкретные числа
тест не проверяет, важно лишь, чтобы `with_timeout` потом менял **только** `timeout`.
</details>

<details><summary>Подсказка 5.2 — with_timeout: какие поля писать руками, какие через ..</summary>

Тебе нужно новое значение, где отличается лишь `timeout`. Явно укажи единственное
меняющееся поле — `timeout: secs` — а все остальные подтяни из исходного конфига
синтаксисом обновления. Помни два правила из Идеи 6: `..base` идёт **последним** в списке
полей, и для не-`Copy` типа базу надо клонировать.
</details>

<details><summary>Подсказка 5.3 — почему именно ..self.clone(), а не ..*self или ..self</summary>

`..self` попыталось бы **переместить** поля (включая `String`) из `self`, но `self` у нас
по ссылке `&self` — перемещать из-за `&` запрещено, компилятор откажет. `..*self` — то же
самое (разыменование ссылки не даёт права на перемещение). Остаётся `..self.clone()`:
клон владеет своими данными, из него перемещать можно. Целиком: `Config { timeout: secs,
..self.clone() }`.
</details>

<details><summary>Подсказка 5.4 — Sentinel::is_active</summary>

Тело — буквально `true`. Сигнатура `fn is_active(&self) -> bool` берёт `&self`, хотя
читать у юнит-структуры нечего: это просто метод, который существует у типа. Создаётся
такой объект без скобок — `let s = Sentinel;` — и вызывается `s.is_active()`.
</details>

Структура — это маленькое обещание: «эти данные ходят вместе и подчиняются вот таким
правилам». Метод `&self` обещает «я только посмотрю», `&mut self` — «я изменю, но объект
останется твоим», `self` — «я это заберу». Когда допишешь — спроси себя по каждому
методу: почему именно такой `self`? Если ответ очевиден из сигнатуры — значит, тип
спроектирован честно. Покажи мне свой `lib.rs`, разберём выбор заимствований вместе.
