# Модуль 14 — Инструменты и качество кода

Большая идея: писать код — половина дела. Вторая половина — **инструменты**, которые
ловят твои ошибки раньше, чем их увидит пользователь, и держат проект в порядке.
Компилятор, тесты, линтер, форматтер и система контроля версий — это не бюрократия, а
твоя страховочная сетка. В этом модуле ты не пишешь логику с нуля: код уже написан, но
**в каждой функции спрятан баг**. Твоя работа — джунская работа на каждый день: запустить
тесты, прочитать красный, найти место расхождения, починить. Это и есть профессия.

> Этот модуль особенный. В стабе нет ни одного `todo!()` — код компилируется и
> «как будто работает». Но тесты красные, потому что в каждой функции — ровно один баг
> (классическая ошибка новичка). Найди и почини.

## Идея 1. `cargo`: build, test, run — и его друзья clippy, fmt, check

`cargo` — это сборщик, менеджер зависимостей и раннер тестов в одном лице.

```bash
cargo build      # собрать проект (проверить, что компилируется)
cargo test       # собрать и прогнать ВСЕ тесты
cargo run        # собрать и запустить бинарник (для проектов с main)
```

### `cargo check` — быстрая петля обратной связи

Когда ты много редактируешь код и хочешь лишь убедиться, что типы сходятся —
**не жди полной сборки**. `cargo check` делает только анализ типов без генерации
бинарника, а значит завершается в разы быстрее:

```bash
cargo check       # только проверка типов; артефакты не генерируются
```

В реальной работе цикл выглядит так: пишешь → `cargo check` (мгновенно) → если
всё чисто → `cargo test` (полная сборка + тесты). Эта привычка экономит минуты каждый день.

### `cargo doc` — документация из кода

Комментарии вида `///` над функцией — это не просто заметки: cargo превращает их
в полноценный HTML-сайт документации, по виду точно такой же, как docs.rs:

```bash
cargo doc            # сгенерировать HTML в target/doc/
cargo doc --open     # сгенерировать И открыть в браузере
cargo doc --no-deps  # только свой крейт, без документации зависимостей
```

**ПОЧЕМУ это бесплатно для тебя.** Ты УЖЕ пишешь `///`-комментарии (по ним сделаны
контракты функций в этом модуле). `cargo doc` просто рендерит их: заголовки, списки,
блоки кода, перекрёстные ссылки на типы. Документация живёт рядом с кодом и обновляется
вместе с ним — не в отдельном устаревающем файле. Открой `cargo doc --open` на этом
модуле и найди `celsius_to_fahrenheit`: увидишь её описание, секцию `# Examples` и
секцию `# Panics` — всё это собрано из `///` автоматически.

### Инструменты качества

```bash
cargo clippy     # ЛИНТЕР: ищет подозрительный код и анти-паттерны
cargo fmt        # ФОРМАТТЕР: приводит код к единому каноническому стилю
```

**ПОЧЕМУ это важно.** Компилятор проверяет, что код *корректен по типам*. `clippy` идёт
дальше и проверяет, что код *написан хорошо*: он замечает `x == x`, бессмысленные
`if b { true } else { false }`, лишние `.clone()` и десятки других «пахнущих» мест.
Часто баг из этого модуля clippy подсветит сам. `fmt` же убирает споры о стиле: отступы,
пробелы, переносы — всё единообразно, и баг проще увидеть глазами, когда форматирование
не мешает.

### Читаем вывод `cargo clippy`

Clippy печатает предупреждения в формате:

```
warning: this comparison is always true
  --> src/lib.rs:42:8
   |
42 |     if x == x { ... }
   |        ^^^^^^
   = note: `#[warn(clippy::eq_op)]` on by default
   = help: consider ...
```

Каждое предупреждение содержит:
- **категорию** (`clippy::eq_op`, `clippy::needless_return`, `clippy::clone_on_copy`…),
- **ссылку** на строку в файле,
- **пояснение** и часто конкретное предложение по замене.

Если предупреждение неуместно в конкретном случае (например, намеренно оставляешь
«избыточный» клон для ясности), можно его заглушить атрибутом прямо над строкой:

```rust
#[allow(clippy::clone_on_copy)]
let x = some_val.clone();
```

> **Совет.** Глушить предупреждения стоит редко. Сначала пойми, о чём предупреждает
> clippy — чаще всего он прав.

Запусти их на этом модуле:

```bash
cd rust/modules/14-tooling
cargo clippy        # прочитай предупреждения — они подсказывают, где искать
cargo fmt           # выровняй стиль
```

## Идея 2. Модули (`mod`) и видимость (`pub`)

Код в Rust разложен по **модулям** — это пространства имён. Всё, что объявлено без
`pub`, **приватно** и видно только внутри своего модуля. `pub` открывает элемент наружу.

```rust
pub fn visible_everywhere() {}   // часть публичного API крейта
fn helper() {}                   // приватная, для внутреннего пользования

mod math {                       // вложенный модуль
    pub fn add(a: i64, b: i64) -> i64 { a + b }   // math::add доступно снаружи модуля
    fn secret() {}                                // видно только внутри math
}
```

**ПОЧЕМУ это важно.** Граница `pub` — это контракт. Всё публичное ты обязан поддерживать
(на это полагаются другие); всё приватное можешь менять свободно. Чем меньше `pub`, тем
меньше поверхность, которую легко сломать. В этом модуле все публичные функции — `pub`,
потому что тесты лежат в отдельном крейте и видят только публичный API
(`use m14_tooling::*;`).

### Как модули отображаются на файлы

Когда пишешь `mod foo;` (с точкой с запятой — это объявление, а не тело), компилятор
ищет исходник в двух местах:

```
src/foo.rs           — плоская структура (проще для маленьких модулей)
src/foo/mod.rs       — если внутри foo есть свои под-модули
```

Пример: проект с несколькими модулями:

```
src/
  lib.rs          ← объявляет: mod utils; mod parser;
  utils.rs        ← тело модуля utils (простой файл)
  parser/
    mod.rs        ← тело модуля parser
    lexer.rs      ← под-модуль parser::lexer (если parser объявляет: mod lexer;)
```

В `lib.rs`:
```rust
mod utils;          // → ищет src/utils.rs
mod parser;         // → ищет src/parser/mod.rs (или src/parser.rs)
```

### Пути (`use`) и реэкспорт (`pub use`)

Чтобы не писать полный путь каждый раз, используй `use`:

```rust
use std::collections::HashMap;  // теперь можно писать просто HashMap

use crate::utils::helper;       // crate:: — корень ТЕКУЩЕГО крейта
use super::sibling_fn;          // super:: — родительский модуль
```

`pub use` позволяет «перенести» имя наружу — так API крейта выглядит плоским, хотя
внутри всё разложено по файлам:

```rust
// в src/lib.rs
mod internal;
pub use internal::SomeStruct;   // пользователь видит: use my_crate::SomeStruct;
                                // а не: use my_crate::internal::SomeStruct;
```

## Идея 3. Крейт, workspace и Cargo.toml / Cargo.lock

**Крейт** — единица компиляции в Rust: один `Cargo.toml` → один крейт (библиотека
или бинарник). У каждого крейта есть имя (`name`), версия (`version`) и список
зависимостей (`[dependencies]`).

```toml
# Cargo.toml этого модуля
[package]
name = "m14_tooling"
version = "0.1.0"
edition = "2021"

[dependencies]
# пока нет внешних зависимостей
```

**Workspace** — когда несколько крейтов живут в одном репозитории и собираются вместе.
В корневом `Cargo.toml` перечислены `members`:

```toml
# rust/Cargo.toml
[workspace]
members = [
    "modules/01-variables",
    "modules/02-ownership",
    # ...
    "modules/14-tooling",
]
```

Команда `cargo test -p m14_tooling`, запущенная из корня workspace, тестирует только
этот крейт, не пересобирая остальные без нужды.

### Cargo.toml vs Cargo.lock

| Файл | Что хранит | Коммитить? |
|------|-----------|-----------|
| `Cargo.toml` | Требования к версиям зависимостей (`"^1.0"`) — это *манифест* | Всегда |
| `Cargo.lock` | Точные версии, которые реально скачаны и собраны — это *снимок* | Для бинарников: да; для библиотек: обычно нет |

Cargo.lock нужен, чтобы два разработчика (или CI) собирали проект с **одинаковыми**
версиями транзитивных зависимостей. Без него одна машина может получить `serde 1.0.195`,
другая — `serde 1.0.196`, и поведение расходится.

## Идея 4. Тесты: юнит (`#[cfg(test)]`) и интеграционные

Тесты в Rust бывают двух видов.

**Юнит-тесты** живут рядом с кодом, в том же файле, в модуле под `#[cfg(test)]`. Они
видят даже приватные функции и компилируются только при `cargo test` (в обычной сборке
их как будто нет):

```rust
fn double(x: i64) -> i64 { x * 2 }

#[cfg(test)]
mod tests {
    use super::*;                 // втянуть всё из родительского модуля

    #[test]
    fn double_works() {
        assert_eq!(double(21), 42);
    }
}
```

### Что такое `#[cfg(test)]` на самом деле

`#[cfg(...)]` — **условная компиляция**: блок включается только если условие истинно.
`test` становится истинным, когда компилятор вызван с флагом `--test` (именно это делает
`cargo test`). В обычном `cargo build` весь блок `#[cfg(test)] mod tests { ... }`
полностью отсутствует в бинарнике — он не занимает ни байта ни времени компиляции.

Тот же механизм используется для платформ: `#[cfg(target_os = "linux")]` или для
фич: `#[cfg(feature = "some_feature")]`.

**Интеграционные тесты** живут в отдельной папке `tests/`. Каждый файл там — отдельный
крейт, который видит твой код **только через публичный API**, как настоящий пользователь:

```rust
// tests/tooling.rs
use m14_tooling::*;

#[test]
fn it_works() {
    assert_eq!(sum_first_n(&[1, 2, 3], 2), 3);
}
```

**ПОЧЕМУ два вида.** Юнит-тест проверяет деталь изнутри (в том числе приватную логику).
Интеграционный проверяет, что снаружи всё собирается в рабочее целое. В этом модуле тесты
в `tests/tooling.rs` — интеграционные: они задают эталонное поведение, к которому ты
приводишь функции.

### Третий вид: doc-тесты (документация, которая не врёт)

Примеры кода внутри `///`-комментариев — это **тоже тесты**. Команда
`cargo test --doc` (или просто `cargo test`, которая гоняет и их) **компилирует и
запускает** каждый блок ` ``` ` в документации:

```rust
/// Переводит градусы Цельсия в Фаренгейт.
///
/// # Examples
///
/// ```
/// use m14_tooling::celsius_to_fahrenheit;
/// assert_eq!(celsius_to_fahrenheit(0.0), 32.0);
/// ```
pub fn celsius_to_fahrenheit(c: f64) -> f64 { c * 9.0 / 5.0 + 32.0 }
```

**ПОЧЕМУ doc-тесты — это уникально полезно.** Обычный пример в документации может
устареть молча: код поменялся, пример в комментарии остался старым, читатель копирует
его — и оно не работает. Doc-тест делает это невозможным: если пример перестал
компилироваться или его `assert_eq!` упал — `cargo test` **краснеет**. Документация
физически не может разойтись с кодом. Это «исполняемая правда» о том, как пользоваться
функцией.

**Подвох 1: doc-тест видит крейт снаружи.** Он выполняется как код пользователя, поэтому
ему нужен `use my_crate::...;` (приватные элементы недоступны). В этом модуле doc-тест
у `celsius_to_fahrenheit` начинается с `use m14_tooling::celsius_to_fahrenheit;`.

**Подвох 2: на стабе `todo!()` doc-тест падает в рантайме.** Тело-заглушка `todo!()`
компилируется (поэтому скелет собирается), но при запуске doc-теста паникует — тест
красный, как и положено, пока ты не починишь функцию.

```bash
cargo test --doc        # запустить ТОЛЬКО doc-тесты
cargo test -p m14_tooling --doc   # только этого крейта
```

### `assert_eq!` vs `assert_ne!` vs `assert!`

Три макроса-проверки, и у каждого своя роль:

| Макрос | Проверяет | При провале печатает |
|--------|-----------|----------------------|
| `assert_eq!(a, b)` | `a == b` (равны) | `left` и `right` — оба значения |
| `assert_ne!(a, b)` | `a != b` (НЕ равны) | `left` и `right` — оба значения |
| `assert!(cond)` | `cond == true` | только текст условия |

**ПОЧЕМУ `assert_eq!` лучше, чем `assert!(a == b)`.** Если `assert!(a == b)` упадёт, ты
увидишь лишь «assertion failed: a == b» — без значений. `assert_eq!` напечатает и
ожидаемое (`right`), и фактическое (`left`) — сразу видно, на сколько разошлось. Поэтому
для равенства всегда бери `assert_eq!`, а не голый `assert!`.

`assert_ne!` нужен, когда важно доказать **различие**: например, что после мутации
значение изменилось, или что два сгенерированных id не совпали:

```rust
let id1 = make_id();
let id2 = make_id();
assert_ne!(id1, id2, "два id не должны совпадать");
```

### `--nocapture`: куда делся мой `println!`

По умолчанию во время `cargo test` **вывод проходящих тестов скрывается**: их `println!`
не показывается, чтобы не засорять отчёт. Это сбивает с толку, когда отлаживаешь тест
печатью — кажется, что `println!` не работает. На самом деле он работает, но cargo
«перехватывает» (captures) поток вывода и показывает его только у **упавших** тестов.

```bash
cargo test -- --nocapture    # НЕ перехватывать — показать ВЕСЬ println! из тестов
```

**ПОЧЕМУ так по умолчанию.** В большом проекте тысячи зелёных тестов залили бы консоль
мусором. Cargo показывает вывод лишь там, где он нужен — у красных. Когда тебе нужно
увидеть печать из зелёного теста, ты явно просишь `--nocapture`. (`--` отделяет флаги
cargo от флагов самого тест-бинарника — `--nocapture` адресован именно ему.)

### Продвинутые возможности тестирования

**`#[should_panic]`** — проверяет, что функция **обязана** паниковать. Тест считается
пройденным, только если паника действительно случилась (и, при `expected`, с нужным
сообщением). Если функция НЕ паникует — тест красный.

```rust
#[test]
#[should_panic(expected = "division by zero")]
fn dividing_by_zero_panics() {
    let _ = checked_div(1, 0);   // ОБЯЗАНА паниковать с этим сообщением
}
```

**ПОЧЕМУ `expected = "..."` важен.** Без него тест ловит *любую* панику — даже не ту,
которую ты имел в виду (например, функция упала по другой причине, а тест «прошёл»).
Подстрока в `expected` фиксирует, что паника именно та, что в контракте. В этом модуле
функция `checked_div(a, b)` паникует на `b == 0` — это документированный контракт
(деление на ноль — нарушение инварианта, а не пользовательская ошибка), и
`#[should_panic(expected = "division by zero")]` это проверяет.

**Сообщения в `assert!`** — добавляй пояснение, что именно не так:

```rust
assert_eq!(result, 42, "при входе {} ожидали {}, получили {}", input, 42, result);
assert!(x > 0, "x должен быть положительным, но равен {x}");
```

**Фильтрация тестов** — запускай только нужные:

```bash
cargo test wf_empty          # все тесты, в имени которых есть "wf_empty"
cargo test prop_             # только property-тесты
cargo test -- --nocapture    # показать println! из тестов (по умолчанию скрыт)
```

**`RUST_BACKTRACE`** — стектрейс при панике:

```bash
RUST_BACKTRACE=1 cargo test  # полный стектрейс при упавшем тесте
```

Полезные макросы: `assert_eq!(a, b)`, `assert!(условие)`, `assert_ne!(a, b)`.

## Идея 5. Git для джуна: status / add / commit / log / diff / branch

`git` — машина времени для кода. Минимальный набор команд, который нужен каждый день:

```bash
git status                    # что изменилось, что готово к коммиту
git diff                      # ПОКАЗАТЬ сами изменения (построчно: было/стало)
git add src/lib.rs            # отметить файл для следующего коммита (в «область подготовки»)
git commit -m "fix: почини off-by-one в sum_first_n"   # зафиксировать снимок
git log --oneline             # история коммитов, по строчке на коммит
git branch                    # список веток; git branch fix-bugs — создать ветку
git switch fix-bugs           # переключиться на ветку (или git checkout fix-bugs)
```

### Три слоя git: рабочее дерево, индекс, HEAD

Git хранит изменения в трёх местах — важно понимать, где что находится:

```
рабочее дерево (working tree)
  │  ← ты редактируешь файлы здесь
  │  git add file.rs
  ▼
индекс / staging area (index)
  │  ← файлы, отмеченные для следующего коммита
  │  git commit -m "..."
  ▼
HEAD (последний коммит текущей ветки)
```

**`git diff`** без аргументов показывает разницу между рабочим деревом и **индексом**
(то, что изменено, но ещё не добавлено в `add`).

**`git diff --staged`** показывает разницу между **индексом** и **HEAD** (то, что будет
зафиксировано в следующем `commit`). Это самопроверка перед коммитом.

```bash
# Типичный workflow проверки перед коммитом:
git diff                # смотришь на незастейдженные изменения
git add src/lib.rs      # добавляешь нужный файл
git diff --staged       # убеждаешься, что собираешься закоммитить именно это
git commit -m "fix: ..."
```

**ПОЧЕМУ это важно.** Коммит — это контрольная точка, к которой всегда можно вернуться.
Чини баг → запускай тесты → как стало зелено, делай `git commit`. Тогда любой следующий
эксперимент безопасен: что бы ты ни сломал дальше, последнее рабочее состояние сохранено.
`git diff --staged` перед коммитом — это самопроверка: ты глазами видишь ровно то, что
меняешь (и часто замечаешь случайную опечатку). Ветки (`branch`) позволяют чинить, не
трогая основную линию, пока правка не готова.

Хороший рабочий цикл этого модуля:

```bash
git switch -c fix-bugs        # отдельная ветка под починку
cargo test                    # увидеть красное
# ... чинишь один баг ...
cargo test                    # один тест позеленел

<details><summary>Подсказка 0. Какие типы и структуры нужны</summary>

Тебе нужен способ считать, сколько раз встречается каждое слово. В Rust для этого
идеально подходит `HashMap<String, usize>` из стандартной библиотеки. Добавь в начало
функции:
```rust

let mut counts: HashMap<String, usize> = HashMap::new();
```
Или импортируй `use std::collections::HashMap;` в верхней части файла.
</details>

<details><summary>Подсказка 1. Как перебрать токены и посчитать их</summary>

`text.split_whitespace()` возвращает итератор по словам, уже пропуская лишние пробелы.
Для каждого токена нужно: получить его в нижнем регистре, найти в HashMap и увеличить
счётчик (или создать с нуля). Есть удобный метод `.entry(...).or_insert(0)`:

```rust
for token in text.split_whitespace() {
    let word = token.to_lowercase();
    *counts.entry(word).or_insert(0) += 1;
}
```

Почему `*`? Потому что `.or_insert(0)` возвращает **изменяемую ссылку** `&mut usize` —
чтобы увеличить значение, нужно разыменовать ссылку звёздочкой.
</details>

<details><summary>Подсказка 2. Как преобразовать HashMap в Vec и отсортировать</summary>

Из HashMap в Vec:
```rust
let mut pairs: Vec<(String, usize)> = counts.into_iter().collect();
```

Для сортировки используй `.sort_by(|a, b| ...)`. Компаратор принимает два элемента
и возвращает `Ordering` (Less / Equal / Greater). Твой порядок:
- сначала по `count` убыванию: `b.1.cmp(&a.1)`,
- при равенстве — по `word` возрастанию: `.then_with(|| a.0.cmp(&b.0))`.

Метод `.then_with(|| ...)` на `Ordering` — это «если сравнение дало Equal, примени
вторичный компаратор».
</details>

<details><summary>Подсказка 1. sum_first_n</summary>

Посмотри на границы цикла. С какого индекса ты начинаешь складывать? Индексы среза идут
с `0`. Если цикл стартует с `1`, самый первый элемент в сумму не попадёт. Проверь себя
тестом `sum_first_n(&[5], 1)` — должно быть `5`.
</details>

<details><summary>Подсказка 2. max_in</summary>

С какого значения честно начинать поиск максимума? Если начать с `0`, то для данных
`[-5, -2, -9]` функция вернёт `0` — а такого числа в данных нет. Начни «текущий максимум»
с **первого элемента** среза (`v[0]`), а сравнивай его с остальными.
</details>

<details><summary>Подсказка 3. average</summary>

Две ловушки в одной строке. Первая: на сколько делим? На **количество** элементов
(`v.len()`), а не на `len - 1`. Вторая: если делить целые числа (`i64 / i64`), Rust
отбросит дробную часть (`7 / 2 == 3`). Приведи и сумму, и количество к `f64` **до**
деления: `sum as f64 / v.len() as f64`.
</details>

<details><summary>Подсказка 4. has_duplicate</summary>

Внутренний цикл сравнивает `v[i]` с `v[j]`. Если `j` начинается с того же `i`, первая же
проверка — это `v[i] == v[i]`, что всегда истина. Пусть `j` начинается **после** `i`, то
есть с `i + 1`. Тогда ты сравниваешь каждый элемент только с идущими следом, и `[1, 2, 3]`
честно даст `false`.
</details>

<details><summary>Подсказка 5. repeat_str</summary>

Сколько раз крутится цикл? Чтобы повторить строку `times` раз, нужно `times` итераций, а
не `times - 1`. Диапазон `0..times` даёт ровно `times` шагов. Проверь `repeat_str("hi", 1)`
— должно быть `"hi"`, а не `""`. Случай `times == 0` уже обработан отдельно — оставь его.
</details>
