# Модуль 14 — Инструменты и качество кода

Большая идея: писать код — половина дела. Вторая половина — **инструменты**, которые
ловят твои ошибки раньше, чем их увидит пользователь, и держат проект в порядке.
Компилятор, тесты, линтер, форматтер и система контроля версий — это не бюрократия, а
твоя страховочная сетка. В этом модуле ты не пишешь логику с нуля: код уже написан, но
**в каждой функции спрятан баг**. Твоя работа — джунская работа на каждый день: запустить
тесты, прочитать красный, найти место расхождения, починить. Это и есть профессия.

> Этот модуль особенный. В стабе нет ни одного `todo!()` — код компилируется и
> «как будто работает». Но тесты красные, потому что в каждой функции — ровно один баг
> (классическая ошибка новичка). Найди и почини.

## Идея 1. `cargo`: build, test, run — и его друзья clippy и fmt

`cargo` — это сборщик, менеджер зависимостей и раннер тестов в одном лице.

```bash
cargo build      # собрать проект (проверить, что компилируется)
cargo test       # собрать и прогнать ВСЕ тесты
cargo run        # собрать и запустить бинарник (для проектов с main)
```

Помимо сборки есть два инструмента качества, которые стоит запускать постоянно:

```bash
cargo clippy     # ЛИНТЕР: ищет подозрительный код и анти-паттерны
cargo fmt        # ФОРМАТТЕР: приводит код к единому каноническому стилю
```

**ПОЧЕМУ это важно.** Компилятор проверяет, что код *корректен по типам*. `clippy` идёт
дальше и проверяет, что код *написан хорошо*: он замечает `x == x`, бессмысленные
`if b { true } else { false }`, лишние `.clone()` и десятки других «пахнущих» мест.
Часто баг из этого модуля clippy подсветит сам. `fmt` же убирает споры о стиле: отступы,
пробелы, переносы — всё единообразно, и баг проще увидеть глазами, когда форматирование
не мешает.

Запусти их на этом модуле:

```bash
cd rust/modules/14-tooling
cargo clippy        # прочитай предупреждения — они подсказывают, где искать
cargo fmt           # выровняй стиль
```

## Идея 2. Модули (`mod`) и видимость (`pub`)

Код в Rust разложен по **модулям** — это пространства имён. Всё, что объявлено без
`pub`, **приватно** и видно только внутри своего модуля. `pub` открывает элемент наружу.

```rust
pub fn visible_everywhere() {}   // часть публичного API крейта
fn helper() {}                   // приватная, для внутреннего пользования

mod math {                       // вложенный модуль
    pub fn add(a: i64, b: i64) -> i64 { a + b }   // math::add доступно снаружи модуля
    fn secret() {}                                // видно только внутри math
}
```

**ПОЧЕМУ это важно.** Граница `pub` — это контракт. Всё публичное ты обязан поддерживать
(на это полагаются другие); всё приватное можешь менять свободно. Чем меньше `pub`, тем
меньше поверхность, которую легко сломать. В этом модуле все пять функций — `pub`, потому
что тесты лежат в отдельном крейте и видят только публичный API (`use m14_tooling::*;`).

## Идея 3. Тесты: юнит (`#[cfg(test)]`) и интеграционные

Тесты в Rust бывают двух видов.

**Юнит-тесты** живут рядом с кодом, в том же файле, в модуле под `#[cfg(test)]`. Они
видят даже приватные функции и компилируются только при `cargo test` (в обычной сборке
их как будто нет):

```rust
fn double(x: i64) -> i64 { x * 2 }

#[cfg(test)]
mod tests {
    use super::*;                 // втянуть всё из родительского модуля

    #[test]
    fn double_works() {
        assert_eq!(double(21), 42);
    }
}
```

**Интеграционные тесты** живут в отдельной папке `tests/`. Каждый файл там — отдельный
крейт, который видит твой код **только через публичный API**, как настоящий пользователь:

```rust
// tests/tooling.rs
use m14_tooling::*;

#[test]
fn it_works() {
    assert_eq!(sum_first_n(&[1, 2, 3], 2), 3);
}
```

**ПОЧЕМУ два вида.** Юнит-тест проверяет деталь изнутри (в том числе приватную логику).
Интеграционный проверяет, что снаружи всё собирается в рабочее целое. В этом модуле тесты
в `tests/tooling.rs` — интеграционные: они задают эталонное поведение, к которому ты
приводишь функции. Полезные макросы: `assert_eq!(a, b)`, `assert!(условие)`,
`assert_ne!(a, b)`.

## Идея 4. Git для джуна: status / add / commit / log / diff / branch

`git` — машина времени для кода. Минимальный набор команд, который нужен каждый день:

```bash
git status                    # что изменилось, что готово к коммиту
git diff                      # ПОКАЗАТЬ сами изменения (построчно: было/стало)
git add src/lib.rs            # отметить файл для следующего коммита (в «область подготовки»)
git commit -m "fix: почини off-by-one в sum_first_n"   # зафиксировать снимок
git log --oneline             # история коммитов, по строчке на коммит
git branch                    # список веток; git branch fix-bugs — создать ветку
git switch fix-bugs           # переключиться на ветку (или git checkout fix-bugs)
```

**ПОЧЕМУ это важно.** Коммит — это контрольная точка, к которой всегда можно вернуться.
Чини баг → запускай тесты → как стало зелено, делай `git commit`. Тогда любой следующий
эксперимент безопасен: что бы ты ни сломал дальше, последнее рабочее состояние сохранено.
`git diff` перед коммитом — это самопроверка: ты глазами видишь ровно то, что меняешь
(и часто замечаешь случайную опечатку). Ветки (`branch`) позволяют чинить, не трогая
основную линию, пока правка не готова.

Хороший рабочий цикл этого модуля:

```bash
git switch -c fix-bugs        # отдельная ветка под починку
cargo test                    # увидеть красное
# ... чинишь один баг ...
cargo test                    # один тест позеленел
git add -A && git commit -m "fix: max_in стартует с первого элемента"
# ... повторяешь, пока всё не зелёное ...
```

## Задания

В `src/lib.rs` пять функций уже написаны и **компилируются**, но в каждой ровно один баг.
Тесты в `tests/tooling.rs` задают ПРАВИЛЬНОЕ поведение. Сделай красное зелёным.

1. `sum_first_n(v: &[i64], n: usize) -> i64` — сумма первых `n` элементов. Баг: **off-by-one**
   (теряется первый элемент).
2. `max_in(v: &[i64]) -> i64` — максимум непустого среза. Баг: **неверная инициализация**
   (старт с `0` ломает случай, когда все числа отрицательные).
3. `average(v: &[i64]) -> f64` — среднее арифметическое. Баг: **неверный делитель и
   целочисленное деление** (дробная часть теряется).
4. `has_duplicate(v: &[i64]) -> bool` — есть ли повтор значения. Баг: **элемент сравнивается
   сам с собой** (всегда «находит» дубликат).
5. `repeat_str(s: &str, times: usize) -> String` — повторить строку `times` раз. Баг:
   **off-by-one** (`times - 1` повторов вместо `times`).

Запуск тестов:

```bash
./rust/run.sh 14
```

### Ступенчатые подсказки

<details><summary>Подсказка 0. С чего вообще начать</summary>

Запусти `cargo test` (или `./rust/run.sh 14`). Каждый упавший тест печатает, что он
ожидал и что получил (`left` / `right` в `assert_eq!`). Открой функцию из имени теста и
сравни тело с тем, что просит тест. Дополнительно прогони `cargo clippy` — некоторые баги
он назовёт прямо.
</details>

<details><summary>Подсказка 1. sum_first_n</summary>

Посмотри на границы цикла. С какого индекса ты начинаешь складывать? Индексы среза идут
с `0`. Если цикл стартует с `1`, самый первый элемент в сумму не попадёт. Проверь себя
тестом `sum_first_n(&[5], 1)` — должно быть `5`.
</details>

<details><summary>Подсказка 2. max_in</summary>

С какого значения честно начинать поиск максимума? Если начать с `0`, то для данных
`[-5, -2, -9]` функция вернёт `0` — а такого числа в данных нет. Начни «текущий максимум»
с **первого элемента** среза (`v[0]`), а сравнивай его с остальными.
</details>

<details><summary>Подсказка 3. average</summary>

Две ловушки в одной строке. Первая: на сколько делим? На **количество** элементов
(`v.len()`), а не на `len - 1`. Вторая: если делить целые числа (`i64 / i64`), Rust
отбросит дробную часть (`7 / 2 == 3`). Приведи и сумму, и количество к `f64` **до**
деления: `sum as f64 / v.len() as f64`.
</details>

<details><summary>Подсказка 4. has_duplicate</summary>

Внутренний цикл сравнивает `v[i]` с `v[j]`. Если `j` начинается с того же `i`, первая же
проверка — это `v[i] == v[i]`, что всегда истина. Пусть `j` начинается **после** `i`, то
есть с `i + 1`. Тогда ты сравниваешь каждый элемент только с идущими следом, и `[1, 2, 3]`
честно даст `false`.
</details>

<details><summary>Подсказка 5. repeat_str</summary>

Сколько раз крутится цикл? Чтобы повторить строку `times` раз, нужно `times` итераций, а
не `times - 1`. Диапазон `0..times` даёт ровно `times` шагов. Проверь `repeat_str("hi", 1)`
— должно быть `"hi"`, а не `""`. Случай `times == 0` уже обработан отдельно — оставь его.
</details>
