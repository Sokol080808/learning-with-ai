# Модуль 08 — Коллекции: Vec, String, HashMap

Большая идея: коллекции — это контейнеры, которые **владеют** своими элементами.
В Rust это не мелочь, а половина смысла. Когда ты кладёшь значение в `Vec` или
`HashMap`, ты не «ставишь ссылку рядом» — ты **передаёшь владение** контейнеру.
Он будет хранить элементы, пока жив сам, и аккуратно их освободит, когда умрёт.
Поэтому, читая сигнатуру функции с коллекцией, первым делом спрашивай себя:
*мы это значение заимствуем (`&str`, `&[T]`) или забираем себе (`String`, `Vec<T>`)?*
Ответ диктует всё остальное.

В стандартной библиотеке коллекций много, но три рабочие лошадки покрывают почти
всё: `Vec<T>` (растущий массив), `String` (растущая строка) и `HashMap<K, V>`
(словарь). Их и осваиваем.

## Идея 1. `Vec<T>` — растущий массив; `String` против `&str`

`Vec<T>` — это динамический массив: непрерывный кусок памяти, который умеет расти.

```rust
let mut v: Vec<i64> = Vec::new();
v.push(10);            // добавить в конец
v.push(20);
let first = v[0];      // индексация (паникует, если выйти за границы)
let len = v.len();     // длина
for x in &v {          // итерация ПО ССЫЛКЕ — вектор остаётся нашим
    println!("{x}");
}
```

Со строками в Rust есть пара, которая поначалу путает: **`String` и `&str`**.

- `&str` (читается «стринг-слайс») — это *заимствованный взгляд* на чьи-то
  UTF-8-байты. Мы их не двигаем и не освобождаем — только смотрим. Литерал
  `"привет"` имеет тип `&str`.
- `String` — это *владеющая*, растущая строка (по сути `Vec<u8>` с гарантией
  корректного UTF-8). Ей мы владеем: можем менять, дописывать, а когда `String`
  выйдет из области видимости — её память освободится.

```rust
let borrowed: &str = "hi";        // смотрим на чужое (тут — на статические данные)
let mut owned: String = String::from("hi");
owned.push_str(" there");         // owned == "hi there"
let view: &str = &owned;          // из String легко получить &str (заимствование)
```

**ПОЧЕМУ так?** Это та же история, что и везде в Rust: тип отвечает на вопрос «кто
освобождает память». `&str` ничего не освобождает (он чужой), `String` освобождает
себя сам. Принимай `&str` в параметрах, когда тебе достаточно *прочитать* строку —
это самая гибкая сигнатура: в неё подойдёт и литерал, и `&String`. А `String`
возвращай/храни, когда значение должно *пережить* вызов.

> Подвох: `String` индексировать как `s[0]` нельзя — Rust не даст. Причина —
> UTF-8: один символ может занимать несколько байт, и «байт номер 0» не обязан быть
> цельным символом. Чтобы ходить по символам, используют `s.chars()`.

## Идея 2. `HashMap<K, V>` и API `entry()`

`HashMap<K, V>` — словарь: по ключу `K` достаём значение `V` за в среднем O(1).

```rust
use std::collections::HashMap;

let mut scores: HashMap<String, i64> = HashMap::new();
scores.insert("alice".to_string(), 10);   // положить/перезаписать
let a = scores.get("alice");              // -> Option<&i64> (Some(&10))
let missing = scores.get("bob");          // -> None
```

`get` возвращает `Option<&V>` — `Some(&значение)` или `None`. Это снова Rust во всей
красе: тип заставляет тебя честно обработать случай «ключа нет», вместо тихого
получения мусора.

Самый частый паттерн с картами — **«взять текущее значение по ключу, а если его нет,
завести по умолчанию»**. Для счётчиков это идеально ложится на `entry()`:

```rust
let mut freq: HashMap<char, usize> = HashMap::new();
for c in "aab".chars() {
    *freq.entry(c).or_insert(0) += 1;     // нет ключа -> заводим 0, затем +1
}
// freq == {'a': 2, 'b': 1}
```

Разберём по косточкам:
- `freq.entry(c)` — «дай мне *место* для ключа `c`», даже если его ещё нет;
- `.or_insert(0)` — «если место пустует, положи туда `0`»; возвращает `&mut V` —
  изменяемую ссылку на значение в карте;
- `*... += 1` — разыменовываем эту ссылку и увеличиваем счётчик прямо внутри карты.

**ПОЧЕМУ `entry`, а не `if let Some(...) = get(...)`?** Потому что `entry` находит
нужную ячейку **за один поиск**, а ручной вариант «проверь, потом вставь» искал бы
ключ дважды и заставил бы тебя бороться с borrow checker (одновременно читать и менять
карту). `entry` — это идиоматичный, дешёвый и безопасный способ.

## Идея 3. Владение значениями внутри коллекции

Когда ключ карты — `String`, а в исходном тексте слова — это `&str` (заимствованные
куски чужой строки), **просто положить `&str` нельзя**: чужие данные могут исчезнуть
раньше карты, и она осталась бы с висячими ссылками. Rust такого не допускает.

```rust
let mut count: HashMap<String, usize> = HashMap::new();
for word in text.split_whitespace() {   // word: &str — кусочек чужого text
    *count.entry(word.to_string())      // .to_string() -> своя String (КОПИЯ)
        .or_insert(0) += 1;             // теперь карта ВЛАДЕЕТ ключом
}
```

`word.to_string()` создаёт **новую `String`**, которой владеет карта. Теперь карта не
зависит от того, жив ли исходный `text`: у неё своя копия данных. Это и есть «коллекция
владеет своими элементами» в действии.

> Подвох: `char` и числа (`i64`, `usize`) — это `Copy`-типы; их карта «копирует»
> бесплатно, и тут думать не о чем. А вот `String`/`Vec` — не `Copy`: их надо явно
> отдать (move) или клонировать. Поэтому в `char_freq` возни с владением нет, а в
> `word_count` — есть.

## Идея 4. Итерация и преобразование

Коллекции удобно перебирать и превращать одну в другую. Есть три способа итерации,
и разница — снова про владение:

```rust
for x in &v      { /* x: &T      — читаем, вектор остаётся нашим */ }
for x in &mut v  { /* x: &mut T  — меняем элементы на месте       */ }
for x in v       { /* x: T       — забираем элементы, v потребляется (move) */ }
```

Полезные методы, которые пригодятся в заданиях:

```rust
let mut v = vec![3, 1, 2, 3, 1];
v.sort();          // на месте, по возрастанию: [1, 1, 2, 3, 3]
v.dedup();         // убирает ПОДРЯД идущие дубликаты: [1, 2, 3]
                   // (поэтому сначала sort, потом dedup!)

let words = ["a", "b", "c"];
let joined: String = words.join(", ");   // "a, b, c" — разделитель только МЕЖДУ
```

**ПОЧЕМУ `dedup` убирает только соседние повторы?** Чтобы работать за один проход и
без аллокаций — он сравнивает каждый элемент только с предыдущим. Если нужно убрать
*все* дубликаты, сперва приводят равные значения в соседство — то есть сортируют.
Связка `sort` + `dedup` и даёт «уникальные по возрастанию».

### Подсказки

<details><summary>char_freq — с чего начать</summary>

Заведи пустую `HashMap::new()`, пройди по `s.chars()` и для каждого символа сделай
`*map.entry(c).or_insert(0) += 1`. Это ровно паттерн-счётчик из Идеи 2. Возвращать
можно прямо собранную карту.
</details>

<details><summary>dedup_sorted — две строчки и одна тонкость</summary>

Раз `v` — твой (по значению), сделай `v` изменяемым (`mut`), затем `v.sort()` и
`v.dedup()`. Важна **последовательность**: `dedup` убирает только соседние повторы,
поэтому без предварительного `sort` он не уберёт разнесённые дубликаты. Верни `v`.
</details>

<details><summary>word_count — почему .to_string()</summary>

Разбей текст на слова через `text.split_whitespace()` — он сам схлопывает любые
пробелы и не отдаёт пустых слов. Каждое слово — это `&str` (кусок чужого `text`), а
ключ карты должен быть `String`, которым карта владеет. Поэтому ключ — это
`word.to_string()`. Дальше — тот же `entry(...).or_insert(0) += 1`.
</details>

<details><summary>join_words — без хитростей и вручную</summary>

Самый короткий путь — `words.join(sep)` (метод срезов делает ровно то, что нужно:
разделитель только между элементами, пустой срез даёт `""`).

Хочешь почувствовать механику руками? Собери результат сам: заведи пустую `String`,
иди по `words.iter().enumerate()` и перед каждым элементом, кроме нулевого (`i > 0`),
дописывай `sep` через `push_str`, затем сам элемент. Сравни оба подхода — поймёшь, что
делает `join` под капотом.
</details>
