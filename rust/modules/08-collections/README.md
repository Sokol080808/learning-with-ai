# Модуль 08 — Коллекции: Vec, String, HashMap

Большая идея: коллекции — это контейнеры, которые **владеют** своими элементами.
В Rust это не мелочь, а половина смысла. Когда ты кладёшь значение в `Vec` или
`HashMap`, ты не «ставишь ссылку рядом» — ты **передаёшь владение** контейнеру.
Он будет хранить элементы, пока жив сам, и аккуратно их освободит, когда умрёт.
Поэтому, читая сигнатуру функции с коллекцией, первым делом спрашивай себя:
*мы это значение заимствуем (`&str`, `&[T]`) или забираем себе (`String`, `Vec<T>`)?*
Ответ диктует всё остальное.

В стандартной библиотеке коллекций много, но три рабочие лошадки покрывают почти
всё: `Vec<T>` (растущий массив), `String` (растущая строка) и `HashMap<K, V>`
(словарь). Их и осваиваем.

## Идея 1. `Vec<T>` — растущий массив; `String` против `&str`

`Vec<T>` — это динамический массив: непрерывный кусок памяти, который умеет расти.

```rust
let mut v: Vec<i64> = Vec::new();
v.push(10);            // добавить в конец
v.push(20);
let first = v[0];      // индексация (паникует, если выйти за границы)
let len = v.len();     // длина
for x in &v {          // итерация ПО ССЫЛКЕ — вектор остаётся нашим
    println!("{x}");
}
```

### Как Vec растёт под капотом: ёмкость и перераспределение памяти

`Vec<T>` хранит три числа: указатель на буфер, **длину** (`len`) — сколько элементов
сейчас, и **ёмкость** (`capacity`) — сколько влезет без нового выделения памяти.

Когда `push` видит, что `len == capacity`, вектор запрашивает у аллокатора вдвое
больший кусок памяти, копирует все старые элементы туда и освобождает старый буфер.
Звучит дорого, но на практике это происходит редко: каждое удвоение «размазывает»
стоимость по всем предыдущим `push`-ам. В итоге амортизированная стоимость одного
`push` — **O(1)**.

```rust
let mut v: Vec<i32> = Vec::new();
// capacity == 0, len == 0
v.push(1);
println!("len={}, cap={}", v.len(), v.capacity()); // len=1, cap=1 (или 4 — зависит от платформы)
v.push(2); v.push(3); v.push(4); v.push(5);
println!("len={}, cap={}", v.len(), v.capacity()); // len=5, cap=8 (после очередного удвоения)
```

Если ты заранее **знаешь** примерное число элементов — скажи об этом вектору:

```rust
let mut v: Vec<i32> = Vec::with_capacity(1000);
// capacity == 1000, len == 0; реаллокаций при первой тысяче push не будет
for i in 0..1000 {
    v.push(i);
}
```

`with_capacity` — не ограничение сверху, а **подсказка** аллокатору. Вектор всё
равно вырастет дальше, если понадобится. Но если тысяча элементов — весь ожидаемый
объём, ты избегаешь нескольких перераспределений и лишних копирований.

> **Подвох: реаллокация инвалидирует ссылки.** После любого `push`, который вызвал
> перераспределение, все ссылки `&v[i]` и итераторы по вектору становятся висячими.
> Rust не даст тебе держать такую ссылку живой через `push` — это защищает от
> целого класса ошибок использования памяти после освобождения.

### `String` против `&str`

Со строками в Rust есть пара, которая поначалу путает: **`String` и `&str`**.

- `&str` (читается «стринг-слайс») — это *заимствованный взгляд* на чьи-то
  UTF-8-байты. Мы их не двигаем и не освобождаем — только смотрим. Литерал
  `"привет"` имеет тип `&str`.
- `String` — это *владеющая*, растущая строка (по сути `Vec<u8>` с гарантией
  корректного UTF-8). Ей мы владеем: можем менять, дописывать, а когда `String`
  выйдет из области видимости — её память освободится.

```rust
let borrowed: &str = "hi";        // смотрим на чужое (тут — на статические данные)
let mut owned: String = String::from("hi");
owned.push_str(" there");         // owned == "hi there"
let view: &str = &owned;          // из String легко получить &str (заимствование)
```

**ПОЧЕМУ так?** Это та же история, что и везде в Rust: тип отвечает на вопрос «кто
освобождает память». `&str` ничего не освобождает (он чужой), `String` освобождает
себя сам. Принимай `&str` в параметрах, когда тебе достаточно *прочитать* строку —
это самая гибкая сигнатура: в неё подойдёт и литерал, и `&String`. А `String`
возвращай/храни, когда значение должно *пережить* вызов.

> Подвох: `String` индексировать как `s[0]` нельзя — Rust не даст. Причина —
> UTF-8: один символ может занимать несколько байт, и «байт номер 0» не обязан быть
> цельным символом. Чтобы ходить по символам, используют `s.chars()`.

### `Vec::get(i)` против `v[i]`: безопасный доступ vs паника

Есть два способа обратиться к элементу вектора по индексу, и они ведут себя
принципиально по-разному:

```rust
let v = vec![10, 20, 30];

// Синтаксис индексации — знакомый, короткий, но жёсткий:
let x = v[1];  // Ok: x == 20
// let y = v[9];  // ПАНИКА в рантайме: index out of bounds

// Vec::get — возвращает Option<&T>, никогда не паникует:
let a: Option<&i64> = v.get(1);   // Some(&20)
let b: Option<&i64> = v.get(9);   // None — спокойно, без паники
```

**ПОЧЕМУ это важно?** `v[i]` удобен, когда ты *знаешь* на 100%, что индекс корректен
(например, итерируешь `0..v.len()`). Но если индекс приходит извне (от пользователя,
из файла, из другой коллекции) — `get` безопаснее: он возвращает `Option`, и
компилятор заставляет тебя обработать случай «индекса нет».

> **Подвох:** `get` возвращает `&T` (ссылку на элемент *внутри* вектора), а не `T`.
> Чтобы получить значение `Copy`-типа, разыменуй: `v.get(1).copied()` или
> `v.get(1).map(|x| *x)`. Для `&mut T` есть пара: `get_mut(i) -> Option<&mut T>`.

### Мутация строк: `push`, `+`, `format!` и `with_capacity`

Есть несколько способов нарастить `String`, и у каждого своя семантика владения:

```rust
// push(char) — добавить один символ в конец (мутация на месте, дёшево)
let mut s = String::from("hel");
s.push('l');
s.push('o');
// s == "hello"

// push_str(&str) — дописать слайс (мутация на месте, без аллокации, если хватает ёмкости)
s.push_str(", world");
// s == "hello, world"

// + — оператор конкатенации; ВАЖНО: он ПОТРЕБЛЯЕТ левый операнд (move)
let s1 = String::from("hello");
let s2 = String::from(", world");
let s3 = s1 + &s2;  // s1 здесь ПЕРЕМЕЩАЕТСЯ в s3; s1 больше недоступна!
// println!("{s1}"); // ← ошибка компилятора: s1 moved

// format! — самый удобный способ склеить несколько строк; НЕ потребляет ни одну из них
let first = String::from("tic");
let second = String::from("tac");
let third = String::from("toe");
let result = format!("{first}-{second}-{third}"); // все три переменные живы
```

Если ты строишь длинную строку в цикле, заранее выдели под неё память — это та же
идея, что `Vec::with_capacity`, только для строк:

```rust
let words = ["один", "два", "три", "четыре"];
// Считаем примерную длину заранее: сумма слов + пробелы между ними.
let cap: usize = words.iter().map(|w| w.len()).sum::<usize>() + words.len();
let mut result = String::with_capacity(cap);
for (i, w) in words.iter().enumerate() {
    if i > 0 { result.push(' '); }
    result.push_str(w);
}
// result == "один два три четыре", и ни одной реаллокации не было
```

## Идея 2. `HashMap<K, V>` и API `entry()`

`HashMap<K, V>` — словарь: по ключу `K` достаём значение `V` за в среднем O(1).

```rust
use std::collections::HashMap;

let mut scores: HashMap<String, i64> = HashMap::new();
scores.insert("alice".to_string(), 10);   // положить/перезаписать
let a = scores.get("alice");              // -> Option<&i64> (Some(&10))
let missing = scores.get("bob");          // -> None
```

`get` возвращает `Option<&V>` — `Some(&значение)` или `None`. Это снова Rust во всей
красе: тип заставляет тебя честно обработать случай «ключа нет», вместо тихого
получения мусора.

Самый частый паттерн с картами — **«взять текущее значение по ключу, а если его нет,
завести по умолчанию»**. Для счётчиков это идеально ложится на `entry()`:

```rust
let mut freq: HashMap<char, usize> = HashMap::new();
for c in "aab".chars() {
    *freq.entry(c).or_insert(0) += 1;     // нет ключа -> заводим 0, затем +1
}
// freq == {'a': 2, 'b': 1}
```

Разберём по косточкам:
- `freq.entry(c)` — «дай мне *место* для ключа `c`», даже если его ещё нет;
- `.or_insert(0)` — «если место пустует, положи туда `0`»; возвращает `&mut V` —
  изменяемую ссылку на значение в карте;
- `*... += 1` — разыменовываем эту ссылку и увеличиваем счётчик прямо внутри карты.

**ПОЧЕМУ `entry`, а не `if let Some(...) = get(...)`?** Потому что `entry` находит
нужную ячейку **за один поиск**, а ручной вариант «проверь, потом вставь» искал бы
ключ дважды и заставил бы тебя бороться с borrow checker (одновременно читать и менять
карту). `entry` — это идиоматичный, дешёвый и безопасный способ.

### Хеш-функция, трейты Hash + Eq и SipHash

Чтобы тип стал ключом `HashMap`, он обязан реализовать два трейта:
- **`Hash`** — функция, превращающая значение в число (хеш-код);
- **`Eq`** — сравнение на полное равенство (`==`).

Почему оба? `HashMap` сначала вычисляет хеш ключа, чтобы быстро найти нужную
«корзину» (bucket), а затем проверяет равенство, чтобы отличить коллизии.
Важное правило: **если `a == b`, то `hash(a) == hash(b)`**. Нарушение этого
инварианта — источник труднейших ошибок (ключ кладётся в одно место, а ищется
в другом).

По умолчанию Rust использует **SipHash** — хеш-функцию, устойчивую к атакам типа
«засыпание таблицы коллизиями» (HashDoS). Это делает `HashMap` безопасным для
хранения пользовательских входных данных. Платим за это чуть большей скоростью
по сравнению с FNV или xxHash — но для большинства задач это незаметно.

```rust
use std::collections::HashMap;
// String, i64, char и прочие примитивы реализуют Hash + Eq «из коробки».
// Для собственного типа — добавь #[derive(Hash, Eq, PartialEq)]:

#[derive(Hash, Eq, PartialEq, Debug)]
struct Point {
    x: i32,
    y: i32,
}

let mut map: HashMap<Point, &str> = HashMap::new();
map.insert(Point { x: 0, y: 0 }, "origin");
println!("{:?}", map.get(&Point { x: 0, y: 0 })); // Some("origin")
```

### `BTreeMap` — упорядоченная альтернатива

Когда тебе важен **порядок** ключей, используй `BTreeMap<K, V>` из `std::collections`:

```rust
use std::collections::BTreeMap;

let mut m: BTreeMap<String, i32> = BTreeMap::new();
m.insert("banana".to_string(), 3);
m.insert("apple".to_string(), 5);
m.insert("cherry".to_string(), 1);

for (k, v) in &m {
    println!("{k}: {v}");
}
// Выводит в алфавитном порядке: apple: 5, banana: 3, cherry: 1
```

`BTreeMap` хранит ключи в B-дереве: поиск, вставка, удаление — всё O(log n) вместо
O(1). Это медленнее для точечных операций, но даёт отсортированный обход и позволяет
делать диапазонные запросы (`range`). Требование к ключу — трейт `Ord` (полный
порядок), а не `Hash`.

> **Правило выбора:** хочешь максимальную скорость точечных операций — `HashMap`;
> нужен отсортированный обход или диапазон ключей — `BTreeMap`.

### `HashSet` — множество уникальных значений

`HashSet<T>` — это по сути `HashMap<T, ()>`: хранит только ключи без значений.
Используется для быстрой проверки «есть ли элемент?» и операций над множествами:

```rust
use std::collections::HashSet;

let mut seen: HashSet<i32> = HashSet::new();
seen.insert(1);
seen.insert(2);
seen.insert(1); // дубликат — тихо игнорируется

println!("{}", seen.contains(&1)); // true
println!("{}", seen.contains(&3)); // false
println!("{}", seen.len());        // 2

// Пересечение двух множеств
let a: HashSet<i32> = [1, 2, 3].iter().copied().collect();
let b: HashSet<i32> = [2, 3, 4].iter().copied().collect();
let intersection: HashSet<_> = a.intersection(&b).collect();
// {2, 3}
```

### Итерация по `HashMap`: ссылочный режим `for (k, v) in &map`

В Идее 4 ниже показаны три режима итерации по `Vec`. У `HashMap` — та же логика, но
чуть другой синтаксис, который стоит зафиксировать отдельно:

```rust
use std::collections::HashMap;

let mut scores: HashMap<String, i64> = HashMap::new();
scores.insert("alice".to_string(), 10);
scores.insert("bob".to_string(), 7);

// Ссылочный режим: карта остаётся нашей, элементы не потребляются.
// k: &String, v: &i64 — каждая итерация даёт пару ссылок.
for (k, v) in &scores {
    println!("{k}: {v}");   // alice: 10 / bob: 7 (порядок не определён)
}
// scores всё ещё здесь, можно пользоваться дальше.

// Мутирующий режим: можно менять значения на месте.
// k: &String, v: &mut i64
for (_k, v) in &mut scores {
    *v += 1;  // каждому добавляем 1
}

// Потребляющий режим: карта уничтожается, мы забираем пары (String, i64).
let items: Vec<(String, i64)> = scores.into_iter().collect();
// scores здесь больше недоступна — она перемещена
```

**ПОЧЕМУ именно так?** `&scores` вызывает `impl IntoIterator for &HashMap<K, V>`, что
даёт итератор по `(&K, &V)`. Это симметрично `&v` для `Vec<T>`: берёшь ссылку на
коллекцию — получаешь ссылки на элементы. Возможность снова использовать `scores`
после цикла — прямое следствие того, что мы заимствовали, а не переместили.

> **Подвох:** порядок обхода `HashMap` **не определён** и может меняться от запуска к
> запуску. Если тебе нужен предсказуемый порядок — используй `BTreeMap` или собери
> пары в `Vec`, отсортируй и итерируй по нему.

## Идея 3. Владение значениями внутри коллекции

Когда ключ карты — `String`, а в исходном тексте слова — это `&str` (заимствованные
куски чужой строки), **просто положить `&str` нельзя**: чужие данные могут исчезнуть
раньше карты, и она осталась бы с висячими ссылками. Rust такого не допускает.

```rust
let mut count: HashMap<String, usize> = HashMap::new();
for word in text.split_whitespace() {   // word: &str — кусочек чужого text
    *count.entry(word.to_string())      // .to_string() -> своя String (КОПИЯ)
        .or_insert(0) += 1;             // теперь карта ВЛАДЕЕТ ключом
}
```

`word.to_string()` создаёт **новую `String`**, которой владеет карта. Теперь карта не
зависит от того, жив ли исходный `text`: у неё своя копия данных. Это и есть «коллекция
владеет своими элементами» в действии.

> Подвох: `char` и числа (`i64`, `usize`) — это `Copy`-типы; их карта «копирует»
> бесплатно, и тут думать не о чем. А вот `String`/`Vec` — не `Copy`: их надо явно
> отдать (move) или клонировать. Поэтому в `char_freq` возни с владением нет, а в
> `word_count` — есть.

### Deref-coercion: `&String → &str` и `&Vec<T> → &[T]`

В Rust есть механизм **deref coercion** («приведение через разыменование»): компилятор
автоматически вставляет нужные `*` и `&`, чтобы типы совпали. Для коллекций это значит:

- `&String` автоматически превращается в `&str` там, где ожидается `&str`;
- `&Vec<T>` автоматически превращается в `&[T]` (срез) там, где ожидается `&[T]`.

```rust
fn print_str(s: &str) {
    println!("{s}");
}

fn sum_slice(xs: &[i32]) -> i32 {
    xs.iter().sum()
}

let owned = String::from("hello");
print_str(&owned);          // &String → &str — компилятор подставляет deref автоматически

let v = vec![1, 2, 3];
let total = sum_slice(&v);  // &Vec<i32> → &[i32]
```

Это **не копирование** — просто компилятор переинтерпретирует указатель. Именно из-за
deref coercion принято писать `fn foo(s: &str)`, а не `fn foo(s: &String)`: первая
принимает и литералы, и `&String`, и `&str` — одна сигнатура покрывает все случаи.

## Идея 4. Итерация и преобразование

Коллекции удобно перебирать и превращать одну в другую. Есть три способа итерации,
и разница — снова про владение:

```rust
for x in &v      { /* x: &T      — читаем, вектор остаётся нашим */ }
for x in &mut v  { /* x: &mut T  — меняем элементы на месте       */ }
for x in v       { /* x: T       — забираем элементы, v потребляется (move) */ }
```

Полезные методы, которые пригодятся в заданиях:

```rust
let mut v = vec![3, 1, 2, 3, 1];
v.sort();          // на месте, по возрастанию: [1, 1, 2, 3, 3]
v.dedup();         // убирает ПОДРЯД идущие дубликаты: [1, 2, 3]
                   // (поэтому сначала sort, потом dedup!)

let words = ["a", "b", "c"];
let joined: String = words.join(", ");   // "a, b, c" — разделитель только МЕЖДУ
```

**ПОЧЕМУ `dedup` убирает только соседние повторы?** Чтобы работать за один проход и
без аллокаций — он сравнивает каждый элемент только с предыдущим. Если нужно убрать
*все* дубликаты, сперва приводят равные значения в соседство — то есть сортируют.
Связка `sort` + `dedup` и даёт «уникальные по возрастанию».

### Идиома `collect()`: итератор → коллекция

В Rust итераторы — ленивые: `map`, `filter`, `flat_map` и другие не создают никаких
промежуточных коллекций, пока ты явно не попросишь. `collect()` — это момент, когда
ты говоришь «хватит откладывать, материализуй всё в конкретный тип»:

```rust
// Итератор → Vec
let squares: Vec<i32> = (1..=5).map(|x| x * x).collect();
// [1, 4, 9, 16, 25]

// Итератор пар (String, usize) → HashMap
use std::collections::HashMap;
let pairs = vec![("one".to_string(), 1usize), ("two".to_string(), 2)];
let map: HashMap<String, usize> = pairs.into_iter().collect();
// {"one": 1, "two": 2}

// HashMap → Vec — противоположное направление; into_iter() потребляет карту
let items: Vec<(String, usize)> = map.into_iter().collect();
// порядок не определён (HashMap не упорядочен)
```

Обрати внимание: тип результата `collect()` почти всегда нужно **аннотировать**
явно (через `: Vec<_>` или turbofish `::<Vec<_>>()`), иначе компилятор не знает,
во что именно «собирать». Подчёркивание `_` говорит: «тип элементов выведи сам».

```rust
// Два равнозначных способа аннотировать тип:
let v1: Vec<_> = (0..5).collect();
let v2 = (0..5).collect::<Vec<_>>();
```

### Подсказки к заданиям 1–4

<details><summary>char_freq — с чего начать</summary>

Заведи пустую `HashMap::new()`, пройди по `s.chars()` и для каждого символа сделай
`*map.entry(c).or_insert(0) += 1`. Это ровно паттерн-счётчик из Идеи 2. Возвращать
можно прямо собранную карту.
</details>

<details><summary>dedup_sorted — две строчки и одна тонкость</summary>

Раз `v` — твой (по значению), сделай `v` изменяемым (`mut`), затем `v.sort()` и
`v.dedup()`. Важна **последовательность**: `dedup` убирает только соседние повторы,
поэтому без предварительного `sort` он не уберёт разнесённые дубликаты. Верни `v`.
</details>

<details><summary>word_count — почему .to_string()</summary>

Разбей текст на слова через `text.split_whitespace()` — он сам схлопывает любые
пробелы и не отдаёт пустых слов. Каждое слово — это `&str` (кусок чужого `text`), а
ключ карты должен быть `String`, которым карта владеет. Поэтому ключ — это
`word.to_string()`. Дальше — тот же `entry(...).or_insert(0) += 1`.
</details>

<details><summary>join_words — без хитростей и вручную</summary>

Самый короткий путь — `words.join(sep)` (метод срезов делает ровно то, что нужно:
разделитель только между элементами, пустой срез даёт `""`).

иди по `words.iter().enumerate()` и перед каждым элементом, кроме нулевого (`i > 0`),
дописывай `sep` через `push_str`, затем сам элемент. Сравни оба подхода — поймёшь, что
делает `join` под капотом.
</details>

## Задание 5. `top_n_words` — конвейер текстового анализа

Это задача побольше: она собирает весь модуль воедино. Тебе нужно реализовать функцию:

```rust
pub fn top_n_words(text: &str, n: usize) -> Vec<(String, usize)>
```

**Контракт:**
- Разбить `text` на слова по `split_whitespace`; регистр не трогать.
- Подсчитать частоту каждого слова (`HashMap`).
- Вернуть не более `n` самых частых слов в виде вектора пар `(слово, количество)`.
- Сортировка: по убыванию частоты; при равной частоте — по алфавиту (возрастание).
- `n == 0` → пустой вектор; пустой/пробельный текст → пустой вектор.

Пример:
```
top_n_words("a b a c b a", 2)  →  [("a", 3), ("b", 2)]
top_n_words("cherry banana apple", 3)  →  [("apple", 1), ("banana", 1), ("cherry", 1)]
// все три по 1 разу — алфавитный тай-брейкинг
```

**Четыре шага конвейера:**
1. Подсчёт частот — `HashMap` + `entry().or_insert(0) += 1` (уже умеешь).
2. Превращение карты в вектор пар — `into_iter().collect()`.
3. Сортировка вектора — `sort_by(...)` с замыканием по двум ключам.
4. Обрезка до `n` элементов — `truncate(n)`.

### Подсказки к `top_n_words`

<details><summary>Шаг 1: почему into_iter, а не iter?</summary>

`map.iter()` отдаёт `(&K, &V)` — ссылки на карту, которая ещё жива. А нам нужны
*владеющие* `(String, usize)` в векторе. `into_iter()` **потребляет** карту и
отдаёт `(String, usize)` с передачей владения — никаких клонирований не нужно.
</details>

<details><summary>Шаг 2: как написать сортировку с двумя ключами?</summary>

`sort_by` принимает замыкание `|a, b| ordering`. Метод `Ordering::then_with` позволяет
связать два сравнения: «сначала по X, при ничьей — по Y». Подумай: для убывания
частоты ты должен поменять `a` и `b` местами (`b.1.cmp(&a.1)`); для алфавитного
порядка — обычный `a.0.cmp(&b.0)`.
</details>

<details><summary>Шаг 3: truncate vs drain vs split_at</summary>

`vec.truncate(n)` обрезает вектор до `n` элементов **на месте**, освобождая хвост.
Это самый простой и дешёвый способ. `drain` и `split_at` дают другие возможности
(потребить хвост или получить два среза), но здесь они избыточны.
</details>
