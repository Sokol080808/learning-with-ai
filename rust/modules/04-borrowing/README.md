# Модуль 04 — Заимствование и ссылки

В модуле 03 ты узнал про **владение**: у каждого значения ровно один владелец, и
передача по значению **перемещает** (move) — старый владелец теряет доступ. Это
безопасно, но неудобно: неужели, чтобы просто посчитать сумму вектора, придётся
отдать его навсегда? Нет. На помощь приходит **заимствование** — ты даёшь функции
ссылку, она временно ею пользуется и возвращает, а владельцем остаёшься ты.

## Большая идея

Ссылка (`&T`, `&mut T`) — это «одолженный взгляд» на чужое значение. Заимствование
**не перемещает** и **не копирует** — оно лишь даёт временный доступ. А главное —
компилятор следит, чтобы этих ссылок не было слишком много «не тех». Это и есть
**borrow checker**, сердце безопасности Rust. То, что выглядит как «компилятор
ругается», на деле — он ловит за руку гонки данных и висячие ссылки ещё до запуска.

## Идея 1. `&T` — читать, `&mut T` — писать

Есть ровно два вида ссылок:

```rust
let v = vec![1, 2, 3];
let r: &Vec<i32> = &v;        // разделяемая ссылка — ТОЛЬКО ЧТЕНИЕ
println!("{}", r.len());      // читать через r — можно
// r.push(4);                 // ОШИБКА: через &T писать нельзя

let mut w = vec![1, 2, 3];
let m: &mut Vec<i32> = &mut w; // эксклюзивная ссылка — можно ПИСАТЬ
m.push(4);                     // менять через &mut — можно
```

`&T` — *разделяемая* (shared): таких ссылок может быть много одновременно, потому
что читать параллельно безопасно. `&mut T` — *эксклюзивная* (exclusive): пока она
жива, она единственная. **Почему?** Если бы кто-то писал, а кто-то одновременно
читал то же значение — это гонка данных. Rust запрещает саму возможность.

## Идея 2. Правило заимствования: много читателей ИЛИ один писатель

В любой момент времени для одного значения допустимо **либо**:

- сколько угодно `&T` (читателей), **либо**
- ровно одна `&mut T` (писатель),

но **никогда не одновременно**.

```rust
let mut v = vec![1, 2, 3];
let a = &v;          // читатель
let b = &v;          // ещё читатель — ОК, читать параллельно безопасно
println!("{} {}", a.len(), b.len());

let m = &mut v;      // писатель — но только когда a и b уже не используются
m.push(4);
// println!("{}", a.len()); // ОШИБКА, если бы a ещё «жил» рядом с m
```

**Почему именно так?** Это правило — «одна запись XOR много чтений» — математически
исключает гонки данных. Не «осторожно их избегаем», а делаем невозможными на уровне
типов. Бесплатный бонус: тот же механизм потом даст потокобезопасность.

## Идея 3. Ссылки не владеют; заимствование не перемещает

Ключевое отличие от move из модуля 03:

```rust
fn len_of(v: &Vec<i32>) -> usize { v.len() }   // берём ССЫЛКУ

let v = vec![1, 2, 3];
let n = len_of(&v);    // одалживаем v на время вызова
println!("{n} {}", v.len()); // v ВСЁ ЕЩЁ НАШ — заимствование вернулось
```

Сравни: если бы `len_of` принимал `Vec<i32>` по значению, вектор бы переместился
внутрь функции и здесь был бы уже недоступен. Ссылка — это «посмотреть, не забирая».
Когда ссылка выходит из области видимости, владелец ничего не теряет: он и не
расставался с владением.

## Идея 4. ПОЧЕМУ borrow checker предотвращает гонки и висячие ссылки

Borrow checker не вредничает — он доказывает две вещи за тебя:

1. **Нет гонок данных.** Правило «писатель эксклюзивен» означает: пока кто-то пишет,
   никто не читает и не пишет то же место. Гонка невозможна по построению.
2. **Нет висячих ссылок.** Ссылка не может пережить данные, на которые указывает:

```rust
fn dangling() -> &i32 {
    let x = 5;
    &x          // ОШИБКА: x умрёт в конце функции, ссылка стала бы висячей
}               // в C++ это undefined behavior; в Rust — ошибка компиляции
```

Поэтому когда компилятор «ругается» — он показывает реальный баг, который в C/C++
тихо выстрелил бы в рантайме. Читай ошибку borrow checker как подсказку, а не как
препятствие. В этом модуле ты на практике увидишь обе стороны: `&[i64]` для чтения и
`&mut Vec<i64>` для записи, а в `max_ref` — как ссылку можно безопасно ВЕРНУТЬ
наружу, и компилятор сам свяжет её время жизни со входным срезом.

## Идея 5. NLL: заимствование заканчивается у последнего использования

Это один из главных источников путаницы у новичков, поэтому разберём подробно.

До Rust 2018 borrow checker рассуждал так: «ссылка жива до закрывающей фигурной
скобки». В современном Rust действует **NLL (Non-Lexical Lifetimes)** — «ссылка
живёт до своего последнего использования», и не дольше. Это делает многие ситуации,
которые раньше не компилировались, корректными:

```rust
let mut v = vec![1, 2, 3];

let first = &v[0];          // разделяемое заимствование начинается здесь
println!("{}", first);      // last use of `first` — заимствование ЗАКАНЧИВАЕТСЯ здесь

v.push(4);                  // теперь OK: first уже не «жив»
```

Без NLL компилятор видел бы конфликт до конца блока. С NLL он понимает, что `first`
больше не используется после `println!` и отпускает заимствование.

**Типичная ловушка:** кажется, что ссылка мертва, но её ещё используют — и компилятор
снова «ругается»:

```rust
let mut map = std::collections::HashMap::new();
map.insert("a", 1);

let val = map.get("a");     // &i32 — разделяемое заимствование
map.insert("b", 2);         // ОШИБКА: val ещё не использован последний раз
println!("{:?}", val);      // если убрать эту строку — ошибки нет
```

Правило большого пальца: **ищи, где ты в последний раз используешь ссылку** — именно
там она «умирает». Если перенести `println!` до `insert` — всё скомпилируется.

## Идея 6. Вывод времён жизни и когда нужно писать `'a` явно

Rust умеет сам выводить времена жизни по трём простым правилам (lifetime elision rules).
Тебе важно знать правило №3, потому что именно оно объясняет `max_ref`.

**Правило 1.** Каждый параметр-ссылка получает своё отдельное время жизни:
```rust
fn foo(x: &i32, y: &i32) // компилятор читает как: fn foo<'a, 'b>(x: &'a i32, y: &'b i32)
```

**Правило 2.** Если входной параметр ровно один — время жизни результата берётся от него:
```rust
fn first_char(s: &str) -> &str  // компилятор читает как: fn first_char<'a>(s: &'a str) -> &'a str
```

**Правило 3 (метод-правило).** Если один из входных параметров — `&self` или
`&mut self`, время жизни результата берётся от него.

Когда правила не справляются — например, два входных параметра-ссылки и
возвращаемая ссылка — компилятор **требует явных аннотаций**, потому что не знает,
к которому из входов привязан результат:

```rust
// ОШИБКА: два входа-ссылки, один выход — компилятор не знает, чьё время жизни у результата
fn longer(a: &str, b: &str) -> &str {
    if a.len() > b.len() { a } else { b }
}

// OK: явно говорим, что результат живёт столько же, сколько оба входа
fn longer<'a>(a: &'a str, b: &'a str) -> &'a str {
    if a.len() > b.len() { a } else { b }
}
```

Одно `'a` — это «оба входа живут хотя бы столько, сколько результат». Компилятор
возьмёт минимум из двух реальных времён жизни. Если ты когда-нибудь видишь
`fn f<'a>(...) -> &'a ...` — значит именно это и происходит: явно сказали,
кому принадлежит возвращаемая ссылка.

## Идея 7. Deref coercion: Rust «распаковывает» ссылки автоматически

Rust умеет автоматически «понижать» ссылки к более обобщённым:

| Что у тебя     | Куда автоматически приводится |
|----------------|-------------------------------|
| `&String`      | `&str`                        |
| `&Vec<T>`      | `&[T]`                        |
| `&Box<T>`      | `&T`                          |

Это называется **deref coercion** (принудительное разыменование). Благодаря ему
функция, принимающая `&str`, одинаково хорошо работает и с `&String`, и с
строковым литералом:

```rust
fn greet(name: &str) { println!("Привет, {name}!"); }

let owned = String::from("Мир");
greet(&owned);          // &String → &str  (deref coercion)
greet("Вселенная");     // &str → &str     (уже OK)
```

А функция с параметром `&[i64]` принимает и `&Vec<i64>`, и `&[i64; N]`:

```rust
fn sum(v: &[i64]) -> i64 { v.iter().sum() }

let vec = vec![1i64, 2, 3];
let arr = [4i64, 5, 6];
sum(&vec);   // &Vec<i64> → &[i64]
sum(&arr);   // &[i64; 3] → &[i64]
```

Именно поэтому в этом модуле `sum_slice` принимает `&[i64]`, а не `&Vec<i64>` —
это более универсальная сигнатура.

**Как deref coercion работает под капотом?** Компилятор применяет `Deref`-трейт
цепочкой: `String` реализует `Deref<Target=str>`, `Vec<T>` реализует
`Deref<Target=[T]>`. При несовпадении типов компилятор ищет такую цепочку
`Deref` вызовов, которая приведёт к нужному типу. Это происходит только при
неявном принуждении (передача в функцию, присваивание) — при явном разыменовании
`*` надо указывать самому.

## Идея 8. Reborrowing: ссылка на ссылку

Иногда ты передаёшь `&mut T` в функцию, а потом снова используешь оригинал. Как
это работает, если `&mut` эксклюзивна?

```rust
fn add_one(x: &mut i32) { *x += 1; }

let mut n = 5;
let r = &mut n;
add_one(r);          // передаём r в функцию
add_one(r);          // и снова — r всё ещё жив!
println!("{n}");     // n == 7
```

Когда ты передаёшь `r` в `add_one`, Rust не *перемещает* `r` (ссылки не `Copy`,
но `&mut T` — особый случай): компилятор создаёт **reborrow** — временно
«суживает» доступ до времени вызова. После возврата из функции `r` снова
активен. Это напоминает стек ссылок: пока вызов идёт, `r` «заморожена» в пользу
функции, а по возврату — «размораживается».

Практически важно: `&mut &mut T` не то же самое, что `&mut T`. Reborrowing
позволяет не думать об этом различии в большинстве случаев, но когда ты явно
передаёшь `&mut ссылку` по значению — владение перемещается:

```rust
fn consume(x: &mut i32) { *x += 1; }

let mut n = 5;
let r: &mut i32 = &mut n;
consume(r);   // reborrow: r по-прежнему жив после вызова
let _moved = r; // r всё ещё доступен
```

## Идея 9. Срезы: паника при индексировании vs. безопасный `.get()`

Срез `&[T]` — это «вид» на последовательную область памяти: указатель + длина.
При обращении по индексу `v[i]` Rust проверяет границы в рантайме и паникует,
если `i >= len`. Это лучше, чем undefined behavior в C, но всё равно «падение»
программы. Безопаснее `.get(i)`, которое возвращает `Option<&T>`:

```rust
let v = vec![10, 20, 30];

// Паника при выходе за границы:
// let x = v[5]; // thread 'main' panicked: index out of bounds: 5 >= 3

// Безопасная альтернатива:
match v.get(5) {
    Some(x) => println!("есть: {x}"),
    None    => println!("нет такого индекса"),
}
```

Аналогично для срезов: `&v[a..b]` паникует при неверных границах, а
`v.get(a..b)` возвращает `Option<&[T]>`. Когда писать `v[i]`? Когда ты
*уверен*, что индекс корректен (например, итерируешь `for i in 0..v.len()`).
Когда писать `.get(i)`? Когда индекс приходит снаружи или ты не знаешь длины —
любой нетривиальный вход.

## Идея 10. Внутренняя мутабельность: `Cell` и `RefCell` (обзор)

Иногда правило «один писатель» кажется слишком строгим: хочется иметь
разделяемую ссылку `&T`, но всё же что-то менять внутри. Для этого в std есть
**interior mutability** — мутабельность «изнутри», сквозь неизменяемую ссылку.

- **`Cell<T>`** (для `Copy`-типов): атомарная замена значения без ссылки на содержимое.
- **`RefCell<T>`**: проверки правила заимствования переносятся из времени компиляции
  в **рантайм** — если правило нарушено, программа паникует, а не не компилируется.

```rust
use std::cell::RefCell;

let shared = RefCell::new(vec![1, 2, 3]);
{
    let mut borrow = shared.borrow_mut(); // проверка в рантайме
    borrow.push(4);
}   // мутабельное заимствование заканчивается здесь
println!("{:?}", shared.borrow()); // [1, 2, 3, 4]
```

**Когда это нужно?** Например, когда у тебя несколько `Rc<RefCell<T>>` (несколько
«владельцев» одного изменяемого значения). Это компромисс: ты обходишь статическую
проверку, но рискуешь паникой. Используй `Cell`/`RefCell` только тогда, когда
статическая проверка действительно невозможна — иначе предпочти обычные ссылки.
Детали — в модуле про умные указатели.

### Подсказки к заданиям 1–4

<details><summary>1. sum_slice — как пройтись по срезу</summary>
Срез итерируется напрямую: <code>for x in v { ... }</code> даёт <code>x: &i64</code>
(ссылку на элемент). Заводи аккумулятор <code>let mut s = 0i64;</code> и прибавляй
<code>*x</code>. Для пустого среза цикл просто не выполнится — вернётся <code>0</code>.
А ещё подумай, почему тут идеально ложится метод-итератор <code>.sum()</code>
(но <code>i64</code> ему придётся подсказать типом).
</details>

<details><summary>2. push_n — добавить count раз</summary>
У <code>Vec</code> есть метод <code>push</code>, который берёт <code>&mut self</code> —
именно поэтому параметр у тебя <code>&mut Vec&lt;i64&gt;</code>. Повтори
<code>v.push(value)</code> в цикле <code>count</code> раз
(например <code>for _ in 0..count { ... }</code>). При <code>count == 0</code> цикл
не выполнится — вектор не изменится, как и требуется.
</details>

<details><summary>3. double_all — менять элементы НА МЕСТЕ</summary>
Ключевой момент — получить <code>&mut</code> на каждый элемент, а не его копию.
Итерируйся так: <code>for x in v.iter_mut() { *x *= 2; }</code>. Здесь
<code>x: &mut i64</code>, и <code>*x *= 2</code> пишет прямо в память вектора.
Если случайно напишешь <code>for x in v.clone()</code> или будешь менять копии —
изменения «уйдут в никуда». Длина при этом не трогается: мы не пушим, а правим.
</details>

<details><summary>4. max_ref — вернуть ссылку или None</summary>
Сначала пустой случай: если <code>v.is_empty()</code> — верни <code>None</code>.
Иначе нужна ссылка на максимум. Ручной путь: заведи <code>let mut best = &v[0];</code>,
пройди <code>for x in v</code> и если <code>x &gt; best</code> — обнови
<code>best = x;</code>, в конце верни <code>Some(best)</code>. Заметь: возвращаешь
ссылку внутрь <code>v</code>, и компилятор сам выведет, что её время жизни — это
время жизни среза. Есть и короткий путь через итераторы: метод <code>.iter()</code>
плюс <code>.max()</code> вернёт <code>Option&lt;&amp;i64&gt;</code> сразу — сверься,
что тип совпал.
</details>

## Задание 5: подсчёт слов с явными временами жизни

Это основное «настоящее» задание модуля. Здесь правила заимствования перестают
быть абстракцией и становятся конкретным инструментом работы.

Реализуй две функции:

```rust
pub fn count_words<'a>(
    text: &'a str,
    counts: &mut std::collections::HashMap<&'a str, usize>,
)

pub fn most_frequent<'a>(
    counts: &'a std::collections::HashMap<&'a str, usize>,
) -> Option<(&'a str, usize)>
```

**Что делают эти функции?**

`count_words` разбивает `text` на слова (по пробелам) и записывает их частоты в
`counts`. Ключи в `HashMap` — не копии строк, а `&str`-срезы **внутрь самого
`text`**. Поэтому у `text` и ключей `HashMap` одно время жизни `'a`: карта не
может жить дольше источника.

`most_frequent` находит слово с максимальной частотой и возвращает `Option` с
парой `(&str, usize)`. Возвращаемая ссылка на слово — срез внутрь `HashMap`,
поэтому её время жизни `'a` совпадает с временем жизни таблицы.

**Почему это нетривиально?** Ты впервые явно пишешь `'a` и говоришь компилятору:
«ключи карты и текст — это одна и та же область памяти, и карта не должна
переживать источник». Без явного `'a` компилятор не сможет связать ключи
`HashMap` с `text` и откажется компилировать.

**Демонстрирует** одновременно: `text` заимствован разделяемо (`&'a str`);
`counts` заимствован эксклюзивно (`&mut HashMap`); они разные объекты — поэтому
оба заимствования существуют одновременно. Это легальная ситуация: `&` и `&mut`
на разные значения — не конфликт.

### Подсказки к заданию 5

<details><summary>Подсказка 1 — разбиение текста на слова</summary>
Метод <code>str::split_whitespace()</code> возвращает итератор по словам, автоматически
пропуская любые пробельные символы (пробел, табуляция, перевод строки). Каждое
слово — это <code>&str</code>, срез внутрь исходного <code>text</code>. Именно
поэтому параметр <code>text: &'a str</code>, а ключи в таблице — <code>&'a str</code>:
они «одалживают» память у <code>text</code>.
</details>

<details><summary>Подсказка 2 — как обновить счётчик в HashMap</summary>
Посмотри на метод <code>HashMap::entry(key).or_insert(default)</code>. Он возвращает
<code>&mut usize</code> — мутабельную ссылку на счётчик (создавая запись с нулём,
если её не было). После этого <code>*count += 1</code> увеличивает счётчик на месте.
В одну строку: <code>*counts.entry(word).or_insert(0) += 1;</code>
</details>

<details><summary>Подсказка 3 — most_frequent: как найти максимум в HashMap</summary>
Метод <code>HashMap::iter()</code> даёт итератор по <code>(&K, &V)</code>-парам.
Для поиска максимума по значению используй <code>.max_by_key(|(_, &count)| count)</code>.
Он вернёт <code>Option&lt;(&amp;&amp;str, &amp;usize)&gt;</code> — двойная ссылка,
потому что итератор сам выдаёт ссылки. Чтобы «распаковать» их в нужный тип,
используй <code>.map(|(&amp;word, &amp;count)| (word, count))</code>: паттерн
<code>&amp;word</code> разыменовывает ссылку прямо в паттерне.
</details>

<details><summary>Подсказка 4 — явные аннотации: зачем писать 'a?</summary>
Попробуй убрать все <code>'a</code> из сигнатуры <code>count_words</code> и
посмотри, что скажет компилятор. Он не сможет доказать, что ключи таблицы не
переживут <code>text</code>, и откажется принять <code>&amp;str</code> из
<code>text.split_whitespace()</code> как ключ. Явное <code>'a</code> говорит:
«я обещаю, что ключи живут не дольше, чем <code>text</code>». Это контракт,
который ты устанавливаешь, и компилятор его проверит.
</details>
