# Модуль 06 — Перечисления, Option, сопоставление

Большая идея: в Rust **перечисление (`enum`)** — это не просто список констант, как в C.
Это полноценный тип «**одно из нескольких**», где каждый вариант может нести **свои
данные**. А компилятор через `match` заставляет тебя обработать **все** варианты — ни
один забыть нельзя. Из этой пары — `enum` + `match` — вырастает то, чем Rust по-настоящему
гордится: `Option<T>` вместо `null`. Ошибки «забыл проверить на пустоту» в Rust просто не
компилируются.

## Идея 1. `enum` может нести данные в вариантах

Вариант перечисления — это как маленький конструктор. Он может быть пустым, а может
хранить значения:

```rust
enum Shape {
    Circle(f64),       // вариант с одним полем — радиус
    Rect(f64, f64),    // вариант с двумя полями — ширина и высота
}

let c = Shape::Circle(2.0);
let r = Shape::Rect(3.0, 4.0);
```

ПОЧЕМУ это важно: `Shape` — это **либо** круг, **либо** прямоугольник, и третьего не
дано. У круга есть радиус, у прямоугольника — стороны, и они не перепутаются. В C++ ты бы
делал это через `union` + тег (и сам следил, чтобы тег и данные совпадали) или через
иерархию классов. Здесь язык даёт это из коробки и безопасно.

### Вариант-структура: именованные поля

Помимо «кортежных» вариантов (`Circle(f64)`) у `enum` бывают **варианты-структуры** с
именованными полями — запись `Shape::Rect { w, h }` вместо позиционного `Shape::Rect(w, h)`:

```rust
enum Shape {
    Circle { radius: f64 },
    Rect   { w: f64, h: f64 },
}

let r = Shape::Rect { w: 3.0, h: 4.0 };

// В match именованные поля тоже именованы:
fn area(s: &Shape) -> f64 {
    match s {
        Shape::Circle { radius: r } => std::f64::consts::PI * r * r,
        Shape::Rect   { w, h }     => w * h,  // краткая форма: w => s.w
    }
}
```

Именованные поля удобны, когда вариант несёт много данных и позиция не очевидна. В задании
используется кортежный стиль (проще для начала), но знать обе формы важно.

## Идея 2. `Option<T>` — `Some`/`None` вместо `null`. ПОЧЕМУ это безопаснее

`Option<T>` — это enum из стандартной библиотеки. Он буквально определён так:

```rust
enum Option<T> {
    Some(T),   // значение есть, вот оно
    None,      // значения нет
}
```

Это замена `null`. И вот **в чём принципиальная разница**. Тони Хоар, придумавший
`null`, назвал его «ошибкой на миллиард долларов»: в языках с `null` *любой* указатель
может внезапно оказаться пустым, и компилятор тебя об этом не предупредит — ты узнаёшь в
рантайме, падением.

В Rust значения, которого «может не быть», имеют **другой тип** — не `T`, а `Option<T>`.
Поэтому:

```rust
let x: Option<i64> = Some(5);
// let y = x + 1;            // ❌ не скомпилируется: Option<i64> — это не i64!
let y = x.unwrap() + 1;      // чтобы добраться до числа, надо ЯВНО достать его
```

Компилятор физически не даст тебе использовать «возможно отсутствующее» значение, не
обработав случай `None`. Забыть проверку невозможно — её отсутствие это ошибка
компиляции, а не падение у пользователя.

### `Result<T, E>` — родной брат `Option`

`Option<T>` отвечает на вопрос «есть значение или нет». А `Result<T, E>` — на вопрос
«получилось или случилась ошибка»:

```rust
enum Result<T, E> {
    Ok(T),   // успех: вот результат
    Err(E),  // ошибка: вот её описание
}
```

Механика ровно та же: `match`, `if let`, комбинаторы. Разница — в семантике: `None` — это
«просто нет», `Err` — это «что-то пошло не так, и вот почему». В реальном Rust-коде
`Result` встречается везде, где может быть IO, парсинг, сетевые операции. `Option` — там,
где «не найдено» — это нормальная ситуация, а не ошибка.

```rust
fn parse_port(s: &str) -> Result<u16, String> {
    s.parse::<u16>().map_err(|e| e.to_string())
}

match parse_port("8080") {
    Ok(port)  => println!("слушаем порт {port}"),
    Err(why)  => println!("неверный порт: {why}"),
}
```

### Комбинаторы `Option`: не всегда нужен `match`

Полный `match` — мощно, но многословно. У `Option<T>` есть удобные методы-комбинаторы:

```rust
let x: Option<i64> = Some(3);

// map: преобразовать значение внутри, если оно есть
let doubled = x.map(|n| n * 2);   // Some(6)
let none: Option<i64> = None;
let still_none = none.map(|n| n * 2); // None — map ничего не делает с None

// and_then: «flatMap» — функция сама возвращает Option
fn safe_sqrt(n: f64) -> Option<f64> {
    if n < 0.0 { None } else { Some(n.sqrt()) }
}
let result = Some(4.0_f64).and_then(safe_sqrt); // Some(2.0)
let bad    = Some(-1.0_f64).and_then(safe_sqrt); // None

// unwrap_or: значение по умолчанию, если None
let val = none.unwrap_or(0);              // 0
let val2 = none.unwrap_or_else(|| 42);   // 42 (ленивое вычисление)
```

**Когда использовать что:**
- `map` — когда надо преобразовать значение внутри `Option`, не выходя наружу;
- `and_then` — когда преобразование само может не дать результат (цепочка «может провалиться»);
- `unwrap_or` / `unwrap_or_else` — когда `None` нужно заменить значением по умолчанию;
- полный `match` — когда логика веток принципиально разная.

### `unwrap()` и когда он опасен

`x.unwrap()` — это «я уверен, что там `Some`, дай мне значение». Если окажется `None` —
программа паникует (аварийно завершается с сообщением об ошибке). В продакшн-коде
`unwrap()` — почти всегда запах: ты перекладываешь ответственность с компилятора на рантайм.

Безопасные альтернативы:

```rust
// expect — то же, что unwrap, но с объяснением:
// паника будет содержать твой текст, что упрощает отладку
let n = x.expect("ожидали число, но получили None");

// ? — оператор «пробрось ошибку выше»; работает в функциях,
// возвращающих Option или Result:
fn double_first(v: &[i64]) -> Option<i64> {
    let first = v.first()?;   // если None — сразу вернуть None из функции
    Some(first * 2)
}
```

Оператор `?` — это синтаксический сахар для «если `None` / `Err` — вернись из функции с
этим же `None` / `Err`». Он делает цепочки проверок читаемыми, не превращая код в пирамиду
`match`.

## Идея 3. `match` по вариантам с извлечением данных; `if let`

`match` — это «супер-switch»: он не просто сравнивает, а **разбирает** значение на части
и достаёт данные из вариантов:

```rust
fn area(s: &Shape) -> f64 {
    match s {
        Shape::Circle(r) => std::f64::consts::PI * r * r,  // r — это радиус из Circle
        Shape::Rect(w, h) => w * h,                        // w, h — стороны из Rect
    }
}
```

Слева от `=>` — **образец** (pattern): он одновременно проверяет, какой это вариант, и
**связывает** внутренние данные с именами (`r`, `w`, `h`). Это и есть «сопоставление с
образцом» (pattern matching).

Когда тебя интересует **только один** вариант, полный `match` многословен. Для этого есть
`if let` — «match на один случай»:

```rust
if let Some(n) = some_option {
    println!("есть число: {n}");   // выполнится, только если внутри Some
}
```

ПОЧЕМУ две формы: `match` — когда нужно обработать все варианты; `if let` — когда нужен
ровно один, а на остальные тебе всё равно. Выбирай по ситуации.

### Продвинутые возможности `match`

**Защитные условия (match guards)** — после образца можно добавить `if условие`:

```rust
match n {
    x if x < 0  => println!("отрицательное: {x}"),
    x if x == 0 => println!("ноль"),
    x           => println!("положительное: {x}"),
}
```

Если условие не выполняется — ветка не срабатывает и `match` проверяет следующую. Важно:
guard не учитывается при проверке исчерпывающности — если последняя ветка имеет guard,
компилятор всё равно потребует `_ =>` или безусловную ветку.

**Альтернативы через `|`** — один образец, несколько вариантов:

```rust
match c {
    'a' | 'e' | 'i' | 'o' | 'u' => println!("гласная"),
    'a'..='z' | 'A'..='Z'       => println!("согласная"),
    _                            => println!("не буква"),
}
```

**Привязка с `@`** — одновременно проверить и сохранить значение:

```rust
match n {
    x @ 1..=10 => println!("в диапазоне [1,10]: {x}"),
    _          => println!("вне диапазона"),
}
// без @ пришлось бы писать отдельный match или if let, чтобы иметь имя x
```

**Игнорирование полей через `..`** — когда тебя интересует только часть данных:

```rust
struct Point { x: f64, y: f64, z: f64 }

let p = Point { x: 1.0, y: 2.0, z: 3.0 };
let Point { x, .. } = p;   // берём только x, остальное игнорируем
println!("x = {x}");
```

В `enum`:
```rust
enum Msg { Move { x: i32, y: i32 }, Write(String), ChangeColor(i32, i32, i32) }

match msg {
    Msg::ChangeColor(r, ..) => println!("красная компонента: {r}"),
    _                       => {}
}
```

### `while let` — цикл до первого `None`

`while let` — это «продолжай цикл, пока образец совпадает»:

```rust
let mut stack = vec![1, 2, 3];

while let Some(top) = stack.pop() {
    println!("{top}");   // напечатает 3, 2, 1 — пока vec не опустеет
}
// pop() возвращает Option<T>; как только стек пуст — None — цикл завершается
```

Удобная альтернатива циклу `loop { if let ... break }`. Как и `if let` — для «одного
интересующего варианта»; просто теперь это цикл, а не условие.

## Идея 4. Исчерпывающность защищает от забытых случаев

Это, пожалуй, главная фишка `match`. Он **обязан** покрыть все варианты:

```rust
match s {
    Shape::Circle(r) => /* ... */,
    // забыли Shape::Rect — ❌ ОШИБКА КОМПИЛЯЦИИ:
    // "pattern `Rect(_, _)` not covered"
}
```

ПОЧЕМУ это подарок, а не придирка: представь, что через полгода ты добавишь в `Shape`
новый вариант `Triangle`. Компилятор тут же подсветит **каждый** `match`, где этот случай
не обработан, — и не даст собраться, пока ты их не починишь. В языке без исчерпывающности
ты бы искал такие места глазами и неизбежно что-то пропустил.

> Подвох: ветка `_ => ...` ловит «все остальные случаи». Иногда она удобна, но она же
> **отключает** проверку исчерпывающности: добавишь новый вариант — `_` молча его
> проглотит, и компилятор уже не напомнит. Поэтому `_` используй осознанно, а не чтобы
> «заткнуть» компилятор.

## Задание 5 (повышенное) — вычислитель выражений

Это задание учит трём вещам сразу: **рекурсивному `enum`**, **рекурсивному `match`** и
**распространению `Option` через рекурсию**. Каждая часть важна по-своему.

### Рекурсивный `enum` и `Box<T>`

Попробуем объявить «выражение»:

```rust
enum Expr {
    Num(f64),
    Add(Expr, Expr),  // ❌ не скомпилируется!
}
```

Компилятор откажет: «recursive type `Expr` has infinite size». Почему? Чтобы выделить
память под `Expr`, нужно знать её размер; `Add(Expr, Expr)` содержит два `Expr`, каждый из
которых может содержать ещё два — и так до бесконечности. Размер не определить статически.

Решение — завернуть во **`Box<T>`**. `Box<T>` — это указатель на данные в куче: сам
указатель всегда имеет фиксированный размер (8 байт на 64-битной платформе), независимо от
того, что в куче:

```rust
enum Expr {
    Num(f64),
    Add(Box<Expr>, Box<Expr>),  // ✅ размер Expr теперь фиксирован
    // ...
}

// Конструируем вложенное выражение:
let e = Expr::Add(
    Box::new(Expr::Num(1.0)),
    Box::new(Expr::Num(2.0)),
);
// Это дерево: Add(1.0, 2.0)
```

`Box::new(x)` перемещает `x` в кучу и возвращает умный указатель. В `match` `Box<Expr>`
разыменовывается автоматически — образец `Expr::Add(l, r)` даст тебе `&Box<Expr>` (или
`Box<Expr>` при потребляющем match), но с ним работают те же паттерны, что и с `Expr`.

### Рекурсивный `match` и Option-пропаганда

Функция `eval(e: &Expr) -> Option<f64>` вычисляет выражение, но возвращает `None`, если
**где-либо** в дереве встретилось деление на ноль — даже на глубине десяти уровней.

Для `Num` всё просто: `Expr::Num(n) => Some(*n)`.

Для `Add(l, r)` нужно вычислить оба поддерева и сложить, **если оба вернули `Some`**.
Здесь пригодится `?`:

```rust
// псевдокод — не смотри на детали, подумай о структуре
Expr::Add(l, r) => Some(eval(l)? + eval(r)?),
```

`eval(l)?` означает: «вычисли левую часть; если `None` — сразу верни `None` из `eval`».
Таким образом, одно деление на ноль в любом месте дерева «пузырится» наверх, не требуя
ручных проверок на каждом уровне.

### Что реализовать

В `src/lib.rs` уже объявлено:
- `pub enum Expr` — пять вариантов: `Num`, `Add`, `Sub`, `Mul`, `Div`;
- `pub fn eval(e: &Expr) -> Option<f64>` — вычисляет выражение или возвращает `None`.

Твои задачи:
- реализовать `eval` через `match`;
- правильно обработать `Div`: при нулевом правом аргументе вернуть `None`, иначе —
  результат деления;
- для всех остальных операций рекурсивно вычислить поддеревья и объединить результаты.
