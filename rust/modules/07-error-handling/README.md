# Модуль 07 — Обработка ошибок: Result и ?

Большая идея: в Rust ошибка — это **обычное значение**, а не исключение, которое
незримо летит мимо тебя к какому-то `catch`. Функция, которая может не справиться,
**обязана** объявить это в своём типе возврата — `Result<T, E>`. Тогда компилятор не
даст тебе забыть про случай неудачи: ты физически не достанешь значение, не разобрав
сначала, не пришла ли вместо него ошибка. «Скрытых путей выхода» нет.

В этом модуле мы научимся возвращать ошибки значениями, красиво пробрасывать их вверх
оператором `?`, создавать собственные типы ошибок и понимать, когда `unwrap`/`expect` —
это нормально, а когда — мина.

## Идея 1. `Result<T, E>` — Ok или Err

`Result<T, E>` — это перечисление (`enum`) с двумя вариантами:

```rust
enum Result<T, E> {
    Ok(T),   // всё хорошо, внутри — значение типа T
    Err(E),  // что-то пошло не так, внутри — описание ошибки типа E
}
```

Функция возвращает `Result`, когда неудача — **ожидаемая и восстановимая** ситуация:
файл может не открыться, строка может не разобраться в число, делитель может оказаться
нулём. Это не «баг», а нормальный ход событий, который вызывающий код должен обработать:

```rust
fn divide(a: i64, b: i64) -> Result<i64, String> {
    if b == 0 {
        Err("division by zero".to_string())  // вернули ошибку как значение
    } else {
        Ok(a / b)                            // вернули успех как значение
    }
}

match divide(10, 0) {
    Ok(n)  => println!("получилось {n}"),
    Err(e) => println!("ошибка: {e}"),       // компилятор ЗАСТАВИЛ обработать оба случая
}
```

**ПОЧЕМУ это лучше, чем `panic!`:** `panic!` рушит программу (или поток). Деление на
ноль по вине пользователя не должно класть весь сервер. `Result` отдаёт решение тому,
кто вызвал, — он сам выберет: показать сообщение, попробовать ещё раз или прокинуть
ошибку выше. `panic!` оставляют для ситуаций «так быть не может, это баг в моём коде»
(нарушение инварианта), а не для ожидаемого плохого ввода.

## Идея 2. Оператор `?` — ранний возврат ошибки

Когда ты вызываешь одну `Result`-функцию из другой, постоянно писать `match` утомительно:

```rust
fn sum_two(a: &str, b: &str) -> Result<i64, String> {
    let x = match parse_int(a) {
        Ok(v)  => v,
        Err(e) => return Err(e),   // вручную пробрасываем ошибку наверх
    };
    let y = match parse_int(b) {
        Ok(v)  => v,
        Err(e) => return Err(e),
    };
    Ok(x + y)
}
```

Оператор `?` делает ровно это, но в один символ:

```rust
fn sum_two(a: &str, b: &str) -> Result<i64, String> {
    let x = parse_int(a)?;   // если Err — СРАЗУ выходим из функции с этой ошибкой
    let y = parse_int(b)?;   // если Ok  — разворачиваем значение и идём дальше
    Ok(x + y)
}
```

`выражение?` читается так: «если внутри `Ok(v)` — дай мне `v`; если внутри `Err(e)` —
немедленно сделай `return Err(e)` из всей функции». Поэтому `?` можно ставить только
внутри функции, которая сама возвращает `Result` (есть куда возвращать ошибку).

**ПОЧЕМУ это удобно:** счастливый путь («всё хорошо») остаётся плоским и читается
сверху вниз, а вся обработка ошибок схлопывается в маленький `?`. При этом — в отличие
от исключений — по `?` ВИДНО, где функция может выйти с ошибкой. Ничего не прячется.

### Как `?` работает под капотом: трейт `From`

Вот тонкость, которую многие упускают. `?` делает не просто `return Err(e)` — он делает
`return Err(From::from(e))`. То есть перед возвратом **конвертирует** тип ошибки через
трейт `From`. Это позволяет использовать `?` там, где тип ошибки в вызываемой функции
и тип ошибки в вызывающей **не совпадают**, — достаточно реализовать конверсию:

```rust
// Стандартная библиотека: ParseIntError -> наш MyError
impl From<std::num::ParseIntError> for MyError {
    fn from(e: std::num::ParseIntError) -> MyError {
        MyError::ParseFailed(e.to_string())
    }
}

fn parse_and_double(s: &str) -> Result<i64, MyError> {
    // s.parse::<i64>() возвращает Result<i64, ParseIntError>
    // ? вызовет From::from(parse_error) -> MyError автоматически
    let n: i64 = s.parse()?;
    Ok(n * 2)
}
```

Без `From` компилятор отклонил бы `?` с ошибкой «типы ошибок не совпадают». С `From` —
конверсия происходит молча и в нужном месте.

## Идея 3. `unwrap` / `expect` — когда можно, чем опасны

`Result` можно «вскрыть» напролом:

```rust
let n = parse_int("42").unwrap();              // вернёт 42
let n = parse_int("oops").unwrap();            // PANIC! программа падает
let n = parse_int("oops").expect("нужно число"); // PANIC с твоим текстом
```

`unwrap()` говорит: «я уверен, что тут `Ok`; если нет — пусть всё рухнет». `expect("…")`
— то же, но с поясняющим сообщением (поэтому почти всегда предпочтительнее `unwrap`).

**Когда это допустимо:**
- в тестах и быстрых прототипах — падение там безвредно и заметно;
- когда ты *логически* доказал, что `Err` невозможен (например, парсишь строковую
  константу из своего же кода).

**Чем опасно:** `unwrap` на данных извне (ввод пользователя, файл, сеть) — это
отложенный `panic!`. Стоит прийти неожиданному значению — и программа падает в проде.
Правило: на границе с внешним миром обрабатывай ошибку (`match`/`?`/`unwrap_or`…), а
`unwrap`/`expect` приберегай для «этого не может случиться».

## Идея 4. ПОЧЕМУ ошибки-в-типах лучше исключений

В языках с исключениями функция `int parse(String)` по сигнатуре выглядит безобидно —
а на деле может «выстрелить» исключением, о котором ты узнаешь только в рантайме (или из
документации, если повезёт). Путь ошибки невидим.

В Rust иначе:

- **Возможность ошибки видна в типе.** `-> Result<i64, String>` прямо говорит: эта
  функция может не справиться. Это часть контракта, а не сюрприз.
- **Игнорировать ошибку нельзя по-тихому.** Чтобы достать значение, ты обязан разобрать
  `Result` (`match`, `?`, `unwrap`…). Компилятор предупредит о неиспользованном
  `Result` (`#[must_use]`).
- **Поток управления честный.** Ошибка возвращается обычным `return` (через `?`), а не
  «телепортируется» через десять кадров стека. Читая код, ты видишь все точки выхода.

Итог: меньше «невидимых» путей, больше гарантий от компилятора. Ошибки становятся
такой же частью дизайна, как и нормальные данные. Это и есть «сделать недопустимые
состояния невыразимыми».

## Идея 5. Комбинаторы Result и Option↔Result

Вместо `match` на каждый `Result` стандартная библиотека даёт богатый набор методов:

```rust
// map: преобразовать Ok-значение, не трогая Err
let doubled = parse_int("5").map(|n| n * 2);   // Ok(10)

// and_then: цепочка операций, каждая из которых может вернуть Err
let result = parse_int("5")
    .and_then(|n| divide(100, n));              // Ok(20)

// unwrap_or: значение по умолчанию при ошибке
let n = parse_int("bad").unwrap_or(0);          // 0

// unwrap_or_else: ленивое значение по умолчанию (вычисляется только при Err)
let n = parse_int("bad").unwrap_or_else(|_| 42); // 42

// map_err: преобразовать Err-значение, не трогая Ok
let r = "abc".parse::<i64>()
    .map_err(|e| format!("плохой ввод: {e}"));  // Err("плохой ввод: ...")
```

**Option и Result дружат:** иногда нужно превратить одно в другое:

```rust
// Option -> Result: ok_or добавляет Err-значение на случай None
let opt: Option<i64> = Some(42);
let res: Result<i64, &str> = opt.ok_or("значение отсутствует"); // Ok(42)

let nothing: Option<i64> = None;
let err: Result<i64, &str> = nothing.ok_or("значение отсутствует"); // Err("значение отсутствует")

// Result -> Option: ok() выбрасывает Err, оставляет только Ok
let maybe: Option<i64> = parse_int("7").ok(); // Some(7)
let gone:  Option<i64> = parse_int("x").ok(); // None

// transpose: Option<Result<T,E>> <-> Result<Option<T>,E>
let or: Option<Result<i64, _>> = Some("5".parse::<i64>());
let ro: Result<Option<i64>, _> = or.transpose(); // Ok(Some(5))
```

Правило: `ok_or` и `ok_or_else` — твои главные мосты между `Option` и `Result`. Помни
их наизусть.

## Идея 6. Проблема нескольких типов ошибок

Реальные функции часто вызывают несколько других функций, каждая из которых возвращает
**свой** тип ошибки. Как объединить их в один `Result`?

### Способ 1: `Box<dyn std::error::Error>`

Быстрый выход — принять любую ошибку через трейт-объект:

```rust
use std::error::Error;

fn read_number(path: &str) -> Result<i64, Box<dyn Error>> {
    let text = std::fs::read_to_string(path)?;  // io::Error -> Box<dyn Error>
    let n: i64 = text.trim().parse()?;           // ParseIntError -> Box<dyn Error>
    Ok(n)
}
```

Оба `?` конвертируют разные типы ошибок в `Box<dyn Error>` автоматически (стандартная
библиотека уже реализует нужные `From`). Плюс — очень быстро писать. Минус — вызывающий
код не знает *конкретный* тип ошибки: сложнее обрабатывать разные случаи по-разному.
Подходит для прототипов, скриптов, `main`.

### Способ 2: объединяющий enum ошибок

Для библиотечного кода обычно делают свой enum, где каждый вариант — отдельный вид
ошибки. Это даёт вызывающему коду возможность `match`-нуть и реагировать точечно:

```rust
#[derive(Debug)]
enum AppError {
    Io(std::io::Error),
    Parse(std::num::ParseIntError),
}

impl From<std::io::Error> for AppError {
    fn from(e: std::io::Error) -> Self { AppError::Io(e) }
}
impl From<std::num::ParseIntError> for AppError {
    fn from(e: std::num::ParseIntError) -> Self { AppError::Parse(e) }
}

fn read_number(path: &str) -> Result<i64, AppError> {
    let text = std::fs::read_to_string(path)?;  // From<io::Error> — сработает
    let n: i64 = text.trim().parse()?;           // From<ParseIntError> — сработает
    Ok(n)
}
```

Чуть больше бойлерплейта, зато полный контроль над типами.

## Идея 7. Трейт `std::error::Error` и цепочка причин

Чтобы твой тип ошибки «вписался» в экосистему Rust, он должен реализовывать три трейта:
`Debug`, `Display` и `std::error::Error`. Первые два описывают, как печатать ошибку;
`Error` добавляет метод `source()` — ссылку на ошибку-причину (та, что «обернула» эту):

```rust
use std::fmt;
use std::error::Error;

#[derive(Debug)]
enum MyError {
    ParseFailed(String),
    OutOfRange { value: i64, max: i64 },
}

impl fmt::Display for MyError {
    fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result {
        match self {
            MyError::ParseFailed(s) => write!(f, "не удалось разобрать: '{s}'"),
            MyError::OutOfRange { value, max } =>
                write!(f, "значение {value} превышает максимум {max}"),
        }
    }
}

impl Error for MyError {
    // source() по умолчанию возвращает None — переопределяй, если ты обернул
    // другую ошибку и хочешь, чтобы её можно было «размотать» цепочкой
    fn source(&self) -> Option<&(dyn Error + 'static)> {
        None
    }
}
```

`source()` важен для инструментов вроде `anyhow`, которые умеют печатать всю цепочку
причин: «ошибка A: вызвана B: вызвана C». Не игнорируй исходную ошибку при оборачивании.

## Идея 8. `thiserror` и `anyhow` — когда std не хватает

Стандартная библиотека даёт все инструменты, но бойлерплейт для `Display` + `Error` +
нескольких `From` быстро надоедает. В экосистеме Rust сложились два популярных крейта:

- **`thiserror`** — макрос `#[derive(Error)]` для твоего enum: генерирует `Display`,
  `Error` и `From` из аннотаций. Идеальна для библиотечного кода, где типы ошибок —
  часть публичного API.

  ```rust
  use thiserror::Error;

  #[derive(Debug, Error)]
  enum MyError {
      #[error("не удалось разобрать: '{0}'")]
      ParseFailed(String),
      #[error("значение {value} превышает максимум {max}")]
      OutOfRange { value: i64, max: i64 },
  }
  ```

- **`anyhow`** — тип `anyhow::Error`, который принимает любую ошибку через `?`, умеет
  хранить контекст (`context("...")`) и красиво печатать всю цепочку причин. Идеален
  для приложений (бинарников), где тебе важно быстро написать `-> anyhow::Result<T>`,
  а конкретный тип ошибки вызывающему коду знать не нужно.

**Когда что выбирать:**
- библиотека → `thiserror` (точные типы в публичном API);
- приложение / прототип → `anyhow` (скорость, контекст, простота);
- учёба / без зависимостей → std-only: `Box<dyn Error>` или ручной enum.

В этом модуле мы работаем в режиме **std-only**, чтобы понять, как всё устроено под
капотом. Освоишь основу — крейты станут для тебя понятным синтаксическим сахаром,
а не магией.

### Подсказки к базовым заданиям

<details><summary>parse_int: как превратить строку в i64</summary>

В стандартной библиотеке уже есть разбор: `s.parse::<i64>()` возвращает
`Result<i64, ParseIntError>`. Тебе нужно превратить его `Err` в свой `String`-вариант.
Удобно через `.map_err(...)`:

```rust
s.parse::<i64>().map_err(|_| format!("not a number: '{s}'"))
```

`map_err` оставляет `Ok` как есть, а у `Err` заменяет содержимое. Заметь: формат
сообщения в тестах — `not a number: '...'` с одинарными кавычками вокруг строки.
</details>

<details><summary>divide: порядок проверки</summary>

Сначала проверь делитель, только потом дели — иначе `a / 0` запаникует РАНЬШЕ, чем ты
успеешь вернуть `Err`. То есть `if b == 0 { return Err(...) }` (или `match`/`if/else`),
и лишь в ветке «не ноль» вычисляй `a / b`.
</details>

<details><summary>element_at: Option превращаем в Result</summary>

У среза есть безопасный `v.get(i)` — он возвращает `Option<&i64>` (`Some`, если индекс
в границах, иначе `None`). Тебе нужно: при `Some(x)` вернуть `Ok(*x)`, при `None` —
`Err` с индексом и длиной (`v.len()`). Можно `match`, можно
`v.get(i).copied().ok_or_else(|| format!("index {i} out of bounds for length {}", v.len()))`.
`copied()` превращает `Option<&i64>` в `Option<i64>` (для `Copy`-типов).
</details>

<details><summary>sum_parsed: тут и живёт оператор ?</summary>

Заведи аккумулятор `let mut total = 0;`, пройди по `parts` циклом и на каждой части
сделай `total += parse_int(part)?;`. Оператор `?` сам выйдет с `Err` на первой плохой
части — до остальных дело не дойдёт. В конце верни `Ok(total)`. Пустой срез просто не
войдёт в цикл — вернётся `Ok(0)`.
</details>

## Задание 5. Реальный CSV-парсер строки: `parse_csv_record`

Это «настоящая» задача модуля — собрать все идеи воедино: собственный тип ошибки-enum,
трейт `std::error::Error`, реализацию `Display`, `From` для автоматической конверсии
через `?`, и наконец функцию, которая использует всё это в деле.

**Что нужно реализовать:**

Тип ошибки `CsvError` — перечисление с двумя вариантами:
- `BadField { col: usize, reason: String }` — конкретная колонка не разобралась;
- `WrongColumnCount { expected: usize, got: usize }` — строка содержит не то количество
  полей.

Плюс функция `parse_csv_record(line: &str, expected_cols: usize) -> Result<Vec<i64>, CsvError>`:
- разбить строку по запятой;
- проверить количество полей;
- разобрать каждое поле в `i64`, используя `?`.

**Тесты:** `tests/csv.rs`.

### Подсказки к заданию 5

<details><summary>Шаг 1: как устроен CsvError</summary>

Начни с enum — два варианта с именованными полями. Дальше тебе понадобятся:
`#[derive(Debug)]`, реализация `Display` через `match self {...}`, реализация
`std::error::Error` (пустое тело — `source()` по умолчанию `None`), и
`impl From<std::num::ParseIntError> for CsvError`.

Зачем `From`? Чтобы `?` мог автоматически превратить `ParseIntError` (который
возвращает `"7".parse::<i64>()`) в твой `CsvError::BadField`.
</details>

<details><summary>Шаг 2: From<ParseIntError> — что писать в теле</summary>

`From::from` получает `ParseIntError`, но не знает номер колонки — номер знает только
тот, кто вызывает `parse`. Как это решить?

Подсказка: `?` можно заменить на `.map_err(|e| CsvError::BadField { col: i, reason: e.to_string() })?`
внутри цикла с индексом. Тогда `From` вообще не нужен — ошибку конвертируешь вручную
с нужным `col`. Это честнее: `From` не знает контекст (номер колонки), а ты — знаешь.
</details>

<details><summary>Шаг 3: логика parse_csv_record</summary>

1. Разбить: `line.split(',').collect::<Vec<_>>()`.
2. Проверить длину против `expected_cols` — если не совпадает, сразу `Err(CsvError::WrongColumnCount { ... })`.
3. Пройти по полям с индексом (`enumerate`), для каждого вызвать `.trim().parse::<i64>()`,
   и при ошибке вернуть `CsvError::BadField { col, reason }`.
4. Вернуть `Ok(vec_чисел)`.

Порядок важен: сначала проверь количество, потом парси — иначе ошибки перемешаются.
</details>
