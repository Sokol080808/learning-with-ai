# Модуль 15 — Конкурентность: потоки, каналы, Arc&lt;Mutex&gt;

Большая идея: до сих пор весь твой код шёл **одним потоком** — инструкции выполнялись
строго по очереди. Конкурентность — это когда несколько частей программы исполняются
«одновременно» (на разных ядрах — по-настоящему параллельно; на одном — чередуясь). Это
быстро, но опасно: два потока, меняющие одни данные без согласования, — это **гонка данных**
(data race), один из самых коварных классов багов: воспроизводится через раз, зависит от
тайминга, исчезает под отладчиком.

Rust называет свой подход **«бесстрашной конкурентностью»** (fearless concurrency). Соль в
том, что гонки данных в безопасном Rust **ловятся на этапе компиляции**, а не «дисциплиной
программиста». Тот же borrow-checker и владение, что в модулях 03–05 (один писатель **или**
много читателей), теперь работают и **между потоками** — через авто-трейты `Send` и `Sync`
(идея 10). Если код компилируется, гонок данных в нём нет. Это не отменяет логических ошибок
(дедлоки, зависший канал) — но целый класс багов закрыт компилятором.

Карта модуля — три способа, как потоки могут общаться, от простого к общему:

```text
thread::spawn ──► каналы (mpsc) ──► Arc<Mutex<T>>  ──► атомики
  «запусти          «передай          «дай всем         «для счётчика
   работу»           результат»        общие данные»     замок избыточен»
```

Финал модуля — `SharedTally`: ты соберёшь руками **ровно ту конструкцию**
`Arc<Mutex<HashMap>>`, на которой стоит `SharedDb` в капстоуне. Бенчмарк теории — глава 16
[The Rust Book](https://doc.rust-lang.org/book/ch16-00-concurrency.html).

## Идея 1. Что такое конкурентность и почему она «бесстрашная»

Конкурентность — это структура программы из независимо продвигающихся задач. Параллелизм
(они реально бегут одновременно на разных ядрах) — частный случай. Зачем это нужно:
загрузить все ядра процессора; не блокировать программу, пока одна часть ждёт диск/сеть.

ПОЧЕМУ это опасно без помощи языка: если два потока пишут в одну переменную без
синхронизации, итог зависит от того, кто успел первым, — **гонка данных**. Классический
пример: счётчик `x += 1` — это на самом деле три шага (прочитать, прибавить, записать). Два
потока могут прочитать одно и то же старое значение, оба прибавить 1 и оба записать — и
вместо `+2` получится `+1`. Один инкремент «потерялся».

ПОЧЕМУ в Rust это «бесстрашно»: язык не разрешит двум потокам одновременно держать
изменяемый доступ к одним данным. Правило владения «один `&mut` **или** много `&`» (модуль
04) распространяется на потоки через трейты `Send`/`Sync`. Тип, который небезопасно
шарить между потоками, просто **не пройдёт компиляцию** в `thread::spawn`. Ты не можешь
случайно написать гонку данных в безопасном Rust — это его главное обещание про
конкурентность.

> Важно различать: **гонка данных** (несинхронизированный доступ к памяти — Rust её
> запрещает) и **состояние гонки/race condition** в широком смысле (логика зависит от
> порядка событий — например дедлок или зависший канал). Второе компилятор не ловит; за
> логику отвечаешь ты (идея 11).

## Идея 2. `thread::spawn` и `JoinHandle::join`

`std::thread::spawn(f)` запускает **новый ОС-поток**, который исполняет замыкание `f`, и
сразу возвращает управление. Результат — `JoinHandle<T>`, ручка к потоку:

```rust
use std::thread;

let handle = thread::spawn(|| {
    // эта работа идёт в отдельном потоке
    42
});

let value: i32 = handle.join().unwrap(); // ждём поток, забираем его результат
assert_eq!(value, 42);
```

`join()` **блокирует** текущий поток, пока дочерний не завершится, и возвращает
`Result<T, _>`: `Ok(возврат_замыкания)` — если поток отработал нормально, `Err(...)` — если
он **запаниковал** (паника в потоке не роняет всю программу, а превращается в `Err`).

ПОЧЕМУ `join` обязателен: когда `main` завершается, программа **убивает все ещё живые
потоки**, не дожидаясь их. Без `join` вывод недетерминирован — поток может не успеть
выполниться вовсе. `join` — это «дождись и собери результат».

Когда потоков много, складываем хендлы в `Vec` и потом проходим по нему циклом:

```rust
let mut handles = Vec::new();
for i in 0..4 {
    handles.push(thread::spawn(move || i * i));
}
let results: Vec<i32> = handles.into_iter().map(|h| h.join().unwrap()).collect();
// results == [0, 1, 4, 9]
```

> Подвох: **порядок завершения потоков не определён**. `results` собирается в порядке
> хендлов (мы сами так написали), но кто из потоков физически закончил первым — не угадать.
> Никогда не полагайся на порядок исполнения потоков.

## Идея 3. `move`-замыкания и проблема разделения

Тело потока обязано быть `'static` — оно может пережить функцию, которая его породила.
Поэтому оно **не может заимствовать** локальные переменные: к моменту, когда поток до них
доберётся, они могли бы уже исчезнуть. Попробуй без `move`:

```rust
let data = vec![1, 2, 3];
let h = thread::spawn(|| {
    println!("{:?}", data); // ❌ ошибка: closure may outlive the current function,
                            //    but it borrows `data`, which is owned by the function
});
```

Компилятор прямо подсказывает: `help: to force the closure to take ownership of `data`,
use the `move` keyword`. Добавляем `move`:

```rust
let data = vec![1, 2, 3];
let h = thread::spawn(move || {
    println!("{:?}", data); // ✅ data ПЕРЕМЕЩЁН во владение потока
});
h.join().unwrap();
// здесь `data` использовать уже нельзя — владение ушло в поток
```

`move` забирает захваченные переменные **во владение** замыкания (мост к модулю 11: тело
потока — это `FnOnce`, оно исполняется один раз). Теперь поток владеет `data` сам, и неважно,
жива ли исходная функция.

ПОЧЕМУ это мотивирует следующие идеи: `move` отдаёт данные **ровно одному** потоку. А если
потокам нужно **передавать друг другу** значения — нужны каналы (идея 4). А если им нужно
**видеть одни и те же** данные — нужен `Arc` (идея 6).

## Идея 4. Каналы `std::sync::mpsc` — «общаться, передавая владение»

Канал — это однонаправленная труба для передачи значений между потоками. `mpsc` =
**multi-producer, single-consumer**: много отправителей, один получатель.

```rust
use std::sync::mpsc;
use std::thread;

let (tx, rx) = mpsc::channel();   // tx — Sender (передатчик), rx — Receiver (приёмник)

thread::spawn(move || {
    tx.send(10).unwrap();          // send ПЕРЕМЕЩАЕТ значение в канал
    tx.send(20).unwrap();
});

let a = rx.recv().unwrap();        // recv БЛОКИРУЕТ, пока не придёт значение
let b = rx.recv().unwrap();
assert_eq!(a + b, 30);
```

Ключевое: `tx.send(v)` **перемещает** `v` в канал — отправитель теряет владение. Поэтому
после отправки никто не может случайно изменить переданное значение: **владение ушло →
гонки данных нет**. Это и есть лозунг Книги: *«share memory by communicating»* (разделяй
память, общаясь) вместо *«communicate by sharing memory»*.

`recv()` **блокирует**, пока не появится значение (или все `tx` не сдропнутся). И `send`, и
`recv` возвращают `Result` — ошибка означает, что другой конец канала уже закрыт.

Receiver удобно перебирать циклом `for`:

```rust
for value in rx {           // итерируется, пока канал открыт
    println!("got {value}");
}                           // цикл завершается, когда ВСЕ Sender сдропнуты
```

ПОЧЕМУ цикл когда-то завершается: `for v in rx` крутится, пока жив хоть один `Sender`. Когда
сдропнут последний — труба «закрывается», `recv` отдаёт `Err`, и цикл выходит. Это подводит
нас к важнейшему подвоху следующей идеи.

## Идея 5. Несколько производителей — `tx.clone()` (и учебный дедлок)

«Multi-producer» означает: `Sender` можно **клонировать**, и каждый клон шлёт в один и тот
же канал. Раздаём по клону каждому потоку:

```rust
use std::sync::mpsc;
use std::thread;

let (tx, rx) = mpsc::channel();

for i in 0..3 {
    let tx = tx.clone();              // свой клон отправителя для каждого потока
    thread::spawn(move || {
        tx.send(i).unwrap();
    });                              // клон tx дропается здесь, на выходе из потока
}

drop(tx);                            // ⚠️ дропаем ИСХОДНЫЙ tx — иначе труба не закроется

let collected: Vec<i32> = rx.collect();  // соберёт все 3 значения и завершится
```

ПОДВОХ (его допускает почти каждый новичок): канал закрывается, только когда сдропнуты **ВСЕ**
`Sender` — включая исходный `tx`, который остался в главном потоке. Если забыть `drop(tx)`
(или не переместить его в поток), то один живой `Sender` будет вечно держать трубу открытой,
и `rx.collect()` (или `for v in rx`) **зависнет навсегда** — это учебный дедлок. Лечится
ровно одной строкой: `drop(tx)` после раздачи клонов.

> Правило: после того как ты раздал клоны по потокам, оставшийся в главном потоке `Sender`
> надо явно дропнуть. Тогда «последний `tx`» исчезнет, когда отработают все потоки, и
> приёмник корректно завершится.

## Идея 6. Разделяемое состояние и `Arc<T>`

Каналы хороши, когда работа **передаётся** между потоками. Но иногда потокам нужно **видеть
и менять одни и те же данные** — общий счётчик, общую `HashMap`. Для этого нужно
**разделяемое владение** между потоками.

Из модуля 13 ты знаешь `Rc<T>` — счётчик ссылок для нескольких владельцев. Но `Rc`
**нельзя** передавать между потоками:

```rust
use std::rc::Rc;
use std::thread;

let data = Rc::new(5);
let d = Rc::clone(&data);
thread::spawn(move || {
    println!("{}", d);   // ❌ `Rc<i32>` cannot be sent between threads safely
                         //    the trait `Send` is not implemented for `Rc<i32>`
});
```

ПОЧЕМУ нельзя: счётчик ссылок в `Rc` обновляется **неатомарно** (ради скорости). Два потока,
одновременно клонирующие/дропающие один `Rc`, испортили бы счётчик — гонка данных. Поэтому
`Rc` — `!Send` и `!Sync`, и компилятор не пускает его в `spawn` (идея 10).

Для потоков есть «брат» `Arc<T>` (**A**tomically **R**eference **C**ounted) — то же самое, но
счётчик атомарный:

```rust
use std::sync::Arc;
use std::thread;

let data = Arc::new(vec![1, 2, 3]);
let mut handles = Vec::new();
for _ in 0..3 {
    let d = Arc::clone(&data);          // дёшево: атомарный инкремент счётчика
    handles.push(thread::spawn(move || {
        println!("{:?}", d);             // все потоки видят ОДИН вектор
    }));
}
for h in handles { h.join().unwrap(); }
```

`Arc::clone` не копирует данные — лишь увеличивает атомарный счётчик; все клоны указывают на
одно хранилище. Атомарность чуть дороже, поэтому в **однопоточном** коде бери `Rc`, в
**многопоточном** — `Arc`.

ВАЖНОЕ ограничение: `Arc<T>` даёт только **общий** доступ `&T`, а не `&mut T`. Несколько
владельцев не могут одновременно менять данные — это была бы гонка. Чтобы менять — нужен
замок поверх данных (идея 7).

## Идея 7. `Mutex<T>` — взаимное исключение и отравление

`Mutex<T>` (**mut**ual **ex**clusion — взаимное исключение) — это замок: он гарантирует, что
в любой момент к данным имеет доступ **только один** поток. Остальные ждут на `lock()`.

```rust
use std::sync::Mutex;

let m = Mutex::new(0);
{
    let mut guard = m.lock().unwrap();   // блокируем; получаем MutexGuard
    *guard += 1;                         // меняем данные через guard
}                                        // guard дропается → замок отпущен
assert_eq!(*m.lock().unwrap(), 1);
```

`lock()` возвращает `MutexGuard` — это **RAII-страж** (тот же принцип Drop, что у `Box`/`String`
из модуля 13): пока guard жив, замок захвачен; как только guard выходит из области видимости —
замок автоматически отпускается. Guard через `Deref`/`DerefMut` отдаёт `&mut T` **из `&Mutex`**
— это потокобезопасная **внутренняя изменяемость** (потоковый аналог `RefCell` из модуля 13,
но `lock()` **блокирует** вместо паники).

`lock()` возвращает `Result`. `Err` означает **отравленный мьютекс** (poisoned): поток,
державший замок, **запаниковал** — данные могли остаться в противоречивом состоянии, и Rust
помечает мьютекс как «возможно испорченный». В учебных заданиях `.unwrap()` / `.expect(...)`
здесь нормально.

> Подвох: держи guard **коротко**. Пока ты держишь замок, остальные потоки ждут — это
> сужает параллелизм. Частая идиома — взять замок в отдельном блоке `{ }`, сделать минимум и
> сразу отпустить. Особенно опасно держать guard через долгие операции (I/O, рекурсию) или
> через `.await` (модуль 16).

## Идея 8. `RwLock<T>` — много читателей **или** один писатель

`Mutex` пускает к данным строго одного — даже если все потоки только **читают**. `RwLock<T>`
(read-write lock) умнее: он различает чтение и запись.

```rust
use std::sync::RwLock;

let lock = RwLock::new(5);

{
    let r1 = lock.read().unwrap();   // много read() могут сосуществовать
    let r2 = lock.read().unwrap();
    assert_eq!(*r1 + *r2, 10);
}                                    // оба читателя отпущены

{
    let mut w = lock.write().unwrap();  // write() эксклюзивен — ждёт всех читателей
    *w += 1;
}
```

`read()` даёт `&T` и пускает **несколько читателей одновременно**; `write()` даёт `&mut T` и
эксклюзивен. Это ровно правило заимствования Rust («много `&` **или** один `&mut`»), только
проверяемое в рантайме и между потоками.

КОГДА брать `RwLock` вместо `Mutex`: когда **читают часто, а пишут редко** — тогда читатели не
блокируют друг друга и параллелизм выше. ЦЕНА: `RwLock` дороже `Mutex` по накладным расходам;
при потоке постоянных читателей писатель может долго не получить доступ
(**write-starvation**, «голодание писателя»).

> Связь с курсом: стретч-цель капстоуна — заменить `Mutex` на `RwLock` в `SharedDb` и
> подумать, что изменится при разной доле чтений и записей.

## Идея 9. Паттерн `Arc<Mutex<T>>` — канонический рецепт

`Arc` даёт несколько владельцев, но только `&T`. `Mutex` даёт `&mut T` через `&self`, но сам
по себе не разделяется между потоками во владении. **Вместе** они дают и то, и другое — это
**самый частый рецепт разделяемого изменяемого состояния** в Rust:

```text
Arc<Mutex<T>>
 │    │    └── T: сами данные
 │    └── Mutex: доступ к данным — строго одному потоку за раз
 └── Arc: разделяемое владение — несколько потоков «держат» одно хранилище
```

```rust
use std::sync::{Arc, Mutex};
use std::thread;

let data = Arc::new(Mutex::new(0));

let mut handles = Vec::new();
for _ in 0..10 {
    let d = Arc::clone(&data);          // каждый поток — свой Arc-клон того же Mutex
    handles.push(thread::spawn(move || {
        let mut guard = d.lock().unwrap();  // взять замок
        *guard += 1;                        // изменить общие данные
    }));                                    // guard дропнут → замок отпущен
}
for h in handles { h.join().unwrap(); }

assert_eq!(*data.lock().unwrap(), 10);  // ровно 10 — ни один инкремент не потерян
```

Разбор слоёв (то же, что в комментарии `rust/capstone/src/shared.rs`): **Arc** = несколько
владельцев-потоков; **Mutex** = доступ к данным одному за раз; **T** = данные. Это **та же
конструкция, что `SharedDb`** в капстоуне — назовём прямо: `SharedDb` это
`Arc<Mutex<Database>>`, а `SharedTally` (задание 5) — `Arc<Mutex<HashMap<String, i64>>>`.

Сравни с `Rc<RefCell<T>>` из модуля 13 — это **однопоточная** версия того же паттерна
(несколько владельцев + изменяемость). `Arc<Mutex<T>>` — её **многопоточная** замена:
`Rc → Arc` (атомарный счётчик), `RefCell → Mutex` (блокирующая синхронизация вместо паники):

| | один поток (м.13) | много потоков (м.15) |
|---|---|---|
| разделяемое владение | `Rc<T>` | `Arc<T>` |
| внутренняя изменяемость | `RefCell<T>` | `Mutex<T>` / `RwLock<T>` |
| оба вместе | `Rc<RefCell<T>>` | `Arc<Mutex<T>>` |

## Идея 10. `Send` и `Sync` — как Rust **доказывает** потокобезопасность

Откуда компилятор знает, что `Arc<Mutex<T>>` можно отдать в поток, а `Rc<T>` — нельзя? Из двух
маркерных авто-трейтов:

- **`Send`** — тип можно **переместить** в другой поток. Почти все типы `Send`.
- **`Sync`** — на тип можно **дать ссылку `&T` из нескольких потоков**. Формально: `T: Sync`
  ⟺ `&T: Send`.

«Авто-трейт» значит, что компилятор **выводит их сам**: структура `Send`, если все её поля
`Send`; аналогично `Sync`. Тебе почти никогда не нужно писать их руками.

Исключения (типы, у которых одного или обоих трейтов нет):

- `Rc<T>` — **ни `Send`, ни `Sync`** (неатомарный счётчик ссылок);
- `RefCell<T>`, `Cell<T>` — **не `Sync`** (рантайм-проверка займов не атомарна);
- `MutexGuard`, сырые указатели `*const`/`*mut` — особые случаи.

`thread::spawn` требует от тела `F: Send + 'static`, а от возвращаемого значения — `Send`.
Поэтому `Rc` в `spawn` не проходит компиляцию — и это та самая ошибка из идеи 6.

Доказать `Send + Sync` для своего типа можно прямо тестом-свидетелем (witness), как в
капстоуне:

```rust
fn assert_send_sync<T: Send + Sync>() {}
assert_send_sync::<SharedTally>();   // компилируется ⟺ SharedTally: Send + Sync
```

Если бы внутри `SharedTally` был `Rc` вместо `Arc`, этот вызов **не скомпилировался бы** —
компилятор не дал бы шарить тип между потоками. Так трейт-система превращает потокобезопасность
из обещания в **доказанный компилятором факт**.

## Идея 11. Дедлоки и логические гонки — что компилятор НЕ ловит

Гонки данных Rust закрывает. Но **дедлок** (взаимная блокировка) — это логическая ошибка, и
её компилятор не видит. Типичные источники:

1. **Два мьютекса в разном порядке.** Поток A берёт `lock(m1)`, потом `lock(m2)`; поток B —
   наоборот, `lock(m2)`, потом `lock(m1)`. Они могут «встретиться» так, что каждый держит один
   замок и вечно ждёт второй.
2. **Повторный `lock()` в одном потоке.** Поток уже держит `Mutex` и снова зовёт `lock()` на
   нём же — `std::sync::Mutex` не реентерабельный, поток заблокирует сам себя.
3. **Забытый `drop(tx)`** из идеи 5 — приёмник ждёт значения, которые никогда не придут.

Обереги (без готового кода — это упражнение на осознанность):

- **единый порядок захвата** замков во всей программе (всегда m1 раньше m2);
- держать замок **минимально** (взял — сделал — сразу отпустил, идея 7);
- предпочитать **каналы** или **scoped-потоки** разделяемому состоянию, где это естественно.

> Дедлок не «крэшит» программу — она просто **зависает**. Если тест на конкурентность не
> завершается (висит вместо красного/зелёного) — почти наверняка дедлок или незакрытый канал.

## Идея 12. Атомики — когда `Mutex` для счётчика избыточен

Для **простого счётчика** (или флага) целый `Mutex` — перебор: блокировка, guard, риск
отравления. Стандартная библиотека даёт **атомарные типы** — `AtomicUsize`, `AtomicBool`,
`AtomicI64` и др., — которые меняются **одной атомарной инструкцией процессора**, без
блокировки:

```rust
use std::sync::Arc;
use std::sync::atomic::{AtomicUsize, Ordering};
use std::thread;

let counter = Arc::new(AtomicUsize::new(0));
let mut handles = Vec::new();
for _ in 0..8 {
    let c = Arc::clone(&counter);
    handles.push(thread::spawn(move || {
        for _ in 0..500 {
            c.fetch_add(1, Ordering::Relaxed);  // атомарный +1, без замка
        }
    }));
}
for h in handles { h.join().unwrap(); }
assert_eq!(counter.load(Ordering::Relaxed), 4000);
```

`fetch_add(1, ...)` атомарно прибавляет и возвращает прежнее значение; `.load(...)` читает
текущее. Заметь: `Mutex` не нужен вовсе — атомики сами `Sync`, поэтому достаточно
`Arc<AtomicUsize>`.

Про `Ordering` коротко: это про то, как операции с атомиком упорядочены **относительно других
переменных**. Для независимого счётчика хватает `Relaxed` (важна только атомарность каждого
`+1`). Если сомневаешься — безопасный дефолт `SeqCst` (последовательная согласованность).
Глубокая memory ordering (`Acquire`/`Release` и модель памяти) — за пределами junior; здесь
достаточно знать, что для счётчика `Relaxed` корректен.

## Шпаргалка — что когда брать

| Нужно | Бери |
|---|---|
| Передать **результат** работы из потока | канал (`mpsc::channel`) |
| Общий **счётчик / флаг** | `Arc<AtomicUsize>` / `Arc<AtomicBool>` |
| Разделяемое **изменяемое** состояние | `Arc<Mutex<T>>` |
| То же, но **читают часто, пишут редко** | `Arc<RwLock<T>>` |
| Короткий проход по **локальным** данным, без `'static` | `thread::scope` (см. ниже) |
| Один владелец на поток, передаём владение | `move`-замыкание |

### Scoped-потоки `thread::scope` (Rust 1.63+)

Есть способ запустить потоки, которым **разрешено заимствовать локальные данные** без `Arc`:

```rust
let data = vec![1, 2, 3];
let total: i32 = std::thread::scope(|s| {
    let h = s.spawn(|| data.iter().sum::<i32>());  // заимствует &data — это ОК
    h.join().unwrap()
});
```

ПОЧЕМУ это безопасно без `'static`: `scope` **гарантирует**, что все порождённые в нём потоки
завершатся (join) до выхода из `scope`. Значит, заимствованные `&data` не переживут владельца —
borrow-checker доволен. **Когда `scope` vs `Arc`:** `scope` — для короткого «развернуть работу
по локальным данным и тут же собрать»; `Arc` — когда поток должен жить дольше текущей функции
или владеть данными сам.

## Задание 5 — `SharedTally`: общий счётчик по ключам

Это **кульминация модуля** — ты собираешь руками ту же конструкцию, что несёт `SharedDb` в
капстоуне. `SharedTally` — потокобезопасное хранилище «ключ → накопленная сумма».

В `src/lib.rs` объявлена структура с **приватным** полем:
`inner: Arc<Mutex<HashMap<String, i64>>>`. Реализуй:

- `pub fn new() -> Self` — внутри `Arc::new(Mutex::new(HashMap::new()))`;
- `impl Default` — `default()` == `new()`;
- `pub fn handle(&self) -> SharedTally` — **второй дескриптор того же хранилища**
  (через `Arc::clone(&self.inner)`), НЕ копия данных;
- `impl Clone for SharedTally` — `clone()` == `handle()`;
- `pub fn add(&self, key: &str, delta: i64) -> i64` — под замком: `entry(key).or_insert(0)`
  прибавить `delta`, вернуть новое значение;
- `pub fn get(&self, key: &str) -> i64` — под замком: значение по ключу или `0`.

Тесты — `tests/shared.rs` (включая witness `assert_send_sync::<SharedTally>()` и конкурентный
тест 8 × 500).

<details><summary>Подсказка 5.1 — почему тип именно <code>Arc&lt;Mutex&lt;HashMap&gt;&gt;</code>?</summary>

Разбери по слоям (идея 9):
- **`HashMap<String, i64>`** — сами данные: ключ → накопленная сумма.
- **`Mutex<…>`** — даёт `&mut HashMap` через `&self` и сериализует доступ: один поток за раз.
  Без него два потока могли бы одновременно менять `HashMap` — гонка.
- **`Arc<…>`** — разделяемое владение между потоками: каждый поток держит свой клон, но все
  указывают на один `Mutex`. Без него `Mutex` нельзя было бы отдать в несколько потоков.

Сравни с `Rc<RefCell<HashMap>>` из модуля 13 — это однопоточная версия. Здесь `Rc → Arc`,
`RefCell → Mutex`.
</details>

<details><summary>Подсказка 5.2 — <code>new</code>, <code>handle</code>, <code>Clone</code></summary>

`new` оборачивает пустую карту в две обёртки:
```rust
SharedTally { inner: Arc::new(Mutex::new(HashMap::new())) }
```
`handle` отдаёт новый `SharedTally` с **клоном `Arc`** (не новой картой!):
```rust
SharedTally { inner: Arc::clone(&self.inner) }
```
`Clone` реализуй **через** `handle`, чтобы `.clone()` и `.handle()` были эквивалентны:
тело `fn clone(&self)` — это `self.handle()`. `Default` — `Self::new()`.

Подумай: в чём разница между `Arc::clone(&self.inner)` (что нам нужно) и
`self.inner.lock().unwrap().clone()` (клон самой `HashMap`)? Почему первое — «тот же ящик», а
второе — «копия содержимого»?
</details>

<details><summary>Подсказка 5.3 — <code>add</code> через <code>entry</code> (и почему он возвращает значение)</summary>

Возьми замок, потом работай с картой через `entry`:
```rust
let mut guard = self.inner.lock().expect("mutex poisoned");
let slot = guard.entry(key.to_string()).or_insert(0);
*slot += delta;
*slot
```
`entry(k).or_insert(0)` — если ключа нет, вставляет `0` и отдаёт `&mut i64` на него; если есть
— просто `&mut` на существующее значение (это паттерн из модуля 08). `*slot += delta` меняет
его на месте, последняя строка возвращает **новое** значение. `get` устроен проще: возьми
замок и верни `guard.get(key).copied().unwrap_or(0)`.
</details>

<details><summary>Подсказка 5.4 — почему конкурентный тест 8×500 обязан давать 4000</summary>

Восемь потоков параллельно зовут `add("shared", 1)` по 500 раз. Если бы `add` не брал замок,
инкременты терялись бы (гонка из идеи 1) и итог был бы случайным числом < 4000. `Mutex`
выстраивает потоки в очередь: каждый `entry/+=/return` происходит атомарно относительно других.
Поэтому теряться нечему — итог ровно `8 * 500 = 4000`. Это тот же экзамен, что
`concurrent_lpush` в капстоуне. А witness-тест `assert_send_sync::<SharedTally>()` не
скомпилируется, если ты случайно положишь внутрь `Rc` вместо `Arc` — компилятор не даст шарить
такой тип между потоками.
</details>
