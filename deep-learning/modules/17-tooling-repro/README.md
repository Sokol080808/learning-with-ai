# Модуль 17 — Воспроизводимость и инженерия

Большая идея модуля: модель, которую нельзя **повторить**, — это не результат, а случайность.
Ты обучил сеть, получил accuracy 0.91, закрыл ноутбук — а назавтра тот же код выдаёт 0.87.
Что изменилось? Может, баг? Может, повезло? Ты не знаешь, потому что не зафиксировал
**источники случайности**. Этот модуль — про инженерную гигиену, которая превращает
эксперимент из «как-то получилось» в «получается каждый раз»: фиксируем сиды, умеем считать
размер модели, **сохраняем и загружаем веса**, не путаем устройство и режим train/eval. Это не
гламурная теория про градиенты — это ремесло, без которого вся теория не доезжает до результата.

> Как и весь курс, держим всё на **CPU**. Воспроизводимость на CPU честнее и проще: одни и те же
> числа у всех, без драйверов и версий CUDA. Идеи переносятся на GPU без изменений.

## Идея 1. Сиды: random, numpy, torch — три источника случайности, не один

Когда сеть «инициализируется случайно» или ты «перемешиваешь батчи», под капотом работает
**генератор псевдослучайных чисел** (PRNG). Он не магия: это функция, которая из текущего
внутреннего состояния (`seed`) считает следующее число и обновляет состояние. Поставил один и тот
же `seed` — получишь **ту же самую** последовательность. В этом вся суть воспроизводимости.

Ловушка в том, что в типичном DL-коде работают **три независимых** генератора, и забыть любой —
значит потерять повторяемость:

- `random` — стандартная библиотека Python (например, `random.shuffle` списка файлов);
- `numpy.random` — почти любая подготовка данных на numpy;
- `torch` — инициализация весов слоёв, `DataLoader` с `shuffle=True`, dropout-маски.

```python
import random, numpy as np, torch

random.seed(0)
np.random.seed(0)
torch.manual_seed(0)

# теперь ОБА вызова ниже дадут одно и то же при каждом запуске программы
torch.randn(3)   # tensor([ 1.5410, -0.2934, -2.1788])  — всегда эти числа при seed=0
```

**ПОЧЕМУ одного `torch.manual_seed` мало?** Потому что генераторы не делят состояние. Зафиксировал
только torch — а данные ты перемешал через `np.random.permutation`, и порядок объектов снова
«поплыл». Один забытый источник случайности рушит воспроизводимость целиком, как одна незакрытая
дверь рушит «герметичность». Поэтому правило: **сидируй все три разом**, одной функцией, в самом
начале программы.

Маленький численный факт, который полезно прочувствовать: вызови `set_seed(0)`, возьми
`torch.randn(4)` — запомни числа. Снова `set_seed(0)`, снова `torch.randn(4)` — **те же** четыре
числа, до последнего бита. А без повторного сидирования второй вызов даст уже **следующие** четыре
числа из последовательности. Сид не «делает числа фиксированными» — он **возвращает генератор в
исходную точку**.

## Идея 2. Сколько в модели обучаемых параметров

Прежде чем обучать сеть, полезно знать её «вес» в параметрах — это и оценка памяти, и грубая
прикидка склонности к переобучению, и просто санити-чек («я правда собрал то, что хотел?»). Параметр
— это любой тензор, который обновляется градиентным спуском: матрица весов `W`, вектор смещений `b`,
параметры нормализаций. У линейного слоя `(D_in → D_out)` параметров ровно `D_in·D_out` (матрица)
плюс `D_out` (смещение).

В PyTorch у любого `nn.Module` есть `model.parameters()` — перебор всех тензоров-параметров. У
каждого тензора `p` есть форма, а `p.numel()` — **число элементов** в нём (произведение размеров
формы). Считаем только **обучаемые** — те, у кого `requires_grad=True` (замороженные слои не в счёт).

```python
import torch.nn as nn

model = nn.Linear(4, 3)          # W: (3, 4) = 12 чисел, b: (3,) = 3 числа
sum(p.numel() for p in model.parameters() if p.requires_grad)   # 12 + 3 = 15
```

**ПОЧЕМУ через `numel()` и `requires_grad`, а не «на глаз»?** Потому что в реальной сети десятки
слоёв, и часть из них может быть **заморожена** (например, при дообучении: тело сети `freeze`, учим
только «голову»). Формула `sum(p.numel() for p in model.parameters() if p.requires_grad)` честно
учитывает только то, что реально учится. Форма каждого `p` — это `(out, in)` для весов и `(out,)`
для смещений; `numel()` просто перемножает размеры формы.

## Идея 3. device (cpu/gpu) и режимы train/eval — две независимые «настройки»

У модели и тензоров есть **где** считать (`device`: `cpu` или `cuda`) и **как** вести себя слоям
(`model.train()` против `model.eval()`). Это две ортогональные вещи, и обе — частый источник тихих
багов.

- **device.** Все участники операции должны лежать на одном устройстве. Перенос — `model.to(device)`
  и `x.to(device)`. В нашем курсе `device = "cpu"` всегда. Грамотный приём — выбирать устройство одной
  строкой и не хардкодить: `device = "cuda" if torch.cuda.is_available() else "cpu"` (у нас это даст
  `"cpu"`).

- **train/eval.** Некоторые слои ведут себя **по-разному** при обучении и при инференсе. Главные двое:
  - **Dropout** в `train()` случайно зануляет часть активаций (регуляризация), а в `eval()`
    выключается полностью — пропускает всё.
  - **BatchNorm** в `train()` считает статистики по текущему батчу, а в `eval()` использует накопленные
    «бегущие» средние.

```python
model.train()            # режим обучения: dropout активен, BN по батчу
model.eval()             # режим инференса: dropout выкл, BN по накопленному

# флаг доступен и для чтения:
model.training           # True после .train(), False после .eval()
```

**ПОЧЕМУ это критично?** Классическая ошибка новичка — мерить качество на валидации, **забыв**
`model.eval()`. Тогда dropout продолжает резать активации, и метрики случайно прыгают от запуска к
запуску, а ты гадаешь, что сломалось. И зеркальная ошибка — обучать, забыв вернуть `model.train()`
после валидации, тогда сеть «учится» с выключенной регуляризацией. Правило-рефлекс: **перед
обучением — `train()`, перед замером/инференсом — `eval()`**. Часто инференс ещё оборачивают в
`torch.no_grad()`, чтобы не строить граф ради экономии памяти.

## Идея 4. Сохранение и загрузка модели через state_dict, и гигиена экспериментов

Обучение длинное — терять веса в конце недопустимо. В PyTorch **канонический** способ сохранить
модель — не пиклить объект целиком, а сохранить только **`state_dict`**: обычный словарь
`{имя_параметра: тензор}`. Это и есть «вся выученность» сети, отделённая от кода её архитектуры.

```python
torch.save(model.state_dict(), "model.pt")          # сохранили ТОЛЬКО веса (словарь тензоров)

# при загрузке: сначала создаём ПУСТУЮ модель той же архитектуры, потом вливаем веса
model2 = MyNet()                                     # новый объект, веса пока случайные
model2.load_state_dict(
    torch.load("model.pt", weights_only=True)        # weights_only=True — современная норма
)
model2.eval()                                        # для инференса — в режим eval
```

> **Заметка о безопасности.** Начиная с PyTorch 2.x, при `torch.load` рекомендуется явно
> передавать `weights_only=True`. Без этого флага PyTorch предупреждает (и в будущих версиях
> изменит умолчание): загрузка произвольного `.pt`-файла через pickle потенциально опасна,
> если файл получен из ненадёжного источника. `weights_only=True` разрешает только тензоры
> и базовые типы Python — этого достаточно для `state_dict`. Всегда используй этот флаг
> при загрузке весов.

**ПОЧЕМУ `state_dict`, а не `torch.save(model)` целиком?** Потому что пикл всего объекта жёстко
привязывает файл к конкретному классу, его модулю и пути импорта: переименовал файл с классом —
загрузка падает. `state_dict` — это просто данные (числа по именам), они переживают рефакторинг кода.
Поэтому контракт загрузки всегда из двух шагов: **создай каркас модели → влей в него
`state_dict`**. После корректной загрузки веса нового объекта **поэлементно совпадают** с исходными
(это проверяется через `allclose`).

Форма данных словами: `state_dict` линейного слоя `nn.Linear(4, 3)` — это словарь с ключами
`weight` (тензор `(3, 4)`) и `bias` (тензор `(3,)`). Загрузка просто копирует эти тензоры на свои
места в новой модели.

**Что такое `torch.no_grad()` и зачем он при инференсе.** По умолчанию каждый тензор
с `requires_grad=True` тянет за собой **граф вычислений** — PyTorch запоминает, из каких
операций тензор получен, чтобы потом `backward()` смог пройти по этому графу назад и
посчитать градиенты. На инференсе (замер качества, финальный loss, генерация) градиенты
**не нужны** — мы ничего не учим. Блок `with torch.no_grad():` говорит автограду «не строй
граф на этих операциях»: это экономит память (не надо хранить промежуточные активации ради
будущего `backward`) и чуть ускоряет проход. Используй его вокруг любого forward, после
которого НЕ будет `backward()`: валидация, тест, инференс, подсчёт метрик.

```python
model.eval()                         # режим инференса (Идея 3)
with torch.no_grad():                # граф не строится — память не растёт
    preds = model(x_val)
    val_loss = criterion(preds, y_val).item()
```

> `model.eval()` и `torch.no_grad()` — **разные** вещи, их часто путают. `eval()` меняет
> поведение слоёв (dropout/BN), но граф всё ещё строится. `no_grad()` отключает граф, но не
> трогает режим слоёв. На честном инференсе нужны **оба**.

**Гигиена экспериментов** — то, что отличает результат от случайности:
- фиксируй сид **в начале** каждого запуска (Идея 1);
- логируй, **что** именно ты запускал: гиперпараметры, версии, сид;
- сохраняй `state_dict` лучшей модели, а не «последней попавшейся»;
- меняй **одну** вещь за эксперимент — иначе не поймёшь, что повлияло.

«Логируй, что запускал» звучит абстрактно — сделаем это конкретным. Минимальный
лог — это словарь-конфиг, дописанный строкой в CSV рядом с метрикой запуска:

```python
import csv, datetime

config = {"seed": 0, "lr": 0.01, "hidden": 16, "n_steps": 5}
val_loss = 0.137                                      # результат запуска

row = {**config, "val_loss": val_loss,
       "ts": datetime.datetime.now().isoformat()}
with open("runs.csv", "a", newline="") as f:
    w = csv.DictWriter(f, fieldnames=row.keys())
    if f.tell() == 0:                                 # пустой файл → шапка
        w.writeheader()
    w.writerow(row)
```

Теперь у каждой строки `runs.csv` есть **сид и гиперпараметры**, по которым запуск
можно повторить, и **метрика**, по которой запуски можно сравнить. Это уже не «как-то
получилось 0.91», а воспроизводимая запись эксперимента — основа любого трекинга
(`tensorboard`, `wandb`, `mlflow` делают то же самое, только богаче).

## Идея 5. Чекпоинты: сохранить не только веса, но и состояние, чтобы продолжить с того же места

В Идее 4 мы сохраняли `state_dict` модели — этого хватает, чтобы **запустить** обученную сеть
на инференс. Но представь длинную тренировку: 200 эпох, прервалась на 137-й (упал сервер,
кончилось время в облаке). Хочется **возобновить с 138-й**, а не учить заново. И тут одних
весов **мало**.

**Что ещё нужно сохранить, кроме весов.** Современные оптимизаторы — не «чистая» функция от
градиента. У них есть **внутреннее состояние**, накопленное за прошлые шаги:

- **SGD с моментумом** хранит `momentum_buffer` — экспоненциально сглаженный «разгон»
  градиента. Шаг зависит не только от текущего градиента, но и от этого буфера.
- **Adam** хранит две оценки — `m` (первый момент, как моментум) и `v` (второй момент,
  для адаптивного масштаба шага по каждому параметру), плюс счётчик шагов для bias-коррекции.

Если при возобновлении залить только веса, а оптимизатор создать заново — он стартует
**«с холода»**: моментум обнулён, у Adam адаптивные масштабы сброшены. Первые шаги после
такого «возобновления» пойдут не туда, куда шла бы непрерывная тренировка, и траектория loss
**разойдётся** с той, что была бы без прерывания. Поэтому **честный** чекпоинт — это словарь
из трёх вещей:

```python
ckpt = {
    "model":     model.state_dict(),        # выученные веса (как в Идее 4)
    "optimizer": optimizer.state_dict(),    # буферы моментума / m,v Adam, счётчики
    "epoch":     epoch,                     # на каком шаге/эпохе мы остановились
}
torch.save(ckpt, "ckpt.pt")
```

Возобновление — зеркальный процесс: создаём **тот же каркас** модели и оптимизатора, вливаем
обратно оба `state_dict` и узнаём, с какой эпохи продолжать:

```python
model = build_skeleton()                                    # та же архитектура
optimizer = optim.SGD(model.parameters(), lr=0.1, momentum=0.9)

ckpt = torch.load("ckpt.pt", weights_only=True)
model.load_state_dict(ckpt["model"])
optimizer.load_state_dict(ckpt["optimizer"])
start_epoch = ckpt["epoch"]                                  # продолжаем с start_epoch + 1
```

**Как это проверить по-честному (оракул этого задания).** Запусти тренировку на `k+m` шагов
непрерывно — запомни финальный loss (baseline). Потом запусти `k` шагов, сохрани чекпоинт,
**с нуля** создай модель+оптимизатор, загрузи чекпоинт и доучи `m` шагов. На CPU с
фиксированным сидом финальный loss обязан совпасть с baseline **до последнего бита**. Если
совпадает — состояние оптимизатора восстановилось верно. Проверка одних весов через `allclose`
этого **не доказывает**: веса можно восстановить, а потерять моментум — и тогда траектория
тихо поедет. Именно поэтому Karpathy в nanoGPT (и любой серьёзный train-скрипт) кладёт в
чекпоинт `optimizer.state_dict()`, а не только веса.

## Идея 6. Перенос обучения: заморозь тело, учи голову

Это самый практичный приём всего DL: почти никто не учит большую сеть **с нуля**. Берут уже
обученную модель и **дообучают** её под свою задачу. Почему так:

1. **Интуиция — нижние слои переиспользуемы.** Сеть, обученная на большом наборе, в нижних
   слоях выучивает **общие** признаки (для картинок: края → текстуры → части объектов). Эти
   признаки полезны почти для любой похожей задачи. Учить их заново на маленьких данных
   дорого и ведёт к переобучению. Поэтому берём готовое **тело** (backbone) и доучиваем только
   тонкую **голову** (head) под свои классы.

2. **Механика — три приёма, которые курс УЖЕ умеет:**
   - **freeze (заморозка).** `p.requires_grad_(False)` — по такому параметру градиент не течёт,
     и оптимизатор его не двигает. Это та самая «заморозка тела» из Идеи 2, где
     `count_parameters` уже учитывает только обучаемое.
   - **feature-extraction (извлечение признаков).** Заморозить **весь** backbone, заменить
     последний `Linear` на **свежий** с нужным числом выходов `num_classes`, и учить **только
     его**. Дёшево: число обучаемых параметров резко падает.
   - **fine-tuning (тонкая доводка).** После того как голова «прогрелась», можно **разморозить**
     backbone и доучить **всё** с **маленьким** `lr` (чтобы не сломать выученные признаки
     большими шагами). В задании мы остаёмся на уровне feature-extraction — он junior-размерный;
     полную доводку оставляем как теорию.

3. **Почему это работает (и как проверить).** У замороженного параметра после `backward()`
   `.grad is None` (он не попал в граф), поэтому шаг SGD его **не меняет** — вес остаётся
   **бит-в-бит** прежним. Это и есть проверяемое сердце переноса обучения: после дообучения
   веса тела обязаны совпадать с весами донора поэлементно, а двигается только голова. А число
   обучаемых параметров (`count_parameters` из Идеи 2) при feature-extraction резко меньше
   полного — это и есть «дёшево».

```python
# 1. загрузить «предобученный» backbone (у нас — крошечная сеть, которую мы
#    сами обучили и сохранили через save_model; НИКАКИХ загрузок из интернета)
model = build_skeleton()
load_model(model, "backbone.pt")

# 2. свежая голова под новое число классов
replace_head(model, new_out=3)        # последний Linear → Linear(in, 3)

# 3. заморозить всё, кроме головы
freeze_backbone(model, n_keep_trainable_tail=1)

# 4. учить только голову — оптимизатор берёт ТОЛЬКО обучаемые параметры
trainable = [p for p in model.parameters() if p.requires_grad]
optimizer = optim.SGD(trainable, lr=0.05)
# ... обычный цикл обучения; веса тела не сдвинутся ни на бит
```

> **«Предобученная» модель без интернета.** В этом курсе всё на CPU и без загрузок, поэтому
> «предобученный backbone» здесь = крошечная `nn.Sequential`, которую мы **сами** обучили на
> шуме и сохранили через `save_model` (Идея 4). Это честная модель того же сценария: на
> практике backbone приходит из `torchvision.models` (ResNet и т.п.), но **механика —
> freeze / replace head / train head — ровно та же**. fast.ai называет перенос обучения
> навыком №1 практического DL: он экономит данные, время и вычисления.

кирпичи, Задание 7 (`save_checkpoint`/`load_checkpoint`) — чекпоинты из Идеи 5, Задание 8
(`freeze_backbone`/`replace_head`/`transfer_finetune`) — перенос обучения из Идеи 6.

1. `set_seed(seed: int) -> None` — зафиксировать **все три** источника случайности: `random`,
   `numpy.random`, `torch`. После вызова два одинаковых блока со случайностью обязаны дать
   идентичные результаты.
2. `count_parameters(model: torch.nn.Module) -> int` — число **обучаемых** параметров модели (сумма
   `numel()` по тем параметрам, у кого `requires_grad=True`).
3. `save_model(model: torch.nn.Module, path: str) -> None` — сохранить **`state_dict`** модели в
   файл `path`.
4. `load_model(model: torch.nn.Module, path: str) -> torch.nn.Module` — загрузить `state_dict` из
   `path` в переданную модель (каркас нужной архитектуры) и **вернуть** эту же модель с залитыми
   весами.
5. `set_mode(model: torch.nn.Module, train: bool) -> torch.nn.Module` — переключить модель в
   режим обучения или инференса и вернуть её же (см. Идею 3 и Задание 5 ниже).
6. `run_reproducible_experiment(seed, ...) -> dict` — интеграционная функция, которая собирает
   весь пайплайн воспроизводимости воедино (см. Задание 5 ниже).
7. `save_checkpoint(model, optimizer, epoch, path) -> None` и
   `load_checkpoint(model, optimizer, path) -> int` — полный чекпоинт обучения (веса + состояние
   оптимизатора + epoch) и его восстановление (см. Идею 5 и Задание 6 ниже).
8. `freeze_backbone(model, n_keep_trainable_tail) -> nn.Module`,
   `replace_head(model, new_out) -> nn.Module` и
   `transfer_finetune(backbone_path, x, y, new_out, n_steps, lr) -> dict` — перенос обучения:
   заморозить тело, поставить свежую голову, доучить только её (см. Идею 6 и Задание 7 ниже).

Запуск:
```bash
./deep-learning/run.sh 17
```

### Подсказки

<details><summary>Задание 1 — что именно сидировать</summary>
Три вызова, по одному на генератор: <code>random.seed(seed)</code>,
<code>np.random.seed(seed)</code>, <code>torch.manual_seed(seed)</code>. Порядок не важен, важно
не забыть ни один. Функция ничего не возвращает (<code>None</code>) — её работа в побочном эффекте:
вернуть все генераторы в исходную точку.
</details>

<details><summary>Задание 1 — как проверить себя руками</summary>
<code>set_seed(0); a = torch.randn(4)</code>, затем снова <code>set_seed(0); b = torch.randn(4)</code>.
Должно быть <code>torch.allclose(a, b)</code>. А если убрать второй <code>set_seed</code> — числа
будут уже другими (генератор ушёл дальше по последовательности). Ровно так и устроен один из тестов.
</details>

<details><summary>Задание 2 — почему именно эта формула</summary>
<code>sum(p.numel() for p in model.parameters() if p.requires_grad)</code>. <code>p.numel()</code> —
число элементов в тензоре-параметре (произведение размеров формы). Фильтр <code>if p.requires_grad</code>
отсекает замороженные слои, чтобы считать только то, что реально обучается. Верни <code>int</code>.
</details>

<details><summary>Задание 3 — что сохранять</summary>
Не сам объект модели, а его словарь весов: <code>torch.save(model.state_dict(), path)</code>.
<code>state_dict()</code> возвращает <code>{имя: тензор}</code> — компактное и устойчивое к рефакторингу
представление выученных весов.
</details>

<details><summary>Задание 4 — два шага загрузки</summary>
Каркас модели тебе уже передали аргументом <code>model</code> (он создан с нужной архитектурой).
Осталось: <code>state = torch.load(path)</code>, затем <code>model.load_state_dict(state)</code>, и
<code>return model</code>. После этого веса <code>model</code> поэлементно совпадут с теми, что были
сохранены (проверяется через <code>allclose</code>).
</details>

### Задание 5 (интеграционное) — `set_mode` + `run_reproducible_experiment`

Задания 1–4 — каждое про одну «гайку». Задание 5 собирает все гайки вместе и проверяет, что
они держат конструкцию: получился ли у тебя настоящий **воспроизводимый эксперимент**.

#### 5a. `set_mode(model, train: bool) -> model`

Оберни вызовы `model.train()` / `model.eval()` в удобную функцию с явным аргументом-флагом.
Она должна:
- при `train=True` переводить модель в режим обучения;
- при `train=False` — в режим инференса;
- возвращать ту же модель (для удобства цепочки вызовов);
- после вызова `model.training` обязан совпадать с аргументом `train`.

Звучит тривиально — и это намеренно. Смысл не в сложности кода, а в том, чтобы сделать
переключение режимов **явным** и читаемым, а не прятать его в магических `.train()` где-то
в глубине функции.

#### 5b. `run_reproducible_experiment(seed, n_features=8, hidden=16, n_steps=5, lr=0.01) -> dict`

Полный пайплайн одним вызовом. Функция обязана:
1. Зафиксировать все три источника случайности через `set_seed(seed)`.
2. Построить двухслойную сеть: `Linear(n_features → hidden) → ReLU → Linear(hidden → 1)`.
3. Сгенерировать случайный батч: `x` формы `(16, n_features)` и `y` формы `(16, 1)`.
4. Обучить модель `n_steps` шагов через **SGD** с `MSELoss` в режиме `train`.
5. Перевести модель в `eval`, посчитать финальный `MSELoss` на том же батче
   (`torch.no_grad()`).
6. Вернуть словарь:
   ```python
   {
       "final_loss":    float,  # MSE после n_steps шагов
       "n_params":      int,    # число обучаемых параметров сети
       "training_mode": bool,   # model.training (должно быть False)
   }
   ```

**Ключевое свойство:** два вызова с одним и тем же `seed` обязаны вернуть **побитово**
идентичный `final_loss`. Если это не так — где-то утекает неконтролируемая случайность.

<details><summary>Подсказка 1 — почему важно вызывать set_seed ДО создания модели</summary>
Инициализация весов `nn.Linear` — тоже случайная операция (PyTorch использует `kaiming_uniform_`
по умолчанию). Если засидировать после создания модели, веса уже «случайные» из внешнего
состояния генератора — и воспроизводимость нарушена. Вопрос: в каком порядке должны идти
<code>set_seed(seed)</code> и <code>nn.Sequential(...)</code>?
</details>

<details><summary>Подсказка 2 — тренировочный цикл</summary>
Шаблон одного шага SGD всегда один и тот же: обнулить градиенты → forward → посчитать loss →
<code>loss.backward()</code> → шаг оптимизатора. Не забудь поставить модель в <code>train</code>-режим
перед циклом: иначе если в сети есть Dropout или BatchNorm, они будут себя вести «по-инференсному»
прямо во время обучения, и ты получишь тихий баг, который не сразу видно. А после цикла —
<code>set_mode(model, False)</code> и <code>torch.no_grad()</code> вокруг финального forward.
</details>

<details><summary>Подсказка 3 — как отлаживать детерминизм</summary>
Запусти функцию дважды с одним сидом и сравни <code>final_loss</code>:
<code>r1["final_loss"] == r2["final_loss"]</code> (не <code>allclose</code> — именно <code>==</code>,
потому что требуем побитовое совпадение). Если не совпадает — добавь принты после каждого
«источника случайности» (инициализация весов, генерация x/y) и найди, где траектории
расходятся. Скорее всего — забытый <code>set_seed</code> или <code>set_seed</code>, сделанный
после создания тензоров.
</details>

### Задание 6 — `save_checkpoint` / `load_checkpoint` (Идея 5)

Задание 3–4 сохраняли только **веса**. Здесь — **полный** чекпоинт, по которому можно
**возобновить** обучение с того же места: веса модели + состояние оптимизатора + номер epoch.

#### 6a. `save_checkpoint(model, optimizer, epoch, path) -> None`

Собери один словарь с тремя ключами (`"model"`, `"optimizer"`, `"epoch"`) и сохрани его одним
вызовом `torch.save`. Ничего не возвращай.

#### 6b. `load_checkpoint(model, optimizer, path) -> int`

Прочитай словарь, влей `state_dict` обратно в `model` **и** в `optimizer`, верни сохранённый
`epoch` как `int`. `model` и `optimizer` — уже готовые объекты нужной архитектуры/типа (как
каркас в `load_model`).

**Ключевое свойство (оракул):** прерванная и продолженная тренировка обязана дать **тот же**
финальный loss, что и непрерывная того же числа шагов. Если оптимизатор с моментумом — это
бит-в-бит на CPU.

<details><summary>Подсказка 1 — какой формы словарь сохранять</summary>
Канонический чекпоинт — это <code>{"model": model.state_dict(), "optimizer":
optimizer.state_dict(), "epoch": epoch}</code>. Сохрани его так же, как в Задании 3 сохранял
веса, только теперь объект сохранения — этот словарь целиком, а не один <code>state_dict</code>.
</details>

<details><summary>Подсказка 2 — почему мало восстановить только веса</summary>
У <code>SGD(momentum=0.9)</code> и у <code>Adam</code> есть внутренние буферы (моментум; у Adam
ещё <code>m</code> и <code>v</code>). Это «память» оптимизатора о прошлых шагах. Если её не
восстановить, оптимизатор стартует «с холода», и траектория loss разойдётся с непрерывной.
Поэтому грузить надо <em>оба</em> <code>state_dict</code>: и в модель, и в оптимизатор. Вопрос:
есть ли у объекта <code>optimizer</code> метод <code>load_state_dict</code>, симметричный
<code>model.load_state_dict</code>?
</details>

<details><summary>Подсказка 3 — как отладить «прерванный == непрерывный»</summary>
Сделай baseline: <code>k+m</code> шагов непрерывно, запомни финальный loss. Потом: <code>k</code>
шагов → <code>save_checkpoint</code> → <em>свежие</em> модель и оптимизатор → <code>load_checkpoint</code>
→ ещё <code>m</code> шагов. Сравни <code>final_loss == baseline</code> (именно <code>==</code>).
Не совпало — почти наверняка не восстановил состояние оптимизатора (грузишь веса, но забыл
<code>optimizer.load_state_dict</code>), либо при возобновлении создал оптимизатор с другими
гиперпараметрами. Что вернуть из функции — это <code>epoch</code>, приведённый к <code>int</code>.
</details>

### Задание 7 — перенос обучения: `freeze_backbone` / `replace_head` / `transfer_finetune` (Идея 6)

Три функции, которые собирают сценарий feature-extraction: загрузить «предобученное» тело →
поставить свежую голову → заморозить тело → доучить только голову. «Предобученный» backbone — это
крошечная сеть, которую тест **сам** строит и сохраняет через `save_model` (без интернета).

#### 7a. `freeze_backbone(model, n_keep_trainable_tail) -> model`

`model` — это `nn.Sequential`. Заморозь параметры всех подмодулей, **кроме** последних
`n_keep_trainable_tail` (это «голова»). У головы параметры остаются обучаемыми. Верни `model`.

#### 7b. `replace_head(model, new_out) -> model`

Найди **последний** `nn.Linear` и замени его свежим `nn.Linear(in_features=старое_in,
out_features=new_out)`. Остальное не трогай. Верни `model`.

#### 7c. `transfer_finetune(backbone_path, x, y, new_out, n_steps=20, lr=0.05) -> dict`

Загрузи backbone в каркас (`load_model`), поставь новую голову, заморозь тело, обучи `n_steps`
шагов SGD + CrossEntropy на `(x, y)`. Верни диагностику: `trainable_params`, `frozen_params`,
`backbone_unchanged` (веса тела бит-в-бит = весам донора), `loss_decreased`.

<details><summary>Подсказка 1 — как «оставить обучаемым только хвост»</summary>
Перебери подмодули <code>nn.Sequential</code> с индексом (он итерируется как список).
Граница хвоста — индекс <code>len(model) - n_keep_trainable_tail</code>: всё, что левее —
заморозить (<code>p.requires_grad_(False)</code>), всё, что правее-или-равно — оставить
обучаемым. Заморозка/разморозка ставится на <em>каждый</em> параметр подмодуля через
<code>for p in submodule.parameters(): ...</code>.
</details>

<details><summary>Подсказка 2 — replace_head: где взять in_features</summary>
Сначала найди индекс последнего <code>nn.Linear</code> (пройди по модулям, запоминая индекс
каждого <code>isinstance(m, nn.Linear)</code>). У старого слоя возьми <code>old.in_features</code>
— размерность признаков тела не меняется, меняется только число выходов. Новый слой
<code>nn.Linear(old.in_features, new_out)</code> присвой на то же место в <code>Sequential</code>
(к нему можно обращаться по индексу: <code>model[idx] = ...</code>).
</details>

<details><summary>Подсказка 3 — transfer_finetune: на что обучать оптимизатор</summary>
После заморозки тела оптимизатор должен видеть <em>только</em> обучаемые параметры, иначе он
попытается двигать замороженные. Собери их фильтром:
<code>trainable = [p for p in model.parameters() if p.requires_grad]</code> и передай именно
этот список в <code>optim.SGD(trainable, lr=lr)</code>. Тогда тело гарантированно не сдвинется.
</details>

<details><summary>Подсказка 4 — как доказать «тело не изменилось»</summary>
До обучения сделай <em>копии</em> весов тела: <code>p.detach().clone()</code> для каждого
параметра тела (всё, кроме головы). После обучения сравни клоны с текущими весами через
<code>torch.equal(before, after)</code> (именно равенство, не <code>allclose</code> — заморозка
обязана дать <em>бит-в-бит</em>). Если хоть один параметр тела сдвинулся — значит, он не был
заморожен, либо оптимизатор получил его в список параметров.
</details>
