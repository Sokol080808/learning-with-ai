# Модуль 12 — MLP на PyTorch: nn.Module и обучение

Большая идея модуля: в прошлых модулях ты руками собирал линейный слой (`x @ W + b`),
руками держал параметры в отдельных переменных, руками их обновлял. Это работало для одного-двух
слоёв. Но как только сеть растёт, такой ручной учёт превращается в кашу: где какие веса, не забыл
ли обнулить градиент, не перепутал ли формы. PyTorch предлагает **порядок**: слой `nn.Linear` сам
хранит свои `W` и `b`, контейнер `nn.Module` собирает слои в одну сеть и **сам знает все её
параметры**, оптимизатор из `torch.optim` обновляет эти параметры за тебя, а autograd (модуль 11)
по-прежнему считает градиенты. Твоя работа сводится к четырём строчкам в цикле:
**forward → loss → backward → step**. В этом модуле ты впервые собираешь настоящую обучаемую сеть
(многослойный перцептрон, MLP) так, как её пишут в реальном коде.

> Как всегда в этом курсе — только **CPU**, крошечные данные, фиксированные сиды. Идеи от железа
> не зависят: тот же класс `MLP` поедет на GPU без единого изменения логики.

## Идея 1. `nn.Module` и слои `nn.Linear` — сеть как объект, который помнит свои веса

`nn.Linear(in, out)` — это тот же линейный слой `x @ W + b`, что ты писал руками, но теперь он
**объект**: внутри уже лежат веса `W` формы `(out, in)` и смещение `b` формы `(out,)`, и они
**сразу** созданы с `requires_grad=True` (то есть готовы обучаться).

```python
import torch.nn as nn
lin = nn.Linear(3, 4)     # вход 3 признака -> выход 4 числа
x = torch.randn(2, 3)     # (N=2, D_in=3)
lin(x)                    # форма (2, 4): каждый из 2 объектов превращён в 4 числа
```

> Ловушка с формой весов. У `nn.Linear(in, out)` веса имеют форму `(out, in)`, а не `(in, out)` —
> внутри слой считает `x @ W.T + b`. Тебе это знать **не обязательно**, потому что ты никогда не
> трогаешь `W` руками: просто зовёшь `lin(x)`. Но если однажды распечатаешь `lin.weight.shape` и
> увидишь `(out, in)` — теперь не удивишься.

Чтобы построить **свою** сеть, наследуются от `nn.Module`. В `__init__` создают слои и сохраняют
их в `self`, а в `forward` описывают, как данные текут сквозь сеть. Главная магия: если слой лежит
в атрибуте `self.чтото`, `nn.Module` **автоматически замечает его параметры**, и `model.parameters()`
вернёт их все скопом.

```python
class Net(nn.Module):
    def __init__(self):
        super().__init__()                 # обязательно! иначе регистрация параметров сломается
        self.fc1 = nn.Linear(4, 8)
        self.fc2 = nn.Linear(8, 3)
    def forward(self, x):
        return self.fc2(torch.relu(self.fc1(x)))

net = Net()
sum(p.numel() for p in net.parameters())   # 4*8+8 + 8*3+3 = 40 + 27 = 67 чисел
```

**ПОЧЕМУ так удобно?** Ты больше не таскаешь `W1, b1, W2, b2` по отдельным переменным и не рискуешь
что-то забыть. Вся сеть — один объект `net`, все её обучаемые числа — один список `net.parameters()`.
Именно этот список ты отдашь оптимизатору, и он будет крутить ровно эти параметры.

**ПОЧЕМУ между слоями нужен ReLU?** Два линейных слоя подряд без нелинейности — это снова один
линейный слой (композиция линейных функций линейна), такая «сеть» не умнее обычной регрессии.
ReLU (`max(0, z)`) ломает линейность и даёт сети способность лепить сложные, изогнутые границы
между классами. ReLU ставят **между** слоями, но **не** после последнего: последний слой выдаёт
сырые числа (логиты), а превратить их в вероятности — забота функции потерь (Идея 2).

Форма данных в MLP, словами: на вход идёт батч `(N, D_in)` — N объектов по D_in признаков; после
`fc1` он становится `(N, hidden)`; ReLU форму не меняет; после `fc2` получаем `(N, out_dim)` —
по одному числу-логиту на каждый из `out_dim` классов для каждого объекта.

## Идея 2. Функция потерь и оптимизатор (`torch.optim`)

**Функция потерь** превращает «насколько сеть ошиблась» в одно число (скаляр), которое мы хотим
уменьшить. Для классификации стандарт — **перекрёстная энтропия** `nn.CrossEntropyLoss()`. Важная
особенность: она принимает **сырые логиты** (выход последнего `nn.Linear`, без softmax!) и
**целочисленные** метки классов `y` формы `(N,)` со значениями `0..C-1`. Softmax и логарифм она
делает внутри сама — это и точнее численно, и удобнее.

```python
loss_fn = nn.CrossEntropyLoss()
logits = model(X)          # (N, C) — сырые числа
loss = loss_fn(logits, y)  # y: (N,), целые метки классов -> скаляр
```

> Частая ошибка новичка: прогнать логиты через softmax и **потом** отдать в `CrossEntropyLoss`.
> Не надо — получится «двойной softmax» и сломанное обучение. Отдаём **логиты как есть**.

**Оптимизатор** — это правило, по которому параметры двигаются против градиента. Самый простой —
`SGD` (стохастический градиентный спуск): `param ← param − lr · grad`. Из модуля 09 ты помнишь и
получше — `Adam`, он часто сходится быстрее. Оптимизатору при создании отдают **список параметров**
(тот самый `model.parameters()`) и шаг обучения `lr`.

```python
import torch.optim as optim
optimizer = optim.SGD(model.parameters(), lr=0.1)   # знает, какие веса крутить
```

**ПОЧЕМУ оптимизатор и параметры связаны заранее?** Потому что при создании ты передаёшь ему
**ссылки** на тензоры-параметры сети. Дальше autograd кладёт градиент в `param.grad`, а вызов
`optimizer.step()` проходит по всем переданным параметрам и обновляет каждый по своему правилу.
Тебе не нужно ни перечислять веса вручную, ни писать формулу шага — всё спрятано за `.step()`.

## Идея 3. Цикл обучения: forward → loss → backward → step

Сердце всего Deep Learning — короткий цикл из четырёх шагов плюс одно обнуление. Один проход по нему
называется **шагом обучения** (или эпохой, если за раз скармливаешь все данные). Вот он целиком:

```python
optimizer.zero_grad()        # 1. обнулить старые градиенты (.grad копит +=, иначе сложатся!)
logits = model(X)            # 2. FORWARD: прогнать данные сквозь сеть -> логиты (N, C)
loss = loss_fn(logits, y)    # 3. LOSS: одно число — насколько плохо
loss.backward()              # 4. BACKWARD: autograd заполняет .grad у всех параметров
optimizer.step()             # 5. STEP: оптимизатор двигает каждый параметр против градиента
```

**ПОЧЕМУ именно такой порядок и зачем `zero_grad`?** Из модуля 11 ты помнишь: `.grad`
**накапливается**, а не перезаписывается. Если не обнулить его перед `backward()`, новые градиенты
сложатся с прошлыми — и шаг пойдёт не туда. Поэтому железное правило: **обнулил → forward → loss →
backward → step**. Перепутаешь порядок (например, `step` до `backward`) — сеть «обновится» по пустым
или старым градиентам и обучаться не будет.

Маленький числовой ориентир. Пусть в начале (веса случайные) на задаче с 3 классами loss около
`ln(3) ≈ 1.10` — это «чистое незнание», сеть угадывает наугад с вероятностью 1/3 на класс. По мере
шагов loss должен **падать** заметно ниже этого, а доля верных ответов (accuracy) — расти к 1.0.
Если loss стоит на месте — почти наверняка забыт `zero_grad`, перепутан порядок шагов или слишком
маленький `lr`.

```python
for epoch in range(100):     # 100 шагов на всём батче
    # ... те самые пять строк выше ...
    pass
# loss: ~1.10  ->  ~0.05   (упал)
# acc:  ~0.40  ->  ~1.00   (вырос)
```

## Идея 4. Режимы `train` / `eval`

У `nn.Module` есть два режима: `model.train()` (по умолчанию) и `model.eval()`. Переключают их
**без аргументов**, одним вызовом. Они влияют на поведение «капризных» слоёв — **Dropout** и
**BatchNorm**, — которые во время обучения ведут себя иначе, чем во время проверки (Dropout случайно
гасит нейроны при обучении и **выключается** при оценке; BatchNorm на обучении считает статистику по
батчу, а на оценке использует накопленную).

```python
model.train()    # режим обучения: dropout активен, batchnorm обновляет статистику
model.eval()     # режим оценки:   dropout выключен, batchnorm заморожен
```

**ПОЧЕМУ это важно даже когда таких слоёв нет?** В нашем простом MLP (только `Linear` + `ReLU`)
режим ничего не меняет — `Linear` и `ReLU` ведут себя одинаково. Но **привычка** правильная и
обязательная: перед измерением качества всегда `model.eval()`, перед обучением — `model.train()`.
Забыть `eval()` в сети с Dropout — классический баг: метрика «скачет» и оказывается заниженной,
потому что часть нейронов случайно отключена прямо во время замера.

Вторая половина правильной оценки — **не считать градиенты** там, где они не нужны. Оборачивай
измерение в `torch.no_grad()`: это отключает построение графа, экономит память и время и явно
говорит «я только смотрю, обучать не буду».

```python
model.eval()
with torch.no_grad():
    logits = model(X)               # граф НЕ строится — быстрее и без лишней памяти
    preds = logits.argmax(dim=1)    # класс = индекс наибольшего логита по каждой строке
```

Форма словами: `logits` имеет форму `(N, C)`; `argmax(dim=1)` берёт максимум **вдоль строки**
(по классам) и даёт `(N,)` — по одной предсказанной метке на объект. Сравниваешь с истинными `y` —
получаешь долю верных, то есть accuracy.

   Метод `forward(self, x)` принимает `x` формы `(N, in_dim)` и возвращает **логиты** формы

   (`zero_grad → forward → loss → backward → step`). Возвращает значение потерь как обычный
   `float` (через `loss.item()`) — число ДО шага, на текущих параметрах.
3. `accuracy(model, X, y) -> float` — доля верных предсказаний на `(X, y)`. Считать в режиме
   `eval()` и под `torch.no_grad()`; предсказанный класс = `argmax` логитов по `dim=1`; вернуть

### Подсказки

<details><summary>Задание 1 — каркас класса MLP</summary>
В <code>__init__</code> ПЕРВОЙ строкой вызови <code>super().__init__()</code> — без него
<code>nn.Module</code> не зарегистрирует слои и <code>parameters()</code> окажется пустым. Затем
создай два слоя и сохрани в <code>self</code>: <code>self.fc1 = nn.Linear(in_dim, hidden)</code> и
<code>self.fc2 = nn.Linear(hidden, out_dim)</code>. ReLU можно либо завести слоем
<code>self.act = nn.ReLU()</code>, либо звать функцией <code>torch.relu(...)</code> прямо в forward —
оба варианта годятся.
</details>

<details><summary>Задание 1 — что возвращает forward</summary>
Поток данных: <code>x (N, in_dim) -> fc1 -> (N, hidden) -> ReLU -> (N, hidden) -> fc2 ->
(N, out_dim)</code>. Верни результат <code>fc2</code> КАК ЕСТЬ — это логиты. Никакого softmax в
конце: <code>CrossEntropyLoss</code> применит его сам. Если добавишь softmax — обучение сломается
(двойной softmax).
</details>

<details><summary>Задание 2 — пять строк и их порядок</summary>
Строго по порядку: <code>optimizer.zero_grad()</code>, затем <code>logits = model(X)</code>, затем
<code>loss = loss_fn(logits, y)</code>, затем <code>loss.backward()</code>, затем
<code>optimizer.step()</code>. Вернуть нужно число ДО шага — посчитай <code>val = loss.item()</code>
сразу после строки с loss (или просто верни <code>loss.item()</code> в конце: <code>.item()</code>
читает значение, уже зафиксированное в loss). Главное — не забыть <code>zero_grad</code> ПЕРВЫМ.
</details>

<details><summary>Задание 3 — как посчитать долю верных</summary>
Сначала <code>model.eval()</code>, потом <code>with torch.no_grad():</code> прогони
<code>logits = model(X)</code>. Предсказание: <code>preds = logits.argmax(dim=1)</code> — форма
<code>(N,)</code>. Доля верных: <code>(preds == y).float().mean().item()</code>. Это уже число в
<code>[0, 1]</code> — ровно accuracy.
</details>

<details><summary>Задание 3 — почему no_grad и eval</summary>
<code>no_grad</code> отключает построение графа — при оценке градиенты не нужны, так быстрее и
экономнее по памяти. <code>eval()</code> в нашем MLP ничего не меняет (нет Dropout/BatchNorm), но
это правильная привычка: в больших сетях без него метрика занижается. Считаем — переводим в eval;
учим — обратно в <code>train()</code>.
</details>

Покажешь мне свой цикл обучения — проверю, в верном ли порядке стоят пять строк (это место, где
ошибаются чаще всего). Главное, что ты должен унести из модуля 12: настоящая сеть — это объект
`nn.Module`, а обучение — это всего пять строк, повторённых много раз. Дальше всё Deep Learning —
вариации на эту тему.
