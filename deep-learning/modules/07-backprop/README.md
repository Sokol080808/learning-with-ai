# Модуль 07 — Обратное распространение с нуля

> Часть II. Модели с нуля. Инструмент: **numpy** (всё руками, без torch).
> Опирается на модуль 03 (производные и цепное правило) и модуль 06 (forward pass).

**Большая идея модуля:** в модуле 06 данные текли *вперёд* — от входа к потере. Теперь мы
пойдём *назад*. Обратное распространение (backpropagation) — это не отдельный хитрый алгоритм,
а просто **цепное правило**, применённое по вычислительному графу сети, слой за слоем от потери
к параметрам. Когда ты однажды посчитаешь градиенты руками, `loss.backward()` в PyTorch
перестанет быть магией: ты будешь точно знать, что там внутри происходит. Это самый важный
модуль курса — не торопись.

---

## Идея 1. backprop = цепное правило по вычислительному графу

**Интуиция.** Сеть — это длинная цепочка операций:

```
x  →  [линейный слой 1]  →  z1  →  [relu]  →  a1  →  [линейный слой 2]  →  z2  →  [softmax]  →  p  →  [cross-entropy]  →  L
```

Каждая стрелка — простая функция. Нас интересует один вопрос: **если чуть-чуть дёрнуть
параметр (например, один вес в `W1`), насколько изменится потеря `L`?** Это и есть градиент
`dL/dW1`. Зная его, мы шагнём весами в сторону уменьшения `L` — и сеть начнёт учиться.

**ПОЧЕМУ цепное правило.** Параметр `W1` влияет на `L` не напрямую, а через цепочку:
`W1 → z1 → a1 → z2 → p → L`. Цепное правило говорит: чтобы получить полную чувствительность,
нужно **перемножить чувствительности на каждом звене**:

```
dL/dW1 = dL/dz2 · dz2/da1 · da1/dz1 · dz1/dW1
```

Бытовая аналогия: курс рубль→доллар→евро→йена. Чтобы узнать, как рубль влияет на йену,
перемножаешь курсы на каждом переходе. backprop — ровно это, только «курсы» — это локальные
производные операций.

**ПОЧЕМУ назад, а не вперёд.** Можно было бы считать чувствительности «слева направо», но тогда
для каждого параметра пришлось бы проходить граф заново. Если идти **справа налево** (от `L` к
входу), то общий «накопленный сверху» множитель `dL/dz` переиспользуется для всех параметров
этого слоя. Один обратный проход — и градиенты по *всем* параметрам готовы. Поэтому это
называют *обратным* распространением: мы распространяем сигнал ошибки от потери назад к весам.

## Идея 2. Градиенты слой за слоем (от потери к входу)

**Интуиция.** Обратный проход — это конвейер. На вход каждого слоя (если смотреть «назад»)
приходит градиент `dL/d(выход слоя)` — насколько потеря чувствительна к выходу этого слоя.
Слой делает две вещи:

1. превращает этот градиент в градиенты по **своим параметрам** (`dW`, `db`) — их мы запомним;
2. пропускает градиент **дальше назад**, к своему входу (`dL/d(вход слоя)`) — его получит
   предыдущий слой.

**Математика для линейного слоя.** Возьмём слой `Y = X @ W + b`. Формы:
`X: (N, D)`, `W: (D, M)`, `b: (M,)`, `Y: (N, M)`. Пусть сверху пришёл градиент
`dY = dL/dY` формы `(N, M)`. Тогда (это три формулы, которые стоит выучить наизусть):

```
dW = X.T @ dY          # (D, N) @ (N, M) -> (D, M)   — форма как у W
db = sum(dY, axis=0)    # (M,)                         — форма как у b
dX = dY @ W.T           # (N, M) @ (M, D) -> (N, D)   — форма как у X (пойдёт дальше назад)
```

> **Трюк проверки форм.** Градиент параметра ВСЕГДА той же формы, что сам параметр. Если у тебя
> `dW` получилась формы `(N, D)`, а `W` — `(D, M)`, ты где-то перепутал транспонирование. Форма
> — твой бесплатный детектор ошибок. Почему в `dW` появляется `X.T`? Просто из требования, чтобы
> формы сошлись: чтобы из `(N, D)` и `(N, M)` получить `(D, M)`, единственный способ — `X.T @ dY`.

**Математика для ReLU.** `a1 = relu(z1) = max(0, z1)`. Производная ReLU равна 1 там, где вход был
положителен, и 0 там, где он был ≤ 0. Поэтому градиент проходит сквозь ReLU как **маска**:

```
dz1 = da1 * (z1 > 0)    # поэлементно; (N, H) * (N, H) -> (N, H)
```

Интуиция: если нейрон был «выключен» (`z1 ≤ 0`, выход 0), то малое изменение его входа ничего не
меняет на выходе — градиент туда не течёт.

**Маленький численный пример (один линейный слой, без батча).**
`x = [1, 2]` (форма `(2,)`), `W = [[0.5], [-1.0]]` (форма `(2, 1)`), `b = [0.0]`.
Тогда `y = x@W + b = 1·0.5 + 2·(-1.0) = -1.5`. Пусть сверху пришёл `dy = [3.0]`.
Тогда `dW = x.T·dy = [[1·3], [2·3]] = [[3], [6]]`, `db = [3.0]`, `dx = dy·W.T = [3·0.5, 3·(-1)] = [1.5, -3.0]`.
Проверь это руками — почувствуй, как «сверху-вниз» множитель `3.0` раздаётся по входам.

## Идея 3. softmax + cross-entropy: красивый градиент (p − y)

**Интуиция.** По-отдельности производная softmax громоздкая (там матрица Якоби `C×C`), а
производная логарифма в cross-entropy даёт деление. Но когда softmax и cross-entropy стоят
**рядом** (а в классификации они всегда вместе), почти всё сокращается, и остаётся поразительно
простой результат.

**Математика.** Пусть `z2: (N, C)` — логиты, `p = softmax(z2)` — вероятности,
`y: (N,)` — правильные метки, а потеря — **средняя** кросс-энтропия по батчу
`L = -1/N · Σ_i log p[i, y[i]]`. Тогда градиент потери по логитам:

```
dz2 = (p − onehot(y)) / N           # форма (N, C)
```

где `onehot(y)` — матрица `(N, C)`, в которой в строке `i` стоит 1 в столбце `y[i]` и 0 в
остальных. Деление на `N` — потому что потеря *средняя* (если бы складывали, а не усредняли,
делить было бы не нужно).

**ПОЧЕМУ это так красиво.** `p − y` буквально читается как «насколько предсказание разошлось с
истиной». Где модель назначила слишком большую вероятность (`p > y`), там градиент толкает логит
вниз; где слишком маленькую правильному классу (`p < 1`), там вверх. Никаких якобианов в коде —
одно вычитание.

**Маленький численный пример.** Один объект, `C = 3`, правильный класс `y = 0`.
Пусть `p = [0.7, 0.2, 0.1]`, тогда `onehot = [1, 0, 0]`, и
`dz2 = (p − onehot)/1 = [−0.3, 0.2, 0.1]`. Сумма всегда 0 (`Σp = 1`, `Σonehot = 1`) — softmax
перераспределяет «массу», а не создаёт её. Знак минуса у правильного класса говорит: «увеличь
логит класса 0».

> Дальше этот `dz2` входит во второй линейный слой как «градиент сверху» — и поехало по формулам
> из Идеи 2: `dW2 = a1.T @ dz2`, `db2 = sum(dz2, axis=0)`, `da1 = dz2 @ W2.T`, потом сквозь ReLU,
> потом первый слой. Деление на `N` делаем ОДИН раз — здесь, в `dz2`; тогда все остальные
> градиенты автоматически становятся градиентами *средней* потери.

## Идея 4. Численная проверка градиента

**Интуиция.** Аналитический backprop легко написать с ошибкой: забыл транспонировать, перепутал
ось суммирования, потерял деление на `N`. Как убедиться, что формулы верны? Сравнить их с
**определением производной** — конечными разностями. Это «эталон», против которого мы калибруем
аналитику.

**Математика.** Производная по определению — это `(f(θ+ε) − f(θ−ε)) / (2ε)` для крошечного `ε`
(центральная разность точнее односторонней). Делаем это для **каждого** элемента параметра
по очереди: чуть-чуть подвинули один вес вверх, замерили loss; подвинули вниз, замерили loss;
поделили на `2ε` — получили численную оценку `dL/dθ_i`. Сложили все — получили численный градиент
той же формы, что параметр.

```
для каждого индекса i в θ:
    f_plus  = forward(θ с θ[i]+ε)
    f_minus = forward(θ с θ[i]−ε)
    grad[i] = (f_plus − f_minus) / (2ε)
    # ВАЖНО: верни θ[i] на место, прежде чем трогать следующий элемент!
```

**ПОЧЕМУ центральная разность, а не `(f(θ+ε) − f(θ))/ε`.** Центральная разность симметрична и
имеет ошибку порядка `ε²`, а односторонняя — порядка `ε`. С `ε ≈ 1e-5` центральная даёт совпадение
с аналитикой до ~5–6 знаков — этого хватает для `np.allclose(..., atol=1e-5)`.

**ПОЧЕМУ так нельзя обучать по-настоящему.** Численный градиент требует ДВУХ прямых проходов
*на каждый параметр*. Для сети с миллионом весов это два миллиона forward'ов на один шаг — вечность.
Поэтому численный градиент — это только **тест корректности** аналитики, не способ обучения.
Аналитический backprop считает все градиенты за один обратный проход — вот почему он незаменим.

> Этот приём (gradient checking) — стандартный инструмент. Написал новый слой — проверь его
> backward численно на крошечных данных, ДО того как удивляться, почему сеть не учится.

## Задания

Открой `backprop.py` и реализуй функции так, чтобы тесты позеленели. Формы тензоров для
двухслойной сети: `x: (N, D)`, `W1: (D, H)`, `b1: (H,)`, `W2: (H, C)`, `b2: (C,)`, `y: (N,)`
(целые метки классов `0..C-1`).

1. **`relu(z) -> np.ndarray`** — ReLU поэлементно (`max(0, z)`), форма сохраняется.

2. **`softmax(z) -> np.ndarray`** — softmax по строкам (`axis=-1`). Численно устойчиво:
   сначала вычти из каждой строки её максимум, потом `exp`, потом раздели на сумму по строке.
   В каждой строке выхода сумма равна 1.

3. **`cross_entropy(p, y) -> float`** — средняя кросс-энтропия:
   `-1/N · Σ_i log p[i, y[i]]`. Лог берём только у вероятности *правильного* класса.

4. **`forward2(x, W1, b1, W2, b2, y) -> float`** — прямой проход
   `z1=x@W1+b1 → a1=relu(z1) → z2=a1@W2+b2 → p=softmax(z2)` и возврат `cross_entropy(p, y)`.
   Эту функцию численная проверка дёргает много раз при чуть изменённых параметрах.

5. **`backward2(x, y, W1, b1, W2, b2) -> dict`** — *сердце модуля*. Верни словарь
   `{"dW1", "db1", "dW2", "db2"}` с градиентами потери по параметрам (формы как у параметров).
   Ключевые шаги: `dz2 = (p − onehot(y))/N` → второй слой (`dW2, db2, da1`) → сквозь ReLU
   (`dz1 = da1 * (z1>0)`) → первый слой (`dW1, db1`).

6. **`numerical_gradient(f, param, eps=1e-5) -> np.ndarray`** — численный градиент скалярной
   функции `f` (без аргументов, возвращает loss при текущем `param`) по массиву `param`,
   центральной разностью. Обязательно восстанавливай каждый элемент после замера.

Главный тест модуля сравнивает аналитические градиенты из `backward2` с численными
(`numerical_gradient` по каждому параметру, через `forward2`) на маленьких seeded данных. Если
они совпали с точностью `atol=1e-5` — твой backprop корректен.

### Подсказки

<details>
<summary>Подсказка 1 — с чего начать (мягкий намёк)</summary>

Сначала сделай три кирпичика и `forward2` — без них ни backprop, ни численную проверку не
проверить. `relu` — это `np.maximum(0.0, z)`. В `softmax` обязательно вычти `z.max(axis=-1,
keepdims=True)` перед `np.exp`, иначе большие логиты дадут `inf`. В `cross_entropy` достань
вероятности правильных классов индексацией `p[np.arange(N), y]`.
</details>

<details>
<summary>Подсказка 2 — порядок обратного прохода в backward2</summary>

Сначала сделай прямой проход и СОХРАНИ промежуточные `z1`, `a1`, `p` — они нужны назад. Потом
иди строго в обратном порядке графа:

```
dz2 = (p - onehot(y)) / N      # (N, C)
dW2 = a1.T @ dz2               # (H, C)
db2 = dz2.sum(axis=0)          # (C,)
da1 = dz2 @ W2.T               # (N, H)
dz1 = da1 * (z1 > 0)           # сквозь ReLU, (N, H)
dW1 = x.T @ dz1                # (D, H)
db1 = dz1.sum(axis=0)          # (H,)
```

Деление на `N` — только в `dz2`. Маска `(z1 > 0)` — это булев массив, при умножении он
работает как 1/0.
</details>

<details>
<summary>Подсказка 3 — onehot и проверка форм</summary>

`onehot` без циклов:

```python
oh = np.zeros((N, C))
oh[np.arange(N), y] = 1.0
```

Проверяй формы на каждом шаге. Если `dW2` получилась не `(H, C)`, а, скажем, `(C, H)` —
ты перепутал, что транспонировать. Помни правило: **градиент параметра имеет форму самого
параметра**. Под это правило подгоняй транспонирования и порядок умножения.
</details>

<details>
<summary>Подсказка 4 — numerical_gradient без боли</summary>

Удобно обходить все элементы любой формы через `np.nditer`:

```python
grad = np.zeros_like(param)
it = np.nditer(param, flags=["multi_index"], op_flags=["readwrite"])
while not it.finished:
    i = it.multi_index
    old = param[i]
    param[i] = old + eps; f_plus  = f()
    param[i] = old - eps; f_minus = f()
    grad[i] = (f_plus - f_minus) / (2 * eps)
    param[i] = old            # восстановить!
    it.iternext()
return grad
```

Если забыть восстановить `param[i]`, ты испортишь параметры и следующие замеры поедут.
А `f` — это замыкание вроде `lambda: forward2(x, W1, b1, W2, b2, y)`: оно читает тот же
объект-массив, который ты меняешь на месте.
</details>
