# Модуль 08 — Свой micrograd: скалярный autograd

Большая идея модуля: в модуле 07 ты считал градиенты **руками** — выводил формулу для
каждого слоя и аккуратно перемножал их цепным правилом. Это работает, но не масштабируется:
для каждой новой архитектуры пришлось бы заново выводить backprop. Сейчас мы сделаем шаг,
который превратил DL из ремесла в инженерию — **автодифференцирование**. Идея почти
наглая в своей простоте: пусть каждое число **само помнит**, из каких чисел и какой
операцией оно получилось. Тогда программа сможет «развернуть» эту память назад и посчитать
все производные **автоматически**, без единой выведенной вручную формулы.

Мы построим `micrograd` — крошечный движок (буквально один класс `Value`), который делает
ровно то, что внутри `loss.backward()` у PyTorch. После этого модуля слово «autograd»
перестанет быть магией: ты будешь знать, что там просто граф и цепное правило.

> Мы работаем со **скалярами** (одиночными числами), а не с тензорами — нарочно. Так видно
> сам механизм, без шума от форм и broadcasting. Тензорный autograd (модуль 11, PyTorch) —
> это та же идея, только узел хранит массив `(N, D)`, а не один float.

---

## Идея 1. Вычислительный граф из узлов

Любое выражение — это **граф**. Возьмём `d = a*b + c`. Чтобы его посчитать, нужно сначала
перемножить `a` и `b`, получить промежуточное `u = a*b`, а потом сложить `u + c`. Эти
шаги — узлы графа, а стрелки — «кто из кого получился»:

```
   a ──┐
       (*) ── u ──┐
   b ──┘          (+) ── d
              c ──┘
```

**ПОЧЕМУ это важно.** Forward pass (вычисление значения) идёт по графу **слева направо**:
от входов к результату. Backprop пойдёт **справа налево**: от результата к входам. Граф —
это и есть «карта», по которой backprop знает, в каком порядке разворачивать цепное правило.
Без записи графа мы бы не знали, какие операции и в каком порядке применялись.

Форма данных тут проще некуда: **каждый узел — это один скаляр** (`float`). Никаких
`(N, D)` — только одно число на узел. Зато узлов в графе будет много, и связи между ними —
самое интересное.

## Идея 2. Узел `Value` хранит `.data` и `.grad`

Весь движок — это один класс `Value`. У каждого узла два главных поля:

- **`.data`** — само число (результат этого узла в forward pass);
- **`.grad`** — производная итогового результата по этому узлу: `d(out)/d(self)`. Сколько
  изменится финальный результат, если этот узел чуть-чуть подтолкнуть. До backprop это `0.0`.

```
a = Value(2.0)   →  a.data = 2.0,  a.grad = 0.0
b = Value(-3.0)  →  b.data = -3.0, b.grad = 0.0
```

Плюс два служебных поля, которые делают всю «магию»:

- **`._prev`** — множество узлов-**родителей** (из кого получился этот узел). У листьев
  (`a`, `b`, `c`) оно пустое; у `u = a*b` это `{a, b}`.
- **`._backward`** — маленькая функция «протолкни мой градиент родителям на один шаг».
  Её замыкание знает локальную производную именно этой операции.

**ПОЧЕМУ две вещи — `data` и `grad` — в одном объекте?** Потому что backprop связывает их:
forward заполняет `.data`, backward заполняет `.grad`, и оба прохода ходят по одному и тому
же графу. Это в точности torch-тензор в миниатюре: у него тоже есть `.data` и `.grad`.

## Идея 3. Локальная производная на каждой операции

Главный трюк автограда: **каждая операция знает только свою маленькую (локальную)
производную** — и больше ничего. Ей не нужно знать про весь граф. Цепное правило соберёт
из этих кусочков общую производную.

Когда мы делаем `out = self * other`, мы тут же привязываем к `out` функцию `_backward`,
которая умеет протолкнуть градиент назад на **один шаг**. Вот локальные производные тех
четырёх операций, что мы реализуем:

| Операция | Значение `out.data` | Локальные производные |
|---|---|---|
| `out = a + b` | `a + b` | `d(out)/da = 1`, `d(out)/db = 1` |
| `out = a * b` | `a * b` | `d(out)/da = b`, `d(out)/db = a` |
| `out = tanh(a)` | `tanh(a)` | `d(out)/da = 1 - tanh(a)^2` |
| `out = relu(a)` | `max(0, a)` | `d(out)/da = 1` если `a > 0`, иначе `0` |

Цепное правило говорит: вклад в `.grad` родителя = (локальная производная) × (`out.grad`).
Поэтому `_backward` всегда выглядит так:

```python
# для out = a * b
def _backward():
    a.grad += b.data * out.grad   # d(out)/da = b,  умножаем на пришедший сверху out.grad
    b.grad += a.data * out.grad   # d(out)/db = a
out._backward = _backward
```

**ПОЧЕМУ `+=`, а не `=`?** Потому что один и тот же узел может участвовать в нескольких
ветвях (например, `x*x` или `a + a` — `x` тут используется дважды). Цепное правило для таких
случаев требует **сложить** вклады со всех ветвей. Если писать `=`, второй вклад затрёт
первый — и градиент будет неверным. (Именно эту ловушку проверяет один из тестов.)

**Маленький численный пример** (`tanh`). Пусть `s = 4.0`. Тогда `tanh(4.0) ≈ 0.99933`, и
локальная производная `1 - 0.99933^2 ≈ 0.00134`. Видишь, какая она крошечная? Это значит:
если на вход `tanh` уже приходит большое число, сигнал градиента почти затухает. Так,
руками, на одном скаляре, ты впервые **видишь** проблему затухающих градиентов, о которой
все говорят применительно к глубоким сетям.

## Идея 4. `backward()` — топологический проход в обратном порядке

Теперь собираем всё вместе. Метод `backward()` (вызываем его обычно от `loss`) должен
заполнить `.grad` у **всех** узлов графа. Алгоритм — три шага:

1. **Топологическая сортировка.** Обойти граф в глубину (DFS) от текущего узла и выписать
   узлы так, чтобы каждый узел шёл **после всех своих родителей**. Получаем список:
   входы → промежуточные → результат.
2. **Завести градиент.** Поставить `self.grad = 1.0` — производная результата по самому
   себе равна 1 (если толкнуть результат на ε, он изменится ровно на ε).
3. **Идти в ОБРАТНОМ порядке** по списку и в каждом узле вызывать `node._backward()`.

```python
def backward(self):
    topo = []
    visited = set()
    def build(v):
        if v not in visited:
            visited.add(v)
            for parent in v._prev:
                build(parent)
            topo.append(v)        # узел добавляется ПОСЛЕ родителей
    build(self)
    self.grad = 1.0
    for node in reversed(topo):   # справа налево!
        node._backward()
```

**ПОЧЕМУ именно обратный топологический порядок?** Когда мы обрабатываем узел, его `.grad`
должен быть уже **полностью** накоплен — иначе он протолкнёт родителям недосчитанное
значение. Топологический порядок гарантирует: к моменту, когда мы дошли до узла, все его
«потомки» (те, кто получился из него) уже отработали и сложили свои вклады в его `.grad`.
Идти приходится именно в обратном порядке: результат → входы.

**Численный пример до конца.** Возьмём `out = ((a*b)+c).tanh()` при `a=2, b=-3, c=10`:

```
u = a*b = -6
s = u + c = 4
out = tanh(4) ≈ 0.99933
g = 1 - tanh(4)^2 ≈ 0.00134            # локальная производная tanh в точке s

backward():
  out.grad = 1.0
  tanh:  s.grad += g * out.grad        → s.grad = g ≈ 0.00134
  (+):   u.grad += 1 * s.grad,  c.grad += 1 * s.grad   → u.grad = c.grad = g
  (*):   a.grad += b.data * u.grad = -3 * g ≈ -0.00402
         b.grad += a.data * u.grad =  2 * g ≈  0.00268
```

То есть `a.grad = g·b`, `b.grad = g·a`, `c.grad = g`. Сверишь это с тестом — там ровно эти
формулы (и плюс независимая численная проверка центральной разностью).

## Задания

Открой `micrograd.py`. Тела помечены `raise NotImplementedError` — это писать тебе. Реализуй
класс `Value`:

1. **`__init__(self, data, _children=(), _op="")`** — сохрани `self.data = float(data)`,
   `self.grad = 0.0`, `self._prev = set(_children)`, `self._op = _op` и
   `self._backward = lambda: None` (у листа графа нечего проталкивать).

2. **`__add__(self, other) -> Value`** — `out = self + other`. Если `other` — число, заверни
   в `Value`. Привяжи `_backward`, проталкивающий `out.grad` обоим слагаемым (`+= 1 * out.grad`).

3. **`__mul__(self, other) -> Value`** — `out = self * other`. Локальные производные:
   `self.grad += other.data * out.grad`, `other.grad += self.data * out.grad`.

4. **`tanh(self) -> Value`** — `out = tanh(self)`. Производная `1 - tanh(self)^2`.

5. **`relu(self) -> Value`** — `out = max(0, self)`. Производная `1` при `out.data > 0`, иначе `0`.

6. **`backward(self) -> None`** — топологическая сортировка от `self`, затем `self.grad = 1.0`
   и проход в обратном порядке с вызовом `node._backward()`. Метод ничего не возвращает —
   только заполняет `.grad` на месте.

Обёртки `__neg__`, `__sub__`, `__radd__`, `__rmul__`, `__rsub__`, `__repr__` уже написаны
через `__add__`/`__mul__` — трогать их не нужно. Поэтому в тестах работают выражения вида
`2 * a`, `a - 1`, `5 - a`, `-x`.

### Подсказки

<details>
<summary>1. С чего начать?</summary>

Сделай сначала `__init__`, `__add__`, `__mul__` — этого хватит, чтобы прошли тесты на
forward (`test_forward_values`, `test_scalars_promoted`). Только потом берись за `backward()`.
Пиши и проверяй маленькими шагами.
</details>

<details>
<summary>2. Как привязать `_backward` к операции?</summary>

Создай `out`, объяви **внутреннюю** функцию-замыкание и присвой её `out._backward`:

```python
def __add__(self, other):
    other = other if isinstance(other, Value) else Value(other)
    out = Value(self.data + other.data, (self, other), '+')
    def _backward():
        self.grad  += out.grad
        other.grad += out.grad
    out._backward = _backward
    return out
```

Замыкание «помнит» `self`, `other` и `out` — поэтому позже, при обратном проходе, оно знает,
куда добавлять градиент. Локальная производная сложения по обоим аргументам равна 1.
</details>

<details>
<summary>3. `tanh` и `relu`</summary>

Для `tanh` посчитай `t = math.tanh(self.data)` **один раз** и используй его и в `out.data`,
и в производной `(1 - t**2)`:

```python
def tanh(self):
    t = math.tanh(self.data)
    out = Value(t, (self,), 'tanh')
    def _backward():
        self.grad += (1 - t**2) * out.grad
    out._backward = _backward
    return out
```

Для `relu` множитель удобно брать через `out.data`: `1.0 if out.data > 0 else 0.0`.
</details>

<details>
<summary>4. Топологическая сортировка в `backward()`</summary>

Классический DFS с множеством `visited`. Узел добавляется в список **после** рекурсии по
всем родителям `v._prev` — тогда родители оказываются раньше в списке:

```python
topo, visited = [], set()
def build(v):
    if v not in visited:
        visited.add(v)
        for parent in v._prev:
            build(parent)
        topo.append(v)
build(self)
```

Затем `self.grad = 1.0` и `for node in reversed(topo): node._backward()`. Обязательно
`reversed` — идём от результата к входам.
</details>

<details>
<summary>5. Почему тест с <code>x*x</code> ловит ошибку?</summary>

Если внутри `_backward` ты написал `self.grad = ...` вместо `self.grad += ...`, то для
`y = x*x` второй вклад затрёт первый, и вместо `2x` получится `x`. Аккумулируй через `+=` —
и `test_grad_accumulates_when_node_reused` станет зелёным.
</details>

<details>
<summary>6. Почему в цикле обучения мы обнуляем <code>.grad</code>?</summary>

`backward()` **прибавляет** к `.grad`. Если не обнулить перед следующим шагом, градиенты
с прошлых итераций накопятся и спуск поедет не туда. Поэтому в `test_training_reduces_loss`
перед каждым `backward()` стоит `w.grad = 0.0`. Ровно это в PyTorch делает `optimizer.zero_grad()` —
теперь ты знаешь, зачем.
</details>
