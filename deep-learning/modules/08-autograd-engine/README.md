# Модуль 08 — Свой micrograd: скалярный autograd

Большая идея модуля: в модуле 07 ты считал градиенты **руками** — выводил формулу для
каждого слоя и аккуратно перемножал их цепным правилом. Это работает, но не масштабируется:
для каждой новой архитектуры пришлось бы заново выводить backprop. Сейчас мы сделаем шаг,
который превратил DL из ремесла в инженерию — **автодифференцирование**. Идея почти
наглая в своей простоте: пусть каждое число **само помнит**, из каких чисел и какой
операцией оно получилось. Тогда программа сможет «развернуть» эту память назад и посчитать
все производные **автоматически**, без единой выведенной вручную формулы.

Мы построим `micrograd` — крошечный движок (буквально один класс `Value`), который делает
ровно то, что внутри `loss.backward()` у PyTorch. После этого модуля слово «autograd»
перестанет быть магией: ты будешь знать, что там просто граф и цепное правило.

> Мы работаем со **скалярами** (одиночными числами), а не с тензорами — нарочно. Так видно
> сам механизм, без шума от форм и broadcasting. Тензорный autograd (модуль 11, PyTorch) —
> это та же идея, только узел хранит массив `(N, D)`, а не один float.

---

## Идея 1. Вычислительный граф из узлов

Любое выражение — это **граф**. Возьмём `d = a*b + c`. Чтобы его посчитать, нужно сначала
перемножить `a` и `b`, получить промежуточное `u = a*b`, а потом сложить `u + c`. Эти
шаги — узлы графа, а стрелки — «кто из кого получился»:

```
   a ──┐
       (*) ── u ──┐
   b ──┘          (+) ── d
              c ──┘
```

**ПОЧЕМУ это важно.** Forward pass (вычисление значения) идёт по графу **слева направо**:
от входов к результату. Backprop пойдёт **справа налево**: от результата к входам. Граф —
это и есть «карта», по которой backprop знает, в каком порядке разворачивать цепное правило.
Без записи графа мы бы не знали, какие операции и в каком порядке применялись.

Форма данных тут проще некуда: **каждый узел — это один скаляр** (`float`). Никаких
`(N, D)` — только одно число на узел. Зато узлов в графе будет много, и связи между ними —
самое интересное.

## Идея 2. Узел `Value` хранит `.data` и `.grad`

Весь движок — это один класс `Value`. У каждого узла два главных поля:

- **`.data`** — само число (результат этого узла в forward pass);
- **`.grad`** — производная итогового результата по этому узлу: `d(out)/d(self)`. Сколько
  изменится финальный результат, если этот узел чуть-чуть подтолкнуть. До backprop это `0.0`.

```
a = Value(2.0)   →  a.data = 2.0,  a.grad = 0.0
b = Value(-3.0)  →  b.data = -3.0, b.grad = 0.0
```

Плюс **три** служебных поля, которые делают всю «магию»:

- **`._prev`** — множество узлов-**родителей** (из кого получился этот узел). У листьев
  (`a`, `b`, `c`) оно пустое; у `u = a*b` это `{a, b}`.
- **`._backward`** — маленькая функция «протолкни мой градиент родителям на один шаг».
  Её замыкание знает локальную производную именно этой операции.
- **`._op`** — короткая строка-метка операции (`'+'`, `'*'`, `'tanh'`, `'relu'`; у листа
  пустая `''`). На сам счёт она не влияет — её роль чисто **bookkeeping**.

**ПОЧЕМУ две вещи — `data` и `grad` — в одном объекте?** Потому что backprop связывает их:
forward заполняет `.data`, backward заполняет `.grad`, и оба прохода ходят по одному и тому
же графу. Это в точности torch-тензор в миниатюре: у него тоже есть `.data` и `.grad`.

**Зачем нужен `._op`?** Это не только «для отладки в repr». Вместе с `._prev` метка `._op`
полностью описывает узел: *какое число* (`data`), *из кого* получено (`_prev`) и *какой
операцией* (`_op`). Этого достаточно, чтобы **нарисовать весь граф вычислений** — например,
обойти его в глубину и распечатать дерево или собрать Graphviz-картинку (ровно это делает
Карпатий в своём ноутбуке, рисуя круглые узлы-данные и прямоугольные узлы-операции). Когда
тебе понадобится *увидеть*, как именно собралось выражение и куда потекли градиенты, ты
поймёшь, что вся нужная информация уже лежит в `data` / `grad` / `_prev` / `_op` — больше
ничего хранить не надо. Мини-набросок такого обхода:

```python
def trace(root):                # вернуть множества узлов и рёбер графа
    nodes, edges = set(), set()
    def build(v):
        if v not in nodes:
            nodes.add(v)
            for parent in v._prev:
                edges.add((parent, v))   # ребро parent → v, подпись = v._op
                build(parent)
    build(root)
    return nodes, edges
```

То есть `Value` — это законченный объект: `data` (forward), `grad` (backward),
`_prev` + `_op` (структура графа), `_backward` (как протолкнуть градиент). Четыре поля — и
весь autograd.

## Идея 3. Локальная производная на каждой операции

Главный трюк автограда: **каждая операция знает только свою маленькую (локальную)
производную** — и больше ничего. Ей не нужно знать про весь граф. Цепное правило соберёт
из этих кусочков общую производную.

Когда мы делаем `out = self * other`, мы тут же привязываем к `out` функцию `_backward`,
которая умеет протолкнуть градиент назад на **один шаг**. Вот локальные производные тех
четырёх операций, что мы реализуем:

| Операция | Значение `out.data` | Локальные производные |
|---|---|---|
| `out = a + b` | `a + b` | `d(out)/da = 1`, `d(out)/db = 1` |
| `out = a * b` | `a * b` | `d(out)/da = b`, `d(out)/db = a` |
| `out = tanh(a)` | `tanh(a)` | `d(out)/da = 1 - tanh(a)^2` |
| `out = relu(a)` | `max(0, a)` | `d(out)/da = 1` если `a > 0`, иначе `0` |

Цепное правило говорит: вклад в `.grad` родителя = (локальная производная) × (`out.grad`).
Поэтому `_backward` всегда выглядит так:

```python
# для out = a * b
def _backward():
    a.grad += b.data * out.grad   # d(out)/da = b,  умножаем на пришедший сверху out.grad
    b.grad += a.data * out.grad   # d(out)/db = a
out._backward = _backward
```

**ПОЧЕМУ `+=`, а не `=`?** Потому что один и тот же узел может участвовать в нескольких
ветвях (например, `x*x` или `a + a` — `x` тут используется дважды). Цепное правило для таких
случаев требует **сложить** вклады со всех ветвей. Если писать `=`, второй вклад затрёт
первый — и градиент будет неверным. (Именно эту ловушку проверяет один из тестов.)

**Маленький численный пример** (`tanh`). Пусть `s = 4.0`. Тогда `tanh(4.0) ≈ 0.99933`, и
локальная производная `1 - 0.99933^2 ≈ 0.00134`. Видишь, какая она крошечная? Это значит:
если на вход `tanh` уже приходит большое число, сигнал градиента почти затухает. Так,
руками, на одном скаляре, ты впервые **видишь** проблему затухающих градиентов, о которой
все говорят применительно к глубоким сетям.

## Идея 4. `backward()` — топологический проход в обратном порядке

Теперь собираем всё вместе. Метод `backward()` (вызываем его обычно от `loss`) должен
заполнить `.grad` у **всех** узлов графа. Алгоритм — три шага:

1. **Топологическая сортировка.** Обойти граф в глубину (DFS) от текущего узла и выписать
   узлы так, чтобы каждый узел шёл **после всех своих родителей**. Получаем список:
   входы → промежуточные → результат.
2. **Завести градиент.** Поставить `self.grad = 1.0` — производная результата по самому
   себе равна 1 (если толкнуть результат на ε, он изменится ровно на ε).
3. **Идти в ОБРАТНОМ порядке** по списку и в каждом узле вызывать `node._backward()`.

```python
def backward(self):
    topo = []
    visited = set()
    def build(v):
        if v not in visited:
            visited.add(v)
            for parent in v._prev:
                build(parent)
            topo.append(v)        # узел добавляется ПОСЛЕ родителей
    build(self)
    self.grad = 1.0
    for node in reversed(topo):   # справа налево!
        node._backward()
```

**ПОЧЕМУ именно обратный топологический порядок?** Когда мы обрабатываем узел, его `.grad`
должен быть уже **полностью** накоплен — иначе он протолкнёт родителям недосчитанное
значение. Топологический порядок гарантирует: к моменту, когда мы дошли до узла, все его
«потомки» (те, кто получился из него) уже отработали и сложили свои вклады в его `.grad`.
Идти приходится именно в обратном порядке: результат → входы.

**Численный пример до конца.** Возьмём `out = ((a*b)+c).tanh()` при `a=2, b=-3, c=10`:

```
u = a*b = -6
s = u + c = 4
out = tanh(4) ≈ 0.99933
g = 1 - tanh(4)^2 ≈ 0.00134            # локальная производная tanh в точке s

backward():
  out.grad = 1.0
  tanh:  s.grad += g * out.grad        → s.grad = g ≈ 0.00134
  (+):   u.grad += 1 * s.grad,  c.grad += 1 * s.grad   → u.grad = c.grad = g
  (*):   a.grad += b.data * u.grad = -3 * g ≈ -0.00402
         b.grad += a.data * u.grad =  2 * g ≈  0.00268
```

То есть `a.grad = g·b`, `b.grad = g·a`, `c.grad = g`. Сверишь это с тестом — там ровно эти
формулы (и плюс независимая численная проверка центральной разностью).

## Идея 5. `.grad` надо ОБНУЛЯТЬ перед каждым шагом (`zero_grad`)

Эта идея на одном уровне с четырьмя предыдущими — и в реальном коде на PyTorch именно она
чаще всего «кусает». Вспомни Идею 3: `_backward` делает `+=`, то есть backprop **прибавляет**
к `.grad`, а не присваивает. Это правильно *внутри одного* обратного прохода (вклады с разных
ветвей суммируются). Но это значит, что у `.grad` **нет «сброса»** между разными вызовами
`backward()`.

**Что ломается.** Цикл обучения — это «forward → backward → шаг → повторить». Если на втором
шаге снова позвать `backward()`, не обнулив `.grad`, новые градиенты **лягут поверх старых**:

```
шаг 1:  grad = 0      → backward → grad = g1            → шаг по g1
шаг 2:  grad = g1     → backward → grad = g1 + g2  (!)  → шаг по (g1+g2), а нужно по g2
шаг 3:  grad = g1+g2  → backward → grad = g1+g2+g3 (!!) → спуск едет не туда
```

Градиент раздувается от шага к шагу, апдейт становится всё больше и не в ту сторону —
обучение расходится или ползёт криво. Поэтому **перед каждым** `backward()` нужно поставить
всем обучаемым параметрам `.grad = 0.0`:

```python
for step in range(n_steps):
    loss = forward()
    for p in params:        # <-- ручной zero_grad: ОБНУЛЯЕМ ПЕРЕД backward
        p.grad = 0.0
    loss.backward()         # backward делает += поверх чистого нуля
    for p in params:
        p.data -= lr * p.grad
```

**Связь с PyTorch.** Ровно это делает `optimizer.zero_grad()` (или `model.zero_grad()`).
Вечный вопрос новичка — «почему torch не обнуляет градиенты сам?» — теперь имеет ответ из
первых принципов: потому что `+=` нарочно оставлен в `backward`, чтобы суммировать вклады
повторно используемых узлов (а ещё — чтобы можно было накапливать градиент по нескольким
мини-батчам перед одним шагом, *gradient accumulation*). Сброс — отдельное, осознанное
действие. Забыл `zero_grad()` — получил тихо неправильное обучение без единой ошибки в
консоли. В Задании 7 ты сделаешь свой `zero_grad()` на `Module`, и тест специально проверит,
что без него градиенты удваиваются.

   `self._backward = lambda: None` (у листа графа нечего проталкивать).

6. **`backward(self) -> None`** — топологическая сортировка от `self`, затем `self.grad = 1.0`

Обёртки `__neg__`, `__sub__`, `__radd__`, `__rmul__`, `__rsub__`, `__repr__` уже написаны
через `__add__`/`__mul__` — трогать их не нужно. Поэтому в тестах работают выражения вида

### Задание 7 (★) — сеть поверх `Value`: `Neuron → Layer → MLP`

Движок готов. Теперь — **вершина** всей истории про micrograd: собрать из `Value`
настоящую нейросеть и обучить её. Важно понять: **никакого нового автограда не нужно**.
`Neuron`, `Layer`, `MLP` — это просто **удобные контейнеры** из уже знакомых `Value`-узлов.
Forward строит граф, `loss.backward()` заполняет `.grad` у *всех* весов, и тот же цикл
«обнули `.grad` → backward → шаг по `-grad`», что и для одного нейрона, обучает многослойную
сеть. Это ровно тот мост от «сырого автограда» к «я умею обучать сеть», который ведёт к
модулю 12 (MLP на PyTorch) — там будут те же `Neuron`/`Linear`/`forward`, просто на тензорах.

Реализуй в `micrograd.py` три класса (плюс крошечный общий родитель `Module`):

7a. **`Module`** — родитель с `parameters() -> list[Value]` (по умолчанию `[]`) и
    `zero_grad()`, который ставит `p.grad = 0.0` каждому параметру. Это твой ручной
    `optimizer.zero_grad()` из Идеи 5.

7b. **`Neuron(n_in, nonlin='tanh')`** — один нейрон. Хранит список весов `w` (по одному
    `Value` на вход) и смещение `b` (`Value`). `__call__(x)` берёт список входов и считает
    `act = b + Σ w_i * x_i`, затем применяет активацию (`'tanh'`, `'relu'` или `'linear'` —
    без неё) и возвращает один `Value`. `parameters()` → `w + [b]`.

7c. **`Layer(n_in, n_out, nonlin='tanh')`** — список из `n_out` независимых `Neuron`, каждый
    видит все `n_in` входов. `__call__(x)` возвращает **список** из `n_out` `Value`.
    `parameters()` — все параметры всех нейронов, сплющенные в один список.

7d. **`MLP(layer_sizes, nonlin='tanh')`** — цепочка слоёв. `layer_sizes = [n_in, h1, …, n_out]`.
    Скрытые слои получают активацию `nonlin`, **последний** слой делается линейным (`'linear'`),
    чтобы выход не зажимался в `(-1, 1)`. `__call__(x)` прогоняет вход через все слои; если на
    выходе ровно один `Value`, верни его одного (а не список из одного).

Чтобы тесты были детерминированными, конструкторы принимают необязательный
`rng: random.Random` и берут стартовые веса из него (`rng.uniform(-1, 1)`); смещение — `0.0`.

Что проверяют тесты (`test_micrograd_props.py`, раздел 16):
- **forward-оракул**: выход `MLP([3,4,4,1])` совпадает с независимым пересчётом тех же весов
  в PyTorch до `1e-6`;
- **grad-check**: на `MLP(2,3,1)` градиент *каждого* веса после `backward()` сходится с
  центральной разностью (`atol=1e-5`) — то есть автоград реально дотягивается до всех весов;
- **обучение**: 50 шагов SGD на игрушечном XOR роняют loss как минимум вдвое, и loss
  **монотонно** падает на трёх контрольных точках (это ловит забытый `zero_grad` и ошибку
  знака в апдейте);
- **структура и `zero_grad`**: число параметров, длины выходов слоя, и что без `zero_grad`
  градиенты удваиваются, а после — обнуляются в ноль.

XOR выбран нарочно: один нейрон его **не** разделит (он не линейно разделим) — нужен скрытый
слой. Так ты на своём движке увидишь, *зачем* вообще нужна глубина.

### Подсказки

<details>
<summary>1. С чего начать?</summary>

Сделай сначала `__init__`, `__add__`, `__mul__` — этого хватит, чтобы прошли тесты на
forward (`test_forward_values`, `test_scalars_promoted`). Только потом берись за `backward()`.
Пиши и проверяй маленькими шагами.
</details>

<details>
<summary>2. Как привязать `_backward` к операции?</summary>

Создай `out`, объяви **внутреннюю** функцию-замыкание и присвой её `out._backward`:

```python
def __add__(self, other):
    other = other if isinstance(other, Value) else Value(other)
    out = Value(self.data + other.data, (self, other), '+')
    def _backward():
        self.grad  += out.grad
        other.grad += out.grad
    out._backward = _backward
    return out
```

Замыкание «помнит» `self`, `other` и `out` — поэтому позже, при обратном проходе, оно знает,
куда добавлять градиент. Локальная производная сложения по обоим аргументам равна 1.
</details>

<details>
<summary>3. `tanh` и `relu`</summary>

Для `tanh` посчитай `t = math.tanh(self.data)` **один раз** и используй его и в `out.data`,
и в производной `(1 - t**2)`:

```python
def tanh(self):
    t = math.tanh(self.data)
    out = Value(t, (self,), 'tanh')
    def _backward():
        self.grad += (1 - t**2) * out.grad
    out._backward = _backward
    return out
```

Для `relu` множитель удобно брать через `out.data`: `1.0 if out.data > 0 else 0.0`.
</details>

<details>
<summary>4. Топологическая сортировка в `backward()`</summary>

Классический DFS с множеством `visited`. Узел добавляется в список **после** рекурсии по
всем родителям `v._prev` — тогда родители оказываются раньше в списке:

```python
topo, visited = [], set()
def build(v):
    if v not in visited:
        visited.add(v)
        for parent in v._prev:
            build(parent)
        topo.append(v)
build(self)
```

Затем `self.grad = 1.0` и `for node in reversed(topo): node._backward()`. Обязательно
`reversed` — идём от результата к входам.
</details>

<details>
<summary>5. Почему тест с <code>x*x</code> ловит ошибку?</summary>

Если внутри `_backward` ты написал `self.grad = ...` вместо `self.grad += ...`, то для
`y = x*x` второй вклад затрёт первый, и вместо `2x` получится `x`. Аккумулируй через `+=` —
и `test_grad_accumulates_when_node_reused` станет зелёным.
</details>

<details>
<summary>6. Почему в цикле обучения мы обнуляем <code>.grad</code>?</summary>

`backward()` **прибавляет** к `.grad`. Если не обнулить перед следующим шагом, градиенты
с прошлых итераций накопятся и спуск поедет не туда. Поэтому в `test_training_reduces_loss`
перед каждым `backward()` стоит `w.grad = 0.0`. Ровно это в PyTorch делает `optimizer.zero_grad()` —
теперь ты знаешь, зачем.
</details>

#### Подсказки к Заданию 7 (`Neuron`/`Layer`/`MLP`)

<details>
<summary>7. С чего начать сеть?</summary>

Снизу вверх. Сначала `Neuron` — самый маленький кирпич. Когда `Neuron.__call__` работает и
`parameters()` возвращает правильный список, `Layer` это просто «список нейронов», а `MLP` —
«список слоёв». Не пиши всё сразу: добейся, чтобы `test_neuron_and_layer_shapes_and_params`
позеленел (он проверяет только структуру, без градиентов), и только потом берись за обучение.
</details>

<details>
<summary>8. Как сложить взвешенную сумму из <code>Value</code>?</summary>

Тебе нужно `act = b + Σ w_i * x_i`. Заведи аккумулятор, стартующий с `b`, и в цикле прибавляй
произведения. Каждое `+` и `*` — это уже перегруженные операции `Value`, так что **граф
строится сам**, тебе достаточно обычной арифметики Python:

```python
act = self.b
for wi, xi in zip(self.w, x):
    act = act + wi * xi
```

`x` может быть списком чисел *или* `Value` — `__mul__` сам завернёт число. После суммы
примени активацию: `act.tanh()`, `act.relu()` или верни `act` как есть для `'linear'`.
(Это не полное решение `Neuron` — инициализацию весов и `parameters()` пишешь сам.)
</details>

<details>
<summary>9. Как собрать <code>parameters()</code> по всей сети?</summary>

У `Neuron` это `self.w + [self.b]`. У `Layer` и `MLP` — «сплющить» параметры детей в один
плоский список. Удобно вложенным генератором:

```python
# Layer:
return [p for n in self.neurons for p in n.parameters()]
```

Зачем плоский список? Потому что цикл обучения и `zero_grad()` хотят просто пробежать по
*всем* обучаемым `Value` без оглядки на структуру. Сделай `zero_grad()` один раз на `Module`
(`for p in self.parameters(): p.grad = 0.0`) — он заработает для всех трёх классов сразу.
</details>

<details>
<summary>10. Цикл обучения на XOR — что именно крутить?</summary>

Структура та же, что в `test_training_reduces_loss`, только параметров много и берём их из
`net.parameters()`:

```python
for step in range(50):
    loss = mean_squared_error_over_4_examples(net, xs, ys)   # forward, это Value
    net.zero_grad()                 # Идея 5 — ПЕРЕД backward
    loss.backward()
    for p in net.parameters():
        p.data -= lr * p.grad       # шаг SGD: против градиента
```

Подбери `lr` (порядка `0.05`) и архитектуру с **скрытым** слоем (например `[2, 8, 8, 1]`):
без скрытого слоя XOR не разделить, и тест на падение loss не пройдёт. Если loss не падает —
проверь знак апдейта (`-=`, не `+=`) и что `zero_grad` стоит **до** `backward`, а не после.
(Тело `mean_squared_error_over_4_examples` и инициализацию сети пишешь сам.)
</details>
