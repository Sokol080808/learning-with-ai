# Модуль 05 — Логистическая регрессия и классификация

**Большая идея модуля:** в модуле 04 модель предсказывала **число** (цену, температуру) —
это регрессия. Теперь мы хотим предсказывать **класс**: спам или не спам, болен или
здоров, кошка или собака (пока — только два класса, бинарная классификация). Оказывается,
для этого достаточно крошечной добавки к линейной модели: пропустить её выход `xW + b`
через **сигмоиду**, которая превращает любое число в **вероятность**. А чтобы модель
училась, мы заменяем MSE на «правильный» для вероятностей loss — **кросс-энтропию**.
Всё остальное (градиенты, шаги спуска) почти такое же, как в линейной регрессии. Это
самый короткий мост от «предсказываю число» к «предсказываю класс».

Формы тензоров держи в голове всё время (половина багов в DL — перепутанные оси):

- `X: (N, D)` — N объектов, у каждого D признаков;
- `w: (D,)` — по одному весу на признак;
- `b` — скаляр, общий сдвиг;
- `y: (N,)` — истинные метки, каждая 0 или 1;
- `p: (N,)` — предсказанные вероятности класса 1.

## Идея 1. Сигмоида превращает число в вероятность

**Интуиция.** Линейная модель выдаёт `z = xW + b` — любое вещественное число от −∞ до +∞.
Но вероятность обязана лежать в `(0, 1)`. Нужна функция-«воронка», которая сожмёт всю
числовую прямую в этот интервал, плавно и монотонно. Такая функция — **сигмоида**:

```
σ(z) = 1 / (1 + e^{-z})
```

S-образная кривая: при больших положительных `z` → почти 1, при больших отрицательных →
почти 0, ровно посередине `σ(0) = 0.5`.

**ПОЧЕМУ именно она.** Во-первых, она монотонна: чем больше логит `z`, тем больше
вероятность — порядок сохраняется. Во-вторых, у неё красивая производная (она нам
понадобится в backprop): `σ'(z) = σ(z)·(1 − σ(z))`. В-третьих, `z` (его называют
**логит**) имеет смысл «лог-шансов»: `z > 0` склоняет к классу 1, `z < 0` — к классу 0.

**Маленький численный пример.**

```
σ(0)  = 1/(1+1)        = 0.5
σ(2)  = 1/(1+e^{-2})   ≈ 0.8808
σ(-2) = 1/(1+e^{2})    ≈ 0.1192
```

Заметь симметрию: `σ(2) + σ(−2) = 1`. Это не случайность: `σ(−z) = 1 − σ(z)`.

**Подводный камень — переполнение.** Если в лоб писать `1/(1+np.exp(-z))`, то при сильно
**отрицательном** `z` (скажем, `z = −1000`) получится `np.exp(1000) = inf`, и дальше всё
поедет в `nan`. Лечится так: используем **две эквивалентные формы** в зависимости от знака
`z`, чтобы под экспонентой всегда стояло число `≤ 0` (а `exp` от неположительного не
переполняется):

```
z >= 0:   σ(z) = 1 / (1 + e^{-z})
z <  0:   σ(z) = e^{z} / (1 + e^{z})
```

Обе ветки дают один и тот же ответ — это та же сигмоида, просто домножили числитель и
знаменатель на `e^{z}`. Собрать ветки можно через `np.where` или булевы маски. Форма
выхода — **ровно такая же, как у входа** `z`.

## Идея 2. Бинарная кросс-энтропия (BCE)

**Интуиция.** Модель выдала вероятность `p` того, что объект из класса 1. Истинная метка —
`y` (0 или 1). Нам нужен loss, который **мал, когда модель уверенно права, и велик, когда
она уверенно ошибается**. Для MSE (как в модуле 04) это работает плохо: квадрат разности
вероятностей почти не наказывает за «уверенную чушь». Правильный инструмент для
вероятностей — **кросс-энтропия**.

Формула для одного объекта:

```
L = −( y·log(p) + (1−y)·log(1−p) )
```

Разберём по случаям — она хитрая только на вид:

- если `y = 1`: остаётся `L = −log(p)`. Чем ближе `p` к 1, тем `L` ближе к 0; при `p → 0`
  loss `→ +∞` (модель уверенно сказала «класс 0», а правда — 1, штраф огромный);
- если `y = 0`: остаётся `L = −log(1−p)`. Зеркально: штраф растёт, когда `p` велико.

Итоговый loss — **среднее** по всем `N` объектам:

```
BCE = (1/N) · Σ_i  −( y_i·log(p_i) + (1−y_i)·log(1−p_i) )
```

**ПОЧЕМУ логарифм.** `−log(p)` — это «сколько мы удивлены», увидев правильный ответ при
уверенности `p`. Уверенная ошибка должна стоить очень дорого (бесконечно дорого в пределе),
и логарифм даёт ровно такое наказание. К тому же в паре «сигмоида + BCE» производная
получается чудесно простой (см. Идею 5) — это не совпадение, эти двое созданы друг для
друга.

**Маленький численный пример.** Пусть `y = 1`.

```
p = 0.9  →  L = −log(0.9) ≈ 0.105   (почти угадал — штраф крошечный)
p = 0.5  →  L = −log(0.5) ≈ 0.693   (монетка — средний штраф)
p = 0.1  →  L = −log(0.1) ≈ 2.303   (уверенно мимо — больно)
```

**Подводный камень — `log(0)`.** Если `p` окажется ровно `0` или `1`, получим `log(0) =
−inf` и весь loss станет `nan`. Защита одной строкой: перед логарифмом **обрезаем** `p` в
`[eps, 1−eps]`, например `np.clip(p, 1e-12, 1 − 1e-12)`. Численно ответ не меняется, а
бесконечностей больше нет.

## Идея 3. Решающая граница

**Интуиция.** Модель выдаёт вероятность `p ∈ (0, 1)`. Чтобы назвать класс, нужен **порог**:
обычно `0.5`. Говорим «класс 1», если `p ≥ 0.5`, иначе «класс 0». А где проходит сама
граница между классами в пространстве признаков?

`p = 0.5` ровно тогда, когда логит `z = xW + b = 0` (ведь `σ(0) = 0.5`). Уравнение
`xW + b = 0` задаёт **прямую** (в 2D), **плоскость** (в 3D) или **гиперплоскость** (в общем
случае). Это и есть **решающая граница**: по одну сторону модель голосует за класс 0, по
другую — за класс 1.

**ПОЧЕМУ это важно.** Логистическая регрессия — **линейный** классификатор: её граница
всегда прямая/плоская. Поэтому два облака точек, которые можно разделить прямой («линейно
разделимые»), она выучит отлично; а вот «бублик внутри бублика» — нет, тут уже понадобятся
нелинейные модели (нейросети из модуля 06+). Знать границы инструмента — половина мастерства.

**Маленький пример.** Пусть `w = (1, 1)`, `b = 0`. Граница — это `x₁ + x₂ = 0`, то есть
прямая под 45°. Точка `(2, 2)`: `z = 4 > 0` → `p ≈ 0.982` → класс 1. Точка `(−2, −2)`:
`z = −4 < 0` → `p ≈ 0.018` → класс 0. Чем дальше от границы — тем увереннее модель.

> Порог `0.5` — не догма. Если ошибка «пропустить класс 1» дороже (например, пропустить
> болезнь), порог снижают, чтобы ловить больше единиц ценой ложных тревог. Поэтому в коде
> порог — параметр `thr`, а не зашитая константа.

## Идея 4. Точность (accuracy)

**Интуиция.** Loss — это для обучения (он гладкий, по нему берут градиент). А человеку
понятнее простая метрика: **какую долю объектов модель классифицировала верно**. Это и есть
accuracy.

Два шага:

1. перевести вероятности в жёсткие классы по порогу: `ŷ = 1, если p ≥ thr, иначе 0`;
2. посчитать долю совпадений `ŷ` с истинными `y`: `accuracy = (число верных) / N`.

**Маленький пример.**

```
p   = [0.9, 0.2, 0.7, 0.4]
ŷ   = [ 1 ,  0 ,  1 ,  0 ]   (порог 0.5)
y   = [ 1 ,  0 ,  0 ,  0 ]
                ↑ ошибка
верных 3 из 4  →  accuracy = 0.75
```

**ПОЧЕМУ нужны обе метрики.** Loss и accuracy измеряют разное. Loss учитывает **уверенность**
(сказать «0.51» там, где правда 1 — почти ошибка по loss, но верный класс по accuracy).
Поэтому при обучении бывает, что loss ещё падает, а accuracy уже застряла — это нормально.
Accuracy удобна для отчёта человеку; loss — то, что реально оптимизируем.

> Осторожно с **несбалансированными** классами: если 95% объектов класса 0, то тупая
> модель «всегда 0» даёт accuracy 0.95, ничего не выучив. В таких задачах смотрят на
> precision/recall — но это уже за пределами этого модуля.

## Идея 5. Градиенты для обучения

Чтобы модель училась, нужен **градиент loss по параметрам** `w` и `b` — он показывает,
куда сдвинуть параметры, чтобы loss вырос; мы пойдём в **противоположную** сторону.

**Чудо пары «сигмоида + BCE».** Если честно продифференцировать BCE по `w` и `b`, проходя
через сигмоиду, почти всё чудесно сокращается, и остаётся та же простая формула, что была в
линейной регрессии. Введём «ошибку на объекте» `err = p − y` (где `p = σ(Xw + b)`). Тогда:

```
dw = (1/N) · Xᵀ · err        # форма (D,)
db = (1/N) · Σ err           # скаляр
```

**Разбор форм** (проговори вслух — это и есть понимание):

- `err: (N,)` — по одной ошибке на объект;
- `Xᵀ: (D, N)`, умножаем на `err: (N,)` → результат `(D,)` — по одному числу на признак;
- делим на `N` (потому что loss — среднее) → `dw: (D,)`;
- `db` — просто среднее ошибок, скаляр.

**Шаг обучения** (это ты сделаешь в тесте `test_training_reduces_loss`, цикл уже написан):

```
w ← w − lr · dw
b ← b − lr · db
```

Повторяем много раз — loss падает, граница встаёт между классами. То самое «красное в
зелёное», только теперь модель учится сама.

> Откуда берётся `Xᵀ`? Та же причина, что в модуле 04: каждый вес `w_j` влияет на loss
> через все объекты, и при дифференцировании по `w_j` сумма по объектам сворачивается в
> скалярное произведение j-го столбца `X` с вектором ошибок. Матрично это и есть `Xᵀ·err`.
> Если хочешь — выведем по шагам на одном объекте, спроси.

### Подсказки

<details><summary>Подсказка 1 — sigmoid без переполнения</summary>

Проблема только при отрицательном `z` (там `e^{-z}` взрывается). Раздели на две ветки и
склей через `np.where`:

```
pos = 1.0 / (1.0 + np.exp(-z))          # безопасно при z >= 0
ez  = np.exp(z)                          # безопасно при z < 0 (z<0 → ez<1)
neg = ez / (1.0 + ez)
return np.where(z >= 0, pos, neg)
```
Тонкость: `np.where` всё равно вычисляет обе ветки, и «опасная» ветка может выдать
warning по `inf`. Это не ошибка (значение всё равно берётся из безопасной ветки), но если
хочешь чисто — посчитай каждую ветку только на своей маске через индексацию.
</details>

<details><summary>Подсказка 2 — bce_loss</summary>

Сначала клип: `p = np.clip(p, eps, 1 - eps)` с `eps = 1e-12`. Затем поэлементно
`−(y*log(p) + (1−y)*log(1−p))` и `np.mean(...)`. Не забудь обернуть результат в `float(...)`.
</details>

<details><summary>Подсказка 3 — predict_proba</summary>

Один линейный слой плюс сигмоида:
```
z = X @ w + b      # (N, D) @ (D,) = (N,), b добавится по broadcasting
return sigmoid(z)
```
Форма `z` — `(N,)`. Если получил `(N, 1)` или скаляр — проверь формы `X` и `w`.
</details>

<details><summary>Подсказка 4 — accuracy</summary>

```
preds = (p >= thr).astype(float)      # 1.0 / 0.0
return float(np.mean(preds == y))
```
`np.mean` булева массива и есть «доля True». Сравнение нестрогое (`>=`), чтобы `p`,
лежащее ровно на пороге, шло в класс 1, — это проверяет один из тестов.
</details>

<details><summary>Подсказка 5 — logreg_gradients (почти решение)</summary>

```
N = X.shape[0]
p = predict_proba(X, w, b)     # (N,)
err = p - y                    # (N,)
dw = (X.T @ err) / N           # (D, N) @ (N,) = (D,)
db = float(np.sum(err) / N)    # скаляр
return dw, db
```
Если `test_gradients_match_numerical` падает — почти наверняка перепутан знак (`y - p`
вместо `p - y`) или забыто деление на `N`. Численная проверка (конечные разности) в тесте
поймает и то, и другое.
</details>
