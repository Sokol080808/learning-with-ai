# Модуль 05 — Логистическая регрессия и классификация

**Большая идея модуля:** в модуле 04 модель предсказывала **число** (цену, температуру) —
это регрессия. Теперь мы хотим предсказывать **класс**: спам или не спам, болен или
здоров, кошка или собака (пока — только два класса, бинарная классификация). Оказывается,
для этого достаточно крошечной добавки к линейной модели: пропустить её выход `xW + b`
через **сигмоиду**, которая превращает любое число в **вероятность**. А чтобы модель
училась, мы заменяем MSE на «правильный» для вероятностей loss — **кросс-энтропию**.
Всё остальное (градиенты, шаги спуска) почти такое же, как в линейной регрессии. Это
самый короткий мост от «предсказываю число» к «предсказываю класс».

Формы тензоров держи в голове всё время (половина багов в DL — перепутанные оси):

- `X: (N, D)` — N объектов, у каждого D признаков;
- `w: (D,)` — по одному весу на признак;
- `b` — скаляр, общий сдвиг;
- `y: (N,)` — истинные метки, каждая 0 или 1;
- `p: (N,)` — предсказанные вероятности класса 1.

## Идея 1. Сигмоида превращает число в вероятность

**Интуиция.** Линейная модель выдаёт `z = xW + b` — любое вещественное число от −∞ до +∞.
Но вероятность обязана лежать в `(0, 1)`. Нужна функция-«воронка», которая сожмёт всю
числовую прямую в этот интервал, плавно и монотонно. Такая функция — **сигмоида**:

```
σ(z) = 1 / (1 + e^{-z})
```

S-образная кривая: при больших положительных `z` → почти 1, при больших отрицательных →
почти 0, ровно посередине `σ(0) = 0.5`.

**ПОЧЕМУ именно она.** Во-первых, она монотонна: чем больше логит `z`, тем больше
вероятность — порядок сохраняется. Во-вторых, у неё красивая производная (она нам
понадобится в backprop): `σ'(z) = σ(z)·(1 − σ(z))`. В-третьих, `z` (его называют
**логит**) имеет смысл «лог-шансов»: `z > 0` склоняет к классу 1, `z < 0` — к классу 0.

**Маленький численный пример.**

```
σ(0)  = 1/(1+1)        = 0.5
σ(2)  = 1/(1+e^{-2})   ≈ 0.8808
σ(-2) = 1/(1+e^{2})    ≈ 0.1192
```

Заметь симметрию: `σ(2) + σ(−2) = 1`. Это не случайность: `σ(−z) = 1 − σ(z)`.

**Подводный камень — переполнение.** Если в лоб писать `1/(1+np.exp(-z))`, то при сильно
**отрицательном** `z` (скажем, `z = −1000`) получится `np.exp(1000) = inf`, и дальше всё
поедет в `nan`. Лечится так: используем **две эквивалентные формы** в зависимости от знака
`z`, чтобы под экспонентой всегда стояло число `≤ 0` (а `exp` от неположительного не
переполняется):

```
z >= 0:   σ(z) = 1 / (1 + e^{-z})
z <  0:   σ(z) = e^{z} / (1 + e^{z})
```

Обе ветки дают один и тот же ответ — это та же сигмоида, просто домножили числитель и
знаменатель на `e^{z}`. Собрать ветки можно через `np.where` или булевы маски. Форма
выхода — **ровно такая же, как у входа** `z`.

## Идея 2. Бинарная кросс-энтропия (BCE)

**Интуиция.** Модель выдала вероятность `p` того, что объект из класса 1. Истинная метка —
`y` (0 или 1). Нам нужен loss, который **мал, когда модель уверенно права, и велик, когда
она уверенно ошибается**. Для MSE (как в модуле 04) это работает плохо: квадрат разности
вероятностей почти не наказывает за «уверенную чушь». Правильный инструмент для
вероятностей — **кросс-энтропия**.

Формула для одного объекта:

```
L = −( y·log(p) + (1−y)·log(1−p) )
```

Разберём по случаям — она хитрая только на вид:

- если `y = 1`: остаётся `L = −log(p)`. Чем ближе `p` к 1, тем `L` ближе к 0; при `p → 0`
  loss `→ +∞` (модель уверенно сказала «класс 0», а правда — 1, штраф огромный);
- если `y = 0`: остаётся `L = −log(1−p)`. Зеркально: штраф растёт, когда `p` велико.

Итоговый loss — **среднее** по всем `N` объектам:

```
BCE = (1/N) · Σ_i  −( y_i·log(p_i) + (1−y_i)·log(1−p_i) )
```

**ПОЧЕМУ логарифм.** `−log(p)` — это «сколько мы удивлены», увидев правильный ответ при
уверенности `p`. Уверенная ошибка должна стоить очень дорого (бесконечно дорого в пределе),
и логарифм даёт ровно такое наказание. К тому же в паре «сигмоида + BCE» производная
получается чудесно простой (см. Идею 5) — это не совпадение, эти двое созданы друг для
друга.

**Маленький численный пример.** Пусть `y = 1`.

```
p = 0.9  →  L = −log(0.9) ≈ 0.105   (почти угадал — штраф крошечный)
p = 0.5  →  L = −log(0.5) ≈ 0.693   (монетка — средний штраф)
p = 0.1  →  L = −log(0.1) ≈ 2.303   (уверенно мимо — больно)
```

**Подводный камень — `log(0)`.** Если `p` окажется ровно `0` или `1`, получим `log(0) =
−inf` и весь loss станет `nan`. Защита одной строкой: перед логарифмом **обрезаем** `p` в
`[eps, 1−eps]`, например `np.clip(p, 1e-12, 1 − 1e-12)`. Численно ответ не меняется, а
бесконечностей больше нет.

## Идея 3. Решающая граница

**Интуиция.** Модель выдаёт вероятность `p ∈ (0, 1)`. Чтобы назвать класс, нужен **порог**:
обычно `0.5`. Говорим «класс 1», если `p ≥ 0.5`, иначе «класс 0». А где проходит сама
граница между классами в пространстве признаков?

`p = 0.5` ровно тогда, когда логит `z = xW + b = 0` (ведь `σ(0) = 0.5`). Уравнение
`xW + b = 0` задаёт **прямую** (в 2D), **плоскость** (в 3D) или **гиперплоскость** (в общем
случае). Это и есть **решающая граница**: по одну сторону модель голосует за класс 0, по
другую — за класс 1.

**ПОЧЕМУ это важно.** Логистическая регрессия — **линейный** классификатор: её граница
всегда прямая/плоская. Поэтому два облака точек, которые можно разделить прямой («линейно
разделимые»), она выучит отлично; а вот «бублик внутри бублика» — нет, тут уже понадобятся
нелинейные модели (нейросети из модуля 06+). Знать границы инструмента — половина мастерства.

**Маленький пример.** Пусть `w = (1, 1)`, `b = 0`. Граница — это `x₁ + x₂ = 0`, то есть
прямая под 45°. Точка `(2, 2)`: `z = 4 > 0` → `p ≈ 0.982` → класс 1. Точка `(−2, −2)`:
`z = −4 < 0` → `p ≈ 0.018` → класс 0. Чем дальше от границы — тем увереннее модель.

> Порог `0.5` — не догма. Если ошибка «пропустить класс 1» дороже (например, пропустить
> болезнь), порог снижают, чтобы ловить больше единиц ценой ложных тревог. Поэтому в коде
> порог — параметр `thr`, а не зашитая константа.

**Геометрия порога — сдвиг той же гиперплоскости.** Менять порог `thr` — это не менять
наклон границы, а **параллельно сдвигать** её. Решающее правило `p ≥ thr` эквивалентно
`σ(z) ≥ thr`, то есть `z ≥ σ⁻¹(thr)`, то есть `xW + b ≥ logit(thr)`, где
`logit(t) = log(t / (1−t))`. При `thr = 0.5` правый край `logit(0.5) = 0` — это исходная
граница `xW + b = 0`. Понизили порог до `thr = 0.2` → `logit(0.2) ≈ −1.386` → условие
становится `xW + b ≥ −1.386`: вся область класса 1 **расширилась** в сторону класса 0
(граница сдвинулась параллельно себе, наклон `W` не тронут). Геометрически это та же
гиперплоскость, просто отодвинутая на постоянный сдвиг `logit(thr) / ‖W‖` вдоль нормали `W`.

**ПОЧЕМУ это полезно при дисбаллансе.** В несбалансированной задаче (редкий класс 1) при
пороге `0.5` модель почти никогда не выдаёт `p ≥ 0.5` для редкого класса — она «ловит» мало
единиц (низкий **recall**). Понижая порог, мы двигаем границу так, чтобы захватить больше
истинных единиц: recall растёт, но в область класса 1 попадает и больше нулей — **precision**
падает. Это и есть **компромисс precision/recall**, и геометрически он — ровно тот самый
параллельный сдвиг границы. Один обученный `W` задаёт целое **семейство** классификаторов,
по одному на каждый порог; кривая precision-recall (или ROC) перебирает их все.

## Идея 4. Точность (accuracy)

**Интуиция.** Loss — это для обучения (он гладкий, по нему берут градиент). А человеку
понятнее простая метрика: **какую долю объектов модель классифицировала верно**. Это и есть
accuracy.

Два шага:

1. перевести вероятности в жёсткие классы по порогу: `ŷ = 1, если p ≥ thr, иначе 0`;
2. посчитать долю совпадений `ŷ` с истинными `y`: `accuracy = (число верных) / N`.

**Маленький пример.**

```
p   = [0.9, 0.2, 0.7, 0.4]
ŷ   = [ 1 ,  0 ,  1 ,  0 ]   (порог 0.5)
y   = [ 1 ,  0 ,  0 ,  0 ]
                ↑ ошибка
верных 3 из 4  →  accuracy = 0.75
```

**ПОЧЕМУ нужны обе метрики.** Loss и accuracy измеряют разное. Loss учитывает **уверенность**
(сказать «0.51» там, где правда 1 — почти ошибка по loss, но верный класс по accuracy).
Поэтому при обучении бывает, что loss ещё падает, а accuracy уже застряла — это нормально.
Accuracy удобна для отчёта человеку; loss — то, что реально оптимизируем.

> Осторожно с **несбалансированными** классами: если 95% объектов класса 0, то тупая
> модель «всегда 0» даёт accuracy 0.95, ничего не выучив. В таких задачах смотрят на
> precision/recall — но это уже за пределами этого модуля.

## Идея 5. Градиенты для обучения

Чтобы модель училась, нужен **градиент loss по параметрам** `w` и `b` — он показывает,
куда сдвинуть параметры, чтобы loss вырос; мы пойдём в **противоположную** сторону.

**Чудо пары «сигмоида + BCE».** Если честно продифференцировать BCE по `w` и `b`, проходя
через сигмоиду, почти всё чудесно сокращается, и остаётся та же простая формула, что была в
линейной регрессии. Введём «ошибку на объекте» `err = p − y` (где `p = σ(Xw + b)`). Тогда:

```
dw = (1/N) · Xᵀ · err        # форма (D,)
db = (1/N) · Σ err           # скаляр
```

**Разбор форм** (проговори вслух — это и есть понимание):

- `err: (N,)` — по одной ошибке на объект;
- `Xᵀ: (D, N)`, умножаем на `err: (N,)` → результат `(D,)` — по одному числу на признак;
- делим на `N` (потому что loss — среднее) → `dw: (D,)`;
- `db` — просто среднее ошибок, скаляр.

**Шаг обучения** (это ты сделаешь в тесте `test_training_reduces_loss`, цикл уже написан):

```
w ← w − lr · dw
b ← b − lr · db
```

Повторяем много раз — loss падает, граница встаёт между классами. То самое «красное в
зелёное», только теперь модель учится сама.

> Откуда берётся `Xᵀ`? Та же причина, что в модуле 04: каждый вес `w_j` влияет на loss
> через все объекты, и при дифференцировании по `w_j` сумма по объектам сворачивается в
> скалярное произведение j-го столбца `X` с вектором ошибок. Матрично это и есть `Xᵀ·err`.

**Вывод по шагам на одном объекте (chain rule).** Раньше мы поверили в формулу `err = p − y`
на слово — теперь выведем её честно. Это первый твой настоящий backprop, и здесь видно,
*откуда* берётся то самое сокращение. Возьмём один объект `x` (вектор `(D,)`), его метку `y`
и параметры `w, b`. Цепочка вычислений:

```
z = x·w + b           (логит, скаляр)
p = σ(z)              (вероятность)
L = −( y·log p + (1−y)·log(1−p) )     (BCE на одном объекте)
```

Нам нужна `∂L/∂w_j`. По цепному правилу идём от конца к началу, через два звена `p` и `z`:

```
∂L/∂w_j = (∂L/∂p) · (∂p/∂z) · (∂z/∂w_j)
```

Разберём каждый множитель отдельно — порознь они простые:

**Звено 1 — `∂L/∂p`.** Дифференцируем `L = −(y·log p + (1−y)·log(1−p))` по `p`
(помним `(log u)' = u'/u`, а у `log(1−p)` внутренняя производная `−1`):

```
∂L/∂p = −( y/p − (1−y)/(1−p) ) = −y/p + (1−y)/(1−p)
```

**Звено 2 — `∂p/∂z`.** Это та самая «красивая производная сигмоиды» из Идеи 1:

```
∂p/∂z = σ'(z) = σ(z)·(1 − σ(z)) = p·(1 − p)
```

**Перемножим звено 1 и звено 2** — здесь происходит чудо, ради которого сигмоида и BCE
созданы друг для друга. Раскрываем скобки и сокращаем:

```
∂L/∂z = (∂L/∂p)·(∂p/∂z)
      = ( −y/p + (1−y)/(1−p) ) · p·(1−p)
      = −y·(1−p) + (1−y)·p          ← p и (1−p) сократились!
      = −y + y·p + p − y·p
      = p − y
```

Вот оно: `∂L/∂z = p − y`. Вся «начинка» BCE и сигмоиды схлопнулась в одну разность —
это и есть наш `err`. Деления на `p` и `(1−p)` (потенциально опасные, см. подводный камень
с `log(0)`) исчезли аналитически — поэтому, кстати, в проде считают BCE *из логитов* (`bce_with_logits`),
минуя нестабильную пару «clip → log».

**Звено 3 — `∂z/∂w_j`.** Тривиально, ведь `z = x·w + b = Σ_k x_k·w_k + b`:

```
∂z/∂w_j = x_j        ∂z/∂b = 1
```

**Собираем всё для одного объекта:**

```
∂L/∂w_j = (p − y)·x_j      →  ∇_w L = (p − y)·x      (вектор (D,))
∂L/∂b   = (p − y)·1        =  (p − y)
```

**Переход к N объектам.** Loss — это среднее `L̄ = (1/N)·Σ_i L_i`, производная среднего —
среднее производных. Складываем по объектам `i` (у каждого свой `err_i = p_i − y_i`):

```
∂L̄/∂w_j = (1/N)·Σ_i (p_i − y_i)·x_{i,j}
```

Сумма `Σ_i err_i · x_{i,j}` по объектам — это ровно скалярное произведение вектора ошибок
`err: (N,)` с j-м **столбцом** матрицы `X`, то есть j-я компонента `Xᵀ·err`. Собрав все `j`
вместе, получаем матричную форму из начала Идеи 5:

```
dw = (1/N)·Xᵀ·err        db = (1/N)·Σ_i err_i
```

Круг замкнулся: матричная формула — это просто N покомпонентных формул `(p−y)·x`, аккуратно
свёрнутых в умножение `Xᵀ·err`. Именно эта механика (локальная производная × приходящий
градиент, потом суммирование по batch) — скелет backprop, который в модулях 08–09 ты
обобщишь на много слоёв.

## Мост к многоклассовой задаче: softmax, категориальная кросс-энтропия, log-sum-exp

До сих пор было **два** класса: одна вероятность `p` (а вероятность класса 0 — это `1−p`,
её даже не считают отдельно). Но классов бывает `K`: цифра 0–9 (`K=10`), порода собаки,
следующий токен в тексте (`K` = размер словаря, как в капстоне с nanoGPT). Бинарная схема
«один логит → сигмоида» не масштабируется. Нужна её прямая `K`-классовая версия — и она
устроена ровно по тем же принципам.

### Softmax — сигмоида для K классов

**Интуиция.** Теперь линейный слой выдаёт не одно число, а **вектор логитов**
`z = (z_1, …, z_K)` — по одному «баллу за класс». Нам нужно превратить эти `K` произвольных
вещественных чисел в **распределение вероятностей**: `K` неотрицательных чисел, сумма
которых ровно `1`. Эту работу делает **softmax**:

```
softmax(z)_k = e^{z_k} / Σ_j e^{z_j}
```

Каждый логит экспоненцируем (становится положительным), затем **нормируем** делением на
сумму всех — получаем доли, складывающиеся в 1. «Soft» (мягкий) max: в отличие от жёсткого
`argmax`, который отдаёт всю массу одному победителю, softmax распределяет вес плавно —
больший логит получает большую долю, но и остальные ненулевые.

**ПОЧЕМУ это обобщает сигмоиду.** Возьмём `K=2` и посчитаем вероятность класса 1:

```
softmax(z)_1 = e^{z_1} / (e^{z_0} + e^{z_1})
             = 1 / (1 + e^{z_0 − z_1})       ← поделили числитель и знаменатель на e^{z_1}
             = σ(z_1 − z_0)
```

То есть бинарный softmax — это сигмоида от **разности** двух логитов. Сигмоида была частным
случаем всё это время: для двух классов важна только разность баллов, поэтому хватало
одного числа. Это и есть смысл фразы «softmax обобщает сигмоиду на K классов».

**Маленький численный пример.** `z = (2, 1, 0)`:

```
e^2 ≈ 7.389,  e^1 ≈ 2.718,  e^0 = 1.000;   сумма ≈ 11.107
softmax ≈ (0.665, 0.245, 0.090)            (сумма = 1, порядок сохранён)
```

**Свойство сдвига (понадобится для стабильности).** Если ко **всем** логитам прибавить одну
и ту же константу `c`, softmax **не меняется**:

```
softmax(z + c)_k = e^{z_k + c} / Σ_j e^{z_j + c} = (e^c · e^{z_k}) / (e^c · Σ_j e^{z_j})
                 = e^{z_k} / Σ_j e^{z_j} = softmax(z)_k
```

`e^c` выносится из суммы и **сокращается** с тем же `e^c` в числителе. Запомни этот факт —
через абзац он спасёт нас от переполнения.

### Категориальная кросс-энтропия (CCE) — BCE для K классов

**Интуиция.** Истинный класс теперь — целое число `y ∈ {0, …, K−1}` (или, что то же,
**one-hot** вектор: единица в позиции `y`, нули в остальных). Модель выдала распределение
`p = softmax(z)`. Loss должен быть мал, когда модель уверенно ставит массу на **правильный**
класс. Формула:

```
L = − Σ_k  one_hot(y)_k · log(p_k) = − log(p_y)
```

One-hot обнуляет все слагаемые, кроме одного — остаётся `−log(p_y)`, «удивление» от того, что
правильным оказался класс `y`, которому модель дала вероятность `p_y`. Это **дословно** та же
идея, что в BCE (`−log p`, когда `y=1`), просто теперь правильный класс выбирается индексом, а
не ветвлением `y / 1−y`. Бинарная CCE при `K=2` numerically совпадает с BCE.

Итоговый loss по batch — среднее `−log(p_{y_i})` по объектам.

### log-sum-exp — канонический способ считать softmax стабильно

**Подводный камень — переполнение.** Считать softmax «в лоб» опасно: `e^{z_k}` переполняется
уже при `z_k ≈ 710` (предел `float64`), а в knnGPT-подобных сетях логиты бывают и больше.
Получаем `inf / inf = nan`. Симметрично, при сильно отрицательных логитах `e^{z_k}` обнуляется
(underflow), и `log(0)` в CCE даёт `−inf`.

**Лекарство — свойство сдвига.** Раз softmax не меняется при вычитании константы из всех
логитов, вычтем **максимум**: `c = max_k z_k`. Тогда наибольший сдвинутый логит станет ровно
`0`, остальные — отрицательными, и все `e^{z_k − c} ∈ (0, 1]` — переполнение невозможно:

```
softmax(z)_k = e^{z_k − c} / Σ_j e^{z_j − c},     где c = max_k z_k
```

Числитель при `k = argmax` равен `e^0 = 1`, а сумма в знаменателе всегда `≥ 1` — никаких
`inf`. Этот приём — вычесть максимум перед экспонентой — называется **log-sum-exp trick**, по
имени устойчивого вычисления логарифма суммы экспонент:

```
log Σ_j e^{z_j} = c + log Σ_j e^{z_j − c},     c = max_k z_k
```

**ПОЧЕМУ именно log-sum-exp, а не просто «поделить softmax».** Для CCE нам нужен `log(p_y)`, а
`p_y` маленькое → `log` маленького числа теряет точность. Поэтому стабильную CCE считают **не**
через `log(softmax(z))`, а напрямую: `log p_y = z_y − logsumexp(z)`. Здесь нет ни деления на
крошечное число, ни `log(0)` — только разность и один устойчивый `logsumexp`. Это и есть смысл
`F.cross_entropy(logits, y)` в PyTorch: он принимает **сырые логиты**, а не вероятности, именно
чтобы пройти через log-sum-exp, минуя нестабильную пару «softmax → log».

**Маленький численный пример отмены `c`.** `z = (1000, 1001, 1002)`. В лоб — `e^{1000} = inf`,
всё `nan`. С трюком `c = 1002`:

```
z − c = (−2, −1, 0);   e^{·} = (0.135, 0.368, 1.000);   сумма = 1.503
softmax = (0.090, 0.245, 0.665)
```

Тот же ответ, что для `z = (−2, −1, 0)` — ведь softmax зависит только от **разностей**
логитов, а вычитание `c` их не трогает. Константа `c` появилась в `e^c` и сократилась —
численно мы спаслись, математически ничего не изменили.

> Этот приём встретится снова: в капстоне (nanoGPT, модуль 17) attention считает softmax по
> строкам матрицы оценок — и делает это через ровно тот же сдвиг на максимум. Стабильный
> softmax через log-sum-exp — фундамент, а не частность; познакомиться с ним проще здесь, на
> трёх числах, чем там, среди тензоров внимания.

> Градиент CCE по логитам не менее красив, чем у BCE: `∂L/∂z = softmax(z) − one_hot(y)`. Это
> та же структура «предсказание минус истина» (`p − y`), что в бинарном случае, только теперь
> `p` — вектор softmax, а `y` — one-hot. Не совпадение: бинарный случай — её частный вид.
> Проверишь это конечными разностями в задании 7.

### Подсказки

<details><summary>Подсказка 1 — sigmoid без переполнения</summary>

Проблема только при отрицательном `z` (там `e^{-z}` взрывается). Раздели на две ветки и
склей через `np.where`:

```
pos = 1.0 / (1.0 + np.exp(-z))          # безопасно при z >= 0
ez  = np.exp(z)                          # безопасно при z < 0 (z<0 → ez<1)
neg = ez / (1.0 + ez)
return np.where(z >= 0, pos, neg)
```
Тонкость: `np.where` всё равно вычисляет обе ветки, и «опасная» ветка может выдать
warning по `inf`. Это не ошибка (значение всё равно берётся из безопасной ветки), но если
хочешь чисто — посчитай каждую ветку только на своей маске через индексацию.
</details>

<details><summary>Подсказка 2 — bce_loss</summary>

Сначала клип: `p = np.clip(p, eps, 1 - eps)` с `eps = 1e-12`. Затем поэлементно
`−(y*log(p) + (1−y)*log(1−p))` и `np.mean(...)`. Не забудь обернуть результат в `float(...)`.
</details>

<details><summary>Подсказка 3 — predict_proba</summary>

Один линейный слой плюс сигмоида:
```
z = X @ w + b      # (N, D) @ (D,) = (N,), b добавится по broadcasting
return sigmoid(z)
```
Форма `z` — `(N,)`. Если получил `(N, 1)` или скаляр — проверь формы `X` и `w`.
</details>

<details><summary>Подсказка 4 — accuracy</summary>

```
preds = (p >= thr).astype(float)      # 1.0 / 0.0
return float(np.mean(preds == y))
```
`np.mean` булева массива и есть «доля True». Сравнение нестрогое (`>=`), чтобы `p`,
лежащее ровно на пороге, шло в класс 1, — это проверяет один из тестов.
</details>

<details><summary>Подсказка 5 — logreg_gradients (почти решение)</summary>

```
N = X.shape[0]
p = predict_proba(X, w, b)     # (N,)
err = p - y                    # (N,)
dw = (X.T @ err) / N           # (D, N) @ (N,) = (D,)
db = float(np.sum(err) / N)    # скаляр
return dw, db
```
Если `test_gradients_match_numerical` падает — почти наверняка перепутан знак (`y - p`
вместо `p - y`) или забыто деление на `N`. Численная проверка (конечные разности) в тесте
поймает и то, и другое.
</details>

<details><summary>Подсказка 6 — softmax (нудный шаг — про оси)</summary>

Главное — **по какой оси** считать максимум и сумму. Для 2-D входа `(N, K)` каждая строка —
свой пример, нормируем по оси `K` (последней), сохраняя ось для broadcasting:

```
z = np.asarray(z, dtype=float)
c = np.max(z, axis=-1, keepdims=True)   # (N, 1) — вычитаем максимум строки
ez = np.exp(z - c)                       # все ≤ 1, переполнения нет
return ez / np.sum(ez, axis=-1, keepdims=True)
```

`axis=-1` сам сделает правильное и для 1-D входа `(K,)`. `keepdims=True` обязателен — без него
формы `(N,)` и `(N, K)` не сложатся по broadcasting. Проверка стабильности из теста (`[1000,
1001, 1002]`) пройдёт ровно из-за вычитания `c`.
</details>

<details><summary>Подсказка 7 — cce_loss через log-sum-exp (НЕ через log(softmax))</summary>

Соблазн — посчитать `softmax`, взять `log`, выбрать нужный класс. Так делать не надо:
`log(маленькое)` теряет точность, а на больших логитах `softmax` уже мог потерять знаки. Считай
`log p_y = z_y − logsumexp(z)` напрямую. Тот же сдвиг на максимум, что в softmax:

```
logits = np.asarray(logits, dtype=float)
c = np.max(logits, axis=-1, keepdims=True)          # (N, 1)
logsumexp = c[:, 0] + np.log(np.sum(np.exp(logits - c), axis=-1))   # (N,)
N = logits.shape[0]
log_p_correct = logits[np.arange(N), y_int] - logsumexp            # (N,) — z_y − lse
return float(np.mean(-log_p_correct))
```

`logits[np.arange(N), y_int]` — это «продвинутая индексация»: для каждой строки `i` берём
элемент в столбце `y_int[i]`, то есть логит истинного класса. Сверка с
`F.cross_entropy(logits, y)` в тесте пройдёт, потому что torch делает ровно это же (он тоже
принимает логиты, не вероятности).
</details>
