# Модуль 11 — Знакомство с PyTorch

Большая идея модуля: до сих пор мы всё считали на **numpy** — и градиенты выводили руками,
шаг за шагом расписывая backprop. Это было важно: ты увидел механику изнутри. Но в реальных
сетях слоёв десятки, и руками дифференцировать невыносимо. PyTorch решает ровно эту боль:
его тензор **сам помнит, как был получен**, и по команде `.backward()` прогоняет backprop
за тебя. Тот же numpy-подобный API — плюс автоматическое дифференцирование (**autograd**).
В этом модуле ты переносишь привычные операции на тензоры и впервые получаешь градиент,
**не считая производную руками**.

> В этом курсе мы держим всё на **CPU** — никаких GPU. Так тесты гоняются быстро и одинаково
> у всех. Идеи от устройства не зависят: научишься на CPU — на GPU поедет тот же код.

---

## Идея 1. `torch.Tensor` — это numpy-массив, который умеет дифференцироваться

Тензор PyTorch — почти тот же `np.ndarray`: есть форма (`shape`), тип элементов (`dtype`),
индексация, broadcasting, `+ - * @`, `sum()`, `mean()`. Если ты знаешь numpy — ты уже на 80%
знаешь torch.

```python
import torch
x = torch.tensor([1.0, 2.0, 3.0])   # форма (3,), dtype = float32
x.shape      # torch.Size([3])
x.dtype      # torch.float32
x + 1        # tensor([2., 3., 4.])
x @ x        # 1+4+9 = 14.0  (скалярное произведение)
```

**ПОЧЕМУ не остаться на numpy?** Потому что у numpy нет памяти о происхождении значений.
numpy-массив `z = x*x` — это просто числа; numpy не знает, что `z` зависит от `x`. А
torch-тензор хранит **граф вычислений** (об этом Идея 3) и потому может посчитать `dz/dx`.
Это единственное, но решающее отличие.

**Про `dtype` — важная ловушка.** По умолчанию `torch.tensor([1, 2, 3])` из целых чисел
даёт **целочисленный** тензор (`int64`), а autograd работает только с **вещественными**
типами. Поэтому в нейросетях почти всё держат во `float32` — это стандарт точности/скорости
для DL. Привыкай явно приводить данные к `float32`.

```python
torch.tensor([1, 2, 3]).dtype          # torch.int64  — НЕ годится для градиентов
torch.tensor([1, 2, 3]).float().dtype  # torch.float32 — то, что нужно
```

Форма словами: тензор данных в DL обычно `(N, D)` — N объектов в строках, D признаков в
столбцах. Эту запись `(N, D)` мы будем использовать постоянно.

## Идея 2. `requires_grad`, `.backward()`, `.grad` — как попросить производную

Чтобы torch отслеживал операции над тензором и умел по нему дифференцировать, ставим флаг
`requires_grad=True`. Дальше считаем какой-нибудь **скаляр** (loss или просто `f(x)`),
вызываем у него `.backward()` — и градиент появляется в `x.grad`.

```python
x = torch.tensor([1.0, 2.0, 3.0], requires_grad=True)
f = (x ** 2).sum()    # f = 1 + 4 + 9 = 14.0  (скаляр!)
f.backward()          # запускаем backprop
x.grad                # tensor([2., 4., 6.])
```

Проверим руками. `f(x) = Σ xᵢ²`, значит `∂f/∂xᵢ = 2·xᵢ`. Для `x = [1, 2, 3]` это
`[2, 4, 6]` — ровно то, что вернул `x.grad`. Autograd посчитал производную **за нас** и
не ошибся.

**ПОЧЕМУ `.backward()` зовут именно у скаляра?** Градиент — это «как меняется *одно* число
при шевелении входов». Loss всегда скаляр, поэтому `∂loss/∂x` имеет ту же форму, что `x`, и
смысл «вклад каждого входа в loss». Если `f` — вектор, непонятно, чью производную брать;
torch потребует уточнить. Правило простое: **сведи к скаляру** (часто `.sum()` или
`.mean()`), потом `.backward()`.

Ещё две тонкости, без которых легко споткнуться:

- `.grad` **накапливается** (`+=`), а не перезаписывается. Перед новым `.backward()` старый
  градиент обнуляют: `x.grad = None` или `x.grad.zero_()`. В цикле обучения это делают
  каждый шаг — иначе градиенты складываются с прошлыми и шаг идёт не туда.
- Форма `x.grad` всегда совпадает с формой `x`. Если `x` имеет форму `(N, D)`, то и градиент
  будет `(N, D)`.

## Идея 3. Граф вычислений строится автоматически

Когда ты пишешь `f = (x**2).sum()`, torch не просто считает число — он попутно строит
**граф**: «`x` возвели в квадрат → получили `t`; `t` просуммировали → получили `f`». Каждый
промежуточный тензор помнит свою операцию-родителя (это видно в атрибуте `grad_fn`).

```python
x = torch.tensor([2.0], requires_grad=True)
t = x ** 2     # t.grad_fn -> <PowBackward0>   (помнит: я родился из возведения в степень)
f = t.sum()    # f.grad_fn -> <SumBackward0>    (помнит: я родился из суммы)
```

`.backward()` идёт по этому графу **в обратную сторону** (отсюда «back») и по цепному
правилу перемножает локальные производные — ровно тот backprop, что ты писал руками в
прошлых модулях, только теперь автоматически. Граф — это и есть «память о происхождении».

**ПОЧЕМУ это удобно?** Тебе больше не нужно держать в голове производную каждого слоя. Ты
описываешь **только прямой проход** (forward) обычным кодом, а обратный (backward) torch
выводит из графа сам. Меняешь архитектуру — backprop пересчитывается автоматически, без
единой строчки про производные.

Маленькая, но частая ловушка: по умолчанию граф **одноразовый** — после `.backward()` он
освобождается. Это нормально для цикла обучения (на каждом шаге forward строит граф заново).
Если же нужно посчитать `.backward()` дважды от одного forward — это отдельная история, в
наших заданиях она не понадобится.

## Идея 4. Устройство (`device`) — у нас всегда `cpu`

У тензора есть атрибут `device` — где физически лежат числа и где идут вычисления: `cpu`
или `cuda` (видеокарта). GPU ускоряет матричные операции в десятки раз, но **в этом курсе мы
сознательно остаёмся на CPU**: так одинаково у всех, без драйверов и без сюрпризов, и тесты
быстрые.

```python
x = torch.tensor([1.0, 2.0])
x.device                 # device(type='cpu')  — по умолчанию CPU
x.to('cpu')              # явно «оставить на CPU» (no-op, но так пишут в реальном коде)
```

**ПОЧЕМУ вообще об этом помнить?** Потому что нельзя смешивать тензоры с разных устройств в
одной операции — torch выдаст ошибку «expected device cpu but got cuda». В большом коде это
частый баг. У нас всё на `cpu`, так что споткнуться негде, — но знать про `device` полезно
заранее, чтобы потом не удивляться.

## Идея 5. Накопление градиентов и `zero_grad` — почему PyTorch так устроен

Ты уже знаешь, что `.grad` не перезаписывается, а **прибавляется**. Разберём, зачем это
нужно и как с этим работать.

### Почему накопление, а не перезапись?

Это **намеренный дизайн**. Представь, что у тебя несколько loss-термов (например, основной
loss и регуляризатор), и ты хочешь посчитать суммарный градиент. С накоплением достаточно
вызвать `.backward()` от каждого терма по очереди — градиенты сложатся сами:

```python
x = torch.tensor([1.0, 2.0], requires_grad=True)

loss1 = (x ** 2).sum()      # первый терм
loss1.backward()
# x.grad = tensor([2., 4.])

loss2 = x.sum()             # второй терм
loss2.backward()
# x.grad = tensor([3., 5.])  — прибавилось [1., 1.], не перезаписалось!
```

Ещё накопление полезно для **gradient accumulation** — технике, когда нельзя вместить весь
батч в память GPU: разбиваешь на части, зовёшь `.backward()` несколько раз, а потом делаешь
один шаг оптимизатора. Градиенты при этом сложились как надо.

### Проблема в цикле обучения

В обычном цикле обучения такое накопление — это **баг**: на каждом шаге ты хочешь градиент
только от текущего батча, а не сумму всех предыдущих. Поэтому перед каждым `.backward()`
градиенты нужно **обнулять**:

```python
# Вручную (когда параметры — просто тензоры):
if x.grad is not None:
    x.grad.zero_()   # обнуляем на месте (in-place)
# или:
x.grad = None        # то же, но освобождает память
```

### `optimizer.zero_grad()` — тот же принцип, но для всех параметров

Когда параметров много (сотни в слоях сети), обнулять каждый вручную неудобно. За тебя это
делает оптимизатор: `optimizer.zero_grad()` пробегается по всем параметрам, которые он знает,
и обнуляет их `.grad`. Это **первая строчка каждого шага** обучения.

```python
optimizer = torch.optim.SGD([W, b], lr=0.01)

# Один шаг обучения:
optimizer.zero_grad()   # 1. обнуляем старые градиенты
pred = x @ W + b        # 2. forward pass
loss = mse(pred, y)     # 3. считаем loss
loss.backward()         # 4. backward: вычисляем градиенты
optimizer.step()        # 5. делаем шаг: W -= lr * W.grad, b -= lr * b.grad
```

Запомни порядок: **zero → forward → loss → backward → step**. Он будет повторяться в каждом
модуле до конца курса.

## Идея 6. `torch.no_grad()` и режим инференса

Когда ты **используешь** обученную модель (не обучаешь — предсказываешь), граф вычислений
строить незачем. Но torch всё равно тратит на это память и время. Контекстный менеджер
`torch.no_grad()` отключает построение графа внутри блока — расчёт идёт быстрее и дешевле:

```python
x = torch.tensor([1.0, 2.0, 3.0], requires_grad=True)

with torch.no_grad():
    y = (x ** 2).sum()   # граф НЕ строится
    # y.grad_fn is None  — у тензора нет связи с графом
    # y.backward()       — вызовет ошибку! нечего дифференцировать

print(y)       # tensor(14.)   — число посчиталось правильно
```

**ПОЧЕМУ это важно?**

1. **Скорость и память.** Хранение графа требует памяти пропорционально числу операций.
   Во время инференса (предсказания на новых данных) градиенты не нужны, поэтому `no_grad`
   экономит и RAM, и время — особенно заметно при больших батчах.
2. **Явное намерение.** `no_grad` сигнализирует читателю кода: «здесь мы только смотрим
   на предсказание, ничего не учим». Это соглашение по стилю в любом PyTorch-коде.
3. **Оценка модели.** Когда после каждой эпохи считаешь метрики на валидационной выборке —
   всегда оборачивай в `no_grad`. Иначе ты случайно создашь граф от всей валидации.

```python
# Типичный паттерн: обучение vs. оценка
model.train()   # включаем train-режим (BatchNorm, Dropout ведут себя иначе)
for x_batch, y_batch in train_loader:
    optimizer.zero_grad()
    loss = criterion(model(x_batch), y_batch)
    loss.backward()
    optimizer.step()

model.eval()    # переключаем в eval-режим
with torch.no_grad():
    for x_val, y_val in val_loader:
        preds = model(x_val)   # просто forward, без графа
        # считаем метрики...
```

> **`torch.inference_mode()`** — более новый и строгий аналог `no_grad` (появился в PyTorch
> 1.9). Он ещё агрессивнее оптимизирует: тензоры внутри него не могут стать листьями графа
> даже если попросить. Для инференса предпочитают его. В этом курсе `no_grad` и
> `inference_mode` взаимозаменяемы.

## Идея 7. `.detach()` — отрезать тензор от графа

Иногда нужно взять значение тензора, но **оборвать его связь с историей вычислений** — чтобы
операции над ним не влекли градиентов назад. Для этого есть `.detach()`.

```python
x = torch.tensor([1.0, 2.0], requires_grad=True)
y = x ** 2         # y.grad_fn = <PowBackward0>

z = y.detach()     # z — тот же массив чисел, но без grad_fn
# z.requires_grad  # False
# z.grad_fn        # None
# y не тронут, x не тронут

z2 = z * 3        # операции над z не строят граф
```

`.detach()` не копирует данные — `z` и `y` разделяют одну память. Это значит: менять `z`
in-place рискованно, лучше вызывать `.detach().clone()` если нужна независимая копия.

**Когда это нужно на практике:**

- **Логирование и отображение.** Хочешь напечатать loss как число — `loss.detach().item()`.
  Если просто вызвать `.item()` без detach, Python числа получишь, но граф от этого call не
  оборвётся (хотя `.item()` сам по себе не строит граф, привычка detach перед numpy/print
  защищает от ошибок).
- **Заморозка части сети.** Хочешь обновлять только последний слой, а предыдущие оставить
  неизменными — detach активации после замороженного блока. Градиент не потечёт дальше.
- **Целевая сеть (target network) в RL.** Классический трюк: параметры target-сети не
  должны получать градиенты от текущего loss — обёртывают их предсказания в `.detach()`.

```python
# Пример: считаем reward-target без градиентов
with torch.no_grad():
    target = reward + gamma * target_net(next_obs)
# или:
target = target_net(next_obs).detach()

loss = F.mse_loss(current_net(obs), target)
loss.backward()   # градиент течёт только через current_net
```

## Идея 8. `.backward()` на не-листовом тензоре и `retain_graph`

### Листья и не-листья

В терминах autograd **лист** (`is_leaf = True`) — это тензор, созданный пользователем
напрямую (например, параметры `W` и `b`). Промежуточные тензоры, полученные из операций,
не являются листьями.

По умолчанию `.grad` заполняется **только у листьев**. Зачем torch такой экономный?
Хранить градиент каждого промежуточного тензора — дорого по памяти. В 99% случаев нужен
только градиент у параметров (листьев), а не у активаций.

```python
x = torch.tensor([2.0], requires_grad=True)  # лист
y = x ** 2      # не лист (получен из x)
z = y.sum()     # скаляр

z.backward()
x.grad          # tensor([4.])   — заполнен, x — лист
y.grad          # None           — не лист, torch не сохраняет
```

Если нужен градиент у промежуточного тензора — вызови `.retain_grad()` на нём **до** запуска
backward:

```python
x = torch.tensor([2.0], requires_grad=True)
y = x ** 2
y.retain_grad()          # теперь torch сохранит y.grad
z = y * 3
z.backward()
y.grad   # tensor([3.])   — ∂z/∂y = 3
x.grad   # tensor([12.])  — ∂z/∂x = ∂z/∂y · ∂y/∂x = 3 · 4 = 12
```

### Граф одноразовый — почему?

После `.backward()` graф освобождается. Если попробовать вызвать `.backward()` ещё раз на
том же скаляре — получишь ошибку:

```
RuntimeError: Trying to backward through the graph a second time ...
```

Это тоже осознанная экономия памяти: хранить граф до явного запроса значит зря занимать RAM.
Чтобы вызвать backward дважды (редкий случай — например, вычисление производной второго
порядка), передай `retain_graph=True`:

```python
x = torch.tensor([1.0, 2.0], requires_grad=True)
f = (x ** 2).sum()

f.backward(retain_graph=True)   # граф остался живым
x.grad                          # tensor([2., 4.])

x.grad.zero_()                  # обнуляем перед вторым вызовом!
f.backward()                    # теперь граф освобождается
x.grad                          # tensor([2., 4.])  — те же значения
```

**Когда `retain_graph` реально нужен?** В основном — при вычислении якобианов, вторых
производных (метрики кривизны, MAML-like meta-learning) или когда несколько loss-термов
зависят от одного forward pass и ты зовёшь backward для каждого по отдельности. В обычном
цикле обучения он не нужен: на каждой итерации строишь новый граф.

## Идея 9. Полный цикл обучения от начала до конца

Собери все детали в одну картину. Вот **минимальный, но полный** цикл обучения на игрушечной
линейной регрессии — ни строчки лишнего, но все шаги на месте:

```python
import torch

# --- Данные ---
torch.manual_seed(0)
N, D_in, D_out = 32, 4, 1
x = torch.randn(N, D_in)           # признаки: (32, 4)
W_true = torch.randn(D_in, D_out)
b_true = torch.randn(D_out)
y = x @ W_true + b_true            # целевые значения: (32, 1)

# --- Параметры (листья с requires_grad=True) ---
W = torch.zeros(D_in, D_out, requires_grad=True)
b = torch.zeros(D_out,       requires_grad=True)

# --- Оптимизатор ---
optimizer = torch.optim.SGD([W, b], lr=0.1)

# --- Цикл обучения ---
for step in range(200):
    # 1. Обнуляем градиенты от прошлого шага
    optimizer.zero_grad()

    # 2. Forward pass: вычисляем предсказание
    pred = x @ W + b                        # (32, 1)

    # 3. Считаем loss (скаляр!)
    loss = ((pred - y) ** 2).mean()

    # 4. Backward pass: autograd считает все градиенты
    loss.backward()

    # 5. Шаг оптимизатора: обновляем W и b
    optimizer.step()

    if step % 50 == 0:
        print(f"step {step:3d}  loss = {loss.detach().item():.4f}")

# step   0  loss = 12.3456
# step  50  loss = 0.0234
# step 100  loss = 0.0001
# step 150  loss = 0.0000
```

Разбор по шагам:

| Шаг | Код | Что происходит |
|-----|-----|----------------|
| 1 | `optimizer.zero_grad()` | Обнуляем `.grad` у W и b; иначе градиенты накопятся |
| 2 | `pred = x @ W + b` | Строится граф вычислений (W и b — листья с `requires_grad`) |
| 3 | `loss = ...mean()` | loss — скаляр; подготовлен к `.backward()` |
| 4 | `loss.backward()` | Backprop по графу: `W.grad` и `b.grad` заполнены |
| 5 | `optimizer.step()` | `W -= lr * W.grad`, `b -= lr * b.grad` (SGD) |

Обрати внимание: в шаге 5 **`optimizer.step()` работает без `torch.no_grad()`** — потому что
`torch.optim.SGD.step()` сам использует `torch.no_grad()` внутри себя. Но если обновляешь
параметры **вручную** (как в заданиях этого модуля), нужно либо самому обернуть в `no_grad`,
либо использовать `with torch.no_grad(): W -= lr * W.grad`.

## Идея 10. `nn.Module` — следующий уровень (карта)

В этом модуле параметры — просто тензоры, граф строится вручную, оптимизатор вызывается
явно. Это хороший способ почувствовать механику. Но реальные сети пишут иначе: вместо
`W = torch.zeros(..., requires_grad=True)` используют **`nn.Module`**.

`nn.Module` — базовый класс PyTorch для любого слоя или модели. Он:
- **хранит параметры** (как `nn.Parameter`, это тензоры с `requires_grad=True`) и
  регистрирует их автоматически,
- предоставляет `parameters()` — итератор по всем параметрам для передачи в оптимизатор,
- инкапсулирует forward-логику в методе `forward()`.

```python
import torch.nn as nn

class LinearModel(nn.Module):
    def __init__(self, D_in, D_out):
        super().__init__()
        # nn.Linear — уже готовый слой: хранит W и b как nn.Parameter
        self.layer = nn.Linear(D_in, D_out)

    def forward(self, x):
        return self.layer(x)

model = LinearModel(4, 1)
optimizer = torch.optim.SGD(model.parameters(), lr=0.1)  # находит все параметры

for step in range(200):
    optimizer.zero_grad()
    loss = ((model(x) - y) ** 2).mean()
    loss.backward()
    optimizer.step()
```

Видишь? Структура **ровно та же**: zero → forward → loss → backward → step. Просто `W` и `b`
спрятаны внутри `model`, а `model.parameters()` отдаёт их оптимизатору. Autograd и граф
работают точно так же, как ты изучил в этом модуле.

Следующие модули курса будут строиться на `nn.Module`. Понимание ручного цикла из этого
модуля — фундамент: ты знаешь, что происходит **внутри** этой абстракции.

## Задания

Файл `torch_intro.py`. Реализуй четыре функции (сигнатуры и контракт — в стабе):

1. `to_tensor(x: ArrayLike) -> torch.Tensor` — превратить вход (список / numpy-массив /
   тензор) в `torch.Tensor` типа **`float32`**. Это «входная дверь»: дальше всё считаем во
   float32, чтобы autograd работал.
2. `grad_of(f: Callable[[torch.Tensor], torch.Tensor], x: torch.Tensor) -> torch.Tensor` —
   вернуть `df/dx` через autograd. `f` принимает тензор и возвращает **скаляр**; нужно
   посчитать градиент по `x` и вернуть тензор той же формы, что `x`. (Не порти исходный `x`:
   работай с копией, у которой включён `requires_grad`.)
3. `linear_forward(x: torch.Tensor, W: torch.Tensor, b: torch.Tensor) -> torch.Tensor` —
   линейный слой `x @ W + b` на тензорах. Формы: `x` — `(N, D_in)`, `W` — `(D_in, D_out)`,
   `b` — `(D_out,)`; результат — `(N, D_out)`. `b` добавляется к каждой строке (broadcasting).
4. `mse_torch(pred: torch.Tensor, y: torch.Tensor) -> torch.Tensor` — среднеквадратичная
   ошибка `mean((pred - y)²)`, возвращает **скалярный** тензор (форма `()`).

### Подсказки

<details><summary>Задание 1 — to_tensor и float32</summary>
Универсальный приём — `torch.as_tensor(x, dtype=torch.float32)`: он принимает список,
numpy-массив и тензор. Можно и `torch.tensor(x, dtype=torch.float32)`, и
`torch.as_tensor(x).float()`. Главное — на выходе `dtype == torch.float32`.
</details>

<details><summary>Задание 2 — как вообще получить df/dx</summary>
Три шага: (1) сделай тензор, по которому дифференцируешь, с `requires_grad=True`; (2) посчитай
скаляр `y = f(x)`; (3) вызови `y.backward()` и верни `.grad`. Чтобы не менять исходный `x`,
возьми `xr = x.clone().detach().requires_grad_(True)` — это «свежая» копия с включённым
отслеживанием.
</details>

<details><summary>Задание 2 — самопроверка на бумаге</summary>
Возьми `f(x) = (x**2).sum()`. Тогда `df/dxᵢ = 2·xᵢ`. Для `x = [1, 2, 3]` ожидаешь
`[2, 4, 6]`. Если твоя функция это вернула — autograd подключён правильно. Именно так и
устроен один из тестов.
</details>

<details><summary>Задание 3 — формы и broadcasting</summary>
`x @ W` даёт форму `(N, D_out)`. `b` имеет форму `(D_out,)` — torch сам «растянет» его по N
строкам (broadcasting), так что просто `x @ W + b`. Если формы не сходятся — перепроверь, что
`W` это `(D_in, D_out)`, а не наоборот.
</details>

<details><summary>Задание 4 — почему скаляр</summary>
`(pred - y) ** 2` — тензор той же формы, что `pred`. Чтобы получить ОДНО число (loss), усредни
все элементы: `.mean()`. Результат — тензор формы `()` (скаляр), от него потом и зовут
`.backward()`.
</details>
