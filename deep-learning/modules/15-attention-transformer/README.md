# Модуль 15 — Внимание и трансформеры

Большая идея модуля: до сих пор сеть обрабатывала каждый объект **по
отдельности** (MLP) или смотрела на соседей через фиксированное окно (CNN). Но
в тексте смысл слова зависит от **других слов**, иногда далеко стоящих: в фразе
«банк реки» и «банк выдал кредит» слово «банк» означает разное — и понять это
можно только посмотрев на соседей. **Внимание (attention)** — это механизм,
который позволяет каждой позиции *самой решить*, на какие другие позиции
смотреть и насколько сильно. Из этого простого механизма собран **трансформер**
— архитектура, на которой работают все современные большие языковые модели.

В этом модуле ты соберёшь сердце трансформера своими руками: scaled dot-product
attention, причинную маску, многоголовое внимание и полный блок трансформера.
Никаких `nn.MultiheadAttention` из коробки — только базовые операции torch,
чтобы ты понимал каждую строчку.

> Как всегда в этом курсе — всё на **CPU**, тензоры крошечные, сиды зафиксированы.
> Идея от размера не зависит: научишься на игрушечных тензорах — тот же код
> поедет на GPT-масштабе.

## Идея 1. Внимание: query, key, value

Представь библиотеку. Ты приходишь с **запросом** (query) — «что-то про реки».
У каждой книги есть **ключ** (key) — короткое описание темы. Ты сравниваешь свой
запрос с ключами всех книг, находишь похожие и берёшь их **содержимое** (value).
Внимание устроено ровно так: три роли — query, key, value.

- **query (q)** — «что я ищу» (одна штука на каждую позицию-получатель);
- **key (k)** — «чем я могу быть полезен» (по одной на каждую позицию-источник);
- **value (v)** — «что я отдам, если меня выберут».

Механика в три шага:
1. **Похожесть.** Для запроса считаем скалярное произведение с каждым ключом:
   `score = q · k`. Больше произведение — больше похожи (ключ «отвечает» запросу).
2. **Веса.** Прогоняем все scores через **softmax** — получаем неотрицательные
   веса, суммирующиеся в 1. Это «доля внимания» на каждый источник.
3. **Смесь.** Берём взвешенную сумму значений: `out = Σ weightᵢ · vᵢ`.

Формы словами. Пусть `T` позиций, размер ключа `d_k`, размер значения `d_v`:
- `q: (T, d_k)`, `k: (T, d_k)`, `v: (T, d_v)`;
- матрица похожестей `scores = q @ kᵀ` имеет форму `(T, T)` — «кто на кого
  смотрит»;
- веса `attn` тоже `(T, T)`, каждая **строка** суммируется в 1;
- результат `out = attn @ v` имеет форму `(T, d_v)`.

В **self-attention** (само-внимание) q, k, v получают из **одного и того же**
входа `x` тремя обучаемыми линейными слоями. То есть последовательность смотрит
сама на себя: каждая позиция собирает информацию со всех остальных.

**ПОЧЕМУ это сильнее свёртки?** У внимания нет фиксированного окна: позиция 1
может напрямую посмотреть на позицию 1000 за **один** шаг. Связь между далёкими
словами — мгновенная, а не «по цепочке соседей».

## Идея 2. Scaled dot-product attention и зачем делить на √d_k

Полная формула одного внимания:

```
Attention(Q, K, V) = softmax( Q Kᵀ / √d_k ) · V
```

Единственное, чего ещё не было — деление на `√d_k`. Разберёмся, **зачем**.

`score = q · k = Σ qᵢ kᵢ` — сумма `d_k` слагаемых. Если компоненты q и k —
случайные числа масштаба ~1, то сумма `d_k` таких произведений имеет разброс
порядка `√d_k`. То есть **чем больше размерность, тем крупнее по модулю
получаются scores**. Для `d_k = 64` они легко вырастают до ±8 и больше.

А теперь вспомним softmax: на больших по модулю входах он **насыщается** —
почти весь вес уходит в один максимум, распределение становится почти one-hot.
Это плохо вдвойне: (1) внимание «зацикливается» на одной позиции; (2) в зоне
насыщения **градиент softmax близок к нулю** — сеть перестаёт учиться.

Деление на `√d_k` возвращает scores к масштабу ~1 **независимо** от размерности.
Softmax остаётся «мягким», градиенты живые.

Маленький численный пример. Пусть `q = k = [1, 1, …, 1]` длины `d_k`. Тогда
`q · k = d_k`. Для `d_k = 64` это `64` — гигантское значение для softmax. После
деления на `√64 = 8` получаем `8` — уже разумно. Вот зачем именно **корень**:
он ровно компенсирует рост разброса суммы.

Формы при батче: на практике добавляют ведущие оси (батч, головы), а внимание
считают по двум последним. То есть `q,k,v: (..., T, d)` — операция работает
поверх любых ведущих осей через broadcasting. Транспонируем именно две последние
оси: `k.transpose(-2, -1)`.

## Идея 3. Причинная (causal) маска — не подглядывать в будущее

Языковая модель учится **предсказывать следующий токен** по предыдущим. Значит,
когда мы считаем представление позиции `t`, ей нельзя видеть позиции `t+1, t+2,
…` — иначе она просто подсмотрит ответ. На обучении вся последовательность
известна сразу, поэтому запрет надо наложить **искусственно** — это и есть
**causal mask**.

Идея реализации проста: в матрице scores `(T, T)` зануляем веса на будущих
позициях. «Зануляем» правильно — ставим score в `-∞` **до** softmax, тогда после
softmax вес там станет `e^(-∞) = 0`, а оставшаяся масса честно распределится по
прошлому (строка по-прежнему суммируется в 1).

Маска — булева матрица `(T, T)`, где `True` означает «**запрещено**». Запрещаем
строго будущее: элемент `[i, j]` запрещён, когда `j > i`. Для `T = 3`:

```
        j=0    j=1    j=2
 i=0 [ False,  True,  True ]   позиция 0 видит только себя
 i=1 [ False, False,  True ]   позиция 1 видит 0 и 1
 i=2 [ False, False, False]   позиция 2 видит всех
```

Это в точности **верхний треугольник без диагонали** — то, что даёт
`torch.triu(ones, diagonal=1)`. Главная диагональ (`j = i`, «смотреть на себя»)
и низ (`j < i`, прошлое) — разрешены.

**ПОЧЕМУ это критично:** без маски модель на обучении читерит, подглядывая в
будущее, и показывает прекрасный loss — а на генерации, где будущего ещё нет,
разваливается. Causal-маска заставляет учиться честно: предсказывать вперёд,
зная только прошлое.

Проверить просто: после softmax под маской вес `attn[i, j]` для `j > i` должен
быть ~0. Ровно это и проверяет один из тестов модуля.

## Идея 4. Многоголовое внимание и блок трансформера

**Многоголовое внимание (multi-head attention, MHA).** Одно внимание усредняет
много связей в один набор весов. Но между словами связей много **разного типа**:
синтаксис, кореференция, темы. Решение — запустить несколько вниманий
параллельно. Делим `d_model` на `n_heads` голов размера `d_head = d_model /
n_heads`, каждая голова считает своё внимание в своём подпространстве, затем
выходы **склеиваем** обратно в `d_model` и пропускаем через выходную проекцию.

Формы по шагам (`B` — батч, `T` — длина, `H = n_heads`, `d_head = d_model / H`):
- вход `x: (B, T, d_model)`;
- проекции `q, k, v: (B, T, d_model)` — три `nn.Linear(d_model, d_model)`;
- «режем» на головы: `view(B, T, H, d_head).transpose(1, 2)` → `(B, H, T,
  d_head)`. Теперь внимание считается отдельно по каждой голове (оси B и H —
  ведущие, broadcasting делает остальное);
- внутри каждой головы — наш `scaled_dot_product_attention` с causal-маской,
  выход `(B, H, T, d_head)`;
- «склеиваем» обратно: `transpose(1, 2).reshape(B, T, d_model)`;
- выходная проекция `w_o`: `(B, T, d_model)`.

Заметь: причинная маска `(T, T)` одна на всех — она одинакова для каждой головы
и каждого объекта батча, поэтому строим её один раз и полагаемся на broadcasting.

**Блок трансформера.** MHA — это «обмен информацией между позициями». Но нужна
ещё и «обработка каждой позиции по отдельности» — за это отвечает маленький
**MLP** (две линейные с нелинейностью между ними, обычно скрытый размер `4 ·
d_model`). Эти два подблока заворачивают в два приёма, без которых глубокие сети
не учатся:

- **Остаточная связь (residual):** `x = x + Подблок(x)`. Градиент получает
  «короткий путь» в обход подблока — поэтому стопку из десятков блоков можно
  обучать без затухания градиента.
- **LayerNorm:** нормирует активации, стабилизируя обучение. В современных
  моделях её ставят **перед** подблоком (**pre-norm**) — так глубокие стеки
  ведут себя устойчивее, чем при классическом post-norm.

Итоговая схема одного блока (pre-norm):

```
x = x + MHA( LayerNorm₁(x) )      # обмен информацией между позициями + residual
x = x + MLP( LayerNorm₂(x) )      # обработка каждой позиции + residual
```

Форма на входе и выходе одинакова: `(B, T, d_model) → (B, T, d_model)`. Именно
поэтому блоки можно складывать **стопкой**: выход одного — вход следующего. Стопка
таких блоков плюс эмбеддинги и выходная голова — это и есть GPT.

**ПОЧЕМУ сохраняется форма:** одинаковая форма входа и выхода — не случайность,
а инженерное удобство: однородные блоки складываются как кубики LEGO, без
переходников. Глубину сети меняешь, просто добавляя блоки.

## Задания

Файл `attention.py`. Реализуй две функции и два класса (сигнатуры и контракт —
в стабе):

1. `scaled_dot_product_attention(q, k, v, mask=None) -> (out, attn_weights)` —
   базовое внимание `softmax(q kᵀ / √d_k) v`. Работает поверх любых ведущих осей
   (батч, головы). При заданной `mask` (где `True` = запрет) ставит в этих
   позициях `-inf` до softmax. Возвращает кортеж: результат и веса внимания.
2. `causal_mask(T) -> torch.Tensor` — булева маска формы `(T, T)`, где `True`
   запрещает смотреть в будущее (`j > i`).
3. `class MultiHeadAttention(nn.Module)` — `__init__(d_model, n_heads)`,
   `forward(x) -> (B, T, d_model)`. Self-attention на `n_heads` головах с
   причинной маской. Хранит проекции `w_q, w_k, w_v, w_o`.
4. `class TransformerBlock(nn.Module)` — `__init__(d_model, n_heads, d_ff=None)`,
   `forward(x) -> (B, T, d_model)`. Pre-LayerNorm: `x + MHA(ln1(x))`, затем
   `x + MLP(ln2(x))`. По умолчанию `d_ff = 4 · d_model`.

### Подсказки

<details><summary>Задание 1 — scaled_dot_product_attention по шагам</summary>

`d_k = q.shape[-1]`. Скоры: `scores = (q @ k.transpose(-2, -1)) / math.sqrt(d_k)`
— форма `(..., T_q, T_k)`. Если `mask` задан, занули запрещённое **до** softmax:
`scores = scores.masked_fill(mask, float('-inf'))`. Веса: `attn =
torch.softmax(scores, dim=-1)` (нормируем по последней оси — по ключам). Результат:
`out = attn @ v`. Верни `(out, attn)`.
</details>

<details><summary>Задание 1 — почему именно √d_k, а не d_k</summary>

Скор — сумма `d_k` произведений; её разброс растёт как `√d_k`, а не как `d_k`.
Поэтому компенсируем именно корнем, чтобы вернуть масштаб к ~1. Деление на сам
`d_k` пере-сожмёт scores, softmax станет почти равномерным.
</details>

<details><summary>Задание 2 — causal_mask одной строкой</summary>

`torch.triu(torch.ones(T, T), diagonal=1)` даёт единицы строго выше диагонали
(будущее). Приведи к bool: `.bool()`. Получишь `True` ровно там, где `j > i`.
Проверь на `T = 3`, что вышла обещанная в README матрица.
</details>

<details><summary>Задание 3 — резка на головы и формы</summary>

`B, T, _ = x.shape`. После `q = self.w_q(x)` (форма `(B, T, d_model)`) режем:
`q = q.view(B, T, self.n_heads, self.d_head).transpose(1, 2)` → `(B, H, T,
d_head)`. То же для k и v. Применяешь `scaled_dot_product_attention(q, k, v,
mask)` — оси B и H броадкастятся сами. Склейка обратно: `out.transpose(1,
2).reshape(B, T, self.d_model)`, затем `self.w_o(...)`. Маску строй
`causal_mask(T)` и при желании перенеси на устройство x: `.to(x.device)`.
</details>

<details><summary>Задание 3 — d_model должен делиться на n_heads</summary>

`d_head = d_model // n_heads`, и нужно `d_model % n_heads == 0`, иначе головы не
сложатся обратно в `d_model`. Полезно проверить это в `__init__` через `assert`.
</details>

<details><summary>Задание 4 — порядок ops в блоке (pre-norm)</summary>

Два подблока с residual:
`x = x + self.attn(self.ln1(x))`, потом `x = x + self.mlp(self.ln2(x))`.
MLP собери как `nn.Sequential(nn.Linear(d_model, d_ff), nn.GELU(),
nn.Linear(d_ff, d_model))`, где `d_ff = 4 * d_model`, если не передали явно.
Обрати внимание: norm **перед** подблоком, а складываем с **исходным** x (а не с
нормированным) — это и есть pre-norm residual.
</details>

<details><summary>Задание 4 — почему форма не меняется</summary>

LayerNorm, MHA и MLP все возвращают `(B, T, d_model)`, а residual `x + ...` форму
не трогает. Поэтому выход блока той же формы, что вход — и блоки складываются
стопкой без переходников.
</details>
