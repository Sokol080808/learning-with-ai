## Что мы строим

**Char-level language model** — модель, которая работает не со словами, а с
**отдельными символами**. Словарь — это просто множество уникальных символов
текста (буквы, пробелы, знаки). Модель получает кусочек текста и для **каждой**
позиции предсказывает, какой символ идёт **следующим**.

Пример (учим на строке `"hello world"`):
- вход (контекст):  `h e l l o _ w o r l`
- цель (сдвиг на 1): `e l l o _ w o r l d`

Для позиции с `h` правильный ответ — `e`; для `e` — `l`; и так далее. Каждая
позиция учит свой переход «текущий символ → следующий». Обучившись на маленьком
тексте, модель ловит его статистику (какие символы за какими идут) и умеет
**генерировать** похожий текст: даёшь ей затравку, она по одному символу
дописывает продолжение.

Почему именно символы, а не слова? Словарь крошечный (десятки символов вместо
десятков тысяч слов), не нужен сложный токенизатор, и всё помещается на CPU.
Идея языковой модели от этого не меняется ни на грамм.

## Архитектура (та же, что у GPT)

```
        idx: (B, T)  — индексы символов
          │
   ┌──────┴───────────────────────────┐
   │ token embedding   (vocab → d)     │   «что за символ»
   │ positional embedding (T → d)      │   «на каком он месте»
   └──────┬───────────────────────────┘
          │  сумма: x = tok_emb + pos_emb        (B, T, d)
          │
     ┌────┴────┐
     │ Block 1 │  causal self-attention + MLP, оба с LayerNorm и residual
     │ Block 2 │
     │   ...   │   ← N блоков трансформера (стопкой)
     │ Block N │
     └────┬────┘
          │
   ┌──────┴───────┐
   │  LayerNorm   │   финальная нормировка
   │  Linear → V  │   «голова»: из d в словарь vocab_size
   └──────┬───────┘
          │
      logits: (B, T, vocab_size)   — для каждой позиции распределение
                                      по следующему символу
```

Узнаёшь? Это ровно `module 15` (внимание + блок трансформера), обёрнутый в
эмбеддинги и линейную голову. Token embedding отвечает на вопрос «**что** за
символ», positional embedding — «на **каком месте** он стоит» (внимание само по
себе порядок не различает, позицию надо добавить явно). Дальше — стопка блоков,
финальный LayerNorm и проекция в словарь: для каждой позиции получаем логиты —
ненормированные «баллы» на каждый возможный следующий символ.

Формы словами (`B` — батч, `T` — длина контекста = `block_size`, `d` —
`d_model`, `V` — `vocab_size`):
- вход `idx: (B, T)` — целые индексы символов;
- после эмбеддингов `x: (B, T, d)`;
- после блоков и головы `logits: (B, T, V)`;
- цель `targets: (B, T)` — те же индексы, сдвинутые на 1 (следующий символ).

Лосс — кросс-энтропия между логитами и целевыми индексами, усреднённая по всем
`B·T` позициям. Снижается лосс → модель всё лучше угадывает следующий символ.

## Майлстоуны

Делай по шагам. Каждый майлстоун завершается зелёным набором тестов — не беги
вперёд, пока предыдущий красный. Привязка к модулям курса — в скобках.

### Майлстоун 1 — токенизатор и данные  *(модули 01, 17)*

Файлы: `nanolm/tokenizer.py`, `nanolm/data.py`.

- `CharTokenizer` — строит словарь из текста (отсортированное множество
  уникальных символов), умеет `encode(text) -> list[int]` и `decode(ids) -> str`,
  отдаёт `vocab_size`. Главное свойство: `decode(encode(text)) == text`
  (round-trip) для текста из обученного алфавита.
- `get_batch(data, block_size, batch_size, seed)` — нарезает из длинного тензора
  индексов случайные обучающие пары `(x, y)`, где `y` — это `x`, сдвинутый на
  один символ вперёд. Формы: `x, y: (batch_size, block_size)`. Сид фиксирует
  выбор позиций — батч воспроизводим.

Зелёным становится `tests/test_tokenizer.py`.

### Майлстоун 2 — модель из блоков трансформера  *(модуль 15)*

Файл: `nanolm/model.py`.

- `TransformerLM(nn.Module)` — собери архитектуру выше: `nn.Embedding` для токенов
  и для позиций, `N` блоков трансформера (**причинное** self-attention + MLP +
  LayerNorm + residual — ровно как в модуле 15), финальный `nn.LayerNorm` и
  `nn.Linear(d_model, vocab_size)`.
- `forward(idx) -> logits` формы `(B, T, vocab_size)`.
- Внимание обязано быть **причинным**: логиты на позиции `t` не должны зависеть от
  символов на позициях `> t`. Это проверяется тестом (меняем будущий токен —
  прошлые логиты не шелохнулись).

Зелёным становятся тесты форм и причинности в `tests/test_model.py`.

### Майлстоун 3 — обучение  *(модули 12, 16)*

Файлы: `nanolm/model.py` (метод не нужен — лосс считаем снаружи), `train.py`.

- Обучающий цикл: `get_batch` → `forward` → кросс-энтропия → `backward` →
  шаг оптимизатора (Adam/AdamW). Это связка из модулей 12 (обучение MLP) и
  16 (практика обучения: оптимизатор, lr, цикл).
- На крошечном seeded корпусе несколько десятков шагов должны **заметно** снижать
  лосс (приёмочный тест требует `итоговый < 0.5 · начального`).
- `train.py` — **готовый рабочий скрипт** (его писать не нужно): он берёт твои
  `CharTokenizer`, `TransformerLM`, `get_batch`, гоняет цикл обучения на
  встроенном маленьком корпусе и в конце что-то генерирует. Запусти его, когда
  модель и данные готовы.

Зелёным становится тест обучения в `tests/test_train.py`.

### Майлстоун 4 — генерация и стретч-цели

Файл: `nanolm/model.py`.

- `generate(idx, max_new_tokens) -> idx` — авторегрессионная генерация: на каждом
  шаге берём логиты **последней** позиции, превращаем в вероятности (softmax),
  **сэмплируем** следующий символ, дописываем его к `idx` и повторяем. Не забудь
  про `block_size`: если контекст длиннее, обрезай его справа (модель видит
  только последние `block_size` символов).
- Запусти `train.py` целиком и посмотри, как модель дописывает текст. На таком
  крошечном корпусе она просто запоминает его — это нормально и наглядно.

Стретч-цели (по желанию, для понимания, без новых тестов):
- **temperature** в `generate`: дели логиты на `temperature` перед softmax
  (`<1` — текст «увереннее», `>1` — разнообразнее).
- **top-k сэмплирование**: оставляй только `k` самых вероятных символов.
- увеличь `n_layers` / `d_model` / `block_size` и посмотри на лосс и на текст.
- посчитай число параметров модели и прикинь, куда они уходят.

## Альтернативный капстоун — классификатор крошечных изображений (CNN)

Если язык тебе менее интересен, чем зрение, — вот равноценная альтернатива на
базе **модуля 13 (CNN)**.

Задача: классификатор маленьких синтетических изображений (например, `8×8`
картинки с простыми фигурами — крест/круг/линия — сгенерированные кодом, без
загрузок из интернета). Архитектура: пара свёрточных слоёв (`Conv2d` + ReLU +
пулинг) → flatten → линейная голова на число классов. Обучение и приёмочные
тесты — те же по духу: формы выходов `(B, n_classes)`, лосс — кросс-энтропия,
несколько шагов на seeded игрушечных данных снижают лосс ниже `0.5 · начального`,
а точность на обучающем наборе заметно растёт.

Контракт остаётся прежним: спецификация + приёмочные тесты, архитектуру
проектируешь сам. Если выберешь этот путь — скажи мне, и мы вместе набросаем
тесты-контракты под CNN-капстоун в том же стиле, что лежат здесь под nanoLM.
