# Модуль 13 — Конкурентность и синхронизация

В модуле 10 ты завёл несколько потоков и впервые увидел **гонку данных**: два
потока полезли в общую переменную — и часть работы потерялась. Там лекарством
был мьютекс. Здесь мы идём дальше: научимся не только **защищать** общие данные,
но и **дожидаться** нужного состояния — «подожди, пока появится место»,
«подожди, пока появится элемент». Это сердце почти любой многопоточной
программы: очереди задач, пулы потоков, конвейеры обработки — всё построено на
том, что ты разберёшь в этом модуле.

## Большая идея

Представь маленькое кафе с одним подоконником выдачи, на котором помещается,
скажем, 4 тарелки.

- **Повар** (производитель) готовит блюда и ставит их на подоконник. Если
  подоконник **полон** — повар не швыряет тарелку на пол и не ставит пятую поверх
  четырёх. Он **ждёт**, пока официант что-нибудь заберёт и освободит место.
- **Официант** (потребитель) забирает тарелки с подоконника и несёт гостям. Если
  подоконник **пуст** — официант не уносит воздух. Он **ждёт**, пока повар
  поставит хотя бы одно блюдо.

Подоконник — это **ограниченный буфер**: общее место, у которого есть предел.
Повар и официант работают **одновременно и независимо**, но через подоконник
они согласованы: никто не суетится впустую и никто ничего не теряет.

Зачем это знать. Так устроена огромная доля реального софта. Веб-сервер: одни
потоки принимают запросы и кладут их в очередь, другие достают и обрабатывают.
Видеоплеер: один поток декодирует кадры в буфер, другой выводит на экран. Везде
один и тот же узор — **производитель-потребитель** через ограниченный буфер.
Научишься делать его правильно — поймёшь скелет промышленных систем.

## Идея 1. Состояние гонки и критическая секция

Напомним кратко (подробно было в модуле 10). Когда два потока трогают **общие
данные**, и хотя бы один — пишет, результат может зависеть от того, кто кого
случайно опередил. Это **состояние гонки (race condition)**.

Невинное `count++` для процессора — это три шага: прочитать, прибавить, записать.
Если два потока выполнят их вперемешку, одно обновление потеряется:

```
Поток A: прочитал count = 5
Поток B: прочитал count = 5      <- A ещё не записал!
Поток A: пишет 6
Поток B: пишет 6                 <- затёр результат A, должно было стать 7
```

Лекарство — **критическая секция**: участок кода, внутри которого гарантированно
**только один** поток. Огораживают её **мьютексом** (mutex, *mutual exclusion* —
«взаимное исключение»):

```c
pthread_mutex_lock(&m);     // взял замок; если занят — ждёт у двери
count++;                    // здесь я ОДИН — никто не вмешается
pthread_mutex_unlock(&m);   // отдал замок следующему
```

ПОЧЕМУ это важно для буфера: у нас несколько полей (`count`, индексы, сам
массив), которые меняются **вместе**. Если их трогать без замка, потоки увидят
буфер в полу-изменённом состоянии — каша. Поэтому **любое** обращение к полям
буфера — только под мьютексом.

Подводный камень: держи критическую секцию **короткой**. Пока ты держишь замок,
остальные ждут у двери. Замок — не место «пожить», а место «быстро сделать дело и
уйти».

## Идея 2. Мьютекс и условная переменная

Мьютекса для буфера **мало**. Смотри, в чём загвоздка.

Официант взял замок, заглянул на подоконник — **пусто**. Что ему делать?

- Крутиться в цикле «пусто? пусто? пусто?», **держа замок** — катастрофа: повар
  не сможет взять замок, чтобы поставить тарелку. Вечное ожидание (deadlock).
- Крутиться в цикле, **отпуская и беря замок** заново — работать-то будет, но это
  **busy-wait**: процессор жжёт 100% впустую, проверяя одно и то же. Как стоять у
  подоконника и каждую миллисекунду орать «ну что, готово?!».

Нужен механизм «**усни, а когда появится повод — я тебя разбужу**». Это и есть
**условная переменная (condition variable)**.

```c
pthread_cond_t not_empty;   // «на подоконнике появился элемент»

// ОФИЦИАНТ ждёт, пока буфер не пуст:
pthread_mutex_lock(&m);
while (count == 0) {                       // пока пусто...
    pthread_cond_wait(&not_empty, &m);     // ...усни (и отпусти замок!)
}
/* ... здесь буфер точно не пуст, замок снова у меня ... */
pthread_mutex_unlock(&m);
```

Магия в `pthread_cond_wait`: он делает **три вещи атомарно** —
1) отпускает замок (чтобы повар мог войти и поставить тарелку),
2) усыпляет поток (процессор не тратится),
3) когда поток будят — **снова берёт замок** перед возвратом.

А будит спящего вот это, со стороны повара:

```c
pthread_mutex_lock(&m);
/* ... поставил тарелку, count++ ... */
pthread_cond_signal(&not_empty);  // «эй, официант, появился элемент!»
pthread_mutex_unlock(&m);
```

ПОЧЕМУ `while`, а **не** `if` — это главная ошибка новичков. Проснувшись, поток
**обязан перепроверить** условие, потому что:
- бывает **ложное пробуждение** (spurious wakeup): ОС имеет право разбудить поток
  «просто так», без причины;
- сигнал мог достаться другому потоку, и пока ты просыпался, он уже забрал
  единственный элемент — и снова пусто.

Поэтому: `while (нельзя продолжать) cond_wait(...);`. Проснулся → перепроверил →
если опять рано, уснул снова. С `if` вместо `while` ты словишь редкий, плавающий,
мучительный баг. Запомни это как мантру.

## Идея 3. Семафор — считающий и бинарный

### Что такое семафор

**Семафор** — это целочисленный счётчик разрешений, у которого есть всего две
атомарные операции: **wait** (забрать одно разрешение, если их нет — спать) и
**post** (вернуть одно разрешение и разбудить одного спящего). Дейкстра,
придумавший семафор в 1965 году, назвал их P и V, но в POSIX прижились `wait` и
`post`.

Аналогия: на парковке 4 места и 4 ключа на стене. Заезжаешь — снял ключ
(`wait`); ключей нет — ждёшь у шлагбаума. Уехал — повесил ключ обратно (`post`).
Мьютекс — это, по сути, семафор с начальным счётчиком 1 («один ключ»).

### POSIX-интерфейс: sem_init / sem_wait / sem_post

```c
#include <semaphore.h>

sem_t s;

/* Инициализировать. Второй аргумент — 0 (семафор между потоками одного процесса,
 * не между процессами). Третий — начальное значение счётчика. */
sem_init(&s, 0, 4);    // 4 разрешения

/* Уменьшить счётчик на 1. Если уже 0 — заблокироваться и ждать. */
sem_wait(&s);

/* Увеличить счётчик на 1 и разбудить одного ждущего (если есть). */
sem_post(&s);

/* Освободить ресурсы. */
sem_destroy(&s);
```

Заметь: `sem_wait` может быть прерван сигналом (`EINTR`) — в реальном коде
оборачивают в цикл `while (sem_wait(&s) == -1 && errno == EINTR);`. В обучающем
коде это обычно опускают.

### Считающий vs бинарный семафор

| Вид | Начальное значение | Что даёт |
|-----|-------------------|----------|
| **Считающий (counting)** | N > 1 | Ограничивает конкурентный доступ к пулу из N ресурсов |
| **Бинарный (binary)** | 1 | Ведёт себя как мьютекс: только один поток «внутри» |
| **Сигнальный (semaphore as signal)** | 0 | Один поток ждёт (`wait`), другой сигналит (`post`) — аналог `signal` без `lock` |

Пример считающего семафора — ограничение числа одновременных соединений к базе:

```c
sem_t db_slots;
sem_init(&db_slots, 0, 10);  // не более 10 одновременных запросов

// в каждом потоке:
sem_wait(&db_slots);
do_db_query();               // не более 10 потоков здесь одновременно
sem_post(&db_slots);
```

### Производитель-потребитель через семафоры

Это альтернатива мьютекс+cond. Для буфера ёмкости N заводим три объекта:

```c
sem_t empty;   // «сколько пустых слотов»   — начальное значение N
sem_t filled;  // «сколько заполненных»     — начальное значение 0
sem_t mutex;   // защита массива            — бинарный, начальное значение 1

sem_init(&empty,  0, N);
sem_init(&filled, 0, 0);
sem_init(&mutex,  0, 1);

// Производитель:
sem_wait(&empty);       // есть ли место? если нет — спим
sem_wait(&mutex);       // войти в критическую секцию
buf[tail] = item;
tail = (tail + 1) % N;
sem_post(&mutex);       // выйти из критической секции
sem_post(&filled);      // сообщить потребителю: появился элемент

// Потребитель (симметрично):
sem_wait(&filled);      // есть ли элемент?
sem_wait(&mutex);
item = buf[head];
head = (head + 1) % N;
sem_post(&mutex);
sem_post(&empty);       // сообщить производителю: появилось место
```

**Критический порядок!** Заметь: `sem_wait(&empty)` всегда **до**
`sem_wait(&mutex)`, а не наоборот. Если ты возьмёшь mutex первым, а потом
уснёшь на `empty` — ты будешь спать, держа mutex. Производитель захочет взять
mutex, чтобы что-то положить и разбудить тебя — но не сможет. Вечный сон.
Это классический deadlock через неправильный порядок семафоров.

### Мьютекс vs семафор vs условная переменная — в чём разница?

Это вопрос, который путает почти всех. Вот главные отличия:

**Мьютекс** — это «замок с владельцем». Только тот поток, который взял замок,
обязан его отпустить. Семантика строгая: нельзя отпустить чужой мьютекс. Это
инструмент для **взаимного исключения** («только я трогаю данные»).

**Семафор** — это «счётчик без владельца». Один поток делает `wait`, другой —
`post`. Это инструмент для **сигнализации** («я готов, можешь продолжать») и для
**ограничения ресурса** («не более N одновременно»). Семафором не всегда
защищают данные — им часто просто говорят «пора».

**Условная переменная** всегда работает **в паре с мьютексом**. Она позволяет
ждать произвольного предиката (не просто «счётчик > 0», а «буфер не пуст»,
«задача завершена» и т.д.). В отличие от семафора, cond_signal не «накапливает»
сигналы: если никто не спит на condvar в момент signal — сигнал теряется. Это
не баг, а особенность: ты защищён мьютексом и проверяешь условие сам.

| | Мьютекс | Считающий семафор | Condvar |
|---|---|---|---|
| **Для чего** | Взаимное исключение | Ограничение / сигнал | Ожидание предиката |
| **Нужен ли отдельный lock?** | Сам и есть lock | Нет (или добавить) | Да, обязан |
| **Кто отпускает?** | Тот, кто взял | Любой поток | Автоматически в wait |
| **Накапливает сигналы?** | — | Да (счётчик растёт) | Нет |
| **Spurious wakeup?** | Нет | Нет | Да → нужен while |

Практическое правило:
- Хочешь защитить данные от параллельного доступа — **мьютекс**.
- Хочешь ограничить число одновременных потоков — **считающий семафор**.
- Хочешь ждать произвольного состояния — **мьютекс + condvar**.
- Хочешь «разбудить кого-то позже» из другого потока — **сигнальный семафор** (начальное значение 0).

В этом модуле мы используем **мьютекс + две условные переменные**: это нагляднее
и точно показывает идею «ждать состояния». Но семафорная версия выше — равно
корректная альтернатива, и знать её полезно.

## Идея 4. Взаимная блокировка (deadlock) — четыре условия Коффмана

### Что такое deadlock

**Взаимная блокировка (deadlock)** — когда потоки встали намертво: каждый ждёт
ресурса, который держит другой. Никто не продвигается. Программа жива (процессы
не упали), но делает ничего.

Классика — два замка, взятые в разном порядке:

```
Поток A: взял замок 1, хочет замок 2 → ждёт
Поток B: взял замок 2, хочет замок 1 → ждёт
```

Ни один не отпустит своё, пока не получит чужое. Круговое ожидание. Бытовая
аналогия: два человека в узком коридоре, каждый ждёт, что другой отойдёт первым.

### Четыре условия Коффмана

В 1971 году Коффман и соавторы доказали: deadlock **возможен тогда и только
тогда**, когда выполняются все четыре условия одновременно. Убери любое одно —
deadlock невозможен.

1. **Взаимное исключение (Mutual exclusion).**
   Ресурс нельзя использовать двумя потоками одновременно. Мьютекс по определению
   такой: только один держатель. _Пример_: один принтер — один документ за раз.

2. **Удержание и ожидание (Hold and wait).**
   Поток уже держит хотя бы один ресурс и при этом ждёт ещё один. Именно здесь
   образуется «цепочка»: держу одно, хочу другое.

3. **Невытесняемость (No preemption).**
   Ресурс нельзя отнять у потока принудительно — только он сам отпустит. Мьютекс
   именно такой: ОС не может «выдернуть» замок у потока. _(Если бы могла — было
   бы вытесняющее освобождение, deadlock бы не возникал.)_

4. **Круговое ожидание (Circular wait).**
   Существует цикл: поток A ждёт поток B, B ждёт C, ..., Z ждёт A. Именно этот
   цикл — последнее звено. Без него цепочка зависимостей конечна и разрешится.

### Как нарушить каждое условие — стратегии предотвращения

Все четыре условия **нужны одновременно**, значит достаточно разрушить хотя бы
одно:

**Нарушить взаимное исключение** — сделать ресурс читаемым несколькими потоками
(read/write lock). Только для ресурсов, где параллельное чтение безопасно.

**Нарушить удержание и ожидание** — брать все замки разом или не брать ничего
(«взять всё или отступить»). Вариант: `pthread_mutex_trylock` — попробовать взять
второй замок; если не вышло — отпустить первый и попробовать снова. Минус:
возможен **livelock** (все постоянно пытаются и уступают, никто не продвигается).

```c
// «Взять всё или отступить»
while (1) {
    pthread_mutex_lock(&m1);
    if (pthread_mutex_trylock(&m2) == 0) break;  // оба взял — готово
    pthread_mutex_unlock(&m1);                   // не вышло — отпустить
    // небольшая пауза или yield перед следующей попыткой
}
```

**Нарушить невытесняемость** — принудительно освобождать замки через таймаут
(`pthread_mutex_timedlock`). Если не взял за T секунд — отпустить всё и
повторить. Ценой — усложнение логики.

**Нарушить круговое ожидание** — самый практичный способ. Введи **глобальный
порядок** на все замки (например, по адресу или по номеру) и бери их всегда
строго по возрастанию:

```c
// ПРАВИЛО: всегда сначала m1 (меньший адрес), потом m2 (больший)
// Поток A:
pthread_mutex_lock(&m1);   // сначала m1
pthread_mutex_lock(&m2);   // потом m2
// ... работа ...
pthread_mutex_unlock(&m2);
pthread_mutex_unlock(&m1);

// Поток B: ТОЖЕ сначала m1, потом m2 — круг не замкнётся
pthread_mutex_lock(&m1);
pthread_mutex_lock(&m2);
// ...
```

Если все потоки берут замки в одном порядке — круговое ожидание
**математически невозможно**: всегда найдётся поток, которому не нужен «следующий»
замок, и он завершит работу, освободив замки для остальных.

### Deadlock vs livelock vs starvation

Пока мы здесь — три важных термина, которые легко перепутать:

- **Deadlock**: потоки стоят и не двигаются совсем.
- **Livelock**: потоки «активно» пытаются, но бесконечно уступают друг другу —
  движение есть, прогресса нет. Аналог: два человека в коридоре, вечно
  синхронно уступающие — оба двигаются, никто не прошёл.
- **Starvation (голодание)**: поток технически может получить ресурс, но другие
  постоянно опережают. Один бедняга никогда не попадает на CPU или не берёт
  замок — не потому что заблокирован, а потому что всегда проигрывает
  планировщику.

### Deadlock в нашем буфере

В нашем буфере **один** мьютекс — классического двухзамкового deadlock'а нет.
Но «зависнуть навечно» всё равно можно:

- Потребитель уснул на `not_empty` (буфер пуст), но производитель забыл сделать
  `signal` — потребитель спит вечно. Это называется **missed wakeup**.
- Если бы условие проверялось в `if`, а не в `while`, и произошло ложное
  пробуждение — поток прочитал бы из пустого буфера.

Поэтому дисциплина строгая: положил → `signal(not_empty)`; забрал →
`signal(not_full)`. Никаких исключений.

## Идея 5. Классическая задача «производитель-потребитель» на ограниченном буфере

Соберём всё вместе. Это и есть твоё задание.

Буфер — это **массив фиксированного размера**, по которому два индекса бегают «по
кругу» (**кольцевой буфер**, ring buffer): дошли до конца — вернулись в начало.
Зачем кольцо: не надо двигать данные. Берём из «головы», кладём в «хвост», оба
индекса по модулю ёмкости ходят по кругу — массив используется снова и снова.

```
ёмкость = 4, в буфере лежит 3 элемента:

индекс:   0    1    2    3
        [ . ][ A ][ B ][ C ]
              ^head          ^... tail смотрит на 0 (по кругу)
```

- `head` — откуда **читаем** (следующий `get`).
- `tail` — куда **пишем** (следующий `put`).
- `count` — сколько сейчас лежит (чтобы отличать «пусто» от «полно», ведь при
  кольце `head == tail` бывает и когда пусто, и когда полно).

Сдвиг по кругу: `tail = (tail + 1) % capacity;` — дойдя до края, индекс сам
вернётся в 0.

Логика на словах:

```
put(item):                          get():
  lock                                lock
  while (count == capacity)           while (count == 0)
      wait(not_full)                      wait(not_empty)
  data[tail] = item                   item = data[head]
  tail = (tail+1) % capacity          head = (head+1) % capacity
  count++                             count--
  signal(not_empty)                   signal(not_full)
  unlock                              unlock
                                      return item
```

Это симметрия двух идей: каждый **проверяет своё условие в while**, делает дело
под замком и **будит противоположную сторону**. Положил элемент — разбуди того,
кто ждёт элемент. Забрал — разбуди того, кто ждёт место.

ПОЧЕМУ это красиво и надёжно: ни busy-wait (потоки спят, а не жгут процессор), ни
потерь (всё под замком), ни рассинхрона порядка (FIFO держится сам собой —
читаем из головы то, что клали в хвост).

## Задания

Файл `src/syncbuf.c` (интерфейс — `include/syncbuf.h`). Тип `BoundedBuffer`
объявлен в заголовке полностью — поля ты заполняешь в `bb_init`, снаружи в них
лазить не надо. Тебе понадобится `#include <pthread.h>` (он уже подтянут через
заголовок) и `#include <stdlib.h>` для `malloc`/`free`. Линковка с потоками в
курсе уже настроена.

Минимум API условных переменных (детали — в `man pthread_cond_wait`):

```c
pthread_cond_t c;
pthread_cond_init(&c, NULL);              // создать
pthread_cond_wait(&c, &m);               // отпустить m, уснуть; проснувшись — снова взять m
pthread_cond_signal(&c);                 // разбудить ОДНОГО ждущего на c
pthread_cond_destroy(&c);                // освободить в конце
```

Реализуй четыре функции:

1. `void bb_init(BoundedBuffer *b, int capacity)` — выделить массив `data` на
   `capacity` элементов, обнулить `head`, `tail`, `count`, запомнить `capacity`,
   инициализировать мьютекс и **обе** условные переменные. Вызывается один раз
   до любых `put`/`get`.

2. `void bb_put(BoundedBuffer *b, int item)` — положить `item`. Под мьютексом:
   **пока** буфер полон (`count == capacity`) — ждать на `not_full`; затем
   записать в `tail`, сдвинуть `tail` по кругу, увеличить `count`, разбудить
   `not_empty`. Порядок строго **FIFO**.

3. `int bb_get(BoundedBuffer *b)` — забрать элемент и вернуть его. Под мьютексом:
   **пока** буфер пуст (`count == 0`) — ждать на `not_empty`; затем прочитать из
   `head`, сдвинуть `head` по кругу, уменьшить `count`, разбудить `not_full`,
   вернуть прочитанное.

4. `void bb_destroy(BoundedBuffer *b)` — `free(data)`, уничтожить мьютекс и обе
   условные переменные.

Запусти тесты:

```bash
./caos/run.sh 13
```

Сначала всё красное: заглушки ничего не делают, а `bb_get` всегда возвращает `0`.
Тесты гоняют производителя и потребителя в **разных потоках** через буфер
ёмкости 4 (и даже 1) — так блокировки реально срабатывают: буфер постоянно то
полон, то пуст. Проверяется, что получены **все** элементы, **в порядке FIFO** и
что **сумма** совпадает. Делай функции по очереди.

### Подсказки (открывай, только если застрял)

<details><summary>С чего начать — bb_init</summary>

Это самая простая функция, но без неё ничего не заработает. Нужно:

```c
b->data     = malloc(sizeof(int) * (size_t)capacity);
b->capacity = capacity;
b->head = b->tail = b->count = 0;
pthread_mutex_init(&b->mutex, NULL);
pthread_cond_init(&b->not_full, NULL);
pthread_cond_init(&b->not_empty, NULL);
```

Никаких потоков тут не запускается — это делает тест. Ты лишь готовишь поле.
</details>

<details><summary>bb_put — скелет</summary>

```c
pthread_mutex_lock(&b->mutex);
while (b->count == b->capacity) {            // полно — ждём место
    pthread_cond_wait(&b->not_full, &b->mutex);
}
b->data[b->tail] = item;
b->tail = (b->tail + 1) % b->capacity;       // сдвиг по кругу
b->count++;
pthread_cond_signal(&b->not_empty);          // разбудили того, кто ждёт элемент
pthread_mutex_unlock(&b->mutex);
```

Обрати внимание: `signal` делаем **до** `unlock` — это нормально и безопасно.
</details>

<details><summary>bb_get — скелет (симметрично put)</summary>

```c
pthread_mutex_lock(&b->mutex);
while (b->count == 0) {                       // пусто — ждём элемент
    pthread_cond_wait(&b->not_empty, &b->mutex);
}
int item = b->data[b->head];
b->head = (b->head + 1) % b->capacity;        // сдвиг по кругу
b->count--;
pthread_cond_signal(&b->not_full);            // разбудили того, кто ждёт место
pthread_mutex_unlock(&b->mutex);
return item;
```
</details>

<details><summary>Почему именно while, а не if (частая ошибка)</summary>

Замени в `bb_get` `while (b->count == 0)` на `if (b->count == 0)` и прогони тесты
несколько раз. Рано или поздно поймаешь сбой: поток проснулся по сигналу или
«просто так», условие уже неверно (элемент перехватили), а он лезет читать из
пустого буфера. С `while` поток перепроверит условие и при необходимости уснёт
снова. Это та самая защита от ложных пробуждений — увидеть её цену полезно
своими глазами.
</details>

<details><summary>Тест то зелёный, то красный / висит</summary>

Если тест **зависает** — скорее всего, ты будишь не ту условную переменную (или
не будишь вовсе): кто-то уснул и его никто не разбудит. Сверься: в `put`
сигналим `not_empty`, в `get` — `not_full`. Если тест **иногда** падает — почти
наверняка где-то `if` вместо `while`, либо обращение к полю буфера **без**
замка. На «плавающих» багах и куётся системный программист.
</details>
