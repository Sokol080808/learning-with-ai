# Модуль 10 — Процессы и потоки

До этого момента твоя программа была одиноким работником: она делала шаги
по очереди, один за другим. Этот модуль — про то, как **запустить несколько
работников сразу** и как они уживаются вместе. Здесь начинается операционная
система: именно ОС решает, кто из программ сейчас работает на процессоре, и
даёт им иллюзию, что они «одни на свете».

## Большая идея

Представь офис.

- **Процесс** — это отдельная *комната* с командой внутри. У комнаты своя мебель,
  свои документы, свой сейф. Команда из соседней комнаты **не может** взять твою
  бумагу со стола — у неё нет доступа в твою комнату. Это изоляция: один процесс
  не лезет в память другого. Когда ты запускаешь браузер и музыкальный плеер — это
  две разные «комнаты», два процесса.
- **Поток** — это *сотрудник* внутри одной комнаты. Сотрудников может быть много,
  и все они работают за **общим столом** в одной комнате: видят одни и те же
  документы, один и тот же сейф. Это и сила, и опасность: общая память — это быстро
  и удобно, но если двое одновременно полезут писать в один документ — будет каша.

Зачем это знать. Любая современная программа, которая «не висит», пока что-то
грузится, — использует потоки или процессы. Игры, серверы, браузеры — всё это
многопоточно. А баги многопоточности (когда «иногда работает, иногда нет») —
самые коварные в профессии. Понять их основу сейчас — значит сэкономить себе
бессонные ночи потом.

## Идея 1. Процесс = программа в исполнении

Файл `program.exe` на диске — это ещё не процесс. Это рецепт, мёртвый текст.
**Процесс** появляется, когда ОС берёт этот рецепт, выделяет ему память и
запускает. У каждого процесса есть своё **адресное пространство** — личный
диапазон адресов памяти, как личная комната.

Ключевой момент: адрес `0x1000` в одном процессе и адрес `0x1000` в другом —
это **разные** ячейки физической памяти. ОС подменяет адреса за кулисами (это
называется виртуальная память, про неё — позже). Поэтому процессы по умолчанию
**не видят** память друг друга. Это защита: упавший плеер не уронит твой браузер.

ПОЧЕМУ это важно: изоляция — фундамент надёжности. Если бы любая программа могла
лезть в память любой другой, один баг рушил бы всю систему. Цена изоляции — за
обмен данными между процессами надо платить (специальные механизмы: каналы,
сокеты, разделяемая память). Внутри одного процесса обмен бесплатный — потому что
память общая.

## Идея 2. Поток = поток исполнения внутри процесса

Внутри процесса есть как минимум один **поток** — «бегунок», который шагает по
коду. Можно завести несколько бегунков, и тогда они шагают по коду
**одновременно** (на разных ядрах процессора — по-настоящему параллельно).

Что потоки **делят** (общее на всю комнату):
- кучу (heap) и глобальные переменные — общий стол и сейф;
- открытые файлы.

Что у каждого потока **своё** (личное место сотрудника):
- свой стек (локальные переменные, цепочка вызовов);
- свой «указатель», на каком шаге кода он сейчас.

```c
// псевдокод идеи: два потока работают с ОДНОЙ переменной total
long total = 0;        // общая, лежит в куче/глобально — видна обоим
// поток A: total += сумма первой половины
// поток B: total += сумма второй половины
// главный поток ждёт обоих и читает total
```

ПОЧЕМУ потоки, а не процессы. Потоки **дешевле** (создать сотрудника быстрее, чем
построить новую комнату) и им **не нужно ничего пересылать** — память уже общая.
Расплата — за общую память надо следить самому (см. Идею 4).

## Идея 3. Как вообще рождаются процессы: fork/exec «на пальцах»

Откуда берётся новый процесс? В мире Unix — двумя шагами.

- **`fork()`** — «клонирование». ОС делает **точную копию** текущего процесса.
  Был один — стало два почти одинаковых: родитель и ребёнок. Аналогия: ты
  ксерокопируешь всю свою комнату вместе с содержимым. Дальше живут раздельно:
  поправил бумагу у себя — у копии не изменилось (комнаты-то разные).

  Хитрый момент: `fork()` возвращается **дважды** — по разу в каждом процессе.
  Родителю он отдаёт номер (PID) ребёнка, ребёнку — `0`. Так одна и та же строка
  кода понимает, кто из них кто:

  ```c
  pid_t pid = fork();
  if (pid == 0) {
      // я — ребёнок
  } else {
      // я — родитель, pid = номер ребёнка
  }
  ```

- **`exec()`** — «переродиться». Процесс выбрасывает свой код и **загружает на его
  место другую программу**. Комната та же, но команда и все документы заменились
  на новые. Возврата из `exec()` нет (если он удался) — старого кода больше не
  существует.

Связка: оболочка (терминал), запуская команду, делает `fork()` (клонирует себя),
а в ребёнке вызывает `exec()` (превращает клон в нужную программу). Так из `bash`
рождается, например, `ls`.

> В заданиях этого модуля мы `fork`/`exec` **не пишем** — это требует Unix и
> аккуратной возни с ожиданием детей. Здесь важно понять *модель*: процессы
> изолированы и порождаются клонированием. А руками поработаем с **потоками** —
> с ними и удобнее, и нагляднее видна главная опасность параллелизма.

## Идея 4. Гонка данных — главный кошмар общей памяти

Два потока делят память. Что плохого может случиться? Возьмём «безобидное»
`counter++`. Для процессора это **не одно** действие, а три:

1. прочитать значение `counter` из памяти в регистр;
2. прибавить 1;
3. записать обратно в память.

Теперь представь, что два потока делают `counter++` почти одновременно, а
счётчик равен `5`:

```
Поток A: читает 5
Поток B: читает 5      <- ещё не записан результат A!
Поток A: 5+1=6, пишет 6
Поток B: 5+1=6, пишет 6   <- затёр результат A
```

Было два инкремента — а счётчик вырос лишь на 1. Одно обновление **потерялось**.
Это и есть **гонка данных (data race)**: результат зависит от того, кто кого
случайно опередил. Самое мерзкое — большую часть запусков «везёт» и всё работает,
а раз в тысячу запусков — нет. Такие баги ловятся месяцами.

Лекарство — **мьютекс** (mutex, от *mutual exclusion* — «взаимное исключение»).
Это замок на дверь сейфа: входить можно только по одному.

```c
pthread_mutex_lock(&m);     // взял замок (если занят — ждёт)
counter++;                  // тут я ОДИН, никто не помешает
pthread_mutex_unlock(&m);   // отдал замок следующему
```

Участок между `lock` и `unlock` — **критическая секция**. Внутри неё гарантированно
один поток. ПОЧЕМУ нельзя «просто посчитать без замка»: потому что без него три
шага инкремента у разных потоков перемешаются — и часть прибавок потеряется. Цена
замка — потоки иногда ждут друг друга у двери; держи критическую секцию короткой.

## Задания

Файл `src/threads.c` (интерфейс — `include/threads.h`). Тебе понадобится
`#include <pthread.h>` — линковка с потоками в курсе уже настроена, ничего
докручивать в сборке не нужно.

Полезный минимум API (детали — в `man pthread_create` и т.п.):

```c
pthread_t t;
pthread_create(&t, NULL, worker, arg); // запустить worker(arg) в новом потоке
pthread_join(t, NULL);                 // дождаться завершения потока t

pthread_mutex_t m;
pthread_mutex_init(&m, NULL);          // создать замок
pthread_mutex_lock(&m); ... pthread_mutex_unlock(&m);
pthread_mutex_destroy(&m);             // освободить замок в конце
```

Функция потока всегда такой формы: `void *worker(void *arg) { ...; return NULL; }`.
Данные внутрь передают через `arg` (обычно — указатель на структуру с аргументами
и местом под результат).

1. `long parallel_sum(const long *arr, size_t n, int num_threads)` — посчитать
   сумму массива, **разбив работу на `num_threads` потоков**. Каждый поток
   суммирует свой кусок массива; главный поток дожидается всех и складывает
   частичные суммы. Гарантируется `1 <= num_threads <= n` (а при `n == 0` ответ
   `0`). Результат обязан совпадать с обычной последовательной суммой —
   при любом числе потоков.

2. `long mutex_counter(int num_threads, int per_thread)` — запустить `num_threads`
   потоков; каждый увеличивает **общий** счётчик `per_thread` раз, и делает это
   **под мьютексом**. Вернуть итог — он обязан быть ровно
   `num_threads * per_thread`. Это задание специально проверяет, что ты защитил
   счётчик: без замка большой тест «потеряет» часть прибавок.

Запусти тесты:

```bash
./caos/run.sh 10
```

Сначала всё красное (заглушки возвращают `0`). Читай вывод GoogleTest: он
покажет, чего ждали и что получили. Делай задания по очереди.

### Подсказки (открывай, только если застрял)

<details><summary>parallel_sum — как передать потоку его кусок</summary>

Потоку нужно знать: где начинается его участок, где заканчивается, и куда положить
результат. Заведи структуру:

```c
typedef struct {
    const long *arr;
    size_t begin;   // индекс начала куска (включительно)
    size_t end;     // индекс конца куска (НЕ включительно)
    long   result;  // сюда поток запишет свою частичную сумму
} Part;
```

Сделай массив из `num_threads` таких структур (и массив `pthread_t`), заполни
границы кусков, создай потоки, дождись всех `pthread_join`, потом сложи все
`result`.
</details>

<details><summary>parallel_sum — как поделить n на куски без потерь</summary>

Простой способ: `base = n / num_threads` элементов каждому, а остаток
`n % num_threads` раскидай по первым потокам (по +1). Тогда границы идут подряд:
конец одного куска = начало следующего. Главное — чтобы куски **не пересекались**
и **покрыли весь** массив: сумма длин кусков == `n`. Проверь это на бумаге для
`n = 7, num_threads = 3`.
</details>

<details><summary>parallel_sum — частая ошибка</summary>

Каждый поток должен писать результат в **свою** ячейку (`parts[i].result`).
Если все потоки пишут в одну общую переменную без мьютекса — это та самая гонка
из Идеи 4, и сумма «поплывёт». Тут мьютекс не нужен: дай каждому своё место,
а сложишь уже в главном потоке после `join`.
</details>

<details><summary>mutex_counter — скелет</summary>

Здесь, наоборот, счётчик **один на всех**. Передай каждому потоку указатель на
общий `long` и на общий `pthread_mutex_t`. В функции потока:

```c
for (int i = 0; i < per_thread; ++i) {
    pthread_mutex_lock(m);
    (*counter)++;
    pthread_mutex_unlock(m);
}
```

Создай потоки, дождись всех, верни `counter`. Не забудь `pthread_mutex_init`
до запуска и `pthread_mutex_destroy` в конце.
</details>

<details><summary>mutex_counter — проверь себя</summary>

Убери `lock/unlock` и прогони большой тест (`8` потоков, `100000` прибавок) —
скорее всего, получишь число **меньше** ожидаемого и не всегда одинаковое. Это
гонка данных вживую. Верни замок — число станет ровным и стабильным. Полезно
увидеть баг своими глазами.
</details>

---

Многопоточность — это первое место в курсе, где «иногда работает» становится
нормой, если не быть аккуратным. Если тест то зелёный, то красный без изменений
в коде — не списывай на случайность: это сигнал о гонке. Приноси такой вывод
мне — на «плавающих» багах учатся думать как системный программист.
