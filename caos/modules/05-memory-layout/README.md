# Модуль 05 — Память: адреса, указатели, стек/куча, выравнивание

## Большая идея

До сих пор переменная была для тебя просто «коробкой со значением». Пора заглянуть
внутрь: **память компьютера — это одна огромная пронумерованная лента байтов**. Каждый
байт имеет номер — **адрес**. Когда ты пишешь `int x = 42;`, машина выбирает где-то на
этой ленте свободное место, кладёт туда число и запоминает адрес.

Зачем это знать? Потому что почти все «странные» баги в C — segfault, порча данных,
утечки — это про то, что *указатель смотрит не туда* или *память не та, что ты думал*.
Поняв адреса, указатели, стек/кучу и выравнивание, ты перестаёшь бояться этих слов и
начинаешь видеть, *что физически происходит*, когда твоя программа работает.

Аналогия на весь модуль: память — это **длинный ряд одинаковых ящиков в камере хранения**,
пронумерованных 0, 1, 2, … Адрес — номер ящика. Указатель — бумажка, на которой записан
этот номер. А выравнивание — правило «крупные вещи кладём только в ящики с номером,
кратным их размеру».

## Идея 1. Адрес и указатель; разыменование

**Адрес** — это номер ячейки в памяти. **Указатель** — переменная, которая хранит такой
номер (адрес другой переменной). Сама бумажка с номером ящика — это указатель; то, что
лежит в ящике, — значение.

```c
int x = 42;        // x лежит где-то в памяти, скажем по адресу 0x7ffe...a4
int* p = &x;       // & — "дай адрес x". Теперь p хранит этот адрес
int  y = *p;       // * — "разыменование": сходи по адресу p и возьми значение -> 42
*p = 100;          // запиши 100 ПО АДРЕСУ p -> теперь x == 100
```

Две операции, которые легко путать:
- `&x` — **взять адрес** переменной (получить номер ящика).
- `*p` — **разыменовать** указатель (сходить по номеру и заглянуть внутрь ящика).

**ПОЧЕМУ это важно:** указатель — единственный способ в C сказать функции «вот *та самая*
переменная, меняй её», а не «вот копия её значения». Без указателей `scanf`, динамическая
память, строки и связные структуры просто не существовали бы.

**Подводный камень:** разыменовать `NULL` (указатель «в никуда») или висячий указатель —
это краш. Указатель сам по себе безобиден; опасно именно `*p`, когда `p` смотрит не туда.

## Идея 2. Стек (автоматическая память) против кучи (malloc/free)

Память для твоих данных берётся из двух разных мест:

- **Стек (stack)** — для локальных переменных. Память выделяется **автоматически** при
  входе в функцию и так же автоматически освобождается при выходе из неё. Быстро,
  но переменная живёт только пока выполняется её блок `{ ... }`.
- **Куча (heap)** — память «по запросу». Ты просишь её сам через `malloc`, и она живёт,
  **пока ты не вернёшь** её через `free`. Гибко, но ответственность на тебе.

```c
#include <stdlib.h>

void demo(void) {
    int a = 5;                       // a на СТЕКЕ: исчезнет в конце demo()
    int* buf = malloc(3 * sizeof(int));  // 3 int на КУЧЕ
    if (buf == NULL) return;         // malloc может вернуть NULL — всегда проверяй
    buf[0] = 10;                     // пользуемся
    free(buf);                       // вернули память. Забыл free -> утечка
}
```

Аналогия: **стек — это стопка тарелок**. Кладёшь сверху (входишь в функцию), снимаешь
сверху (выходишь) — строго по очереди, мыть не надо. **Куча — это склад**: берёшь ячейку
любого размера, но обязан сам вернуть ключ, иначе склад постепенно забьётся (утечка).

**ПОЧЕМУ это важно:** на стеке нельзя вернуть из функции адрес локальной переменной —
она умрёт сразу после `return`, и адрес станет мусором. Большие или «долгоживущие»
данные обязаны лежать на куче.

**Подводные камни:** (1) забыл `free` — утечка памяти; (2) сделал `free` дважды или
пользуешься памятью после `free` — порча/краш; (3) вернул адрес переменной со стека —
висячий указатель.

> В заданиях этого модуля мы НЕ зовём `malloc`/`free` — сначала надо твёрдо понять адреса
> и арифметику указателей. Но знать эти два «этажа» памяти нужно уже сейчас.

## Идея 3. Выравнивание (alignment) и набивка (padding) в структурах

Процессору удобно читать число с адреса, **кратного его размеру**: 4-байтовый `int` —
с адреса, кратного 4; 8-байтовый `double` — кратного 8. Это и есть **выравнивание**.
На многих платформах чтение «вкривь» либо медленнее, либо вообще запрещено.

Из-за этого правила компилятор иногда **вставляет пустые байты** между полями структуры —
это **набивка (padding)**, чтобы каждое поле начиналось на «правильном» адресе:

```c
struct A {
    char  c;   // 1 байт по смещению 0
    // ... 3 байта НАБИВКИ, чтобы int встал на смещение, кратное 4 ...
    int   n;   // 4 байта по смещению 4
};             // sizeof(struct A) == 8, хотя данных всего 5 байт!
```

Поэтому `sizeof(struct A)` часто больше, чем сумма размеров полей. Переставь поля
от больших к меньшим — и набивки нередко станет меньше.

Аналогия: правило камеры хранения «холодильник (большой) ставим только в ящик с номером,
кратным 4». Если перед ним лежит маленькая коробочка в ящике 0, ящики 1–3 остаются
пустыми — это и есть padding.

**ПОЧЕМУ это важно:** объясняет, почему структуры «весят больше ожидаемого», почему
порядок полей влияет на размер, и почему `align_up` (твоё первое задание) — настоящая
рабочая операция: именно так аллокаторы и компиляторы вычисляют, куда положить следующий
объект.

## Идея 4. Арифметика указателей

Указатели можно складывать с числами — но **шаг считается в элементах, а не в байтах**.
`p + 1` сдвигает на `sizeof(*p)` байт, то есть на один целый элемент.

```c
int a[3] = {10, 20, 30};
int* p = a;          // имя массива "распадается" в указатель на a[0]
*(p + 1);            // == a[1] == 20   (сдвиг на ОДИН int, не на 1 байт)
p[2];                // == a[2] == 30   (индексация — это и есть *(p + 2))

long d = (a + 2) - a; // разность указателей == 2 (в ЭЛЕМЕНТАХ, не в байтах)
```

То есть `&a[2] - &a[0]` даёт `2`, хотя в байтах между ними `2 * sizeof(int)`. Компилятор
делит байтовую разницу на размер элемента за тебя.

**ПОЧЕМУ это важно:** так работает любой обход массива, строки, буфера. И именно поэтому
тип указателя имеет значение: `int*` и `char*`, указывая на один и тот же байт, по `+1`
шагнут на разное число байт.

**Подводный камень:** вычитать имеет смысл только указатели **в один и тот же массив**;
«расстояние» между указателями в разные массивы — неопределённое поведение.

## Задания

Файл `src/memory.c` (интерфейс — `include/memory.h`). Реализуй четыре функции. В стабе
сейчас заглушки — они компилируются, но возвращают неверный результат, поэтому тесты
красные.

1. `size_t align_up(size_t n, size_t align);` — округлить `n` ВВЕРХ до кратного `align`
   (`align` — степень двойки). `align_up(13, 8) == 16`, `align_up(16, 8) == 16`,
   `align_up(0, 8) == 0`.
2. `int is_aligned(const void* p, size_t align);` — вернуть `1`, если адрес `p` кратен
   `align`, иначе `0` (`align` — степень двойки).
3. `int* nth_element(int* base, size_t n);` — вернуть указатель на `base[n]`
   (то есть `base + n`), **без** разыменования.
4. `long ptr_distance(const int* a, const int* b);` — расстояние `a - b` в **элементах**
   (может быть отрицательным).

Запуск тестов модуля:

```bash
./caos/run.sh 05
```

Прочитай вывод GoogleTest: он показывает, что ожидалось и что получилось. Делай функции

### Подсказки (открывай по одной, только если застрял)

<details><summary>Задание 1 — align_up без циклов</summary>

Для степени двойки округление вверх — это классический трюк «прибавить (align − 1) и
обрезать младшие биты». Прибавь `align - 1` к `n`, а затем обнули младшие биты результата.
Обнулить младшие биты помогает маска `~(align - 1)` и оператор `&`. Проверь на `n == 0`
и на уже кратных значениях — должно остаться как есть.
</details>

<details><summary>Задание 2 — is_aligned через адрес как число</summary>

Адрес — это по сути большое целое число. В C его можно положить в целочисленный тип
`uintptr_t` (из `<stdint.h>`): `uintptr_t a = (uintptr_t)p;`. Адрес кратен `align`, когда
`a % align == 0`. Для степени двойки тот же ответ даёт `(a & (align - 1)) == 0`. Верни
`1`/`0` соответственно.
</details>

<details><summary>Задание 3 — nth_element</summary>

Это буквально арифметика указателей из Идеи 4: нужный адрес — это `base + n`. Никакого
`*` не нужно — мы возвращаем сам указатель, а не значение по нему.
</details>

<details><summary>Задание 4 — ptr_distance</summary>

Компилятор уже умеет вычитать указатели одного типа и сам делит на размер элемента:
`a - b` для `const int*` даёт разницу в элементах. Результат имеет тип `ptrdiff_t` —
приведи его к `long` для возврата. Проверь знак: если `a` левее `b`, ответ отрицательный.
</details>
