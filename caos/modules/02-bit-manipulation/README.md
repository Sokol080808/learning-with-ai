# Модуль 02 — Битовые операции: маски, сдвиги, флаги

В модуле 01 ты разобрался, что число внутри компьютера — это просто набор битов
(нулей и единиц). Теперь научимся этими битами **управлять напрямую**: включать,
выключать, переключать и переставлять. Это первый момент в курсе, где ты по-настоящему
говоришь с машиной на её языке.

## Большая идея

Любое число `uint32_t` — это **32 лампочки в ряд**, каждая горит (1) или погашена (0).
Привычная арифметика (`+`, `-`, `*`) смотрит на ряд как на одно большое число.
А битовые операции смотрят на **каждую лампочку по отдельности** и позволяют щёлкать
выключателями. Зачем это нужно? Чтобы хранить много «да/нет» в одном числе (компактно),
проверять и менять их за один такт процессора (быстро), и понимать, что происходит на
уровне железа (а в этом весь смысл курса CAOS).

Договоримся о нумерации: биты считаем **справа налево, с нуля**. Самый правый
(младший) бит — это бит 0. В числе `0b1010` бит 0 = 0, бит 1 = 1, бит 2 = 0, бит 3 = 1.

```
число 0b1010 (это десятичное 10):

  бит:   3   2   1   0
         1   0   1   0
         ^               ^
      старший          младший
```

## Идея 1. Четыре базовые операции: AND, OR, XOR, NOT и сдвиги

Эти операции работают **побитово** — сравнивают i-й бит одного числа с i-м битом
другого и кладут результат в i-й бит ответа.

| Операция | Знак в C | Правило | Бытовая аналогия |
|---|---|---|---|
| AND (И) | `&` | 1, только если **оба** бита 1 | «фильтр»: пропускает только там, где у маски 1 |
| OR (ИЛИ) | `\|` | 1, если **хотя бы один** бит 1 | «включатель»: где у маски 1 — там точно загорится |
| XOR (исключающее ИЛИ) | `^` | 1, если биты **разные** | «переключатель»: где у маски 1 — там бит щёлкнет |
| NOT (НЕ) | `~` | инвертирует **каждый** бит | «инверсия»: все лампочки наоборот |

```c
0b1100 & 0b1010  ==  0b1000   // 1 только там, где у обоих 1
0b1100 | 0b1010  ==  0b1110   // 1 там, где хоть у одного 1
0b1100 ^ 0b1010  ==  0b0110   // 1 там, где биты разные
~0b00000000u      ==  0xFFFFFFFFu  // все нули стали единицами (для 32 бит)
```

Ещё два инструмента — **сдвиги**:

- `x << n` — сдвиг **влево** на `n` позиций. Все биты уезжают влево, справа дописываются
  нули. Это то же самое, что умножить на 2ⁿ.
- `x >> n` — сдвиг **вправо** на `n`. Биты уезжают вправо. Для беззнаковых
  (`uint32_t`) слева дописываются нули. Это деление на 2ⁿ.

```c
1u << 0   ==  0b0001   // единица на месте 0
1u << 3   ==  0b1000   // единица «уехала» на место 3
0b1000u >> 2  ==  0b0010
```

> **ПОЧЕМУ это важно.** Связка `1u << i` — твой главный инструмент в этом модуле.
> Она строит число, где горит **ровно один** бит — i-й. Имея такую «иголку», можно
> добраться до любого конкретного бита: проверить его, включить, выключить или
> переключить.

> **Подводный камень.** Сдвигать на число `>= 32` (ширина типа) — это
> **неопределённое поведение** в C: результат может быть любым. Не пиши `1u << 32`.
> И обязательно используй `1u` (беззнаковую единицу), а не `1`: у знакового `int`
> сдвиг в старший бит — тоже ловушка.

## Идея 2. Маски: упаковываем флаги в одно число

**Маска** — это просто число, которое ты используешь как трафарет: его биты говорят,
«с какими позициями работаем». Это позволяет хранить набор «галочек» в одном числе.

Представь настройки: жирный шрифт, курсив, подчёркивание. Можно завести три отдельные
переменные `bool`. А можно — три бита в одном `uint32_t`:

```c
#define BOLD      (1u << 0)   // 0b001
#define ITALIC    (1u << 1)   // 0b010
#define UNDERLINE (1u << 2)   // 0b100

uint32_t style = 0;
style |= BOLD;      // включить жирный  -> 0b001
style |= UNDERLINE; // включить подчёрк -> 0b101
if (style & ITALIC) { /* курсив включён? нет */ }
style &= ~BOLD;     // выключить жирный -> 0b100
```

Заметил приёмы? `OR` (`|=`) включает флаг, `AND` с маской (`& FLAG`) проверяет,
`AND` с инверсией (`&= ~FLAG`) выключает. Эти же три приёма ты реализуешь в заданиях.

> **ПОЧЕМУ это важно.** Так устроены реальные API: права доступа к файлу
> (чтение/запись/исполнение), флаги открытия файла, регистры устройств. Один
> `int` вместо десятка `bool` — это и память, и скорость, и совместимость с железом.

## Идея 3. Зачем биты вообще: компактность и скорость

Два аргумента, почему это не «трюк ради трюка»:

1. **Компактность.** 32 «да/нет» помещаются в одно 4-байтовое число вместо 32
   отдельных байтов (а `bool` в C++ обычно занимает целый байт). Это в 8 раз меньше
   памяти — критично для сетевых протоколов, форматов файлов, встраиваемых систем.
2. **Скорость.** `&`, `|`, `^`, `<<` — это **одна инструкция процессора**. Проверить
   25 флагов одним `&` — мгновенно. А ещё есть фокусы: `x & (x - 1)` гасит младший
   единичный бит, `x << 1` умножает на 2 быстрее, чем `* 2` в наивном коде.

## Идея 4. Полезные трюки на битах

Парочка приёмов, которые выглядят как магия, пока их не разберёшь:

- **Проверка степени двойки.** У степени двойки (1, 2, 4, 8, …) установлен **ровно
  один** бит. Хитрость: `x & (x - 1)` обнуляет самый младший единичный бит. Если после
  этого получился 0 — значит, единичный бит был один-единственный. Но не забудь
  отдельно отсечь `x == 0` (у нуля битов нет вообще, это не степень двойки).

  ```c
  // x = 0b1000 (это 8). x - 1 = 0b0111. x & (x-1) = 0b0000 -> степень двойки!
  // x = 0b1010 (это 10). x - 1 = 0b1001. x & (x-1) = 0b1000 -> НЕ степень двойки.
  ```

- **Перестановка байтов.** Число из 4 байт `[B3 B2 B1 B0]` иногда нужно перевернуть в
  `[B0 B1 B2 B3]`. Это пригодится в модуле 03, когда будем разбирать **порядок байт**
  (endianness) — почему один и тот же файл по-разному читается на разных машинах.
  Каждый байт вытаскивается сдвигом и маской `& 0xFF`, а потом ставится на новое место
  сдвигом в другую сторону.

  ```c
  // 0x11223344  ->  0x44332211
  // младший байт 0x44 уезжает в старший, и наоборот.
  ```

## Идея 5. Логические vs побитовые операторы: ловушка, которую все наступают

В C есть **два** семейства операторов, которые выглядят похоже, но делают разное:

| Семейство | Операторы | Что делают |
|---|---|---|
| Побитовые | `&`, `\|`, `^`, `~` | Работают с **каждым битом** отдельно |
| Логические | `&&`, `\|\|`, `!` | Смотрят на число целиком — ноль или нет |

Классический баг: ты хочешь проверить флаг `MASK`, а пишешь `&&` вместо `&`:

```c
#define MASK  0x02u   // бит 1

uint32_t flags = 0x02u;

// НЕВЕРНО — логический AND: любое ненулевое число -> true (1)
if (flags && MASK) { /* всегда выполняется, если flags != 0 */ }

// ВЕРНО — побитовый AND: проверяет именно бит 1
if (flags & MASK)  { /* выполняется, только если бит 1 установлен */ }
```

Почему опасно? `flags && MASK` вычисляет «`flags` ненулевой? ДА» и «`MASK` ненулевой? ДА»,
и возвращает `1` — но это не имеет никакого отношения к тому, установлен ли конкретный
бит. Если `flags = 0x01u` (бит 0), то `flags & 0x02u == 0`, но `flags && 0x02u == 1`.

> **Правило:** работаешь с битовыми полями — всегда используй `&`, `|`, `^`, `~`. Логические
> `&&`/`||` — только для булевых условий вида «оба условия верны».

## Идея 6. Арифметический сдвиг знаковых чисел

Выше мы сдвигали **беззнаковые** числа (`uint32_t`): вправо всегда вдвигаются нули.
Для **знаковых** (`int32_t`) правый сдвиг (`>>`) — это **арифметический сдвиг**:
слева вдвигается **знаковый бит** (0 для положительных, 1 для отрицательных).

```c
int32_t pos =  8;  //  0b...0001000
int32_t neg = -8;  //  0b...1111000 (дополнение до двух)

pos >> 2  ==  2    // 0b...0000010  — слева нули, деление на 4
neg >> 2  == -2    // 0b...1111110  — слева единицы, «деление на 4» для отрицательных
```

Это позволяет делать целочисленное деление на степень двойки для отрицательных чисел
(с округлением к -∞). Компилятор часто заменяет деление на сдвиг ровно по этой причине.

> **Подводный камень — UB при сдвиге на >= ширины типа.** Сдвиг `x >> 32` или `x << 32`
> для 32-битного типа — это **неопределённое поведение** в C11. Многие думают, что
> получат 0, но на x86 ЦП **маскирует счётчик сдвига до 5 бит** (для 32-бит операций):
> инструкция `SHL eax, 32` выполняется как `SHL eax, 0` и не сдвигает вообще ничего!
> Разные компиляторы дают разные результаты. Никогда не полагайся на сдвиг >= ширины.
>
> ```c
> uint32_t x = 1u;
> int n = 32;
> // x << n  ->  UB; на x86 может дать 1, а не 0!
> // Безопасно: if (n < 32) result = x << n; else result = 0;
> ```

## Идея 7. Законы Де Моргана для битовых операций

Из булевой алгебры: «НЕ (А И Б) = (НЕ А) ИЛИ (НЕ Б)» — и то же самое работает побитово:

```
~(a & b) == (~a) | (~b)      // закон Де Моргана 1
~(a | b) == (~a) & (~b)      // закон Де Моргана 2
```

Это выглядит как формула, но у неё есть практическое применение. Вспомни выключение флага:

```c
// Выключить биты FLAGS в переменной x:
x &= ~FLAGS;
// По Де Моргану эквивалентно:
// x = x & (~FLAGS) = x & ((~(bit0)) | (~(bit1)) | ...)
```

Ещё пример — инвертировать группу флагов, сохранив остальные:

```c
#define MASK 0b1010u
uint32_t x = 0b1111u;
x ^= MASK;           // -> 0b0101  (только биты по маске щёлкнули)
// Проверка через Де Моргана: x & ~MASK не изменился,
// а x & MASK стал побитово инвертированным.
```

Законы Де Моргана используют при упрощении условий и при оптимизации компиляторами.

## Идея 8. Извлечение и вставка многобитовых полей

До сих пор мы работали с **одиночными битами**. Но часто в одном числе нужно хранить
**несколько полей** по нескольку бит каждое. Пример из реальной жизни — пиксель формата
RGB888: красный, зелёный и синий канал по 8 бит в одном 32-битном слове.

**Извлечение поля** шириной `k` бит, начиная с позиции `p` (позиция младшего бита поля):

```c
// Сдвигаем нужные биты на место 0, затем маскируем лишнее.
uint32_t field = (x >> p) & ((1u << k) - 1u);
//                            ^^^^^^^^^^^^^^^^
//                            маска из k единиц: при k=3 получаем 0b111 = 7
```

Как работает маска `(1u << k) - 1u`? Сдвиг даёт единицу в позиции k: `0b1000`.
Вычитаем 1: `0b0111` — ровно k единиц, которые «пропускают» только k младших бит.

**Вставка поля** (установить биты `p..p+k-1` в значение `v`, остальное не трогать):

```c
uint32_t mask = ((1u << k) - 1u) << p;   // маска поля на нужном месте
x = (x & ~mask) | ((v & ((1u << k) - 1u)) << p);
//   ^^^^^^^^^^ обнуляем поле    ^^^^^^^^^^^^^^^^^^^^^^^^^^^^ ставим новое значение
```

Два шага: сначала стираем старое поле через `AND ~mask`, потом записываем новое через `OR`.

```c
// Пример: вставить значение 5 (0b101) в биты 4..6 числа x
uint32_t x = 0b11111111u;
uint32_t mask = ((1u << 3) - 1u) << 4;  // маска: 0b01110000
x = (x & ~mask) | ((5u << 4) & mask);   // x -> 0b10101111
```

> **Практика.** Именно так работают RGBA/BGR пиксели, поля IP-заголовков, аппаратные
> регистры — везде, где один `uint32_t` хранит несколько независимых числовых полей.

## Идея 9. Подсчёт установленных битов (popcount) через x & (x-1)

Уже видели трюк `x & (x - 1)` для проверки степени двойки. Он делает ровно одно:
**гасит самый младший единичный бит** числа `x`.

Доказательство: пусть `x = ...1 0...0` (самый младший единичный бит на позиции p,
дальше идут нули). Тогда `x - 1 = ...0 1...1` — бит p погас, все биты правее стали 1.
При `x & (x-1)` биты правее p (у x они нули) дают нули; бит p у x равен 1, у x-1 равен 0
— даёт 0; биты левее p не изменились. Итог: самый правый единичный бит стёрт.

Применение — **подсчёт установленных битов (popcount)** за O(popcount(x)) итераций,
а не за O(32):

```c
int popcount_kernighan(uint32_t x) {
    int count = 0;
    while (x) {
        x &= x - 1u;   // гасим один бит за итерацию
        ++count;
    }
    return count;
}
// popcount(0b10110) = 3: итерации дают 0b10100 -> 0b10000 -> 0b00000, count=3
```

Если установлено k битов, цикл выполнится ровно k раз — это оптимально, когда k мало.
В реальном коде используют встроенную функцию `__builtin_popcount(x)` или инструкцию
процессора `POPCNT`, но понимать механизм важно.

> **Задумайся:** можешь ли ты написать `popcount` без цикла, используя только сдвиги
> и маски? (Подсказка: разбей биты на пары, потом четвёрки, потом восьмёрки — это так
> называемый «параллельный popcount».)

## Задание 7. Упаковка и распаковка многобитовых полей

Ты только что узнал, как извлекать и вставлять числовые поля. Теперь применим это
для реальной задачи: **упаковать четыре числа в одно 32-битное слово**.

Интерфейс (`include/bits.h`):

```c
uint32_t pack_fields(uint8_t flags, uint8_t r, uint8_t g, uint8_t b);
void     unpack_fields(uint32_t packed,
                       uint8_t *flags, uint8_t *r, uint8_t *g, uint8_t *b);
```

Формат слова — каждое поле занимает ровно 8 бит:

```
биты: 31..24   23..16   15..8    7..0
      flags      r        g       b
```

`pack_fields` собирает слово из четырёх байтов; `unpack_fields` разбирает его обратно.
Если сделать правильно, `unpack_fields(pack_fields(f, r, g, b), &f2, &r2, &g2, &b2)`
должна вернуть ровно те же четыре числа.

### Подсказки (открывай по одной, только если застрял)

<details><summary>Подсказка 1 — как расположить поля</summary>

Вспомни Идею 8: чтобы поставить значение `v` в биты `p..p+7`, нужно сдвинуть `v` влево
на `p` позиций: `(uint32_t)v << p`. Какое смещение `p` у каждого из четырёх полей?
Взгляни на таблицу формата выше.
</details>

<details><summary>Подсказка 2 — как собрать слово из частей</summary>

Каждое поле уже стоит на своём месте после сдвига. Теперь нужно объединить четыре
непересекающихся «кусочка» в одно число. Какая операция объединяет биты, не затрагивая
соседей? Это `OR` (`|`). Сложи все четыре сдвинутых значения через `|`.
</details>

<details><summary>Подсказка 3 — как извлечь поле обратно</summary>

Используй формулу из Идеи 8: `(packed >> p) & 0xFFu`. Сдвиг ставит нужный байт в
позицию 0, маска `0xFFu` оставляет только 8 бит. Запиши результат по указателю:
`*r = (uint8_t)((packed >> 16) & 0xFFu);` — и так для каждого поля.
</details>
