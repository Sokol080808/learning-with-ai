# Модуль 14 — Инструменты, ввод-вывод и отладка

Этот модуль — особенный. Раньше ты писал код с нуля. Здесь код **уже написан, но в нём
баги**. Твоя работа — как у настоящего программиста: найти, почему оно врёт, и починить.
А заодно ты познакомишься с инструментами, которые превращают отладку из «вглядывания в
код до боли в глазах» в спокойное расследование с уликами.

## Большая идея

Программист тратит на **поиск багов** больше времени, чем на написание нового кода. И
ключевой навык — не «думать сильнее», а **смотреть на факты**: что реально лежит в памяти,
какие значения у переменных, куда ушло время, какие системные вызовы сделала программа.
Эти факты дают инструменты. Тот, кто умеет ими пользоваться, чинит баг за минуты; тот, кто
не умеет, — часами перечитывает один и тот же цикл, не замечая `<=` вместо `<`.

В этом модуле четыре маленькие функции, и в каждой по одному багу. Три бага — «логические»
(программа считает не то), один — «про память» (выход за границу массива). Логические
ловятся отладчиком и тестами; «про память» — санитайзером или valgrind, и это особенно
важно: такие баги могут «работать» на твоей машине и падать у пользователя.

## Идея 1. Откуда вообще берётся ввод-вывод: системные вызовы

Когда ты пишешь в файл или читаешь с клавиатуры, твоя программа **не имеет прямого доступа**
к железу. Она просит операционную систему: «прочитай мне столько-то байт». Эта просьба —
**системный вызов** (syscall). Три фундаментальных:

- `open(path, flags)` — открыть файл, получить **файловый дескриптор** (fd) — просто
  маленькое целое число, «номерок из гардероба», по которому ОС знает, о каком файле речь.
- `read(fd, buf, n)` — прочитать до `n` байт в буфер `buf`.
- `write(fd, buf, n)` — записать `n` байт из `buf`.

Три дескриптора есть у каждой программы с рождения: `0` — stdin (ввод), `1` — stdout
(вывод), `2` — stderr (ошибки). Когда ты пишешь `printf(...)`, под капотом в итоге
происходит `write(1, ...)`.

**Зачем это знать.** `printf` не печатает мгновенно — он копит данные в **буфере** и
сбрасывает их пачкой (это быстрее, чем дёргать ОС на каждый символ). Поэтому при отладке
через `printf` сообщение может «не успеть» напечататься перед крахом — буфер не сбросился.
Лекарство: `fprintf(stderr, ...)` (stderr не буферизуется), либо `fflush(stdout)`, либо —
лучше — отладчик вместо `printf`. Понимание «вывод буферизуется» спасает от ложного вывода
«до сюда дошло, дальше нет» (на самом деле дошло и дальше, просто буфер потерялся при краше).

```c
// Грубо говоря, printf("hi\n") в конце концов делает что-то вроде:
write(1, "hi\n", 3);   // 1 == stdout
```

## Идея 2. Флаги компилятора — что включает каждый

Прежде чем запускать отладчик или санитайзер, нужно правильно **скомпилировать** программу.
Флаги gcc/clang — это не украшение, каждый делает конкретную работу.

### Предупреждения: `-Wall`, `-Wextra`, `-Werror`

```bash
gcc -Wall -Wextra -Werror -std=c11 -o prog prog.c
```

- **`-Wall`** — включает «стандартные» предупреждения: неинициализированная переменная,
  неиспользуемый результат функции, сравнение знаковых с беззнаковыми, подозрительные
  конструкции в `if`/`switch`, неявное приведение типов, забытый `return`. Название
  обманчиво: это **не** *все* предупреждения — просто разумное умолчание.
- **`-Wextra`** — добавляет ещё один слой: предупреждает о параметрах функции, которые не
  используются, о сравнениях, которые всегда истинны/ложны по типу, об упущенных ветках в
  `switch`. Работает поверх `-Wall`.
- **`-Werror`** — превращает **все** предупреждения в ошибки компиляции. Программа не
  соберётся, пока в ней есть «желтые» сообщения. Это жёстко, но именно так работают
  настоящие проекты: «предупреждение — это баг, который ещё не взорвался».

Хорошее правило: **всегда компилировать с `-Wall -Wextra`**, а в коммите иметь нулевое
число предупреждений. Флаг `-Werror` добавляй по желанию — он полезен в CI, чтобы случайно
не «накопить» предупреждения.

### Уровни оптимизации: `-O0`, `-O1`, `-O2`, `-O3`

```bash
# Для разработки и отладки:
gcc -O0 -g -std=c11 -o prog prog.c

# Для «финального» бинаря:
gcc -O2 -std=c11 -o prog prog.c
```

- **`-O0`** (по умолчанию, «без оптимизаций») — компилятор переводит код почти
  дословно: каждая строка C становится несколькими инструкциями, переменные живут в
  памяти, а не в регистрах. Результат медленный, но **отлаживается идеально**: в gdb ты
  видишь ровно те переменные и строки, что написал.
- **`-O1`** — лёгкая оптимизация, убирает очевидные лишние операции. Код быстрее, ещё
  отлаживаемый.
- **`-O2`** — агрессивная оптимизация: инлайнинг функций, перестановка инструкций,
  устранение мёртвого кода. Так собирают «релизы». Под gdb можно столкнуться с
  «оптимизированными» переменными, которые «не существуют» в данный момент.
- **`-O3`** — ещё агрессивнее, иногда включает векторизацию. Для повседневной работы
  разница с `-O2` редко ощутима.

**Ключевое правило:** для отладки всегда используй `-O0 -g`. При `-O2` отладчик иногда
показывает «непонятное», потому что компилятор переставил инструкции или убрал переменные.

### Отладочная информация: `-g`

Флаг `-g` просит компилятор вложить в бинарь **таблицу соответствий**: номер инструкции ↔
строка в исходном коде, имена переменных, типы. Без `-g` gdb/lldb знает только адреса;
с `-g` он говорит `buggy.c:41` и показывает `int i = 3`.

Размер бинаря вырастает, но на диск уходит лишь несколько мегабайт — для разработки это
не проблема. В продакшне `-g` тоже полезен: его можно выкладывать отдельно (`.dSYM`,
`.debug`) и подключать, когда программа упала на сервере.

```bash
# Типичный набор для «изучаю код и отлаживаю»:
gcc -std=c11 -Wall -Wextra -O0 -g -o prog prog.c

# Типичный набор для «проверяю производительность»:
gcc -std=c11 -O2 -o prog prog.c
```

> **Почему в этом курсе CMakeLists говорит `-O0 -g`?** Именно для того, чтобы gdb и
> санитайзеры работали чисто. Если захочешь проверить, что твой код быстр, пересобери
> руками с `-O2`.

## Идея 3. GDB — отладчик под микроскопом

GDB (GNU Debugger) позволяет **остановить** программу в произвольном месте и посмотреть,
что происходит внутри. На macOS его заменяет LLDB — команды почти те же.

### Типичный сеанс

```bash
# 1. Компилируй с -g (без неё gdb покажет только адреса, не строки)
gcc -std=c11 -O0 -g -o buggy buggy.c

# 2. Запусти под отладчиком
gdb ./buggy
# (или на macOS: lldb ./buggy)
```

Внутри gdb:

```
(gdb) break sum_first_n        # точка останова: гdb остановится при входе в функцию
(gdb) break buggy.c:19         # точка останова на конкретной строке
(gdb) run                      # запустить программу (остановится на breakpoint)
(gdb) print i                  # напечатать переменную i
(gdb) print a[0]               # или элемент массива
(gdb) next                     # шаг вперёд (не заходить в вызываемые функции)
(gdb) step                     # шаг вперёд (заходить внутрь вызовов)
(gdb) continue                 # продолжить до следующего breakpoint
(gdb) backtrace                # показать стек вызовов (особенно полезно после краша)
(gdb) quit                     # выйти
```

**Ситуация: программа упала (segfault)**. Запусти её под gdb, дождись краша, набери
`backtrace` — увидишь точную функцию и строку, где произошло падение, плюс всю цепочку
вызовов, которая к ней привела.

```text
Program received signal SIGSEGV, Segmentation fault.
0x0000... in reverse_array (a=0x..., n=5) at buggy.c:41
41          a[j] = tmp;
(gdb) backtrace
#0  reverse_array (a=0x..., n=5) at buggy.c:41
#1  main () at main.c:10
```

**Ситуация: функция возвращает не то значение**. Поставь breakpoint перед `return`,
распечатай все локальные переменные (`info locals`) или нужные `print sum`, `print i`.
Шагай по итерациям цикла и наблюдай, когда значение начинает расходиться с ожидаемым.

> Совет: `display <выражение>` (вместо `print`) автоматически печатает выражение
> на каждом шаге — удобно для отслеживания `i` и `sum` в цикле.

## Идея 4. Valgrind — детектор памяти без перекомпиляции

Valgrind запускает программу **в своём эмуляторе** и отслеживает каждый доступ к памяти.
Главный инструмент внутри — **Memcheck**.

```bash
# Запустить программу под valgrind:
valgrind --leak-check=full --error-exitcode=1 ./buggy

# Или с подробным выводом о каждом блоке памяти:
valgrind --leak-check=full --show-leak-kinds=all --track-origins=yes ./buggy
```

Что умеет Valgrind/Memcheck:

| Что произошло | Сообщение Valgrind |
|---|---|
| Читаем за концом массива | `Invalid read of size N` |
| Пишем за концом массива | `Invalid write of size N` |
| Используем freed память | `Invalid read/write ... freed` |
| Освобождаем дважды | `Invalid free / double free` |
| Утечка памяти | `definitely lost: N bytes` |
| Читаем неинициализированное | `Use of uninitialised value` |

```text
# Типичный вывод Valgrind при выходе за границу:
==12345== Invalid write of size 4
==12345==    at 0x... reverse_array (buggy.c:41)
==12345==    by 0x... main (main.c:10)
==12345==  Address 0x... is 0 bytes after a block of size 16 alloc'd
```

**Плюсы Valgrind:** работает без перекомпиляции, совместим со старыми бинарями.
**Минусы:** медленно (5–50x замедление), только Linux/macOS (не Windows), не работает
с санитайзерами одновременно.

> **На macOS:** Valgrind официально поддерживается плохо (и вообще не работает на Apple
> Silicon). Используй санитайзеры — они там работают отлично.

## Идея 5. Санитайзеры — охрана прямо в бинаре

Санитайзеры — это **инструментация компилятора**: при `-fsanitize=...` компилятор
вставляет в каждый доступ к памяти, каждую арифметическую операцию или вызов потока
«проверочный» код. Это быстрее и точнее Valgrind, но требует перекомпиляции.

### AddressSanitizer (ASan) — `-fsanitize=address`

Ловит **все виды некорректных обращений к памяти**:

- выход за границу кучи или стека
- use-after-free (доступ к освобождённой памяти)
- use-after-return (доступ к стеку умершей функции)
- double-free

```bash
gcc -std=c11 -O1 -g -fsanitize=address -o buggy buggy.c
./buggy
```

```text
# Типичный вывод ASan при use-after-free:
==12345==ERROR: AddressSanitizer: heap-use-after-free on address 0x...
READ of size 4 at 0x... thread T0
    #0 0x... in freq_lookup src/buggy.c:87
    #1 0x... in main src/main.c:12
previously freed at:
    #0 0x... in freq_destroy src/buggy.c:70
```

ASan показывает **точную строку** где произошёл плохой доступ и где память была
освобождена — это то, что без инструментов заняло бы часы. Замедление ~2x — почти
незаметно при разработке.

### UndefinedBehaviorSanitizer (UBSan) — `-fsanitize=undefined`

Ловит **неопределённое поведение** по стандарту C — то, что компилятор вправе сделать
как угодно:

- переполнение знакового `int` (в C это UB! беззнаковые переполняются корректно)
- сдвиг на отрицательное или слишком большое число
- разыменование `NULL`
- выход за границу массива (в дополнение к ASan)
- вызов функции через неверный указатель на функцию

```bash
gcc -std=c11 -O1 -g -fsanitize=undefined -o buggy buggy.c
./buggy
```

```text
# Типичный вывод UBSan при знаковом переполнении:
buggy.c:55:23: runtime error: signed integer overflow: 2147483647 + 1 cannot be
represented in type 'int'
```

**Важно:** знаковое переполнение `int` — **undefined behavior**, а не просто «большое
число». Компилятор при оптимизации вправе убрать проверки, зная что «overflow невозможен
по стандарту». UBSan ловит это в рантайме.

### ThreadSanitizer (TSan) — `-fsanitize=thread`

Ловит **гонки данных** (data races) в многопоточном коде: два потока одновременно
обращаются к одной переменной, хотя бы одно из обращений — запись, без синхронизации.

```bash
gcc -std=c11 -O1 -g -fsanitize=thread -o buggy buggy.c -lpthread
./buggy
```

```text
# Типичный вывод TSan:
WARNING: ThreadSanitizer: data race (pid=12345)
  Write of size 4 at 0x... by thread T2:
    #0 counter_increment buggy.c:33
  Previous read of size 4 at 0x... by thread T1:
    #0 counter_read buggy.c:28
```

TSan не совместим с ASan в одном бинаре. Используй его отдельно, когда отлаживаешь
многопоточный код.

### Комбинации на практике

```bash
# Максимальная «охота на баги» в однопоточном коде:
gcc -std=c11 -O1 -g -fsanitize=address,undefined -o buggy buggy.c

# Для многопоточного кода — только TSan (несовместим с ASan):
gcc -std=c11 -O1 -g -fsanitize=thread -o buggy buggy.c -lpthread
```

> **Подсказка:** флаги санитайзеров работают и с clang, и с gcc. На macOS Apple Silicon
> используй Homebrew clang (он свежее) — он поддерживает все три.

### Сравнение: Valgrind vs санитайзеры

| Критерий | Valgrind | ASan/UBSan/TSan |
|---|---|---|
| Нужна перекомпиляция? | Нет | Да |
| Замедление | 5–50x | 1.5–5x |
| Работает на macOS ARM? | Плохо / нет | Да |
| Точность сообщений | Хорошая | Отличная (строка кода) |
| Одновременно с gdb | Да | Да |

Для новых проектов рекомендуется запускать тесты с `-fsanitize=address,undefined` в CI —
это ловит целый класс скрытых багов при почти нулевой цене.

## Идея 6. Инструменты отладки — обзор арсенала

Не нужно знать их наизусть. Нужно знать, **какой инструмент для какого вопроса**:

| Вопрос, который ты задаёшь | Инструмент |
|---|---|
| «Что лежит в переменной прямо сейчас? Где упало?» | **gdb** (отладчик) |
| «Я где-то лезу за границу массива / читаю мусор?» | **санитайзеры (ASan)**, **valgrind** |
| «Какие системные вызовы делает программа? Какой файл не открылся?» | **strace** (вызовы ОС), **ltrace** (вызовы библиотек) |
| «На что программа тратит время? Где горячий цикл?» | **perf** |
| «Сколько памяти ест? Какие у процесса fd, потоки?» | **/proc/<pid>/** |

Коротко, что это:

- **gdb** — ставишь точки останова (`break`), запускаешь (`run`), смотришь переменные
  (`print x`), шагаешь по строкам (`next`/`step`). После краха `backtrace` показывает,
  через какие функции пришли в точку падения.
- **Санитайзеры** — компилятор по флагу `-fsanitize=address` встраивает в программу
  «охрану», которая кричит точным адресом и строкой при выходе за границу, use-after-free
  и т.п. Почти бесплатно по сравнению с поиском такого глазами.
- **valgrind** — то же про память, но без перекомпиляции (запускает программу в эмуляторе).
  Медленнее санитайзеров, но иногда удобнее.
- **strace** — печатает каждый syscall: увидишь `openat("config.txt", ...) = -1 ENOENT`
  и сразу поймёшь, что файла нет, вместо гадания «почему не читается».
- **perf** — профайлер: `perf stat`, `perf record`/`perf report` показывают, где время.
- **/proc** — виртуальная папка про живые процессы: `/proc/<pid>/fd`, `/proc/<pid>/maps`,
  `/proc/<pid>/status`. Никаких файлов на диске — ОС отвечает на лету.

> Заметка про платформу: `gdb`, `strace`, `perf`, `/proc`, `valgrind` — это мир Linux.
> На macOS аналоги: `lldb` вместо `gdb`, `dtruss`/Instruments вместо `strace`/`perf`.
> **Санитайзеры (`-fsanitize=...`) работают и там, и там** — это самый переносимый из
> инструментов, начни с него.

## Идея 7. Почему отладчик и санитайзер быстрее, чем «вглядывание в код»

Когда баг найден глазами — отлично. Но глаз **достраивает то, что ожидает увидеть**: ты
читаешь `i <= n` и мозг говорит «ну да, до n», не замечая, что это на один шаг больше, чем
надо. Инструмент не достраивает — он показывает факт.

Сравни два пути для бага «функция возвращает не то число»:

- **Глазами:** перечитываешь цикл, проговариваешь его «в голове», возможно, с той же
  ошибкой, что и в коде. Можешь сидеть так долго.
- **Под gdb:** ставишь точку останова, шагаешь по итерациям, на каждой печатаешь `i`, `j`,
  `sum`. Видишь, что на последней итерации `i` дошло до `n` (а должно было до `n-1`), и
  баг сам показывает себя.

А для бага «про память» (выход за границу) глаз почти бессилен — код *выглядит* правильно,
просто индекс на единицу больше. Зато ASan напечатает прямым текстом:
`heap-buffer-overflow ... WRITE of size 4` и номер строки. Минута против часа.

```text
# Так выглядит крик ASan (сокращённо) — он буквально показывает строку с багом:
==12345==ERROR: AddressSanitizer: stack-buffer-overflow
WRITE of size 4 at 0x... thread T0
    #0 0x... in reverse_array src/buggy.c:41
```

**Зачем это знать.** Привычка «сначала инструмент, потом догадки» экономит тебе годы
суммарно. Это не признак слабости — это профессионализм.

## Идея 8. Git для junior: твоя машина времени

Когда чинишь баги, легко самому что-нибудь сломать. Git позволяет **возвращаться** и
**видеть, что именно ты изменил**. Минимальный набор на каждый день:

- `git status` — что изменено, что готово к коммиту. Смотри его постоянно.
- `git diff` — построчно показывает, **что** ты поменял с последнего коммита. Перед тем
  как сохранить правку — глянь diff: часто там видно случайную опечатку.
- `git add <файл>` — пометить файл «хочу сохранить эти изменения».
- `git commit -m "почему"` — сохранить снимок. Сообщение пиши про *смысл* («fix off-by-one
  in reverse_array»), а не «изменил файл».
- `git log` — история коммитов: когда и что менялось.

**Зачем это знать.** Чинишь баг, делаешь хуже — `git diff` покажет, что ты тронул, а в
худшем случае всегда можно откатиться к последнему рабочему коммиту. Без git ты боишься
менять код; с git — экспериментируешь смело, потому что есть «отмена».

<details><summary>Задание 1 — sum_first_n</summary>

Подставь `n == 1`. Какие индексы переберёт цикл `for (int i = 0; i < n - 1; i++)`?
Ни одного — `0 < 0` ложно. Значит единственный элемент не сложили. Сколько элементов
ты обязан сложить: с `a[0]` по `a[n-1]` включительно — это какое условие на `i`?
</details>

<details><summary>Задание 2 — max_in</summary>

Если массив `{-7, -2, -9, -5}`, а `best` стартует с `0`, то ни один отрицательный

начинаться поиск максимума, чтобы ответ всегда был элементом самого массива? Подсказка:
с первого элемента.
</details>

<details><summary>Задание 3 — reverse_array (это про память!)</summary>

Парный (зеркальный) к индексу `i` — это какой индекс? Если массив длины `n`, последний
валидный индекс — `n - 1`, а не `n`. Сейчас `j = n - i`: при `i == 0` получаем `j == n`,
то есть `a[n]` — на один ЗА концом массива. Это и есть выход за границу.

И второе: сколько обменов нужно? Чтобы развернуть `n` элементов, меняем местами `n / 2`
пар. Условие `i <= n / 2` делает на один обмен больше — он либо лишний (для чётных вернёт
всё назад в исходное), либо снова трогает центр. Почини и индекс `j`, и условие цикла.
</details>

<details><summary>Задание 4 — count_char</summary>

`count = 1;` — это «поставить флажок: символ встречался», а не «увеличить счётчик».
Поэтому для `"hello"` и буквы `'l'` (две штуки) вернётся `1`. Какой оператор увеличивает
число на единицу на каждом совпадении?
</details>

<details><summary>Проверь себя инструментом, а не только тестом</summary>

память не тронута за границей. Многие баги «про память» молчат на маленьких примерах и

</details>

## Задание 5 — «Найди и почини» в счётчике частот слов

Файл `src/buggy.c` содержит небольшую хеш-таблицу, которая считает, сколько раз каждое
слово встречается в тексте. Код компилируется и даже что-то считает — но в нём посажены
**три намеренных бага**:

1. **use-after-free** — функция обращается к памяти после того, как та была освобождена.
   Этот баг **ловит ASan** (`-fsanitize=address`): он напечатает «heap-use-after-free» и
   точную строку. Глазами его почти не видно — код «выглядит нормально».

2. **signed overflow в хеш-функции** — хеш-функция накапливает значение в знаковом `int`,
   и при длинных словах оно переполняется. Это **неопределённое поведение** по стандарту C.
   Ловит **UBSan** (`-fsanitize=undefined`): напечатает «signed integer overflow» с номером
   строки. На практике такой баг иногда приводит к неверным хешам → слова попадают в
   неверные корзины → счётчик врёт.

3. **логическая ошибка** — функция проверяет возвращаемое значение не того вызова.
   Тесты поймают это на конкретных входных данных: частота слова будет неверной даже
   после «исправления» первых двух багов.

Твоя задача: найти все три, починить, добиться зелёных тестов.

### Подсказки (открывай по одной)

<details><summary>Как начать: стратегия поиска трёх багов</summary>

1. Скомпилируй с санитайзерами и запусти тесты:
   ```bash
   # Временно, вручную — чтобы почувствовать инструмент:
   cc -std=c11 -O1 -g -fsanitize=address,undefined \
      -Iinclude src/buggy.c -o freq_test
   ```
   ASan сразу остановится и напечатает строку с use-after-free.
   UBSan напечатает строку с переполнением.

2. Для третьего бага запусти тесты (`./caos/run.sh 14`) и посмотри, у какой функции
   значение не сходится с ожидаемым. Затем прочитай её внимательно: она проверяет
   `if (result_of_A != ...)`, хотя должна проверять результат другого вызова.
</details>

<details><summary>Подсказка к багу 1 — use-after-free</summary>

Посмотри на функцию `freq_lookup`. Есть ли там обращение к полю структуры или к строке
**после** того, как для них могло быть вызвано `free`? Порядок операций важен: сначала
читай, потом освобождай — а не наоборот. ASan скажет тебе точную строку.
</details>

<details><summary>Подсказка к багу 2 — signed overflow</summary>

Посмотри на `freq_hash`. Хеш копит сумму в переменной типа `int`. При длинном слове
(или символах с кодом > 100) это значение может выйти за `INT_MAX`. Чем отличается
`unsigned int` от `int` в плане переполнения? Что произойдёт, если заменить тип
накопителя на беззнаковый?
</details>

<details><summary>Подсказка к багу 3 — неверная проверка return</summary>

Найди место, где функция делает два вызова подряд и проверяет их результат.
Если написано `if (strcmp(a, b) != 0)` там, где нужно `if (strcmp(a, c) != 0)` —
условие будет всегда неверным или всегда верным на определённых входных данных.
Перечитай, что именно сравнивается, и совпадает ли это с тем, что нужно.
</details>
