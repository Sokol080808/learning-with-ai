# Модуль 00 — Setup: как программа становится битами, и что такое компьютер на самом деле

**Большая идея.** Компьютер не понимает ни C, ни Python, ни английский. Он понимает
только **числа** — нули и единицы. Когда ты пишешь `int add(int a, int b)`, кто-то
должен превратить эти буквы в набор команд, которые умеет крутить процессор. Этот
«кто-то» — целый конвейер инструментов. Цель CAOS (Computer Architecture & Operating
Systems) — понять, **что машина делает с твоим кодом** и **как операционная система
этим управляет**. А цель этого, нулевого, модуля скромнее: чтобы у тебя **работал
инструмент** и ты один раз прошёл весь цикл «написал → собрал → проверил тестом →
починил». Без рабочего инструмента всё дальнейшее — магия, а магию мы не любим: мы её
разбираем на части.

Если ты раньше не программировал на компилируемом языке — не пугайся слов вроде
«линковка» или «ассемблер». Ниже всё с нуля, на бытовых аналогиях. Веду за руку.

## Идея 1. Конвейер: как текст превращается в исполняемый файл

Языки бывают двух больших семейств. Python/JavaScript — *интерпретируемые*: есть
программа-переводчик (интерпретатор), которая читает твой код строчка за строчкой
и тут же исполняет. C (на нём ты будешь писать задания) — *компилируемый*: твой
текст заранее, целиком, переводится в машинный код, и получается отдельный файл,
который процессор исполняет уже без посредника.

Этот перевод — не один шаг, а **конвейер** из четырёх станций. Представь сборку
мебели на фабрике: доски → детали → собранный шкаф → шкаф у тебя дома.

1. **Препроцессор.** Обрабатывает строки, начинающиеся с `#` (`#include`, `#define`).
   `#include "warmup.h"` буквально **вставляет** содержимое заголовочного файла в
   твой `.c`. Это чистая работа с текстом: ножницы и клей, никакой «умной» логики.
   На выходе — один большой кусок текста (его называют *translation unit*,
   «единица трансляции»).

2. **Компиляция.** Превращает этот текст в **ассемблер**, а затем в машинные команды
   конкретного процессора. Здесь компилятор проверяет грамматику языка: правильно ли
   расставлены скобки, существует ли функция, которую ты зовёшь, совпадают ли типы.
   Если что-то не так — это **ошибка компиляции**.

3. **Ассемблирование.** Ассемблер (текстовое представление машинных команд) переводится
   в собственно байты — **объектный файл** (`.o`). Это уже почти машинный код, но с
   «дырами»: там, где твой код зовёт функцию из *другого* файла, пока стоит заглушка
   «адрес подставят позже».

4. **Линковка (компоновка).** Линкер сшивает все объектные файлы и библиотеки в одну
   программу и **подставляет реальные адреса** вместо тех «дыр». Если функция объявлена,
   но нигде не реализована — это **ошибка линковки**.

И есть пятый шаг, который происходит уже не при сборке, а при **запуске**:

5. **Загрузка.** Когда ты запускаешь программу, операционная система (вот она, «OS» из
   названия курса!) берёт исполняемый файл с диска, раскладывает его по памяти и
   говорит процессору: «начинай отсюда». Об этом — весь дальнейший курс.

**ПОЧЕМУ это важно знать.** Потому что ошибки на разных станциях конвейера — это
**разные** ошибки, и лечатся по-разному:

- `'foo' undeclared` (или `was not declared`) — этап 2, компиляция. Ты зовёшь `foo`,
  но компилятору про неё *не рассказали*: нет объявления. Лечится — подключи нужный
  заголовок или объяви функцию.
- `undefined reference to 'foo'` — этап 4, линковка. Компилятор *знает*, что `foo`
  существует (объявление видел), но никто её *не реализовал* — нет тела. Лечится —
  добавь `.c`-файл с реализацией в сборку.

Эти два сообщения новички постоянно путают и тратят часы. Ты теперь будешь видеть
разницу: «не объявили» против «не реализовали».

> Если хочешь увидеть конвейер своими глазами (это не обязательно для задания, но
> очень полезно): возьми любой `.c` и прогони руками по станциям —
> ```bash
> gcc -E file.c            # 1. только препроцессор — увидишь раскрытые #include
> gcc -S file.c            # 2. остановиться на ассемблере (получится file.s)
> gcc -c file.c            # 3. объектный файл file.o (байты с «дырами»)
> gcc file.o -o file       # 4. линковка в исполняемый файл
> ```

## Идея 2. Что изучает CAOS и зачем это тебе

CAOS = **Computer Architecture & Operating Systems**, «архитектура компьютера и
операционные системы». Двумя предложениями:

- **Архитектура** — *как устроена и как исполняет код сама машина*: процессор, регистры,
  память, как число `42` хранится в виде битов, почему сложение быстрое, а деление
  медленное.
- **Операционные системы** — *кто всем этим распоряжается*: как ОС запускает твою
  программу, делит между программами процессор и память, не даёт им мешать друг другу,
  обслуживает потоки и файлы.

**ПОЧЕМУ это знать, если можно просто писать код, который работает?** Несколько честных
причин:

- **Баги, которые иначе необъяснимы.** Почему программа «иногда» падает? Почему два
  потока испортили общий счётчик? Почему число вдруг стало отрицательным после
  умножения? Все эти «мистики» — следствие того, как работает железо и ОС. Зная
  механику, ты их не угадываешь, а *видишь*.
- **Скорость.** Программа на одном и том же языке может работать в 10 раз быстрее или
  медленнее в зависимости от того, дружит ли она с тем, как устроена память. Это не
  фокусы — это понимание машины.
- **Фундамент.** Языки и фреймворки меняются каждый год. Принцип «как байт лежит в
  памяти и кто его туда положил» не меняется десятилетиями. Это знание не устаревает.

Не нужно держать всё это в голове сейчас — просто знай, *куда* мы идём. Этот модуль —
вход в здание.

## Идея 3. Схема «железа»: что есть внутри компьютера и как куски общаются

Прежде чем идти глубже в ПО, полезно один раз нарисовать машину как систему.
CS:APP (Bryant & O'Hallaron, глава 1) даёт такую схему:

```
+------+     шина     +-------+     шина     +------+
| CPU  |<============>|  RAM  |<============>| disk |
|      |              |       |              |  I/O |
| ALU  |              | DRAM  |              |      |
| reg  |              | (ОЗУ) |              |      |
| PC   |              |       |              |      |
+------+              +-------+              +------+
   |
   |  шина расширения / I/O-шина
   +--------> контроллер монитора, USB, сеть…
```

**Ключевые части и их роли:**

- **CPU (процессор)** — единственное место, где что-то *считается*. Внутри —
  **ALU** (арифметико-логическое устройство, делает +, –, AND, OR, …),
  **регистры** (сверхбыстрые ячейки прямо на кристалле, штук 16–32, каждая
  64 бита) и **PC** (program counter / счётчик команд — указывает, какую
  инструкцию исполнить следующей).

- **RAM (ОЗУ)** — оперативная память, гигабайты. CPU не может «смотреть» туда
  напрямую: данные нужно *загрузить* в регистр, поработать, *сохранить* обратно.
  Каждый байт в RAM имеет **адрес** — число от 0 до N.

- **Шина** — общая «дорога», по которой CPU, RAM и устройства обмениваются
  данными, адресами и управляющими сигналами. Одновременно по шине может ехать
  только одно «письмо», поэтому шина — потенциальное узкое место.

- **Диск / I/O** — медленные, но дешёвые устройства. Диск хранит данные, когда
  питание выключено; монитор/клавиатура/сеть — точки входа/выхода.

**ПОЧЕМУ это знать.** Когда твоя программа читает переменную — она путешествует:
диск → RAM → кэш → регистр → ALU. Каждый переход стоит времени. Это не
абстракция — это физика, которую ты будешь чувствовать, когда займёшься темой
«кэш-локальность» в модуле 08.

## Идея 4. Иерархия памяти: скорость против цены

Вот самое важное практическое знание об архитектуре, которое влияет на скорость
*твоего конкретного* кода. Разные виды памяти отличаются по скорости в сотни и
тысячи раз — и одновременно по цене и ёмкости.

```
            быстро, дорого, мало
                     ▲
         регистры    │  ~16–32 × 64 бит,  ~0.3 нс
        ─────────────┤
         L1-кэш      │  ~32 КБ,           ~1 нс
        ─────────────┤
         L2-кэш      │  ~256 КБ,          ~4 нс
        ─────────────┤
         L3-кэш      │  ~8 МБ,            ~10 нс
        ─────────────┤
         RAM (ОЗУ)   │  ~8–32 ГБ,         ~100 нс
        ─────────────┤
         SSD/NVMe    │  ~512 ГБ–4 ТБ,     ~50 мкс
        ─────────────┤
         HDD         │  ~4–16 ТБ,         ~10 мс
                     ▼
            медленно, дёшево, много
```

**Ключевые выводы для программиста:**

1. **Разрыв колоссальный.** Регистр — в ~300 раз быстрее RAM. HDD — в ~30 000 раз
   медленнее RAM. Это не нюансы, это порядки величины.

2. **Кэши — компромисс.** Мы хотим скорость регистров и ёмкость диска. Кэши —
   промежуточная «полка»: CPU держит там недавно использованные данные. Если данные
   нашлись в кэше — это **cache hit**, быстро. Не нашлись (первый раз или вытеснили) —
   **cache miss**, данные тащатся из RAM, долго.

3. **Принцип локальности.** Кэш работает, потому что программы склонны снова и снова
   обращаться к *тем же* адресам (**временна́я локальность**) и к *соседним* адресам
   (**пространственная локальность**). Если пишешь цикл по массиву подряд — кэш доволен.
   Прыгаешь по случайным адресам — кэш плачет.

4. **Скорость ≠ количество операций.** Два цикла с одинаковым числом умножений могут
   отличаться по времени в 10 раз, потому что один «дружит» с кэшом, а другой нет.
   Мы измерим это в модуле 08.

```c
// Пример: оба цикла делают N умножений, но…
// Дружит с кэшом: идём по строкам (подряд в памяти)
for (int i = 0; i < N; i++)
    for (int j = 0; j < N; j++)
        sum += a[i][j];

// Не дружит: идём по столбцам (каждый шаг — «прыжок» в памяти)
for (int j = 0; j < N; j++)
    for (int i = 0; i < N; i++)
        sum += a[i][j];
```

На большой матрице первый вариант будет быстрее второго в несколько раз — не потому
что он «умнее», а потому что кэш-линии CPU читаются блоками и первый вариант их
использует, а второй выбрасывает.

## Идея 5. Три большие абстракции операционной системы

Аппаратура — дикая, разнообразная, не умеет «делиться». Операционная система —
это слой, который прячет детали железа за тремя ключевыми абстракциями. Понять их —
значит понять, зачем вообще нужна ОС.

### Абстракция 1: процесс

**Процесс** — это работающая программа плюс всё, что ей нужно: текущие значения
регистров, собственная память, открытые файлы. ОС создаёт **иллюзию**, что твоя
программа владеет всем процессором и всей памятью целиком, хотя на одной машине
работают десятки программ одновременно.

Как? Через **переключение контекста**: каждые несколько миллисекунд ОС сохраняет
состояние одного процесса (все регистры, PC) в память, загружает состояние другого
и отдаёт ему процессор. Это происходит тысячи раз в секунду — ты этого не видишь.

```c
// fork() — создать копию текущего процесса
pid_t pid = fork();
if (pid == 0) {
    // мы — дочерний процесс
} else {
    // мы — родительский, pid — ID дочернего
}
```

### Абстракция 2: виртуальная память

Каждый процесс думает, что у него есть **свои** адреса от 0 до 2⁶⁴. Адрес `0x7fff...`
в одном процессе и тот же адрес в другом — это **разные** физические ячейки RAM.
ОС (совместно с железом — через MMU, Memory Management Unit) автоматически переводит
виртуальные адреса в физические.

Это даёт две вещи:
- **Изоляцию**: процесс не может случайно (или намеренно) испортить память другого.
- **Гибкость**: процесс может использовать больше памяти, чем есть RAM — «лишнее»
  ОС временно сбрасывает на диск (swap).

```
Адресное пространство процесса (упрощённо, снизу вверх):
  [ kernel (ядро ОС)    ]  — только ОС
  [ стек (stack)        ]  ← растёт вниз, локальные переменные, адреса возврата
  [ ...                 ]
  [ куча (heap)         ]  ← растёт вверх, malloc/free
  [ данные (.bss, .data)]  — глобальные и статические переменные
  [ код (.text)         ]  — байты инструкций
```

### Абстракция 3: файл

**Файл** — это последовательность байт с именем. За этой простой идеей ОС прячет
разнообразие устройств: жёсткий диск, SSD, терминал, сетевой сокет, труба (`pipe`)
— всё это открывается и читается через единый интерфейс `open / read / write / close`.

Идея называется «всё есть файл» (UNIX) — один из принципов, сделавших UNIX таким
живучим. Не нужно знать, что за устройство за файловым дескриптором: одинаковые
вызовы работают для всего.

```c
int fd = open("data.txt", O_RDONLY);
char buf[128];
ssize_t n = read(fd, buf, sizeof(buf)); // читает с диска
close(fd);

// Тот же интерфейс — для стандартного ввода:
ssize_t n2 = read(STDIN_FILENO, buf, sizeof(buf)); // читает с клавиатуры
```

**ПОЧЕМУ три именно такие.** Все три абстракции — ответ на один вопрос: *как
совместно использовать ресурсы, не ломая друг другу жизнь?* Процессы делят CPU,
виртуальная память делит RAM, файлы делят устройства I/O. CAOS-курс — это разбор
каждой из этих абстракций изнутри.

## Идея 6. Линковка изнутри: разрешение символов и релокация

В «Идее 1» мы прошли конвейер на уровне «что происходит». Теперь про этап 4
(линковку) чуть глубже — потому что ошибки линкера сбивают с толку дольше всего.

### Объектный файл: что внутри

После компиляции каждого `.c` файла получается `.o` — объектный файл. Он содержит:
- **секцию `.text`** — байты машинных команд этого файла;
- **секцию `.data`** — инициализированные глобальные переменные;
- **секцию `.bss`** — неинициализированные глобальные переменные (занимают ноль байт в файле);
- **таблицу символов** — список имён функций и переменных, объявленных или используемых в этом файле.

```bash
nm warmup.o          # показывает символы: T = код, U = неопределённый (нужен извне)
objdump -d warmup.o  # дизассемблировать (увидеть машинные команды)
```

### Шаг 1: разрешение символов (symbol resolution)

Линкер собирает все `.o`-файлы и проходит по каждому неопределённому символу (`U`
в `nm`). Для каждого ищет, в каком файле этот символ *определён* (`T`). Если нашёл —
связывает. Если нет — **ошибка: `undefined reference to 'foo'`**.

```
файл a.o:  определяет bar(), использует baz()    ← baz = U
файл b.o:  определяет baz(), использует bar()    ← bar = U
линкер:    bar.U ← a.o,  baz.U ← b.o  → ОК
```

### Шаг 2: релокация (relocation)

Пока мы в `.o`-файле, все адреса — *относительные* (от начала секции) или вовсе
заглушки. Линкер знает, куда какую секцию положит в итоговом исполняемом файле,
и **переписывает** все «дыры» реальными адресами. Это и называется релокацией.

```c
// В warmup.c зовём printf() — адрес printf в libc нам неизвестен при компиляции.
// Линкер вставит правильный адрес или запись в PLT (таблицу динамической линковки).
printf("hello\n");
```

Вот почему `undefined reference` — это линкерная ошибка, а не компиляторная: при
компиляции достаточно *знать, что функция существует* (это заголовок). При линковке
нужно *найти тело функции* и поставить на него адрес.

```
Компилятор видит:   int add(int, int);   ← объявление, ОК
Линкер ищет:        add(…) { … }        ← тело. Нет тела → ошибка.
```

**Практическое правило:** если видишь `undefined reference to 'foo'`, не смотри в
заголовки — ищи, добавлен ли в сборку `.c`-файл с телом `foo`. Для стандартных
библиотек (math: `-lm`, pthreads: `-lpthread`) — проверь флаги компоновки.

## Идея 7. Путь программы с диска в процессор: загрузка и запуск

Итак, исполняемый файл готов. Что происходит, когда ты пишешь `./program`?

1. **Shell зовёт `execve`** — системный вызов к ОС: «загрузи и запусти этот файл».
2. **ОС читает ELF-заголовок** файла. ELF (Executable and Linkable Format) — стандарт
   формата исполняемых файлов в Linux. Там написано: «секция `.text` лежит вот тут,
   положи её в виртуальный адрес 0x401000; стек сделай там».
3. **ОС создаёт адресное пространство** (виртуальную память) нового процесса,
   «отображает» секции файла на адреса (через механизм page mapping — не копирует
   весь файл сразу, только при первом обращении).
4. **CPU передаёт управление на `_start`** — крошечный код runtime, который
   инициализирует стек, передаёт аргументы командной строки и зовёт `main()`.
5. `main()` **работает**. Когда возвращает значение, runtime зовёт `exit()`, ОС
   уничтожает процесс и освобождает ресурсы.

```
файл на диске          память процесса
─────────────          ──────────────
ELF header         →   (служебное)
.text     ─────────→   виртуальный адрес 0x401000…  (код)
.data     ─────────→   виртуальный адрес 0x404000…  (данные)
(нет в файле)      →   стек  ← создаёт ОС
(нет в файле)      →   куча  ← выделяет malloc
```

**Зачем это нужно знать.** Отладчик (`gdb`), профилировщик (`perf`), анализ
падений (core dump) — всё это работает именно с этой картиной памяти. Когда
программа «segfault» — это значит, что CPU пытался обратиться по адресу, которого
нет в таблице отображений процесса. ОС это видит и убивает процесс.

## Идея 8. Цикл build → test → fix и что такое GoogleTest

В этом курсе ты пишешь код на C, а проверяют его **тесты** на C++. Не пугайся двух
языков: ты тесты не пишешь и не трогаешь — они уже готовы и служат «эталоном правильного
поведения». Твоя работа — сделать так, чтобы они стали зелёными.

**Тест** — это маленькая программа, которая зовёт твою функцию и сверяет результат с
ожидаемым. **GoogleTest** — популярный фреймворк для таких проверок. Выглядит так:

```cpp
TEST(Warmup, AddBasic) {
    EXPECT_EQ(add(2, 3), 5);   // ожидаю, что add(2, 3) вернёт 5
}
```

- `EXPECT_EQ(a, b)` — проверить, что `a == b`. Если нет — тест «красный», но проверка
  идёт дальше (другие `EXPECT_` в этом же тесте тоже выполнятся).
- `ASSERT_EQ(a, b)` — то же, но при провале **обрывает** тест на месте.

Когда проверка не прошла, GoogleTest печатает, *что ожидалось* и *что получилось по
факту* — это и есть твоя обратная связь. Не угадывай, читай вывод.

**ПОЧЕМУ через тесты, а не «запусти и посмотри глазами».** Глаз устаёт, забывает
проверить крайние случаи (а ноль? а отрицательное?), и не повторишь проверку завтра.
Тест проверяет всё и всегда одинаково, за долю секунды. Это и есть цикл, в котором живёт
любой инженер:

> **build** (собрать) → **test** (прогнать тесты) → **fix** (починить) → снова build…

Пока есть красный тест — работа не закончена. Этот ритм мы повторим десятки раз за курс,
и нулевой модуль — чтобы ты прошёл его впервые без отвлечения на сложную тему.

## Идея 9. Программа и данные в памяти — это просто байты

Вот, пожалуй, главная мысль всего курса, и хорошо встретить её сразу. В памяти
компьютера **нет** «чисел», «букв» или «команд» как отдельных сущностей. Там есть только
**байты** — ячейки, каждая хранит число от 0 до 255. Всё остальное — это *договорённость*,
как эти байты толковать.

- Байты `01001000 01101001` — это число? Или текст «Hi»? Или часть машинной команды?
  **Зависит от того, как мы условились их читать.** Сами байты молчат.
- Твоя функция `add` после компиляции — это тоже просто байты в памяти, лежащие рядом
  с байтами данных, над которыми она работает. Код и данные сделаны из одного материала.

**Бытовая аналогия.** Представь длинную полку с пронумерованными ячейками, в каждой —
число. Одна и та же ячейка может быть «возрастом», «кодом буквы» или «шагом инструкции» —
смысл задаёт не ячейка, а тот, кто её читает. Процессор — очень быстрый, но абсолютно
буквальный читатель: он делает ровно то, что сказано в текущей ячейке-команде, и берёт
данные ровно из тех ячеек, на которые ему указали.

**ПОЧЕМУ это знать с самого начала.** Потому что почти каждая тема дальше — это вариация
на эту мысль: «как именно мы условились толковать байты». Представление чисел, символы,
указатели (адреса тех самых ячеек), стек вызовов — всё отсюда. Сегодня просто запомни:
**в памяти лежат байты; смысл им придаёт интерпретация.** Мы будем возвращаться к этому
постоянно.

### Задание 3. Байты, структуры и порядок: смотрим на память своими глазами

Это задание — прямое продолжение «Идеи 9»: раз в памяти только байты, давай
*буквально* посмотрим на них. Три небольших функции, каждая открывает одну грань
того, как C раскладывает данные по адресам.

Объявления уже в `include/warmup.h`, реализуй их в `src/warmup.c`:

```c
/* Возвращает размер структуры, вычисленный через offsetof + sizeof последнего поля.
   Результат должен совпадать с sizeof(struct PackedRecord).
   Встретишься с padding — выравниванием байт компилятором. */
size_t packed_record_size(void);

/* Раскладывает 32-битное целое по байтам в порядке хранения в памяти.
   out[0] — байт по меньшему адресу. На little-endian машине для 0x01020304
   out = {0x04, 0x03, 0x02, 0x01}. */
void uint32_to_bytes(uint32_t value, unsigned char out[4]);

/* Возвращает 1, если машина little-endian, 0 — если big-endian.
   Реализуй через union или указатель — без __BYTE_ORDER__ и системных макросов. */
int is_little_endian(void);
```

Структура для `packed_record_size` объявлена в том же заголовке:

```c
struct PackedRecord {
    uint8_t  a;   /* 1 байт */
    uint32_t b;   /* 4 байта — но компилятор может вставить дырку перед ним! */
    uint16_t c;   /* 2 байта */
};
```

### Подсказки к заданию 3

<details><summary>Подсказка 1 — с чего начать (packed_record_size)</summary>

Прежде чем писать код, ответь себе на вопрос: какой адрес имеет поле `b` внутри
структуры? Скорее всего, не 1 (сразу после `a`). Почему — смотри в `offsetof`.
Попробуй написать на бумаге, сколько байт займёт каждое поле с учётом «дырок»,
а потом проверь через `sizeof` в отдельной маленькой программке.
</details>

<details><summary>Подсказка 2 — uint32_to_bytes и порядок байт</summary>

Задача: не вычислять сдвигами `(value >> 8) & 0xFF`, а именно *прочитать из памяти*.
У тебя есть `value` — это `uint32_t`, 4 байта, лежит в памяти прямо сейчас.
Возьми адрес этой переменной, приведи к `unsigned char *` и скопируй 4 байта подряд.
Что получишь? Зависит от того, little- или big-endian у тебя машина. Именно это и
будет наблюдением.
</details>

<details><summary>Подсказка 3 — is_little_endian через union</summary>

`union` — это объект, в котором несколько полей занимают *одно и то же место* в
памяти. Заведи `union { uint32_t i; unsigned char b[4]; }`, положи в `i` значение 1.
Посмотри, что в `b[0]`. Если там 1 — значит, младший байт лежит по меньшему адресу.
Это и есть определение little-endian. Можно сделать то же самое через указатель:
`unsigned char *p = (unsigned char *)&i; return p[0] == 1;`
</details>
