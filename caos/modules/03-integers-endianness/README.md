# Модуль 03 — Целые: переполнение, порядок байт, расширение знака

В прошлых модулях мы смотрели, как числа **записаны** в битах. Теперь — что
происходит, когда эти биты **не вмещают** ответ, когда одно и то же число
**по-разному лежит** в памяти двух машин, и как «дорастить» короткое число до
длинного, не сломав знак.

Это не абстракция. Ровно на этих трёх вещах ломаются программы, которые читают
файлы, ходят в сеть и общаются с железом. Здесь ты впервые увидишь, что у целых
чисел в C есть **острые края**, и научишься их не резать.

## Идея 1. Переполнение: знаковое против беззнакового

«Переполнение» — это когда результат не помещается в тип. Аналогия: одометр в
машине. На пятизначном табло после `99999` идёт `00000` — лишняя цифра просто
теряется. С целыми числами в компьютере — то же самое, но есть важная развилка.

- **Беззнаковое (`unsigned`)** переполнение определено стандартом: всё считается
  **по модулю** 2ⁿ. Для `uint8_t` (8 бит, диапазон 0..255):
  ```c
  uint8_t x = 255;
  x = x + 1;   // == 0, гарантированно. Просто «крутится» как одометр.
  ```
  Это законно и предсказуемо. Иногда этим даже пользуются специально.

- **Знаковое (`signed`)** переполнение — **неопределённое поведение (UB)**.
  ```c
  int32_t x = INT32_MAX;  // 2147483647
  x = x + 1;              // UB! НЕ «станет INT32_MIN» — формально «что угодно».
  ```
  Стандарт C не обещает тут *ничего*. Компилятор вправе предположить, что такого
  не бывает, и выкинуть твою проверку, переставить код, выдать любой результат.

**ПОЧЕМУ это важно.** Классический баг: ты пишешь `if (a + b < a) переполнилось();`,
рассчитывая поймать переполнение по «обёртке». Для знаковых это UB, и компилятор,
зная, что UB «не случается», может вычеркнуть весь `if`. Проверять переполнение
надо **до** сложения, не выходя за границы типа. Именно это ты и сделаешь в первом
задании.

> Подводный камень: в C маленькие типы при арифметике сначала «продвигаются» до
> `int` (integer promotion). Поэтому `uint8_t + uint8_t` считается в `int`, а
> «обёртка по модулю» случится только при обратном присваивании в `uint8_t`.

## Идея 2. Порядок байт (endianness)

Число `0x01020304` занимает 4 байта: `01`, `02`, `03`, `04`. Вопрос на миллион:
**в каком порядке** они лежат в памяти, от младшего адреса к старшему?

- **Big-endian** («тупоконечный») — сначала старший байт: `01 02 03 04`. Так
  пишут люди слева направо, так же исторически устроены сетевые протоколы.
- **Little-endian** («остроконечный») — сначала младший байт: `04 03 02 01`. Так
  работают x86 и почти все современные процессоры, на которых ты сидишь.

Аналогия: дату можно записать как `2026-06-19` (от крупного к мелкому — big) или
как `19.06.2026` (от мелкого к крупному — little). Число одно, порядок «цифр в
памяти» разный.

**ПОЧЕМУ это важно.** Пока число живёт внутри одной программы — без разницы, она
сама себя понимает. Беда начинается, когда байты **выходят наружу**: в файл, в
сетевой пакет, на диск. Машина-отправитель little-endian записала `04 03 02 01`,
машина-получатель big-endian прочитала это как `0x04030201` — и получила мусор.
Поэтому в сетях принят единый «сетевой порядок» (big-endian), а программы
**переставляют байты** на входе и выходе. Этим и заняты задания `swap16`/`swap32`.

```text
uint32_t v = 0x01020304;   адреса:  +0   +1   +2   +3
little-endian в памяти:             04   03   02   01
big-endian   в памяти:              01   02   03   04
```

## Идея 3. Расширение знака

У тебя есть 8-битное знаковое число и нужно положить его в 32-битную переменную.
Просто дописать нули слева нельзя: `0xFF` как 8-битное знаковое — это `-1`, а если
дополнить нулями до `0x000000FF`, получится `+255`. Знак потеряется.

Правильный приём — **расширение знака (sign extension)**: смотрим на старший
(знаковый) бит короткого числа и дублируем его во все добавленные старшие биты.

```text
 8-битное  0x80  = 1000 0000  (старший бит = 1, значит число -128)
расширяем до 32:  1111 1111 1111 1111 1111 1111 1000 0000  = 0xFFFFFF80 = -128 ✔

 8-битное  0x7F  = 0111 1111  (старший бит = 0, число +127)
расширяем до 32:  0000 0000 ... 0111 1111                  = 0x0000007F = +127 ✔
```

**ПОЧЕМУ это важно.** Процессор делает это автоматически, когда ты приводишь
`int8_t` к `int`, и ты обычно не замечаешь. Но как только ты сам распаковываешь
данные из байтов (формат файла, дизассемблер, эмулятор, протокол с полями по
N бит) — расширять знак приходится руками. Забудешь — отрицательные значения
тихо превратятся в большие положительные. Задание `sign_extend` — про этот ровно
момент.

## Идея 4. Преобразования типов: явные и скрытые ловушки (CS:APP §2.2.3–2.2.6)

### Явное приведение (каст) между signed и unsigned

Когда ты пишешь `(unsigned int)x` или `(int)u`, битовый паттерн в памяти
**не меняется** — меняется только то, как C его интерпретирует.

```c
int32_t  a = -1;
uint32_t b = (uint32_t)a;  // битовый паттерн тот же: 0xFFFFFFFF
                            // но значение теперь 4294967295u, а не -1
```

Формально: если `x` — знаковое значение в диапазоне `[0, INT32_MAX]`, то
`(uint32_t)x == x` — значение не изменилось. Если `x < 0`, то по правилу C
добавляется `2^32`: `(uint32_t)(-1) == UINT32_MAX`, `(uint32_t)(-128) == 2^32 - 128`.

В обратную сторону: если беззнаковое `u > INT32_MAX`, то `(int32_t)u`
**implementation-defined** по стандарту C. На практике x86 просто
переинтерпретирует те же биты как дополнение до двух. Но рассчитывать на это
в переносимом коде нельзя.

### Усечение при сужении (truncation)

Когда значение больше не помещается в меньший тип, лишние старшие биты просто
отбрасываются. Остаётся всего `n` младших бит:

```c
uint32_t big = 0x12345678u;
uint16_t small = (uint16_t)big;  // 0x5678 — старшие 16 бит пропали
uint8_t  tiny  = (uint8_t)big;   // 0x78  — остался только последний байт
```

Для знаковых: если результат усечения в старшем бите оказывается `1`, число
«переходит» в отрицательное:

```c
uint32_t u = 0x180u;  // 384 — не помещается в uint8_t (>255)
uint8_t  x = (uint8_t)u;   // 0x80 = 128 как unsigned, или -128 как signed
int8_t   s = (int8_t)u;    // -128: 0x80 в 8-битном знаковом = -128
```

Вот почему «обрезать» длинный сетевой адрес или размер до маленького типа —
классическая ошибка: число тихо «переворачивается».

### Неявные преобразования в смешанных выражениях

Это самая коварная группа. Когда в одном выражении встречаются разные целочисленные
типы, C автоматически применяет **integer promotion** и **usual arithmetic conversions**.

**Integer promotion:** типы `char`, `short`, `int8_t`, `int16_t` и их беззнаковые
аналоги сначала «повышаются» до `int` (или `unsigned int`, если не помещаются
в `int`). Это происходит **до** любой операции:

```c
uint8_t a = 200, b = 100;
// Оба a и b продвигаются до int, сложение — в int: 300 (не «обернётся» в uint8_t!)
int result = a + b;  // 300, а не 44

uint8_t c = a + b;   // 300 & 0xFF = 44 — усечение при присваивании
```

**Usual arithmetic conversions (смешанные выражения):** если типы операндов
разные, C «поднимает» меньший к большему по иерархии: `int → unsigned int → long →
unsigned long → long long → unsigned long long`. Ключевое правило:

> Если один операнд `unsigned`, а другой `signed` **того же или меньшего ранга**,
> знаковый приводится к беззнаковому — тихо и без предупреждений.

```c
int           a = -1;
unsigned int  b = 1u;
// В выражении a + b: a приводится к unsigned: (unsigned)(-1) == UINT_MAX
// Результат: UINT_MAX + 1u == 0 (переполнение беззнакового, wraparound)
unsigned int r = a + b;  // 0 — не 0 ли ожидал ты? Правильно, но по другой причине.
```

### Знаковое расширение (sign extension) — механика

Мы уже видели знаковое расширение в Идее 3. Теперь стоит объяснить, **как**
C делает это автоматически при расширении знакового типа:

```c
int8_t  s8  = -5;       // 0xFB в 8 битах: 1111 1011
int16_t s16 = s8;       // 0xFFFB: 1111 1111 1111 1011  ← знак расширен автоматически
int32_t s32 = s8;       // 0xFFFFFFFB — все старшие биты 1, значение всё равно -5
```

А вот беззнаковое расширение (zero extension) — просто дополняет нулями:

```c
uint8_t  u8  = 0xFBu;   // 251 в 8 битах
uint32_t u32 = u8;      // 0x000000FB = 251 — нули, значение сохранено
```

Проблема возникает при смешении: если расширить `uint8_t` до `int32_t`, дополнение
нулями сохраняет значение (251 → 251). Но если сначала сделать каст `uint8_t → int8_t`,
а потом расширить знаком — значение уже другое (251 → -5 → -5). Порядок кастов имеет
значение.

### Почему сравнение signed/unsigned опасно

Это, пожалуй, самый популярный источник тихих багов в C:

```c
int   count = -1;
unsigned int limit = 10u;

if (count < limit) {    // ЛОВУШКА!
    // Перед сравнением count приводится к unsigned:
    // (unsigned)(-1) == UINT_MAX == 4294967295 > 10
    // Условие ЛОЖНО — ты ожидал истину!
    printf("не попадёшь сюда\n");
}
```

Другой классический сценарий — цикл с беззнаковым счётчиком «вниз»:

```c
for (unsigned int i = 10; i >= 0; i--) {
    // Когда i == 0, i-- даёт UINT_MAX (wraparound)
    // Условие i >= 0 ВСЕГДА истинно — бесконечный цикл!
}
```

Ещё один вариант: функции типа `strlen` возвращают `size_t` (беззнаковый). Если
ты сравниваешь результат с отрицательным числом:

```c
int n = -1;
if (strlen(s) > n) { ... }  // n → (size_t)(-1) == SIZE_MAX, всегда истина
```

**Правило:** при сравнении знакового и беззнакового явно приводи к нужному типу
(предварительно проверив диапазон) или используй только один семейства типов.

> Компилятор предупреждает о таких сравнениях флагом `-Wsign-compare` (входит
> в `-Wall`). Включай предупреждения и читай их — они экономят часы отладки.

## Идея 5. Зачем это всё на практике

Три идеи выше — не школьная экзотика, а ежедневная реальность системного кода:

- **Сеть.** Заголовки TCP/IP хранят длины и порты в big-endian. Прочитал порт без
  перестановки байт на little-endian машине — соединился не туда. Функции
  `htons`/`ntohl` из реального кода делают ровно твои `swap16`/`swap32` (но
  только если машина того требует).
- **Файлы.** PNG, ZIP, ELF, шрифты — у каждого формата зафиксирован порядок байт.
  Парсер обязан его соблюдать, иначе читает мусор.
- **Безопасность.** Целочисленное переполнение при подсчёте размера буфера —
  это классическая дыра: посчитал размер, он «обернулся» в маленькое число,
  выделил мало памяти, записал много — переполнение буфера.

Понимая эти три механизма, ты перестаёшь бояться «странных» багов в бинарных
данных и начинаешь их предвидеть.

### Задание 6. Сериализация сетевого заголовка (network byte order)

Сетевые протоколы хранят числа в **big-endian** («сетевой порядок байт» — network
byte order, NBO). Когда твоя программа отправляет структуру через сокет, она обязана
записать поля в проводной формат: старший байт первым. Если забыть — другая машина
прочтёт мусор.

Структура `PacketHeader` описывает заголовок пакета:

```c
typedef struct {
    uint16_t version;    /* 2 байта */
    uint16_t port;       /* 2 байта */
    uint32_t length;     /* 4 байта */
    uint32_t checksum;   /* 4 байта */
} PacketHeader;          /* итого 12 байт в памяти; 12 байт на проводе (2+2+4+4) */
```

Реализуй два преобразования:

```c
/* Кодирует заголовок pkt в big-endian байты в buf.
   buf должен быть не менее 12 байт.
   Возвращает 12 при успехе; -1, если buf == NULL или buf_size < 12. */
int packet_encode(const PacketHeader *pkt, uint8_t *buf, size_t buf_size);

/* Декодирует big-endian байты из buf обратно в *out.
   Возвращает 12 при успехе; -1, если buf == NULL, out == NULL или buf_size < 12. */
int packet_decode(const uint8_t *buf, size_t buf_size, PacketHeader *out);
```

На проводе каждое многобайтовое поле записывается **big-endian**: старший байт
лежит по меньшему адресу. Это именно то, что делают `htons`/`htonl` в реальном
сетевом коде.

#### Подсказки (открывай по одной, только если застрял)

<details><summary>Шаг 1 — как записать uint16_t в big-endian вручную?</summary>

Возьми `uint16_t v`. Старший байт: `(v >> 8) & 0xFF`. Младший: `v & 0xFF`.
В big-endian старший идёт первым: `buf[0] = старший`, `buf[1] = младший`.
Не используй `memcpy` напрямую — ты не знаешь endian-ность машины. Используй сдвиги.
</details>

<details><summary>Шаг 2 — как обобщить на uint32_t?</summary>

Для `uint32_t v` четыре байта, от старшего к младшему:
`(v >> 24) & 0xFF`, `(v >> 16) & 0xFF`, `(v >> 8) & 0xFF`, `v & 0xFF`.
Запиши их последовательно в `buf[i]`…`buf[i+3]`.
</details>

<details><summary>Шаг 3 — как декодировать обратно?</summary>

Декодирование — обратная операция: берёшь байты из буфера и
собираешь значение сдвигами влево и ИЛИ:
`uint16_t v = ((uint16_t)buf[0] << 8) | buf[1];`
Аналогично для `uint32_t` — четыре байта, сдвиги на 24, 16, 8, 0.
</details>

1. `int add_overflows(int32_t a, int32_t b);` — вернуть `1`, если знаковое
   сложение `a + b` переполнит `int32_t`, иначе `0`. **Проверяй до сложения** —
   не вычисляй `a + b` в `int32_t` (это и было бы UB).
2. `uint16_t swap16(uint16_t x);` — обменять местами два байта: `0xAABB → 0xBBAA`.
3. `uint32_t swap32(uint32_t x);` — развернуть порядок четырёх байт:
   `0xAABBCCDD → 0xDDCCBBAA`.
4. `int32_t sign_extend(uint32_t x, int bits);` — расширить знак `bits`-битного
   значения (лежит в младших битах `x`) до 32 бит. `bits` в диапазоне `1..32`.
5. `int is_little_endian(void);` — вернуть `1`, если машина little-endian, иначе `0`.
6. `int packet_encode(const PacketHeader*, uint8_t*, size_t);` и
   `int packet_decode(const uint8_t*, size_t, PacketHeader*);` — сериализовать и
   десериализовать заголовок пакета в big-endian. Описание и подсказки — в

### Подсказки (открывай, только если застрял)

<details><summary>Задание 1 — как поймать переполнение, не переполняясь</summary>

Думай про границы. Если `b > 0`, то результат «вылезет вверх» ровно когда
`a > INT32_MAX - b` (это вычитание безопасно: `INT32_MAX - b` точно в диапазоне,
ведь `b` положительно). Если `b < 0`, симметрично: вылезет «вниз», когда
`a < INT32_MIN - b`. Если `b == 0` — переполнения быть не может. Все сравнения —
до всякого сложения.
</details>

<details><summary>Задание 2 — swap16 через сдвиги и маски</summary>

Старший байт — это `(x >> 8)`. Младший байт — это `(x << 8)`. Сложи их через `|`.
Следи за типом: считай в `uint16_t`/`unsigned` и при необходимости обрежь
результат до 16 бит (`& 0xFFFF`), чтобы лишние биты не «всплыли».
</details>

<details><summary>Задание 3 — swap32 по байтам</summary>

Достань каждый из четырёх байт маской `0xFF` и сдвигом (`x & 0xFF`,
`(x >> 8) & 0xFF`, …), потом расставь их на зеркальные позиции сдвигами влево.
Байт 0 уезжает на позицию 3 (сдвиг на 24), байт 1 — на позицию 2 (сдвиг на 8),
и наоборот. Собери всё через `|`. Пиши суффикс `u` у констант: `0xFFu`.
</details>

<details><summary>Задание 4 — расширение знака одной формулой</summary>

Есть короткий приём. Пусть `m = 1u << (bits - 1)` — это маска знакового бита.
Сначала обрежь `x` до нужной ширины: `x = x & ((bits == 32) ? 0xFFFFFFFFu :
((1u << bits) - 1))`. Тогда `(int32_t)((x ^ m) - m)` даёт расширение знака:
если знаковый бит был установлен, вычитание «протащит» единицы в старшие разряды.
Подумай, почему `bits == 32` стоит обработать отдельно (что делает `1u << 32`?).
</details>

<details><summary>Задание 5 — как машина сама себя выдаёт</summary>

Положи в память `uint32_t` со значением `1` и посмотри на **самый первый байт**.
Если там `1` — младший байт лежит первым, это little-endian. Удобно сделать через
`union { uint32_t i; unsigned char c[4]; }` или через приведение адреса к
`unsigned char*`. Никаких сдвигов — здесь вопрос именно про раскладку в памяти.
</details>
