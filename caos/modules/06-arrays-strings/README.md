# Модуль 06 — Массивы, си-строки и безопасность буферов

Здесь мы знакомимся с тем, как в C устроен **текст**. В Python строка — это
самостоятельный объект, который знает свою длину, умеет сам расти и за всем следит. В
C такой роскоши нет: строка — это просто кусок памяти, и **вся ответственность на
тебе**. Звучит страшно, но именно отсюда растёт половина громких уязвимостей в истории
(переполнения буфера). Понять этот модуль — значит понять, *почему* такие баги вообще
возможны, и научиться их не делать.

## Большая идея

В C нет отдельного «типа строка». Есть **массив `char`** (массив байтов-символов), а то,
где строка заканчивается, помечает специальный байт — **нулевой байт `'\0'`** (число 0,
не символ `'0'`!). Все функции работы со строками — `strlen`, `printf`, `strcmp` —
просто идут по памяти от начала и **останавливаются, когда видят `'\0'`**.

Отсюда два следствия, которые надо прочувствовать:

1. Если `'\0'` нет — функция не остановится и пойдёт читать чужую память (баг или краш).
2. Массив **не знает свой размер** — поэтому размер буфера приходится передавать руками.

Аналогия: представь поезд из вагонов-символов. `'\0'` — это табличка «конец состава».
Пока проводник идёт по вагонам, он смотрит, не висит ли табличка. Забыл повесить
табличку — проводник пойдёт в соседний поезд и дальше в депо. Вот и весь «ужас» C-строк.

## Идея 1. Строка в C — это массив `char` с нулевым байтом на конце

```c
char s[] = "hi";   // в памяти лежит: 'h' 'i' '\0'  — ТРИ байта, не два!
```

Литерал `"hi"` компилятор сам дополняет завершающим `'\0'`. Длина строки — это число
символов **до** `'\0'`, сам `'\0'` в длину не входит. Поэтому под строку из 5 видимых
символов нужен массив минимум на 6 `char`.

**Почему это важно знать:** когда ты сам выделяешь буфер под строку, всегда закладывай
+1 байт под `'\0'`. Классическая ошибка новичка — `char buf[5]` под слово из пяти букв:
букв влезет пять, а табличке «конец» места не останется.

Обход строки выглядит так — идём, пока не упрёмся в `'\0'`:

```c
size_t i = 0;
while (s[i] != '\0') {
    /* s[i] — очередной символ */
    i++;
}
// здесь i == длина строки
```

## Идея 2. Чем опасны `strcpy`/`gets`: переполнение буфера

Старые функции `strcpy(dst, src)` и `gets(dst)` копируют/читают, **пока не встретят
`'\0'`** (или конец строки) — и им **всё равно**, сколько места в `dst`.

```c
char buf[8];
strcpy(buf, "это очень длинная строка");  // КАТАСТРОФА
```

`buf` рассчитан на 8 байт, а копируется ~25. Лишние байты лягут **за границей массива** —
поверх соседних переменных, а то и поверх служебных данных на стеке (например, адреса
возврата из функции). Это и есть **переполнение буфера (buffer overflow)**.

**Почему это важно знать:** это не абстракция. Подсунув в такую программу специально
подобранную «слишком длинную строку», атакующий может перезаписать адрес возврата и
заставить процессор прыгнуть в свой код. Так работали целые классы эксплойтов
(Morris worm, Code Red и многие другие). Функцию `gets` за это в итоге **вычеркнули из
стандарта C** — ею просто нельзя пользоваться безопасно в принципе.

Аналогия: `strcpy` — это как наливать воду в стакан с закрытыми глазами, полагаясь
только на то, что в кувшине «когда-нибудь кончится вода». Если кувшин больше стакана —
зальёшь весь стол.

## Идея 3. Работа в границах буфера; почему размер передают явно

Лекарство простое: **всегда знай размер буфера и не выходи за него**. Раз массив сам
свой размер не помнит, мы передаём его параметром — отсюда и `n` в `f(char* buf, size_t cap)`.

```c
void f(char* dst, size_t cap, const char* src);
//                  ^^^ сколько ВСЕГО байт в dst, включая место под '\0'
```

Безопасная запись соблюдает два правила:

- пишем не больше `cap - 1` символов (один байт резервируем под `'\0'`);
- в конце **обязательно** ставим `'\0'`, даже если пришлось обрезать.

**Почему это важно знать:** обрезать строку неприятно, но это контролируемо и безопасно.
Уехать за границу буфера — это undefined behavior: может «работать», может молча портить
данные, может рухнуть в самый неподходящий момент. Лучше предсказуемая обрезка, чем
непредсказуемый хаос. Именно так устроены «безопасные» аналоги: `strncpy`, `snprintf`,
`strlcpy` — все они принимают размер.

## Идея 4. Обход массива (и как ходить «с двух концов»)

Большинство операций со строками — это аккуратный проход по индексам. Иногда вперёд
(посчитать длину, сравнить), иногда **навстречу друг другу с двух концов** (развернуть):

```c
size_t i = 0, j = n - 1;
while (i < j) {
    /* поменять местами s[i] и s[j] */
    i++;
    j--;
}
```

Два бегунка стартуют с краёв и сходятся к центру. Когда `i` и `j` встретились —
работа сделана. Это базовый приём, он встретится тебе ещё много раз (и в C, и дальше).

**Почему это важно знать:** умение точно ходить по индексам, не вылезая за `[0, n-1]`, —
это и есть «безопасность буферов» на практике. Один лишний шаг `j` за край — и ты снова
читаешь/пишешь чужую память.

## Задания

Открой `src/cstrings.c`. Там четыре функции-заглушки (компилируются, но возвращают
ерунду — поэтому тесты красные). Реализуй их по контракту. Интерфейс — в
`include/cstrings.h`, эталон поведения — в `tests/test_cstrings.cpp` (трогать тесты не нужно).

1. `size_t my_strlen(const char* s);`
   Длина строки — число символов до `'\0'`.

2. `size_t safe_copy(char* dst, size_t cap, const char* src);`
   Скопировать **не более `cap - 1`** символов из `src` в `dst`, **всегда** завершить
   результат `'\0'` (если `cap > 0`), и вернуть длину `src`. По возвращённой длине
   вызывающий поймёт, было ли усечение (усечение, если результат `>= cap`).

3. `void reverse_string(char* s);`
   Развернуть строку **на месте** (in-place), без второго буфера. `'\0'` остаётся на конце.

4. `int my_strcmp(const char* a, const char* b);`
   Сравнить как `strcmp`: вернуть `<0`, `0` или `>0`.

### Подсказки (открывай, только если застрял)

<details><summary>Задание 1 — my_strlen</summary>

Заведи счётчик `size_t len = 0`. Иди по строке, пока `s[len] != '\0'`, увеличивая `len`.
Когда цикл закончился — `len` и есть ответ. `'\0'` не считаем.
</details>

<details><summary>Задание 2 — safe_copy, с чего начать</summary>

Сначала отдельно посчитай длину `src` (можешь вызвать свой `my_strlen`) — её ты вернёшь
в конце, **независимо** от того, влезло всё или нет. Это и есть «сигнал об усечении».
</details>

<details><summary>Задание 2 — safe_copy, само копирование</summary>

Если `cap == 0` — писать некуда вообще, просто верни длину `src` и ничего не трогай.
Иначе копируй символы в цикле, пока выполнены **оба** условия: не дошли до `'\0'` в `src`
И уже записано меньше `cap - 1` символов. После цикла поставь `dst[written] = '\0'`.
Так ты гарантированно не вылезешь за буфер и всегда завершишь строку.
</details>

<details><summary>Задание 3 — reverse_string</summary>

Сначала узнай длину `n` (тем же приёмом, что в задании 1). Потом два индекса:
`i = 0` и `j = n - 1`. Пока `i < j`: поменяй местами `s[i]` и `s[j]` через временную
переменную, затем `i++`, `j--`. Пустую строку (`n == 0`) трогать не нужно — цикл сам
не выполнится. `'\0'` остаётся на своём месте, его не двигаем.
</details>

<details><summary>Задание 4 — my_strcmp</summary>

Иди по обеим строкам одновременно индексом `i`. Пока символы совпадают И текущий символ
не `'\0'` — двигайся дальше (`i++`). Как только символы различаются ИЛИ дошёл до конца —
верни **разность** этих двух символов. Важная тонкость: приводи символы к
`unsigned char` перед вычитанием — настоящий `strcmp` сравнивает байты как беззнаковые
(тест `UnsignedCharSemantics` это проверяет). Если строки равны, цикл дойдёт до `'\0'`
в обеих, и разность будет 0.
</details>
