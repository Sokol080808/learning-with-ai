# Модуль 06 — Массивы, си-строки и безопасность буферов

Здесь мы знакомимся с тем, как в C устроен **текст** и как в памяти лежат **массивы**.
В Python строка — это самостоятельный объект, который знает свою длину, умеет сам расти
и за всем следит. В C такой роскоши нет: строка — это просто кусок памяти, и **вся
ответственность на тебе**. Звучит страшно, но именно отсюда растёт половина громких
уязвимостей в истории (переполнения буфера). Понять этот модуль — значит понять,
*почему* такие баги вообще возможны, и научиться их не делать.

Мы опираемся на CS:APP §3.8 (массивы) и §3.10 (атаки через стек).

## Большая идея

В C нет отдельного «типа строка». Есть **массив `char`** (массив байтов-символов), а то,
где строка заканчивается, помечает специальный байт — **нулевой байт `'\0'`** (число 0,
не символ `'0'`!). Все функции работы со строками — `strlen`, `printf`, `strcmp` —
просто идут по памяти от начала и **останавливаются, когда видят `'\0'`**.

Отсюда два следствия, которые надо прочувствовать:

1. Если `'\0'` нет — функция не остановится и пойдёт читать чужую память (баг или краш).
2. Массив **не знает свой размер** — поэтому размер буфера приходится передавать руками.

Аналогия: представь поезд из вагонов-символов. `'\0'` — это табличка «конец состава».
Пока проводник идёт по вагонам, он смотрит, не висит ли табличка. Забыл повесить
табличку — проводник пойдёт в соседний поезд и дальше в депо. Вот и весь «ужас» C-строк.

## Идея 0. Как массив живёт в памяти: row-major и адресная арифметика

Прежде чем перейти к строкам, разберёмся с тем, как C вообще раскладывает массивы в
памяти — это фундамент для понимания всего остального.

### Одномерный массив

Объявление `int a[5]` резервирует в памяти **5 последовательных** `int`-ячеек.
Адрес элемента с индексом `i`:

```
адрес(a[i]) = base + i * sizeof(int)
```

Именно поэтому обращение `a[i]` в C — это сокращение для `*(a + i)`:
компилятор вычисляет смещение сам, пользуясь размером элемента.

```c
int a[5] = {10, 20, 30, 40, 50};
// a[2] == 30; его адрес == (char*)a + 2*sizeof(int)
int* p = a + 2;   // p указывает ровно на a[2]
// *p == 30
```

### Двумерный массив: row-major раскладка

Объявление `int m[3][4]` — это **3 строки по 4 столбца**. В памяти все элементы лежат
**подряд, строка за строкой** (row-major order, CS:APP §3.8). Никаких «таблиц» в памяти
нет — только непрерывный кусок из `3 * 4 = 12` ячеек:

```
m[0][0] m[0][1] m[0][2] m[0][3]  m[1][0] m[1][1] ...  m[2][3]
|<--------- строка 0 ---------->| |<--- строка 1 --->| ...
```

Адрес произвольного элемента `m[i][j]` при `ncols = 4`:

```
адрес(m[i][j]) = base + (i * ncols + j) * sizeof(int)
```

Развернём для примера: `m[2][1]` при `ncols = 4`:
смещение = `(2 * 4 + 1) * 4` = `36` байт от начала массива.

```c
int m[3][4];
// m[i][j] == *(*(m + i) + j) == *((int*)m + i*4 + j)
// Обе записи эквивалентны.
```

**Почему это важно знать:** именно из row-major раскладки растут все рассуждения о
кэш-дружелюбности (следующий модуль), а также все ошибки «вышел за строку и попал в
следующую». Если обходить двумерный массив по столбцам (неправильный порядок), кэш-промахи
умножатся и программа станет в разы медленнее.

**Многомерный случай** обобщается естественно: каждое новое измерение добавляет уровень
умножения. Для `T a[d0][d1][d2]` адрес элемента `a[i][j][k]`:

```
base + (i*d1*d2 + j*d2 + k) * sizeof(T)
```

### Строка как одномерный массив `char`

C-строка — это в точности одномерный массив `char` с нулевым байтом-ограничителем.
Та же адресная арифметика, только `sizeof(char) == 1`, поэтому смещение `i`-го символа
равно просто `i` байт от начала.

## Идея 1. Строка в C — это массив `char` с нулевым байтом на конце

```c
char s[] = "hi";   // в памяти лежит: 'h' 'i' '\0'  — ТРИ байта, не два!
```

Литерал `"hi"` компилятор сам дополняет завершающим `'\0'`. Длина строки — это число
символов **до** `'\0'`, сам `'\0'` в длину не входит. Поэтому под строку из 5 видимых
символов нужен массив минимум на 6 `char`.

**Почему это важно знать:** когда ты сам выделяешь буфер под строку, всегда закладывай
+1 байт под `'\0'`. Классическая ошибка новичка — `char buf[5]` под слово из пяти букв:
букв влезет пять, а табличке «конец» места не останется.

Обход строки выглядит так — идём, пока не упрёмся в `'\0'`:

```c
size_t i = 0;
while (s[i] != '\0') {
    /* s[i] — очередной символ */
    i++;
}
// здесь i == длина строки
```

## Идея 2. Чем опасны `strcpy`/`gets`: переполнение буфера

Старые функции `strcpy(dst, src)` и `gets(dst)` копируют/читают, **пока не встретят
`'\0'`** (или конец строки) — и им **всё равно**, сколько места в `dst`.

```c
char buf[8];
strcpy(buf, "это очень длинная строка");  // КАТАСТРОФА
```

`buf` рассчитан на 8 байт, а копируется ~25. Лишние байты лягут **за границей массива** —
поверх соседних переменных, а то и поверх служебных данных на стеке (например, адреса
возврата из функции). Это и есть **переполнение буфера (buffer overflow)**.

**Почему это важно знать:** это не абстракция. Подсунув в такую программу специально
подобранную «слишком длинную строку», атакующий может перезаписать адрес возврата и
заставить процессор прыгнуть в свой код. Так работали целые классы эксплойтов
(Morris worm, Code Red и многие другие). Функцию `gets` за это в итоге **вычеркнули из
стандарта C** — ею просто нельзя пользоваться безопасно в принципе.

Аналогия: `strcpy` — это как наливать воду в стакан с закрытыми глазами, полагаясь
только на то, что в кувшине «когда-нибудь кончится вода». Если кувшин больше стакана —
зальёшь весь стол.

## Идея 3. Работа в границах буфера; почему размер передают явно

Лекарство простое: **всегда знай размер буфера и не выходи за него**. Раз массив сам
свой размер не помнит, мы передаём его параметром — отсюда и `n` в `f(char* buf, size_t cap)`.

```c
void f(char* dst, size_t cap, const char* src);
//                  ^^^ сколько ВСЕГО байт в dst, включая место под '\0'
```

Безопасная запись соблюдает два правила:

- пишем не больше `cap - 1` символов (один байт резервируем под `'\0'`);
- в конце **обязательно** ставим `'\0'`, даже если пришлось обрезать.

**Почему это важно знать:** обрезать строку неприятно, но это контролируемо и безопасно.
Уехать за границу буфера — это undefined behavior: может «работать», может молча портить
данные, может рухнуть в самый неподходящий момент. Лучше предсказуемая обрезка, чем
непредсказуемый хаос. Именно так устроены «безопасные» аналоги: `strncpy`, `snprintf`,
`strlcpy` — все они принимают размер.

### `strcpy` vs `strncpy`: в чём разница и почему оба небезопасны по-своему

```c
// ОПАСНО: не знает о размере dst
char dst[8];
strcpy(dst, src);              // уедет за буфер, если strlen(src) >= 8

// Лучше, но с ловушкой: если src не влезает, '\0' НЕ добавляется!
strncpy(dst, src, sizeof(dst)); // без гарантии '\0' при обрезке
// Всегда добавляй руками после strncpy, если не уверен:
dst[sizeof(dst) - 1] = '\0';

// Ещё лучше — snprintf: всегда завершает '\0', возвращает исходную длину
snprintf(dst, sizeof(dst), "%s", src);
```

Запомни порядок аргументов: `strncpy(dst, src, n)` и `snprintf(dst, n, fmt, ...)` — `n` это
**весь** размер буфера, а не «сколько копировать». Это частая путаница, приводящая к
off-by-one.

## Идея 3.5. Переполнение буфера и атаки через стек (CS:APP §3.10)

Этот раздел — о том, почему переполнение буфера не просто «программа падает», а
**полноценный вектор атаки**. Нужно понять механику, чтобы писать безопасный код осознанно.

### Как выглядит кадр стека

Когда функция вызывается, процессор создаёт на стеке **кадр (stack frame)**. Упрощённо
(x86-64, стек растёт вниз):

```
высокий адрес
 ┌──────────────────────┐
 │  аргументы (если >6) │
 ├──────────────────────┤
 │  адрес возврата (RA) │  ← сохранённый %rip, куда вернуться
 ├──────────────────────┤
 │  сохранённый %rbp    │
 ├──────────────────────┤
 │  локальные переменные│
 │  (в т.ч. char buf[]) │
 └──────────────────────┘
низкий адрес (в сторону роста стека)
```

Ключевой момент: **буфер `buf` находится ниже адреса возврата**, но массив растёт вверх
(в сторону старших адресов). Значит, если записать в `buf` слишком много байтов — запись
«поднимется» по стеку и **перезапишет адрес возврата**.

### Что делает атакующий

```c
void vulnerable(const char* user_input) {
    char buf[64];
    strcpy(buf, user_input); // <-- нет проверки длины!
    // ... обработка buf
}
```

Если `user_input` длиннее 64 байт, лишние байты перезапишут:
1. сохранённый `%rbp` (фрейм-поинтер),
2. **адрес возврата** — вот тут начинается магия.

Атакующий специально подбирает строку так, чтобы байты на позиции адреса возврата
указывали на его **шеллкод** (вставленный туда же, в переполненный буфер, или в другое
место). Когда функция завершается (`ret`), процессор прыгает не в вызывающий код, а в
код атакующего. Именно так работал **червь Морриса (1988)** и сотни последующих эксплойтов.

### Защитные механизмы (современные системы)

Сегодня операционная система и компилятор активно борются с таким классом атак:

| Механизм | Как работает |
|---|---|
| **Stack canary** (`-fstack-protector`) | Компилятор вставляет «канарейку» — случайное значение между буфером и RA. Перед `ret` проверяет: если канарейка изменилась — сигнал и аварийное завершение. |
| **ASLR** (Address Space Layout Randomization) | ОС загружает стек, кучу, библиотеки по случайным адресам при каждом запуске. Атакующий не знает, куда прыгать. |
| **NX / W^X** (non-executable stack) | Страницы памяти помечаются либо «записываемые», либо «исполняемые», но не оба сразу. Шеллкод в стеке — не выполнится. |
| **ROP mitigations** | Против обходов NX через цепочки «гаджетов» (Return-Oriented Programming) существуют CFI и shadow stack. |

Эти механизмы не делают переполнение буфера безвредным — они лишь **поднимают планку** для
атакующего. Правильный ответ — **не допускать переполнения в коде**.

### Out-of-bounds: тихий или громкий?

Выход за границу массива — это **undefined behavior** в C. Компилятор вправе
предположить, что этого никогда не происходит, и выбросить или трансформировать код
любым образом. На практике это выглядит так:

```c
int a[4] = {1, 2, 3, 4};
int x = a[4];  // UB: чтение за границей
int y = a[-1]; // UB: отрицательный индекс
a[10] = 42;    // UB + потенциальная запись в соседние переменные
```

Читай об этом как о **трёх возможных исходах**:
- Программа продолжает работать с «мусорными» данными (самое опасное — нет сигнала об ошибке).
- Программа падает с SIGSEGV (хотя бы заметно).
- Программа делает что-то непредсказуемое из-за оптимизаций компилятора.

**Инструменты, которые ловят UB во время разработки:**
- `clang -fsanitize=address,undefined` — AddressSanitizer + UBSan: ловят out-of-bounds,
  use-after-free, переполнения типов.
- `valgrind --tool=memcheck` — альтернатива, без перекомпиляции, но медленнее.

## Идея 4. Обход массива (и как ходить «с двух концов»)

Большинство операций со строками — это аккуратный проход по индексам. Иногда вперёд
(посчитать длину, сравнить), иногда **навстречу друг другу с двух концов** (развернуть):

```c
size_t i = 0, j = n - 1;
while (i < j) {
    /* поменять местами s[i] и s[j] */
    i++;
    j--;
}
```

Два бегунка стартуют с краёв и сходятся к центру. Когда `i` и `j` встретились —
работа сделана. Это базовый приём, он встретится тебе ещё много раз (и в C, и дальше).

**Почему это важно знать:** умение точно ходить по индексам, не вылезая за `[0, n-1]`, —
это и есть «безопасность буферов» на практике. Один лишний шаг `j` за край — и ты снова
читаешь/пишешь чужую память.

## Задание 5 — `StrBuf`: ограниченный строковый буфер (продвинутое)

Все четыре функции выше работают с **сырыми** C-строками: буфер снаружи, размер снаружи,
забота о `'\0'` снаружи. Настоящие библиотеки берут это на себя, пряча детали в
**структуру**. В этом задании ты построишь простейший вариант такой абстракции.

Интерфейс, который нужно реализовать (объявлен в `include/cstrings.h`):

```c
typedef struct {
    char*  buf;   /* указатель на выделенный буфер                  */
    size_t len;   /* текущая длина строки (без '\0')                 */
    size_t cap;   /* ёмкость: сколько байт выделено (включая '\0')  */
} StrBuf;

/* Инициализировать: выделить буфер на cap байт через malloc.
 * buf[0] = '\0', len = 0. Если cap == 0 или malloc вернул NULL — buf = NULL. */
void strbuf_init(StrBuf* sb, size_t cap);

/* Дописать строку s в конец буфера sb.
 * Вписать не более (cap - 1 - len) символов (место до конца без '\0').
 * Всегда NUL-терминировать результат.
 * Вернуть фактически дописанное число байт (может быть меньше strlen(s) при обрезке). */
size_t strbuf_append(StrBuf* sb, const char* s);

/* Сбросить содержимое в пустую строку (buf[0] = '\0', len = 0).
 * Память не освобождать. */
void strbuf_clear(StrBuf* sb);

/* Освободить buf через free, обнулить все поля. */
void strbuf_free(StrBuf* sb);
```

### Подсказки (открывай, только если застрял)

<details><summary>С чего начать: какие поля нужны и зачем</summary>

Подумай: зачем нам три поля вместо одного указателя?
- `buf` — туда пишем символы;
- `cap` — мы не можем знать в `strbuf_append`, сколько места вообще выделено, если не
  хранить это число; именно поэтому его кладут в структуру;
- `len` — чтобы не пересчитывать длину при каждом `append` за O(n): мы знаем, куда
  дописывать — это `buf + len`.

Попробуй нарисовать на бумаге состояние структуры после `init(sb, 8)`, затем после
`append(sb, "hello")`, затем после `append(sb, " world")`.
</details>

<details><summary>strbuf_append: как посчитать, сколько байт влезает</summary>

Свободного места в буфере — `cap - 1 - len` байт (вычти 1 под `'\0'` и уже занятое).
Скопируй не более этого числа символов из `s`, начиная с позиции `buf + len`.
После копирования обязательно поставь `buf[len + copied] = '\0'` и обнови `len`.

Что вернуть? Ровно `copied` — сколько байт реально дописали.
</details>

<details><summary>strbuf_free: почему важно обнулить поля после free</summary>

После `free(buf)` указатель `buf` становится «висячим» (dangling pointer): память
освобождена, но адрес в переменной остался старый. Случайное обращение через него —
undefined behavior. Хороший приём: сразу после `free` присвоить `buf = NULL`, `len = 0`,
`cap = 0`. Тогда повторный `free(NULL)` безопасен (стандарт гарантирует no-op), и
случайное разыменование даст SIGSEGV вместо тихого повреждения данных.
</details>
