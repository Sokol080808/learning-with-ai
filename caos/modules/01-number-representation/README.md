# Модуль 01 — Представление чисел: двоичная, шестнадцатеричная, дополнительный код

## Большая идея

Внутри компьютера **нет** цифр «5» или «−3». Есть только провода, в каждом из которых
ток либо есть, либо нет — то есть **0 или 1**. Один такой провод — это **бит**. Всё
остальное (числа, буквы, картинки, твой код) — это договорённость о том, *как читать*
группы битов.

В этом модуле мы учимся читать и записывать числа так же, как их «видит» железо.
Зачем это знать: без этого ты не поймёшь, почему `int` переполняется и становится
отрицательным, почему адреса памяти пишут как `0x7ffd`, почему «сдвиг влево» — это
умножение на 2, и откуда берутся загадочные `-1 == 0xFFFFFFFF`. Это фундамент, на
котором стоит весь курс.

## Идея 1. Позиционные системы: основание 2 и 16

Ты уже умеешь читать числа в системе с основанием **10**. Запись `253` означает:

```
2·100 + 5·10 + 3·1  =  2·10² + 5·10¹ + 3·10⁰
```

Каждая позиция — это степень десятки, а цифры берутся из набора `0..9`. Ничего
священного в десятке нет — просто у нас десять пальцев. Можно взять **любое**
основание.

**Двоичная система (основание 2).** Цифры только `0` и `1`, позиции — степени двойки:

```
1011₂  =  1·2³ + 0·2² + 1·2¹ + 1·2⁰  =  8 + 0 + 2 + 1  =  11
```

Бытовая аналогия: представь набор гирь номиналом 1, 2, 4, 8, 16, … кг. Бит — это
ответ «класть гирю или нет». `1011` = «гирю 8 — да, 4 — нет, 2 — да, 1 — да» = 11 кг.
Любое число собирается из таких гирь ровно одним способом.

**Шестнадцатеричная (основание 16).** Цифр уже 16: `0..9`, дальше `A=10, B=11, C=12,
D=13, E=14, F=15`. Позиции — степени шестнадцати:

```
1A₁₆  =  1·16 + A·1  =  1·16 + 10  =  26
```

**Почему вообще нужен hex.** Двоичная запись честная, но длинная и нечитаемая:
`11011110` глазами не схватишь. Но 16 = 2⁴, поэтому **каждая hex-цифра — это ровно
4 бита** (одна «тетрада», по-английски *nibble*). Перевод бесплатный, без арифметики:

```
1101 1110₂   ->   D    E     ->  0xDE
(13)  (14)
```

Поэтому адреса, байты, цвета (`#FF8800`) пишут в hex: компактно и при этом группы
битов видны на глаз. Префикс `0x` — это просто маркер «дальше шестнадцатеричное»
(как `0b` для двоичного).

## Идея 2. Беззнаковые и знаковые целые

У числа в железе есть **ширина** — сколько битов под него отведено: 8, 16, 32, 64.
Это как количество окошек на табло: больше окошек — больше разных значений
помещается, но оно конечно.

**Беззнаковое** (`unsigned`, у нас `uint32_t`) — все биты идут на само число, знака
нет. На `n` битах помещаются значения от `0` до `2ⁿ − 1`. Для 8 бит это `0..255`.

Что будет, если прибавить 1 к максимуму? Табло «перещёлкивает» через ноль —
как одометр в машине, который после `999999` показывает `000000`. Это называется
**переполнение по модулю** (wrap-around), и для `unsigned` это не ошибка, а
определённое поведение: `255 + 1 == 0` для 8-битного.

А вот как на тех же битах хранить **отрицательные** числа — об этом следующая идея.
Зачем тебе это: перепутать знаковость — классический баг. `unsigned a = 0; a - 1`
даёт не `−1`, а огромное число, и цикл `for (unsigned i = n; i >= 0; i--)` крутится
вечно.

## Идея 3. Дополнительный код, или почему вычитание — это сложение

Нам нужно хранить отрицательные числа теми же битами. Наивная идея «отдать один бит
под знак» (как «−» перед числом) работает плохо: получаются *два* нуля (`+0` и `−0`),
и сложение приходится делать по особым правилам. Железо так не любит.

Реальные компьютеры используют **дополнительный код** (англ. *two's complement*).
Правило для `n` бит: чтобы получить запись числа `−x`, берём `2ⁿ − x`.

Возьмём 8 бит (`2⁸ = 256`). Число `−1` хранится как `256 − 1 = 255 = 11111111₂`.
Число `−2` — как `254 = 11111110₂`. Старший бит при этом всегда `1` у отрицательных
и `0` у неотрицательных — удобный «флажок знака», но это *следствие*, а не сама суть.

Зачем такая странность — вот ради чего. Проверим `5 + (−5)` на 8 битах:

```
   00000101   (5)
 + 11111011   (−5, это 256−5 = 251)
 -----------
  100000000   (= 256)
```

Девятый бит «убегает» за пределы 8 окошек и просто теряется (тот самый wrap-around).
Остаётся `00000000` — то есть `0`. **Сошлось!** И это не совпадение: вычитание `a − b`
железо считает как `a + (−b)`, обычным сложением. Не нужна отдельная схема для
вычитания — один и тот же сумматор. Вот зачем человечество выбрало дополнительный код:
он делает арифметику со знаком *бесплатной*.

Полезный фокус: чтобы инвертировать число в дополнительном коде, **переверни все биты
и прибавь 1**. Проверь на `1`: `00000001` → инверсия `11111110` → `+1` → `11111111` =
`−1`. Работает.

## Идея 4. Диапазоны при разной ширине типа

Ширина задаёт «потолок и пол». Для знаковых дополнительный код делит значения почти
пополам: половина под неотрицательные, половина под отрицательные (отрицательных на
одно больше — нуль «занимает место» на положительной стороне).

| Тип        | Бит | Беззнаковый диапазон      | Знаковый диапазон (доп. код)         |
|------------|-----|---------------------------|--------------------------------------|
| `int8_t`   | 8   | 0 … 255                   | −128 … 127                           |
| `int16_t`  | 16  | 0 … 65 535                | −32 768 … 32 767                     |
| `int32_t`  | 32  | 0 … 4 294 967 295         | −2 147 483 648 … 2 147 483 647       |
| `int64_t`  | 64  | 0 … ~1.8·10¹⁹             | ~−9.2·10¹⁸ … ~9.2·10¹⁸               |

Формула под рукой: знаковый `n`-битный тип хранит `−2ⁿ⁻¹ … 2ⁿ⁻¹ − 1`, беззнаковый —
`0 … 2ⁿ − 1`.

Почему это важно практически: выбор типа — это выбор диапазона. Счётчик пользователей
в `int16_t` сломается на 32 768-м. Хранить размер файла в `int32_t` — и файл больше
2 ГБ «уйдёт в минус». Понимание границ заранее экономит тебе ночные дебаги.

## Идея 5. Порядок байт (endianness): big-endian и little-endian

Тип `uint32_t` занимает **4 байта** в памяти. Вопрос: в каком порядке эти 4 байта
лежат по адресам? Представь число `0x12345678`. Его старший байт — `0x12`, младший —
`0x78`.

**Big-endian («старший вперёд»):** байты идут от старшего к младшему, как цифры
слева направо в книге:

```
адрес:  base+0  base+1  base+2  base+3
байт:    0x12    0x34    0x56    0x78
```

**Little-endian («младший вперёд»):** байты идут от младшего к старшему — именно так
устроены все современные процессоры x86/x64 и большинство ARM:

```
адрес:  base+0  base+1  base+2  base+3
байт:    0x78    0x56    0x34    0x12
```

Запомни мнемонику: little-endian — это как если бы ты записывал число «задом наперёд»
побайтово. На практике это не мешает, пока ты работаешь в одной системе, — процессор
сам читает правильно. Проблема возникает **на границах**: сеть, файловые форматы
(PNG, ELF, TCP/IP) используют big-endian (его называют *network byte order*), а твой
x86-компьютер — little-endian. Значит, при отправке числа по сети нужно явно переставить
байты.

Как узнать порядок байт своей машины прямо в коде?

```c
#include <stdint.h>

int host_is_little_endian(void) {
    uint32_t x = 1;          /* биты: ...00000001 */
    uint8_t  b = *(uint8_t *)&x;  /* читаем самый первый байт */
    return b == 1;            /* 1 => little-endian, 0 => big-endian */
}
```

Ещё элегантнее — через `union`:

```c
int host_is_little_endian(void) {
    union { uint32_t u32; uint8_t b[4]; } probe = { .u32 = 1 };
    return probe.b[0] == 1;
}
```

Оба способа законны в C11 (чтение `union` через другой член — defined behavior,
в отличие от C++). `union`-вариант принято считать более идиоматичным.

## Идея 6. Битовые операторы C: побитовые vs логические

В C есть **два** разных набора операторов, которые часто путают:

| Оператор | Категория   | Что делает                          |
|----------|-------------|-------------------------------------|
| `&`      | побитовый   | AND над каждым битом                |
| `\|`     | побитовый   | OR над каждым битом                 |
| `^`      | побитовый   | XOR над каждым битом                |
| `~`      | побитовый   | инверсия всех битов                 |
| `&&`     | логический  | AND: 0 или 1 (short-circuit)        |
| `\|\|`   | логический  | OR: 0 или 1 (short-circuit)         |
| `!`      | логический  | NOT: 0 или 1                        |

Побитовые работают с **каждым битом** независимо:

```c
uint8_t a = 0b11001010;   /* 0xCA */
uint8_t b = 0b10110011;   /* 0xB3 */

uint8_t and_res = a & b;  /* 0b10000010 = 0x82 */
uint8_t or_res  = a | b;  /* 0b11111011 = 0xFB */
uint8_t xor_res = a ^ b;  /* 0b01111001 = 0x79 */
uint8_t not_a   = ~a;     /* 0b00110101 = 0x35 */
```

Логические работают со **всем значением** как с булевым (0 = ложь, всё остальное = истина)
и возвращают строго 0 или 1:

```c
int x = 5, y = 0;
int p = x && y;  /* 0 — потому что y == 0 */
int q = x || y;  /* 1 — потому что x != 0 */
int r = !x;      /* 0 — потому что x != 0 */
```

Классическая ловушка: `if (flags & 0x04)` — проверяет один бит; `if (flags && 0x04)` —
всегда истина (потому что `0x04 != 0`). Именно побитовый `&` нужен для масок.

**Сдвиги (`<<` и `>>`)** тоже битовые операции, но с особенностями:

- `x << k` — сдвиг влево на `k` бит, эквивалентно умножению на `2ᵏ` (при отсутствии
  переполнения).
- `x >> k` — сдвиг вправо на `k` бит. Для `unsigned` это **логический** сдвиг: старшие
  биты заполняются нулями; эквивалентно целочисленному делению на `2ᵏ`. Для знакового
  `int` поведение **зависит от компилятора**: большинство реализаций делают
  **арифметический** сдвиг (заполнение знаковым битом), но стандарт C11 оставляет это
  implementation-defined. Вывод: сдвигай вправо знаковые числа осторожно, лучше
  кастуй к `unsigned` или используй деление.

```c
int32_t neg = -8;         /* 0xFFFFFFF8 в доп. коде */
int32_t ars  = neg >> 2;  /* -2 на большинстве платформ (арифм.) */
uint32_t lrs = (uint32_t)neg >> 2;  /* 0x3FFFFFFE — логический */
```

## Идея 7. Преобразования типов, знаковое расширение и ловушки

### Расширяющий каст (widening)

Когда значение переходит в **более широкий** тип, происходит следующее:

- `unsigned` → `unsigned` шире: старшие биты заполняются нулями — ок.
- `signed` → `signed` шире: старшие биты заполняются **знаковым битом** (sign extension) —
  число сохраняется. Например, `int8_t` `−1` (`0xFF`) → `int32_t` `−1` (`0xFFFFFFFF`).
- `signed` → `unsigned` (любой ширины): результат берётся по модулю `2ⁿ`. `-1` →
  `UINT32_MAX`. Это определённое поведение, но часто неожиданное.

```c
int8_t   s8 = -1;          /* 0xFF */
int32_t  s32 = s8;         /* 0xFFFFFFFF = -1   (знаковое расширение) */
uint32_t u32 = s8;         /* 0xFFFFFFFF = 4294967295 (по модулю 2^32) */
```

### Смешивание signed и unsigned

Когда в одном выражении встречаются `signed` и `unsigned`, C **молча** приводит
`signed` к `unsigned` (стандарт: «обычные арифметические преобразования»). Это источник
коварных багов:

```c
int a = -1;
unsigned b = 1;
if (a < b)     /* ЛОЖЬ! -1 стал огромным unsigned */
    puts("ожидаемо");
else
    puts("сюрприз");   /* выполнится это */
```

Компилятор с флагом `-Wsign-compare` предупредит. Всегда включай предупреждения и
относись к ним серьёзно.

### Знаковое переполнение — это неопределённое поведение (UB)

Переполнение `unsigned` — корректно: wrap-around по модулю `2ⁿ`. Переполнение
`signed` — **undefined behavior** (UB) по стандарту C11. Компилятор вправе считать,
что оно **никогда не происходит**, и оптимизировать код исходя из этого допущения:

```c
int x = INT_MAX;
x += 1;  /* UB — компилятор может выбросить проверку "x > 0" как "всегда истинную" */
```

Реальный пример: цикл `for (int i = 0; i < i + 1; i++)` при `-O2` может стать
бесконечным — оптимизатор «знает», что `i` не переполнится, а значит условие всегда
истинно. Используй беззнаковые типы или `__builtin_add_overflow` для безопасной
проверки.

### Асимметрия INT_MIN

Самое отрицательное знаковое число — `INT_MIN` = `−2ⁿ⁻¹`. Его модуль на 1 **больше**,
чем `INT_MAX` = `2ⁿ⁻¹ − 1`. Почему? Потому что ноль занимает место на «положительной»
стороне. Следствия:

```c
int x = INT_MIN;
int y = -x;   /* UB! |INT_MIN| не помещается в int */
int z = abs(INT_MIN);  /* тоже UB, результат зависит от реализации */
```

И ещё: `INT_MIN / -1` тоже UB — результат `INT_MIN + 1 + 1 = INT_MAX + 1`, которого
нет в `int`.

```c
#include <limits.h>
printf("%d %d\n", INT_MIN, INT_MAX);  /* -2147483648  2147483647 */
/* |INT_MIN| = 2147483648 — не помещается в int32_t */
```

Всегда проверяй крайние значения; особенно опасны алгоритмы с `abs()` и отрицанием
над `signed`.

</details>

## Задание 5 — Endianness-сериализатор

Числа нужно не только хранить, но и **передавать**: в файл, по сети, в другой процесс.
Для этого нужен контракт: в каком порядке байты числа окажутся в памяти/потоке.
Стандарт де-факто для сети — big-endian (RFC 1700 — «network byte order»); бинарные
форматы типа PNG и ELF явно указывают свой порядок. Твоя задача — написать портируемый
сериализатор для `uint32_t`, который работает правильно независимо от типа процессора.

Реализуй в `src/numrep.c` четыре функции и одну вспомогательную:

```c
void     pack_u32_be(uint32_t v, uint8_t out[4]);
uint32_t unpack_u32_be(const uint8_t in[4]);
void     pack_u32_le(uint32_t v, uint8_t out[4]);
uint32_t unpack_u32_le(const uint8_t in[4]);
int      host_is_little_endian(void);
```

Контракты:
- `pack_u32_be(v, out)` — записать `v` в `out` в **big-endian** порядке: `out[0]` =
  старший байт (`v >> 24`), `out[3]` = младший (`v & 0xFF`).
- `unpack_u32_be(in)` — обратная операция: собрать `uint32_t` из 4 байт, где `in[0]`
  — старший.
- `pack_u32_le` / `unpack_u32_le` — то же самое, но в **little-endian**: `out[0]` =
  младший байт, `out[3]` = старший.
- `host_is_little_endian()` — вернуть `1` если машина little-endian, `0` если
  big-endian. Использовать `union` или приведение указателя.

Пример правильного вывода для числа `0x12345678`:

```
BE: [0x12, 0x34, 0x56, 0x78]
LE: [0x78, 0x56, 0x34, 0x12]
```

### Подсказки (открывай, только если застрял)

<details><summary>Подсказка 1 — как достать отдельный байт</summary>

Число `uint32_t v` содержит 4 байта. Старший байт находится в битах 31..24.
Как его достать? Сдвинь вправо на 24 и возьми младший байт маскировкой:
`(uint8_t)(v >> 24)`. Следующий байт — `(uint8_t)(v >> 16)`, и так далее.
Для BE пиши их в порядке убывания: `out[0] = v >> 24`, `out[1] = v >> 16`, ...
</details>

<details><summary>Подсказка 2 — как собрать число обратно</summary>

`unpack_u32_be` — зеркало `pack_u32_be`. Каждый байт нужно поставить на своё
место, «сдвинув» его в нужную позицию и склеив через `|`. Например, `in[0]` должен
встать в старший байт результата — значит его нужно сдвинуть влево на 24:
`result = ((uint32_t)in[0] << 24) | ((uint32_t)in[1] << 16) | ...`
</details>

<details><summary>Подсказка 3 — host_is_little_endian через union</summary>

Создай `union { uint32_t u32; uint8_t b[4]; }` и запиши в `u32` значение `1`.
Теперь прочитай `b[0]`. Если `b[0] == 1`, значит младший байт числа хранится
по наименьшему адресу — это и есть little-endian. Если `b[0] == 0` — big-endian.
</details>
